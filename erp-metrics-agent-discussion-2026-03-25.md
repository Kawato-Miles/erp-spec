## Round 1 — Senior PM 初稿

### 北極星指標（初稿）

**每月在系統內完成完整生命週期的訂單數**

定義：一筆訂單從「需求單建立」到「出貨完成」的所有狀態轉換均在 ERP 系統內完成，且無任何步驟依賴 Slack 主導。

**選擇理由**：
- 商業 Phase 2 核心使命是「取代 Ragic + Slack + 紙本組合」，完整生命週期在系統內完成，強制讓所有角色真正進入系統
- 以「訂單數」為單位是業務語言，管理層可直觀理解
- 包含「出貨完成」條件，確保整個流程走完（而非只有前段）

### 被放棄的 Target（初稿）

| 指標候選 | 放棄理由 |
|---------|---------|
| 報價準確率 | Phase 1 無法量測（需完整 BOM 成本計算，Phase 1 不覆蓋）|
| NPS / 使用者滿意度 | ERP 是內部工具，受管理政策影響大，無法反映系統設計品質 |
| 出貨準時率 | 影響因素超出 ERP 範疇（供應商、師傅效率、急單），無法歸因 |
| 系統登入天數（DAU/WAU）| 虛榮指標，登入不等於解決問題 |
| 工單建立數量 | 產出指標（Output）非結果指標（Outcome）|
| Ragic 存取次數歸零時間點 | 政策決定非系統決定，指標受管理層宣布時機影響 |

### Senior PM 推導邏輯

**使用框架**：G.U.N.（Growth / User / North Star）+ 四層指標結構（目標 / 運營 / 防禦 / 生態）

**從痛點推導步驟**：
1. 先找根因：工單散落根因是「Slack 是阻力最小的路徑」；產能不透明根因是「師傅報工不在系統」；角色分工不清根因是「無系統角色綁定」
2. 找到「最難改變的行為」：工單異動走 Slack → 成為北極星的核心強制條件
3. 為每個模組設置防禦配對（目標指標 + 對應防禦指標）

## Round 2A — CEO Reviewer 審查意見

### 整體評估

> 這份指標設計的最大風險是：三個月後所有指標都綠燈，你卻仍然無法回答「這套 ERP 解決了什麼問題」。採用率不等於問題解決。

### 意見 1：北極星定義讓你在錯誤方向慶祝勝利

**問題**：業務可能在 Slack 達成共識後，事後把結果補輸系統。這類訂單照樣被計入北極星，但問題未解決。

**修正建議**：加入「無系統外補輸異動記錄」的反向條件；或用「工單異動來源為 Slack 轉述的比例」當代理指標。

### 意見 2：「報價天數縮短 20%」無 Baseline 是虛值

**問題**：沒有 Baseline 的改善目標只有兩種結果：自我感覺達成，或目標淪為裝飾品。

**修正建議**：上線前先從 Ragic / Slack 紀錄拉過去 3 個月資料建立 Baseline；或「首月建立 Baseline，第二個月起才計算縮短幅度」。

### 意見 3：「工單完成度計算準確率 100%」是技術指標放錯位置

**問題**：此指標幾乎必然達成（計算邏輯確定性），達成後也說明不了任何業務問題是否被解決。

**修正建議**：移至技術驗收清單；改用「印務每日需要人工核對或修正系統計算結果的次數」。

### 意見 4：「生管每日登入率 80%」是虛榮指標，且與你自己的「被放棄 Target」自相矛盾

**問題**：你在 Rejected Target 說「系統登入天數是虛榮指標」，然後設定了「生管每日登入率 80%」，本質相同。

**修正建議**：改為「生管透過系統分派或確認生產任務的比率」，直接衡量有效使用行為。

### 意見 5：「訂單建立放棄率 < 15%」目標方向反了

**問題**：放棄率應是診斷工具，不是 KPI 目標。設 < 15% 等於告訴設計團隊「放棄 15% 是被允許的」。

**修正建議**：改為「各步驟中斷點分布分析」；設警示機制：超過 10% 觸發設計審查。

## Round 2B — ERP Consultant 審查意見

### 整體評估

> 這份方案的主要風險集中在：（1）部分指標在技術層面缺乏可量測性；（2）指標設計與現行狀態機的對應關係沒有明確建立。

### 意見 1：北極星指標定義與狀態機衝突（EC 路徑無需求單）

**問題**：Notion 狀態機顯示 EC 訂單不經需求單，直接從「等待付款」進入工單流程。若北極星起點是「需求單建立」，EC 訂單永遠無法被計入，分母有系統性漏洞。

**修正建議**：分拆起點——線下以「訂單確認」、EC 以「付款完成」為起點；或縮限指標為「訂單狀態機走完的比率」。

### 意見 2：「工單異動在系統內完成的比率 80%」分母不清

**問題**：若異動狀態機已設計好，理論上 100% 的正式異動都應走系統。那 80% 的分母是什麼？剩下 20% 是口頭確認未回寫、直接改紙本工單，還是什麼？

**修正建議**：先定義「哪些異動可能系統外發生」，再設目標；或改用「透過系統發起工單異動次數趨勢」作為代理指標。

### 意見 3：「工單完成度計算準確率 100%」混淆計算邏輯與輸入資料品質

**問題**：計算邏輯是確定性的（輸入相同則輸出相同），不存在「準確率」語意。真正導致「不準確」的是輸入資料有問題（quantity_per_work_order 填錯、QC 入庫數漏填）。

**修正建議**：替換為（a）計算結果與人工核對一致率；（b）每月因計算異常發起系統修正次數。

### 意見 4：「訂單建立放棄率 < 15%」缺乏技術判定標準

**問題**：印刷業接單情境中，業務常多次進出訂單頁面等待規格確認，這會被誤計為「放棄」。

**修正建議**：先定義「放棄」的技術判定標準（如停留 > X 分鐘未提交）；或替換為「訂單建立平均完成時間」。

### 意見 5：「生管每日登入率 80%」是活躍度虛榮指標

**問題**：登入行為不等於有效業務操作，且與被放棄的「系統登入天數」自相矛盾。

**修正建議**：替換為「工管透過系統發起任務分派或確認異動的次數」（直接對應狀態機事件「已交付」、「已確認異動內容」）。

## 最終結論

### 北極星指標（修訂版）

**訂單流程完整完成率**

定義：在系統中從訂單起點（線下訂單：「訂單確認」；EC 訂單：「付款完成」）到「出貨完成」的完整流程均在系統內完成的訂單比率。

目標：
- 商業 Phase 2 上線後第 1 個月：≥ 60%
- 商業 Phase 2 上線後第 3 個月：≥ 80%

衡量方式：系統事件追蹤（狀態機完整走完的訂單數 ÷ 當月進入系統的訂單總數）

**北極星選擇策略邏輯**：
商業 Phase 2 的核心使命是「取代 Ragic + Slack + 紙本組合」。選擇「訂單完整走完流程的比率」而非「功能採用率」，是因為：（1）訂單是業務語言，跨角色可理解；（2）「走完完整流程」強制讓所有環節角色真正進入系統；（3）出貨完成是業務承諾兌現的時刻，不達此點不算真正解決問題。

### 被放棄的 Target（最終版）

| 指標候選 | 放棄理由 |
|---------|---------|
| 報價準確率 | Phase 1 無法量測（需完整 BOM 展算，Phase 1 不覆蓋）|
| NPS / 使用者滿意度 | ERP 是內部工具，採用受管理政策驅動，NPS 無法反映設計品質 |
| 出貨準時率 | 影響因素超出 ERP 範疇（供應商、師傅效率），無法歸因 |
| 系統登入天數（DAU/WAU）| 虛榮指標，登入不等於解決問題 |
| 工單建立數量 | 產出指標（Output）非結果指標（Outcome）|
| Ragic 存取次數歸零時間點 | 政策決定非系統決定，指標受管理層宣布時機影響 |
| 工單完成度計算準確率 100% | 技術驗收項目，非 KPI；計算邏輯確定性不存在準確率語意 |
| 訂單建立放棄率 < 15% | 缺乏技術判定標準；印刷業多次進出頁面是正常行為；改為診斷工具 |
