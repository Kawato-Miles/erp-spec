# Linear 交付標準模板（自包含）

> **本模板是精煉出的標準結構，不引用任何特定 issue / 需求作為對照**（實例會被修改或刪除即失效）。
> 產出時依此模板填空；模板更新走版本控管（見末「版本」）。
> 正本分工：wiki＝商業內容正本（概述與 Use Case 取材處）、spec＝功能行為正本（Feature 商業邏輯與驗收取材處）、**Prototype＝介面與互動正本（操作動線與版型的取材處與交付參照，不再文字重述）**、**Project＝薄的功能目錄＋商業脈絡（What）**、**Issue＝實作範圍與具體工作（How）**。
> 通篇自包含（Linear 為對開發的交付正本；wiki 為欄位與狀態正本、開發直接查閱），跨單據只以 Linear 識別碼互指，**不出現 `openspec/...` 路徑或內部檔名**。

Prototype 為準：頁面結構與版型、操作動線、元件選用與狀態呈現、欄位在介面上的位置。

發現 Prototype 與本文件不一致時：回報 PM、不自行裁決——不一致即缺陷，會修正本。

## 狀態機

### <實體> 狀態機

​```mermaid
stateDiagram-v2
    [*] --> <狀態A>: <觸發>
    <狀態A> --> <狀態B>: <觸發 / 條件>
​```
```

### 填寫要點

- **概述**：商業目標＋現行問題與痛點，取材 wiki（痛點卡、商業邏輯卡「現況」段）。不寫功能清單（那是 Scope 的事）。
- **Use Case**：業務敘事白話句、角色 × 業務動作；可跨 project 講完整業務故事。取材 wiki 業務情境。
- **Key Feature / Scope**：維度是「功能」不是模組（模組在 project 層已用完）。一個 Scope 項＝一個可獨立交付、可獨立驗收的功能，對應 spec 一至多個 Requirement。巢狀兩節點：主要功能／次要功能（次要＝可後做）。phase 標示非固定產出——僅 wiki 階段規劃或 Miles 拍板有明確階段時標註，MUST NOT 用 phase 區分 Scope / Out of Scope。
- **Key Feature 章**：每 Feature 一章，章內依 Function（觸發顆粒：一種人工操作或系統觸發）分節；單 Function 的 Feature 直接寫兩段、不立 Function 標題。每節只有**商業邏輯＋驗收條件**。
- **商業邏輯**：留「業務規則」（誰、什麼條件、發生什麼、為什麼），實作規格（演算法細節、欄位、事件結構）下放 Task 票；介面呈現與操作動線以 Prototype 參照承載、不文字重述。
- **Prototype 參照**（Feature 層）：有介面的 Feature 必附——頁面路由＋一句可實際操作的行為；無介面的純系統行為 Feature 免列。
- **Prototype 參照邊界**：整段固定文字照模板抄錄、不逐案改寫；Prototype 即介面與互動正本。
- **驗收條件**：可勾稽、二元判定；流程用「假設／當／則」情境式、約束用規則式條列。素材自 spec Scenario 轉譯；缺料記 OQ 標另案、不自編；不混通用品質門檻（Definition of Done）。
- **測試決策**：受測介面（測試觀察點）2-4 個、交付時預先議定——票層驗收測試都必須落在這些介面上，測試火力集中在關鍵路徑而非每個邊角。
- **書寫原則（全篇）**：(1) 白話完整句、業務主管唸得順，不用「術語標籤＋符號壓縮」；(2) 只寫現在式的「系統是什麼」，**禁設計對比語言**（合一 / 統一 / 取代 / 不再 / 原本）；(3) 規則本體只在一處、他處一句引用；(4) 禁中英夾雜（英文識別碼用「中文（英文）」格式）；(5) 成品禁出現內部路徑、檔名、內部問題編號。
- **不放進 Project 的**：操作方式與畫面動線（→ Prototype 參照）、實作細節（→ Task 票）、資料欄位表（→ wiki 實體卡，開發直接查閱）、職能分工區塊與票號對照（票自己指回 Project；職能由開票時指派的 team 承載）。

## 二、Task issue 描述模板（How 層）

### Task 票（按功能範圍切票；職能由開票時指派的 team 承載，模板不分職能，也不出設計票——操作方式與介面由 Prototype 承載）

```markdown
## 概述

1. 範圍可參考本 project「Key Feature：<功能名>」章（多功能票列多章）
2. 本 issue 負責 <一句範圍>

## 實作細節

<可依主題分組>
- [ ] <實作要點：API / 狀態機 / 計算規則 / 頁面 / 區塊 / 互動；自 Project 下放的演算法細節、欄位、事件結構寫在這>

## 驗收測試

> 開發依測試先行處理：每條先落地為失敗測試、再實作至通過。測試落在 Project「測試決策」章列出的受測介面；期望值取自 Project 對應 Feature 的驗收條件，不得由實作重算。

- [ ] 假設 <前置情境>，當 <動作>，則 <可觀測結果>（受測介面：<名稱>）

## Blocked by

- <阻擋票的 Linear 識別碼與票名>，或「無——可立即開工」

## 依賴與範圍

- [ ] <依賴其他模組 API：標明「只消費、不實作」>
- [ ] <Out of Scope 註記（不含項與歸屬）>
```

### Task issue 填寫要點

- 票名與 Function 用功能本名，不帶設計過程的對比修飾。
- **概述**：指回 Project「Key Feature：<功能名>」章＋一句負責範圍。不重複 Project 已寫的商業邏輯。
- **切票軸線**：按功能範圍切、不按職能分模板；職能由開票時指派的 Linear team 承載（依規先開 PM team 再轉對應 team，見 SKILL.md 操作 know-how）。
- **實作細節**：該票範圍的 How checklist，含自 Project 下放的實作規格；**不可只剩「指回 project」一句而無實作要點**。操作動線與介面呈現不寫進票——以 Project 各 Feature 的 Prototype 參照為準。
- **驗收測試**：3-5 條、每條可獨立驗證；先紅後綠（先落地失敗測試再實作）；每條標明受測介面（必須在 Project「測試決策」章已議定的清單內）；期望值取自 Project 驗收條件——獨立正本，防「用程式同樣方式重算期望值」的自我循環測試。
- **Blocked by**：開票順序 blockers-first（先開被依賴票取得識別碼，再開依賴票）；用 Linear 原生 blocking 關係建立，票內本段同步列出；無阻擋者寫「無——可立即開工」。
- **依賴註記**：跨模組依賴標「只消費不實作」；Out of Scope 項標清楚。
- 進行中（In Progress）且已有內容的 issue **不覆寫**（含既有設計票遺產），只由 project 描述引用。

## 三、狀態機規範

- 狀態密集模組（多實體各有狀態）**每個狀態機附一張 UML**；狀態列舉與轉換正本在 wiki 狀態機卡，交付圖忠實轉繪、**不畫來源未定義的轉換**（缺料記 OQ 標另案）。
- 形式：Mermaid ` ```mermaid ` + `stateDiagram-v2`；每張圖含狀態節點＋轉換箭頭＋觸發 / 條件標籤；關鍵終態 / 不可逆用 `note` 標。
- 雙維度狀態分開各畫一張；視圖層 project 沿用核心 project 的圖、不另繪，以文字交代相關轉換。
- 中文狀態代號無標點符號；轉換標籤避免半形括號（用「·」或全形）。
- 寫入 Linear 後 MUST 請 Miles 確認渲染成圖。

## 四、Milestone 與開票順序

- **Milestone＝縱向整合節點**：每個 milestone 是一條可展示的端到端路徑（例「手動分派走通」），把該路徑相關的票掛入；milestone 的先後即整合順序，讓跨票協作的先後在 Linear 介面上直接可見。
- **開票順序 blockers-first**：先開被依賴的票（取得 Linear 識別碼），再開依賴它的票並建立 Linear 原生 blocking 關係；票內「Blocked by」段同步列出。所有阻擋已完成的票即為可立即開工的前沿。
