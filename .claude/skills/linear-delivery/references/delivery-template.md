# Linear 交付標準模板（自包含）

> **本模板是精煉出的標準結構，不引用任何特定 issue / 需求作為對照**（實例會被修改或刪除即失效）。
> 產出時依此模板填空；模板更新走版本控管（見末「版本」）。
> 兩層結構：**project 描述 = 模組層 What**、**Task issue = 角色層 How**。
> 通篇自包含（Linear 為對開發的唯一正本，不依賴外部 BRD 頁面），跨單據只以 Linear 識別碼互指，**不出現 `openspec/...` 路徑或內部檔名**。

```markdown
## 概述

<一句模組定位：解決什麼問題、核心結構、取代什麼舊系統>
<一句範圍邊界（Out of Scope 摘要）：不含什麼 / 不匯入什麼歷史資料>

## 使用情境

1. <角色> 可於 <系統> 完成 <端到端流程一句話>

   2. <子情境 B>

   1. <子情境>

### Key Feature

1. **<功能名>**：<一句說明（含關鍵規則）>
2. **<功能名>**：<一句說明>

- <不做項>：<歸屬——延後批次 / 其他 project / 其他平台>

## Spec

### Design

* User flow（Figma：<連結，或指向負責設計的 Task issue>）：
  * <流程名>（<角色>）：

### 功能邏輯說明

* **<規則名>**：<具體公式 / 格式 / 觸發條件 / 邊界>（要可驗證，不寫「比較好」這種模糊話）
* **<規則名>**：<...>

### 驗收條件

> 每條都是可勾稽的驗收情境（達成 / 未達成二元判定），描述「使用者或系統可觀測到什麼結果」，不寫怎麼做。
> 流程 / 狀態轉換類用情境式（Given / When / Then）；介面約束 / 系統規則 / 清單型要求用規則式條列。

**<功能 / 流程名>**

- 假設 <前置情境>，當 <角色採取某動作>，則 <可觀測的結果 + 可量化閾值>
- 假設 <前置情境>，當 <觸發條件>，則 <系統狀態變化 / 拒絕並提示>

**<規則 / 介面約束名>**

- <系統行為規則一句，含明確門檻 / 格式 / 範圍>
- <系統行為規則一句>

### FE

* <前端負責範圍重點（頁面 / 區塊 / 互動）>

### BE

* <後端負責範圍重點（狀態機 / 計算 / 串接）>

<付款狀態等「來源未定義」者：標「依 X 推導 / 另案處理」，不畫捏造的轉換>

### project 描述填寫要點

- **概述**：精簡。自包含：一句定位 + 一句範圍邊界。不貼大段背景（背景與細節由使用情境 / 功能邏輯段承載）。
- **Use Case**：編號 + 巢狀子情境（角色 → 任務 → 子情境）。
- **Key Feature**：粗體功能名 + 一句說明。**phase 標示非固定產出**——僅在 wiki 階段規劃或 Miles 拍板有明確階段時標註，MUST NOT 用 phase 區分 Scope / Out of Scope；Scope 外的項目列「Out of Scope」清單並標歸屬（延後批次 / 其他 project / 其他平台）。
- **功能邏輯說明**：每條「**粗體規則名**：具體公式 / 格式 / 條件」。可驗證，禁模糊。
- **驗收條件**：可勾稽的驗收情境，達成 / 未達成二元，**描述結果不描述做法**（做法歸各 Task issue 的 How）。
  - **格式（依素材性質分流，不強制單一）**：有明確「前置→動作→結果」的流程 / 狀態轉換用「假設 / 當 / 則」情境式；介面約束、系統規則、清單型要求用規則式條列。
  - **條數**：每模組（或每個獨立功能群組）控制在 **3-5 條**。可量化處給明確閾值（份數 / 金額 / 天數 / 狀態名）。超過 5 條多半是「該拆功能群組」的訊號，不是塞更多條件。
  - **與「功能邏輯說明」分工**：功能邏輯說明寫**規則本體**（公式 / 觸發條件 / 邊界怎麼算）；驗收條件寫**可勾稽的驗收情境**（在什麼前置下、做什麼、應看到什麼結果）。同一規則一處定義、一處驗收，不互相複寫。
  - **品質判準**：(1) 可驗證、二元 pass/fail；(2) outcome 導向，寫「看到什麼」不寫「怎麼建」；(3) 各條彼此獨立；(4) 無歧義、人人同解。
  - **素材來源**：自業務情境的步驟判準與規格 Scenario 轉譯為自包含條目；缺料處標「另案處理」並記 OQ，**不自編**。**轉譯後成品禁出現任何內部路徑或檔名**。
  - **不混 Definition of Done**：驗收條件只放「這張單特有、業務可觀測的結果判準」；通用品質門檻（測試覆蓋 / code review / 文件齊備）屬與開發另立的 Definition of Done，不在每模組重述。
- **資料欄位**：**不寫入交付內容**——欄位正本在 wiki 實體卡（`05-entities/`），開發端直接取得；project 描述不複寫欄位表（複寫即雙正本，欄位異動時必失步）。
- **FE / BE**：各角色粗略範圍（細節在對應 issue）。
- **狀態機**：見 § 三。

### 完整示例（印刷 ERP 真實格式 · 訂單異動）

> 此示例同時含情境式與規則式兩種格式，作為「多樣化引導」的示範。

```markdown
### 驗收條件（示例 · 訂單異動）

**送出異動申請**

- 假設訂單 #ORD-2026-0431 狀態為「生產中」，當業務送出「加印 500 份」的異動申請，則系統建立一筆狀態為「待主管審核」的異動單，並凍結該訂單後續排程動作直到審核結果產生
- 假設訂單 #ORD-2026-0431 已有一筆「待主管審核」的異動單，當業務再送出第二筆異動申請，則系統拒絕並提示「尚有未結案的異動申請」，不建立新異動單

**異動核准後的金額與工單重算**

- 假設異動單「加印 500 份、加收新台幣 8,400 元」經主管核准，當核准生效，則訂單應收金額自新台幣 25,200 元更新為 33,600 元，且受影響工單份數依新數量重算
- 假設異動內容為「縮減交期」不涉數量，當核准生效，則訂單應收金額不變、僅交期欄位更新

**異動單顯示規則**

- 異動單列表每筆顯示異動類型、申請人、異動前後差異（份數或金額）、目前狀態四項
- 異動前後金額差異以新台幣顯示至元，正值標「補收」、負值標「退費」
```

## 二、Task issue 描述模板（角色層 · How）

```markdown
## 概述

1. 範圍可參考本 project「<project 名>」描述（Phase <N>）的 <對應段落>
2. 本 issue 負責 <一句角色範圍>

## 實作細節

### <Backend | Frontend>

<可依主題分組>
- [ ] <實作要點：含 API / 狀態機 / 計算規則 / 頁面 / 區塊 / 互動>
- [ ] <實作要點>

依賴與範圍
- [ ] <依賴其他模組 API：標明「只消費、不實作」>
- [ ] <Out of Scope 註記（不含項與歸屬）>
```

### Task issue 填寫要點

- **概述**：兩點 —— 指回 project 描述對應段落 + 一句負責範圍。不重複 project 已寫的 What。
- **實作細節**：該角色的 How checklist；複雜 issue 可分主題群組。**不可只剩「指回 project」一句而無實作要點**（過度精簡 = 開發看不到該做什麼）。
- **依賴註記**：跨模組依賴標「只消費不實作」；Out of Scope 項標清楚。
- 進行中（In Progress）且已有內容的 issue（如設計工作）**不覆寫**，只由 project 描述引用。

## 三、狀態機規範

- 狀態密集模組（如訂單：訂單 / 印件 / 金流 / 發票 等多個狀態機）**每個狀態機附一張 UML**。
- 形式：Mermaid ` ```mermaid ` + `stateDiagram-v2`。
- 每張圖 MUST 含：**狀態節點 + 轉換箭頭 + 觸發 / 條件（箭頭 label）**；關鍵終態 / 衍生值 / 不可逆用 `note` 標。
- **狀態值、轉換、觸發 MUST 來自規格正本**（state-machines spec 等）；來源未定義者**不畫**，改記 OQ + 標「另案處理」。
- 雙維度狀態（如審稿維度 / 印製維度、開票維度 / 收款維度）**分開各畫一張**。
- 中文狀態 id 無標點符號；轉換 label 避免半形括號（用「·」或全形）。
- 寫入 Linear 後 **MUST 請 Miles 確認渲染成圖**（非程式碼）。
