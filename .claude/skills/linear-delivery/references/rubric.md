# 交付給開發文件 · 評分標準（Rubric）

> 用法：交付前由**評分者**（與實作者分離的 sub-agent，結構性交付才開；不需 senior-pm——交付內容自 BRD / PRD 整理、上游已定案，評分聚焦 rubric 符合性而非重跑上游查證）逐維度評分；未達合格分數 → 修正 → **重新評分**，以 goal 機制迭代直到達標（硬上限 3 輪、卡住問 Miles、不放水）。
> 重點是 Step 4「**具體拒絕案例（絕對不要做的事）**」—— 禁令越精確，評分者越好抓違規；評分 MUST evidence-anchored（引用草稿具體位置，非空泛評語）。
> **評分尺度（2026-07-06 分數化、2026-07-15 加 D5）**：每維度 **通過 = 2 分 / 部分 = 1 分 / 未通過 = 0 分**，滿分 10 分。**合格分數 = 10 分，且 D4 必須 2 分（一票否決）**；未達 10 分不發布。
> 案例皆取自 2026-06-01 訂單管理交付的真實事件（D5 自 2026-07-15 工作流 v2.0 改版新增）。

---

## 評分輸出格式

```
[交付評分 · 第 N 輪]　總分：N / 10
D1 正本邊界：2 / 1 / 0　← 違反禁令：<引用草稿具體位置> / 修正方向：<...>
D2 分層與顆粒度：...
D3 完整性：...
D4 真實性（不捏造）：2 / 0（一票否決，無 1 分）
D5 交付結構完整性：...
達標判定：是（10 / 10）/ 否
總結：可發布（達標）/ 需迭代（列必修項 → 交回實作者修正後 MUST 重新評分，不得自審放行）
```

## D1 · 正本邊界

**定義**：Linear 為對開發的唯一正本，交付內容自包含、跨單據只以 Linear 識別碼互指（Notion 已停用為 BRD，不引用）。OpenSpec spec 是 PM 內部工作版本，不外露；資料欄位表不寫入交付內容（欄位正本在 wiki 實體卡，開發端直接取得）。

**絕對不要**：
- 絕對不要在交付內容寫 `openspec/specs/...` 或任何 `.md` 內部檔路徑
- 絕對不要把「規格正本」指向開發打不開的內部檔
- 絕對不要引用 Notion 頁面 / DB 連結當交付依據（Notion 已停用為 BRD）
- 絕對不要把 PM 內部流程術語（change / delta / archive / OQ 編號）寫進給開發的內容

**正例**：資料欄位段以自包含欄位表呈現（業務語言、只列本批相關欄位）；跨 issue 以 Linear 識別碼（如 FE-259）互指。

**反例（2026-06-01 真實）**：資料欄位段寫「規格正本：openspec/specs/order-management/spec.md § Data Model」、各 Backend issue `## Ref` 列 `openspec/specs/...`。Miles 指出對開發與公司而言 Linear 上的內容才是正本，內部工作版本引用不該出現。

**評分**：無任何內部引用 = 通過；殘留 1-2 處 = 部分；多處或核心指向內部檔 = 未通過。

---

## D2 · 分層與顆粒度

**定義**：project 描述寫完整模組層 What；各 Task issue 寫該角色 How 實作要點。What 不下沉到 issue 重複、How 不上浮到 project；issue 不過度精簡到空洞。

**絕對不要**：
- 絕對不要把實作細節（API / 欄位級 / 互動）寫進 project 描述
- 絕對不要讓 Task issue 只剩「範圍見 project」一句而無該角色實作 checklist
- 絕對不要在 Task issue 整段複製 project 已寫的使用情境 / 功能邏輯

**正例**：project 寫 Use Case + Key Feature + 功能邏輯（What）；BE issue 寫「概述指回 project 段落 + 實作細節 checklist（狀態機 / 計算 / 串接）」。

**反例（本次真實）**：第二輪把 5 個 BE/FE issue 收斂到只剩「## 概述 指回 project + 一句範圍」，移除實作細節 → Miles 指出「BE/FE 的實作 task 可以再多寫實作細節」。（另一極端：第一輪 What 與 How 都很詳細、有重複風險。）

**評分**：分層清楚且 issue 有足夠 How = 通過；某幾個 issue 過簡或局部重複 = 部分；issue 空洞或 What/How 大量錯置 = 未通過。

## D3 · 完整性

**定義**：交付含模組所需的必要區塊；**狀態密集模組每個狀態機附 UML**（狀態 + 轉換 + 觸發）；**有業務可觀測結果的功能附驗收條件**（可勾稽情境、3-5 條、達成 / 未達成二元）。

**絕對不要**：
- 狀態密集模組（多實體各有狀態），絕對不要交付沒有任何狀態流轉圖的文件
- 雙維度狀態（如審稿 / 印製、開票 / 收款），絕對不要擠成一張圖
- 絕對不要漏掉模板規定的必要區塊（概述 / 使用情境 / Spec）；絕對不要把資料欄位表寫進交付內容（欄位正本在 wiki 實體卡）
- 有明確業務結果的功能，絕對不要交付沒有任何驗收條件的 project 描述；絕對不要把通用品質門檻（測試 / code review / 文件）當該模組的驗收條件混進來

**正例**：訂單交付補上 8 類狀態機（訂單 / 印件雙維度 / 打樣 / 訂單異動 / 售後 / 發票 / 折讓 / 請款期次雙維度）各一張 Mermaid 圖。

**反例（本次真實）**：交付初稿涵蓋訂單 / 印件 / 金流 / 發票多個狀態機的模組，卻完全沒有狀態流轉圖 → Miles 指出「只要是狀態的都需要附上 UML」。

**評分**：必要區塊齊 + 狀態機 UML 齊 + 驗收條件齊（可勾稽、未混 DoD）= 通過；缺 1-2 張狀態圖、驗收條件單薄或混入通用門檻、某區塊單薄 = 部分；狀態密集卻無 UML、或缺核心區塊、或有業務結果的功能完全無驗收條件 = 未通過。

---

## D4 · 真實性（不捏造）｜一票否決

**定義**：交付內容的規則 / 狀態 / 數值 MUST 來自規格正本。來源未定義者，停下來記 OQ + 標「另案處理」，不自編。

**絕對不要**：
- 來源（spec）未定義某邏輯時，絕對不要自編一套規則填入交付文件
- 絕對不要用「應該 / 大概 / 通常」的推測當定案寫入
- 絕對不要為了「看起來完整」而補上沒有依據的狀態轉換 / 公式 / 欄位 / 驗收條件情境

**正例（本次真實）**：要畫付款狀態圖時，發現 spec 未定義「部分退款 / 已退款」判定條件 → 停下來問 Miles + 記 `oq-manage` ORD-027 + 交付文件標「依收款累計推導、完整定義另案處理」，**不畫捏造的轉換**。

**反例**：spec 沒寫就自行假設「收到任一筆退款 = 部分退款、退滿 = 已退款」當定案畫進圖裡、發布給開發。

**評分**：全部有據、缺口已顯性化 = 通過；任一處捏造 = **未通過（整體一票否決，不可發布）**。

---

## D5 · 交付結構完整性（v2.0 結構件）

**定義**：交付模板 v2.2 的結構件齊備——Project 含「測試決策」章（受測介面 2-4 個＋測試先行說明，只此一處）；每張 Task 票含「驗收測試」段（直接列清單、每條標受測介面，不重複引言）與「Blocked by」段；milestone 已建立且票已掛入、順序反映整合先後。

**絕對不要**：
- 絕對不要漏掉「測試決策」章；絕對不要另附「Prototype 參照邊界」表（為準／不為準邊界正本在 CLAUDE.md 分工宣告，交付不重述）
- 絕對不要交付沒有「驗收測試」段或「Blocked by」段的 Task 票；票內絕對不要重複測試先行引言（該說明只在「測試決策」章）
- 驗收測試絕對不要落在「測試決策」章未議定的受測介面上（先回頭補議定，不偷跑）
- 驗收測試的期望值絕對不要用「與實作同一套方式重算」得出（自我循環、永遠會過）——期望值必須取自 Project 驗收條件
- 絕對不要另開設計票（DE 票）——操作方式與介面由 Prototype 承載

**評分**：結構件全齊 = 通過；零星缺漏 1-2 處（如個別票漏 Blocked by）= 部分；系統性缺失（全部票無驗收測試、無測試決策章）= 未通過。
