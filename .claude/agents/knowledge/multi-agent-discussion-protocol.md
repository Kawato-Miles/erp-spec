# 多 Agent 輪次討論協議

> 適用情境：凡兩個以上 agent 共同審查或討論同一議題時，均依此協議執行。
> 不限定參與 agent 數量或組合，由呼叫方在啟動時指定本次參與的 agent 清單。

---

## 核心原則

多個 agent 不是「各說各話後由 Claude 彙整」，而是**自主討論至收斂，再把結論交給 Miles**。
Claude 的角色是輪次協調者，負責帶入上一輪結果並判斷是否收斂，不提前彙整。

---

## 背景知識載入規則

每個 agent 只載入與其視角直接相關的資源，不重複載入其他 agent 已覆蓋的範圍。

| Agent | 載入範圍 | 不需主動載入 |
|-------|---------|------------|
| senior-pm | 產品目標、User Story DB、使用者情境（角色權責）| 狀態機細節、技術流程 |
| ceo-reviewer | BRD 本體、KPI DB、商業流程（高層摘要）| User Story 細節、狀態機 |
| erp-consultant | 狀態機（上層 + 下層）、商業流程（完整）、資料欄位 DB | 產品目標、User Story DB |

跨視角的關聯問題（如「角色到不了這個狀態」）不依賴同時載入，而是透過 Round 2 的「跨視角質疑」步驟顯式提出。

---

## 執行流程

### Round 1（平行啟動）

所有參與 agent 依上方載入規則完成背景載入後，獨立輸出初始立場。

每個 agent 的 Round 1 輸出必須包含：
1. **核心立場**：針對本次議題，從自身視角提出最重要的 2-3 個觀察或問題
2. **預期分歧點**：哪些地方你預期其他 agent 可能與你看法不同？為什麼？
3. **前提假設**：你的立場依賴哪些假設？若假設不成立，你的結論會如何改變？

---

### Round 2+（平行，帶入前一輪全文）

每個 agent 收到**所有其他參與 agent** 的前一輪完整輸出後回應。

每個 agent 的輸出必須包含：

**對每位其他參與 agent 的前一輪立場逐點回應：**
- 同意：說明為什麼同意，有無補充觀點
- 部分同意：說明哪部分同意、哪部分有保留，保留的理由是什麼
- 不同意：說明不同意的具體理由，並提出你認為更合理的方向

**跨視角質疑（每輪必填）：**
你的前一輪立場中，有沒有任何結論需要另一個 agent 的專業視角才能確認或推翻？
若有，明確指出：「這個問題需要 [agent 名稱] 從 [狀態機 / 使用者情境 / 商業目標] 角度確認。」

**新增觀察（若有）：**
看到其他 agent 的立場後，是否發現自己前一輪遺漏的問題？

**本輪立場摘要：**
經過交叉討論後，你目前的核心立場是什麼？是否有任何調整？

---

### 收斂判斷（Claude 執行，不告知 Miles）

每輪結束後，Claude 判斷：

| 情況 | 處置 |
|------|------|
| 所有核心議題都達到「同意」或「部分同意（差異可接受）」 | 進入最終彙整 |
| 仍有明確「不同意」且影響結論 | 執行下一輪 |
| 討論陷入循環（各方無新論點，立場未移動）| 強制進入彙整，標記為「未解爭議」 |

**最少執行兩輪**（Round 1 + Round 2）。第三輪起視收斂情況決定是否繼續，最多執行至第三輪後強制彙整。

---

### 最終彙整（Claude → Miles）

所有輪次完成後，Claude 輸出給 Miles 的格式：

```
[多 Agent 討論結論]

參與 agent：[列出本次參與的 agent 名稱]
討論進行：X 輪

共識結論（所有 agent 同意）：
1. [結論] — 依據：[各 agent 共同認可的理由]
2. ...

修正立場（初始有分歧，討論後收斂）：
- [議題]：原本 [A 的立場] vs [B 的立場] → 收斂為 [最終共識]，收斂理由：[...]

必須處理的行動項目：
- [行動] — 優先級：[P0/P1/P2] — 處置方式：[BRD 修改 / 轉為 OQ / 記錄風險]

未解爭議（交 Miles 決定）：
- [議題]：[Agent A] 認為 [...]；[Agent B] 認為 [...]
  → Miles 需要在 [A 方向] vs [B 方向] 之間決定；A 的影響是 [...]；B 的影響是 [...]
```

---

### 討論後寫入流程（Claude 執行）

最終彙整輸出給 Miles 後，若「必須處理的行動項目」中包含 BRD 內容修改，Claude 依以下流程處理：

**步驟一：確認是否需要 senior-pm 寫入**

Claude 判斷行動項目屬於哪一類：

| 類型 | 處置 |
|------|------|
| BRD 內容修改（段落更新、新增說明）| 進入步驟二，觸發 senior-pm 寫入流程 |
| 轉為 OQ（待確認事項）| Claude 觸發 `oq-manage` skill |
| 記錄為已知風險（不修改 BRD）| Claude 在 BRD 適當位置新增風險說明後，請 Miles 確認 |

**步驟二：觸發 senior-pm Mode B — Phase 1（規劃）**

```
Claude 呼叫 senior-pm，傳入：
- [MODE: PLAN]
- 最終彙整結論全文
- BRD Notion 頁面連結
- 本次需處理的行動項目清單
```

senior-pm 返回結構化「寫入計畫」後，Claude 將計畫呈現給 Miles：

```
[Senior PM 寫入計畫 — 待確認]

以下修改將由 Senior PM 直接寫入 Notion，請確認後執行：

操作 N：[目標頁面] § [段落]
  理由：[...]
  變更前：[...]
  變更後：[...]

確認執行？（確認後 Senior PM 將直接寫入，不再經由 Claude 轉述）
```

**步驟三：Miles 確認後，觸發 senior-pm Mode B — Phase 2（執行）**

```
Claude 呼叫 senior-pm，傳入：
- [MODE: EXECUTE | plan]
- Phase 1 完整寫入計畫
```

senior-pm 執行完畢，返回已寫入清單。Claude 將清單摘要回報 Miles。

---

## 行為規範

- **不允許** 任何 agent 在 Round 2 以後對所有其他 agent 的觀點都說「同意」——必須提出至少一個真實的質疑或補充
- **不允許** 空洞回應如「這個觀點很有道理」，必須說明具體的同意或不同意理由
- **不允許** Claude 在討論未達最少兩輪前就彙整給 Miles
- Round 3 之後無論是否收斂，都必須輸出給 Miles——剩餘分歧透明列出，不靜默跳過
- 整個討論過程 Miles 不可見；只有最終彙整才呈現給 Miles
