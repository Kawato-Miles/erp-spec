## 一、印刷業 ERP 的核心設計原則

**Job-shop 生產模式的本質：**
印刷業是 Job-shop，每張訂單工藝路線不同，與 Flow-shop（汽車流水線）有根本性差異。這決定了 ERP 的設計取向：
- 工藝路線（Routing）必須依訂單規格動態生成，不能標準化
- 機台排程需要考慮換線時間（製版切換是印刷業的主要換線成本）
- 物料消耗難以精確預估（廢張率因機台狀況、操作員、材質不同而差異大）

**多層單據結構的必要性：**
印刷業的生產有天然層次：需求單 → 訂單 → 工單 → 印件 → 生產任務。每一層都有獨立的生命週期與狀態，不能合併。這是業界設計的基本共識，不是過度設計——合併會導致無法支援複雜的部分完工、部分出貨情境。

**數量換算的複雜性：**
印刷業的數量換算非直觀：
- 訂單數量（客戶要的份數）≠ 印刷數量（需加廢張、測試張）
- 工單數量可能因拼版而是訂單數量的整數倍
- 每個生產任務的「每工單倍數」(quantity_per_work_order) 必須在任務層獨立記錄，不能從上層繼承——這是印刷業特有的設計需求

## 二、狀態機設計原則與常見陷阱

**核心原則：狀態是「穩定的等待節點」，不是「執行步驟」**
- 每個狀態代表「這個單據目前在等什麼才能繼續」
- 每個狀態必須有明確的進入條件（前置狀態）與離開條件（觸發事件）
- 所有狀態轉換必須可追溯（誰、何時、觸發了什麼動作）

**木桶原則（最少工單原則）：**
父層狀態由子層中「進度最落後的那個」決定。一個工單有 5 個印件，只要 1 個未完成，工單就不能標記完成。這是 Job-shop ERP 的核心邏輯，設計時最容易被忽略，上線後最常出問題。

**常見陷阱：**

1. **缺少中間確認狀態**
   直接從「進行中」跳到「完成」，沒有「待 QC 確認」或「待客戶確認」的中間節點。現實中品質問題就在這個跳躍中被遺漏。

2. **回滾路徑缺失**
   狀態只能前進不能後退。現實中退單、重工、客戶取消是常態——每個狀態都需要定義回滾路徑與觸發條件，以及回滾後關聯資料的處理方式。

3. **自動觸發無前置條件校驗**
   自動狀態推進（如「所有印件完成 → 工單自動完成」）沒有校驗必填資料是否完整（如出貨地址、發票資訊），導致流程推進但下游操作缺少必要資訊。

4. **並行更新導致狀態競爭**
   多個印件同時完成時，父層工單的狀態計算必須是原子操作。若兩個並行更新同時讀取「未完成」狀態再各自推進，可能導致狀態計算錯誤。

5. **狀態語意模糊**
   「處理中」、「進行中」這類狀態名稱，在不同人眼中代表不同意思。狀態名稱必須反映業務語意（如「等待審稿」「等待排程」「等待製版」），不能用技術性詞彙。

## 三、資料模型設計考量

**關聯設計的關鍵決策：**
- 外鍵關係必須明確定義刪除行為（CASCADE 或 RESTRICT）。印刷業中，訂單不能因客戶被刪除而消失——所有製造業 ERP 的業務資料應使用 RESTRICT
- 軟刪除（soft delete）是製造業 ERP 的必要設計，歷史資料不能真刪——稽核需求與糾紛處理都依賴歷史記錄
- 版本控制：報價、工藝參數、稿件確認都需要版本歷史，不能只保留最新版

**金額與計算欄位的設計原則：**
- 金額欄位必須記錄計算時的基準（如報價時的紙張單價），不能只存計算結果——日後單價改動，歷史報價必須可重現
- 所有衍生計算欄位（如訂單總金額）應記錄計算時的版本或公式參照，避免公式改動後歷史資料無法重算
- 涉及多幣別或稅率的金額，必須記錄原始幣別與匯率

**欄位完整性規範：**
- 「可空」欄位的設計必須有業務理由——「之後再填」的欄位在系統層面應標記為哪個流程節點必填
- 時間戳必須統一 timezone 處理，資料庫層儲存 UTC，顯示層轉換本地時間

## 四、跨模組整合的挑戰

**整合節點的常見設計問題：**

| 整合節點 | 常見問題 |
|---------|---------|
| 訂單確認 → 工單建立 | 觸發條件不明確（哪個訂單狀態才建立工單？手動還是自動？） |
| 工單完工 → 庫存入帳 | 完工認定標準模糊（全部印件完成？部分完工是否入帳？） |
| QC 通過 → 出貨許可 | QC 結果與出貨單之間的資料流沒有定義（哪些資訊隨附出貨單？） |
| 出貨完成 → 應收帳款 | 部分出貨的帳款拆分邏輯未設計 |

**跨模組操作的原子性：**
當一個動作影響多個模組（如訂單取消同時影響工單、庫存、出貨），必須確保：
1. 所有相關模組的狀態都能回滾
2. 整個操作是原子性的（全部成功或全部失敗）
3. 失敗時有明確的錯誤訊息與人工介入路徑（不能靜默失敗）

## 五、成功 ERP 實施的關鍵因素

根據印刷業導入案例的共同觀察：

**成功因素：**
1. **資料遷移的品質**：歷史資料品質決定系統上線後的可信度。遷移前必須清理髒資料，不能「帶著問題一起遷移」
2. **關鍵使用者的設計參與**：生管、業務主管必須在設計階段就參與，不能事後驗收——他們最了解例外情境
3. **分階段上線**：先跑通核心流程（訂單 → 出貨），再逐步加入成本分析、自動排程。大爆炸式上線是高風險路線
4. **平行跑帳期**：新舊系統並行至少 1 個月，在真實業務壓力下建立信任

**常見失敗模式：**
- 設計過於理想化，沒有為「例外是常態」設計路徑
- 上線前培訓不足，使用者在壓力下放棄系統
- 客製化過多，導致升級困難與維護成本失控

## 六、CRM 與忠誠度管理的系統設計考量

這家公司有兩條訂單渠道，帶來兩套不同的 CRM 資料流，設計時必須同時支撐。

**雙渠道的資料架構挑戰：**

| 渠道 | 訂單來源 | 客戶行為特徵 | CRM 資料需求 |
|------|---------|------------|------------|
| 傳統業務接單 | 業務手動建單 | B2B 採購，重複規格，信用交易 | 累計訂單量、毛利貢獻、信用額度、沉默偵測 |
| 線上自助訂單（圖編）| 客戶自行下單 | 偏 B2C，自助設計，即時付款 | 回購率、模板使用記錄、首單後追蹤 |

**客戶身份統一（Customer Master）：**
同一家公司可能同時有業務接的訂單（公司帳戶）與員工自助下的線上訂單（個人帳戶）。設計上必須：
- 支援「企業帳戶 → 子帳戶」的層級結構，避免訂單分散在不同身份下導致忠誠度計算不完整
- 訂單合併歸屬的規則必須明確（以公司稅號、域名、或人工關聯？）

**忠誠度計算的資料模型考量：**
- 忠誠度等級的計算基礎（累計訂單金額？訂單次數？毛利？）必須明確定義，且在訂單完成時原子性更新
- 等級升降的觸發條件需要版本化記錄——客戶等級變更必須有歷史快照（「何時從 Bronze 升 Silver，依據哪筆訂單」）
- 跨渠道的訂單應合併計算忠誠度，不能各算各的

**忠誠度功能與 ERP 核心流程的整合節點：**

| 整合節點 | 設計考量 |
|---------|---------|
| 訂單完成 → 更新忠誠度積分 / 累計金額 | 必須在訂單狀態轉「完成」的同一個 transaction 中更新，不能非同步 |
| 忠誠度等級變更 → 觸發優惠設定生效 | 定價規則的生效時間點必須明確（立即生效？下筆訂單生效？） |
| 客戶沉默偵測 → 業務提醒 | 需要一個定期作業（非即時）偵測超過 N 天未下單的客戶，觸發通知 |
| 線上訂單 → 匯入 ERP | 圖編的訂單匯入 ERP 時，必須帶入客戶身份以正確計算忠誠度，不能建成匿名訂單 |

**常見設計錯誤：**
- 忠誠度積分計算邏輯分散在多處（下單時加一次、退款時未扣除）→ 必須有單一計算入口
- 退單後未回滾忠誠度積分 → 客戶退貨還能保留積分升等
- 優惠計算在前端做而非 ERP 做 → 線上與線下的優惠邏輯不一致

## 七、當前趨勢（由 knowledge-updater 定期更新）

> 以下內容每次更新時，knowledge-updater 會綜合最新搜尋結果重寫此段。穩定的 Domain Know（§ 一至五）不在更新範圍。

**ERP 架構趨勢：**
- Cloud-native ERP（微服務、API-first）成為中型企業主流選擇，避免 monolithic 系統的升級困境
- Composable ERP 概念興起：企業不再選擇單一供應商的全套系統，而是組合最佳模組
- Low-code 配置（vs 傳統客製化開發）降低中小型印刷廠的導入與維護門檻

**製造業 ERP 設計新思維：**
- 即時生產進度可視化（IoT 設備自動回報機台狀態）開始進入印刷業
- 事件驅動架構（Event-driven）取代傳統輪詢設計，狀態變化即時傳播至相關模組
- AI 輔助排程（預測廢張率、自動排機優化）在高端 MIS 系統中出現
