## Purpose

訂單異動模組 — 管理訂單成立後因規格變更、加印追加、退印、折扣、補運費、急件費等原因導致的金額異動。

**問題**：
- 訂單成立後客戶仍可能追加、退印、或因品質問題退款，這些金額增減需要有獨立的審核軌跡，不能直接改訂單明細（否則無法追溯「原本報多少、後來改了什麼」）
- 補收（客戶加單）與退款（錢退出去）的風險不對稱——退款一旦出錯難追回，補收是送上門的生意——所以需要不同的審核門檻

**目標**：
- 補收正項免審直達確認可執行（不阻擋業務接單效率），退款負項須業務主管核可（現金流出強把關）
- 每筆異動有獨立狀態機（草稿 → 待主管審核 → 已核可 → 確認可執行），不阻擋主訂單狀態推進

- 相依模組：[order-management](../order-management/spec.md)（共用 Order 實體）、[order-billing](../order-billing/spec.md)（異動影響應收、退款牽動發票折讓）、[after-sales-ticket](../after-sales-ticket/spec.md)（售後退款走 ticket 容器建異動單）
- 欄位正本：[wiki 訂單異動實體卡](../../../memory/Sens_wiki/wiki/erp/05-entities/訂單異動.md)
- 狀態列舉正本：[wiki 訂單異動狀態機卡](../../../memory/Sens_wiki/wiki/erp/06-state-machines/訂單異動狀態.md)

## Requirements

### Requirement: 訂單異動（OrderAdjustment）建立與審核

業務 / 諮詢於訂單詳情頁建立訂單異動，記錄訂單成立後的金額增減（規格變更 / 加印 / 退印 / 折扣 / 運費 / 急件費等）。訂單異動單有獨立狀態機（草稿 → 待主管審核 → 已核可 / 已退回 → 確認可執行 / 已取消），不影響主訂單狀態推進。補收正項免審直達確認可執行，退款負項須業務主管核可。「確認可執行」（終態、不可逆）的推進與應收認列時點依路徑：補收於建單生效時、一般退款於主管核可當下、諮詢取消退費於諮詢人員確認金額時；實際退款金流（退款收款紀錄完成）MUST NOT 回寫單據狀態，由對帳應退差額盯住。訂單進入終態後售後異動改走售後服務單。

**Priority**: P0

**Rationale**: 訂單成立後加印、退印、折扣等金額增減是印刷業常態，需要有獨立審核軌跡（不直接改訂單明細），且補收與退款因風險不對稱需不同審核門檻。

#### Scenario: 業務建立加印追加異動（不變，沿用既有）

- **GIVEN** 訂單 SO-001 狀態 = 生產中
- **WHEN** 客戶要求加印 200 份，業務於訂單詳情頁點擊「建立訂單異動單」
- **THEN** 系統 SHALL 建立訂單異動單、UI 標題顯示「訂單異動單」
- **AND** 業務 SHALL 可選異動類型 = 加印追加
- **AND** 業務新增明細「明細類型 = 印件，描述 = 加印 200 份，金額 = +20,000」
- **AND** 異動金額 SHALL 自動加總為 +20,000
- **AND** 訂單異動單狀態 SHALL = 草稿
- **AND** 關聯售後服務單 SHALL = NULL
- **AND** 業務點擊「提交審核」後狀態 SHALL → 待主管審核

#### Scenario: 業務於 AfterSalesTicket 內建關聯 OrderAdjustment（不變，沿用既有）

- **GIVEN** 售後服務單 AS-001 狀態 = 處理中、解決方案 = 退款
- **WHEN** 業務於售後服務單內點「建立退款異動單」
- **THEN** 系統 SHALL 建立訂單異動單、預填異動類型 = 退印、關聯售後服務單 = AS-001
- **AND** 業務填入異動金額 = -5000、明細描述
- **AND** 訂單異動單狀態 SHALL = 草稿，後續走原狀態機（提交審核 → 主管核可當下即推進確認可執行並認列應收；實際退款由業務另建退款收款紀錄，完成後由對帳應退差額核銷）

#### Scenario: 已完成 / 已取消訂單禁止於訂單詳情頁新增訂單異動單（add-after-sales-ticket 補驗收）

- **GIVEN** 訂單狀態 ∈ {訂單完成、已取消}
- **AND** 使用者為業務 / 諮詢
- **WHEN** 使用者進入訂單詳情頁「訂單異動」Tab
- **THEN** 「新增訂單異動單」按鈕 SHALL disabled
- **AND** 按鈕 SHALL 顯示引導提示（Tooltip / title）：已完成 → 「訂單已完成，售後異動請至『售後服務』Tab 建立售後服務單」；已取消 → 「訂單已取消，無法新增訂單異動」
- **AND** 既有訂單異動單（完成前建立的存量）SHALL 仍可檢視、審核、編輯金額、管理關聯收款紀錄（不受新增閘門影響）
- **AND** 訂單完成後的金額異動 SHALL 改於售後服務單內建立（見 [after-sales-ticket spec](../after-sales-ticket/spec.md)）

#### Scenario: 業務主管核可 OrderAdjustment（不變，沿用既有）

- **GIVEN** 訂單異動單狀態 = 待主管審核、異動金額 = -5000
- **WHEN** 業務主管於訂單詳情頁的異動清單點擊「核可」
- **THEN** 訂單異動單狀態 SHALL → 已核可
- **AND** 系統 MUST 記錄核可者、核可時間、核可金額快照 = -5000

#### Scenario: 業務主管退回 OrderAdjustment（不變，沿用既有）

- **GIVEN** 訂單異動單狀態 = 待主管審核
- **WHEN** 業務主管點擊「退回」並填入退回原因
- **THEN** 訂單異動單狀態 SHALL → 已退回
- **AND** 業務 SHALL 可修改後重交審核

#### Scenario: 業務於已核可狀態調整金額（沿用 refine-after-sales-refund 設計）

- **GIVEN** 訂單異動單狀態 = 已核可、核可金額快照 = -5000、當前金額 = -5000
- **WHEN** 業務發現實際退款金額應為 -4800（客戶談判），於 OA 編輯介面點擊「編輯金額」
- **THEN** 系統 SHALL 允許業務修改當前金額 = -4800
- **AND** 訂單異動單狀態 SHALL 維持「已核可」（不需重新送審）
- **AND** 對照欄位 SHALL 顯示「主管核可金額 -$5,000（{核可時間}）｜當前金額 -$4,800｜業務已調整 +$200」
- **AND** 稽核紀錄 SHALL 記錄此異動（調整前金額 = -5000, 調整後金額 = -4800, 調整時狀態 = 已核可）

#### Scenario: 退款 Payment 切已完成不改變 OA 狀態（金流完結由對帳盯住）

- **GIVEN** 訂單異動單狀態 = 確認可執行（主管核可時已推進並認列）、當前金額 = -5000
- **AND** 業務已在 OA 編輯介面建立關聯退款收款紀錄 P-001（金額 = -5000, 付款方式 = '退款', 收款狀態 = '處理中'）
- **WHEN** 業務於 P-001 編輯 dialog 補付款日期、上傳對帳附件、切收款狀態 → '已完成'、點擊「儲存」
- **THEN** 系統 SHALL 通過驗證並寫入 P-001 收款狀態 = '已完成'、完成時間 = now
- **AND** 訂單異動單狀態 MUST 維持「確認可執行」（退款金流完結不回寫單據）
- **AND** 對帳「應退差額」SHALL 核銷歸零
- **AND** 請款期次（見 [order-billing](../order-billing/spec.md)）SHALL NOT 自動變動

#### Scenario: 補收 OA 建單即推進確認可執行（免審、不綁收款）

- **GIVEN** 業務建立訂單異動單 OA-002（異動類型 = 加印追加、異動金額 = +20000）
- **WHEN** 業務點擊「儲存並馬上生效」
- **THEN** OA-002 狀態 SHALL 直接 = 確認可執行（approved_by = 業務本人、executed_at = now）
- **AND** 訂單應收總額 MUST 即時 +20000
- **AND** 系統 MUST NOT 自動建立補收收款紀錄；補收的收款另循訂單正常收款流程，不對應到本單
- **AND** 系統 MUST NOT 要求任何收款紀錄完成才推進本單

#### Scenario: 已核可狀態的操作入口（依異動類型分流）

- **GIVEN** 訂單異動單狀態 = 已核可
- **WHEN** 使用者打開 OA 編輯介面
- **THEN** 諮詢取消退費型 SHALL 顯示「確認」按鈕（諮詢人員確認金額後推進確認可執行並認列）與「取消」按鈕（決定不退）
- **AND** 系統 SHALL 顯示「新增收款紀錄」按鈕（退款收款紀錄的建立入口；其完成與否不影響單據狀態）

#### Scenario: 確認可執行 OA 鎖定金額（不變，沿用既有）

- **GIVEN** 訂單異動單狀態 = 確認可執行
- **WHEN** 業務嘗試打開 OA 編輯金額
- **THEN** 系統 SHALL NOT 顯示「編輯金額」按鈕
- **AND** 金額欄位 SHALL 為唯讀（disabled）

#### Scenario: 訂單異動不阻擋主訂單推進（不變，沿用既有）

- **GIVEN** 訂單異動單狀態 = 待主管審核
- **AND** 訂單主狀態 = 生產中
- **WHEN** 工單 / 印件層級觸發 bubble-up 推進主訂單至「出貨中」
- **THEN** 系統 SHALL 允許主訂單推進，訂單異動單仍維持其獨立狀態

#### Scenario: 訂單異動執行後生產內容變更提示（修訂觸發機制描述）

- **GIVEN** 訂單異動單含印件類型明細（例如加印追加、規格變更）
- **WHEN** 訂單異動單推進至「確認可執行」
- **THEN** 系統 SHALL 顯示提示「此異動涉及生產內容，請至訂單詳情頁編輯印件以接續審稿 / 工單流程」
- **AND** 提示為非阻擋式（業務可關閉提示繼續），系統 NOT 自動建立或修改 PrintItem

### Requirement: OrderAdjustment.adjustment_type 完整 enum

訂單異動單的異動類型 SHALL 採用以下完整 enum 列舉，不再依階段限制可選範圍：

| 異動類型 | 適用情境 | 建立方式 |
|----------------|---------|---------|
| 規格變更 | 訂單期間客戶變更印件規格導致金額調整 | 業務手動 |
| 加印追加 | 訂單期間客戶要求加印 | 業務手動 |
| 退印 | 退印 / 退款（訂單期間或售後皆可）| 業務手動 |
| 折扣 | 業務給予客戶折扣 | 業務手動 |
| 加運費 | 訂單成立後補收運費 | 業務手動 |
| 急件費 | 訂單成立後補收急件費 | 業務手動 |
| 補退 | 售後補印收費 / 訂單期間補退 | 業務手動 |
| **諮詢取消退費** | **諮詢取消觸發的半額退款（諮詢費 × 50%）；僅由系統於諮詢取消觸發點自動建立**| **系統內生（業務 UI 不顯示此選項）**|
| 其他 | 不屬上述類別 | 業務手動 |

業務透過 UI 與 API 皆 SHALL 可選用「業務手動」類別的任一異動類型，系統不再依訂單狀態推算限制。「諮詢取消退費」為系統內生類型 — 業務 UI 的異動類型下拉選單 MUST NOT 包含此選項，僅由系統於 [consultation-request spec § 諮詢取消觸發建諮詢訂單與退費](../consultation-request/spec.md) 流程自動建立。

當業務於售後服務單內建關聯訂單異動單時，UI 仍 SHALL 預填合理的異動類型（例：解決方案=退款 → 預填退印；解決方案=補印 → 預填補退），但業務可改選（限「業務手動」類別內選項）。

**Priority**: P0

**Rationale**: 明確列舉所有異動類型（規格變更 / 加印 / 退印 / 折扣 / 運費 / 急件費 / 補退 / 諮詢取消退費 / 其他），業務選用時不會選到不適用的類型，系統自動建的「諮詢取消退費」不暴露給業務。

#### Scenario: 業務於 AfterSalesTicket 內建關聯 OrderAdjustment 預填 adjustment_type

- **GIVEN** 售後服務單解決方案 = 退款
- **WHEN** 業務於售後服務單內點「建立退款異動單」
- **THEN** 系統 SHALL 預填異動類型 = 退印
- **AND** 業務可改選為 折扣 / 補退 / 其他（「業務手動」類別內）
- **AND** 業務下拉選單 MUST NOT 顯示「諮詢取消退費」選項

#### Scenario: 業務於訂單期間自由選 adjustment_type

- **GIVEN** 訂單狀態 = 生產中
- **WHEN** 業務建立訂單異動單
- **THEN** 業務 SHALL 可從「業務手動」8 個列舉（規格變更 / 加印追加 / 退印 / 折扣 / 加運費 / 急件費 / 補退 / 其他）選擇
- **AND** 業務下拉選單 MUST NOT 顯示「諮詢取消退費」選項

#### Scenario: 系統自動建立諮詢取消退費 OA

- **GIVEN** ConsultationRequest 狀態 = 待諮詢、諮詢人員觸發取消諮詢確認流程
- **WHEN** 系統執行「諮詢取消觸發建諮詢訂單與半額退費」流程
- **THEN** 系統 SHALL 自動建立訂單異動單（異動類型 = 諮詢取消退費、異動金額 = -1000、狀態 = 已核可、核可者 = system、執行時間 = NULL、是否需主管審核 = false）
- **AND** OA 建立 MUST NOT 經過業務 / 主管的 UI 操作
- **AND** 業務 SHALL 可於 OA 已核可狀態調整退款金額（沿用既有「已核可後修改不需重審」）；退款收款紀錄切已完成累計達 -1000 推進 OA 確認可執行
- **AND** 應收認列已核可 OA(-1000)，應收公式 = 訂單額外費用(2000) + ∑確認可執行或已核可訂單異動單(-1000) = 1000

### Requirement: 業務主管批次審核 OrderAdjustment（user story）

業務主管 SHALL 可於後台「待審核訂單異動」頁批次查看所有狀態 = 待主管審核的訂單異動單，依負責業務 / 異動類型 / 訂單編號篩選，逐筆核可 / 退回。

**Priority**: P1

**Rationale**: 業務主管需要在後台集中檢視所有待審核異動單，逐筆核可或退回，避免散落在各訂單詳情頁逐單處理。

#### Scenario: 業務主管查看待審核異動清單

- **WHEN** 業務主管登入後台進入「待審核訂單異動」頁
- **THEN** 系統 SHALL 列出所有狀態 = 待主管審核的訂單異動單
- **AND** 主管 SHALL 可依異動類型篩選

### Requirement: OrderExtraCharge vs OrderAdjustment.fee 時間邊界

訂單額外費用（資料模型實體，見 [order-management § OrderExtraCharge](../order-management/spec.md)）SHALL 限於「訂單未進入終態」時使用。凍結錨點為**訂單狀態進入終態集合 {訂單完成, 已取消}**（與印件金額同步，鎖定點統一為訂單完成終態；取代舊版「線下自審核通過起凍結」——審核通過完全不鎖定明細金額）：

- **線下訂單**：訂單額外費用可於訂單完成前任一狀態（草稿 / 待業務主管審核 / 審核通過 / 報價待回簽 / 已回簽 / 製作中 / 出貨中等）由業務 / 諮詢直接新增 / 編輯 / 刪除（含調降費用金額，運費 / 急件費等），不需業務主管審核；**自進入「訂單完成 / 已取消」終態起凍結**，之後費用異動走訂單異動的明細類型 = 費用明細。
- **線上訂單**：訂單額外費用由 EC 結帳帶入、無業務手動新增視窗；訂單完成後新增費用走訂單異動。
- **諮詢訂單**：訂單額外費用為建立時的諮詢費；進入終態後費用變更走訂單異動。

訂單額外費用調降（費用金額減少 / 刪除致應收減少）時 MUST 寫活動紀錄「完成前金額調降」（弱把關、不阻擋）。

UI SHALL 在訂單進入終態後隱藏「新增訂單額外費用」按鈕，引導走訂單異動。系統 SHALL 在 API 層拒絕終態後的訂單額外費用寫入請求。

**Priority**: P0

**Rationale**: 訂單額外費用（建立時即確定，不需審核）與訂單異動（成立後變動，需審核）語意不同但都影響應收，必須明確以訂單完成終態為分界，避免業務混淆「該走哪條路徑加費用」。

#### Scenario: 業務於製作中加運費走訂單額外費用

- **GIVEN** 線下訂單 SO-001 處於「製作中」狀態（未進入終態）
- **WHEN** 業務新增 200 元運費
- **THEN** 系統 SHALL 建立訂單額外費用（費用類型 = 運費, 費用金額 = 200）、應收即時 +200
- **AND** 不需業務主管審核

#### Scenario: 訂單完成後加運費走訂單異動

- **GIVEN** 線下訂單 SO-001 處於「訂單完成」終態
- **WHEN** 業務需補收 200 元運費
- **THEN** UI SHALL 隱藏「新增訂單額外費用」按鈕
- **AND** 業務 SHALL 建立訂單異動單（異動類型 = 加運費，明細：明細類型 = 費用，異動金額 = 200）

#### Scenario: API 拒絕終態後新增訂單額外費用

- **GIVEN** 線下訂單 SO-001 處於「訂單完成」或「已取消」終態
- **WHEN** 系統收到訂單額外費用寫入請求
- **THEN** 系統 SHALL 拒絕並回傳 400 錯誤
- **AND** 錯誤訊息 SHALL 為「訂單已進入終態，新增費用請走訂單異動單流程」

### Requirement: 訂單異動（OrderAdjustment）狀態機

OrderAdjustment SHALL 為**獨立狀態機**，不影響主訂單狀態。狀態定義：

| 狀態 | 說明 |
|------|------|
| 草稿 | 業務 / 諮詢建立但未提交審核 |
| 待主管審核 | 業務提交後等待業務主管核可（僅退款負項 OA 經此態；補收正項免審）|
| 已核可 | 審核已通過、金額待確認；停留於此態的為諮詢取消退費 OA（系統代審建立，諮詢人員確認前可調金額或取消）；一般退款由主管核可當下即推進確認可執行、不停留 |
| 已退回 | 業務主管退回（含原因），業務可修改後重交 |
| 確認可執行 | 終態（不可逆）；金額於進入此態時認列進應收（補收＝建單生效時、一般退款＝主管核可時、諮詢取消＝諮詢人員確認金額時）；實際退款金流不回寫本狀態 |
| 已取消 | 業務主動取消（草稿或已退回階段）（終態） |

狀態轉換：

```
草稿 ─ 提交審核（退款 OA）─────────────────▶ 待主管審核
草稿 ─ 儲存並馬上生效（補收正項，免審）────▶ 確認可執行
草稿 ─ 取消 ──────────────────────────────▶ 已取消
待主管審核 ─ 主管核可（當下即生效）────────▶ 確認可執行
待主管審核 ─ 退回 ────────────────────────▶ 已退回
已退回 ─ 修改後重交 ──────────────────────▶ 待主管審核
已退回 ─ 取消 ────────────────────────────▶ 已取消
（系統建單，諮詢取消退費）─────────────────▶ 已核可
已核可 ─ 諮詢人員確認金額（應收於此認列）──▶ 確認可執行
已核可 ─ 取消（決定不退）──────────────────▶ 已取消
```

OrderAdjustment 處於非終態時 SHALL NOT 阻擋主訂單狀態推進。

**「確認可執行」推進機制（認列即生效、不綁 Payment 累計）**：

- **補收正項 OA**（amount > 0、免審）：建立並「儲存並馬上生效」時直接推進「確認可執行」，應收即時 +N。
- **退款負項 OA**（amount < 0、送審）：業務主管核可當下推進「確認可執行」（executed_at = 核可時點），應收即時 −N；不等退款 Payment 切已完成。
- **諮詢取消退費 OA**（系統內生）：系統代審建為「已核可」（負向須審核不破例、金額固定故由系統代行）；諮詢人員確認金額時推進「確認可執行」、應收於此刻 −N；確認前可調金額（不重審）或取消（決定不退）。

**分流規則（四類 OA × 路徑）**：

| OA 類型 | 審核 | 狀態流轉 |
|---------|------|---------|
| 補收正項（amount > 0） | 免審 | 草稿 → 確認可執行（業務儲存並馬上生效）|
| 退款負項（amount < 0） | 業務主管 | 草稿 → 待主管審核 →（核可當下）確認可執行 |
| 諮詢取消退費（系統內生）| 系統代審 | 已核可（系統建）→（諮詢人員確認金額）確認可執行；或 → 已取消（決定不退）|
| 規格變更（amount = 0） | - | 不建 OA |

**Priority**: P0

**Rationale**: OA 狀態機為訂單帳務的核心控制點，補收免審與退款送審的分流對齊印刷業風險不對稱的實務需求。

#### Scenario: 退款 OA 核可後核可即生效

- **GIVEN** 退款 OrderAdjustment OA-001（status = 已核可、amount = -5000）
- **WHEN** 業務主管核可
- **THEN** 系統 SHALL 自動推進 OA-001.status → 確認可執行、executedAt = 核可時點
- **AND** 系統 MUST 重算訂單應收總額（含此筆確認可執行 OA，應收 −5000）
- **AND** OA 推進確認可執行 MUST NOT 等待退款 Payment 切已完成

#### Scenario: 補收正項 OA 儲存並執行直達確認可執行

- **GIVEN** 補收 OrderAdjustment OA-002（amount = +8000、adjustment_type = 加印追加、免審）
- **WHEN** 業務點「儲存並執行」
- **THEN** OA-002.status SHALL 跳過待主管審核 / 已核可、直接 = 確認可執行（approved_by = 業務、executed_at = now）
- **AND** 應收即時 +8000、MUST NOT 綁 Payment 切已完成

#### Scenario: 取消退款 Payment 後 OA 維持確認可執行（無回退）

- **GIVEN** 退款 OA-001 確認可執行（應收已 −5000）、退款 Payment P-001 已完成
- **WHEN** 業務取消 P-001
- **THEN** OA-001.status SHALL 維持「確認可執行」（應收維持 −5000、不回退已核可）
- **AND** 對帳面板 SHALL 重現「應退差額 5000」、引導業務重建退款 Payment
