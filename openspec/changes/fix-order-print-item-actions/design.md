## Context

Miles 在 Prototype 操作訂單詳情頁時回報兩個痛點：
1. 印件列的「刪除」按鈕點擊沒反應
2. 不知道如何編輯印件

經三輪討論釐清後，Miles 明定以下業務規則：
- **編輯規則**：非金額類可編輯（製作細節、單位等），金額類不可編輯（成本、售價）
- **金額變動路徑**：統一走「建立補收款訂單」；補收款支援正負項（增加 = 正值補收款、減少 = 負值補收款 / 退款）；僅「客戶要求全然取消訂單」或「業務誤建整筆訂單」才走取消訂單
- **印件為 source of truth，訂單為唯一編輯入口**：印件資料貫穿訂單 / 工單 / 生產任務 / 出貨單；為確保變更源頭單一，印件編輯只開放於訂單端，下游單據讀取印件即時值
- **報價待回簽的印件修改**：需連動報價單（產生新版次），本 change 採 ActivityLog 提示、不做自動連動

這些規則使得「訂單階段的印件層直接刪除」**沒有任何合法業務情境**：
- 加項 → 補收款（正值）
- 減項 → 補收款（負值 / 退款）
- 改內容（不影響金額）→ 編輯（本 change）
- 合格後改內容 → 棄用新建（後續 change）
- 誤建整單 → 取消訂單
- 全然取消 → 取消訂單

因此本 change 的正確解法是：
1. **完全移除 Trash 按鈕**（非 disabled、非導引、直接拿掉）
2. **新增 Pencil 編輯按鈕**，白名單限制編輯範圍於非金額欄位
3. **新增訂單頂部變更路徑導引 info icon**，補償 Trash 移除後的業務心智錨點
4. **移除 Eye 按鈕**（與印件名稱 link 功能重疊）

## Goals / Non-Goals

**Goals**：
- 移除訂單印件列的 Trash 按鈕，消除「誤導業務以為可直接刪除」的互動債
- 新增編輯 Panel，支援非金額欄位的修改，填補「不知道如何編輯」的規格缺口
- 移除 Eye 按鈕，以印件名稱 link 作為唯一檢視入口，簡化操作欄
- 明定編輯欄位白名單，金額類永遠唯讀、其他欄位在非終態訂單下可改
- 以印件為 source of truth、訂單為唯一編輯入口，下游單據即時同步
- 訂單頂部新增變更路徑導引 info icon，協助業務快速識別正確路徑
- 建立 ActivityLog 事件類型 `PRINT_ITEM_EDITED`，記錄編輯行為供稽核

**Non-Goals**：
- 不處理金額類欄位編輯（數量、單位、成本、售價）
- 不實作補收款 / 取消訂單流程（各為獨立既有流程）
- 不實作報價單自動連動（跨模組範疇，僅於 ActivityLog + Toast 提示）
- 不實作印件異動觸發下游業務邏輯重算（僅確保顯示值同步，不重算排程 / 成本）
- 不處理合格後的「棄用 + 新建」流程（後續 change `add-print-item-deprecate`）
- 不修改 `/print-items/:id` 詳情頁的顯示內容

## Decisions

### D1：Trash 按鈕完全移除（不改為 disabled、不改為導引 Dialog）

**決策**：直接刪除 `OrderDetail.tsx` 印件列操作欄的 Trash 按鈕程式碼與對應 icon import。

**為何**：
- 所有訂單狀態下，印件層刪除皆無合法業務情境（金額變動統一走補收款、全然取消走取消訂單），保留按鈕只會誤導
- 導引 Dialog 方案會讓使用者產生「原來可以刪除但被系統擋住」的錯覺；訂單頂部的 info icon（D6）是更清楚的業務教學
- 金額變動的正確路徑（補收款 / 取消訂單）在訂單頁已有入口

### D2：Eye 按鈕移除，印件名稱 link 為唯一檢視入口

**決策**：移除印件列操作欄的 Eye 按鈕；檢視印件詳情統一透過「印件名稱 text link」進入。

**為何**：
- Eye 按鈕與印件名稱 link 目的地完全相同（皆 `navigate('/print-items/:id')`），業務容易混淆「點哪個」
- 印件名稱 link 符合 Web 心智模型（text link 代表可檢視）
- 操作欄空間需留給補件、編輯

### D3：編輯欄位白名單採「非終態訂單」二分，不再依印製狀態細分

**決策**：編輯 Panel 的欄位依訂單狀態簡單二分：
- **終態訂單**（已取消、訂單完成）：所有欄位禁止編輯
- **非終態訂單**：白名單欄位皆可編輯；唯一例外是「預計交貨日期」於印件已送達後鎖定

**為何**：
- 原先設計曾考慮依印製狀態（製作中 / 部分工單製作中）分段鎖定規格類欄位，以保護已下料 / 已投入工時的成本
- 但 Miles 的 source of truth 原則（D7）已確保下游單據同步印件最新值；分段鎖定反而阻斷業務合理的變更（如客戶改包裝說明、更新產線）
- 「成本已動、不該改」的顧慮，應由會計 / 主管流程管控（如編輯審核），不由系統強制鎖定所有規格欄位
- 「預計交貨日期」為唯一例外：已送達代表實體已寄出，交期失去意義

### D4：金額類欄位永遠唯讀（任何狀態皆然）

**決策**：購買數量、單位、成本總額、報價總額 SHALL 在任何訂單狀態下都於 Panel 中以 disabled 呈現。

**為何**：
- 直接對齊 Miles 明定規則「金額類不可編輯」
- 數量與單位雖非「金額本身」，但直接決定計價基礎（數量 × 單價 = 金額），視為金額類
- 金額變動有專屬路徑（補收款正負項 / 取消訂單），在 Panel 內開放編輯會繞過業務規則

**UX 處理**：不隱藏金額欄位，保留顯示但 disabled，並於該區塊下方顯示引導文字至正確路徑。讓業務在編輯時看到「這裡有金額但不能改」的提示，比完全隱藏更能強化規則認知。

### D5：ActivityLog 寫入粒度為「整筆編輯事件」

**決策**：每次儲存編輯 Panel 產生**一筆** ActivityLog 事件（而非每個欄位一筆），事件 content 包含該次修改的所有欄位差異。

**為何**：
- 業務實際操作是「一次改多個欄位後按確認」，以單次操作為 log 單位更貼近業務語意
- 多欄位改動視為同一決策的副作用，分拆會破壞追溯上下文

**欄位差異摘要規則**：
- 文字類欄位（規格備註、備註）：超過 80 字以 `...` 截斷
- 列表類欄位（預計產線）：完整記錄前後差集
- 日期 / 數字：完整記錄
- Boolean（免審稿）：記錄 true/false 轉換

### D6：訂單頂部新增「變更路徑導引」info icon

**決策**：於訂單詳情頁頂部「取消訂單」按鈕旁新增 Lucide `Info` 圖示按鈕（14px），hover 時顯示業務變更決策樹。

**為何**：
- Trash 移除後，業務看到「印件想刪」時沒有視覺錨點告訴他去哪；僅靠 Release Note 或口頭培訓效力有限
- info icon 是系統內建的業務規則教學，使用成本低、效果持久
- 四條路徑（加項 / 減項 / 改規格 / 全然取消）覆蓋訂單層所有印件相關變更情境

### D7：印件為 source of truth，訂單為唯一編輯入口

**決策**：印件資料由 PrintItem 物件統一管理，訂單詳情頁為唯一編輯入口；工單、生產任務、出貨單等下游單據 MUST NOT 保留印件欄位的獨立快照，顯示時即時讀取印件最新值。

**為何**：
- Miles 明定：印件資料貫穿多張單據；變更源頭必須單一，否則會出現「印件在訂單頁改了但工單頁顯示舊值」的資料打架
- Prototype 以共享 store 的 reactive 更新承擔同步（Zustand 的 selector 自動響應）；正式系統需以資料庫 JOIN read path 設計
- 下游自動化邏輯（如預計產線改動觸發工單排程重算）不在本 change 範疇；本 change 僅確保顯示值同步

**對 D3 的連動影響**：正因為 source of truth 原則確保下游同步，「製作中後鎖定規格欄位」的保守分段變成不必要（不會有「工單快取舊值」的風險），因此 D3 採二分而非三段。

## Risks / Trade-offs

**[Risk A] 業務習慣用 Trash 按鈕當作「想刪」的視覺標記**
- 情境：移除按鈕後業務困惑「我想刪怎麼沒地方點」
- Mitigation：D6 的 info icon 提供路徑教學；Lovable 上線需配合 Release Note

**[Risk B] 編輯白名單可能漏掉某些欄位**
- 情境：業務想改「印件類型（打樣/大貨）」或其他未列入的欄位
- Mitigation：白名單為保守起點；UAT 期間追蹤業務反映，後續 change 擴充

**[Risk C] 生產中改動規格類欄位可能造成成本基準落差**
- 情境：印件已進入生產，業務改「難易度」或「規格備註」；若難易度影響成本估算，改動後不回溯成本會造成帳實不符
- Mitigation：本 change 的編輯僅改顯示值、不回溯計算；會計 / 主管審核流程為補救機制；若 UAT 發現高頻影響，獨立 change 處理難易度鎖定時機

**[Risk D] Source of truth 的 reactive 同步在 Prototype 僅靠 Zustand store**
- 情境：若未來改為獨立頁面 server-side render，reactive 同步機制失效
- Mitigation：本 change 僅 Prototype 範圍；正式系統階段需重新設計 read path

**[Risk E] 編輯併發未處理**
- 情境：業務 A 開啟編輯 Panel 時，業務 B 同時改訂單狀態或稿件
- Mitigation：列為 OQ（本 change 不處理）；Prototype 假設單人操作

**[Trade-off] 放棄「編輯歷程查看 UI」**
- 本 change 僅寫入 ActivityLog，不新增「查看此印件歷次編輯」的專屬 UI
- 若業務需要「篩選特定印件的歷程」，獨立 change 處理

## Migration Plan

本 change 為 Prototype 新增 / 移除功能，無資料遷移。部署步驟：
1. 於 `sens-erp-prototype` 實作：
   - 移除 `OrderDetail.tsx` 的 Trash 與 Eye 按鈕
   - 新增 `components/order/EditOrderPrintItemPanel.tsx`
   - 新增 `useErpStore.updateOrderPrintItem` action + `ActivityLog` 寫入邏輯
   - 訂單頂部新增變更路徑導引 info icon + tooltip
2. 依 tasks.md 逐項實作與自測
3. Push 至 Lovable main 分支，由 Miles 於 Lovable 環境實測
4. 將 delta specs 透過 `opsx:archive` 合併回 main specs

**Rollback**：若編輯 Panel 出現欄位覆寫錯誤（例如誤把金額欄位寫入），UI 層 disabled 為第一道防線，`updateOrderPrintItem` action 的白名單過濾為第二道防線（實作時建議 throw error 而非靜默丟棄，便於測試抓漏）。

## Open Questions

1. **編輯時「預計交貨日期」改至更早日期是否觸發急件費提示**：本 change 不處理急件費邏輯，OQ 追蹤。
2. **編輯時「出貨方式」變更是否影響運費重算**：訂單有獨立運費欄位，OQ 追蹤。
3. **難易度在工單建立後改動是否應鎖定**：目前保留可改（僅更新顯示、不回溯成本）；若 UAT 發現高頻成本落差，獨立 change 處理。
4. **業務誤建是否也可走負數補收款（而非強制取消訂單）**：Miles 目前規則為「誤建走取消訂單」，CEO 審查建議彈性化；OQ 追蹤。
5. **數量微調 +/- 5% 的高頻場景是否應放寬白名單**：目前數量鎖定；若 UAT 發現業務頻繁踩到（走補收款太重），獨立 change 處理。
6. **編輯併發保護**：Prototype 單人假設；OQ 追蹤。
7. **ActivityLog 是否新增 `targetType / targetId` 欄位**：若未來支援「只看某印件的編輯歷程 UI」需要，OQ 追蹤。
