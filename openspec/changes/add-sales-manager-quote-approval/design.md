## Context

需求單流程「已評估成本 → 議價中」轉換目前由業務單方推進，缺少業務側的收款條件審核機制（[quote-request spec](../../specs/quote-request/spec.md) US-QR-002）。報價提供給客戶前，業務與部門主管之間的收款條件討論未走系統流程，造成：

- 業務在收款條件未對齊下對外報價（無稽核軌跡）
- 「業務主管」這個角色目前不存在於 [user-roles spec](../../specs/user-roles/spec.md)，部門治理無法落地至 ERP
- 與印務主管的指定機制不對稱（印務主管已在建立需求單時指定，但業務側無對應審核者）

本 change 在「已評估成本 → 議價中」之間插入強制 gate，並對稱新增「業務主管」角色，補齊業務側的審核閉環。範疇刻意限縮為「最小但完整的把關」（範疇 A），不擴散至 KPI / Dashboard / 訂單模組，避免一次改太多。

設計決策已與 Miles 完成 8 點對齊（範疇、強度、欄位、退回、平台、指定、Prototype 模擬、收款備註填寫時機）。三視角審查（Senior PM / CEO / ERP 顧問）後再追加 6 項補強決策（條件化預留、組織規模假設、Slack 通知對齊、US 拆分、防擺爛三層機制、Supervisor 解鎖機制）。

## Goals / Non-Goals

**Goals：**

- 為「已評估成本 → 議價中」轉換新增強制業務主管 gate，業務無法跳過
- 新增「業務主管」角色至 [user-roles spec](../../specs/user-roles/spec.md)，平台歸屬中台，需求單 R/W、其他模組 X
- 新增需求單欄位 `approved_by_sales_manager_id`、`payment_terms_note`、`approval_required`（條件化預留）
- 業務主管核可動作可被稽核（ActivityLog 含核可、不核可 / 退回討論、首次查看時間三類事件）
- 業務主管核可通知對齊印務主管的 Slack Webhook 機制
- Prototype 模擬業務從業務平台進入只看「需求單 + 訂單」、業務主管從中台進入只看「需求單」
- Phase 1 即提供 Supervisor 解鎖機制（重新指定業務主管），避免離職 / 請假卡單

**Non-Goals：**

- 不擴散至訂單模組（收款 / 發票仍走訂單模組既有設計）
- 不擴散至 KPI Dashboard（業務主管的 KPI 視角為 Phase 2，本 change 僅做埋點）
- 不提供業務主管「退回至待評估成本」的狀態轉換（既有 US-QR-006 由業務發起重新評估，不變；業務主管可留「退回討論」comment 但不改狀態）
- 不結構化收款條件（保持 free text，待後續報價單模組規劃時再評估）
- 不開放業務主管跨自己被指定範圍核可他人需求單（避免決策歸屬模糊）
- 不調整「議價中 → 成交 / 流失」的角色（仍為業務）
- 不做 Phase 1 條件化 gate（金額閾值 / 客戶類型觸發）— 但預留 schema

## Decisions

### 決策 1：範疇限縮為「單純把關 → 議價中」（範疇 A）

**選擇**：業務主管只介入「已評估成本 → 議價中」這一個轉換點，不進入需求確認、成本評估、議價對談、成交、流失等其他流程環節。

**Alternatives**：

- **B 部門主管角色**：業務主管同時擁有部門所有業務的需求單檢視權限，看下屬 KPI、流失分析。
- **C 業務職能全鏈**：業務主管同時審核流失、轉訂單，並擁有訂單模組權限。

**Rationale**：B 與 C 都觸發部門階層 / 跨模組設計，現階段未確認業務組織結構（人數、分組、跨業務協作關係）。先做 A 驗證業務流程的核可關卡是否對齊真實工作節奏，再決定下一步擴散方向。範疇蔓延的成本遠高於後續加功能。

### 決策 2：強制 gate（業務看不到「進入議價」按鈕）

**選擇**：「已評估成本」狀態下業務 UI **不**顯示「進入議價」按鈕，僅業務主管看得到「核可進入議價」按鈕。API 層面業務角色嘗試推進此狀態時回傳權限不足。

**Alternatives**：

- **軟控**：業務仍看到按鈕，UX 提示「需業務主管確認」，業務按下時觸發通知給業務主管。
- **雙按鈕**：業務按「送業務主管審核」→ 業務主管看「核可」。

**Rationale**：軟控等於沒做（業務可繼續按，無法強制）。雙按鈕新增中間狀態（「待業務主管核可」），增加狀態機複雜度且無價值（已評估成本本身已可承擔此語意）。直接讓業務看不到按鈕最乾淨，並對齊既有「印務主管評估」的 UI 設計（業務在「待評估成本」狀態看不到「評估完成」按鈕）。

### 決策 3：收款備註為 QuoteRequest 級 free text（任何時候可填）

**選擇**：新增 `payment_terms_note`（text，最長 500 字，選填），位於 QuoteRequest 級（非 QuoteRequestItem 級），業務在草稿到議價中任一狀態皆可編輯，終態（成交 / 流失）後鎖定。

**Alternatives**：

- **結構化欄位**：`payment_terms`（LOV：月結 / 即期 / 預付 / 分期）+ 分期比例
- **印件級**：`payment_terms_note` 放 QuoteRequestItem 層
- **限定狀態填寫**：只能在「已評估成本」狀態填寫

**Rationale**：

- Miles 確認「收款條件後續會變成報價單的內容」，報價單對應整張需求單，故放 QuoteRequest 級
- Free text 而非 LOV：印刷業 B2B 收款條件多元，結構化會限制業務描述
- 任何時候可填：業務跟客戶討論收款條件的時機不固定，限制狀態填寫反而造成業務在實際對話與系統登記之間擠壓記憶

**Trade-off**：free text 無法直接做收款條件分析。Phase 1 不追求此 KPI，可接受。

### 決策 4：業務主管不開新狀態轉換，但保留「退回討論」comment 動作

**選擇**：業務主管沒有「退回至待評估成本」的狀態轉換按鈕（單向核可），但提供「退回討論」非狀態轉換動作 — 寫入 ActivityLog（事件描述 + 退回原因 free text），需求單仍維持「已評估成本」狀態，業務看到此 ActivityLog 後決定是否走 US-QR-006 重新評估。

**Alternatives**：

- **β 業務主管可退回**：新增按鈕「退回重新評估」→ 狀態回「待評估成本」
- **γ 中間態**：新增「退回待議」狀態
- **α 完全無退回**：業務主管「不核可」就只是「不按按鈕」，業務看不到任何退回意見

**Rationale**：

- 三視角審查指出 α 會造成黑箱（業務不知為何被卡），β 增加狀態機複雜度
- 「退回討論」comment 動作折衷兩者：保持狀態機簡潔（單向核可），同時補溝通管道（業務主管表達不同意 + 理由）
- 業務看到 comment 後若認同，主動走 US-QR-006 重新評估；若不認同，可直接找業務主管討論。重新評估的決策仍由業務發起（最了解客戶反應）

### 決策 5：業務主管 = 中台（與印務主管對稱）

**選擇**：業務主管平台歸屬為中台。

**Rationale**：對稱印務主管設計（中台 + 需求單 R/W）。Miles 確認業務主管為專職中階主管，每日進系統處理待辦，符合中台介面定位（非 owner / 副總一週進兩次的角色形態）。

### 決策 6：建立時指定業務主管（與印務主管 `estimated_by_manager_id` 對稱）

**選擇**：新增 `approved_by_sales_manager_id`（FK 必填，建立時指定，進入待評估成本後鎖定）。業務主管的待辦清單僅顯示自己被指定範圍的需求單。

**Rationale**：對稱印務主管設計，降低「審核者責任分配」的歧義，並支援後續 KPI（業務主管平均審核時長、核可率）。「先到先審」會造成多業務主管同時看到所有 pending 需求單，責任歸屬不明。

### 決策 7：Prototype 平台模擬範圍

**選擇**：

- 業務角色登入 → 業務平台 → 側選單只顯示「需求單、訂單」
- 業務主管角色登入 → 中台 → 側選單只顯示「需求單」
- 角色切換選單新增「業務主管」選項
- 業務主管列表頁預設篩選 `approved_by_sales_manager_id = 當前使用者` + `status = 已評估成本`（即「待辦清單」的具體實作）

**Rationale**：Prototype 目前 ERP 全模組對所有角色開放，無法呈現平台分流的真實情境。本 change 同步調整側選單模擬，讓 UAT 時 Miles 可以實際走業務 / 業務主管視角驗證流程。

### 決策 8：收款備註不必填，但空備註核可需二次確認

**選擇**：`payment_terms_note` 為選填，業務主管核可時若該欄位為空，UI MUST 跳出 Confirm Dialog「此需求單無收款條件備註，確認已與業務口頭對齊？」需業務主管二次確認後才推進狀態。

**Alternatives**：

- **必填 gate**：核可按鈕在 `payment_terms_note` 為空時 disabled
- **完全無提示**：業務主管直接核可，無任何提醒

**Rationale**：

- 必填會造成業務隨便填字反而失去意義
- 完全無提示讓 gate 流於蓋章（三視角審查中 Senior PM 質疑 C + CEO 質疑 4 共同指出）
- Confirm Dialog 折衷：不阻擋實際口頭對齊的情況，但提高擺爛成本（需點兩次 + 系統留下「業務主管確認口頭對齊」的 ActivityLog）

### 決策 9：條件化 gate 預留 schema（Phase 2 升級路徑）

**選擇**：新增 `approval_required` boolean 欄位（Phase 1 預設 true，所有需求單必經業務主管核可），但 Data Model 與 Scenario 預留條件化升級空間。

**Alternatives**：

- **不預留**：Phase 1 與 Phase 2 schema 不同，未來條件化時所有歷史資料需回填
- **Phase 1 即條件化**：依金額 / 折扣率設規則

**Rationale**：

- 業界（NetSuite SuiteFlow / SAP Sales Approval）成熟做法為條件化審核，always-on gate 是反模式（ERP 顧問審查指出）
- Phase 1 即條件化的成本過高（需先定義條件規則，但目前缺乏資料支撐）
- 預留欄位 schema 成本極低（一個 boolean），但避免 Phase 2 改成條件化時需要對所有 in-flight + 歷史需求單回填的技術債
- Phase 2 升級路徑：依「報價金額」、「客戶類型」、「折扣率」、「收款條件特殊度」等規則動態決定 `approval_required = true / false`，false 時自動跳過 gate

### 決策 10：組織規模假設（業務主管每日使用 ERP）

**選擇**：本 change 假設業務主管為「每日進系統處理待辦的中階主管」，而非「一週進兩次的 owner / 副總」。

**Rationale**：

- CEO 視角審查指出「業務主管 = 中台 + 強制 gate」的設計對小型印刷廠（業務主管 = 老闆）不適用，會造成所有報價單卡死
- Miles 確認本 ERP 預期客戶為「業務主管每天開電腦使用 ERP」的組織形態
- 此假設限制本 change 的適用範圍：業務團隊規模 ≥ 業務主管屬「專職中階主管」的組織。若未來導入小型印刷廠（業務主管 = owner），需透過獨立 change 提供「Supervisor 兼任業務主管核可」或「金額閾值跳過 gate」等機制

**留下的 OQ**：本 change 不提供小廠模式切換；若 Miles 後續需支援多種組織形態，需另開 change 設計「角色合併規則」或「條件化 gate」。

### 決策 11：業務主管核可通知對齊印務主管 Slack Webhook 機制

**選擇**：需求單進入「已評估成本」狀態時，系統 SHALL 透過 Slack Webhook 通知指定業務主管，機制與 quote-request spec § Slack Webhook 通知（US-QR-005「待評估成本通知」）對齊。

**Rationale**：

- ERP 顧問審查指出「印務主管有 push、業務主管沒有」造成業務主管 lead time 失控
- Miles 確認業務主管使用 ERP 的方式與印務主管相同（含 Slack Webhook）
- 通知 permalink 寫回需求單既有 `slack_thread_url` 欄位（複用，不新增欄位）

### 決策 12：US-QR-002 拆分，業務主管核可獨立成 US-QR-015

**選擇**：US-QR-002 聚焦「業務管理需求單進度」（含填收款備註、看待辦狀態、執行成交 / 流失）；新增 US-QR-015「業務主管核可需求單進入議價」獨立 user story。

**Rationale**：

- Senior PM 審查指出單一 US 跨多角色（業務 + 業務主管）違反「單一 US 單一意圖」原則
- 拆分後 acceptance criteria 更清晰，UAT 可分別由業務、業務主管視角驗證

### 決策 13：防擺爛三層機制（ActivityLog + Confirm Dialog + 等待可見性）

**選擇**：本 change 同時實作三個防擺爛層：

- **層 1 — ActivityLog 三類事件**：業務主管首次查看需求單時間、不核可 / 退回討論動作（含理由）、核可動作。為 Phase 2 KPI（lead time、核可率、退回率）預留資料。
- **層 2 — 空備註 Confirm Dialog**：業務主管核可時 `payment_terms_note` 為空，需二次確認（決策 8）
- **層 3 — 業務側等待可見性**：業務於需求單詳情頁可見「等待 [業務主管姓名] 核可中（已等待 X 天）」，將業務主管的時間壓力顯性化

**Rationale**：

- 三視角審查共識：強制 gate 若無防擺爛機制，會退化為蓋章流程，反而比沒 gate 更糟（讓業務以為有審核而放鬆警覺）
- 三層機制成本低（埋點 + Dialog + UI 文字），但合計能顯著降低 gate 流於形式的風險
- Phase 1 不做 KPI Dashboard，但 ActivityLog 資料須先收集，否則 Phase 2 想分析時無歷史資料

### 決策 14：業務主管離職 / 請假由 Supervisor 解鎖（Phase 1 即實作）

**選擇**：Supervisor 角色 SHALL 擁有「重新指定需求單的 `approved_by_sales_manager_id`」權限，可解鎖既有鎖定欄位，將需求單轉派給其他業務主管。此操作 MUST 寫入 ActivityLog。

**Alternatives**：

- 留 OQ 給 Phase 2：業務主管離職時透過資料庫直接改

**Rationale**：

- ERP 顧問審查指出「現必填 + 鎖定的設計，業務主管離職那一刻所有 in-flight 需求單就卡住」
- 既有 user-roles spec 中 Supervisor 已有「不直接編輯業務資料但可全模組 R/W」的定位，對「重新指派」這種非業務動作合適
- Phase 1 不做完整代理人機制（複雜），但提供 Supervisor 後門避免上線首月就踩雷

## 欄位 Lifecycle 表

對齊 ERP 顧問審查建議，列出本 change 涉及的指定類欄位在各狀態下的可編輯性。包含既有 `estimated_by_manager_id` 以補齊 spec 一致性。

| 狀態 | `estimated_by_manager_id`（印務主管） | `approved_by_sales_manager_id`（業務主管） | `payment_terms_note` | `approval_required` |
|------|--------------------------------------|-----------------------------------------|---------------------|---------------------|
| 草稿 | 可編輯（必填） | 可編輯（必填） | 可編輯（選填） | 預設 true（系統設定，不可手動改） |
| 需求確認中 | 可編輯（必填） | 可編輯（必填） | 可編輯（選填） | 預設 true |
| 待評估成本 | 唯讀（鎖定） | 唯讀（鎖定，Supervisor 可解鎖重指） | 可編輯（選填） | 唯讀（系統設定） |
| 已評估成本 | 唯讀 | 唯讀（同上） | 可編輯（選填） | 唯讀 |
| 議價中 | 唯讀 | 唯讀（同上） | 可編輯（選填） | 唯讀 |
| 成交 / 流失 | 唯讀（終態） | 唯讀（終態） | 唯讀（終態鎖定） | 唯讀 |

**註**：`estimated_by_manager_id` 既有 spec 未明寫鎖定時機，本 change 順帶補齊（user-roles + quote-request spec 內呈現），對稱兩個欄位的行為。

## 資料可見範圍規則

對齊 ERP 顧問審查補洞。

| 角色 | 需求單可見範圍 |
|------|--------------|
| 業務 | `created_by = self`（自己建立的需求單）|
| 業務主管 | `approved_by_sales_manager_id = self` AND `status ∈ {已評估成本, 議價中, 成交, 流失}`（自己被指定且已進入評估後階段的需求單）|
| 兼具兩角色（資深業務 + 業務主管） | 上述兩條件取聯集 |
| Supervisor | 所有需求單（既有規則）|
| 訂單管理人 | X（既有規則：需求單模組無權限）|

**待辦清單定義**：業務主管登入中台後的需求單列表頁，預設套用篩選 `approved_by_sales_manager_id = self AND status = 已評估成本`，並按 `已評估成本進入時間` ASC 排序（最久的優先）。此篩選可由業務主管手動切換為其他狀態查詢。

## Risks / Trade-offs

| Risk | Mitigation |
|------|-----------|
| 業務主管成為瓶頸：請假 / 離職造成卡單 | Supervisor 解鎖機制（決策 14），Phase 1 即實作；通知對齊印務主管 Slack Webhook（決策 11）降低 lead time |
| 業務主管擺爛 / 蓋章：核可流於形式 | 防擺爛三層機制（決策 13）：ActivityLog 埋點 + Confirm Dialog + 等待可見性 |

## Open Questions

下列 OQ 不阻擋本 change 完成，但需於後續迭代中釐清。建立此 change 時已搜尋 Notion OQ DB（[Follow-up DB](https://www.notion.so/32c3886511fa808e9754ea1f18248d92)），與本 change 範疇直接相關 / 部分回應的 OQ：

- [需求單檢視權限設計](https://www.notion.so/32c3886511fa81938415ed25260db279)：原 OQ 提到「業務助理或主管的協作需求」。本 change 透過新增業務主管角色 + 指定機制部分回應「主管協作」，但「主管完整檢視部門所有業務需求單」（即跨指定範圍的部門級檢視）仍未涵蓋（Phase 2）。

新衍生 OQ（建議於本 change 歸檔後新增至 OQ DB）：

1. **業務主管是否可在進入待評估成本後變更（解鎖）？**
   - 本 change 已實作 Supervisor 解鎖機制（決策 14），但業務主管自己 / 原業務能否申請變更尚未定義
   - 對稱印務主管同樣鎖定，建議共通設計
   - 影響：需求單編輯權限規則

2. **收款備註後續帶入報價單 / 訂單時的同步 vs 脫鉤策略？**
   - 對齊既有 `client_note` 設計模式（quote-request spec § 需求單印件稿件備註）：成交轉訂單時帶入後脫鉤
   - 待報價單 / 訂單模組延伸時釐清

3. **業務主管核可比例 / 平均核可時長如何度量？**
   - Phase 2 KPI Dashboard 範疇
   - 本 change 已埋 ActivityLog 三類事件作為資料基礎

4. **多印件混合不同收款條件如何處理？**
   - 目前 `payment_terms_note` 為 QuoteRequest 級，假設整張需求單收款條件一致
   - 若實際業務發現需印件級，需升級為 QuoteRequestItem 級欄位

5. **預設業務主管偏好設定**：
   - 業務每次建立需求單需手動指定業務主管，實務上多半固定向同一主管報告
   - 是否新增 user 級偏好設定（每個業務 user 設定預設業務主管）？建立需求單時自動帶入但可改

6. **業務主管「不核可的理由」溝通管道升級**：
   - 本 change 提供「退回討論」comment（ActivityLog 記錄），但無 threaded discussion / 通知業務的 push 機制
   - 後續若實務上發現主管 / 業務在 ActivityLog 來回對話頻繁，考慮升級為正式 comment thread

7. **報價單對外 PDF 的核可印記**：
   - 三視角審查中 CEO 提出「把把關點下沉到對外文件，比擋住內部狀態機更實際」
   - 待後續報價單模組規劃時納入考量（PDF 印「業務主管核可：XXX、核可時間：YYYY-MM-DD」）

8. **業務系統外繞過行為的監測**：
   - 業務若先口頭報價再補建單，ERP 內無法檢測
   - 是否需在 Phase 2 設計「報價單發送時間」欄位作為對外時間戳，與系統內核可時間 cross-check？

9. **小型印刷廠（業務主管 = owner）支援**：
   - 本 change 限定組織規模假設（決策 10）
   - 未來若需支援小廠，獨立 change 設計：(a) Supervisor 兼任業務主管核可路徑；(b) 條件化跳過 gate（金額 / 客戶類型）
