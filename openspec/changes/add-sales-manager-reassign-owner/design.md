## Context

本設計為序列協作（PM → CEO → ERP 顧問 → PM 收斂）Phase 1-4 的產出，加上 Miles 在 Phase 4 後對三個邊界 OQ 的拍板。

商業需求：業務主管需能改派諮詢單 / 需求單 / 訂單三種前段業務單據的負責業務，解決業務離職 / 長假名下單據無人接手（漏單痛點）與日常負荷不均。

當前狀態：三單據皆已有負責人欄位（諮詢單負責諮詢人員、需求單與訂單負責業務），但詳情頁皆純文字顯示、無改派機制；store 無改派 action。既有「分享」機制（職務代理 / 檢視授權，不改負責人）已存在（沿用權限管理元件）。

商業層 ground truth：[訂單](../../../memory/erp/ERP_Vault/05-entities/訂單.md)、[需求單](../../../memory/erp/ERP_Vault/05-entities/需求單.md)、[諮詢單](../../../memory/erp/ERP_Vault/05-entities/諮詢單.md)、[業務主管](../../../memory/erp/ERP_Vault/03-roles/業務主管.md)。

## Goals / Non-Goals

**Goals:**
- 業務主管於三單據詳情頁逐張改派負責業務（改負責人），即時生效
- 「改派」（改負責人，限主管）與「分享」（不改負責人，業務可做）明確區分
- 改派理由結構化分類，作為「通知 / 分享成員處置 / 稽核留痕」共同樞紐
- 改派全程留痕（五要素），支撐績效歸屬時間切分
- 完成訂單可改派，售後歸屬隨之轉移
- 改派候選人以角色模組權限篩選

**Non-Goals:**
- 批次改派、自動改派規則（後驗）
- 改審核業務主管 / 改印務主管（不同語意的轉單，不在本次）
- 分享機制本身的修改
- 名下單據停滯主動偵測（dashboard 類後驗 epic）

## Decisions

### D1：改派理由結構化分類為共同樞紐（CEO + 顧問）
- 決策：新增改派理由 `reassign_reason_category` 五值（離職交接 / 長假代理 / 工作負荷平衡 / 客戶要求換窗口 / 其他，必填）+ `reassign_note`（選填）。此單一分類驅動四個下游：通知對象（D6）、分享成員處置（D5）、稽核留痕（D7）、改派事件本身。
- 為何：避免把通知 / 分享 / 留痕做成四個獨立功能；對齊既有 `cancel_reason_category`（諮詢取消）/ `lost_reason`（需求單流失）的結構化分類模式。
- 落地位置：`reassign_reason_category` 落地為**活動紀錄事件 payload**，而非單據持久化欄位。理由：一張單可被改派 N 次、每次理由不同，存單據只能保留最後一次，存事件才能追溯歷史。對齊既有 OrderAdjustment `audit_log[]` 模式。
- Alternatives：純自由文字理由（駁回：無法驅動 D5/D6 分流、稽核不可解釋）；落地單據欄位（駁回：無法保留多次改派歷史）。

### D2：改派只換人，MUST NOT 改變任何單據狀態（顧問）
- 決策：負責人維度與單據狀態維度正交。改派只換負責人，不回退、不推進狀態。
- 為何：owner 是「誰負責」，狀態是「流程走到哪」，兩者獨立。改派觸發狀態轉移會破壞流程完整性。
- 參照：state-machines spec 三單據狀態機。

### D3：完成訂單可改派 + 售後負責人衍生自訂單負責人（Miles Phase 4 拍板，修正顧問設計）
- 決策：**訂單未取消即可改派（含已完成）**；售後服務的負責人衍生自訂單負責人，改派訂單負責人連帶轉移售後歸屬。已取消訂單禁改派（無售後意義）。需求單 / 諮詢單維持終態禁改派（成交已轉訂單 / 流失 / 諮詢完成 / 取消，皆無售後繼承問題）。
- 為何：顧問原設計「終態一律禁改派」，理由是「終態無後續業績」。Miles 指出顧問當時不知道的既有業務規則——**售後服務單掛在訂單下、售後負責人即訂單負責人**。因此業務離職後，完成訂單的售後要有人接，就必須能改派該完成訂單的負責人。這也直接回應顧問 challenge 1（終態訂單售後真空）與 AFT-1 的前段部分。
- 對齊：order-management v1.13「訂單未取消即可編輯」粗粒度規則。
- Alternatives：終態一律禁改派（顧問原案，駁回：完成訂單售後無人接）；售後服務單獨立負責人欄位（駁回：Miles 明示售後負責人衍生自訂單，不需獨立欄位）。

### D4：改派候選人以角色模組權限篩選（Miles Phase 4 拍板，修正顧問設計）
- 決策：ERP 無「部門」概念，使用者透過 Role 設定可存取的模組。改派候選新負責人 = 具對應模組權限的使用者（訂單 → 具訂單管理權限 Role、需求單 → 具需求單權限 Role、諮詢單 → 具諮詢權限 Role）；改派操作者 = 具業務主管權限 Role 的使用者。
- 為何：顧問原寫死「候選=業務+諮詢」。Miles 澄清權限模型是 Role-模組制（非角色寫死），候選應以「能存取該模組的 Role」動態篩選，更通用且對齊既有權限架構。
- Alternatives：寫死角色清單（顧問原案，駁回：不符 Role-模組權限模型、未來新角色須改 code）。

### D5：分享成員處置依理由分類（CEO + Miles Phase 4 拍板）
- 決策：改派後既有分享成員（協作檢視關係）——離職交接 → 清空；其餘全部（長假代理 / 工作負荷平衡 / 客戶要求換窗口 / 其他）→ 保留。
- 為何：CEO 提「依理由分類處置」；Miles 拍板「負荷 / 客戶要求 / 其他一律保留」。理由：清空是不可逆的「讓人看不到」，保留最安全；真要清，主管可事後手動移除。僅離職（原負責人已走、協作關係失效）才清空。
- Alternatives：一律清空（駁回：誤砍合理協作）；負荷 / 客戶要求清空（Miles 駁回：不可逆風險）。

### D6：改派後通知依理由分類（CEO + 顧問）
- 決策：新負責人一律通知（接手者須知道，否則製造新漏單）；原負責人依理由分類——離職交接 → 不通知（已離職無意義）；其餘 → 知會（在職者有權知道單據被移走）。管道：Slack mock（console.warn）+ 活動紀錄。
- 為何：對齊既有 Slack mock 通知模式（諮詢取消 / 大額補收）。

### D7：全公司範圍 + 稽核留痕兜底越權（CEO + Miles Phase 4 拍板）
- 決策：改派範圍全公司（任一業務主管可改任一單據負責人），不做部門 gating。標註「改派 = 管理動作（全公司）≠ 核可 = 部門內」。越權風險由稽核留痕兜底（五要素含操作主管，可追查）。
- 為何：ERP 無部門概念（Miles 拍板），硬做部門限制做不到；業務離職接手常需跨組支援。對齊 ORD-024「業務主管中台全公司可見」。
- 稽核留痕五要素：原負責人 / 新負責人 / 改派時間 / 理由分類+補述 / 操作主管。

### D8：業務「轉單」收斂為「僅分享」（顧問，supersession）
- 決策：user-roles spec 既有「業務角色可轉單」（業務可將單據指派給其他業務 / 諮詢）收斂為「業務僅可分享（不改負責人），改派負責人收歸業務主管」。
- 為何：與「改派限主管」需求直接衝突，屬解既有 spec 內部矛盾。
- delta op 紀律：MUST 標 **MODIFIED**（非 ADDED）——archive sync 按 exact-title 只增不刪，ADDED 取代會導致主 spec「業務可轉單」與「業務僅分享」並存矛盾。

### D9：設計模式對照（顧問）
- 已套用：狀態碼結構化（reason_category 樞紐）、稽核鉤子（ActivityLog 五要素剛性）、合格/完成終態（終態規則，惟訂單完成因售後例外可改派見 D3）。
- 不適用：當前版本指針（改派無多版本歷史實體）、B2C/B2B 分流（改派為內部管理動作）。

## Risks / Trade-offs

- [全公司範圍越權：任一主管可改非自己管轄單據] → 稽核留痕兜底（D7），可事後追查；無部門概念下這是務實選擇。
- [三單據終態規則不一致：訂單完成可改派、需求單/諮詢單終態不可] → 有明確業務理由（只有訂單有售後負責人繼承），非設計疏漏；在 spec 與 Prototype 明確標註差異。
- [reason_category 落地事件而非單據欄位，查詢「當前改派理由」需讀最新事件] → 對齊既有 OrderAdjustment audit_log 模式，可接受；改派理由本就是歷史性資訊。
- [候選人以 Role 模組權限動態篩選，依賴 Role-模組權限資料正確] → 沿用既有權限架構，不新增權限模型。

## Migration Plan

Prototype 階段，無資料遷移：三單據負責人欄位既有，僅語意升級為「可改派」；新增活動紀錄事件型別與改派理由 LOV。無 rollback 複雜度（純新增能力 + 一處 supersession 收斂）。
