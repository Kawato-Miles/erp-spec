## Context

目前訂單建立僅有「線下單由需求單轉入」與「EC 線上單 API 同步（Phase 2）」兩條路徑。EC 會員來電要求印件調整時，業務無法在 ERP 直接建立訂單。需新增「客製單」類型與手動建立功能。

現有 Prototype 狀態：
- `pages/OrderList.tsx`：已有「建立訂單」按鈕（line 66-69），但未連接任何面板
- `types/order.ts`：`OrderType = '線下單' | '線上單EC'`，需新增「客製單」
- `data/mockOrders.ts`：已有線下單 mock 資料，需新增客製單範例
- `components/quote/CreateQuotePanel.tsx`：右側 Sheet 面板建立模式，可作為 UI 參考

## Goals / Non-Goals

**Goals:**
- 業務可在訂單列表頁手動建立客製單（含多筆印件）
- 客製單走線上單狀態機路徑（等待付款 → 已付款 → 共用段）
- 印件支援免審稿選項
- Phase 1 付款狀態由業務手動推進

**Non-Goals:**
- ERP → EC 反向同步（Phase 2）
- EC 線上單 API 自動同步（Phase 2）
- 客製單的付款自動偵測（Phase 2，依賴 EC 同步）
- 訂單編輯功能（本次僅做新增）

## Decisions

### D1: 訂單類型新增「客製單」

order_type 從二選一改為三選一：線下單 / 線上單EC / 客製單。

**理由**：客製單的建立方式（ERP 手動）和狀態機入口（等待付款）都與線下單不同，明確區分可避免後續統計和流程判斷混淆。

**替代方案**：歸類為「線上單EC」並用標記區分 → 放棄，因為 EC 線上單預期為 API 自動同步，手動建立的客製單在來源和建立方式上有本質差異。

### D2: 狀態機走線上單路徑

客製單初始狀態為「等待付款」，與 EC 線上單共用同一條前段路徑。

```
客製單：手動新增 → 等待付款 → 已付款 → 稿件未上傳 → ...（共用段）
```

**理由**：客製單的業務場景是 EC 會員來電，付款仍在 EC 端進行。Phase 1 由業務手動標記「已付款」，Phase 2 改為 EC 付款自動觸發，路徑不變。

**OrderStatus 需擴充**：現有 Prototype 的 `OrderStatus` 缺少「等待付款」「已付款」狀態，需新增以對齊 state-machines spec。

### D6: OrderStatus 類型擴充

現有 Prototype `OrderStatus` 需對齊 state-machines spec：

```
現有：'草稿' | '待確認' | '已確認' | '生產中' | '出貨中' | '已完成' | '已取消'

擴充為（對齊 spec）：
'報價待回簽' | '已回簽' |           // 線下前段
'等待付款' | '已付款' |             // 線上 / 客製前段
'稿件未上傳' | '等待審稿' |         // 共用段（審稿）
'製作等待中' | '工單已交付' |       // 共用段（製作）
'製作中' | '製作完成' |             // 共用段（製作）
'出貨中' | '訂單完成' |             // 共用段（出貨）
'已取消'                           // 終態
```

需同步更新 `OrderStatusBadge.tsx` 的樣式對應。

### D7: CreateOrder 改為全頁表單

原設計 D3 採右側 Sheet 面板（560px 寬），實作中改為全頁表單（路由 `/orders/new`）。

**理由**：客製單欄位多（顧客資訊、發票設定、物流設定、印件清單），Sheet 面板空間不足，全頁可容納更完整的欄位佈局。

**表單區塊順序**：訂單與顧客資訊 → 交期 → 備註 → 發票設定 → 物流設定 → 印件清單 → 付款設定 + 訂單金額試算（50/50 佈局）

### D8: 印件清單改為可編輯 Table（erp-table 樣式）

原設計 D4 為動態卡片列，實作中改為 `erp-table` 可編輯列（`row-editable` class）。

**理由**：表格形式利於多欄位並列比較，配合後續規格品帶入，UX 更直覺。

**技術決策**：可編輯 Table 內的下拉/Combobox 元件，統一使用 `PortalCombobox`（`components/shared/PortalCombobox.tsx`），透過 ReactDOM Portal + fixed 定位解決 overflow-x-auto 截切問題。後續所有可編輯 Table 內的搜尋下拉均應沿用此元件。

### D9: 規格品（ProductTemplate）快速帶入與成本估算

業務輸入印件名稱時，系統顯示規格品清單供選取，選取後自動帶入：單位、單位成本、單位售價、規格備註。

**數量聯動計算**（選取規格品後）：
- 成本 = 單位成本 × 數量（數量變更時自動重算）
- 金額（含稅）= 單位售價 × 數量（數量變更時自動重算）
- 業務可手動覆寫成本與金額欄位進行微調

**目的**：讓業務在建立客製單時可快速估算印件成本與報價，降低手動計算負擔。

### D10: 顧客資訊與訂單資訊合為一個區塊

顧客資訊（聯絡人、地址、統編等）與訂單資訊（客戶名稱、接單業務等）合併在同一個表單卡片中。

**理由**：客製單情境下，「客戶名稱」即為選取的 EC 會員，顧客資訊是對該會員的補充，分開卡片反而增加操作跳躍感。

**搜尋邏輯**：客戶名稱欄支援依姓名、手機、統編、信箱、公司抬頭篩選 EC 會員，選取後自動帶入顧客資訊各欄位（會員類型除外，不可修改）。

### D11: 訂單金額試算 50/50 佈局

付款設定（折扣、紅利折抵）與訂單金額試算並排為 50/50 grid，讓業務調整折扣時可即時看到總金額變化。

**試算公式**：總金額 = 商品金額 - 折扣 - 紅利折抵 + 運費

### D12: 建立後導向訂單詳情頁

確認建立後 `navigate('/orders/${newOrderId}')`，直接進入該訂單詳情頁，不返回列表。

**理由**：業務通常在建立後需立即確認資料或繼續操作（如標記付款），減少多餘的頁面跳轉。

### D13: 訂單詳情頁新增顯示區塊

為對應新收集的欄位，OrderDetail 新增：
- 顧客資訊卡片（含會員類型）
- 發票設定卡片（有開立發票才顯示）
- 物流設定卡片（有設定出貨方式才顯示）
- 金額卡片新增紅利折抵列

## Risks / Trade-offs

**[風險] OrderStatus 擴充影響現有 mock 資料與 UI** → 需同步更新 mockOrders 的 status 值與 OrderList 的篩選器選項。影響範圍：OrderList、OrderDetail、OrderStatusBadge。

**[風險] 客製單無需求單來源，訂單詳情頁「來源需求單」區塊為空** → Prototype 已有 quote_request_id nullable 設計，UI 顯示「無」或隱藏即可。

**[取捨] Phase 1 手動標記付款 vs 自動偵測** → Phase 1 接受手動操作的效率損失，換取不依賴 EC 同步即可上線。Phase 2 補齊後自動化。

**[取捨] 免審稿跳過「等待審稿」狀態** → 若所有印件皆為免審稿，訂單從「稿件未上傳」可直接跳至「製作等待中」。此邏輯需在 state-machines spec 明確記載，避免實作時遺漏。
