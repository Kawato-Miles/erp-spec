## Context

審稿模組經 add-prepress-review / refine-prepress-review-scope / add-order-review-rejected-status / refactor-review-round-model 四次迭代後，資料結構（ReviewRound 雙端聚合）與業務流程（補件迴圈）已穩定。但審稿主管的**營運監控工具**與審稿員的**跨部門對帳工具**仍不足。本 change 基於穩定的資料結構加上工具層。

## Design Decisions

### D1：列表保持印件粒度（不改為 Round 粒度）

**選項**：
- A. 印件粒度（一印件一列，現況）
- B. Round 粒度（一 Round 一列，審稿員對帳可見每次審稿紀錄）
- C. 切換模式（UI 提供粒度切換）

**決定：A（不改）**

**理由**：
1. **主管日常使用**：主管每天看待審印件壓力，語意是「有哪些印件壓在系統裡」，印件粒度直觀
2. **對帳情境可解**：審稿員對帳 = 用 reviewerId + 時間區間 + status=合格 filter 出印件清單 → 點印件看歷史輪次即可逐輪確認
3. **避免複雜化**：切換粒度會增加 UI 複雜度，對現有主管用法造成干擾
4. **Round 粒度重複率**：同一印件多輪審稿若都在時間區間內，Round 粒度會重複顯示，對主管造成雜訊

**反對意見**：審稿員對帳時若同一印件多輪都合格（不存在的情境），印件粒度看不到每輪筆數。但實務上「合格」是終態，同一印件只會有一次 `result = '合格'` Round，此顧慮無實質影響。

### D2：時間區間篩選邏輯（以 ReviewRound 為判斷依據）

**選項**：
- A. 依印件建立時間（createdAt）
- B. 依印件 ReviewRound 任一 `reviewedAt` 落在區間
- C. 依印件 `submittedAt`（首輪業務送審）

**決定：B**

**理由**：
- A 不貼近業務語意（「今日新進稿」跟印件建立時間無關）
- C 只看首輪，補件後重審若跨區間會被遺漏
- B 最接近「今日有動到這件」的審稿員語言；對帳情境也適用

**實作**：`items.filter(pi => pi.reviewRounds?.some(r => r.reviewedAt && isInRange(r.reviewedAt, range)))`

### D3：Summary bar 計算以印件為單位（非 Round）

**原則**：summary bar 顯示的是「印件數量」而非「Round 數量」。

- 待處理 N：當前 reviewDimensionStatus ∈ {等待審稿, 不合格, 已補件} 的印件數
- 合格 N：該印件於時間區間內有任一輪 result='合格' 的印件數
- 不合格 N：該印件於時間區間內有任一輪 result='不合格' 的印件數

**邊界**：同一印件若在區間內先不合格後合格（補件迴圈），**同時計入合格 + 不合格**（反映該印件這段時間經歷過兩種事件）。Miles 確認這個 double-count 是合理的（業務語言說「今天有 5 件合格、2 件不合格」，不排斥重疊）。

### D4：處理時間公式 α（含排隊時間）

**公式**：`avg(reviewedAt - submittedAt)` 以 reviewerId 分組計算。

**特性**：
- 含排隊時間（業務送審 → 審稿員打開 → 審完）
- 審稿員負擔高時數字膨脹，負擔低時數字接近真實審稿時間
- 不需要新增資料欄位（openSpec 已有的欄位即可算）

**限制**：
- 抓不到「刷工時」的偷懶（短時間連續送出）
- 負擔異常高時，數字高 ≠ 偷懶（可能只是工作量爆）

**折衷**：本 change 標示為「參考指標」，不作為異常判定依據。排序仍以退件率為主。未來若需要精準抓偷懶，補 β 公式（`reviewedAt - 上件 reviewedAt`）作為進階指標。

### D5：不標示異常 ⚠️ 徽章（純展示 + 排序）

**問題**：Baseline 未知，硬寫閾值（退件率 > 50%、處理時間 < 1 分鐘 = ⚠️）會誤導。

**決定**：個人對比表純顯示數字 + 依退件率降冪排序。主管肉眼看排序前段的人自行判斷。

**理由**：
- 公司無「退件率應為多少」的歷史基準
- 審稿員總數可能不多（3-5 人），樣本數不足以算統計異常
- UAT 跑一段時間後可依實際資料定 proxy 閾值（留批次 2）

### D6：時間範圍預設「今日」

**選項**：
- A. 預設「今日」（主管每天看）
- B. 預設「本週」（主管一週回顧）

**決定：A**（Miles 確認）

**理由**：
- 主管訪談提的核心情境是「每天早上看」
- 週回顧可透過手動切換「本週」達成
- 預設今日 = 主管打開就看到最新資訊；預設本週 = 主管打開看到昨天以前的舊資訊

### D7：L1 今日卡片不與個人對比表分 Tab

**選項**：
- A. L1 卡片放 SupervisorDashboard 頂部（Tabs 上方常駐）
- B. L1 卡片獨立 Tab（新增「今日」Tab）

**決定：A**

**理由**：
- L1 是「永遠可見的巡檢數字」，放 Tabs 上方每次切換 Tab 都看得到
- 分 Tab 會讓主管進頁面要先切 Tab 才看到這些數字，違反 10 秒巡檢語意

### D8：個人對比表 Tab 位置

**選項**：
- A. 「審稿人員」Tab 擴充（現有負擔表下方加對比表）
- B. 新增獨立「個人對比」Tab
- C. 取代現有「審稿人員」Tab 的負擔表（合併為一個表）

**決定：C**

**理由**：
- 現有「審稿人員」Tab 的負擔表只有進行中 / 完成總數 + 派案等級編輯；個人對比表包含更豐富指標（退件率、處理時間、時間範圍切換）
- 合併為單一表，欄位包括：審稿員、派案等級（含編輯按鈕）、進行中件數、時間區間內件數、退件率、平均處理時間
- 減少 UI Tab 數量，資訊在同一視圖內

**反對意見**：合併後欄位變多，表格可能擁擠。緩解：欄位固定順序、可響應式折疊次要欄位。

## Alternatives Considered

### A1：在 ReviewerInbox 加「完成紀錄」Tab

審稿員本人從自己收件匣切 Tab 看完成清單，不用去 InProgressItems。

**拒絕理由**（Miles 確認）：
- 重複 UI（InProgressItems 已提供相同能力）
- 審稿員對帳情境低頻，不需為之新建 Tab
- 保持 ReviewerInbox 純淨（待審清單單一職責）

### A2：列表改為 Round 粒度

審稿員對帳直接看每筆 Round。

**拒絕理由**（見 D1）：破壞主管日常使用語意；印件粒度已能解對帳情境。

### A3：硬寫退件率異常閾值（> 50% 顯示 ⚠️）

**拒絕理由**（見 D5）：Baseline 未知，誤導風險高。

### A4：處理時間用 β 公式（`reviewedAt - 上件 reviewedAt`）

**拒絕理由**（見 D4）：需跨日 / 斷點規則，實作複雜；本 change 優先可用性，β 作為未來進階。
