## Context

印務為工單排定生產時程時，需要處理多道工序之間的先後順序。現況是逐筆手動填開工日期，沒有系統輔助回推。生產任務 spec 已規劃 Phase 2 自動派工建議，本次先以 prototype 驗證 UX。

核心模型：「分階段」管理工序相依性。

```
階段 1（平行）    階段 2         階段 3       階段 4
┌──────────┐    ┌──────────┐   ┌──────────┐  ┌──────────┐
│ 封面印刷  │    │ 上霧膜   │   │ 裁切     │  │ 膠裝     │
│ 內頁印刷  │ →  │          │ → │          │→ │          │
└──────────┘    └──────────┘   └──────────┘  └──────────┘
  同階段平行       等前置完成     等前置完成     等前置完成
```

## Goals / Non-Goals

**Goals:**
- 驗證「分階段 + 自動回推」UX 是否直覺、可操作
- 驗證時間軸呈現是否足夠讓印務快速判斷排程合理性
- 驗證階段管理的互動模式（拖拉 or 下拉選單）

**Non-Goals:**
- 不實作後端 API
- 不處理週末/假日（Phase 1 用日曆天）
- 不實作設備負載衝突檢測
- 不依數量自動推算工期（Phase 1 印務手動填）
- 不影響現有工單狀態機流程

## Decisions

### D1: 階段管理互動 — 下拉選單指定階段

**決策**：每筆生產任務用下拉選單選擇所屬階段（1、2、3...），不用拖拉。

**替代方案**：拖拉式看板 → 實作成本高，且生產任務列表已有表格模式，切換為看板不一致。

**理由**：下拉選單與現有表格相容，操作簡單，prototype 驗證夠用。

### D2: 回推演算法 — 從交貨日倒推

**決策**：
```
演算法：
1. 取工單交貨日（expectedDate）為終點
2. 從最後一個階段開始倒推
3. 每個階段的持續天數 = max(該階段所有任務的 estimated_duration_days)
4. 該階段完成日 = 下一階段開始日（最後階段完成日 = 交貨日）
5. 該階段開始日 = 完成日 - 持續天數
6. 各任務的 scheduled_date = 所屬階段開始日
7. 各任務的 planned_end_date = scheduled_date + 該任務 estimated_duration_days
```

**理由**：倒推邏輯直覺，同階段取 max 確保平行任務都能在階段結束前完成。

### D3: 時間軸呈現 — 水平甘特圖

**決策**：排程結果以水平甘特圖呈現，X 軸為日期，Y 軸為各生產任務。以階段色塊區分，標示交貨日紅線。

**理由**：甘特圖是排程的標準視覺化，印務容易理解。

### D4: UI 位置 — 工單詳情頁新增「排程規劃」Tab

**決策**：不建新頁面，在工單詳情頁的 Tab 列新增「排程規劃」，與生產任務 Tab 並列。

**理由**：排程是工單層級的操作，放在工單詳情頁符合資訊架構。

### D5: 工期來源 — 印務手動填 + 工序預設值

**決策**：每筆生產任務有 estimated_duration_days 欄位，系統依工序給預設值（如數位印刷=2天、上光=1天），印務可覆寫。

**理由**：Phase 1 最簡方案，後續可依設備和數量自動推算。

## Risks / Trade-offs

- **日曆天 vs 工作日** → Phase 1 先用日曆天，排程可能跨週末。Mitigation：後續加工作日設定。
- **階段數量不固定** → 有的工單只有 2 階段，有的可能 6-7 階段。Mitigation：動態新增/刪除階段。
- **回推結果可能在過去** → 如果工單交貨日太近，回推的開始日可能是今天以前。Mitigation：UI 標示警告「排程起始日已過」。
