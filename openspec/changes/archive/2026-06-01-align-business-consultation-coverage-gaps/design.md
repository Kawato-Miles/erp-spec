## Context

本輪業務 / 諮詢日常工作情境盤點識別出 12 項 spec 缺口，加上 user story 拆乾淨後的設計細化（訂單前段流程明確化、退款流程三組件組合明示）。Notion User Story DB 已含 35 個新 user story（CR/ORD/PO/AS/SP/PI 六組），本 change 補齊 spec 對齊這些 user story 與設計決議。

既有 spec 已有相關 Requirement：
- `order-management` § 業務主管核准訂單（推進至「報價待回簽」一步到位）
- `order-management` § 付款計畫變更觸發訂單回業務主管審核（變更即回審）
- `state-machines` § 訂單狀態機（線下路徑：報價待回簽 → 已回簽 → 共用段）

本輪設計決議改變這些既有 Requirement 的行為：
1. 業務主管核可 → **不再直接推進至「報價待回簽」**，先進入「審核通過」中間狀態
2. 業務透過外部管道送報價單給客戶後 → 手動推進至「報價待回簽」
3. 付款計劃變更稽核 → **分階段邏輯**（主管審核通過前不稽核 / 通過後直接編輯但記稽核），取代「變更即回審」

## Goals / Non-Goals

**Goals**：

- 訂單前段流程細化為四階段（草稿 / 待主管審核 / 審核通過 / 報價待回簽），業務知道每張單在哪個階段
- 退款流程三組件邊界明確：訂單異動單審批 / 退款金流 / 發票異動三者互不重疊、各自可獨立驗證
- 發票作廢 vs 折讓的 UI 與後端規則清楚分離（UI 業務自主選擇 / 後端依第三方平台實際回應引導）
- 多品項發票稅額計算規則明示（每品項分別計算 + 差額集中最後品項 + 禁止平均分攤）
- 客戶資料 relation 帶出規則明示（廠客資料異動自動同步 + 已開發票保留快照）
- 訂單複製功能補完（reorder 場景 + 新訂單 source_order_id 追蹤）

**Non-Goals**：

- 本 change 不實作 Prototype（後續另開實作 task）
- 不重新設計訂單前段流程的 UI（沿用既有訂單詳情頁，僅增加按鈕與狀態顯示）
- 不調整工單 / 製作階段 / 出貨段相關 spec（聚焦業務 / 諮詢日常工作範圍）
- 不處理諮詢單模組獨立 spec 化（CR 模組仍以 OrderType=諮詢訂單方式併入 order-management，未來再開 change 拆出獨立 spec）

## Decisions

### A. 訂單前段流程細化為四階段

**問題**：既有訂單狀態機線下路徑（archived `relocate-sales-manager-approval-from-quote-to-order`）為「待業務主管審核 → 報價待回簽 → 已回簽」一步到位，有兩個缺口：(1) 業務主管核可後到客戶實際收到報價單之間存在隱性「審完未送」階段，業務無法區分哪些單已審完但還沒送客戶；(2) 轉訂單後直接進「待業務主管審核」，業務沒有調整帶入內容的緩衝、且「草稿」起點只畫在流程圖未落正式條款（state-machines L188 線下路徑列舉漏列、US-ORD-001 仍寫進報價待回簽）。

**決策**：將線下前段細化為完整四階段 **草稿 → 待業務主管審核 → 審核通過 → 報價待回簽 → 已回簽**：
1. 轉訂單以「**草稿**」為初始狀態（取代既有 US-ORD-001 / L188 入口描述）；業務於草稿態調整帶入內容後點「**送主管審核**」推進至待業務主管審核（新增 `submitted_for_review_at`）
2. 業務主管核可推進至「**審核通過**」，業務送出報價單後手動點「已送報價單」推進至「報價待回簽」（新增 `quote_sent_at`）
3. **編輯邊界**：草稿 + 待業務主管審核兩態業務可改訂單全部內容（核准前為內部初版），進入「審核通過」起鎖定成交條件
4. **鎖定錨點前移**：`payment_terms_note` / `approved_by_sales_manager_id` 自「進入報價待回簽後鎖定」前移至「進入審核通過後鎖定」（核准當下即鎖），修補新增中間態後「核准後、外發前竄改收款條件繞過把關」的漏洞
5. **駁回沿用現況**：業務主管不核准維持「待業務主管審核」+ Slack 討論，不設退回鈕、無「退回草稿」動作（業務於待審核態可持續修改）；業務主管於草稿建立時即指派

**理由**：
- 明確區分草稿（業務可編輯）/ 待審核 / 審核完成（審完未送）/ 已送客戶四階段，避免業務漏送、給業務調整緩衝
- 鎖定前移確保 `lastApprovedPaymentTermsNote` 快照與實際鎖定時點一致，維持業務主管把關完整性
- 沿用既有 UI 與權限機制（業務主管核准動作不變，只是目標狀態調整）

**既有主 spec staleness 的修法（MODIFIED 整條取代，非延後人工校正）**：主 spec § 訂單狀態機 線下路徑列舉（L188「報價待回簽 → 已回簽」+ L234 過時備註）與 § 訂單建立 US-ORD-001 入口（「進入報價待回簽」）為 archived `relocate-sales-manager-approval-from-quote-to-order` 遺留的不一致。OpenSpec sync 以 exact-title 整條 Requirement 比對，ADDED 不會觸及既有 Requirement 內文的散落字句——若僅靠 ADDED + 文字宣告「取代」，archive 後主 spec 會新舊路徑並存矛盾（erp-consultant 審查點 4）。故改以 MODIFIED 整條納入 § 訂單狀態機 與 § 訂單建立（byte-faithful 重製 + 僅改目標行），讓 sync 自動取代、零人工依賴。代價是 delta 較長，但消除「上線後才發現主 spec 矛盾」的風險。

**Alternatives 拒絕的選項**：
- 「業務主管核准直接進報價待回簽」（既有）：無法區分審完未送 vs 已送，業務追蹤工作困難
- 「審核通過後系統自動寄送報價單」：未來功能，目前報價單透過外部管道送出，系統不參與寄送
- 「業務送審後可自撤回草稿」：與單向推進精神衝突；改以「待審核態仍可編輯」滿足業務改錯需求（Miles 2026-06-01 拍板）
- 「審核完畢 / 外發報價單 / 提交審核」另命名：沿用本 change 既有「審核通過 / 已送報價單 / 送主管審核」，零改名成本（Miles 2026-06-01 拍板）

### B. 退款流程三組件組合而非單一動作

**問題**：實務退款涉及三類動作：金額異動審批 / 金流動作 / 稅務動作。早期 user story 把三類混在一個「退款流程」中，造成 user story 內容互相重疊、難以驗證。

**決策**：拆乾淨三組件為三個獨立 user story，業務需依場景組合：
1. **訂單異動單 + 業務主管審核**（金額異動審批）
2. **退款款項處理**（金流：負值收款紀錄）
3. **開立折讓單 或 作廢發票**（稅務：依跨齊報稅期狀態擇一）

**理由**：
- 切乾淨後每 user story 可獨立寫測試案例
- 業務操作對應實際 ERP 不同入口（訂單異動 UI / 付款管理 UI / 發票異動 UI）
- 跨齊報稅期判斷在發票異動組件內處理，金流組件不受影響

**Alternatives 拒絕的選項**：
- 「一鍵退款」自動同時建立三組件：違反稅務規則「每動作獨立紀錄」要求，且難以稽核

### C. 發票作廢 vs 折讓：UI 兩按鈕 + 後端規則

**問題**：跨齊報稅期判斷規則對業務不直覺，但 ERP 系統不知道發票實際是否跨期（由第三方發票平台判斷）。

**決策**：
- **UI 維持作廢 / 折讓兩按鈕**：業務依直覺選擇（最常見動作為作廢）
- **後端規則由第三方平台判斷**：作廢失敗時顯示明確錯誤訊息「此發票已跨齊報稅期無法作廢，請改開折讓單」並引導改點折讓
- **業務不需了解術語**：依錯誤訊息指引操作即可

**理由**：
- 避免在 ERP 內維護一套「跨齊報稅期」判斷邏輯（風險：與第三方平台不一致）
- 業務認知負擔最小（先點作廢，失敗再轉折讓）
- 第三方平台為單一事實來源，規則變更不需改 ERP

**Alternatives 拒絕的選項**：
- 「ERP 先判斷再呼叫平台」：維護兩套判斷邏輯，平台規則變更時 ERP 跟不上
- 「只顯示一個按鈕（系統判斷後決定）」：業務無法依情境主動選擇（例如客戶要求不作廢只折讓）

### D. 付款計劃變更分階段稽核

**問題**：既有「付款計畫變更觸發訂單回業務主管審核」邏輯太重（製作中、出貨後客戶協商付款延期都要回審核）。但完全取消稽核又會失去軌跡。

**決策**：分階段邏輯：
- **業務主管審核通過前**（訂單為內部初版）：變更不觸發稽核、不回主管審核（業務內部修正正常）
- **業務主管審核通過後**（訂單為正式版）：變更不回主管審核，但自動記錄 `original_expected_date`（首次變更時） + `change_count + 1`

**理由**：
- 初版階段業務有正常迭代需求（規格 / 金額 / 付款條件可能多次調整），不應卡稽核
- 正式版後客戶協商付款延期是業務權責，不需主管每次核可，但需稽核軌跡
- `original_expected_date` 為首次變更前的值（不被後續變更覆寫），保留客戶最初承諾日期

**Alternatives 拒絕的選項**：
- 「所有變更都回主管審核」（既有）：阻擋業務日常彈性
- 「所有變更不留稽核」：失去責任歸屬軌跡

### E. 多品項發票稅額算法：每品項分別計算 + 差額集中最後品項

**問題**：「總額算稅額後分攤」會造成每品項稅額除不盡的小數誤差，違反中華民國稅務規則「每品項分別計算稅額」要求。

**決策**：
1. 每品項各自計算未稅金額 → 稅額（無條件進位）→ 含稅金額
2. 加總所有品項含稅金額 = 「品項加總」
3. 與「總額一次計算」比對，差額（通常 1 元）集中於最後一個品項
4. **禁止平均分攤**（破壞每品項稅額正確性）

**理由**：
- 符合中華民國稅務規則
- 多品項時通常品項加總比總額多 1 元，差額需具體分配
- 集中於最後品項是業界常見做法（業務 / 客戶易理解）

**參考**：[創嶼會計師事務所 - 發票含稅金額回推 1 塊錢尾差](https://www.tron-island.com/blog/fapiao-rounding-issue)

### F. 訂單客戶資料 relation（非 snapshot）

**問題**：客戶改名 / 統編變更時，既有訂單若用 snapshot 會保留舊資料，需逐張訂單手動修改。但已開發票依稅務規則需保留當時資料。

**決策**：
- `order.customer` 為 Customer entity relation（非 snapshot）
- 廠客資料異動自動同步至既有訂單顯示
- **例外**：已開發票上的買受人資訊 MUST 保留快照（稅務規則要求）

**理由**：
- 訂單顯示一致性（避免新舊資料混雜）
- 發票稅務合規（一旦開立其買受人為法定紀錄）

### G. 訂單複製功能 + source_order_id 追蹤

**問題**：老客戶「照上次一樣」reorder 場景，業務需要快速建單但不希望從頭輸入規格。

**決策**：
- 業務於訂單列表執行「複製訂單」 → 建立新訂單帶入客戶 / 印件規格 / 價格 / 付款條件
- 新訂單 `source_order_id` 指向原訂單
- 新訂單進入正常流程起點（草稿 → 待業務主管審核）

**理由**：
- 老客戶 reorder 是常見場景，業務希望「一鍵」操作
- `source_order_id` 追蹤 reorder 關係供日後分析（哪些客戶常 reorder）
- 新訂單走正常流程確保業務主管仍核可（不跳審核）

### H. 多印件分次出貨追蹤（既有設計補完）

**決策**：
- 印件層 `shipment_quantity` 累計欄位
- 印件出貨狀態三態：未出貨 / 部分出貨 / 已出貨
- 出貨單列表呈現印件級分次出貨進度

**理由**：
- 既有設計（印件狀態 + 出貨單 + 印件出貨數量）已涵蓋邏輯，本 change 明示視覺呈現規則
- 避免業務 / 客戶混淆訂單整體出貨狀態 vs 各印件出貨狀態

## Risks / Trade-offs

### 風險 1：「審核通過」新狀態影響既有 Prototype

**風險**：既有 Prototype 訂單狀態流轉為「待業務主管審核 → 報價待回簽」一步到位，加入「審核通過」中間狀態需修改 Prototype 訂單詳情頁、訂單列表頁、業務主管審核頁、ActivityLog。

**緩解**：
- spec 中明示「審核通過」狀態與動作對應的 UI 變更（業務主管核准按鈕目標、業務「已送報價單」按鈕）
- Prototype 補實作另開 task（不在本 change 範圍）
- 既有「業務主管核准訂單」Requirement MODIFIED 明示目標狀態變更，archive 時 main spec 自動 merge

### 風險 2：付款計劃變更分階段邏輯與既有 prototype 不一致

**風險**：既有 prototype 已實作「變更觸發回主管審核」邏輯（OrderPaymentSection.tsx:154-171）。本 change 取代為分階段稽核。

**緩解**：
- spec 明示分階段邏輯與既有邏輯的取代關係
- Prototype 補實作另開 task，包含拆除既有回審邏輯 + 加入稽核欄位邏輯
- 過渡期間：既有訂單在主管審核通過前 / 後的判斷以 spec 為準

### 風險 3：發票作廢 / 折讓 UI 業務操作偏差

**風險**：業務不熟稅務規則，可能習慣性點作廢直到失敗才轉折讓，效率較差。

**緩解**：
- 錯誤訊息明確引導改開折讓
- 後續可加 UI 提示（如「此發票已開立超過 X 天，建議改開折讓單」），但不在本 change 範圍
- 業務培訓階段提示「跨期發票直接點折讓」

### 取捨：spec 中保留多個 MODIFIED Requirement vs 拆多個 change

**取捨**：本 change 涵蓋 9 項補齊內容跨 4 個 spec。優點是一次性對齊本輪設計決議，缺點是 change 範圍較大、archive merge 複雜度增加。

**選擇**：採一個大 change 整合的方案，主因：
- 9 項補齊內容邏輯相關（訂單前段流程 / 退款流程都涉及訂單模組與發票模組）
- 業務 / 諮詢看 user story 已含這些內容的 user story 編號，spec 對齊一致性高
- 三視角審查可一次涵蓋所有變更
