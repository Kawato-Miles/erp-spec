## Context

退款款項建立後狀態為「處理中（待出金）」，公司月底由會計統一出金。現況無集中列管入口，靠業務主管 Excel 人工彙整，且退款收款帳號未於款項建立時結構化記錄。商業規則正本見 wiki [待出金退款清單組成](../../../memory/Sens_wiki/wiki/erp/04-business-logic/營運規則/帳務/待出金退款清單組成.md)。

既有基礎（不新建）：

- 收款紀錄（Payment）已有 `款項類型`（收款／退款）、`款項狀態`（處理中／已完成／已取消）、附件與「已完成」切換機制（complete-payment-status change 已歸檔）。
- finance 導覽已有「待開發票」「應收款項」兩個唯讀列管頁，[Receivables.tsx](../../../../sens-erp-prototype/src/pages/finance/Receivables.tsx) 為現行列表範式。
- [PaymentEditPanel.tsx](../../../../sens-erp-prototype/src/components/order/PaymentEditPanel.tsx) 已具備款項編輯（含已完成 + 附件上傳）。

## Goals / Non-Goals

### Goals

- 新增唯讀列管頁「待出金退款」，逐筆列出處理中退款款項，供會計執行出金。
- 退款款項建立時結構化記錄「退款收款帳號」，並顯示於清單與出金操作面板。
- 會計於清單就地（side panel）完成出金（切已完成 + 掛匯款證明），完成即移出清單。

### Non-Goals

- 防漏退提醒（「應收已調降但未建退款款項」）——屬另一 UX 議題，不在本 change。
- 系統排程批次出金——「月底」是會計人工集中作業時點，非定時 job（ORD-034／BI-8 已拍板）。
- 對帳明細／CSV——清單驅動「未發生的出金」，與對帳核對「已發生」目的不同，沿用既有對帳工具。
- order-billing spec 退款舊模型殘留文字清理、legacy finance 頁與廢棄 change 清理——另案。

## Decisions

### 決策 1：清單為衍生查詢視圖，不新增狀態 / 實體

清單＝過濾 `款項狀態 = 處理中 且 款項類型 = 退款` 的收款紀錄逐筆呈現。出金完成即沿用既有「處理中 → 已完成」切換，款項自然脫離過濾條件而移出清單。不引入新狀態、新實體或新批次單據。

替代方案（否決）：建「待出金批次單」實體統管一輪出金——增加單據與狀態維護成本，與「會計隨時手動產出、逐筆切已完成」的實務不符。

### 決策 2：退款收款帳號為 Payment 單一自由文字欄位

於 `types/payment.ts` Payment 新增一欄位（戶名／銀行／帳號合一的自由文字），退款款項建立時填。欄位正本在 wiki [帳務](../../../memory/Sens_wiki/wiki/erp/05-entities/帳務.md) § 收款紀錄，spec / 型別不複寫欄位表。

替代方案（否決）：拆「戶名／銀行／帳號」三欄或建客戶收款帳號主檔——退款帳號每筆當次由客戶提供、可能與付款帳號不同，且非高頻結構化查詢需求，單一自由文字欄足夠（YAGNI）。

### 決策 3：出金操作於清單頁 side panel 重用 PaymentEditPanel

會計點清單列「檢視」開 side panel，內嵌既有款項編輯能力（切已完成 + 上傳匯款證明）。不跳轉訂單詳情頁、不在清單列 inline 複製完成邏輯。

替代方案（否決）：(a) 跳訂單詳情頁——會計批次出金時頻繁跳頁、脫離清單脈絡；(b) 清單列 inline 完成——需把已完成 + 附件上傳邏輯複製一份到清單，重複維護。side panel 兼顧不脫離清單與重用既有機制。

### 決策 4：角色可見範圍維持粗粒度

待出金退款清單對「會計」（全公司，執行出金）與「業務主管」（彙整）可見；會計可執行出金。不做「個人／部門／全公司」三層細粒度矩陣。

替代方案（否決）：比照待收款／待開發票的三層 view 矩陣——出金是會計集中作業 + 業務主管彙整，無個別業務逐筆操作需求，細粒度矩陣為過度設計。

### 決策 5：出金切已完成與退款核銷分配解耦（Miles 2026-06-14 拍板）

會計於待出金清單切「已完成」不以退款核銷分配（PaymentAllocation 至期次）為前置；唯一硬前置為匯款證明附件（沿用既有完成機制）。核銷分配是業務獨立於訂單詳情頁完成的動作，帳務一致性靠事後對帳。

替代方案（否決）：設閘門要求核銷分配完成才可切已完成——會增加會計出金的前置依賴、把業務的核銷責任綁進會計的出金動作，違反「彈性 + 事後對帳」原則（[[feedback_invoice_reconciliation_flexibility]]）。短暫的「期次已退金額未累計」由既有對帳差額警示抓出。

## Risks / Trade-offs

- PaymentEditPanel 將於兩處渲染（訂單詳情頁 + 清單 side panel）→ 確保完成機制單一來源、行為一致；side panel 切已完成後清單需即時移出（依賴 store reactivity，apply 時驗證）。
- 退款收款帳號為選填 → 既有處理中退款款項可能無帳號；清單顯示「未填」不阻擋顯示，由會計向業務補。

## Migration Plan

- 退款收款帳號為新選填欄位，既有 mock 資料不需回填。
- 新增 finance 路由與導覽入口，不影響既有頁面。
- 回滾：移除新頁路由 / 導覽項與欄位即可，不影響既有 Payment 流程。
