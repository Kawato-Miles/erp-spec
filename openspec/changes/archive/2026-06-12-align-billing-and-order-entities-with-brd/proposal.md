## Why

### Background

在印刷業 ERP 訂單與帳務處理的商業模型中，訂單的款項計畫、收款登記與發票開立需嚴格服從業務實務。近期 Wiki 進行了大量商業需求（BRD）的重構：
- [訂單實體卡](../../../../memory/Sens_wiki/wiki/erp/05-entities/訂單.md) 統一了帳務公司與接單公司，並明確了複製來源等追溯欄位。
- [帳務實體卡](../../../../memory/Sens_wiki/wiki/erp/05-entities/帳務.md) 對請款期次與收款紀錄的屬性進行了調整。
- [對帳一致性](../../../../memory/Sens_wiki/wiki/erp/04-business-logic/營運規則/帳務/對帳一致性.md) 與 [訂單異動規則](../../../../memory/Sens_wiki/wiki/erp/04-business-logic/營運規則/訂單到交付/訂單異動規則.md) 重新定義了已核可退款之認列與實際退金流（款項紀錄）脫鉤的業務關係。
- 此外，開放問題 [ORD-027-訂單付款狀態是否新增部分付款態](../../../../memory/Sens_wiki/wiki/erp/08-open-questions/ORD-027-訂單付款狀態是否新增部分付款態.md) 亦暴露了訂單與請款期次付款狀態定義重疊與狀態機推導逻辑的不一致。

### Problem Statement

目前 OpenSpec 系統行為規格 (PRD) 與 Prototype 實作程式碼，在訂單與帳務實體欄位上與上述最新拍板的 BRD 存在多處矛盾，導致系統與實務脫節：
1. **付款狀態推導邏輯不符**：訂單的付款狀態在業務上應由下屬請款期次的收款狀態「由下而上」進行齊套（取最落後者）推導，但現行 PRD 卻使用金額百分比推導。且代碼中將訂單付款狀態宣告為已棄用，缺失了「已退款」狀態。
2. **退款金流表示不合會計直覺**：舊有系統設計要求在收款紀錄中以負數輸入退款金額，且收款/付款方式中包含「退款」此一金流媒介，實務上應以獨立的「款項類型 = 收款/退款」表示，且金額一律為正，款項/付款方式中不應存在名為「退款」的金流。
3. **核銷分配自動化之錯置**：舊規格與 BRD 誤描述了有系統自動核銷分配邏輯，但實務上為全手動分配，且請款期次實體中缺少了收款金額與退款金額的核心彙整欄位，導致期次收款狀態無法動態推導。
4. **追溯與紅利等商業欄位缺失**：訂單的「複製來源」、「主訂單」、「是否為補收款」與「可獲得紅利點數」已在 BRD 定義，但 PRD 與實作完全缺失。
5. **PRD 內留有大量技術與商業對照**：PRD 中保留了大量的英文欄位對照，違反「PRD 內不留任何商業欄位資料，且系統外鍵描述不可使用英文」之 SSD 規格約束。

## What Changes

### 1. 訂單實體（Order Entity）修訂
- **PRD 重構**：
  - 於 `order-billing/spec.md` 中重構付款狀態推導，改為「請款期次齊套推導」中文規則。刪除對 `Order.paymentStatus` 的 `@deprecated` 標記。在代碼與 PRD 狀態列舉中補上「已退款」。
  - 移除 PRD 中所有 `accountCompany` 與 `billingCompanyId` 之英文對照，僅以中文描述「帳務公司」的手動選擇關係。
  - 補齊「複製來源」、「主訂單」、「是否為補收款」與「可獲得紅利點數」之行為規格，外鍵關聯以中文描述為：「以原始訂單編號作為 FK」、「以主訂單編號作為 FK」。

### 2. 請款期次實體（Billing Installment Entity）修訂
- **BRD 修正**：
  - 新增 `預計金額（未稅）`、`已收金額` 與 `已退金額` 三個中文欄位。
  - 保留 `變更次數`、`原始預計收款日`、`原始預計開票日` 三個稽核欄位。
- **PRD 重構**：
  - 徹底移除 PRD 中所有金額英文欄位表，僅使用中文描述「預計金額（含稅）」與「預計金額（未稅）」。
  - 定義期次的 `已收金額` 與 `已退金額` 的動態彙整手動核銷邏輯，用以推導期次收款狀態（未收/部分收款/已收訖）。
  - 清理 PlannedInvoice 與 PaymentPlan 舊術語，期次與訂單關聯改寫為：「以訂單編號作為 FK」。

### 3. 收款紀錄實體（Payment Entity）修訂
- **BRD 修正**：
  - 新增 `款項類型` 欄位（單選：收款 / 退款）。
  - 將原 `收款方式` 改名為 `款項方式`，單選範圍修正為 `現金 / 信用卡 / 銀行轉帳 / 支票 / 其他`（徹底刪除「退款」金流）。
  - 將 `收款狀態` 改名為 `款項狀態`，可用範圍調整為 `處理中 / 已完成 / 已取消`。
  - 新增 `取消備註` (文字) 欄位，供記錄取消原因。
  - 修正 `金額` 欄位說明，退款一律記錄正數。
  - 移除 `關聯訂單異動` 欄位。
  - 不收錄取消時間等技術戳記。
- **PRD 重構**：
  - 移除 Legacy 的 `PaymentType` (第一期/尾款) 英文對照，重構為「收款」與「退款」中文規格。
  - 重構 `paymentMethod` 描述為「款項方式」，並排除退款金流。
  - 移除邏輯刪除布林 `是否已取消`，改由款項狀態 `'已取消'` 來標記金流失效事實。
  - 補充「退款（款項類型 = 退款）」手動核銷至期次之連動計算規格。

### 4. 收款核銷分配實體（PaymentAllocation Entity）修訂
- **BRD 修正**：
  - 刪除 `是否自動分配` 與 `是否手動覆寫` 二技術欄位。
- **PRD 重構**：
  - 刪除所有「系統自動依序填滿」之規格，改為「全手動核銷分配」規格。
  - 外鍵關係描述完全移除英文，改寫為：「以請款期次編號作為 FK，且以收款紀錄編號作為 FK」。

### 5. 發票與折讓實體（Invoice & SalesAllowance Entity）修訂
- **BRD 修正**：
  - 發票表中刪除 `藍新商店代號` 技術欄位。
  - 折讓單中將 `藍新折讓號` 重命名為 `折讓單號` 並予保留（供對帳使用）。
- **PRD 重構**：
  - 清空發票與折讓單的英文變數與 API 欄位對照。
  - 外鍵關係描述完全移除英文，改寫為：「以請款期次編號作為 FK」、「以發票號碼作為 FK」、「以折讓單編號作為 FK」。

## Capabilities

### New Capabilities

無。

### Modified Capabilities

- **order-management**：訂單付款狀態齊套推導與複製追溯、主從訂單連動。
- **order-billing**：全手動請款核銷、收款與退款雙金流方向支援、請款期次已收與已退金額累計。

## Impact

### 1. 訂單實體修改之商業問題與風險評估
- **若不補齊「複製來源/主訂單/是否補收款」之風險**：系統中將缺乏主從訂單的關聯鏈，當印務或出貨部門在追查補印與補收款時，無法與原訂單交叉核對，容易造成重複發貨或漏收差額的實質資金損失。
- **若未補齊付款狀態「已退款」之問題**：列表篩選器將無法過濾出已完成退款的訂單。業務與會計必須逐單開啟確認，極易產生漏退款或重覆退款的財務合規風險。

### 2. 請款期次實體修改之商業問題與風險評估
- **若不新增「預計金額（未稅）」之問題**：B2B 開立發票時，品項單價與稅額無法在期次規劃階段預先對齊，容易導致開立發票時產生四捨五入的尾差，引起客戶對帳疑慮或稅務申報不合規的法律風險。

### 3. 收款紀錄實體修改之商業問題與風險評估
- **「退款改為正值 + 款項類型判定」造成的統計錯誤風險**：移除負值金額設計後，系統內所有涉及金額合計、對帳面板計算的公式，必須從原本直接 `sum(金額)` 改為依據 `款項類型` 進行條件運算（如：收款為加、退款為減）。若重構時遺漏了部分財務報表或 CSV 導出邏輯，將會把退款金額誤算為收款，導致對帳淨額嚴重虛胖的財務風險。在歷史資料遷移（Migration）時，必須撰寫腳本將舊有的負值金額轉換為正值，並將 `款項類型` 補填為 `'退款'`，以防歷史帳目混亂。
- **移除「關聯訂單異動」造成的資金安全風險**：移除此關聯後，系統無法在技術上限制作作廢/退款必須有已核可的異動單為依據，業務將可以直接手動建立已完成的退款款項。這產生了業務可能繞過「業務主管核可」而私自退款、虛報支出的財務漏洞，需在會計操作流程上增加二次審查把關。
- **移除「關聯訂單異動」造成的對帳核銷追查困難**：多筆退款與多筆退款異動單同時存在時，會計在月結時無法進行單筆反查。一旦對帳出現總額不符，會計必須人工逐筆核對銀行水單與異動原因，大幅增加人工查帳的作業成本。
- **重構「款項類型」造成的舊資料損壞風險**：直接修正 `type` 欄位會造成舊系統中已將收款分類為「第一期」或「尾款」的歷史數據損壞，導致追查舊帳時無法辨識舊收款的期次歸屬。必須在代碼變更前撰寫資料搬移腳本，將舊有 `type` 資料完全對應並轉換寫入到新的請款期次核銷分配中。

### 4. 收款核銷分配實體修改之商業問題與風險評估
- **改為「全手動核銷分配」帶來的會計操作負擔與輸入錯誤風險**：取消自動分配後，會計每一次登記收款都必須手動點選並逐筆分配金額至各期次。在面對大量請款單時，這會顯著增加人工作業時間，且一旦會計漏勾選或手動輸入金額錯誤，將直接導致期次的收款狀態與實際對帳不符，增加財務核帳的失誤風險。

### 5. 發票與折讓實體修改之商業問題與風險評估
- **若未處理好發票流水號遞增之開票碰撞風險**：若流水號沒有在技術上強制作廢重開遞增，與藍新 API 對接時會發生號碼碰撞。這將直接導致發票開立失敗，使出貨程序卡住，面臨客戶因延遲交貨而索賠或流失訂單的風險。
