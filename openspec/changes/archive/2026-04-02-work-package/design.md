## Context

生管的派工流程已有 prototype 驗證（task-dispatch-prototype change），驗證了「日期軸 + 工序分組 + 批次派工 + 報工」的基本模式。但目前批次派工是一次性動作，派出後各生產任務散開獨立追蹤，缺乏「這批是一起派的」持久化記錄。

工作包（Work Package）將「一次派工」升級為顯式實體，成為生管派工與師傅接單的核心操作單位。

相關 specs 狀態：
- 生產任務 spec（v0.2）：已定義 assigned_operator 欄位、派工流程
- 狀態機 spec：指派師傅 MUST NOT 觸發狀態變更，狀態「製作中」僅由首次報工觸發
- 任務分派介面 spec（task-dispatch-board）：已定義工序分組、批次派工、報工等功能

## Goals / Non-Goals

**Goals:**
- 定義 WorkPackage Data Model，作為無狀態的派工分組容器
- 將現有「批次派工」升級為「建立工作包」，一步完成
- 進行中區從「依師傅分組」改為「依工作包卡片」，提供展開/收合
- 生產任務列表全面改為工作包視角，提供工作包 Key 搜尋
- 支援工作包操作：移出任務、整包移轉師傅、刪除工作包
- 支援以工作包為單位列印工作單（細節待後續定義）

**Non-Goals:**
- 不定義工作單列印的詳細欄位與版面（本次只確認「可印」）
- 不實作後端 API 或資料庫（純前端 prototype）
- 不改動排程中心（印務層仍以單筆生產任務排入設備佇列）
- 不改動報工流程（報工仍以單筆生產任務為單位）
- 工作包不參與狀態機（無狀態、不觸發向上傳遞）

## Decisions

### D1: WorkPackage 為無狀態分組容器

**決策**：WorkPackage 不設狀態欄位，不參與狀態機。它只是一組生產任務的命名容器，綁定一位師傅。工作包的「進度」由包內各生產任務的狀態動態計算（如 2/5 製作中），不儲存於工作包本身。

**替代方案**：給工作包狀態（未開工 / 進行中 / 完成） -> 增加狀態同步複雜度，且與生產任務狀態機產生冗餘

**理由**：工作包的價值是分組與可搜尋性，不是流程控制。進度從包內任務即時推導即可，避免引入額外狀態同步邏輯。

### D2: Key 格式 WP-{工序}-{MMDD}-{流水號}

**決策**：Key 由系統自動產生，格式為 `WP-{工序名稱}-{建立日期MMDD}-{當日該工序流水號三碼}`。例：`WP-印刷-0402-001`。

**替代方案**：
- 純流水號（WP-20260402-001） -> 無法從 Key 識別工序
- 使用者自訂名稱 -> 命名不一致，搜尋困難

**理由**：Key 需同時服務搜尋（快速定位）與列印（工作單上可讀）。工序 + 日期 + 流水號的組合讓生管看到 Key 就知道「哪天的什麼工序」，搜尋時也可用工序名稱篩選。

### D3: 建立工作包 = 升級版批次派工（一步完成）

**決策**：取代現有的「派工確認對話框」，改為「建立工作包對話框」。生管勾選待處理任務 -> 點擊「建立工作包」 -> 對話框中選擇師傅、填寫備註、為各任務填寫確樣需求 -> 確認。系統一次完成：建立 WorkPackage、設定各任務的 work_package_id + assigned_operator + sample_notes + actual_start_date。

**替代方案**：先建工作包再指派師傅（兩步分離） -> 增加操作步驟，不符合生管「派完就走」的工作節奏

**理由**：工作包本質是「持久化的批次派工」，不應增加操作成本。一步完成與現有派工體驗一致，只是多了備註和確樣需求欄位。

### D4: 進行中區改為工作包卡片視角

**決策**：進行中區移除原本的「依師傅分組」層，改為平鋪顯示工作包卡片。每張卡片顯示：Key、師傅名稱、備註、進度摘要（如 2/5 製作中）。卡片可展開查看包內生產任務明細表格。

**替代方案**：保留師傅分組 -> 工作包為三層（師傅 -> 工作包 -> 任務），UI 層級過深

**理由**：工作包已綁定師傅，卡片上直接顯示師傅名稱即可。生管搜尋時以工作包 Key 為主，不需要先找師傅再找包。減少一層 UI 巢狀，操作更直覺。

### D5: 生產任務列表全面以工作包為單位

**決策**：所有已派工的生產任務在列表中以工作包為單位呈現（可展開）。未派工的生產任務不會出現在工作包列表中（它們在待分派區）。提供獨立搜尋欄位，支援以工作包 Key、師傅名稱進行搜尋。

**理由**：Miles 確認「日後不會有沒有工作包的生產任務」，因此所有已派工任務必然屬於某工作包，可以全面採用工作包視角。

### D6: 工作包操作 — 移出 / 移轉 / 刪除

**決策**：
- **移出任務**：將生產任務從工作包移出，清除 work_package_id + assigned_operator + sample_notes + actual_start_date，任務退回待分派區
- **整包移轉師傅**：更新 WorkPackage.operator_id 與包內所有 ProductionTask.assigned_operator
- **刪除工作包**：刪除 WorkPackage 記錄，包內所有任務按「移出」邏輯處理（退回待分派）

**約束**：已進入「製作中」狀態（已有報工）的生產任務 MUST NOT 允許移出或因工作包刪除而退回待分派。此類任務須先處理（完成或作廢）後才能刪除工作包。

**理由**：已開工的任務有報工紀錄，退回待分派會造成資料不一致。工作包刪除應為安全操作，不影響已開工的生產進度。

### D7: 工作單列印（範圍限定）

**決策**：提供「列印工作單」按鈕，以工作包為單位輸出可列印內容。本次 change 只確認功能存在與入口位置（工作包卡片上的操作按鈕），不定義工作單的詳細欄位與版面設計。

**理由**：列印格式需與現場實際需求對齊，後續獨立 change 處理。

## Risks / Trade-offs

- **工序 Key 中文命名**：Key 含中文工序名稱（如 WP-印刷-0402-001），若工序名稱過長可能影響可讀性。Mitigation：工序名稱使用簡稱（由工序定義表控制）。
- **已開工任務的操作限制**：刪除工作包時若包內有「製作中」任務會被阻擋，生管可能不理解為什麼不能刪。Mitigation：刪除前顯示明確提示，列出阻擋原因與建議處理方式。
- **與排程中心的銜接**：排程中心仍以單筆生產任務操作，工作包在排程中心不可見。若印務調整排程影響到已建工作包內的任務，需透過異動確認機制處理。此為現有機制，本次不新增。
