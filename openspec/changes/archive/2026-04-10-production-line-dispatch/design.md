## Context

目前生產任務的 production_line_id 為選填，由印務在工單詳情頁手動指定。多位生管各自負責不同產線，日程面板雖已有產線篩選器，但因產線覆蓋率不完整（選填 = 可能為空），篩選效果不佳。

Miles 確認：BOM 管理已實作，材料 / 工序 / 裝訂的維護介面已在線上運作。BOM 行項目即決定產線歸屬，生產任務建立時直接帶入，不允許下游修改。

相關 changes：
- production-flow-optimization（完成）：建立 ProductionLine 資料表與 production_line_id 欄位
- scheduling-center（完成）：排程中心設計，產線為唯讀顯示
- task-dispatch-prototype（完成）：日程面板 prototype，含產線篩選器

角色分工（參照 user-roles spec）：
- 印務 / 印務主管：BOM 維護時指定產線，工單建立時帶入
- 生管：日程面板依產線篩選任務、分派師傅、報工
- 業務：不涉及產線操作

## Goals / Non-Goals

**Goals：**
- BOM 行項目新增 production_line_id，確保源頭即指定產線
- 生產任務建立時從 BOM 自動帶入 production_line_id，帶入後唯讀
- production_line_id 從選填改為必填，保證 100% 覆蓋率
- 外包廠 / 中國廠商的任務統一歸入「外包產線」
- 日程面板篩選器支援偏好記憶（記住上次選擇）

**Non-Goals：**
- 生管與產線的系統綁定（不加 manager_id，不做權限隔離）
- 產線產能計算（未來需求）
- BOM 管理介面改造（僅新增欄位，不重新設計介面）
- 行動裝置版（ERP 僅桌機版）

## Decisions

### Decision 1：產線由 BOM 行項目決定，非設備或工序層

**選擇：BOM 行項目層指定 production_line_id**

BOM 維護時，印務 / 印務主管在每個行項目（材料 / 工序 / 裝訂）指定產線。同一道工序在不同 BOM 可能指向不同產線。

替代方案 A：Process -> ProductionLine（工序層 1:1）-- 過度限制，同一工序可能分屬不同產線。
替代方案 B：Equipment -> ProductionLine（設備層 N:1）-- 排程後才確定，太晚。
選擇 BOM 行項目層的理由：印務在 BOM 維護時已具備產線判斷能力，且此時的決策最穩定（不會因排程調度而改變）。

### Decision 2：production_line_id 帶入後唯讀

**選擇：系統帶入後不可修改**

生產任務的 production_line_id 由 BOM 帶入後，印務和生管均不可修改。理由：
- 產線歸屬是 BOM 級的設計決策，不應被下游推翻
- 若允許修改，會導致日程面板篩選結果不穩定（生管分派後任務跑到別人的產線）
- 需要改產線時，應修改 BOM 後重新建立生產任務

替代方案：允許印務在排程時調整 -- 增加混亂風險，且破壞產線覆蓋率保證。

### Decision 3：外包廠與中國廠商分別建立產線

**選擇：建立「外包廠」與「中國廠商」兩條產線，依 factory_type 自動帶入**

理由：
- 外包廠與中國廠商的管理流程不同（中國廠商多「已送集運商」中間態）
- 分開後篩選更精確，避免兩類任務混雜
- 未來外包量增加時，可進一步依供應商拆分，擴展性更好

替代方案 A：統一一條「外包」產線 -- 混雜不同管理流程的任務，篩選粒度不足。
替代方案 B：外包不指定產線（production_line_id = null） -- 破壞 100% 覆蓋率。

### Decision 4：篩選器偏好以 localStorage 實作

**選擇：瀏覽器 localStorage 記住生管上次選擇的產線**

理由：
- 不需動資料模型（不加使用者偏好資料表）
- 實作成本最低，行為直觀
- 換電腦 / 清快取時重新選擇一次即可，影響極小

替代方案：使用者偏好存後端 -- 過度工程化，生管通常固定使用同一台電腦。

### Decision 5：既有資料回填策略

**選擇：上線前一次性回填，依工單對應的 BOM 補齊 production_line_id**

步驟：
1. 掃描所有 production_line_id 為空的 ProductionTask
2. 根據其 task_id -> Work Order -> BOM 行項目，取得對應 production_line_id
3. 無法自動判斷的（如舊資料無 BOM 對應），標記為「外包」或由印務手動補齊

替代方案：不回填，舊資料允許 null -- 篩選器會遺漏舊任務，不可接受。

## Risks / Trade-offs

**[BOM 行項目必填產線] -> 增加 BOM 維護成本**
每次建立 / 修改 BOM 行項目需多選一個欄位。
緩解：大部分 BOM 的工序產線固定，印務只需設定一次。可考慮「產線預設值」（依工序自動帶入）降低操作成本。

**[產線唯讀不可改] -> 排程調度彈性降低**
若排程時發現應調整產線（如某產線滿載），無法在生產任務層修改。
緩解：印務應在 BOM 層調整後重新建立生產任務。實務中產線調度頻率極低。

**[localStorage 偏好] -> 換機 / 清快取遺失**
生管更換電腦或清除瀏覽器資料後需重新選擇。
緩解：操作成本極低（選一次即可），無需為此建立後端儲存。

**[外包拆兩條產線] -> 未來供應商更多時可能仍不夠細**
目前拆為「外包廠」與「中國廠商」兩條，若未來有東南亞等新供應商類型，需再擴充。
緩解：ProductionLine 為獨立資料表，新增產線成本極低。
