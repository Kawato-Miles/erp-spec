## Context

生管（生產管理）是工廠平台的核心角色，負責接收印務交付的生產任務、分派給師傅、追蹤進度與代替報工。目前系統已有工單 prototype（含生產任務表、QC 紀錄），但缺少生管視角的「每日派工總覽」介面。

現況痛點：
- 生管需逐一翻看各工單才能知道今天有哪些任務要處理
- 同工序的任務分散在不同工單中，無法一次看到全貌
- 師傅頻繁切換工序導致設備調校浪費

相關 specs 狀態：
- 生產任務 spec（v0.2）已定義任務分派介面、設備選擇、報工紀錄等功能
- 狀態機 spec 已定義生產任務狀態路徑（待處理 → 製作中 → 已完成）
- 使用者角色 spec 已定義生管權責（接收、分派、報工、異動確認）
- [XM-003](https://www.notion.so/32c3886511fa815085a8e5699718cef5) 工序合併規則尚未定案

## Goals / Non-Goals

**Goals:**
- 建立生管每日任務分派介面 prototype，驗證資訊架構是否支援高效派工決策
- 驗證「依工序分組」的顯示模式是否符合生管心智模型
- 驗證批次派工操作流程（選取 → 確認 → 派工）的可用性
- 驗證報工紀錄介面的欄位設計與操作流程
- 提供足夠的 mock data 涵蓋多工單、多工序、多工廠的真實情境

**Non-Goals:**
- 不實作後端 API 或資料庫（純前端 prototype）
- 不實作設備負載甘特圖（已有 EquipmentGantt 元件，本次不重複）
- 不實作 QC 流程（屬 QC 模組範疇，待 PT-002 定案）
- 不實作合併規則邏輯（XM-003 未定案，本次僅做分組顯示）
- 不實作外包廠 / 中國廠的運送狀態路徑（本次聚焦自有工廠派工）

## Decisions

### D1: 頁面資訊架構 — 日期軸 + 工序分組

**決策**：任務分派介面以「選定日期」為主軸，頁面頂部有日期選擇器（預設今日），主內容區依工序分組顯示該日期所有待處理 + 製作中的生產任務。

**替代方案**：
- A. 依工單分組（現有工單詳情頁已有） → 無法跨工單看同工序任務，不符合生管需求
- B. 看板模式（依狀態分欄） → 生管關注「做什麼」多於「什麼狀態」，看板不適合派工決策
- C. 甘特圖模式 → 過於複雜，且已有 EquipmentGantt 元件

**理由**：生管每天早上需要決定「今天師傅要做哪些事」，以日期 + 工序為軸最符合決策順序。同工序放在一起，生管可一次派工給同一位師傅。

### D2: 工序分組卡片設計

**決策**：每個工序為一張卡片（Card），卡片標題顯示工序名稱 + 任務數量 + 合計目標數量。卡片內以表格列出該工序下的所有生產任務，顯示：所屬工單號、印件名稱、目標數量、工廠、設備、狀態。

**理由**：表格適合同時比較多筆任務屬性，生管需要看到工單來源以判斷優先級（交期關聯）。

### D3: 批次派工操作

**決策**：每張工序卡片內支援勾選多筆「待處理」任務 → 點擊「派工」按鈕 → 彈出確認對話框（顯示已選任務摘要、指定師傅） → 確認後任務狀態從「待處理」轉為「製作中」。

**替代方案**：逐筆點擊派工 → 操作效率太低，每日 20-30 筆任務不適合

**理由**：批次操作符合生管實際工作模式（同工序一起派給同一位師傅），減少重複操作。

### D4: 報工入口設計

**決策**：在任務分派介面的「製作中」任務行上提供「報工」按鈕，點擊後彈出報工對話框。報工欄位依生產任務 spec：報工數量（必填）、報工工時（選填，分鐘）、缺陷數量（選填）、備註（選填）。

**理由**：報工是生管在任務分派介面上最常執行的第二個操作（第一是派工），放在同一介面避免跳轉。

### D5: 複用現有元件與型別

**決策**：
- 複用 `ProductionTaskStatusBadge`（狀態徽章）
- 複用 `src/types/workOrder.ts` 的 `ProductionTask`、`Task` 等型別定義
- 複用 `src/data/processContentCatalog.ts` 的工序定義
- 新增 `src/data/mockDispatch.ts` 專門的派工情境 mock data

**理由**：避免重複定義，確保型別一致性。

### D6: 側邊欄定位 — 工廠平台入口

**決策**：在側邊欄新增「任務派工」項目，位於工單管理之後。此為生管角色的主要工作入口。

**理由**：生管日常操作以任務分派介面為主，需要獨立的導覽入口。

## Risks / Trade-offs

- **XM-003 未定案** → 本次只做「依工序分組顯示」，不實作合併邏輯。若後續合併規則需要額外維度（紙張規格、顏色等），卡片設計可能需要調整。Mitigation：卡片設計預留擴展空間，分組欄位可配置。
- **Mock data 覆蓋度** → 真實情境可能有 20+ 工序、50+ 任務同日並存，mock data 難以完全模擬。Mitigation：提供至少 3 種工序、10+ 筆任務的情境，覆蓋基本驗證需求。
- **角色切換** → 現有 prototype 以印務視角為主，新增生管視角可能造成導覽混亂。Mitigation：側邊欄明確標示角色區隔，後續可依角色過濾選單。
