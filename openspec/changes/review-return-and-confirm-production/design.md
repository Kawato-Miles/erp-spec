## Context

印件審稿維度狀態機是印件從稿件到生產的核心控制點。現行設計「合格」為終態、合格後立即觸發工單建立。本次變更在合格與工單建立之間插入「業務確認」閘門，並開放「退回重審」路徑。

Prototype 技術棧：React + TypeScript + Tailwind + shadcn/ui。狀態機邏輯集中在 store 與 utils，UI 在各模組 panel / page。

核心約束：
- 印件狀態 enum 新增 2 值（已確認可製作、待改稿），影響 store 中所有使用 `reviewDimensionStatus` 的判斷
- 訂單 Bubble-up 派生規則從 4 條改為 5 條
- B2C / B2B 分流邏輯需在自動確認處分岔

## Goals / Non-Goals

### Goals

- 印件審稿維度新增「已確認可製作」「待改稿」兩狀態 + 3 條轉換規則
- 工單建立觸發從「合格」後移至「已確認可製作」
- 訂單 Bubble-up 派生規則更新
- 訂單詳情頁印件卡片新增「確認可製作」「退回重審」操作
- 審稿待辦清單區分一般件與重審件
- 訂單列表警示標記（待改稿 / 待確認印件數）

### Non-Goals

- 不修改印製維度狀態機
- 不廢除棄用 + clone 路徑
- 不新增 B2C 業務手動確認流程（B2C 一律自動跳過）
- 不新增退回重審輪次上限（暫不設限）
- 不修改打樣流程（打樣 NG-稿件問題仍走棄用 + clone）

## Decisions

### D1：「已確認可製作」與「待改稿」放在審稿維度內

**選擇**：在既有審稿維度 enum 中新增 2 個狀態值

**替代方案**：
- 新增獨立「確認維度」— 過度設計，跨維度重置邏輯複雜
- 復用「不合格」狀態加欄位區分 — 語意模糊，統計篩選複雜

**理由**：審稿維度已有 5 個狀態，新增 2 個後仍為 7 個，管理成本低。「已確認可製作」是品質確認後的業務決策，「待改稿」是退回等待改稿，兩者語意清晰。

### D2：退回重審預設原審稿人員，不重新自動分配

**選擇**：退回重審後預設回原審稿人員

**替代方案**：重新走自動分配 — 原審稿人員對同一印件最熟悉，重新分配降低重審效率

**理由**：與既有「補件回原審稿人員」邏輯一致（prepress-review spec § 補件回原審稿人員）。原審稿人員不在崗時走既有審稿主管覆寫機制。

### D3：退回原因存在印件 ActivityLog，不存 ReviewRound

**選擇**：退回原因記錄在印件 ActivityLog 的「退回重審」事件中

**替代方案**：在 ReviewRound 新增 `return_reason` 欄位 — 退回重審不產生新 Round（Round 在客戶上傳新稿時才建），退回原因與 Round 生命週期不對齊

**理由**：ActivityLog 是事件流紀錄的正確位置。審稿人員查看退回原因時從 ActivityLog 讀取最近一筆「退回重審」事件。

### D4：訂單製作段不退回審稿段

**選擇**：訂單一旦進入製作段，不因單一印件退回而退回審稿段，搭配警示標記

**替代方案**：訂單退回審稿段 — 多印件訂單中大部分印件已在生產，退回會造成狀態失真

**理由**：線下單印件數通常 3-10 件以上，退回 1 件就退回整張訂單會嚴重失真。警示標記讓業務知道有事要處理，不影響其他印件生產。

## Risks / Trade-offs

- **B2B 印件合格後業務忘記確認** → 訂單會卡在審稿段。緩解：訂單列表已有審稿段子狀態顯示，業務可識別。未來可考慮待辦提醒，但本次不做
- **退回重審無輪次上限** → 理論上可無限退回。緩解：實務上客戶改稿次數有限，暫不設限。若未來需要，可新增 OQ 討論上限機制
- **免審印件退回後走正常審稿** → 免審印件原本不需審稿，退回後卻走正常審稿可能讓審稿人員困惑。緩解：實務上免審印件被退回是極端情境，審稿流程本身不變，只是退回原因會解釋為什麼退回

## Migration Plan

Prototype 層變更（無生產環境部署）：
1. 更新印件狀態 enum（新增 `confirmed_for_production`、`pending_revision`）
2. 更新訂單 Bubble-up 派生邏輯（5 條規則）
3. 訂單詳情頁印件卡片新增操作按鈕
4. 審稿待辦清單新增「重審」標示
5. 訂單列表新增警示標記
6. 更新 e2e 測試

回滾策略：git revert 即可（Prototype 層無生產資料庫遷移）。
