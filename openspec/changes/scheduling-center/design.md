## Context

印務每日排程 20-30 筆生產任務，目前在各工單詳情頁逐筆設定開工日期與設備，缺乏跨工單全局視圖。多個印務無法看到設備衝突，業務無法即時回答客戶部分交貨時間。

現有相關功能：
- SchedulePlanner（工單內排程回推 prototype）：階段管理 + 自動回推，已驗證 UX 可行性
- SchedulePanel（生管日程面板 prototype）：生管分派 + 報工，為排程中心的下游
- scheduling-completion-date change：定義 estimated_duration_days、planned_end_date 欄位與完工推算規則

本次設計排程中心作為印務的主要排程工作介面，整合上述概念。

角色分工（參照 user-roles spec）：
- 印務：排程中心主要使用者，操作設備佇列排序、設定工期、追蹤外包
- 印務主管：同印務權限，另可查看全局統計
- 生管：排程中心唯讀，主要在生管日程面板操作分派
- 業務：透過訂單/印件頁面查看預估完成日，可使用部分交貨試算

## Goals / Non-Goals

**Goals：**
- 提供跨工單的設備佇列排程介面，取代逐筆手動填日期
- 佇列式自動日期推算，減少印務手動計算
- 階段式相依性提示，避免排程錯誤
- 印件層交期預警，即時識別延遲風險
- 部分交貨試算，輔助業務回答客戶

**Non-Goals：**
- 設備產能管理（每日可用時數、最大負載等）-- Phase 2
- 自動排程演算（系統自動決定最佳排程）-- Phase 2
- 跨工單任務合併（XM-003 OQ 待確認）-- 待 OQ 解答
- 師傅指派（由生管在日程面板操作，非排程中心範疇）
- 行動裝置版（ERP 僅支援桌機瀏覽器）

## Decisions

### Decision 1：佇列式排程 vs 甘特圖拖拉排程

**選擇：佇列式排程**

印務的核心操作是「決定某台設備上任務的先後順序」，而非在時間軸上精確定位。佇列式設計讓印務只需決定一件事 -- 順序，日期由系統自動推算。

替代方案：甘特圖拖拉排程 -- 印務在時間軸上直接拖拉任務色塊到特定日期。優點是直觀，缺點是操作成本高（每筆都要精確定位），且印務不習慣使用甘特圖軟體。

佇列式搭配甘特圖唯讀視覺化是最佳折衷：操作簡單（拖入佇列），結果直觀（甘特圖呈現）。

### Decision 2：日期推算策略 -- 緊密排列為預設

**選擇：緊密排列為預設，允許手動覆蓋開工日產生空隙**

預設假設設備連續運作（前一任務完工次日即接續下一任務），這在印刷業是常態。若印務知道某任務需等材料或有特殊原因，可手動覆蓋開工日。

替代方案 A：每筆都手動填日期 -- 與現有做法相同，無改善。
替代方案 B：固定間隔（如每筆中間留 1 天緩衝）-- 過度保守，降低設備利用率。

### Decision 3：相依性管理 -- 階段概念 + 提示不阻擋

**選擇：沿用 schedule-backtrack-prototype 的階段概念，提示但不阻擋**

工單內的生產任務以 stage_order 分階段。同階段可平行（如同時印正反面），跨階段必須依序（如印完才能上光）。排入設備佇列時：
- 前置階段已排程：顯示提示「前置：印刷預計 4/3 完工」，自動調整開工日
- 前置階段未排程：顯示提示「前置：印刷 尚未排程」，不阻擋排入

不阻擋的理由：印務可能有策略性考量（如先佔設備佇列位置），阻擋會降低靈活度。

替代方案：嚴格阻擋（前置未排不可排後置）-- 過度限制，印務實務中常需同時規劃多階段。

### Decision 4：外包任務的排程模型

**選擇：外包任務不進設備佇列，獨立追蹤區**

外包任務不佔用自有設備，其進度由廠商控制，印務無法排序。但外包的回廠日影響後續自有工序的排程（如外包燙金回來後才能裁切），因此需要追蹤。

設計：
- 外包追蹤區顯示外包任務狀態 + 預計回廠日（expected_return_date）
- 日期推算時：若後續自有工序相依於外包任務，以 expected_return_date + 1 為最早可開工日
- expected_return_date 由印務與廠商確認後手動填寫

替代方案：外包也進佇列（以「虛擬外包設備」呈現）-- 增加不必要的複雜度，外包本質不是設備排程問題。

### Decision 5：部分交貨試算 -- 線性推算 v1

**選擇：線性推算**

公式：某工單完成 X 個所需時間 = (X / 總目標) x 該工單最晚完工生產任務的工期 + 該任務的 scheduled_date。

線性假設偏差：印刷實務中前期有校版試印，後期速度較快，線性推算會低估前 20% 的時間、高估後 80% 的時間。但 v1 粗估已足夠回答業務 80% 的問題。

替代方案 A：精確排程模型（考慮校版時間、良率等）-- 需要大量參數，v1 不切實際。
替代方案 B：系統累積歷史數據後自動校正 -- Phase 2 優化方向。

### Decision 6：排程中心頁面佈局

**選擇：左右分欄佈局**

```
┌──────────────────┬──────────────────────────────────┐
│ 待排區（左側）     │ 設備佇列 + 時間軸（右側）          │
│ - 待排任務清單     │ - 設備佇列（預設視圖）              │
│ - 依交期排序       │   或 時間軸甘特圖（可切換）          │
│ - 可拖入右側佇列   │ - 外包追蹤（下方摺疊區）            │
├──────────────────┤                                    │
│ 交期預警（左下）   │                                    │
│ - 印件交期風險     │                                    │
│ - 部分交貨試算入口 │                                    │
└──────────────────┴──────────────────────────────────┘
```

右側可在「設備佇列」與「時間軸」之間切換：
- 設備佇列：操作用，拖拉排序
- 時間軸：視覺化用，甘特圖呈現結果

### Decision 7：SchedulePlanner 功能定位調整

**選擇：從排程操作入口調整為排程預覽（唯讀）**

現有 SchedulePlanner 在工單詳情頁提供階段管理 + 回推排程功能。排程中心上線後，排程操作統一在排程中心進行。

SchedulePlanner 調整為：
- 顯示該工單各生產任務在排程中心的排程狀態（設備、日期、佇列位置）
- 顯示回推時間軸（唯讀甘特圖）
- 不再支援修改排程（改為連結到排程中心）

### Decision 8：scheduled_date 從必填改為選填

**選擇：建立生產任務時不要求填寫 scheduled_date**

現有 spec 將 scheduled_date 標記為必填，這強迫印務在建立任務時就決定開工日期。排程中心的兩階段設計改為：
- 建立時：scheduled_date 留空
- 排入佇列時：系統自動推算填入

向下相容：scheduled_date 欄位保留，只是不再於建立階段強制要求。任務交付前置條件（所有任務需有 scheduled_date）在排程中心排程完成後自然滿足。

### Decision 9：與既有 change 的整合策略

- **scheduling-completion-date**：直接沿用其 estimated_duration_days、planned_end_date 欄位定義與完工推算規則（含跳過週日）。排程中心在佇列日期推算時複用相同邏輯。
- **schedule-backtrack-prototype**：階段概念（stage_order）整合為排程中心的相依性基礎。SchedulePlanner prototype 的回推演算法保留作為排程預覽的計算引擎。
- **task-dispatch-prototype**：不受影響。排程中心排好後，任務交付給生管，生管在日程面板分派操作員。

## Risks / Trade-offs

**[設備佇列不含產能] -> 可能超載排程**
印務可能在同一天排太多任務到同一台設備。v1 僅顯示「已排 N 天」摘要，不做產能限制。
緩解：印務本身有經驗判斷，且摘要資訊已比現狀（完全不可見）好。Phase 2 再加產能限制。

**[線性推算不精確] -> 部分交貨時間可能偏差**
印刷前期校版試印較慢，線性推算偏樂觀。
緩解：v1 結果標注「估算值」，供參考而非承諾。Phase 2 以歷史數據校正。

**[多印務同時操作佇列] -> 潛在排序衝突**
兩個印務同時拖拉同一設備佇列的任務。
緩解：Prototype 階段使用 mock data，不需處理並發。正式實作時需考慮樂觀鎖定或操作鎖定機制。

**[排程與狀態機獨立] -> 排程變更不影響狀態**
生產任務進入「製作中」後重新排程（改設備/日期），不影響狀態轉換。這是設計選擇（排程是計畫，狀態是事實），但可能造成排程與實際不一致。
緩解：排程中心 SHALL 以不同顏色標示「製作中」任務的排程為「參考用」。

## Open Questions

- 工期預設值表是否需要系統管理員 UI 調整？還是由開發團隊在系統設定中維護？-- 暫定系統設定，Phase 2 再評估
