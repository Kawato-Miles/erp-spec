## Context

工單異動流程是工單管理模組的核心功能之一，涵蓋「交付後需修正工單內容」的所有情境。目前狀態機 spec 與工單 spec 對異動路徑的描述不一致，且 OQ 已決議的規則（WO-004/WO-009/WO-010/PT-002）尚未完整反映在規格中。

現況：
- 狀態機 spec 描述異動為「直接回退至重新確認製程」，無獨立異動狀態
- 工單 spec 描述異動為「進入異動狀態 → 生管確認後返回原狀態」，有獨立異動狀態
- 兩份文件互相矛盾，需統一

前提：
- 收回（Withdraw）機制已定義完整，不在本次修改範圍
- 任務層異動有「已確認異動內容」中間態，工單層不需要
- supplier-portal change 引入供應商自助報工，異動需考慮外包廠情境

參照規格：
- 狀態機 spec：[狀態機](../../specs/state-machines/spec.md)
- 工單 spec：[工單管理](../../specs/work-order/spec.md)
- 生產任務 spec：[生產任務](../../specs/production-task/spec.md)
- 使用者角色 spec：[使用者角色](../../specs/user-roles/spec.md)

## Goals / Non-Goals

**Goals:**
- 統一狀態機 spec 與工單 spec 對異動流程的描述
- 設計異動分流機制：印務選擇「是否需重新審核製程」，系統依此決定返回路徑
- 定義異動期間下游任務行為（繼續執行、完成度持續計算）
- 新增 WorkOrderModification 資料表記錄異動歷史
- 定義供應商平台對異動的呈現行為

**Non-Goals:**
- 不修改收回（Withdraw）機制
- 不修改任務層異動流程（任務層的「已確認異動內容」中間態維持不變）
- 不設計異動的審批層級（如印務主管核准異動）
- 不設計異動的自動化觸發（如 QC 不通過自動發起異動）

## Decisions

### Decision 1：異動分流 — 以「是否需重新審核製程」區分路徑

印務發起異動時選擇「是否需重新審核製程」（是/否），系統依此決定後續路徑：

| 選項 | 適用情境 | 生管確認後的目標狀態 |
|------|---------|-------------------|
| 否（不需重新審核） | QC 不通過補印、追加數量、作廢/報廢任務 | 返回發起時的原狀態（工單已交付 或 製作中） |
| 是（需重新審核） | 修改製程、變更材料規格、大幅調整生產任務結構 | 回到「重新確認製程」，走製程審核流程 |

**理由**：是否需要印務主管重新審核製程，是印務的專業判斷。系統無法自動判斷（例如「新增一個生產任務」可能是簡單補印，也可能是製程變更）。以二元選擇取代「輕量/重量」命名，語意更明確。

**替代方案**：
- 系統依操作內容自動判斷 → 放棄，因為無法準確區分情境
- 所有異動都走重新審核 → 放棄，QC 補印等高頻操作會被不必要的審核拖慢

### Decision 2：異動期間下游任務不停擺

工單進入「異動」狀態期間：

| 項目 | 行為 |
|------|------|
| 已在「製作中」的生產任務 | 繼續執行，不受影響 |
| 已交付但未開工的生產任務 | 維持待處理，生管可繼續分派 |
| 新增的生產任務 | 生管確認後即可指派師傅、開工 |
| 作廢/報廢的生產任務 | 送出異動時立即生效 |
| 完成度計算 | 持續運作 |
| 師傅/供應商報工 | 不受阻擋 |

**理由**：現實中異動發生時，工廠不會停線等待確認。印務通知生管後，生管會立即安排。生管確認只是系統記錄（「我知道了，我會安排」），不是審批卡關。

**替代方案**：
- 異動期間暫停所有下游任務 → 放棄，與現場作業不符，且會導致生產停滯

### Decision 3：路徑 B（需重新審核）已交付任務不收回

當異動選擇「需重新審核」時，工單回到「重新確認製程」。此時已交付的任務繼續執行，不收回：

- 已交付且未受影響的任務群組 → 繼續，不收回
- 新增的任務群組 → 審核通過後才能交付
- 工單狀態不因已有任務在跑而矛盾（「重新確認製程」代表部分製程需重新確認，不代表全部重來）

**理由**：全部收回會中斷已在進行的外包廠生產，造成不必要的成本與延遲。

### Decision 4：異動 UX 流程 — 單一入口

印務在工單詳情頁點擊「異動」 → 進入生產任務編輯介面 → 調整生產任務 → 設定「是否需重新審核」 → 填寫異動原因 → 送出。

送出後工單才進入「異動」狀態。編輯階段是 UI 行為，不影響工單狀態。

### Decision 5：WorkOrderModification 記錄結構

每次異動建立一筆 WorkOrderModification 紀錄，記錄：
- 異動類型（不需重新審核 / 需重新審核）
- 異動原因（必填文字）
- 異動內容快照（JSON：新增/作廢/修改了哪些生產任務）
- 發起人（印務）、發起時間
- 確認人（生管）、確認時間
- 發起時的工單狀態（工單已交付 / 製作中）

### Decision 6：供應商平台異動呈現

| 情境 | 供應商平台行為 |
|------|--------------|
| 已分派的待處理任務被作廢 | 任務從清單消失，顯示通知「任務 XX 已取消」 |
| 已開工的任務被報廢 | 任務移至「已結束」分類，標記「報廢」，可查看不可操作 |
| 新增任務分派到外包廠 | 正常出現在「待處理」，與一般分派無差異 |
| 工單處於異動狀態 | 供應商看不到工單狀態，只看到自己的生產任務狀態（不受影響） |

**理由**：供應商只關心自己的生產任務，不需要知道工單層的異動狀態。作廢/報廢直接反映在任務狀態上。

## 成功指標

| 指標 | 說明 | 目標 |
|------|------|------|
| 異動結構化記錄率 | 透過系統發起的異動（vs 紙本/口頭） | 上線後 >= 80% |
| 異動確認時效 | 印務送出到生管確認的平均時間 | < 2 小時 |

## Risks / Trade-offs

**[印務誤選「不需重新審核」]** → 實際需要重新審核的異動被跳過。緩解：系統記錄完整異動內容，事後可追溯。Phase 2 可考慮印務主管覆核機制。

**[異動期間任務繼續的協調風險]** → 生管可能尚未確認就有師傅開始做新增任務。緩解：新增任務需生管指派後才能報工；「待處理 → 製作中」由首次報工觸發，未指派的任務不會被報工。

**[路徑 B 已交付任務不收回的語意矛盾]** → 工單狀態為「重新確認製程」但有任務在製作中，可能造成混淆。緩解：UI 明確標示「部分製程重新確認中，已交付任務持續執行」。
