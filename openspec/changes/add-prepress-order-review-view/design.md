## Context

審稿以「印件」為單位進行（一張訂單下多件印件，每件各走審稿輪次）。現行審稿頁面（我的待審、待審印件）以印件粒度呈現扁平清單，訂單僅是其中一欄。審稿人員要了解「一張訂單整體卡在哪、其他印件進度、整體交期」只能私下用 Slack 問業務，形成資訊斷點（見 [審稿人員角色卡](../../../memory/Sens_wiki/wiki/erp/03-roles/審稿人員.md)）。

本變更新增一個「按訂單分組」的查詢視圖。技術上是純讀取的視圖重組——訂單與印件的關聯靠既有 `PrintItem.order_id` 反查（審稿輪次 ReviewRound 本身無 `order_id`），指標靠既有審稿時間戳與訂單交期聚合，零新增欄位、零狀態機變更。經序列協作三視角（PM / CEO / ERP 顧問）設計收斂，唯一具結構份量的變更是審稿角色對訂單模組權限的放寬。

## Goals / Non-Goals

**Goals:**
- 讓審稿人員與審稿主管從訂單視角監看待審內容（哪張訂單卡在審稿、卡多久、是否逼近交期）
- 補上「審稿端看不到訂單脈絡」造成的跨部門資訊斷點
- 零資料模型改動（純 read model）

**Non-Goals:**
- 不在此視圖執行審稿動作（送審 / 判合格 / 退件）——維持單一審稿入口（我的待審）
- 不引入訂單之間的父子關係（現行無子母訂單；「母子」指既有的訂單→印件層級）
- 不改造既有審稿頁面（並存不動）
- 不開放審稿角色看訂單的金額 / 帳務 / 異動資訊
- 不新增任何資料欄位或狀態轉換

## Decisions

### D1：純讀取視圖（read model），零資料改動
訂單脈絡靠 `ReviewRound.printItemId → PrintItem.order_id` 兩跳反查取得，沿用 prepress-review spec § 對帳資料可得性 已宣告足夠的反查鏈。
- 替代方案：在 ReviewRound 加 `order_id` 冗餘欄位加速查詢 → 否決，會造成雙寫不一致風險，且既有反查鏈已足夠。

### D2：母集合 = 有待審印件的訂單，排除已取消訂單與已棄用印件
主清單只放「底下至少一件印件審稿維度 ∈ {等待審稿, 已補件, 不合格}」的訂單；全合格訂單走搜尋/篩選回查。
- 排除已棄用印件（打樣 NG 後 clone）：否則棄用印件的審稿維度狀態保留作稽核軌跡、停留永不脫離，會污染停留天數指標。
- 排除已取消訂單：取消訂單的審稿已無意義。

### D3：查詢 + 跳轉動線，不就地審（Miles 拍板）
審稿人員對自己負責印件點「去審」跳回既有「我的待審」審稿詳情頁，此視圖不承載審稿動作。
- 理由：避免變成第三個審稿入口，導致「待審計數 / 績效統計」兩邊對不起來。對齊業界（Good2Go 區分「單件審」與「整單查」兩動線）。
- 替代方案：就地展開審稿 → 否決（CEO 管理立場反對 + Miles 拍板）。

### D4：主管與審稿列表一致，差異用篩選處理（Miles 拍板）
不為審稿主管做專屬視圖；主管與審稿人員的列表欄位、排序、操作一致，「看自己 vs 看全部」的差異以篩選功能處理。
- 此決策化解了序列協作中 CEO 提出的 OQ-A（主管首要動作=監看卡關 vs 監看分派的主視圖分歧）——既然不做專屬主視圖，就沒有「主排序軸要選哪個」的問題。
- 替代方案：主管專屬「按停留天數排序」主視圖 + OQ-A 切換點 → 否決（Miles 判斷先做一致列表即可，實務差異用篩選）。

### D5：審稿角色對訂單模組權限 X → R（限定唯讀），引入第三級權限
原權限模型只有 R/W 與 X 兩級。為讓審稿角色唯讀訂單脈絡（編號/客戶/交期/印件清單 + 限訂單層級詳情）但不開放訂單模組完整功能，引入「R（限定唯讀）」第三級。
- delta op：MODIFIED user-roles spec § 模組存取權限模型 + § 審稿主管與審稿角色階段限制（supersession 不可 ADDED，否則主 spec「審稿訂單=X」與「審稿可讀訂單脈絡」並存矛盾）。
- 連帶放寬 US-AR-007 成功條件 1「非分派給自己不可見」為「同訂單脈絡內唯讀可見」。
- 替代方案：維持訂單模組 X，把訂單資料視為「審稿模組內呈現的跨實體資料」不算訂單模組存取 → 否決（定案⑧「點進訂單詳情」實際進入訂單模組頁面，觸及 spec「X→不顯示入口」，無法迴避，須誠實 MODIFIED 權限）。

### D6：訂單審稿停留天數採牆鐘時長口徑（Miles 拍板）
首件送審（首輪 submittedAt）→ 全部審畢（最終 reviewedAt；未全部審畢計至今日）的日曆工作天。
- 理由：零新增欄位即可算（守 D1）；業界 SLA 系統的 Elapsed Time 為基礎款。
- 替代方案：累計在審時長（純反映審稿工時）→ 否決，需新增「印件審稿維度狀態每次進出時間戳」事件流欄位，破壞零新增欄位前提。
- 偽指標防護：停留天數 MUST 與退件率配對顯示，不可單獨作審稿人員績效（避免「為降天數草率放行」）。

### D7：逾期訂單數改距交期口徑
「距客戶交期 < 安全緩衝且仍在母集合」，用既有 Order.客戶交期，不依賴 D6 的停留天數口徑，且更貼「會不會爆交期」的管理痛點。閾值未定（見 Open Questions）。

### D8：訂單詳情入口限訂單層級三頁籤（Miles 拍板）
資訊（訂單編號/客戶/交期/案名）+ 印件清單 + 活動紀錄；隱藏金額/發票/對帳/異動/工單/QC/出貨。
- 與顧問建議差異：顧問建議含「審稿紀錄」頁籤，但訂單詳情頁目前無此頁籤，本次先不新增（審稿紀錄維持在印件層查看）→ 落地為三頁籤。

## Risks / Trade-offs

- [權限放寬被誤解為開放訂單編輯] → 緩解：明確定義「R（限定唯讀）」第三級，spec 寫明不可編輯、僅限查詢視圖內限定欄位 + 限訂單層級詳情閹割。
- [牆鐘時長含「等客戶補件的等待時間」、不純反映審稿員工時] → 緩解：與退件率配對顯示，且不作組員績效 KPI；若未來需純工時口徑再評估是否值得加事件流欄位。
- [指標②逾期安全緩衝閾值未定] → 緩解：prototype 先採距交期 ≤ 2 工作天暫定值，正式值待確認（見 Open Questions）。
- [審稿人員看到非自己負責印件，可能誤動] → 緩解：非自己負責印件唯讀、無「去審」動作。

## Migration Plan

- Prototype 新增「審稿訂單脈絡查詢視圖」頁面元件，沿用既有依登入角色切資料範圍的範式（如 PrintItemDashboard 的 role-based 過濾）。
- 審稿平台與中台側邊欄各新增入口。
- 訂單詳情頁新增審稿平台限定唯讀版（三頁籤閹割）。
- 無資料遷移（零新增欄位）。
