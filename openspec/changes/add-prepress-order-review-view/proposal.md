## Why

審稿人員需要從「訂單視角」查看稿件所關聯的整張訂單脈絡——哪些訂單卡在審稿關卡、卡了多久、會不會爆交期。但現行審稿頁面只有「印件粒度」的扁平清單（訂單僅是其中一欄），審稿員要了解一張訂單的整體狀況（同訂單其他印件進度、整體交期）只能私下用 Slack 問業務，形成資訊斷點（見 [審稿人員角色卡 § 預期阻力](../../../memory/Sens_wiki/wiki/erp/03-roles/審稿人員.md)）。本變更新增一個「按訂單分組」的查詢視圖，把既有的「訂單（母）→ 印件（子）」層級（[印件實體卡](../../../memory/Sens_wiki/wiki/erp/05-entities/印件.md)：一張訂單下多件印件、審稿以印件為單位）重組成審稿端可直接監看的訂單脈絡。

## What Changes

- 新增「審稿訂單脈絡查詢視圖」獨立子模組頁面，與既有「我的待審」「待審印件」並存（既有頁不動）。
- 以「訂單（母）→ 印件（子）」兩層呈現。母集合 = 底下至少有一件印件審稿維度狀態 ∈ {等待審稿 / 已補件 / 不合格} 的訂單；**排除已取消訂單與已棄用印件**（避免棄用印件停留永不脫離污染指標）。
- 訂單母列顯示：訂單編號 / 客戶 / 客戶交期 / 距交期天數 / 訂單審稿停留天數（與退件率配對顯示）。**不顯示「待審 N / 共 M」這類審稿狀態彙總標籤**（交期屬訂單既有識別屬性、非彙總標籤）。
- 展開訂單列出該訂單下「全部」印件（含已合格、以審稿狀態標示）。自己負責的印件可點「去審」跳回「我的待審」審稿詳情頁（**不在此頁就地審**）；非自己負責的印件唯讀。
- 審稿人員只見「底下有自己負責印件」的訂單；審稿主管見全部訂單。**審稿主管與審稿人員的列表先做成一致，不為主管做專屬視圖；看自己 vs 看全部的差異以「篩選功能」處理。**
- 訂單母列可點進訂單詳情，**限訂單層級**：僅顯示「資訊（訂單編號 / 客戶 / 交期 / 案名）」「印件清單」「活動紀錄」三頁籤，隱藏金額 / 發票 / 對帳 / 訂單異動 / 帳務 / 工單 / QC / 出貨（詳見 [ORD-039](../../../memory/Sens_wiki/wiki/erp/08-open-questions/ORD-039-審稿平台版訂單詳情頁顯示範圍.md)，已拍板）。
- **MODIFIED（業務規則放寬）**：審稿人員 / 審稿主管對訂單模組權限由「無存取」放寬為「限定唯讀」（可讀訂單編號 / 客戶 / 交期 / 印件清單，不可讀金額 / 帳務 / 異動，不可編輯）。連帶放寬 [US-AR-007 成功條件 1](../../../memory/Sens_wiki/wiki/erp/13-user-stories/prepress-review/US-AR-007-執行印件審稿.md)「非分派給自己不可見」為「查詢視圖內非自己負責印件唯讀可見」。
- 新增審稿環節訂單級指標：**訂單審稿停留天數**（牆鐘時長：首件送審 → 全部審畢的日曆工作天，與退件率配對）、**審稿關卡逾期訂單數**（距客戶交期 < 安全緩衝且仍在母集合，閾值見 [AR-14](../../../memory/Sens_wiki/wiki/erp/08-open-questions/AR-14-審稿關卡逾期訂單距交期緩衝閾值.md)）、**訂單視角使用率**（上線後驗證，prototype 不強求）。
- **零實體 / 狀態機 / 欄位變更**：純讀取視圖（read model），訂單關聯靠印件反查（`PrintItem.order_id`），審稿輪次（ReviewRound）本身無 order_id；指標靠既有審稿時間戳聚合。

## Capabilities

### New Capabilities

（無新增 capability；本變更落在既有審稿模組與角色權限模型內。）

### Modified Capabilities

- `prepress-review`: 新增「審稿訂單脈絡查詢視圖」Requirement（訂單母→印件子兩層查詢視圖、母集合定義、查詢+跳轉動線、訂單母列顯示欄位）+ 審稿環節訂單級指標（停留天數 / 逾期訂單數 / 視角使用率）。與既有審稿總覽 / 待審印件並存、不取代。
- `user-roles`: 審稿人員 / 審稿主管對訂單（order-management）模組的存取權限由「無存取（X）」MODIFIED 為「審稿脈絡查詢視圖內限定唯讀」；補審稿主管「按訂單軸監看」職責。
