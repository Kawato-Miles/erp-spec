## Context

印件詳情頁已存在於 `/work-orders/print-items` 路徑下，目前承載的功能：填寫預計產線（order-management spec § 印件預計產線）、上傳成品縮圖（同 spec § 印件成品縮圖上傳）、顯示稿件檔案。本次擴充新增「工單與生產任務」區塊與報工入口。

印務角色定位（依 user-roles spec § 印務角色職責）原本是「單工單負責人」，跨印件僅能透過印件總覽唯讀其他工單。本次將印務的責任邊界明確化：「印件 PM」負責印件層追蹤，但對非自己工單仍維持唯讀；報工權限以工單負責人為唯一守門條件。

報工行為已在 production-task spec 多處定義（§ 師傅自助報工、§ 供應商自助報工、§ 生管代替報工、§ 批次報工操作）。本次不新增報工邏輯，僅新增觸發位置並重用工單詳情頁批次報工元件。

## Goals / Non-Goals

**Goals:**
- 為印務提供以印件為主語的跨工單追蹤視角，減少在多工單詳情頁切換成本
- 報工入口從工單頁延伸到印件頁，讓印務在追蹤狀況的當下即可報工
- 跨印務印件透明化：完整可見其他印務的工單進度，但操作邊界明確
- 守門規則 UI / API 雙層檢查，避免權限洩漏

**Non-Goals:**
- 不引入印件層的進度百分比或聚合完成度數字（沿用既有狀態詞）
- 不在印件詳情頁加入 QC 單建立、工單異動、生產任務新增 / 編輯（仍於工單詳情頁進行）
- 不變更既有報工邏輯（首次報工觸發、累加規則、狀態向上傳遞、ProductionTaskWorkRecord 結構皆不動）
- 不新增 Data Model 欄位
- 不在印件詳情頁加入稿件版本歷程、出貨狀態 Tab（驗證後再決定）
- 不對師傅自助 / 供應商自助 / 生管代報工流程做任何變動
- 不為印務主管設計跨工單代報工特權

## Decisions

### 1. 進度呈現採用狀態詞 + 齊套性三欄（不做百分比）

**選擇**：印件 / 工單 / 生產任務各層級顯示既有狀態詞（依 state-machines spec），生產任務額外顯示「預計 / 完成 / 入庫」三欄原始數字 + QC 狀態徽章。

**理由**：
- 狀態機已是團隊溝通共通語，不需新增解讀層
- 百分比涉及加權邏輯（任務數平均？產量加權？BOM 工序權重？）每種都有反例
- 齊套性三欄為既有 work-order spec § 工單詳情頁印件區塊既定設計，印件詳情頁直接複用同口徑，使用者習慣一致
- QC 徽章（最近一次 QC 結果：通過 / 不合格 / 未檢）解決「PT 完成數達標但 QC fail 重印」造成的戰情失真風險（若僅看完成數會誤判進度）
- 三欄差距即可判讀齊套性瓶頸：完成 - 入庫 = QC 重印缺口；目標 - 完成 = 待生產缺口

**替代方案**：
- A：簡單「完成任務數 / 總任務數」── 但「報 99% 的 10 個任務」會顯示 0%，誤導
- B：依目標數量加權的完成率 ── 計算合理但跨工序權重不直觀
- C：BOM 工序順序加權 ── 最貼近真實產出，但需新欄位且 BOM 權重來源未定義
- D：僅顯示完成數 / 目標數 + 狀態詞（無入庫欄與 QC 徽章）── 第一版會撞到 QC 失敗造成戰情失真

### 2. 跨工單可見性採完全透明（方案 β）

**選擇**：印務開啟印件詳情頁時，他人負責工單下的所有生產任務以完整明細呈現（編號、工序、目標數、已完成數、狀態），僅報工操作禁用。

**理由**：
- 印件詳情頁定位為「戰情室」，戰況不能模糊化
- 跨印務協作時印務需判斷「印件卡在哪一步」，僅看狀態條無法定位瓶頸
- 已透過守門規則限制操作，可見不可改不會造成資料風險

**替代方案**：方案 α 僅顯示工單狀態條 ── 隱私強，但失去戰情室價值

**風險**：印務之間可能有「被監看」觀感 → 透過 user-roles spec 明確規範與印務溝通

### 3. 報工權限守門以「工單負責人」為唯一條件

**選擇**：報工按鈕 / 勾選框啟用條件為 `WorkOrder.assigned_to == current_user`，UI 與 API 雙層檢查。印務主管不享跨工單代報工特權。

**理由**：
- 與既有 user-roles 規範一致（印務只負責自己工單），不開新例外
- 印務主管若需報工，須先成為該工單負責人，符合既有指派流程
- 避免「跨工單代報」變成新的代報工管道，與 § 生管代替報工 重疊產生語意混淆

**替代方案**：
- A：印務主管可代報所有工單 ── 與生管代報工角色重疊，責任歸屬模糊
- B：印務可代報同印件下他人工單 ── 違反 user-roles 跨印務唯讀規則，需大改

### 4. 重用工單詳情頁的批次報工元件

**選擇**：印件詳情頁批次報工 UI 與 production-task spec § 批次報工操作 完全一致，僅在每筆生產任務多顯示「所屬工單編號」標註。

**理由**：
- 報工流程一致性，避免兩處 UX 走鐘
- 印件頁下勾選的任務可能跨多個工單，必須標註工單編號避免報錯
- 元件重用降低實作與維護成本

### 5. 入口維持單一路徑

**選擇**：印件詳情頁僅從印件總覽（`/work-orders/print-items`）點擊進入，不額外從訂單詳情、工單詳情建立入口。

**理由**：
- 印件總覽已是印務的標準入口，不需擴散
- 訂單詳情頁顯示印件清單時可導航至印件詳情頁，但本次 change 不修改訂單頁，保持 scope 收斂
- 後續若驗證需多入口再追加

### 6. ProductionTaskWorkRecord 新增 source_channel 欄位（必修）

**選擇**：所有報工記錄寫入時必須帶入 `source_channel` enum（craftsman_platform / supplier_platform / dispatch_board / work_order_detail / print_item_detail），由系統依提交來源自動填入，使用者不可修改。

**理由**：
- 報工管道從 4 個變 5 個後，未來資料異常無從歸因（「這筆報工從哪裡提交」無從查證）
- 採用率分析需要：印件詳情頁推出後，需驗證使用者是否真採用此入口，無此欄位無法判斷
- 防呆與稽核：API 層守門被觸發時（繞過 UI 直接呼叫），需寫入稽核日誌，與 source_channel 配套形成完整可追溯性
- 既有報工紀錄遷移：歷史資料可預設 `legacy` 或依規則回填，不阻擋本 change 上線

**替代方案**：
- A：僅在 UI 層加追蹤事件（如 GA / 埋點）── 不寫入資料庫無法做後續分析查詢，且異常稽核需要持久化記錄
- B：用 `reported_by` 反推 channel ── 同一使用者可能跨多管道（印務在工單頁 vs 印件頁），無法區分

### 7. 印件狀態讀取路徑統一

**選擇**：印件總覽列表與印件詳情頁顯示的印件印製狀態 MUST 來自同一讀取路徑（同一查詢函式或 cache 來源）。

**理由**：
- 印件狀態本來就紀錄在印件實體上（透過 bubble-up 規則維護），兩處若用不同路徑取值（如總覽用 cache、詳情即時讀），可能出現「總覽顯示等待中、詳情顯示製作中」
- 不一致顯示會讓使用者懷疑系統壞掉，瓦解戰情室的可信度
- 此為工程實作層的明文規範，避免實作時各自詮釋

## Risks / Trade-offs

- **印務主管請假代報工的斷鏈**：本次決定主管不享跨工單代報特權；現實中印務請假時需有代理機制 → **後續另起 change**：採分享權限模式（類似需求單），由請假印務或主管臨時將工單分享給另一位印務，期間享報工權，不開「主管特權」例外
- **守門 UI 層繞過風險**：若 API 層未檢查，惡意使用者可繞過 UI 直接呼叫 ProductionTask 報工 API → spec 明確要求 UI / API 雙層守門 + 稽核日誌
- **工單編號標註的視覺干擾**：報工面板每筆顯示所屬工單會增加資訊密度 → 採用次要文字樣式（小字、淡色），避免干擾數量輸入主視線
- **印件層級若被誤解為「上層代理」**：可能被誤用為跨工單操作集中地（如建 QC、發起異動）→ 透過 spec 明確 Non-Goals，UI 上不放這些操作入口
- **既有報工紀錄無 source_channel**：上線前需決定遷移策略 → 建議歷史資料統一回填 `legacy` 或依 reported_by 角色推測，不阻擋本 change 上線
- **批次提交部分失敗**：印件頁勾選跨多 PT 時，若提交當下其中一筆守門失敗（如工單剛改派），採個別提交策略（成功的記、失敗的明確告知），不原子回滾 → 此為工程實作邊界，沿用既有 § 批次報工操作 規則
- **沒有百分比指標的長期影響**：若團隊後續需數據化追蹤，需另起 change 設計加權邏輯 → 留待 Phase 2 / 3 視 KPI 需求再評估
