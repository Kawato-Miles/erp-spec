## Why

印務是印件印製階段的 PM，但現況缺少「以印件為主語」的統整視角：一個印件常拆成多張工單（打樣 / 大貨 / 加工），印務必須在多個工單詳情頁之間切換才能追蹤整體進度，跨印務協作的印件更難看清對方進度。同時，報工只能從工單詳情頁切入，印務在追蹤印件狀況時要離開印件語境去操作。

更深層的問題：印件作為「跨工單聚合單位」原本就應該有 PM 視角的儀表板，本次補上的不只是效率，而是**印件交期可視化的責任歸屬視圖**——讓印件這個層級真正有人在看。

## What Changes

- 在既有「印件總覽」頁（`/work-orders/print-items`）點擊印件進入的**印件詳情頁**擴充「工單與生產任務」區塊
- 新區塊呈現該印件下所有工單，依工單分組展開生產任務，顯示工單狀態、生產任務狀態、預計 / 完成 / 入庫齊套性三欄、QC 狀態徽章
- 生產任務列提供**報工入口**，權限守門：僅自己負責工單（`WorkOrder.assigned_to == current_user`）下的生產任務可報工，其他工單可讀不可操作
- 支援批次報工：印件詳情頁可勾選自己負責工單下的多筆生產任務，一次開啟報工面板（重用工單詳情頁既有報工元件）
- 印務主管權限與印務一致（受工單負責人約束）
- ProductionTaskWorkRecord 新增 `source_channel` 欄位（資料溯源與稽核），既有四個報工管道與本次新增的印件詳情頁管道皆需正確標記
- API 層守門被觸發時寫入稽核日誌
- 印件狀態讀取路徑統一：印件總覽與印件詳情頁共用同一讀取來源，避免兩處狀態不一致
- 非本次範圍：QC 單建立、出貨狀態 Tab、異動歷史、稿件版本歷程、停滯天數、進度百分比（第一版不放，驗證後再決定）
- 後續另起 change：印務主管臨時代理機制（採分享權限模式，類似需求單）

## Capabilities

### New Capabilities

無。本次不引入新 capability。

### Modified Capabilities

- `order-management`：印件詳情頁新增「工單與生產任務」區塊（跨工單彙整含齊套性三欄 + QC 徽章 + 報工入口）；新增印件狀態讀取路徑統一規則
- `production-task`：報工入口新增一處（印件詳情頁）；ProductionTaskWorkRecord 新增 `source_channel` 欄位；API 層守門寫入稽核日誌
- `user-roles`：印務 §「跨印務印件」場景補充印件詳情頁適用同規則（可看不可改）

## Impact

- 前端：新增印件詳情頁的「工單與生產任務」區塊 UI（含齊套性三欄與 QC 徽章顯示）；重用工單詳情頁的報工 Modal / Sheet 元件
- 後端：ProductionTaskWorkRecord 新增 `source_channel` 欄位（必修）；API 層守門加入稽核日誌；報工 API 權限檢查沿用工單負責人邏輯；五處既有 / 新增報工管道均需正確帶入 source_channel
- 規格文件：三個 spec 新增 delta（order-management / production-task / user-roles）
- 使用者：印務與印務主管於跨工單印件的追蹤效率與責任歸屬可見性提升，不變更既有工單詳情頁操作
- 風險：source_channel 欄位需既有四個報工管道一併補齊資料（既存報工紀錄可考慮預設值如 `legacy`）
