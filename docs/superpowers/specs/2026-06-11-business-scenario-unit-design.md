## 一、背景與目的

wiki 是 BRD 商業需求正本（公司怎麼做生意的單一可信描述，第一讀者是不參與開發的 PM）。03 / 04 / 06 已依憲章重構；07-scenarios 與 13-user-stories 仍是舊式範本，且兩層存在實證的重複維護（13 的「業務流程」段重述 07 的接力順序）。本設計把兩單元合併為單一「業務情境」單元，補齊商業行為描述中「目標完成過程」這個維度，並接住盤點發現的三個無家內容（定期事務、外部系統參與的過程、外部介面契約的過程面）。

## 三、單元定位（範本「定位節」內容綱要）

依憲章「單元定位模式」四題：

1. **本質**：業務情境卡 = 商業知識在「目標完成過程」維度的投影——一個業務目標從觸發到收斂，誰（角色 / 外部系統 / 時間）依什麼順序做什麼、每步憑什麼可觀測的業務結果判斷成立。過紙本測試：純手工時代這套過程存在於簽核單與口頭默契。
2. **投影分工**（正本只有一處，引用不複寫）：
   - 對 04 服務藍圖：藍圖是型（這類生意一般怎麼跑，5-10 階段、不含判準）、情境是實例（這一個目標怎麼走完，含判準）。接力型卡頭標「具現自 [[某藍圖]] 階段 N-M」。
   - 對 04 商業規則：判準本體（門檻數字、計算邏輯、為什麼）正本在規則卡；情境步驟的判準 = 一句關卡描述 + wiki link，禁抄準則與數字。
   - 對 03 角色：動作歸誰（與為什麼歸他）正本在角色卡；情境只寫動作的時序接力，角色名 wiki link。
   - 對 06 狀態機：狀態詞彙與轉換正本在狀態機卡；情境只引用狀態名（一字不差），禁轉換箭頭、守衛條件、狀態定義。
   - 對 05 實體：欄位正本在實體卡；情境點名實體 wiki link，不抄欄位表。
   - 對 01 痛點：痛點正本掛接點唯一在藍圖痛點段（歸因 pain-points）；情境只寫 how，要連痛點以 wiki link 指回，不重述現象。
3. **已知陷阱**：寫成第二份藍圖（自己在描述「所有變體共通的骨架」即越界）；把規則門檻抄進判準；把痛點寫進情境；每條路徑各開一張卡（應主成功過程為骨幹、岔路掛接）；以 OpenSpec 為情境正本（舊 07 README 的違憲現況，本次廢除）。
4. **伺候的讀者情境**：商業自足理解（一卡讀懂一件事怎麼做完）、規劃自查（改規則前反查哪些過程經過它）、AI 載入依據（防發明不存在的分支）、訪談確認（與業務核對實際做法）。

## 四、三個變體（判斷表分流，同卡型同治理）

| 變體 | 觸發者 | 範圍 | 例 |
|------|--------|------|---|
| 接力型 | 角色或外部方，跨多位置交棒 | 一個業務事件的回應接力，綁定藍圖路徑 | 訂單取消退款、諮詢取消退費 |
| 能力型 | 單一角色，一次坐定完成 | 單角色目標（含不掛任何接力的獨立能力） | 訂單複製、登錄收款核銷 |
| 排程型 | 時間或系統事件（非人類角色） | 定期 / 批次事務，過程常為人機混合 | 月結對帳、逾期警示 |

- 行為者明訂可含**外部系統 / 外部方**（金流平台、surveycake 可為步驟行為者）。
- 外部介面契約本體（payload 對應、重試）歸 04 外部約束（照 ezPay 卡模式）；情境只承載「互動過程」。

## 五、卡的產出格式（段落骨架）

| 段落 | 內容 | 必/選 |
|------|------|-------|
| 概述 | 目標一句（誰要達成什麼 + 為什麼值得）+ 觸發 + 變體 + 接力型標「具現自 [[藍圖]] 階段 N-M」+ 正本歸屬聲明 | * |
| 主成功過程 | 編號步驟：行為者 + 業務動作 + 成立判準（有規則正本 → 一句關卡 + 引用；純程序結果 → 直接寫可觀測業務結果；狀態名與 06 一字不差） | * |
| 延伸岔路 | 掛載步 + 條件 + 處置（Cockburn additive；處置可引其他情境卡，禁複述主線；共用過程「自第 N 步起同 [[主情境]]」） | ? 無岔路可省 |
| 收斂狀態 | 成功保證：做完後世界處於什麼狀態（引 06 狀態名） | * |
| 範圍外 | 不涵蓋什麼、屬於誰；易被實作腦補的主題點名「存在但不在本卡」 | * |
| 相關卡 | 依分類 wiki link（藍圖 / 規則 / 角色 / 實體 / 狀態機 / 情境），只列有連結的分類 | * |

frontmatter：

```yaml
type: scenario            # 沿用既有 type，wiki-schema 同步改語意
variant: 接力型 | 能力型 | 排程型
module: [模組，中文值]
business-domain: [六領域之一或跨領域]
source:                   # 往上：藍圖 / 04 規則卡 / 拍板 OQ / 外部依據；禁同層、下層、實作文件
implemented-by:           # 往下：導航中繼資料，可留空
status: draft | active
last-reviewed: YYYY-MM-DD
```

廢除欄位：`us-id`、`priority`、`prerequisites`、`related-test-cases`、`stage`（交付導向欄位，BRD 不收；依賴關係由過程順序與引用表達）。
