# 角色卡撰寫範本

> 設計依據：[[index]] § Wiki 憲章（本質、第一讀者、兩判準、單元定位模式）＋ 與 `06-state-machines/_template-state-machine.md` 同構的治理機制。
> 本範本依「雙向開發」迭代：每張卡走查時發現的缺口回饋範本。

---

## 〇、角色卡是什麼（先懂定位再動筆）

> 本節目的：每條規則都從定位推導。Wiki 的本質、第一讀者、兩判準（紙本測試／商業語言）、BRD 自足與「範圍外＝防護欄」見 **[[index]] § Wiki 憲章**——本節只講角色單元的特定定位。

**角色卡是公司分工的正本**——這個位置的人做什麼、把關什麼、不碰什麼。公司用紙本與口頭默契運作時這套分工就存在（職務說明、誰簽核誰執行），完全過紙本測試。

**它是商業知識的「人」視角投影**（與鄰層分工，投影不互相複寫）：

- 分權規則的「為什麼」（如補收免審、退款送審的理由）正本在 `04-business-logic/`，本卡只寫**動作歸屬**（「業務建補收單直接生效；退款送業務主管核可」）。
- 跨角色的接力順序屬服務藍圖與情境卡；單一角色的操作步驟屬使用者故事。

**收什麼**：公司分工中的「位置」——內部職位（業務、印務、會計…）、外部協力（外包廠商、中國廠商——他們承擔業務動作）、高階監督（Supervisor＝高階主管，經營監督視角）。**不收**：純系統帳號概念（過不了紙本測試）、客戶端使用者（B2C 會員依既有決議不納正式角色）。

**已知陷阱**：把跨角色流程寫進角色卡（那是情境的事）；**把帳號權限矩陣當角色定義**——角色＝職責分工，不＝系統權限；職務範圍用商業語言三分法表達，細粒度權限屬實作規格。

**本單元主要伺候的讀者情境**：商業自足理解（這家公司怎麼分工）、規劃自查（設計功能時誰該操作什麼、誰把關）、訪談確認（與角色本人核對職責）。

## 一、撰寫流程

```
辨識 → 歸位 → 確認前提 → 撰寫 → 稽核 → 定案
```

| 步驟 | 做什麼 | 這一步的產出 | 卡住時 |
|------|--------|------------|--------|
| **1. 辨識** | 這個「人」需要開角色卡嗎？依 § 二判斷 | 開卡與否 | 走不通 → § 六 |
| **2. 歸位** | 檔名＝角色名（繁中語意化），放 `03-roles/` | 檔名與落點 | 走不通 → § 六 |
| **3. 確認前提** | 依 § 三逐項確認 | 已確認清單（任一缺 → 停下問） | 停下問使用者 |
| **4. 撰寫** | 依 § 八格式填入，遵守 § 四規則 | 卡草稿 | 需要假設 → § 七 |
| **5. 稽核** | 依 § 十二維度自檢 | 逐項勾稽結果 | 不通過 → 回 3 或 4 |
| **6. 定案** | commit ＋ 追加 wiki/log.md（逐卡 `[[卡名]]`＋動機） | — | — |

## 二、判斷表

> 本節目的：不是每個稱呼都值得開卡——別名、兼任、純帳號概念開了卡會讓分工圖失真。

| 順序 | 問自己 | 否的話 |
|------|--------|--------|
| 1 | 這個位置有**獨立的職責歸屬**嗎？（有自己負責或把關的業務動作，不是另一個角色的別名） | 不開卡，在既有角色卡註明別稱 |
| 2 | 它承擔的是**業務動作**嗎？（過紙本測試——純系統帳號層級、登入身分概念不算） | 不開卡，屬實作規格 |
| 3 | 是公司分工的一員嗎？（內部職位／外部協力／高階監督） | 客戶端使用者不開正式角色卡 |

兼任（一人多角色）不影響開卡——卡寫的是「位置」不是「人」；同一人兼業務與諮詢，兩張卡照開。

## 三、前提確認（MUST 全部已確認才能動筆）

> 本節目的：答不出來的是默會知識缺口——停下問，不准編。

| 前提 | 確認狀態 |
|------|---------|
| 職責清單的來源依據在哪？（訪談／角色權責表／管理層拍板） | 已確認 / 待確認 → 停下 |
| 這個角色把關哪些節點？把關失敗會擋住什麼？ | 已確認 / 待確認 → 停下 |
| 它的職務範圍三分法（負責什麼／看得到什麼／不接觸什麼）？ | 已確認 / 待確認 → 停下 |
| 有沒有容易混淆的近似角色？權責怎麼切？ | 已確認 / 確認沒有 |
| 分權動作背後的規則正本在哪張商業規則卡？ | 已確認 / 找不到 → 先補 04 或開 OQ |
| 導入系統時這個角色的預期阻力？ | 已確認 / 訪談未提及（如實標注，不編） |

---

## 四、撰寫規則

> 本節目的：MUST 不容裁量、SHOULD 偏離須說明、MAY 自由。每條附理由。

| 強度 | 規則 | 理由 |
|------|------|------|
| MUST | 每段內容過紙本測試與商業語言兩判準（§ 〇）；正文零實作規格引用 | 本卡是 BRD，第一讀者讀不到實作文件 |
| MUST | 職務範圍用**商業語言三分法**：「負責的單據（可建可改）／看得到但不能動／不接觸」，不用 R/W/X 縮寫、不用固定模組行 | 縮寫是系統語言、商業讀者要翻譯；固定行跟不上模組成長 |
| MUST | 職責寫**動作歸屬層**；分權動作回引 `04-business-logic/` 規則卡，不複寫判斷準則 | 規則只寫一次；角色卡答「誰做」、規則卡答「為什麼」 |
| MUST | 職責邊界每條附「為什麼這樣切」 | 沒有理由的邊界會被下一次組織調整無意推翻 |
| MUST | 有易混淆的近似角色時，概述寫明區隔（如「業務主管 ≠ Supervisor」） | 同義詞混用是分工圖失真的主因 |
| MUST | 寫「範圍外」段；**細粒度權限矩陣點名「存在但屬實作規格」** | 範圍外是防下游過度設計的防護欄 |
| SHOULD | 非顯然的職責附一句營運動機 | 防止後人無意推翻有理由的安排 |
| SHOULD | 預期阻力具體到「什麼改變引發什麼抗拒」；訪談未提及就寫「（訪談未提及）」 | 空洞阻力（「可能不適應」）無資訊量；編造阻力比沒有更糟 |
| MAY | 職責單純的角色（純檢視、單一動作）可省略職責邊界段，邊界併入職務範圍 | 顯然的事不必開段落 |

## 五、修改既有卡前的檢查

> 本節目的：職責分工是組織決策，改動是商業行為不是編輯行為。

| 修改類型 | 改之前要確認什麼 |
|---------|----------------|
| 新增／移除職責、改把關點 | 組織決策依據（管理層拍板／訪談定案）；沿引用反查情境卡與使用者故事的連帶 |
| 純措辭修正 | 不需依據，但仍須通過語言紀律稽核 |

## 六、邊際情境處理

| 卡住的情境 | 標準處理 |
|-----------|---------|
| 疑似別名（兩個稱呼一套職責） | 問使用者哪個是正名，另一個在卡內註明別稱，不開兩張卡 |
| 一人身兼多角色 | 各角色照開卡（卡寫位置不寫人），兼任事實在相關卡互註 |
| 純系統帳號概念（登入身分、權限群組） | 不開卡，屬實作規格 |
| 外部協力的職責界線不明 | 停下問（外包廠商做到哪、誰驗收） |
| 職責清單與權責表矛盾 | 停下，比對呈報使用者裁決，不自行調和 |

## 七、error handle（出現任一即停下）

1. **無來源依據**：職責只有推測 → 停下
2. **矛盾**：與權責表、服務藍圖、其他角色卡的分工矛盾 → 停下，列出矛盾兩方
3. **需要假設**：必須假設「這應該是他做的」才能繼續 → 停下
4. **邊界講不出理由**：某條邊界為什麼這樣切說不出來 → 停下問
5. **超出已知**：要描述的職責超出已確認範圍 → 停下

處理方式：標記具體問題、附已知事實提問；**禁止**用「一般來說」「通常」填充。改寫舊卡發現的問題，保留舊內容另開 OQ。

## 八、產出格式

一張卡＝一個分工位置。**檔名即標題**（卡內不寫 `# 標題`）。**正文零實作規格引用、無正文來源段**（正確性來源由 frontmatter `source` 承載；歷史查 wiki/log.md 與 git）。

**段落結構**（七段；「為誰」＝伺候的讀者情境）：

| 段落 | 內容 | 為誰 | 必/選 |
|------|------|------|-------|
| 概述 | 誰、哪類（內部職位／外部協力／高階監督）、守哪道關、與近似角色的區隔、參與哪幾段流程 | 第一次接觸的 PM | * |
| 職務範圍 | 商業三分法：負責的單據（可建可改）／看得到但不能動／不接觸 | 分工速查 | * |
| 主要職責 | 逐項動作歸屬；非顯然附動機；分權動作回引 04 | 規劃自查 | * |
| 職責邊界 | 每條附「為什麼這樣切」 | 防推翻有理由的分工 | * 職責單純可併入職務範圍 |
| 關切點 | 這角色在意系統提供什麼（訪談素材；角色本人讀了能說「對，我就在意這個」） | 訪談確認 | ? |
| 預期阻力 | 什麼改變引發什麼抗拒；訪談未提及就如實標注 | 導入規劃 | ? |
| 範圍外＋相關卡 | 不涵蓋什麼、屬於誰；易腦補主題點名；相關卡依分類列（規則／狀態機／情境／實體／角色） | 防下游捏造＋互連 | * |

**frontmatter**：

```yaml
type: role
module: [模組，中文值，見 wiki-schema § 二]
business-domain: [六領域之一或跨領域；多數角色屬跨領域]
source:                # 往上＝正確性根據：角色權責表／管理層拍板／04 分權規則卡；禁指同層、下層、實作文件
  - "[[<分權規則卡>]] 或 Notion 角色權責表（連結記 related-notion）"
implemented-by:        # 往下＝導航中繼資料：實作規格檔層；可多值、可留空
  - "openspec/specs/<對應模組>/spec.md"
related-notion: <Notion 核心角色權責 DB 連結>   # 若有
status: draft | active
last-reviewed: YYYY-MM-DD
```

## 九、不收清單

> 本節目的：越界內容的路由表。發現越界：判斷歸屬 → 確認目標卡存在 → 已有則刪、沒有則搬移後刪。

| 不收 | 歸屬 | 判斷訊號 |
|------|------|---------|
| 跨角色端到端流程 | 服務藍圖／`07-scenarios/` | 看到多角色接力敘述 |
| 分權規則的判斷準則 | `04-business-logic/` | 看到「為什麼免審／送審」的論證 |
| 實體欄位定義 | `05-entities/` | 看到欄位名、型別 |
| 單一角色的操作步驟 | `13-user-stories/` | 看到「第 1 步…第 2 步…」 |
| **帳號權限矩陣、平台部署歸屬** | 實作規格層（不過紙本測試） | 看到 R/W/X 縮寫、欄位級權限、系統平台分配 |
| 介面操作 | 原型實作層 | 看到按鈕、頁面、儀表板元件 |

## 十、撰寫紀律

1. **動作歸屬只寫一次**：同一個把關動作不在兩張角色卡重複宣告主責（協作關係用「配合」語意註明）
2. **規則回引不複寫**；**禁中英夾雜**；**禁實作術語當主詞**
3. **不確定項開 OQ**（`oq-manage` mode B），不留 inline 標注
4. **檔名即標題**；**正文零迭代史**
5. **不編造**：關切點與預期阻力只收訪談與權責表來源，無則如實標注

---

## 十一、擴張準則

| 情境 | 做法 |
|------|------|
| 公司新設職位／新外部協力 | 新開卡（過 § 二判斷） |
| 組織調整（職責搬家） | 兩張卡同輪修改＋互查引用，不留同一動作雙主責 |
| 卡明顯過長（粗標：正文超過 100 行） | 先過 § 九路由表檢查越界內容並外移；歸位後仍長 → 與使用者確認是否職責過載（組織問題的訊號，非文件問題） |

## 十二、稽核維度（vault-audit 用；寫完逐項勾）

| # | 維度 | 通過標準 |
|---|------|---------|
| 1 | 結構對齊 | 七段齊備；檔名即標題；無正文來源段、無「備註」雜項段 |
| 2 | 職務範圍 | 商業三分法呈現；無 R/W/X 縮寫、無固定模組行 |
| 3 | 動作歸屬 | 職責為動作歸屬層；分權判斷準則回引 04、無複寫 |
| 4 | 邊界附理由 | 職責邊界每條有「為什麼這樣切」 |
| 5 | 近似角色區隔 | 有易混角色時概述已寫明區隔 |
| 6 | BRD 自足 | 正文零實作規格引用；每段過紙本測試 |
| 7 | 語言紀律 | 商業語言；無中英夾雜、無實作術語當主詞、無介面措辭 |
| 8 | 範圍外防護 | 範圍外段存在；細粒度權限矩陣已點名「存在但屬實作規格」 |
| 9 | 不編造 | 關切點／預期阻力有來源；無來源處如實標注 |
| 10 | 連結方向 | `source` 指權責表／拍板／04 卡；相關卡雙向可達、無斷鏈 |
