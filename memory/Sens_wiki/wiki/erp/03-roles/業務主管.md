# 業務主管

> [!success] OpenSpec spec 狀態：**已有**
> `openspec/specs/user-roles/spec.md` § Requirement: 業務主管角色職責（L572-614）已完整定義。本卡從 spec 提取（非自編）。
> 注意：業務主管 ≠ [[Supervisor]]（Supervisor 為 Dashboard / Forecast / KPI 追蹤角色，偏產能總覽）。

## 基本資料

- **平台**：中台（與印務主管 / 審稿主管 / Supervisor / 訂單管理人 / EC商品管理對稱）
- **利害關係程度**：高（「待業務主管審核 → 報價待回簽」為訂單流程關鍵推進節點，未通過則報價單無法外發）
- **參與階段**：評估（訂單建立後、報價單外發前）
- **任務操作**：X
- **報價單 / 訂單**：R/W（成交條件審核 + 訂單異動核可）
- **工單**：X
- **需求單**：R（只讀，提供部門總覽；MUST NOT 編輯內容 / 推進狀態 / 執行成交流失）

## 主要工作職責

每日進系統處理三類待辦（對齊印務主管模式）：

1. **訂單審核（成交條件審核）**：訂單建立後、報價單外發前，與業務確認收款條件、報價金額、交期合理後，將線下訂單從「待業務主管審核」推進至「報價待回簽」
2. **退款訂單異動核可（限退款，amount < 0）**：核可 / 退回退款負項訂單異動單（草稿 → 待主管審核 → 已核可 / 已退回，對應 [[訂單異動狀態]]）。**補收正項訂單異動單（amount > 0）由業務直接執行、主管不經手**——退款這個現金流出方向一旦出錯收不回來才需主管把關，補收是客戶加單不該被審核卡住交期（補收免審 vs 退款送審的不對稱分權判斷準則見 [[訂單異動規則]]）。不分是否關聯 AfterSalesTicket（訂單期間加掛與售後 ticket 內加掛的退款異動皆走相同核可流程）。大額補收（建議閾值 > 50000 NTD）採事後 Slack 監督 + CEO 收款變更率指標追蹤，不阻擋業務當下操作（避免大額加單卡審核流失生意）
3. **售後服務檢視**：中台「售後服務」頁（路由 `/sales-manager/after-sales`）唯讀檢視全公司所有 AfterSalesTicket，作為 Slack 跟催售後進度的決策入口
4. **改派負責業務**（add-sales-manager-reassign-owner change，2026-05-30）：於諮詢單 / 需求單 / 訂單詳情頁改派負責業務（改 owner）。限業務主管；改派理由分類（離職交接 / 長假代理 / 工作負荷平衡 / 客戶要求換窗口 / 其他）驅動分享成員處置與通知；改派只換人、不改單據狀態。詳見 [[XM-008-業務主管改派負責業務的權限邊界與機制|XM-008]]

## 職責邊界（關鍵）

- **核准決策範圍 = 部門內**：限「訂單指定的業務主管 = 自己」的訂單；MUST NOT 跨業務主管核准他人指定範圍
- **檢視範圍 = 全公司**：售後服務頁唯讀檢視全公司 ticket（識別逾期久未跟催 / 跨業務責任不清 / 公司認賠金額異常），但 MUST NOT 編輯非自己管轄 ticket
- **改派負責業務範圍 = 全公司**（add-sales-manager-reassign-owner change）：改派屬管理動作（非核可決策），可改派全公司任何單據的負責業務，與「核可決策範圍 = 部門內」不同層級（ERP 無部門維度、對齊售後檢視的全公司範圍）
- **不介入需求單流程**：需求確認 / 成本評估 / 議價 / 成交 / 流失皆由業務獨立執行（改派負責業務屬管理動作、不在此限）
- **不參與 AfterSalesTicket 系統流程**：無 ticket 核可關卡；售後決策影響力透過「ticket 內加掛 OA 的金額核可」+「Slack 線下討論」

## 關切點

- 成交條件審核（收款條件 / 報價金額 / 交期）有系統支援，不靠口頭 / Slack
- OA 金額核可有留存記錄（approved_by / approved_at）
- 全公司售後可視性（跨部門協作識別異常）

## 預期阻力

審核流程從口頭 / Slack 改為系統操作，初期可能因操作不熟悉而增加確認時間（對齊印務主管導入阻力）
