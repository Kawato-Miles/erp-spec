# US-ORD-011 業務取消訂單與退款申請

> **重寫紀錄（2026-05-22）**：原卡涵蓋「業務取消 + 系統 top-down 連鎖 + 業務退款申請 + 會計執行退款 + 已退款」跨多角色多動作，違反「user story 單角色單動作」紀律。本卡重新定位為**業務動作**（取消 + 退款申請）；會計執行退款部分拆出為 [[US-ORD-013-會計執行退款處理]]；端到端跨角色串接由 07-scenarios 情境 16 處理。

## 業務情境

### 作為
[[業務]]

### 我希望
能取消訂單並對已收款訂單提退款申請

### 以便
正式終止訂單並啟動退款流程，下游工單 / 任務 / 生產任務由系統連鎖終止避免製作浪費

### 前置條件
- 訂單存在（任意狀態均可取消）
- 業務為訂單擁有者或職務代理人

### 業務流程

1. 業務於訂單詳情執行「取消訂單」
2. 訂單狀態轉「已取消」終態（不可逆）；系統自動 top-down 連鎖終止下游（屬系統行為，業務無需手動處理）：
   - 所屬工單全數轉「已取消」
   - 任務全數轉「已作廢」
   - 生產任務依生產進度分流：尚未進入生產 → 「已作廢」/ 已進入生產 → 「報廢」（成本依已報工數量計算）
3. 業務判斷該訂單是否需退款（依收款淨額是否大於應收決定，對帳面板顯示「應退差額」警示）
4. 若需退款，業務於訂單異動 Tab 建立退款訂單異動（負額）送業務主管核可；核可後該訂單異動即生效（已執行＝核可後應收調整生效，不需等退款款項）
5. 業務建立退款付款紀錄（負額）、上傳匯款證明並切「已完成」以核銷應退差額；退款付款紀錄切「已完成」即為退款完成錨點，對帳面板應退差額歸零為結果呈現。跨月稅務折讓（銷貨折讓單關聯退款付款紀錄）為會計後續動作，詳見 [[US-ORD-013-會計執行退款處理]]

### 成功條件

1. 業務可於任意狀態執行「取消訂單」，訂單狀態轉「已取消」終態不可逆
2. 系統自動 top-down 連鎖終止下游工單 / 任務 / 生產任務（業務無需手動處理）
3. 已進入生產的生產任務轉「報廢」（成本依已報工數量計算）；尚未進入的轉「已作廢」
4. 業務可對收款淨額大於應收的訂單建立退款訂單異動（負額，經業務主管核可後生效），並以退款付款紀錄（負額、上傳匯款證明後切已完成）核銷對帳面板的應退差額；退款訂單異動建立與退款付款紀錄寫入活動紀錄供事後追蹤
5. 取消動作與下游連鎖終止清單寫入活動紀錄供事後稽核
6. 退款付款紀錄切「已完成」即視為退款完成、對帳面板應退差額歸零；應退差額警示不提供「忽略此差額」選項，缺口只能靠實際退款消除
