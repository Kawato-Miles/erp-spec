## 現況

工單資料分散在 Ragic 與紙本，異動不回寫系統。師傅依紙本工單生產，中途規格變更時容易照舊版執行。

## 規則

印刷業 ERP 的生產單據遵循四層結構，上層建立後才產生下層：

```
訂單（含多個印件）→ 工單（含多個任務）→ 任務 → 生產任務
```

- 一張訂單可拆成多張工單（不同工廠、不同工序）
- 一張工單僅對應一位印務負責
- 一張訂單的多張工單可分派給不同印務

工單建立方式依訂單來源不同：

- B2B 線下單：系統建空工單草稿，印務主管指派印務後，印務手動規劃生產任務
- B2C 線上單：系統依商品主檔工序定義自動建工單並帶入生產任務

## 為什麼這樣定

印刷品常由多道工序組合而成，每道工序可能在不同工廠執行。四層結構讓系統能追蹤到每一道工序的進度，解決「工單散落、無法追蹤」的核心問題。

B2B 全客製訂單的工序組合每單不同，必須由印務手動規劃。B2C 規格品的工序固定，可自動帶入。

## 領域知識

- 印務主管決定工單怎麼拆（哪些工序、派給哪家工廠）
- 印務收到指派後，規劃工單內每道生產任務的細節
- 生管負責任務的排程派工、與機台師傅溝通
- 師傅執行生產並報工
- [[廠務]] 負責「流動」：派送工單給各產線（品檢也是產線）、段序間運送印件、管理待處理區、接收外廠回件——產線與品檢只管做和驗，東西在哪由廠務負責

### 三種印件完成路徑（現場實況）

| 路徑 | 過程 |
|------|------|
| 完全廠內 | 各段序完成（產線回報＋照片）→ 廠務送下一站 → 一路到品檢（終站）→ 出貨 |
| 廠內＋外發 | 廠內印製完成 → 廠務放**待處理區**並通知印務 → 印務負責外發作業 → 回廠（印務自取放回待處理區，或外廠配送、印務提前一天告知、廠務收件拍照）→ **回廠後品檢點數量＋初步確認狀態** → 廠務送下一段 → 最終品檢 → 出貨 |
| 完全外發 | 外發完成 → 回廠 → 品檢 → 出貨（無廠內製作段） |

- 外發＝一筆生產任務：外發出去的那道工序就是一筆生產任務，它引用的 BOM 掛外包廠（或中國廠），透過任務層按工廠分組被切開；外包／中國廠是整個任務交付出去、等對方做完回報，不經生管派工（生管只派廠內）
- 待處理區＝外發前與回廠後的實體暫存區，由廠務管理（是否成為系統概念待評估，見 [[PT-008-外發回廠流轉的系統設計範圍|PT-008]]）
- 場外回場內的系統表達（用轉交單還是生產任務運送中狀態）待設計評估，見 [[PT-008-外發回廠流轉的系統設計範圍|PT-008]]
- 品檢有兩種：每張印件入庫前必有一次**最終品檢**；生產過程特定斷點的**中間品檢**（外包回廠成品要組合做下一個工作時、或公司認為該工序良率要特別注意），由印務建立工單時規劃。回廠的「數量清點」與「品質品檢」現場為兩個動作，系統是否各記一筆待評估（見 [[PT-008-外發回廠流轉的系統設計範圍|PT-008]]）
- 生產數量可含緩衝（如加印 10%），由印務決定，但不可低於印件所需數量
- 品檢通過的數量才算入庫，入庫數量決定可出貨上限
- 出貨支援分批，累計出貨超過目標數量時系統警告但不阻擋（印刷業因良率緩衝，實際可出貨數量可能超過原需求）

## 具體例子

例子 1：客戶訂 1000 本型錄。印務主管拆成 2 張工單：工單 A 委外印刷（工廠甲）、工單 B 場內裝訂。印務甲負責工單 A，印務乙負責工單 B。各自規劃生產任務、追蹤進度。

例子 2：B2C 客戶訂購 500 張標準名片（EC 規格品）。系統依商品主檔自動建立工單與生產任務，印務不需手動規劃。
