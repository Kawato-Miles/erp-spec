# 審稿分配規則（業務規則）

> 本卡記載**業務規則**：候選集篩選原則、優先順序、破例派工觸發條件。
>
> **不記載**：演算法的逐步實作細節（屬實作層，留 `prepressReview.ts` 程式碼）。

## 一、自動分配觸發點

```
訂單付款 → hook 觸發
├── 印件 skipReview = true → 走 [[免審決策樹]]
└── 印件 skipReview = false → 觸發自動分配
```

## 二、候選集篩選原則

審稿人員須同時滿足以下三條件才進入**合格候選集**：

| 條件 | 業務含義 |
|------|---------|
| `active = true` | 啟用中（未停用） |
| `availableStatus = '在崗'` | 在崗（非休假 / 離職） |
| `maxDifficultyLevel ≥ printItem.difficulty` | 能力範圍涵蓋此印件難易度 |

→ 難易度匹配邏輯見 [[難易度機制]]。

## 三、合格候選集選擇優先序

當合格候選集非空時，選擇優先序：

1. **能力範圍最貼近**：選 `maxDifficultyLevel` 最小且 ≥ 印件難易度的人（避免高能力人員處理簡單案）
2. **同能力者按負載**：若多人 `maxDifficultyLevel` 相同，選**目前負載最少**的
3. **負載相同按 ID 穩定排序**：確保結果可重現

## 四、破例派工觸發條件

當合格候選集為**空**時（沒有審稿人員能力 ≥ 印件難易度）：

- 啟動「破例派工」邏輯
- 從**所有 `active` 且在崗**的審稿人員中選 `maxDifficultyLevel` **最高**者
- 記錄 note：「被指派者能力 X < 印件難易度 Y」（供[[審稿主管]]覆寫決策）

## 五、主管覆寫

[[審稿主管]]可主動覆寫自動分配結果：

- 覆寫事件記入 ActivityLog
- 用於計算「自動分配命中率」（1 - 覆寫率）

**覆寫能力門檻（resolve-prepress-review-gaps-ar-10-ar-12 拍板）**：

| 項目 | 規則 |
|------|------|
| 候選清單預先過濾 | 能力不足者（`maxDifficultyLevel < printItem.difficulty`）+ 離職停用者 SHALL **不出現於主管覆寫的候選清單**（UI 層阻擋，比 spec 既有「事後拒絕」更嚴格）|
| 可選清單為空時 | UI SHALL 顯示 actionable 提示「無可選審稿人員，請先調整審稿員能力等級或恢復原審稿員」 |

**設計理由（為何主管覆寫嚴格擋，自動派工卻允許破例）**：
- 自動派工的「破例派工」（見 § 四）是**系統不得已的降級** — 無人能力夠時保證印件不卡住，留 ActivityLog 給主管事後監看做人力決策
- 主管手動覆寫是**主動行為**，不應比自動派工更寬鬆 — 避免主管隨意覆寫造成審稿品質問題
- 兩條 path 規則不同是因動機不同（系統降級 vs 主管主動），不是設計矛盾

**Prototype 既有狀態**：`overrideAssignment` 採事後拒絕（`throw new Error('目標審稿人員能力不足')`），待 Prototype 後續改為 UI 層阻擋（覆寫 Dialog 候選清單預先過濾）。

## 六、自動分配命中率（KPI）

| 指標 | 計算 |
|------|------|
| 命中率 | (總自動分配次數 - 被覆寫次數) / 總自動分配次數 |
| 目標 | 越高越好（代表自動分配的業務邏輯接近實際決策）|

→ 此 KPI 應納入 `01-products/kpi/`（Phase B Stage 5 補）。

## 八、退回重審指派規則

業務退回重審後，印件重新進入審稿流程時，預設指派原審稿人員（該印件最近一輪合格 ReviewRound 的 `reviewer_id`），不重新執行自動分配演算法。

- 原審稿人員在崗 → 印件直接加入其待審列表，待審清單標示「重審」
- 原審稿人員不在崗 → 印件出現在[[審稿主管]]覆寫待辦清單，由主管轉指派

**設計理由**：與既有「補件回原審稿人員」邏輯一致（見 § 五 主管覆寫）。原審稿人員對同一印件最熟悉，重新分配降低重審效率。
