## 現況

過去工廠靠紙本工單，師傅做完在紙上註記、進度全靠人工回報，主管看不到即時狀況、產能不透明。這次 ERP 的核心目標之一就是把報工納入系統：誰做、做了多少、附現場照片，做完就回報——系統據此即時算出成品完成數量、推進工單進度、支援分批出貨。本卡定義誰能報工、報什麼、報工怎麼推進狀態。報工是現場回報體系（師傅報工／廠務轉交／品檢／揀貨裝箱／出貨）的其中一種，體系全貌見 [[生產流程]] § 現場回報體系。

## 規則與營運動機

### 報工的五個管道

| 項目 | 內容 |
|------|------|
| 規則 | 同一筆報工依提交來源記錄管道，共五種：師傅自助（任務平台）、供應商自助（供應商平台）、生管代報（日程面板）、印務於工單詳情頁、印務於印件詳情頁。管道由系統依來源自動記錄，使用者不可手選或修改 |
| 營運動機 | 紙本時代報工集中在少數人轉抄，瓶頸又失真；開放各角色從自己的工作介面直接報，並記錄來源管道供日後分析各管道採用率、稽核溯源 |

### 報工的內容與照片

| 項目 | 內容 |
|------|------|
| 規則 | 一筆報工記錄實際產出三欄——**良品數＋不良品數＋不良原因**（固定選項，與品檢端原因架構共用邏輯，細節待 [[QC-003-QC不通過原因固定選項|QC-003]]／[[QC-004-報工不良與品檢缺口取數口徑|QC-004]]）——與提交者（2026-07-22 拍板）；現場狀態回報附照片（段序完成照、成品照、外廠收件照；裝箱照歸出貨單揀貨回報，見 [[出貨單]]）。品檢不是報工——印件最終品檢的通過數量記在 [[品檢紀錄]] |
| 營運動機 | 記實際產出而非只記完成數：不良品數與原因是放損率調整（按工序×原因統計）的資料來源。照片是現場的驗證語言——SOP 裡每個段序完成、成品、收件都要附照，讓上游與品檢「看得到」而非只看數字；取代紙本工單的口頭與手寫佐證 |

### 首次報工觸發製作中

| 項目 | 內容 |
|------|------|
| 規則 | 生產任務在「待處理」收到第一筆報工，狀態自動轉「製作中」並向上帶動任務／工單／印件／訂單同步反映（見 [[生產任務狀態]]）。指派師傅只是欄位更新，不觸發狀態 |
| 營運動機 | 以「開始做（報第一筆工）」為動工事實的觸發點，不是以「被指派」——排程定了不等於動工了，狀態忠於現場 |

### 報工數量上限（超過警示、不阻擋）

| 項目 | 內容 |
|------|------|
| 規則 | 同一生產任務的報工累計超過該任務的目標數量（含放損緩衝由印務於規劃時定，見 [[印件生產流程]]）時，系統**警示但不阻擋**（2026-07-22 拍板，由硬性阻擋改為警示）；累計達標即視為該任務完成 |
| 營運動機 | 報工要忠於實際產出——硬擋會逼現場少報、失真；保留真實產出資料，才能按工序×原因統計做放損率調整。超報以警示提醒查核，不以阻擋犧牲資料真實性 |

### 報工權限守門

| 項目 | 內容 |
|------|------|
| 規則 | 各管道只能報自己範圍的任務：師傅只見自己被指派的任務、供應商只見分派給該廠商的任務、印務（含印務主管）只能報自己負責工單下的生產任務。守門在介面與系統兩層都檢查，繞過介面直接呼叫一樣擋下並寫稽核日誌 |
| 營運動機 | 報工直接動完成數量與狀態，錯報影響齊套與出貨；把報工權綁在工作歸屬上，責任清楚、防誤報 |

## 具體例子

例子 1（師傅首次報工）：師傅在任務平台對「待處理」的印刷任務報第一筆工 500 張並附段序完成照，生產任務轉「製作中」，所屬任務、工單同步反映製作中。

例子 2（供應商標記完畢觸發在途）：外包廠在供應商平台把製作中的裱貼任務標記製作完畢，狀態轉「運送中」（外包廠路徑）；中國廠商同動作轉「已送集運商」。

例子 3（印務代報守門）：印務甲在印件詳情頁勾選工單 #4（印務乙負責）的生產任務報工，按鈕為禁用；即使繞過介面呼叫，系統在後端擋下並記稽核日誌。

## 邊界

- 報工數量怎麼往上彙整成完成數量、怎麼算做齊 → [[齊套邏輯]]（規則正本）
- 品檢的通過數量判定與不符合處置 → 屬品檢規則（正本見 [[品檢紀錄]]；品檢不是報工）
- 報工是否開放行動裝置（師傅在機台旁手機報工）→ [[PT-001-師傅報工行動裝置例外|PT-001]]（現行政策桌機限定，待拍）
- 報工介面的欄位與操作細節 → 屬實作規格
