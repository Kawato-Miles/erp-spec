## 現況

印刷廠接單後金額常臨時變動：客戶在製作中追加印量、改急件、補運費，公司多收錢（稱補收）；或因品質、規格不符要退費，公司退錢出去（稱退款）。這兩個方向的營運風險不對稱——錢退出去一旦出錯難追回，客戶加單則是送上門的生意。所以公司的分權原則是：[[業務主管]] 把關「退款」這個現金流出方向，「補收」這個客戶加單方向由 [[業務]] 自己拍板、不必等主管，避免客戶等審核而流失加單機會。

另有一種特例：諮詢取消要退客戶一半費用（退 1000 元），這是公司既定規則、系統自動生成這筆退款異動單，不需主管再判斷該不該退。

退款的認列與退款的物理動作分成兩件事：主管一核可這張退款異動單，它當下就生效（記為確認可執行、應收當下減去這筆金額），不必等實際退錢；之後業務真的把錢匯出去、那筆退款款項標成「已完成」並掛上匯款證明，是在對帳階段把「應該退的差額」核銷歸零（見 [[對帳一致性]]），不再回頭改異動單的狀態。退款的金流記在收款款項那一條（退款款項記負數），不掛在異動單上。

主管讀本卡要能驗證的是：這套「補收免審、退款送審」的分權，以及「主管核可即生效、實際退錢另在對帳核銷」的認列規則，是否符合公司實際的現金管控需求。

## 規則與營運動機

> 十二條規則，依「分權（補收免審／退款送審／諮詢取消退費系統建已核可）→ 認列（退款確認可執行認列／應收總額認列公式）→ 編輯與建單前提（金額編輯閘門／零金額不建單）→ 動機與來源標記（補收退款不對稱／來源售後服務單標記）→ 事後監督（大額補收事後監督）→ 完成後與完成前善後路徑（訂單完成後金額異動路徑分流／款項紀錄直接建退款）」排列。每條規則的四要素（觸發條件 / 審核路徑 / 終態 / 一定要成立的規則）見「具體例子」與「不變條件」段。

### 補收免審

| 項目 | 內容 |
|------|------|
| 意圖 | 客戶主動加錢不被主管流程卡住，保住加單生意與出貨時程 |
| 規則 | 補收這種加錢（正的）異動單（金額是加錢，且類型為加印追加／加運費／急件費／補退正項／規格變更正項）：免主管審核，從「草稿」直接推進到「確認可執行」（略過待主管審核與已核可）；核可人記為業務本人、執行時間記為當下；建單當下，這筆要收的錢馬上加進應收總額，而且不用去對任何一筆退款 |
| 營運動機 | 客戶臨時加印是送上門的生意，等主管核可會拖延出貨、可能流失加單；加錢方向風險低（公司是收錢方），授權業務當場拍板。服從 [[現金流出把關]]——把關的鬆緊跟著錢的流向走（進來的可鬆） |

四句話看懂：什麼時候會用到＝客戶加錢時；要不要主管審＝不用；走到最後是什麼狀態＝確認可執行（錢已處理）；要守的規則＝補收這種加錢的不走退款、不對應退差額（見「退款確認可執行認列」的一定要成立的規則）。

### 退款送審

| 項目 | 內容 |
|------|------|
| 意圖 | 退錢出去前先由主管把關金額與理由；主管一核可這筆退款就生效認列，實際退錢另在對帳階段核銷 |
| 規則 | 退款（負的）異動單（金額是退錢，業務發起）：走路徑——草稿 → 待主管審核 → 已核可 → 確認可執行；主管一核可，這張單當下就推進到「確認可執行」、記下執行時間、應收當下減去這筆退款金額，不必等實際退錢。之後業務真的把錢匯出去、那筆退款款項標成「已完成」並掛上匯款證明，是在對帳階段把「應該退的差額」核銷歸零（見 [[對帳一致性]]），不再回頭改這張異動單的狀態 |
| 營運動機 | 錢退出去出錯難追回，由業務主管把關金額與理由是否合理（把關擺在核可這一關）；核可即生效讓應收即時反映該退的金額，避免「帳上還沒認列、客戶已在催退」的落差。實際退沒退改由對帳的「應退差額」盯住（退款款項標已完成才核銷歸零），把「該退」與「退好沒」分兩層各自把關。服從 [[現金流出把關]]——出去的要嚴（嚴在主管核可這一關）|

四句話看懂：什麼時候會用到＝品質／規格退費；要不要主管審＝要，業務主管核可；走到最後是什麼狀態＝確認可執行（主管一核可就到、應收當下減去這筆）；要守的規則＝退款確認可執行＝主管已核可（與實際退錢脫鉤）；實際退錢由對帳的「應退差額」盯住，退款款項標已完成才核銷歸零。

### 諮詢取消退費系統建已核可

| 項目 | 內容 |
|------|------|
| 意圖 | 諮詢取消退一半是既定規則、不需人工判斷審核，但金額確認仍由諮詢人員把關 |
| 規則 | 諮詢取消退費異動單（系統自己建的、不是業務手動開，金額固定 −1000）：「負向異動須審核」的規則不破例——因退費金額固定，**審核由系統代行**，建單即自動推進「已核可」（核可人掛在系統名下、執行時間先空著）。已核可階段諮詢人員可調整退款金額（沿用「金額編輯閘門」），也可取消這張單（決定不退了）。諮詢人員**確認金額**後，單據推進「確認可執行」——**應收於此刻認列減去**；之後的開立折讓（或作廢重開發票）與實際退款，由對帳把發票淨額、收款淨額對回應收淨額。此類型為系統專用，業務不可手動建立 |
| 營運動機 | 退一半是公司既定規則、不需主管再判斷（審核由系統代行，主管的退款審核佇列只留線下單）；建為已核可（而非直接確認可執行）讓諮詢人員在認列前仍能調整金額或決定不退；認列（確認可執行）與物理退款分兩層，退好沒由對帳應退差額盯住 |

四句話看懂：什麼時候會用到＝諮詢取消（系統自己建）；要不要主管審＝要審但由系統代行（建單即已核可）；走到最後是什麼狀態＝確認可執行（諮詢人員確認金額後到、應收於此刻減去；確認前可調金額或取消不退）；退好沒由對帳應退差額盯住。

### 退款確認可執行認列

| 項目 | 內容 |
|------|------|
| 意圖 | 退款的「認列」跟著主管核可走，實際退錢另由對帳盯住，兩層分開 |
| 規則 | 退款（負的）：主管一核可，系統當下就把異動單推進到確認可執行、記下執行時間（執行時間＝核可時點）、應收當下減去這筆金額，不再等實際退錢、不綁退款款項累計。諮詢取消退費：系統代審建為已核可後，由諮詢人員確認金額推進確認可執行、應收於確認時減去，見「諮詢取消退費系統建已核可」。實際退錢是另一層：業務把錢匯出去、退款款項標成「已完成」並掛上匯款證明，把對帳的「應退差額」核銷歸零（見 [[對帳一致性]]），不回頭改這張異動單。退款款項的金流記在收款款項那一條（記負數）、不掛在異動單上，也不去對任何一張請款期次。若退款款項建錯：取消它，對帳的「應退差額」會重新出現、由差額提醒引導業務重退，異動單維持確認可執行、應收不回退 |
| 營運動機 | 把「該退多少」（主管核可、認列進應收）與「退好沒」（退款款項已完成、對帳核銷）分兩層各自把關：核可即認列讓應收即時反映該退金額，不卡在退款動作上；退沒退改由對帳應退差額盯住，比用單據狀態硬扣更貼合實務（退款常分次、可能建錯重退）。脫鉤前舊設計用「退款款項累計達金額才算確認可執行、取消款項回退已核可」綁死兩層，遇分次退款與建錯重退會把異動單狀態反覆拉扯，已於訂單收退款模型重構 改為本脫鉤版（取消退款款項是否回退異動單，此點已決議，見 [[ORD-003-取消退款Payment是否回退OA\|取消退款記錄是否回退異動單]]）|

一定要成立的規則（系統一定守住）：退款異動單為「確認可執行」，代表已經有權者確認（一般退款＝業務主管核可；諮詢取消＝系統代審後諮詢人員確認金額）、應收已減去這筆金額；至於錢退好沒，由對帳的「應退差額」盯住——退款款項標成「已完成」且帳能對平，應退差額才歸零（補收這種加錢的不走退款、也不對應退差額）。

### 應收總額認列公式

| 項目 | 內容 |
|------|------|
| 意圖 | 應收總額即時反映金額增減，但不替業務決定分幾期收 |
| 規則 | 應收總額 ＝ 所有印件費加總 ＋ 所有其他費用加總 ＋ 所有「確認可執行」異動單金額加總（印件費＝每個印件下單數量 × 報價單價，逐件加總）。進入「確認可執行」的時點依路徑：補收於業務建單生效時、一般退款於主管核可時、諮詢取消退費於諮詢人員確認金額時——三條路徑統一在此認列，無特例。異動單推進到確認可執行時觸發應收總額重算，但系統不自動建立或修改 [[分期請款狀態\|請款期次]] |
| 營運動機 | 應收要即時反映真實欠款／應退，但「分幾期收、何時開票」是業務依客戶狀況另行規劃，系統自動動期次反而打亂業務既有安排。本公式須服從對帳底線（應收 ＝ 發票淨額 ＝ 收款淨額，正本見 [[對帳一致性]] 共用規則卡）|

### 金額編輯閘門

| 項目 | 內容 |
|------|------|
| 意圖 | 退款做出去之前，讓業務還有機會最後修一下金額，但已經生效的就不能再亂改 |
| 規則 | 草稿：可改，改後重新送審。待主管審核：不可改。已退回：可改，改後重新送審。已核可：可改且不需重新送審（退款前最後修，卡片同時顯示「核可的金額」跟「目前金額」兩個對照）；諮詢取消退費單於已核可亦可取消（決定不退了）。確認可執行：鎖住只能看不能改（不會出現修改的地方）。已取消：鎖定。每次改金額寫一筆稽核記錄，必含：異動時間／操作者／改前金額／改後金額／異動當時狀態 |
| 營運動機 | 已核可後現實常變（客戶談妥的退款數字微調），允許不重審以免流程空轉（此窗口主要落在諮詢取消這種系統建已核可、尚未確認執行的單上；一般退款由主管核可當下即生效確認可執行）；確認可執行代表應收已認列，再改會破壞對帳，故鎖定。已核可可改是否需通知主管見 [[AFT-3-OA已核可改金額是否通知主管\|異動單已核可改金額是否通知主管]]、是否設金額門檻重審見 [[ORD-002-OA核可金額閾值閘門\|異動單核可金額門檻閘門]] |

### 零金額不建單

| 項目 | 內容 |
|------|------|
| 意圖 | 沒有金額增減的規格調整不該佔用一張金額異動單 |
| 規則 | 純規格變更若金額 ＝ 0：不建立異動單。製作後的印件規格異動（非金額類，如規格備註／印件下單數量／單位／難易度）由 [[業務]] 直接編輯，系統自動通知相關 [[印務]]／[[印務主管]] 並寫活動紀錄，不走異動單 |
| 營運動機 | 異動單是金額載具，規格調整無金額時走通知＋活動紀錄即可，避免無金額的空單據混淆對帳 |

### 補收退款不對稱

| 項目 | 內容 |
|------|------|
| 意圖 | 明確記錄「為什麼補收免審、退款送審」這條對稱破壞的營運根據，供主管驗證前提是否成立 |
| 規則 | 補收（補收免審）與退款（退款送審）採不對稱分權：主管把關現金流出方向（退款風險高、出錯難追回），不把關客戶下單追加方向（補收是送上門生意，等審核會拖延出貨、流失加單）。對齊台灣印刷業實務分權 |
| 營運動機 | 對稱設計（兩方向都送審或都免審）會在「擋不該擋的加單」與「放掉該管的退款」之間二擇一犯錯；不對稱分權讓風險與管控強度匹配。這正是 [[現金流出把關]] 的展開——把關鬆緊跟著錢的流向走，不跟著金額有沒有變動走。這條一邊免審一邊送審的不對稱，怎麼講才清楚見 [[BI-9-補收OA立即執行對稱破壞表述\|補收異動單立即執行的不對稱怎麼表述]] |

### 來源售後服務單標記

| 項目 | 內容 |
|------|------|
| 意圖 | 訂單完成後的退款沿用同一套異動流程，僅以一個標記區分它是從哪張售後服務單來的 |
| 規則 | 異動單上可以標一個記號，註明它是從哪一張售後服務單來的：沒標記號＝訂單未完成前 [[業務]] 在訂單內直接建立；已標記號＝訂單完成後從 [[售後服務]] 單內建立的退款負項異動單（售後單內可同時放補印印件）。標記與否走同一套退款審核規則（退款送審／金額編輯閘門一致），這個記號一經標上去就不能再改 |
| 營運動機 | 訂單完成後的客訴退費沿用同一套退款異動流程，不另開例外路徑，主管審核與認列邏輯一致、降低維護負擔；標記只供反查來源，不改變規則本身。注意完成後「向客戶加錢」不走這條：完成後補收要另開一張新訂單收費（見「訂單完成後金額異動路徑分流」），不在售後單內建補收異動單 |

### 大額補收事後監督

| 項目 | 內容 |
|------|------|
| 意圖 | 補收免審後，主管仍能在事後看到異常大額的補收，但不重新成為流程瓶頸 |
| 規則 | 補收金額超過大額門檻（建議起始 50000 元）：寫一筆活動紀錄並標紅，同時在公司通訊軟體（Slack）上發通知給 [[業務主管]]。屬事後監督，不阻擋業務操作（補收仍免審直達確認可執行） |
| 營運動機 | 免審換到了出貨速度，但完全不可見會讓主管失去對大額現金流入異常（如誤填、舞弊）的察覺；事後紅標＋通知補上可見性，又不退回審核流程。大額門檻的最終取值見 [[BI-4-補收OA大額閾值定義\|補收異動單的大額金額門檻]] |

### 訂單完成後金額異動路徑分流

| 項目 | 內容 |
|------|------|
| 意圖 | 訂單一旦完成（進入終態），金額就不在訂單內直接改，改走哪條善後路徑要分清楚「退錢」與「加錢」 |
| 規則 | 訂單進入「訂單完成」終態後，明細金額鎖定（見 [[明細時點分界]]），之後的金額異動依方向分流：（一）要退錢給客戶、或要替客戶補印一批貨 → 開一張 [[售後服務]] 單，售後單內放退款負項異動單（走退款送審）與補印印件；（二）要向客戶加收錢（完成後補收）→ 另開一張新訂單收費，不在售後單內、也不在原訂單內建補收異動單 |
| 營運動機 | 退錢與補印是針對原單瑕疵的善後，歸到售後服務單集中受理、留客訴脈絡；向客戶加收錢則是一筆新生意（性質同重新接單），開新訂單才能獨立報價、開票、收款，不會把已結算的原單帳目又拉開。對齊「訂單完成後客訴退費沿用既有流程、不另開例外路徑」與「完成後補收幾乎不加、真加另循新訂單」的實務（見 [[ORD-030-完成前明細直接增減真實業務流程待訪談\|ORD-030 業務訪談]] 第 d 點）|

### 款項紀錄直接建退款

| 項目 | 內容 |
|------|------|
| 意圖 | 完成前因調降而要把已多收的錢退回去，讓業務直接在款項紀錄退、不必再開一張退款異動單送主管核可 |
| 規則 | 訂單完成前（未進入終態）若因明細調降（少收）導致已收的錢超過應收，業務可直接在款項紀錄建一筆退款款項（記負數、掛匯款證明），不需開退款異動單、不需業務主管核可。此路徑只用於「完成前減量退款」這個軟兜底；訂單完成後的客訴退款仍走 [[售後服務]] 單內的退款異動單（退款送審） |
| 營運動機 | 完成前的調降本身就是業務當場可決定的事（弱把關、事後可見，見 [[明細時點分界]]），調降後把多收的錢退回去屬同一筆操作的收尾，再要求開異動單送主管核可會把當場能結的事拖成兩段流程。把關擺在「調降」這個源頭（主管事後看得到），退錢動作不再重複送審。推翻 2026-05-07「退款必走訂單異動單」的舊規定——該規定未區分「完成前減量收尾」與「完成後客訴退款」，把前者一併綁進送審 |

### 異動單只管金額、售後事件走售後服務單

| 項目 | 內容 |
|------|------|
| 意圖 | 異動單回歸「純金額異動載具」定位，不再兼任售後容器 |
| 規則 | 異動單只處理應收金額的增減（補收與退款），所有 adjustment_type（規格變更／加印追加／退印／折扣／加運費／急件費／補退／其他）皆可於任何訂單狀態下選用，不依訂單狀態推算適用範圍。訂單完成後的客訴、不良、規格不符等售後事件改建 [[售後服務]] 單，售後單內部視決議再建關聯異動單（退款）或補印印件 |
| 營運動機 | 舊設計讓異動單同時承擔「金額異動」與「售後受理」雙重身份，造成業務搞不清楚什麼時候該開異動單、什麼時候該開售後單。拆開後：異動單只管錢、售後單只管事件追蹤與決議，分工清楚 |

### 金額有無變動決定走哪條流程

| 項目 | 內容 |
|------|------|
| 意圖 | 業務遇到變更需求時，一個判斷就能選對流程：「有沒有向客戶補收或退費」|
| 規則 | 訂單期間有金額變動（向客戶補收或退費）→ 走異動單（[[訂單異動狀態]]）。訂單已完成後的售後事件 → 走 [[售後服務]] 單（售後單內依決議建關聯異動單或補印印件）。無金額變動（含成本上升公司吸收）→ 走工單異動流程（工單層面處理，不建立金額為零的異動單，原因見「零金額不建單」）。「公司吸收成本」在訂單已完成後仍走售後服務單（補印、公司認賠），不建異動單，售後單提供事件追蹤與品質成本切片分析的價值 |
| 營運動機 | 金額異動以外的變更（換紙、調工序、補印公司吸收）本質是生產端的調度，讓它們留在工單流程處理，不把異動單系統塞滿金額為零的空單據。「金額有無變動」是最直覺的分界線——業務當場就能判斷，不必先去查訂單狀態。服從「零金額不建單」規則 |

例子 1（補收免審，正例）：訂單 ORD-2026-0512 製作中，客戶臨時把名片從 500 張加到 800 張。業務建補收異動單 +6000 元（類型：加印追加），儲存後該單直接成為確認可執行，核可人記為業務本人，訂單應收當場 +6000。整批貨照常生產，不必停下來等主管，也不綁任何收款款項（收款另循正常流程）。

例子 2（退款送審，正例）：訂單 ORD-2026-0488 因色差客訴，業務建退款異動單 −5000 元送審，業務主管一核可，這張單當下就是確認可執行、執行時間記為核可時點、訂單應收當場減去 5000；此時對帳出現「收款淨額比應收多 5000」的應退差額。業務之後實際匯款並建退款款項 −5000 元、上傳匯款證明、標成「已完成」，對帳的應退差額歸零——退款款項已完成是退款做完的物理錨點，不回頭改異動單狀態。

例子 3（退款送審，分次退款對帳未平，邊界反例）：退款異動單 −10000 元主管核可後即為確認可執行、應收減去 10000、對帳出現應退差額 10000。業務先退第一筆 5000 元（退款款項 −5000 標成已完成），對帳的應退差額剩 5000、還沒歸零——錢只退了一半，差額提醒會繼續盯著這 5000，但異動單一直維持確認可執行（認列在核可那一刻就定了，不隨退款進度跳動）。

例子 4（退款確認可執行認列，建錯重退，邊界反例）：訂單 ORD-2026-0488 的退款異動單 −5000 元已是確認可執行、應收已減 5000。業務發現退款款項建錯、取消那筆已完成退款款項，對帳的應退差額重新出現 5000、由差額提醒引導業務重建一筆正確的退款款項；異動單維持確認可執行、應收不回退（該退的金額在核可時就認列定了，建錯的是退款動作、不是該不該退）。

例子 5（諮詢取消退費系統建已核可，正例）：客戶諮詢後取消，諮詢費原收 2000 元，依規則退一半。系統自動建諮詢取消退費異動單 −1000 元、審核由系統代行直接落「已核可」（核可人記為系統）。此時應收仍 ＝ 2000（退費尚未確認、不認列）。諮詢人員與客戶確認金額後，將單據推進「確認可執行」——應收於此刻 ＝ 2000 − 1000 ＝ 1000、對帳出現應退差額 1000；後續開立折讓（或作廢重開發票）並實際退款 1000 元、退款款項標成已完成，對帳的應退差額歸零，發票淨額、收款淨額與應收淨額三方對平（異動單維持確認可執行不變）。

例子 6（來源售後服務單標記，正例）：訂單 ORD-2026-0470 已完成後客戶反映外觀瑕疵，業務在售後服務單內建退款異動單 −3000 元，系統自動寫入來源售後服務單標記。這張異動單走「草稿 → 待主管審核 → 確認可執行」（與訂單期間退款同一套退款送審規則），主管在異動單層級核可金額、核可當下即推進確認可執行、應收減 3000；實際退錢由對帳應退差額盯住，退款款項標已完成才歸零。

例子 7（大額補收事後監督，正例）：訂單 ORD-2026-0531 客戶追加大量印量，業務建補收異動單 +80000 元（類型：加印追加）。金額超過 50000 門檻，系統照補收免審直達確認可執行（不卡業務），同時寫一筆紅標活動紀錄並在公司通訊軟體（Slack）通知業務主管，主管事後可檢視這筆異常大額補收。

例子 8（零金額不建單，邊界反例）：訂單製作後客戶把某印件的紙張磅數從 200 改 250（不影響售價，金額差 ＝ 0），業務直接修改該印件規格，系統推通知給工單負責印務並寫活動紀錄，全程不建任何異動單。

例子 9（訂單完成後金額異動路徑分流，正例）：訂單 ORD-2026-0470 已完成。情況甲——客戶反映其中一款外觀瑕疵要退費並重印：業務開一張售後服務單，單內建退款負項異動單 −3000 元（走退款送審）並掛一批補印印件。情況乙——客戶看了成品很滿意、當場追加一批新名片：這是新生意，業務另開一張新訂單獨立報價收費，不在原訂單或售後單內建補收異動單。

例子 10（款項紀錄直接建退款，正例）：訂單 ORD-2026-0512 製作中，原應收 50000、客戶已全額預收 50000。客戶臨時砍掉一款印件、業務直接把該印件下單數量改為 0（明細調降），應收降為 42000，主管事後在活動紀錄看得到這筆調降。多收的 8000 元，業務直接在款項紀錄建一筆退款款項 −8000、掛匯款證明、標已完成，不開退款異動單、不送主管核可；對帳的應退差額隨之歸零。

例子 11（金額有無變動決定走哪條流程，正例）：訂單 ORD-2026-0550 製作期間紙廠停產換紙，客戶接受替代紙、成本差異由公司吸收（不向客戶補收）。印務走工單異動修改紙材，不建任何異動單——因為金額沒有變動，成本變化由工單流程承擔。

例子 12（金額有無變動決定走哪條流程，邊界反例）：訂單 ORD-2026-0470 已完成後客戶反映瑕疵，公司認賠補印 100 份不收費。雖然金額為零，但因訂單已完成，業務走售後服務單（補印、公司認賠），不建異動單。售後單仍提供事件追蹤與品質成本切片分析。

## 一定要成立的規則

> 這是系統一定要守住、用來核對帳目的鐵則。業務規則卡的這些鐵則須服從本領域對帳底線（應收 ＝ 發票淨額 ＝ 收款淨額）。

1. 退款異動單為「確認可執行」 → 代表主管已核可（諮詢取消為系統建已核可並確認）、應收已減去這筆金額；不再要求背後有累計達標的退款記錄（認列與實際退錢已脫鉤）。實際退好沒由第 5 條的對帳應退差額盯住。
2. 補收只要一建單，要收的錢馬上加進應收，而且不用去對任何一筆退款。
3. 應收總額 ＝ 所有印件費加總 ＋ 所有其他費用加總 ＋ 所有「確認可執行」異動單金額加總（退款異動單於主管核可時即計入）；諮詢取消特例改認列已核可異動單。
4. 來源售後服務單記號一經標上去就不能再改。
5. 退款做完的物理錨點 ＝ 退款款項標成「已完成」且帳能對平 → 對帳的「應退差額」歸零；只要還有應退差額沒歸零，就代表還有該退而沒退好的錢（補收這種加錢的不走退款、不對應退差額）。
