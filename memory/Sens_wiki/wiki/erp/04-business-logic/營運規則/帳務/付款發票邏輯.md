## 現況

印刷業的收付款情境極為多樣（預開發票、拆票、跨月收款、頭尾款、線下補收等），目前全靠人工處理，月結時會計需大量時間比對。

## 規則

收付款涉及 13 種主要業務情境，分三大類（索引概覽，各情境展開見 [[payment-invoice-scenarios]]）：

## 一、三大分類

| 分類 | 數量 | 範圍 |
|------|------|------|
| **特殊發票開立方式** | 8 個（F1-F8）| 預開、拆多筆、有 / 無統編、指定品名、指定日期、跨月、作廢重開、折讓 |
| **特殊收款方式** | 4 個（C1-C4）| 頭尾款、申請後付款、頭尾款指定日期、線下單補收（含異動補收）|
| **訂單異動情境** | 3 個（A1-A3）| 異動不收費（公司吸收）、異動補收、異動退款 |
| **不在範疇** | 2 個 | 跨訂單混合匯款（人工）、重新報價（屬需求單議價）|

## 二、特殊發票開立方式（F1-F8）

| 編號 | 情境 | 關鍵設計 |
|------|------|---------|
| F1 | 預開發票 + 拆 2 筆 + 實際金額調整 | Payment 可不入帳任何收款項目（預收桶）；售後 OA + SalesAllowance（折讓只關聯發票）|
| F2 | 拆多筆指定金額發票 | 拆票 = 拆期（一期一票）+ PaymentAllocation 入帳 N:M |
| F3 | 拆多筆有統編 / 無統編 | Invoice.category 分 B2B / B2C |
| F4 | 指定品名 / 作廢重開 | 作廢需填原因（中文 6 字 / 英文 20 字內，藍新限制）|
| F5 | 指定日期開立 | BillingInstallment.expected_invoice_date 排程提醒；手動開立 |
| F6 | 分 2 筆不同月份 | BillingInstallment 拆 2 期、各填 expected_invoice_date |
| F7 | 作廢重開（多種子情況）| 統編 / 金額錯誤 / 已申報等 |
| F8 | 折讓單情況 | 過期發票不可作廢；SalesAllowance（折讓只關聯發票，退款 Payment 不關聯）|

## 三、特殊收款方式（C1-C4）

| 編號 | 情境 | 關鍵設計 |
|------|------|---------|
| C1 | 頭尾款 50% 指定日期 | BillingInstallment 2 期、各 due_date |
| C2 | 申請後付款（出貨後 N 天）| 先開後付，Invoice 可無 Payment 對應 |
| C3 | 頭尾款 + 指定開票日期 + 付款時間 | 兩 BillingInstallment 各自 due_date + expected_invoice_date |
| C4 | 線下單補收（含異動）| OrderAdjustment（phase=during_order）+ 業務手動加 PrintItem |

## 四、訂單異動情境（A1-A3）

> **重要**：訂單**期間** vs **已完成後** 路徑不同。完成後走 AfterSalesTicket（add-after-sales-ticket change）。

| 編號 | 情境 | 訂單期間 | 訂單已完成後 |
|------|------|---------|-------------|
| A1 | 異動不收費（公司吸收）| 工單異動，不建 OrderAdjustment | AfterSalesTicket（responsibility=公司認賠）+ 補印免費 |
| A2 | 異動補收 | OrderAdjustment（補收）+ BillingInstallment + 補收發票 | AfterSalesTicket + 關聯 OA（補退）+ 補印 PrintItem |
| A3 | 異動退款 | OrderAdjustment（退款）+ 折讓 / 作廢重開 | AfterSalesTicket + 關聯 OA（退印）+ 折讓 / 作廢 + 退款 Payment |

## 五、跨情境共用實體

| 實體 | 用途 | 對應實體卡（05-entities）|
|------|------|---------|
| BillingInstallment（收款項目 / 請款期次）| 應收規劃單一實體（合併原 PaymentPlan + PlannedInvoice 雙頭維護），雙維度狀態（開票維度 / 收款維度）| 訂單 § 付款 |
| Payment | 實際收款 / 退款紀錄（可負數） | 訂單 § 付款 |
| PaymentAllocation（收款入帳明細）| N:M 關聯 Payment ↔ BillingInstallment，業務手動勾選收款項目 + 填金額入帳（取代 PaymentInvoice junction；sum ≤ 收款金額、溢收進預收桶）| 訂單 § 付款 |
| Invoice | 發票，可 B2B / B2C；期次↔發票業務 1:1（一張發票對應唯一一個請款期次）| 訂單 § 發票 |
| SalesAllowance | 折讓單，僅關聯 Invoice（退款 Payment 不建關聯，反查走訂單活動紀錄；帳平以三軸總額比對）| 訂單 § 售後 |
| OrderAdjustment | 訂單異動單，phase=during_order / after_completion | 訂單 § 異動 |
| AfterSalesTicket | 售後容器（add-after-sales-ticket change）| 售後服務 |

## 五B、七實體連帶矩陣（2026-05-28 新增）

> 任何「款項 / 發票 / 退款 / 補收 / OA」異動 MUST 連帶檢查下列影響範圍。
> 受 Miles 第六輪反饋拍板：「改退款的相關功能，就要去連帶檢查是否有影響收帳款、發票等規劃」。

### 5B.1 主動實體 → 連帶影響表

| 主動實體 | 連帶實體 | 影響規則 |
|---------|---------|---------|
| **Invoice 開立** | InvoiceItem（五欄硬約束）/ PaymentAllocation（核銷分配）/ BillingInstallment（鏈式預填來源、取代 PlannedInvoice） | 五欄硬約束（ItemCount × ItemPrice = ItemAmt）+ B2B 未稅 / B2C 含稅切換 |
| **Invoice 作廢** | SalesAllowance（超過 14 天時限改折讓）/ Payment（退款 Payment 反向）| 14 天作廢期 + 超期 MUST 走折讓單 |
| **SalesAllowance 確認** | Invoice（折讓金額對應）| 折讓只掛 Invoice（不關聯退款 Payment）+ 隔日上傳財政部 |
| **Payment 退款（負值）** | OA.adjustment_type=退款（執行觸發）/ SalesAllowance（金額總額對齊，不建 FK）/ Invoice（作廢或折讓）| OA 推進條件 + 折讓與退款金額在對帳總額層對齊 |
| **OrderAdjustment 確認可執行** | Payment（退款核銷對帳應退差額）/ Invoice（作廢或折讓）/ SalesAllowance（建立）| 核可即生效（OA 確認可執行不綁 Payment 累計推進、不回退）|
| **BillingInstallment 自動建**（取代 PlannedInvoice）| Order（諮詢訂單收尾**不做大貨（source_type=consultation_end_no_production）/ 需求單流失（source_type=quote_lost）**兩情境 / 補收 OA）/ Invoice（一鍵開票繼承 items[]）| 鏈式預填 + 不自動開立發票；**諮詢取消情境 MUST NOT 自動建待開發票**（2026-05-30 converge-consultation-cancel 收斂，留存 1000 收入由業務手動開票、未開票由對帳差額警示兜底；source_type=consultation_cancellation enum 保留供手動標示）|
| **AfterSalesTicket 退款** | OA（responsibility=公司認賠 / 補退）/ Payment / SalesAllowance | 跨售後 ticket 容器 |

### 5B.2 七實體連帶網絡圖

```
                Payment ────────┐
              ↑      ↑          │
              │      │          ↓
              │   OrderAdjustment ────────┐
              │      ↑                    │
PaymentPlan ──┘      │                    │
                  AfterSalesTicket        │
              ┌──────┴──────┐             │
              ↓             ↓             ↓
            Invoice ←─ SalesAllowance ←──┤
              ↑                          │
              │                          │
       PaymentInvoice junction ─────────┘
              ↑
       BillingInstallment（取代 PlannedInvoice + PaymentPlan）
```

> 註：BillingInstallment（收款項目 / 請款期次，規劃層）+ PaymentAllocation（收款入帳明細，核銷層）已於 unify-billing-installment-and-reconciliation-csv（2026-05-28）+ 收斂 Step 2b（2026-05-29）統一取代舊 PaymentPlan / PlannedInvoice / PaymentInvoice junction 三實體。**上方網絡圖節點請讀作：PaymentPlan → BillingInstallment、PaymentInvoice junction → PaymentAllocation**（圖為舊標籤，語意以本註與 § 五 實體表為準）。收款入帳改業務手動勾選（非系統依序填滿）、發票↔期次業務 1:1。

### 5B.3 連帶檢查紀律（規劃時必做）

任何 ERP 規劃 / 設計涉及下列議題 MUST 跑連帶檢查：

| 議題類型 | 必檢實體 |
|---------|---------|
| 改退款邏輯 | Payment（退款）+ OA（執行條件）+ SalesAllowance（金額對齊）+ Invoice（作廢或折讓）+ AfterSalesTicket（若已完成）|
| 改發票開立 | Invoice（五欄硬約束）+ InvoiceItem + PaymentAllocation（核銷分配）+ BillingInstallment（鏈式預填、取代 PlannedInvoice）+ ezPay 法規 |
| 改 OA | Payment（退款核銷對帳應退差額，訂單收退款模型重構 不綁累計）+ Invoice（補收 / 折讓）+ SalesAllowance + AfterSalesTicket（若已完成）|
| 改收款 | Payment（入帳明細 PaymentAllocation）+ BillingInstallment（收款維度推進）+ Invoice（對應）+ OA（推進）|
| 改補印（after-sales）| AfterSalesTicket + OA（補退）+ Payment + SalesAllowance + 補印 PrintItem |
| 改 BillingInstallment（取代 PlannedInvoice）| Order（諮詢收尾不做大貨 / 需求單流失觸發、**諮詢取消不自動建**見 5B.1 / 補收觸發）+ Invoice（一鍵開票繼承）+ InvoiceItem（五欄） |

### 5B.4 外部硬約束連帶

任何 Invoice / InvoiceItem 異動 MUST 對齊 [[發票法規硬約束-ezPay-MIG]] 規範：
- 五欄硬約束（ItemName / ItemCount / ItemUnit / ItemPrice / ItemAmt）
- 作廢期限 14 天（超期走折讓）
- B2B 未稅 vs B2C 含稅切換
- 字軌 2 月為 1 期
- 上傳財政部時程（每日 01:00 上傳前日資料）

## 五C、發票異動三情境（作廢、折讓、改買受人）

業務遇到發票異動時，依情境選擇對應流程：

| 情境 | 適用時機 | 業務判斷規則 |
|------|---------|-------------|
| **作廢** | 同期未交付未申報、發票錯誤可修正 | 業務直接點作廢，系統呼叫第三方平台；成功則發票狀態變作廢、字軌號碼不重用 |
| **折讓** | 跨期、客戶不需作廢、只需減額 | 業務開立折讓單，折讓金額不超過該發票剩餘可折讓金額；折讓與退款款項不建立關聯（反查走訂單活動紀錄）|
| **改買受人（統編打錯）** | 買受人資訊有誤 | 業務作廢原發票 → 開新發票帶入正確買受人；不額外設計「換開」捷徑 |

跨齊報稅期規則：依中華民國稅務規則，發票一旦跨齊報稅期不可作廢，只能透過折讓單抵銷。業務不需自行判斷是否跨期——直接依直覺選擇作廢（最常見），第三方平台回應失敗時系統引導改開折讓單。詳見 [[發票法規硬約束-ezPay-MIG]] § 四。

折讓與退款分離原則：折讓單僅關聯發票（不關聯退款款項），退款款項記在收款紀錄（記負數）；折讓與退款在對帳總額層對齊即可，不需逐筆關聯。

## 五D、處理中款項不開折讓

業務建立處理中退款款項時（尚未補齊對帳附件切「已完成」），系統不自動建立折讓單，也不顯示提示。折讓相關提示僅在退款款項切「已完成」後觸發——符合會計實務「未實際發生的款項不開折讓」的對帳邏輯。處理中款項不計入收款淨額、不影響收款項目收款維度、不推進關聯異動單。

## 五E、請款與核銷四步流程

業務在規劃階段一次建立收款項目（請款期次），同時定義「何時收、何時開票、開什麼品項」三個面向，取代舊版分散操作：

```
規劃收款項目（填應收日 / 預計開票日 / 金額 / 品項 / 備註）
   ↓
一鍵開票（從收款項目繼承品項等資訊開立發票，一個期次對應一張發票）
   ↓
登錄款項（業務手動勾選收款項目 + 填入帳金額，入帳合計不超過收款金額）
   ↓
款項切已完成 → 收款項目收款維度自動更新
```

收款入帳由業務手動勾選（系統不自動依序填滿），溢收進預收桶。

## 六、三方對帳

每個情境的最終驗收都要對齊：

```
應收（訂單應收總額）= 發票淨額（已開發票合計 - 已確認折讓）= 收款淨額（Payment 加總，含負數退款）
```

差額為零代表「三方對齊」。差額不為零時觸發**對帳警示 banner**。

**收款淨額限縮**（add-payment-status-and-decouple-oa-execution change 2026-05-21）：

`收款淨額 = ∑ Payment.amount WHERE paymentStatus = '已完成'`（含正值一般收款與負值已完成退款）。處理中 Payment 不計入收款淨額（業務先填一半時對帳不受影響）、不影響 BillingInstallment 收款維度狀態、不推進關聯 OA。對帳面板新增「處理中（合計，不計入）」資訊軸顯示，差額 hint 加註「另含處理中款項 X 元，齊備後將計入」。

詳見 [order-management spec § Requirement: 三方對帳檢視面板](../../../../openspec/specs/order-management/spec.md)。

## 八、不做系統化的範疇外

| 情境 | 原因 |
|------|------|
| 跨訂單金額混合匯款對帳 | 走人工處理（業務 / 會計拆分銀行入帳）|
| 同訂單開指定數量發票 + 混單出貨 | 不做系統化 |
| 重新報價 / 重開報價單 / 重打樣 | 屬需求單議價階段，不在訂單模組範疇 |
