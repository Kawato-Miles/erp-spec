# 印件檔案備註上限（client_note）

> **業務規則：500 字上限、非必填、跟印件走、不跟輪次**。

## 一、定義

`client_note`（印件檔案備註）：

- **所有人**：業務（B2B 訂單）/ 會員（B2C / EC 訂單）
- **對象**：寫給審稿人員的說明
- **方向**：單向（業務 → 審稿），不是對話
- **規模**：1:1（印件對應一份備註）

## 二、字數上限

```typescript
export const CLIENT_NOTE_MAX_LENGTH = 500;
```

- **上限**：500 字
- **判定**：以 JavaScript String length 計算（中英混合不分權重）
- **超過行為**：UI 即時阻擋輸入超過 500 字

## 三、生命週期

| 階段 | 行為 |
|------|------|
| **需求單階段** | 業務於印件首次建立時填寫 |
| **成交轉訂單** | 帶入 `OrderPrintItem.clientNote`，**帶入後各自獨立編輯（脫鉤型）** |
| **訂單階段** | 可繼續編輯 client_note |
| **補件階段** | **不再更新**（補件不會新建 client_note） |

> [!info] 脫鉤型欄位
> client_note 在「成交轉訂單」時複製一份到訂單，**之後 Quote 與 Order 各自獨立編輯**，不加入 `TRACKED_PARITY_FIELDS`（不會自動同步）。

## 四、為什麼跟印件走、不跟輪次

| 設計理由 | 說明 |
|---------|------|
| 業務 / 會員只填一次 | 不需要每輪都重填 |
| 補件不重填 | 補件方知道規格沒變，不必重複說明 |
| 一個印件一份 | 業務責任歸屬清晰 |

→ 與審稿備註（`review_note`，跟輪次走）對照見 [[稿件管理規則]]。

## 五、相關欄位

- `OrderPrintItem.clientNote`（訂單階段，同上）
- `CLIENT_NOTE_MAX_LENGTH = 500`（前端常數）
