# 商業邏輯卡撰寫範本

> `04-business-logic/` 是 BRD 的營運層——記錄「公司怎麼做生意、怎麼決定事情、產業有什麼限制」。
> 核心紀律：**規則只寫一次**（Ronald Ross 原則），其他地方用 `[[wiki link]]` 引用，不複寫。

---

## 一、目錄結構

```
04-business-logic/
├── _template-business-logic.md  ← 本檔（撰寫準則）
├── 服務藍圖/        ← A 類：端到端業務鏈
├── 外部約束/        ← B 類：法規 / 第三方規格（不可自行更改）
├── 領域知識/        ← B 類：印刷業製程事實（不是決策，是事情本來就是這樣）
└── 營運規則/        ← B 類：公司決策規則（可改，需管理層拍板）
    ├── 售前/        ← 客戶下單前的互動
    ├── 訂單到交付/   ← 下單後到收到商品
    └── 帳務/        ← 貫穿全程的金流
```

### 分類判斷表

| 問自己 | 答案 | 放哪裡 |
|--------|------|--------|
| 這是一條從頭到尾的服務流程嗎？ | 是 | `服務藍圖/` |
| 這是法規或第三方強制的規格嗎？（公司無法自行決定） | 是 | `外部約束/` |
| 這是印刷業的製程事實嗎？（不是公司決策，是產業本來就這樣） | 是 | `領域知識/` |
| 這是公司自己定的營運規則嗎？（可以改，但要管理層拍板） | 是 | `營運規則/` 依服務階段放子目錄 |

營運規則子目錄判斷：

| 問自己 | 答案 | 放哪裡 |
|--------|------|--------|
| 客戶還沒下單前的事？（諮詢、報價、議價） | 是 | `營運規則/售前/` |
| 下單後到客戶收到商品？（審稿、生產、出貨、售後） | 是 | `營運規則/訂單到交付/` |
| 金流相關？（收付款、開票、對帳、退款） | 是 | `營運規則/帳務/` |

## 二、兩種卡型

| 卡型 | frontmatter `type` | 用途 | 只在哪個子目錄 |
|------|-------------------|------|--------------|
| **A 類：服務藍圖** | `service-blueprint` | 端到端業務鏈（公司怎麼服務客戶） | `服務藍圖/` |
| **B 類：商業規則** | `business-rule` | 獨立的決策邏輯、領域知識、外部約束 | `外部約束/`、`領域知識/`、`營運規則/` |

**A 類與 B 類的關係**：服務藍圖描述流程階段與角色交接，每個階段適用的決策規則用 `[[wiki link]]` 引用 B 類卡。B 類規則獨立於流程存在——同一條規則可能在多個流程中被引用。

---

## 三、A 類骨架：服務藍圖

一張卡 = 一條端到端業務鏈。以「客戶的目標」定義服務終點。

**段落結構**（全部必填）：

| 段落 | 內容 |
|------|------|
| 概述 | 一段話說明這條業務鏈的商業目的與服務終點 |
| 流程階段 | 5-10 個階段，每階段標註角色、交接點、決策點 |
| 現況痛點 | 目前的問題，歸因到 [[pain-points]]（隨訪談迭代擴充） |
| 適用規則 | 引用 B 類卡，標註在哪個階段生效（不複寫規則內容） |
| 範圍外 | 明確排除不涵蓋的範圍 |
| 相關卡 | 情境、角色、使用者故事等 wiki link |

**frontmatter**：

```yaml
type: service-blueprint
module: [模組]
business-domain: [領域或 cross-domain]
status: draft | active
last-reviewed: YYYY-MM-DD
```

---

## 四、B 類骨架：商業規則

一張卡 = 一條決策邏輯、領域知識或外部約束。

**段落結構**（必填標 *，選填標 ?）：

| 段落 | 內容 | 必/選 |
|------|------|-------|
| 現況 | 一句話描述目前公司怎麼處理這件事 | * |
| 規則 | 用商業語言描述（一段話講清楚） | * |
| 為什麼這樣定 | 營運動機（公司決策 / 法規 / 產業慣例） | * |
| 適用範圍 | 連結到服務藍圖的哪個階段生效 | * |
| 領域知識 | 印刷業特有的背景 | ? |
| 具體例子 | 真實業務數字說明（1-3 個） | * |
| 例外情境 | 什麼時候不適用 | ? |
| 相關卡 | 角色、實體、狀態機等 wiki link | * |

**frontmatter**：

```yaml
type: business-rule
module: [模組]
business-domain: [領域或 cross-domain]
status: draft | active
last-reviewed: YYYY-MM-DD
source: [規則來源]
```

## 五、不收清單（B 類卡排除的內容）

| 不收 | 應去的位置 | 判斷方式 |
|------|----------|---------|
| 欄位名稱與定義 | `05-entities/` 實體卡 | 看到 `field_name`、底線命名、駝峰命名 |
| 計算公式（程式碼層級） | `05-entities/` 或 OpenSpec | 看到 `floor()`、`min()`、`sum()`、程式碼片段 |
| 狀態枚舉值 | `06-state-machines/` 狀態機卡 | 看到 `status = '已完成'`、狀態清單 |
| API 規格（欄位長度、加密方式） | OpenSpec 或外部文件 | 看到 `Varchar(30)`、`AES-256` |
| Prototype 檔案路徑或行號引用 | OpenSpec `implemented-by` | 看到 `src/types/`、`L58-61` |
| 步驟級操作細節 | `07-scenarios/` 或 `13-user-stories/` | 看到「第 1 步...第 2 步...」 |

**發現實作細節時的處置**：
1. 判斷該內容的正確歸屬位置
2. 確認目標位置的卡片存在
3. 如果目標位置已有該內容 → 直接從 B 類卡刪除
4. 如果目標位置沒有 → 搬移過去後再從 B 類卡刪除

## 六、撰寫紀律

1. **規則只寫一次**：B 類卡是規則的唯一正本。A 類、情境、使用者故事只能引用
2. **痛點掛在流程上**：現況痛點寫在 A 類（流程級別），B 類只寫一句「現況」
3. **禁中英夾雜**：技術詞括號附註，不直接混入中文句子
4. **禁實作術語當主詞**：用角色、業務動作、金額描述，不用欄位名或程式碼物件
5. **不確定項開 OQ**：觸發 `oq-manage` 開獨立 OQ 卡，不在正文內留 inline 標注
6. **範圍外必寫**：A 類和 B 類都要明確說不涵蓋什麼
7. **檔名即標題**：卡片內容不重複寫 `# 標題`，Obsidian 以檔名顯示

## 七、輸入確認（寫卡前必須完成）

撰寫任何一張卡之前，必須先確認以下問題。**答不出來的就停下來與使用者確認，不可猜測填充。**

資訊來源不限（外部參考、內部邏輯推導、訪談、既有文件等皆可），重點是**資訊是否已被確認**。

### A 類服務藍圖的輸入

| 必須確認 | 確認狀態 |
|---------|---------|
| 這條服務的客戶目標是什麼？（定義服務終點） | 已確認 / 待確認 → 停下 |
| 從哪裡開始？（觸發點） | 已確認 / 待確認 → 停下 |
| 經過哪些角色交接？ | 已確認 / 待確認 → 停下（不可假設角色職責） |
| 有哪些決策分叉？每個分叉走向？ | 已確認 / 待確認 → 停下 |
| 目前有什麼痛點？ | 已確認 / 暫無 → 可留空標「待訪談」 |
| 哪些不在範圍內？ | 已確認 / 待確認 → 停下（不可自行假設範圍） |

### B 類商業規則的輸入

| 必須確認 | 確認狀態 |
|---------|---------|
| 這條規則解決什麼問題？（為什麼存在） | 已確認 / 待確認 → 停下 |
| 目前公司怎麼處理？（現況） | 已確認 / 待確認 → 停下（不可假設現況） |
| 規則的性質？（公司決策 / 法規 / 產業慣例） | 已確認 / 待確認 → 停下 |
| 在哪些流程階段生效？ | 對照服務藍圖 / 如無藍圖 → 標記「待補」 |
| 有沒有例外情境？ | 已確認 / 暫無 → 可留空標「待確認」 |

## 八、邊際情境處理

| 卡住的情境 | 標準處理 |
|-----------|---------|
| 不確定放哪個子目錄 | 用判斷表走一遍。走完仍無法確定 → 開 OQ（`oq-manage` mode B），描述「內容是 X，不確定屬 Y 還是 Z，因為...」，暫放最接近的目錄，等使用者確認後搬移 |
| 一張卡跨兩個分類 | 問使用者：「這張卡 80% 的使用情境屬哪個分類？」放主要分類，另一個分類用 wiki link 引用 |
| 現有分類都不適合 | 向使用者提議新子目錄，說明為什麼既有分類不夠，等確認後再建 |
| 不確定是領域知識還是營運規則 | 問：「這件事是印刷業本來就這樣（領域知識），還是公司自己決定這樣做（營運規則）？」 |
| 不確定是外部約束還是營運規則 | 問：「公司可以自己改這條規則嗎？」可以 → 營運規則；不可以 → 外部約束 |
| 改寫舊卡時不確定某段保留還是下沉 | 問：「這段回答的是『公司怎麼決定事情』（保留），還是『系統怎麼實作』（下沉）？」答不出來 → 停下問使用者 |

## 九、error handle（卡住時的標準處理）

### 觸發條件（出現任一即停下）

1. **無來源依據**：要寫的內容沒有已確認的資訊支持，只有推測 → 停下
2. **矛盾**：新內容與既有卡片矛盾 → 停下，列出矛盾兩方，問使用者哪個對
3. **需要假設**：必須假設「公司應該是這樣做的」才能繼續 → 停下
4. **分類不明**：判斷表走完仍無法確定 → 開 OQ，停下
5. **超出已知**：要描述的流程或規則超出目前已確認範圍 → 停下

### 處理方式

- 標記卡住的具體問題（「我不確定 X，因為 Y」）
- 向使用者提問，附上已知事實與不確定部分
- **禁止**用「一般來說」「通常」「應該是」填充未確認內容
- 如果是改寫舊卡時發現的問題，保留舊內容不動，另開 OQ 追蹤

## 十、擴張準則

### 何時新建卡 vs. 併入既有卡

| 情境 | 做法 |
|------|------|
| 新發現一條獨立的決策規則 | 新建 B 類卡 |
| 新發現的規則只是既有規則的補充說明 | 併入既有卡 |
| 一張卡超過 200 行 | 考慮拆分（但訂單異動規則等複雜規則例外） |
| 同一張卡涵蓋兩個不相干的主題 | 拆成兩張 |

### 何時拆子目錄

| 情境 | 做法 |
|------|------|
| 單一子目錄超過 10 張卡 | 考慮依子主題再拆一層 |
| 新的服務流程（如 EC 訂單） | 在 `服務藍圖/` 新建卡 |
| 新的外部約束（如新法規） | 在 `外部約束/` 新建卡 |
| 新的產業知識 | 在 `領域知識/` 新建卡 |

### 新服務藍圖的判斷

只有「客戶的目標不同」時才開新的服務藍圖：
- 諮詢服務：客戶目標 = 得到諮詢建議
- 商品販售：客戶目標 = 收到商品
- 帳務流程：不是客戶目標驅動，但獨立運作 → 可開藍圖

「同一個服務中的分支」（如打樣/大貨）不另開藍圖，在既有藍圖中標為可選步驟。

## 十一、稽核維度（vault-audit 用）

| # | 維度 | 通過標準 |
|---|------|---------|
| 1 | 分類正確 | 卡片放在對的子目錄 |
| 2 | 卡型對齊 | frontmatter type 是 service-blueprint 或 business-rule |
| 3 | 結構對齊 | 段落結構符合 A 類或 B 類骨架 |
| 4 | 不收內容 | 無欄位名、計算公式、API 規格、Prototype 引用 |
| 5 | 規則唯一性 | 同一規則只寫一次，無跨卡重複 |
| 6 | 引用完整 | 藍圖有引用規則；規則有引用適用範圍 |
| 7 | 語言紀律 | 無中英夾雜、無實作術語當主詞 |
| 8 | 範圍外 | 有寫「範圍外」或「例外情境」段 |
