# 商業邏輯卡撰寫範本

> `04-business-logic/` 是 BRD 的營運層——記錄「公司怎麼做生意、怎麼決定事情、產業有什麼限制」。
> 核心紀律：**規則只寫一次**（Ronald Ross 原則），其他地方用 `[[wiki link]]` 引用，不複寫。

---

## 一、撰寫流程（所有卡型共用）

新資料進來時，依以下步驟處理。**每一步都必須完成才能進入下一步。**

```
辨識 → 歸位 → 確認前提 → 撰寫 → 稽核 → 定案
```

| 步驟 | 做什麼 | 卡住時 |
|------|--------|--------|
| **1. 辨識** | 這筆資料的性質是什麼？依 § 二分類判斷表走一遍 | 走不通 → 見 § 六邊際情境 |
| **2. 歸位** | 依判斷結果決定卡型 + 子目錄 + `mutability` | 走不通 → 見 § 六邊際情境 |
| **3. 確認前提** | 依對應卡型的撰寫規則（§ 三/四）逐項確認前提 | 有未確認項 → 停下問使用者 |
| **4. 撰寫** | 依對應卡型的產出格式（§ 八/九）填入內容 | 需要假設才能寫 → 停下（§ 七） |
| **5. 稽核** | 依 § 十二稽核維度自檢 | 不通過 → 回步驟 3 或 4 修正 |
| **6. 定案** | commit | |

## 二、分類判斷表

### 兩種卡型

| 卡型 | `type` | 用途 |
|------|--------|------|
| **服務藍圖** | `service-blueprint` | 端到端業務鏈（公司怎麼服務客戶） |
| **商業規則** | `business-rule` | 決策邏輯、領域知識、外部約束 |

### 三種可變性（`mutability`，商業規則專用）

| `mutability` | 意義 | 子目錄 | 誰能改 |
|---|---|---|---|
| `external` | 外部約束（法規/第三方規格） | `外部約束/` | 只有外部來源變更時（如法規修訂、API 改版） |
| `domain` | 領域知識（產業事實） | `領域知識/` | 產業本身改變時（極少） |
| `internal` | 營運規則（公司決策） | `營運規則/` | 訪談、管理層拍板可改 |

### 判斷流程

依序問以下問題，**第一個命中的就是答案**：

| 順序 | 問自己 | 是 | 卡型 | `mutability` | 子目錄 |
|------|--------|---|------|-------------|--------|
| 1 | 這是一條端到端的服務流程嗎？（有明確的客戶目標、起點、終點） | 是 | 服務藍圖 | 不適用 | `服務藍圖/` |
| 2 | 這是法規或第三方強制的規格嗎？（公司無法自行決定） | 是 | 商業規則 | `external` | `外部約束/` |
| 3 | 這是印刷業的製程事實嗎？（不是公司決策，是產業本來就這樣） | 是 | 商業規則 | `domain` | `領域知識/` |
| 4 | 這是公司自己定的營運規則嗎？（可以改，但要管理層拍板） | 是 | 商業規則 | `internal` | `營運規則/` 再依下表分流 |

**判斷第 3 題 vs 第 4 題的方法**：問「公司可以自己改這條規則嗎？」
- 不行，換任何一家印刷廠都一樣 → 領域知識（`domain`）
- 可以，是我們公司自己決定這樣做 → 營運規則（`internal`）

**營運規則子目錄分流**：

| 問自己 | 是 | 子目錄 |
|--------|---|--------|
| 客戶還沒下單前的事？（諮詢、報價、議價） | 是 | `營運規則/售前/` |
| 下單後到客戶收到商品？（審稿、生產、出貨、售後） | 是 | `營運規則/訂單到交付/` |
| 金流相關？（收付款、開票、對帳、退款） | 是 | `營運規則/帳務/` |

## 三、服務藍圖撰寫規則

**觸發條件**：辨識到一條端到端的業務鏈（客戶有明確目標、有起點和終點）。

### 前提確認（MUST 全部已確認才能開始撰寫）

| 前提 | 確認狀態 |
|------|---------|
| 客戶目標（服務終點）是什麼？ | 已確認 / 待確認 → 停下 |
| 從哪裡開始？（觸發點） | 已確認 / 待確認 → 停下 |
| 經過哪些角色交接？ | 已確認 / 待確認 → 停下（不可假設角色職責） |
| 有哪些決策分叉？每個分叉走向？ | 已確認 / 待確認 → 停下 |
| 目前有什麼痛點？ | 已確認 / 暫無 → 可留空標「待訪談」 |
| 哪些不在範圍內？ | 已確認 / 待確認 → 停下（不可自行假設範圍） |

### 撰寫規則

| 強度 | 規則 |
|------|------|
| MUST | 每個階段標註角色與交接對象 |
| MUST | 決策分叉只標方向，細節引用商業規則卡（不複寫規則內容） |
| MUST | 寫「範圍外」段 |
| SHOULD | 痛點歸因到 [[pain-points]]（暫無痛點可標「待訪談」） |
| MAY | 同一服務的可選子步驟（如打樣）寫在階段內，不另開卡 |

### 產出格式

見 § 八。

## 四、商業規則撰寫規則

**觸發條件**：辨識到一條決策邏輯、產業事實、或外部約束。

### 可變性分流

撰寫規則和產出格式相同，但子目錄和 `mutability` 不同。用以下判斷分流：

| 問自己 | 答案 | `mutability` | 子目錄 |
|--------|------|-------------|--------|
| 公司可以自己改嗎？ | 不行，是法規/第三方規定 | `external` | `外部約束/` |
| 公司可以自己改嗎？ | 不行，是產業本來就這樣 | `domain` | `領域知識/` |
| 公司可以自己改嗎？ | 可以，是公司決定的 | `internal` | `營運規則/` |

### 前提確認（MUST 全部已確認才能開始撰寫）

| 前提 | 確認狀態 |
|------|---------|
| 這條規則解決什麼問題？（為什麼存在） | 已確認 / 待確認 → 停下 |
| 目前公司怎麼處理？（現況） | 已確認 / 待確認 → 停下（不可假設現況） |
| 規則可變性？（外部約束 / 領域知識 / 營運規則） | 已確認 → 設定 `mutability` 分流 / 待確認 → 停下 |
| 在哪些流程階段生效？ | 對照服務藍圖 / 如無藍圖 → 標記「待補」 |
| 有沒有例外情境？ | 已確認 / 暫無 → 可留空標「待確認」 |

**外部約束額外前提**（`mutability: external` 時追加）：

| 前提 | 確認狀態 |
|------|---------|
| 約束來源（哪個法規/第三方）？ | 已確認且有外部文件佐證 / 待確認 → 停下 |
| 公司資料與外部的對應關係？ | 已確認且可在 `05-entities/` 找到對應 / 找不到 → 開 OQ |

### 撰寫規則

| 強度 | 規則 |
|------|------|
| MUST | 規則用商業語言描述，禁實作術語當主詞 |
| MUST | 現況段具體描述目前的工具與方式，不寫抽象描述 |
| MUST | 附至少一個具體例子（真實業務數字） |
| MUST | 標註適用範圍（引用服務藍圖的階段） |
| MUST | 外部約束卡（`mutability: external`）須標明外部來源文件（法規條號、API 文件版本） |
| SHOULD | 標註例外情境 |
| SHOULD | 標註規則來源（誰決定的、什麼依據） |
| SHOULD | 相關卡段依分類列出（角色/實體/狀態機/規則/OQ），只列有連結的分類 |
| MAY | 領域知識段補充印刷業背景 |
| MAY | 複雜規則（多條並列）可用「意圖/規則/營運動機」三欄表格呈現每條規則 |

### 產出格式

見 § 九。

## 五、修改商業規則前的可變性檢查

修改任何商業規則卡之前，MUST 先檢查 frontmatter `mutability`：

| `mutability` | 可以改嗎 | 改之前要確認什麼 |
|---|---|---|
| `external` | **原則上不可改** | 外部來源真的變了嗎？（法規修訂、API 改版）→ 須附外部變更依據 |
| `domain` | **原則上不可改** | 產業事實真的變了嗎？→ 須確認不是訪談中的誤解 |
| `internal` | **可以改** | 確認是管理層拍板或訪談確認的變更 |

**觸發 error handle 的情境**：訪談中有人說要改某條規則，但該卡 `mutability` 為 `external` 或 `domain` → 停下，與使用者確認是「外部來源真的變了」還是「對規則的理解有誤」。

## 六、邊際情境處理

| 卡住的情境 | 標準處理 |
|-----------|---------|
| 分類判斷表走完仍無法確定 | 開 OQ（`oq-manage` mode B），描述「內容是 X，不確定屬 Y 還是 Z，因為...」，暫放最接近的目錄，等使用者確認後搬移 |
| 一張卡跨兩個分類 | 問使用者：「這張卡 80% 的使用情境屬哪個分類？」放主要分類，另一個分類用 wiki link 引用 |
| 現有分類都不適合 | 向使用者提議新子目錄，說明為什麼既有分類不夠，等確認後再建 |
| 改寫舊卡時不確定某段保留還是下沉 | 問：「這段回答的是『公司怎麼決定事情』（保留），還是『系統怎麼實作』（下沉）？」答不出來 → 停下問使用者 |
| 同一規則疑似在別張卡已存在 | 先搜尋確認。已存在 → 併入不新建；不確定是否重複 → 問使用者 |

## 七、error handle（卡住時的標準處理）

### 觸發條件（出現任一即停下）

1. **無來源依據**：要寫的內容沒有已確認的資訊支持，只有推測 → 停下
2. **矛盾**：新內容與既有卡片矛盾 → 停下，列出矛盾兩方，問使用者哪個對
3. **需要假設**：必須假設「公司應該是這樣做的」才能繼續 → 停下
4. **分類不明**：判斷表走完仍無法確定 → 開 OQ，停下
5. **超出已知**：要描述的流程或規則超出目前已確認範圍 → 停下
6. **可變性衝突**：要修改 `mutability` 為 `external` 或 `domain` 的卡 → 停下，確認外部來源或產業事實是否真的改變

### 處理方式

- 標記卡住的具體問題（「我不確定 X，因為 Y」）
- 向使用者提問，附上已知事實與不確定部分
- **禁止**用「一般來說」「通常」「應該是」填充未確認內容
- 如果是改寫舊卡時發現的問題，保留舊內容不動，另開 OQ 追蹤

## 八、服務藍圖產出格式

一張卡 = 一條端到端業務鏈。以「客戶的目標」定義服務終點。

**段落結構**（全部必填）：

| 段落 | 內容 |
|------|------|
| 概述 | 一段話說明這條業務鏈的商業目的與服務終點 |
| 流程階段 | 5-10 個階段，每階段標註角色、交接點、決策點 |
| 現況痛點 | 目前的問題，歸因到 [[pain-points]]（隨訪談迭代擴充） |
| 適用規則 | 引用商業規則卡，標註在哪個階段生效（不複寫規則內容） |
| 範圍外 | 明確排除不涵蓋的範圍 |
| 相關卡 | 依分類列出 wiki link（情境/角色/使用者故事/實體/狀態機），只列有連結的分類 |

**frontmatter**：

```yaml
type: service-blueprint
module: [模組]
business-domain: [售前 / 訂單管理 / 印前審稿 / 生產執行 / 履約與售後 / 款項與發票 / 跨領域]
status: draft | active
last-reviewed: YYYY-MM-DD
```

## 九、商業規則產出格式

一張卡 = 一條決策邏輯、領域知識或外部約束。

**段落結構**（必填標 *，選填標 ?）：

| 段落 | 內容 | 必/選 |
|------|------|-------|
| 現況 | 一句話描述目前公司怎麼處理這件事 | * |
| 規則 | 用商業語言描述（一段話講清楚） | * |
| 為什麼這樣定 | 營運動機（公司決策 / 法規 / 產業慣例） | * |
| 適用範圍 | 連結到服務藍圖的哪個階段生效 | * |
| 領域知識 | 印刷業特有的背景 | ? |
| 具體例子 | 真實業務數字說明（1-3 個） | * |
| 例外情境 | 什麼時候不適用 | ? |
| 相關卡 | 依分類列出 wiki link（角色/實體/狀態機/規則/OQ），只列有連結的分類 | * |

**frontmatter**：

```yaml
type: business-rule
module: [模組]
business-domain: [售前 / 訂單管理 / 印前審稿 / 生產執行 / 履約與售後 / 款項與發票 / 跨領域]
mutability: external | domain | internal
status: draft | active
last-reviewed: YYYY-MM-DD
source: [規則來源]
```

## 十、不收清單（商業規則卡排除的內容）

| 不收 | 應去的位置 | 判斷方式 |
|------|----------|---------|
| 欄位名稱與定義 | `05-entities/` 實體卡 | 看到 `field_name`、底線命名、駝峰命名 |
| 計算公式（程式碼層級） | `05-entities/` 或 OpenSpec | 看到 `floor()`、`min()`、`sum()`、程式碼片段 |
| 狀態枚舉值 | `06-state-machines/` 狀態機卡 | 看到 `status = '已完成'`、狀態清單 |
| API 規格（欄位長度、加密方式） | OpenSpec 或外部文件 | 看到 `Varchar(30)`、`AES-256` |
| Prototype 檔案路徑或行號引用 | OpenSpec `implemented-by` | 看到 `src/types/`、`L58-61` |
| 步驟級操作細節 | `07-scenarios/` 或 `13-user-stories/` | 看到「第 1 步...第 2 步...」 |

**發現實作細節時的處置**：
1. 判斷該內容的正確歸屬位置
2. 確認目標位置的卡片存在
3. 如果目標位置已有該內容 → 直接從卡中刪除
4. 如果目標位置沒有 → 搬移過去後再刪除

## 十一、撰寫紀律

1. **規則只寫一次**：商業規則卡是規則的唯一正本。服務藍圖、情境、使用者故事只能引用
2. **痛點掛在流程上**：現況痛點寫在服務藍圖（流程級別），商業規則只寫一句「現況」
3. **禁中英夾雜**：技術詞括號附註，不直接混入中文句子
4. **禁實作術語當主詞**：用角色、業務動作、金額描述，不用欄位名或程式碼物件
5. **不確定項開 OQ**：觸發 `oq-manage` 開獨立 OQ 卡，不在正文內留 inline 標注
6. **範圍外必寫**：服務藍圖和商業規則都要明確說不涵蓋什麼
7. **檔名即標題**：卡片內容不重複寫 `# 標題`，Obsidian 以檔名顯示
8. **修改前檢查可變性**：修改商業規則卡前先看 `mutability`，外部約束和領域知識不可因訪談而改

## 十二、擴張準則

### 何時新建卡 vs. 併入既有卡

| 情境 | 做法 |
|------|------|
| 新發現一條獨立的決策規則 | 新建商業規則卡 |
| 新發現的規則只是既有規則的補充說明 | 併入既有卡 |
| 一張卡超過 200 行 | 考慮拆分（但複雜規則如訂單異動規則例外） |
| 同一張卡涵蓋兩個不相干的主題 | 拆成兩張 |

### 何時拆子目錄

| 情境 | 做法 |
|------|------|
| 單一子目錄超過 10 張卡 | 考慮依子主題再拆一層 |
| 新的服務流程 | 在 `服務藍圖/` 新建卡 |
| 新的外部約束 | 在 `外部約束/` 新建卡 |
| 新的產業知識 | 在 `領域知識/` 新建卡 |

### 新服務藍圖的判斷

只有「客戶的目標不同」時才開新的服務藍圖：
- 諮詢服務：客戶目標 = 得到諮詢建議
- 商品販售：客戶目標 = 收到商品
- 帳務流程：不是客戶目標驅動，但獨立運作 → 可開藍圖

「同一個服務中的分支」（如打樣/大貨）不另開藍圖，在既有藍圖中標為可選步驟。

## 十三、稽核維度（vault-audit 用）

| # | 維度 | 通過標準 |
|---|------|---------|
| 1 | 分類正確 | 卡片放在對的子目錄 |
| 2 | 卡型對齊 | frontmatter type 是 service-blueprint 或 business-rule |
| 3 | 可變性標記 | 商業規則卡有 `mutability` 欄位（external/domain/internal） |
| 4 | 結構對齊 | 段落結構符合對應產出格式 |
| 5 | 不收內容 | 無欄位名、計算公式、API 規格、Prototype 引用 |
| 6 | 規則唯一性 | 同一規則只寫一次，無跨卡重複 |
| 7 | 引用完整 | 藍圖有引用規則；規則有引用適用範圍 |
| 8 | 語言紀律 | 無中英夾雜、無實作術語當主詞 |
| 9 | 範圍外 | 有寫「範圍外」或「例外情境」段 |
