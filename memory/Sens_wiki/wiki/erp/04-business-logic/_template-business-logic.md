# 商業邏輯卡撰寫範本

> `04-business-logic/` 有兩種卡型，依內容性質選用。
> 核心紀律：**規則只寫一次**（Ronald Ross 原則），其他地方用 `[[wiki link]]` 引用，不複寫。

## 一、兩種卡型

| 卡型 | frontmatter `type` | 用途 | 判斷口訣 |
|------|-------------------|------|---------|
| **A 類：服務藍圖** | `service-blueprint` | 描述一條端到端業務鏈（公司怎麼服務客戶） | 「這是一條從頭到尾的流程嗎？」 |
| **B 類：商業規則** | `business-rule` | 描述一條獨立的決策邏輯或領域知識 | 「這是一條決策原則或產業慣例嗎？」 |

**A 類與 B 類的關係**：服務藍圖描述流程階段與角色交接，每個階段適用的決策規則用 `[[wiki link]]` 引用 B 類卡。B 類規則獨立於流程存在——同一條規則可能在多個流程中被引用。

## 二、B 類明確不收的內容

以下內容不屬於商業規則卡，須放在對應的下游位置：

| 不收 | 應去的位置 |
|------|----------|
| 欄位名稱與定義 | `05-entities/` 實體卡 |
| 計算公式（程式碼層級） | `05-entities/` 或 OpenSpec spec |
| 狀態枚舉值 | `06-state-machines/` 狀態機卡 |
| API 規格（欄位長度、加密方式） | OpenSpec 或外部文件 |
| Prototype 檔案路徑或行號引用 | OpenSpec `implemented-by` |
| 步驟級操作細節（第 1 步...第 2 步...） | `07-scenarios/` 情境卡或 `13-user-stories/` |

## 三、A 類骨架：服務藍圖

一張卡 = 一條端到端業務鏈。

```markdown
---
type: service-blueprint
module:
  - <模組，可多選>
business-domain:
  - <領域或 cross-domain>
status: draft
last-reviewed: YYYY-MM-DD
---

# <業務鏈名稱>

## 概述

一段話說明這條業務鏈的商業目的。

## 流程階段

[角色] 做什麼 → 交接給 [角色] → ...
（5-10 個階段，每階段標註：誰做、交接給誰、主要決策點）

## 現況痛點

目前這條流程的問題（隨訪談迭代擴充）。
歸因到 [[pain-points]] 的對應核心問題。

## 適用規則

- [[規則卡名]] — 在哪個階段生效
（引用 B 類卡，不複寫規則內容）

## 範圍外

本流程不涵蓋的範圍（明確排除）。

## 相關卡

- 情境：[[07-scenarios 卡]]
- 角色：[[03-roles 卡]]
- 使用者故事：[[13-user-stories 卡]]
```

## 四、B 類骨架：商業規則

一張卡 = 一條決策邏輯或領域知識。

```markdown
---
type: business-rule
module:
  - <模組，可多選>
business-domain:
  - <領域或 cross-domain>
status: draft
last-reviewed: YYYY-MM-DD
source:
  - "<規則來源：公司決策 / 法規 / 產業慣例>"
---

# <規則名稱>

## 現況

一句話描述目前公司怎麼處理這件事。

## 規則

用商業語言描述這條規則（一段話講清楚）。

## 為什麼這樣定

來源：公司決策 / 法規 / 產業慣例。
（說明營運動機，不只是「規定如此」）

## 適用範圍

- [[服務藍圖卡名]] § 階段 X
（連結到哪些服務藍圖的哪個階段生效）

## 領域知識

印刷業特有的背景說明（如有）。

## 具體例子

用真實業務數字說明（1-3 個）。

## 例外情境

什麼時候不適用（如有）。

## 相關卡

- 角色：[[03-roles 卡]]
- 實體：[[05-entities 卡]]
- 狀態機：[[06-state-machines 卡]]
```

## 五、撰寫紀律

1. **規則只寫一次**：B 類卡是規則的唯一正本。A 類、情境、使用者故事只能引用。
2. **痛點掛在流程上**：現況痛點寫在 A 類（流程級別），B 類只寫一句「現況」。
3. **禁中英夾雜**：技術詞括號附註，不直接混入中文句子。
4. **禁實作術語當主詞**：營運背景段用角色、業務動作、金額描述，不用欄位名或程式碼物件。
5. **不確定項開 OQ**：識別到待確認內容，觸發 `oq-manage` 開獨立 OQ 卡，不在正文內留 inline 標注。
6. **範圍外必寫**：A 類和 B 類都要明確說不涵蓋什麼。
