# AR-7 審稿主管監控異常閾值定義

## 問題

US-AR-005「監控當日審稿工作量」與 US-AR-006「比對審稿人員績效」都描述「審稿主管依指標判斷異常」，但 spec 與 user story 均未定義「異常」的具體閾值：

- **US-AR-005**：「未合格數異常多 → 補強審稿員」中的「異常多」是多少？絕對值（如 > 20 件）？比率（如 > 30%）？對比基準（如高於昨日 50%）？
- **US-AR-006**：「異常退件審稿員」中的「異常」是多少？高於部門平均 N%？高於中位數 1.5 倍？

spec L632 明示「低樣本數不標異常」設計決策（系統不自動標異常徽章，由主管肉眼判斷），但「肉眼判斷」需要主管心中有閾值基準。

## 影響

- 影響 US-AR-005 / US-AR-006 的「以便」量化描述（spec § 五新紀律要求量化）
- 影響後續 Notion KPI DB 是否要建「異常監測閾值」指標卡
- 影響審稿主管培訓內容（「怎樣算異常」需要 SOP）
- spec L632「不自動標異常」設計決策仍保留，但需提供主管參考基準

## 選項

| 選項 | 規則 |
|------|------|
| A | 系統不定義閾值，由主管肉眼判斷（保持 spec L632 現狀，user story 不量化）|
| B | 系統提供參考閾值（如：不合格率 > 30% / 退件率高於部門中位數 1.5 倍）但不自動標異常 |
| C | 系統定義硬閾值並自動標示（違反 spec L632，需 spec 級變更） |
| D | 由審稿主管自訂個人化閾值（每位主管可設不同閾值，系統自動標示） |

## 暫定處理

- US-AR-005 / US-AR-006 「以便」段保持定性描述（「即時掌握異常」「辨識異常退件審稿員」），不量化
- 待 Miles 解答後決定是否補 user story 量化條件或建議閾值
