# AR-12 打樣後新稿件實體機制與根因判定

## 問題

US-AR-011「打樣後重新處理稿件」涉及兩個關鍵設計議題，spec 與業務邏輯卡均未明示：

### 議題 1：新稿件與原印件的實體關聯方式

打樣不合格且根因為稿件問題時，業務取得客戶新稿件後，系統如何處理？

| 方向 | 規則 | 對齊 |
|------|------|------|
| A 棄用 + 建新印件 | 原打樣印件狀態轉「已棄用」，建立新印件（FK 指向原印件追溯）；新印件審稿從第 1 輪重新開始 | 對齊 spec L244「合格為終態，後續變更內容透過棄用 + 建新印件」 |
| B 同印件追加新審稿輪次 | 同印件下追加新 ReviewRound，類似補件迴圈但語意改為「重提」 | 違反 spec L244「合格為終態」原則 |

不同方向對「業界 ERP 統計打樣後重提次數」「報價是否需重收打樣費」「ActivityLog 事件命名」皆有顯著影響。

### 議題 2：根因判定機制（誰判定？依據什麼？）

業務流程描述「打樣結果經分析確認問題來自稿件」，但未明示：

- **判定者**：印務 / 業務 / QC 品檢 / 跨角色會議？
- **判定依據**：客觀標準是什麼？「客戶設計稿色彩偏差」可能同時是「稿件 CMYK 設定錯誤」+「打樣機色彩管理偏差」雙重根因
- **判定流程**：是否需要書面記錄判定理由？是否需要主管核可？

## 影響

- 影響 US-AR-011 業務流程描述（誰執行 step 1 根因判定）
- 影響 US-AR-011 acceptance criteria（「原打樣印件如何處理」「新印件如何標記來源」）
- 影響 [[印件]] 資料模型（是否需要 `previous_print_item_id` FK？是否新增 `is_supplementary` 標記？）
- 影響 [[打樣流程]] § 情境 3 描述（是否要更精細化）
- 影響打樣費用計算（重提是否重收）— 跨訂單管理模組

## 決議（2026-05-23）

apply 階段對照 Prototype 既有實作（[src/types/order.ts:38 SampleResult enum](../../../../sens-erp-prototype/src/types/order.ts)、[src/pages/WorkOrderDetail.tsx](../../../../sens-erp-prototype/src/pages/WorkOrderDetail.tsx)、[src/store/useErpStore.ts:2482-2592](../../../../sens-erp-prototype/src/store/useErpStore.ts)）發現 Prototype 已實作完整業務判定 + 棄用 + clone 機制；OQ 議題 1 + 2 實質已被 Prototype 解掉。本 change 工作為「補 spec 對齊既有實作」+「補結構化追溯 FK」。

### 議題 1 拍板：方向 A（棄用 + 建新印件）— 對齊 Prototype 既有實作

**Prototype 既有實作**（`rebuildPrintItemForSampleNG`）：
- 原打樣印件 `printItemStatus = '已棄用'`（用既有 `PrintItemStatus` enum）
- clone 新印件（沿用規格 + reset 審稿狀態）
- 追溯關聯透過 `notes` 文字「（由 [原印件 ID] NG-稿件問題重建）」

**本 change 補強**：
- 新增 `PrintItem.derived_from_print_item_id` FK PrintItem nullable — **結構化追溯關聯**，補 Prototype 既有 notes 文字追溯的反向查詢能力（`WHERE derived_from_print_item_id = X` 找衍生印件）
- notes 與 FK 並存：notes 人類可讀、FK 機器可查詢

**取消的設計**（過度設計）：
- ~~`lifecycle_status` enum 獨立欄位~~ → 用既有 `PrintItemStatus = '已棄用'`
- ~~印件層 ActivityLog 新增「打樣後棄用 / 自原印件建立」事件型別~~ → 用既有訂單層 `Order.activityLogs` 弱型別文字（對齊 Prototype）

### 議題 2 拍板：業務直接在打樣 WorkOrder 詳情頁判定 sampleResult enum

**Prototype 既有實作**：
- 判定者：業務（current user，order owner）
- 觸發位置：打樣 WorkOrder 詳情頁（**不是印件詳情頁**）
- 觸發條件：`WorkOrder.type = 打樣 && WorkOrder.status = 已完成 && sampleResult = 待確認`
- UI：「填打樣結果」按鈕 → Dialog 三選一
- enum：`'待確認' | 'OK' | 'NG-製程問題' | 'NG-稿件問題'`（含初始「待確認」狀態，**中文措辭**）

**各 enum 分支後續流程**：
- `OK` → 業務後續手動建大貨工單（系統不自動觸發）
- `NG-製程問題` → 業務 UI 自行建新打樣 WorkOrder（系統不自動建，保留業務決定權；下游自動化待 OQ AR-13）
- `NG-稿件問題` → 系統自動觸發棄用 + clone 流程（議題 1 邏輯）

### 取消的設計（從 spec 對齊 Prototype）

| 項目 | 原 spec 設計 | 對齊後 |
|------|--------------|--------|
| 欄位命名 | `proofing_result` | `sampleResult` |
| Enum 值 | `ok / ng_artwork / ng_process` 英文 | `'待確認' / 'OK' / 'NG-製程問題' / 'NG-稿件問題'` 中文 |
| 觸發位置 | 印件詳情頁 | 打樣 WorkOrder 詳情頁 |
| 補充備註欄位 | `proofing_result_note` 自由文字 | 無此欄位（取消） |
| 棄用狀態欄位 | `lifecycle_status` 獨立 enum | 既有 `PrintItemStatus = '已棄用'` |
