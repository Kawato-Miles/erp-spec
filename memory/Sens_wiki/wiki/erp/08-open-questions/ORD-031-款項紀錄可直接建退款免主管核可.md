# ORD-031：退款是否一律須走訂單異動單，或款項紀錄可直接建退款且免主管核可

## 問題描述

2026-05-07 既有決策定「退款一律走訂單異動單」（建負項訂單異動 → 主管核可 → 推進已執行）。訂單收退款模型重構（訂單收退款通用方案）追加需求：完成前的減量退款應可在訂單詳情頁「款項紀錄」直接建一筆退款款項，無需主管核可，作為完成前減量的軟兜底入口。

兩者衝突：5-07 把退款收斂到單一審核入口（訂單異動單 + 主管核可）；訂單收退款模型重構 要在款項紀錄開一條免核可的直建退款路徑。

## 涉及範圍

- 模組：order-management（款項紀錄區塊 / 訂單異動入口）、payment（退款款項建立）
- 相關卡：[[../04-business-logic/訂單異動規則]]、[[../05-entities/訂單]]
- 影響：完成前減量退款的發起動線、主管核可把關強度、退款留痕與對帳

## 討論記錄

- 5-07 決策：退款金額異動屬報價總額變動，須走訂單異動單 + 業務主管核可，避免業務私自退款。
- 訂單收退款模型重構 重新定調：完成前（訂單未進終態）金額調降本就採弱把關（主管事後可見、不阻擋，見 [[ORD-030-完成前明細直接增減真實業務流程待訪談|ORD-030]] 訪談結論），退款作為調降的金流落地，沒有理由比調降本身更嚴。要求款項紀錄可直接建退款。
- 兩條路徑並存的疑慮：免核可入口是否被濫用、與訂單異單退款路徑的職責如何切分。

## 決議與理由（2026-06-02，訂單收退款模型重構 序列協作 Phase 4）

**決議**：採訂單收退款模型重構——訂單詳情頁「款項紀錄」可直接建退款款項，無需主管核可，**推翻 2026-05-07「退款一律走訂單異動單」決策**。

兩條退款路徑並存且職責切分明確：

- **款項紀錄直建退款**：完成前減量（少收 / 退已收）的金流落地，免主管核可，弱把關事後可見（對齊 [[ORD-030-完成前明細直接增減真實業務流程待訪談|ORD-030]] 訪談：完成前調降弱把關、主管事後知會不擋操作）。
- **訂單異動單退款（負項訂單異動）**：需主管核可，承載「應收調整生效」的審核閘門；核可即推「已執行」+ 應收減 N（移除「退款款項累計達訂單異動單金額才推進」回退機制，見 [[ORD-003-取消退款Payment是否回退OA|ORD-003]]）。

**理由**：完成前金額調降採弱把關（訪談確認主管事後可見、不阻擋），退款是該調降的金流落地，把關強度應與調降本身一致，不宜更嚴。款項紀錄直建退款讓業務在完成前快速處理少收 / 退款，符合實務節奏；需主管審核的應收調整另由負項訂單異動承載。完成後的退款仍走售後服務單（內含負項訂單異動），不適用本路徑。

**決策者**：Miles（2026-06-02，序列協作 Phase 1-4 收斂）

**落實**：本 change `refactor-order-receivable-refund-model`（退款金流遷至款項紀錄、`createRefundPayment` 操作；款項紀錄區塊提供直建退款入口）
