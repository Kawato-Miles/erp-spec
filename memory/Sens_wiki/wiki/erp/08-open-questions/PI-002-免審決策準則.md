# PI-002：免審決策準則

## 問題描述

`PrintItem.skipReview` 為布林欄位，業務 / 諮詢於需求單階段決定，免審路徑跳過審稿人員分配，直接產生 `source=免審稿` 的合格 Round。

但 **spec 未明確定義「哪些情境算免審」**，由業務 / 諮詢主觀判斷，可能造成：

- 不該免審的案件被免審 → 製作後發現稿件問題
- 該免審的案件走完整審稿 → 浪費審稿產能

## 涉及範圍

- 模組：prepress-review、quote-request
- 相關卡：[[免審決策樹]]、[[難易度機制]]、[[審稿分配規則]]、[[稿件管理規則]]
- 影響範圍：
  - 業務 / 諮詢免審決策一致性
  - 審稿命中率與產能
  - 客訴風險（免審後出問題）

## 討論記錄

Phase B 撰寫 04-business-logic/免審決策樹.md 時識別出此議題。建議釐清以下情境是否屬免審：

- 重複下單（同一規格已審過）？
- B2B 大客戶免審政策？
- 標準規格商品（無設計變動）？
- EC 自動商品（無自由設計欄位）？

## 待解答

- [ ] 列出免審情境清單（規約化）
- [ ] 免審決策的責任歸屬（業務 vs 諮詢 vs 主管）
- [ ] 免審後若出問題，事後追溯與責任分攤

## 候選方案

### 方案 A：建立免審情境白名單（規約化）

- 與業務 / 審稿主管 / 諮詢主管討論，列出明確免審情境
- 行動：寫入 [[免審決策樹]] 並由系統檢核（業務勾選免審 → 顯示適用情境清單）
- 優點：客觀、可稽核
- 缺點：難窮舉，邊界案件多

### 方案 B：免審需主管核可

- 業務勾選免審 → 觸發審稿主管 / 業務主管核可
- 優點：把關效果好
- 缺點：增加流程環節，違背「快速路徑」原意

### 方案 C：依客戶類型 / 歷史記錄自動建議

- 系統依客戶過去稿件審核結果，自動建議是否免審
- 優點：減輕業務判斷負擔
- 缺點：需歷史資料 + 邏輯設計
