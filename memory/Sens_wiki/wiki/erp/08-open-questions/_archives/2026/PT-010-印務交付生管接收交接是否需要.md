# 問題描述

生產模組架構設計**拿掉 task（任務）層**後，原任務狀態的「待交付 → 已交付」所承載的「**印務交付 → 生管接收分派**」交接動作，是否為真實現場需求？

此交接點由 PM 設想、**未經現場確認**。若現場沒有這個正式交接（印務在排程中心把任務排進設備佇列後，生管直接建工作包派工即可），拿掉 task 後就不需要補任何交接機制；若現場確實有「印務正式交付一批、生管正式接收」的動作，則需在工單或生產任務層補對應的狀態/通知。

# 待解答

- [ ] 現場是否有「印務交付 → 生管接收」的正式交接點？還是印務排程後生管直接建工作包即可？
- [ ] 若有，這個交接對應什麼實際動作（通知？簽收？）？掛在工單還是生產任務層？

# 候選方案

- 方案 A（傾向）：**無正式交接**——印務排程（設備佇列）排好後，生管直接從生產任務建工作包派工，不需顯式「交付/接收」動作。拿掉 task 後不補機制。
- 方案 B：**有正式交接**——需在工單或生產任務層補「待派工 → 已派工」或交付通知。

# 決議（2026-06-14，Miles 拍板）

**採方案 B 方向：交付追蹤真實需要、任務（交付包）層保留**（原一度誤判為「拿掉 task 層」，已修正）。

- 任務（交付包）＝印務按**負責廠商**分包（自有一包、各外包一包）的交付追蹤單元；外包據此獨立追進度、發紙本派單。
- 「工單已交付」＝印務確認**各負責廠商都收到**後才標（取最落後：全部任務交付才推進）、下游才看得到。
- 任務（印務交付追蹤）與 [[工作包]]（生管派工給師傅）是**並存的兩條正交分組、無直接關係**。
