# PT-019：EC 消費端計價公式實作核對

## 背景

2026-07-23 全面梳理後端 sens-print-core 確認：後端只承載計價設定與 options 發布（主檔 → S3 JSON → 消費端），本身沒有成本計算引擎；實際售價計算在消費端平台（EC 前台，程式不在已盤點的兩個 repo）。[[BOM結構]] 的材料計價公式依資料模型參數與紙業慣例整理（重量計噸價公式、令價計令重磅數換算 0.000711、單張計直取），與模型一致但未經消費端程式碼直接對證。

## 問題

1. EC 消費端對重量計／令價計的實際計算式是否與 [[BOM結構]] 記載一致（含 0.000711 常數、一令 500 張）？
2. 「每台頁數」（預設 8）在消費端公式中怎麼用：書冊頁數 → 台數 → 用紙張數的換算鏈是否如推定？
3. 數位印刷 399mm 小圖／大圖分界、平版每色價 × 色數、大圖才數折扣乘數的套用順序，消費端實作為何？
4. 後端 weight_entry 價格欄的 db 註解寫「磅價（元/t）」，單位標注與令價計（元/磅）／重量計（元/t）的雙語意混用，建議後端修註解。

## 解答（2026-07-23，sensation-api dev 分支 product/pricing 逐式對證）

1. 計價引擎在 EC 後台 sensation-api `product/pricing/`（非前台）：calc-price 供後台商品設定驗算，前台下單共用同一份計算，售價一致（Miles 確認）。M1 重量計（面積㎡×克重÷1,000,000×噸價）、M2 令價計（0.00071117、一令 500 張）與 [[BOM結構]] 記載完全一致。
2. 每台頁數用途確認：書冊內頁大紙張數＝ceil(頁數 ÷ 每台頁數) × 本數，由引擎於拼版階段算出；亦為 B1 台數裝訂的台數換算輸入（B1 現況未啟用）。
3. 套用順序確認：拼版選優（成本計＝逐方案材料＋工序＋設備總成本最低；數量計＝母版張數最少＋區間表選設備）→ 部件成本加總 → 加成（成本加成 ×(1+r%) 或毛利率加成 ÷(1−r%)）→ 急件 ×(1+adj%) → 四捨五入取整。E1 每色貢獻（CMYK 固定 4 色、特別色與金屬色以彩色價乘倍率）、E2 399mm 任一邊判定、E3 才數 ceil(長cm×寬cm÷900) 均已入 [[BOM結構]]。
4. 後端 weight_entry 價格欄 db 註解「磅價（元/t）」單位混用仍建議後端修正（唯一殘留，屬程式註解衛生、不影響商業邏輯）。

**誰接住**：完整計算框架已回填 [[BOM結構]]「計價引擎計算框架（正本）」段；引擎中標注 pending 的三項（平版版費與特規加價、數位單雙面倍率、B1 台數裝訂）在該段逐項標明「現況未實作」。

## 觸發解決時機建議

- 已解決。原建議：取得消費端程式碼時；工單成本計算引擎設計前。
