# AR-10 主管覆寫分派是否允許破例派工

## 問題

OpenSpec spec v1.5 對「覆寫分派」與「自動分派」採取**不一致的能力門檻策略**：

- **自動分派（spec § L77-86「能力不足破例派工」）**：當無任何審稿員能力等級足夠承擔印件難易度時，系統 SHALL 派工給能力最高者，並在 ActivityLog 記「破例派工」事件
- **主管覆寫分派（spec § L102-108「覆寫至能力不足者被拒」）**：當主管選擇能力不足的審稿員時，系統 SHALL 拒絕並提示能力不足

erp-consultant + senior-pm 雙視角同步指出此規格內部不一致：自動可破例，但人工不可破例。

## 影響

- **業務情境衝突**：原審稿員請假、急件需資深審稿員處理特殊客戶、特殊客戶指定審稿員時，主管覆寫常面臨「目標審稿員能力 < 印件難易度」的情況；目前 spec 拒絕，業務上不切實際
- **使用者繞道**：若拒絕覆寫，主管的繞道是先到 US-AR-003 提高該員 max_difficulty_level（影響範圍：其他印件分派門檻也跟著變），再回到 US-AR-004 覆寫，再降回原等級 — 流程不合理
- **設計矛盾**：自動分派的「破例派工」設計精神（找最佳可用人選優於不派）為何不適用於主管手動操作？

## 選項

| 選項 | 規則 | 影響 |
|------|------|------|
| A | 主管覆寫嚴格門檻（保持 spec 現狀）| 主管須繞道 US-AR-003，流程不合理 |
| B | 主管覆寫允許破例 + 強制填「破例原因」+ 警示確認 | 一致性：與自動分派精神對齊；主管承擔判斷責任 |
| C | 主管覆寫允許破例 + 不警示但留 ActivityLog | 不阻擋業務動作；責任靠事後稽核釐清 |
| D | 主管覆寫允許破例 + 需業務主管核可（雙簽核）| 高責任機制，但流程拉長 |

## 暫定處理

- US-AR-004 業務流程 step 4 補一段：「能力門檻驗證策略待 [[AR-10-主管覆寫分派是否允許破例派工|AR-10]] 解答（spec 現狀拒絕，但與自動分派『破例派工』設計矛盾）」
- US-AR-004 成功條件 3 維持「驗證能力足夠」描述，待解答後修正

## 決議（2026-05-23）

**拍板選項 A（嚴格門檻）+ UI 層阻擋強化**：審稿主管重新分配時，**不能選擇能力不足的審稿人員 — 候選清單先擋著**（UI 層預先過濾，比 spec 既有「事後拒絕」更嚴格）。

**設計理由**（補入 spec）：
- 自動派工的「破例派工」（spec L41, L77-84）是**系統不得已的降級** — 無人能力夠時保證印件不卡住，留 ActivityLog 給主管事後監看做人力決策
- 主管手動覆寫是**主動行為**，不應比自動派工更寬鬆 — 避免主管隨意覆寫造成審稿品質問題
- 兩條 path 規則不同是因動機不同（系統降級 vs 主管主動），不是設計矛盾

**spec 修訂內容**：
- MODIFIED「審稿主管覆寫分配」Requirement：
  - 條文補「能力不足者 SHALL 不出現於主管覆寫的審稿人員候選清單」（UI 層阻擋）
  - 補設計理由段 + 實作層落差說明（Prototype 既有 `overrideAssignment` 採事後拒絕，需後續改為 UI 層阻擋）
  - Scenario「覆寫至能力不足者被拒」rename 為「覆寫候選清單預先過濾能力不足者」
  - 新增 Scenario「可選清單為空時 UI 提示」（無人能力夠 + 原審稿員離職時 UI 顯示 actionable 提示）

**Prototype 既有狀態**：
- `overrideAssignment`（[src/utils/prepressReview.ts:461](../../../../sens-erp-prototype/src/utils/prepressReview.ts)）採事後拒絕（`throw new Error('目標審稿人員能力不足')`）
- **待 Prototype 後續改動**：覆寫 Dialog 候選清單預先過濾 `maxDifficultyLevel < printItem.difficultyLevel` 的審稿員
