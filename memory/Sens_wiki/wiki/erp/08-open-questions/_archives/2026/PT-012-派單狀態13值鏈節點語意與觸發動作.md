# PT-012：派單狀態 13 值鏈的節點語意與觸發動作

## 背景

2026-07-23 現況同步發現大陸處理狀態已從舊 14 值鏈重構為 13 值鏈（前端 main、後端 enum、dev 實際資料三方一致），打樣與補印從狀態轉為工單製作類型。wiki [[派單狀態]] 已依現況改寫，但節點的觸發動作主體（誰在什麼時點推進哪一態）系統未強制、也無文件明定，現況為 ERP 端人工下拉推進。

## 問題

1. 「已交付工單」與「已發稿」的動作主體與先後語意：誰做了什麼動作分別觸發這兩態？（介面提示「發稿日期＝管理人員派案給中國供應商的時間」，那「已交付工單」是供應商承接確認？還是內部交辦完成？）
2. 「分批出貨中」的推進時機：第一批出貨即切換？由供應商端或 ERP 端更新？
3. 海外直發三態（分批／全出／送大陸客戶）的使用時機與收尾對帳差異：三態各自的金流與物流收尾有何不同？
4. 各節點典型由誰推進（印務／管理員／中國廠商端）？除端別權限（中國廠商端不可寫入已交付工單／已發稿／已到貨品檢／台灣已入庫、入庫後鎖定）外是否有未寫進系統的作業慣例？

## 解答（2026-07-23，Miles 提供關係圖）

1. 「已交付工單」＝訂單管理人檢查工單內容無誤後切換；「已發稿」＝印務（凹豆）指派供應商時系統自動切換並記錄發稿時間。兩關分離：內容確認在前、派案在後。
2. 「分批出貨中」通常由外包供應商處理，有出貨即可切，後續以 [[貨運單]] 追蹤。
3. 海外直發三態與台灣已入庫為派單終態，後續對帳以撈取終態派單為基準。MES/ERP 設計注意：初步構想派單與工單無關——工單是印件層展開的 BOM（工序、裝訂與材料），派單是因工單中有外包工序要外發，概念上是「派生產任務出去」（已記入 MES 規劃記憶）。
4. 派單管理主責印務（凹豆），廠商負責回報。權限：派案管理頁切狀態＝印務／訂單管理人；貨運單管理頁切狀態＝品檢出貨人員。自動觸發：指派供應商→已發稿、供應商送出運單→運送中分流（海外直發直達終態）、貨物抵台認列→已到貨品檢、QC 通過→台灣已入庫並同步 EC 稿件狀態。

**誰接住**：節點語意與觸發者已全數回填 [[派單狀態]] 正本卡（狀態列舉表、狀態機圖、權限與強制規則段）；MES 構想記入 memory `project_mes_architecture_planning`。
