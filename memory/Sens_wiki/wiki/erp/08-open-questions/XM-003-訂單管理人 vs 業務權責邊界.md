# XM-003：訂單管理人 vs 業務權責邊界

## 問題描述

Notion 核心角色權責 DB 同時定義 [[業務]] 與 [[訂單管理人]] 兩個角色，**兩者都有訂單 R/W 權限**，但職責邊界不清晰：

- **工單異動執行責任**在誰？業務、印務主管、訂單管理人？
- 訂單管理人是否能修改業務建立的訂單？反之？
- 客戶溝通職責歸屬（補單 / 變更請求）

## 涉及範圍

- 模組：order-management、work-order、cross-module
- 相關卡：[[業務]]、[[訂單管理人]]、[[訂單]]、[[工單]]、[[印件生產流程]]
- 影響範圍：訂單異動流程設計、工單異動發起角色、合批派工功能設計（OQ XM-004 相關）

## 討論記錄

Phase A 角色清單對齊識別出此議題。Notion 角色 DB 中：

- [[業務]]：「補單與規格異動是日常常態」 — 暗示業務發起異動
- 訂單管理人：「工單分配效率、整體進度可見、合批派工功能減少重複整理」 — 暗示訂單管理人負責跨工單合併

但實際操作時 Slack / 紙本流程可能讓兩者都動訂單與工單，責任歸屬不明。

## 部分拍板（2026-06-11）

Miles 裁決：**訂單管理人類似印務主管，現階段當同一個位置，卡片合併**——訂單管理人卡已併入 [[印務主管]]（追蹤訂單進度、確認印務分配狀況入其職責）並刪除。原待解中已隨其他拍板解消的：訂單異動發起者＝業務／諮詢（金額異動，見 [[訂單異動規則]]）；工單異動＝印務發起、生管確認（見 [[任務狀態]]）；跨業務接手＝業務主管改派負責業務（XM-008 已定）。

## 殘留待解

- [ ] **印務主管（原訂單管理人）對訂單的編輯範圍**：[[訂單]] 實體卡「可以編輯的人」沿襲舊權責表含訂單管理人，但印務主管的調度視角應為唯讀追蹤還是可編輯訂單內容，待確認
- [ ] 合批派工的負責角色與合併邏輯 → [[XM-009-合批派工合併邏輯|XM-009]]

## 候選方案

### 方案 A：業務發起，訂單管理人協調

- 業務：所有訂單 / 工單異動的主要發起人
- 訂單管理人：協調跨訂單派工、不主動異動

### 方案 B：兩者都可發起，依異動類型分工

- 規格 / 數量異動 → 業務
- 派工 / 時程 / 工序合併 → 訂單管理人
