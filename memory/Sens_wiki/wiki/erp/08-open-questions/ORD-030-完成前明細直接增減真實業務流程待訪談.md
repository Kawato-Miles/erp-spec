# ORD-030：完成前明細直接增減的真實業務操作流程待訪談

## 問題描述

訂單收退款模型重構（訂單收退款通用方案）假設「訂單狀態 = 訂單完成前，明細（印件費 / 其他費用）可直接增減（含調降）」。但此操作模式在 wiki 全無 know-how 卡記載（pre-check 雙領域稽核 2026-06-02 攔截：屬「憑空新增概念、無既有 know-how 比對基礎」反模式）。須先業務訪談確認真實狀況，否則進 propose 寫 spec 會架空，序列協作 ERP 顧問亦無設計依據。

## 涉及範圍

- 模組：order-management（明細編輯）+ 跨 billing-cash（減量退款）
- 相關卡：[[../04-business-logic/訂單異動規則]]、[[../05-entities/訂單]]、[[../05-entities/印件]]、[[../04-business-logic/對帳一致性]]
- 影響：訂單收退款模型重構 明細時點分界的可行性、退款保護強度、序列協作設計依據

## 待解答（業務訪談）

- [x] (a) 完成前（含製作中 / 出貨中）誰在什麼情境直接改明細金額？多常見？
- [x] (b) 調降（少收）實際要不要經會計 / 主管把關？降價的理由 / 額度，現有沒有紀錄或核准慣例？（關係訂單收退款模型重構「明細調降放棄送審」是否丟失現有把關）
- [x] (c) 完成前已收款後減量退款的實務怎麼走？完成前就收滿款多常見（多是訂金未收滿，還是常全額預收）？收滿後減量退錢誰核准、多久退？
- [x] (d) 售後補印向客戶加錢的實際頻率與走法？（訂單收退款模型重構 假設「多由公司吸收、極少加錢」）

## 阻擋關係

- **阻擋 propose 寫 spec**（明細時點分界為訂單收退款模型重構 核心設計依據）
- 連動 [[ORD-027-訂單額外費用凍結時點與審核通過成交鎖定對齊|ORD-027（報價鎖定時點）]]、[[ORD-025-完成後非售後補收入口歸屬|ORD-025（完成後補收入口）]]、[[BI-9-補收OA立即執行對稱破壞表述|BI-9（OA 已執行語意）]]

## 決議與理由（2026-06-02 業務訪談）

**決議**：訪談 confirm 原本訂單收退款模型重構 成立，用原本訂單收退款模型重構、不另加閘門。

訪談實況（Miles 回答 a/b/c/d）：
- (a) 情境：客戶加印 / 追加數量、改規格連動價格、報價打錯修正、議價調降（增減都有）；業務 + 諮詢都會改。
- (b) 調降弱把關：主管事後知會 / 看得到、不擋操作；紀錄看情況、不一定留。
- (c) 完成前多數訂金未收滿 → 減量多是「少收（調降應收）」、退已收罕見；真要退已收的錢要主管核可，走原本訂單收退款模型重構 的 OA 退款路徑。
- (d) 完成後補收幾乎不加（公司吸收為主）→ OA 正向極少；真加錢走法看情況（次要、保彈性）。

**對訂單收退款模型重構 的影響**：完成前明細直接改管「少收（調降應收）」、弱把關事後可見；退已收的錢走 OA 退款路徑（既有主管核可）。OA 收斂為單向退款 + 2C 整筆。訪談未推翻訂單收退款模型重構，亦不需新增閘門。
