# PI-005 大貨參考打樣稿件的免審觸發機制

## 問題描述

打樣與大貨一律拆兩個印件、兩份稿件。但實務上打樣通過後，大貨稿件不重新上傳，業務在大貨稿件寫「參考打樣稿件」以省審稿人力——本質是大貨印件免審。

問題在觸發時點：現有 [[免審決策樹]] 的免審是**需求單階段由業務標記、轉訂單後鎖定不可改**；而大貨「參考打樣稿件」的免審是**打樣通過後（訂單早已成立）**才產生，現有機制的時點接不住。與 AR-8／PI-002 不同——那兩張講「哪些情境算免審、怎麼核可」（適用條件與準則），本卡講「打樣後晚觸發的免審，現有機制塞不進去」（觸發機制與時點）。

## 待解答

- [ ] 大貨免審走哪條路：擴充 [[免審決策樹]]（放寬「轉訂單後鎖定」、新增打樣後晚觸發免審），還是當 [[打樣決策與重新打樣]] 的打樣 OK 收斂動作獨立處理、不歸免審決策樹管？
- [ ] 大貨免審是系統自動（打樣判 OK 即自動讓大貨免審直通合格／可製作），還是業務手動標記？
- [ ] 純打樣訂單（無大貨）、純大貨訂單（無打樣）下，此機制是否各有差異？

## 候選方向（待拍板，不預設答案）

- 方向 A：擴充免審決策樹，免審邏輯集中一處；代價是破壞該卡「需求單階段鎖定」前提，恐自相矛盾。
- 方向 B：當打樣決策收斂動作，打樣 OK 後由業務把大貨印件審稿維度標合格／可製作、稿件填「參考打樣稿件」；觸發脈絡乾淨、不污染免審決策樹。

（釐清階段 Claude 傾向 B，理由：重複品項免審與打樣後參考是兩種觸發脈絡。最終由系統設計 change 裁決。）

## 待 change 回補標記（本 OQ 拍板後執行）

- [[打樣流程]] 規則段已於 2026-06-13 更正商業現況（大貨參考打樣稿件、不重新上傳重審）；待本 OQ 拍板的是「此免審的系統觸發機制」（方向 A／B），非商業描述。
- [[打樣決策與重新打樣]] 情境卡步驟 3-4 現寫「打樣 OK → 大貨印件開放上傳大貨稿件 → 進入審稿」，與現況不符——待本 OQ 拍板後回補。
