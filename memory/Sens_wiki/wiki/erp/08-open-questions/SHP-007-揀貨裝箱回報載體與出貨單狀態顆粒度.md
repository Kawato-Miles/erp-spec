# SHP-007：揀貨裝箱回報的系統載體與出貨單狀態顆粒度

## 問題描述

生產階段流程重整（2026-07-08 拍板）已定商業層：成品最終品檢通過即入庫（可出貨，無庫存概念）、業務建立出貨單即揀貨指令、[[揀貨人員]]依單打包裝箱並回報裝箱內容、出貨單狀態顆粒度朝 EC 現況四態融合（未處理／打包中／待出貨／已出貨）。**系統載體與操作流程尚未設計**，Miles 明示另議。

## 待解答

- [ ] 裝箱內容回報記在哪：出貨單上的欄位／動作、獨立裝箱明細、或沿用 [[報工紀錄]] 既有「入箱照」掛點？（三者語意重疊，須收斂為一處）
- [ ] 出貨單狀態列舉的四態融合細節：新列舉全集、與既有「異常／已作廢」的關係、揀貨回報如何推進狀態（[[出貨單狀態]] 現行三態正本待此拍板後改寫）
- [ ] 揀貨人員的操作介面與流程：在哪個畫面看到揀貨指令、怎麼回報（箱數／每箱內容／照片）
- [ ] [[報工紀錄]] 的「入箱照」欄位與提交管道 enum 是否隨五角色回報體系擴充或搬移
- [ ] 揀貨錯誤（裝錯箱、漏裝）的更正流程

## 討論記錄

**2026-07-08（Miles）視角拍板＋現況素材**：

- **揀貨以訂單為角度進行**：情境＝業務要求出貨 a／b 印件各 200 個，揀貨依此執行——與既有 [[出貨單狀態]]「業務於訂單建立出貨單、選印件×數量（可同訂單多印件合箱）」的訂單層模型一致，載體設計以出貨單為中心展開。
- **現況 Slack 回報欄位雛形**（Miles 提供截圖，品檢完成訊息）：工單編號／案名／業務・印務標註／款式／**到貨數量／出貨數量／箱數／箱規（如馬克杯箱）**＋成品照、秤重照、裝箱照。現況此回報由品檢完成時一併發出（舊「打包即入庫」實態）；新流程拆為品檢回報（通過數）與揀貨回報（裝箱內容）兩段，欄位分家是本卡載體設計的核心題。
- **[[SHP-005-分批出貨觸發節點|SHP-005]] 併場**：現況實務為業務告知印務要先出多少；系統表達與本卡同場討論。
- 揀貨要清點數量已拍板（[[SHP-008-揀貨是否驗數量|SHP-008]] 決議，已封存）。

**2026-07-20（Miles）裝箱實務補充＋前提裁決**：

- [[SHP-009-出貨單與印件關聯基數矛盾|SHP-009]] 已拍板封存：合箱成立，出貨單＝主單＋出貨明細（印件×數量多筆），本卡載體設計以此為前提。
- **裝箱方式的決定權**：業務有指定就照業務的（如「200 個一箱」，填在出貨單告訴揀貨）；業務沒指定，揀貨人員自行依公版箱規判斷。
- **揀貨回報三要素**：裝幾箱、共多少個、附照片（現況另有秤重照）。
- **箱規＝公司公版箱體**，對應第三方物流的箱數計價（大箱每箱 x 元、小箱每箱 y 元）。**運費的帳先不做**——訂單通常已先收運費；箱數記錄的用途是**每月與第三方物流對帳**：拿自己出貨的箱數與第三方請款結果比對，確認雙方認知運費相同（對帳機制屬後續設計，本階段只確保箱數／箱規有留紀錄）。
- **品檢與裝箱回報確定拆開**：現況兩者混在同一則 Slack 訊息發（品檢完成訊息夾出貨數量／箱數／箱規），造成出問題時難以定位；新流程拆為品檢回報（通過多少）與揀貨回報（裝了幾箱共多少個）兩段。
