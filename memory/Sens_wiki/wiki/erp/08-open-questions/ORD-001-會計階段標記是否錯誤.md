# ORD-001：會計階段標記是否錯誤

## 問題描述

Notion 核心角色權責 DB 標記 [[會計]] 角色「**僅參與審稿階段**」，但實際職責是：

- 線下金流查帳
- 發票開立

兩者與「審稿階段」無直接關聯。需確認此標記是否為：

1. **錯誤標記**（應改為「訂購階段」「打樣 / 印製階段」「出貨階段」）
2. **特殊業務邏輯**（如審稿階段才確認帳款？）
3. **資料模型誤解**（「階段」一詞在 Notion DB 有特殊定義）

## 涉及範圍

- 模組：order-management（付款 / 發票）、prepress-review（若真有關聯）
- 相關卡：[[會計]]、[[付款發票邏輯]]、[[訂單狀態]]
- 影響範圍：Notion 角色 DB 修正、會計操作流程 spec 設計

## 討論記錄

Phase A 建 03-roles/會計.md 時注意到此不一致。會計實際操作集中在「訂單應收記錄、發票開立、退款 / 折讓處理」，這些都跨越評估 / 訂購 / 審稿 / 打樣 / 出貨 全程，不應只限「審稿階段」。

## 待解答

- [ ] 確認 Notion DB 的標記是否錯誤
- [ ] 若錯誤，應改為哪些階段
- [ ] 若不是錯誤，背後業務邏輯為何

## 候選方案

### 方案 A：Notion DB 標記錯誤 → 修正為「全階段」

- 行動：更新 Notion 核心角色權責 DB 該欄位
- 影響：Vault 03-roles/會計.md 同步更新

### 方案 B：保留現狀，補充說明欄位語意

- 行動：在 Vault 卡中補註「審稿階段」實際含義（可能指「審稿階段付款 / 發票確認」）
- 影響：Vault 卡補強，不動 Notion

## 決議（2026-06-11）

採方案 A 的精神並隨範本演進自然消解：Miles 拍板（2026-06-11）會計**主責公司現金流**（月結對帳、依業務主管月退款報表執行退款、特例才協助開票）——涉及錢的環節全程參與，「僅審稿階段」確為錯誤標記。角色範本 v2 已廢除「參與階段」欄位（平台與階段屬實作層），[[會計]] 卡依 v2 重寫後概述直接寫明職責全貌，問題載體消失。2026-05-28 對談（月結對帳節奏）為佐證來源。Notion 角色權責 DB 的對應欄位待下次對外同步時一併修正。
