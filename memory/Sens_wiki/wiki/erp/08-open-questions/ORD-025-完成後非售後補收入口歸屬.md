# ORD-025：訂單完成後「非售後性質補收」的入口歸屬

## 問題描述

2026-05-29 bug fix 依 spec L1302 擋掉「訂單完成 / 已取消」訂單於訂單詳情頁「訂單異動」Tab 的「新增訂單異動單」入口，完成後金額異動統一改走「售後服務」Tab 建立 AfterSalesTicket（其內再建關聯 OrderAdjustment）。

但 AfterSalesTicket 的 case_category 為客訴 / 不良 / 規格不符等「售後問題」性質。當「客戶在訂單完成後又要求加印 / 追加數量」這類**非售後問題的補收**發生時，語意上不全屬「售後」，卻因新增閘門被導向售後服務單，可能造成 case_category 語意污染或業務認知混淆。

## 涉及範圍

- 模組：order-management（訂單異動入口）、after-sales-ticket（售後容器 case_category）
- 相關卡：[[售後服務]]、[[付款發票邏輯]]
- 影響範圍：完成後補收的發起動線、AfterSalesTicket.case_category 語意純度、對帳與發票歸屬

## 討論記錄

- spec L1302 既定：訂單已完成後售後事件改走 AfterSalesTicket，業務不再於訂單詳情頁直接建 OA。
- 本次修正將此規則落實到 UI gating（既往 prototype 漏接，導致完成後仍可直接於訂單異動 Tab 新增 OA）。
- 落實後浮現語意邊界問題：「完成後加印」屬新需求補收，非客訴 / 不良 / 規格不符。

### 2026-06-02 訂單收退款模型重構 引導文案校正（方案 B 方向確立、補收 OA 實作未動）

訂單收退款模型重構（`refactor-order-receivable-refund-model`）釐清完成後金額異動路徑，實質傾向方案 B：

- **退款 / 補印** → 建立售後服務單（內含負項訂單異動 + 補印印件）。
- **補收（向客戶收錢）** → 建立新訂單（視為新需求），非走訂單異動 Tab、非補收訂單異動。

本次僅改 Prototype 引導文案（把完成後補收導向開新訂單），**spec 既有「補收訂單異動（正項）」實作未動**。因此本 OQ 仍有未收斂張力（見下「待解答」），不結案。

## 待解答

- [ ] 完成後「非售後問題補收」（如客戶事後加印）應走：(a) 售後服務單（沿用既有唯一完成後入口）/ (b) 新開訂單（視為新需求）/ (c) 其他入口？（訂單收退款模型重構 引導文案已傾向 (b)，待正式拍板）
- [ ] 若走售後服務單，是否需為此類補收新增 case_category 值（如「客戶追加」），避免污染既有售後語意？
- [ ] 若走新訂單，與原訂單的關聯 / 對帳 / 客戶體驗如何處理？
- [ ] spec 既有「補收訂單異動（正項）」與實務「補收 → 開新訂單」認知衝突：訂單收退款模型重構 只改引導文案、未動補收訂單異動實作。補收訂單異動是否應廢除（完成後補收一律開新訂單），或保留作其他情境（如完成前補收）？兩者並存的邊界為何？

## 候選方案（初步）

### 方案 A：一律走售後服務單（現況）
- 優點：單一完成後入口、實作最簡、對齊 spec L1302
- 缺點：case_category 語意污染（加印非售後問題）

### 方案 B：非售後補收改新開訂單
- 優點：語意乾淨（新需求 = 新訂單）、對帳獨立
- 缺點：與原訂單關聯斷裂、客戶可能期待同訂單延續

### 方案 C：售後服務單擴 case_category 值（如「客戶追加」）
- 優點：保留單一入口 + 語意可辨識
- 缺點：售後容器承載非售後語意，概念邊界仍模糊
