# 帳務（Order Billing）

> 一句話定位：管理一張訂單的「應該收多少」「開了多少發票」「實際收到多少」三條線，讓業務有操作彈性、靠對帳確保最終一致。本卡承載帳務領域所有實體的業務可見欄位表。行為規則正本在 [[對帳一致性]]、[[付款發票邏輯]]、[[訂單異動規則]]，本卡只引用不重述。

## 這張卡要回答的問題

- 帳務領域有哪些單據？彼此怎麼關聯？
- 一張發票上會記哪些資料？
- 收款紀錄怎麼跟請款期次對應？
- 折讓單是什麼、什麼時候開？

## 營運背景

印刷廠一筆訂單的錢分三條線各自記：業務先算出「這筆訂單客戶應該付多少」（應收）、開了幾張發票合計多少（發票）、實際收到客戶匯來多少（收款）。這三條線是不同人、不同時間、用不同單據各自填進系統的，很容易某一條漏記或填錯。公司的做法不是事先用硬性規定把三條線綁死，而是讓業務照客戶實際狀況自由處理，最後靠對帳把三個數字湊在一起比對——對得起來代表帳沒問題，對不上就要有人去查。「三個數字最後一定要相等」就是這個領域不能退讓的底線。

## 欄位（業務可見）

> 按實體分子表。純技術欄位（id / FK / 時間戳記）不列入。

### 帳務公司（BillingCompany）

帳務公司是發票開立的法律主體，每家對應一個藍新（ezPay）商店帳號。系統管理員維護，業務 / 諮詢 / 會計僅讀取選用。

| 介面欄位 | 說明 | 欄位形態 | 來源 | 可否修改 |
|---------|------|---------|------|---------|
| 公司名稱 | 帳務主體名稱 | 文字 | 系統管理員設定 | 可（系統管理員） |
| 統一編號 | 營利事業統一編號 | 文字 | 系統管理員設定 | 可（系統管理員） |
| 藍新商店代號 | 對應藍新（ezPay）商店帳號；發票開立／折讓／作廢時送藍新平台使用，同時間須唯一（兩家不可共用，以免違反藍新自訂編號不可重覆） | 文字 | 系統管理員設定 | 可（系統管理員） |
| 地址 | 公司登記地址 | 文字 | 系統管理員設定 | 可（系統管理員） |
| 電話 | 聯絡電話 | 文字 | 系統管理員設定 | 可（系統管理員） |
| 是否預設 | 同時間僅允許一筆為預設 | 是 / 否 | 系統管理員設定 | 可（系統管理員） |
| 啟用狀態 | 停用後不出現在下拉選項 | 是 / 否 | 系統管理員設定 | 可（系統管理員） |

### 發票（Invoice）

發票透過藍新平台開立，對應台灣統一發票。一張發票掛在一張訂單下、對應一個請款期次。

| 介面欄位 | 說明 | 欄位形態 | 來源 | 可否修改 |
|---------|------|---------|------|---------|
| 發票種類 | B2B / B2C | 單選 | 請款期次預填或業務選擇 | 開立前可改 |
| 買受人名稱 | 客戶公司或個人名稱 | 文字 | 訂單客戶資料預填 | 開立前可改 |
| 買受人統編 | B2B 必填、B2C 留空 | 文字 | 訂單客戶統編預填 | 開立前可改 |
| 買受人地址 | 選填 | 文字 | 訂單客戶地址預填 | 開立前可改 |
| 買受人信箱 | 選填 | 文字 | 訂單客戶信箱預填 | 開立前可改 |
| 載具類別 | 手機條碼 / 自然人憑證 / ezPay 載具（B2C 適用）| 單選 | 業務選擇或客戶預設 | 開立前可改 |
| 載具編號 | 載具類別對應的編號 | 文字 | 業務輸入 | 開立前可改 |
| 捐贈碼 | B2C 捐贈發票用；3-7 碼純數字。與載具互斥等開立規則見 [[發票法規硬約束-ezPay-MIG]] | 文字 | 業務輸入 | 開立前可改 |
| 課稅別 | 應稅 / 零稅率 / 免稅 | 單選 | 訂單課稅設定預填 | 開立前可改 |
| 稅率 | 依課稅別自動帶入 | 數字 | 系統依課稅別自動帶入 | 不可（隨課稅別變動） |
| 銷售額（未稅）| 系統自動計算 | 金額（未稅） | 系統依商品明細自動計算 | 不可 |
| 稅額 | 系統自動計算 | 金額 | 系統自動計算（銷售額 x 稅率） | 不可 |
| 發票金額（含稅）| 發票總計金額；業務可微調 | 金額（含稅） | 系統自動計算或業務微調 | 開立前可改 |
| 備註 | 自由填寫 | 文字 | 業務輸入 | 開立前可改 |
| 狀態 | 發票目前的開立狀態（開立 / 作廢） | 單選 | 系統自動與手動觸發 | 依狀態機規則 |
| 發票號碼 | 藍新金流回傳；開立後唯讀 | 文字（唯讀） | 藍新金流回傳 | 不可 |
| 作廢原因 | 作廢時必填（限中文 6 字或英文 20 字） | 文字 | 作廢操作時業務輸入 | 作廢時可改 |
| 折讓累計 | 該發票所有已確認折讓單金額合計 | 金額 | 系統自動累計 | 不可 |
| 剩餘可折讓 | 發票金額 - 折讓累計 | 金額 | 系統自動計算 | 不可 |
| 折讓衍生標示 | 無折讓 / 已部分折讓 / 已完全折讓 | 單選 | 系統自動計算 | 不可 |
| 商品明細 | 見下方「發票品項」；可從期次品項一鍵繼承或從訂單印件預填 | 子結構 | 請款期次預填或業務編輯 | 開立前可改 |

**發票品項**（InvoiceItem，內嵌於發票的商品明細）：

| 介面欄位 | 說明 | 欄位形態 | 來源 | 可否修改 |
|---------|------|---------|------|---------|
| 商品名稱 | 上限 30 字；例：彩色名片 | 文字 | 預填或業務輸入 | 開立前可改 |

| 商品單位 | 張 / 本 / 冊 / 份 / 個 / 卷 / 盒 / 套 / 批 / 式 / 組（共 11 項） | 單選 | 預填或業務選擇 | 開立前可改 |
| 商品單價 | B2B 為未稅、B2C 為含稅 | 整數 | 預填或業務輸入 | 開立前可改 |

### 折讓單（SalesAllowance）

折讓單是發票已開立後的部分減額憑證（依台灣統一發票使用辦法第 20 條）。折讓本身不自動建立退款，業務先建退款收款紀錄再開折讓，兩者以訂單活動紀錄串聯。

| 欄位 | 形態 | 說明 |
|------|------|------|
| 折讓金額（含稅）| 金額 | 業務輸入正數 |
| 折讓金額（未稅）| 金額 | 雙欄計價 |
| 折讓後剩餘發票金額 | 金額 | 藍新回傳（自動計算）|
| 折讓原因 | 文字 | 必填 |
| 狀態 | 單選 | 已確認 / 已作廢 |
| 折讓單號 | 文字 | 用於對帳的折讓單號碼；開立後唯讀 |
| 作廢原因 | 文字 | 作廢時填寫 |
| 回簽附件 | 檔案上傳（多檔）| 折讓回簽文件（合併自 SalesAllowanceFile）|

### 收款紀錄（Payment）

款項紀錄記錄客戶每一筆實際付款或退款。款項類型為「退款」時與一般收款分軸。

| 欄位     | 形態       | 說明                                                                                   |
| ------ | -------- | ------------------------------------------------------------------------------------ |
| 款項類型   | 單選       | 定義此筆款項為收款或退款（收款 / 退款）                                                                |
| 款項狀態   | 單選       | 處理中 / 已完成 / 已取消；處理中與已取消不計入收款淨額                                                       |
| 金額     | 金額       | 款項金額（一律以正值輸入）                                                                        |
| 款項方式   | 單選       | 現金 / 信用卡 / 銀行轉帳 / 支票 / 其他                                                            |
| 付款序號   | 文字       | 銀行入帳序號 / 信用卡授權碼 / 支票號碼（入帳憑證）                                                         |
| 退款收款帳號 | 文字       | 款項類型為退款時填：客戶收款帳號（戶名／銀行／帳號）。每筆退款當次由客戶提供、可能與付款帳號不同；收款情境留空。與「付款序號」語意分清——前者是出金目標、後者是入帳憑證 |
| 款項時間   | 日期時間     | 退款情境對應退款日期                                                                           |
| 紀錄人    | 文字       | 紀錄操作者姓名                                                                              |
| 備註     | 文字       | 退款原因 / 對帳備註                                                                          |
| 取消備註   | 文字       | 款項狀態為已取消時之說明                                                                         |
| 對帳附件   | 檔案上傳（多檔） | 款項狀態為已完成時，最少需一個檔案                                                                    |

> 處理中超過 7 天的款項紀錄會顯示「老化 N 天」提示。

### 請款期次（BillingInstallment）

請款期次把一張訂單的應收款拆成若干期（如訂金 30%、尾款 70%），每期各自追蹤開票與收款進度。開票維度與收款維度完全獨立，支援先收後開 / 先開後收。

| 介面欄位 | 說明 | 欄位形態 | 來源 | 可否修改 |
|---------|------|---------|------|---------|
| 期別 | 1 / 2 / 3 ...；拆期後兩筆獨立平輩期次各佔一個序號 | 整數 | 系統依建立順序自動給定 | 不可 |
| 描述 | 訂金 30% / 尾款 70% / 加印款 / 諮詢費 等 | 文字 | 業務填寫或預設值 | 可 |
| 預計金額 | 含稅；等於品項明細合計（品項為空時保留手動填入值） | 金額（含稅） | 系統依品項明細自動計算或業務手動填寫 | 可（已開立發票或已收款核銷後不可修改） |
| 預計金額（未稅） | 預計金額的未稅值 | 金額（未稅，唯讀） | 系統自動計算 | 不可 |
| 已收金額 | 該期次已手動核銷之收款金額（系統自動累計，唯讀） | 金額（唯讀） | 系統由核銷分配資料自動累計 | 不可 |
| 已退金額 | 該期次已手動核銷之退款金額（系統自動累計，唯讀） | 金額（唯讀） | 系統由核銷分配資料自動累計 | 不可 |
| 預計付款方式 | 匯款 / 現金 / 信用卡 / 支票 / 其他（規劃值，非事實） | 單選 | 業務建立時選擇 | 可 |
| 預計收款日 | 應收日；核銷「依序填滿」依此由早到晚排序 | 日期 | 業務填寫 | 可 |
| 預計開票日 | 可空；支援尚未決定開票時點的情境 | 日期 | 業務填寫 | 可 |
| 開票狀態 | 未開立 / 已開立 / 已作廢 | 單選（唯讀） | 系統依發票開立狀態自動推導 | 不可 |
| 收款狀態 | 未收 / 部分收款 / 已收訖（依核銷分配累計自動推導） | 單選（唯讀） | 系統依核銷累計金額自動推導 | 不可 |
| 品項明細 | 同發票品項結構；鏈式預填（訂單印件 → 期次 → 發票） | 子結構 | 訂單印件明細帶入或業務編輯 | 可 |
| 備註 | CSV 對帳資料來源；拆期時自動帶前綴 | 文字 | 業務填寫 | 可 |
| 來源類型 | 手動建立 / 諮詢取消 / 諮詢結束無生產 / 報價流失 / 拆期 | 單選（唯讀） | 系統建立期次時依流程標記 | 不可 |
| 原始預計收款日 | 訂單審核通過時凍結為基準 | 日期（唯讀） | 系統在訂單核准時將當時預計收款日寫入 | 不可 |
| 原始預計開票日 | 同上凍結基準 | 日期（唯讀） | 系統在訂單核准時將當時預計開票日寫入 | 不可 |
| 變更次數 | 收款日 + 開票日累計變更次數（自動計算；拆期不計入） | 整數（唯讀） | 系統在收款日或開票日修改時自動累算 | 不可 |
| 是否取消 | 拆期 / 業務取消時標記；取消後不入狀態推導與應收計算 | 是 / 否 | 系統因拆期操作或業務手動取消 | 不可直接修改，需透過取消操作變更 |
| 取消原因 | 取消時填寫（如「拆兩期」「業務取消」） | 文字 | 取消操作時業務填寫 | 取消時可填 |

### 收款核銷分配（PaymentAllocation）

收款核銷分配是款項紀錄與請款期次之間的多對多橋接，由業務或會計手動核銷與分配。

| 介面欄位 | 說明                        | 欄位形態       | 來源          | 可否修改       |
| ---- | ------------------------- | ---------- | ----------- | ---------- |
| 分配金額 | 該筆收款/退款分配至該期次的核銷金額（一律為正值） | 金額         | 業務或會計手動核銷輸入 | 款項狀態未鎖定前可改 |

> 稽核事件（BillingActivityEvent）為純稽核用途的結構化事件記錄，記錄期次建立、收款日變更、開票日變更、拆期、取消、核銷覆寫等操作，供 KPI 量測與審計查詢，不另建欄位表。
