# 諮詢單（ConsultationRequest）

> 一句話定位：客戶透過 surveycake webhook 自動建單，由 [[諮詢]] 人員自我認領處理。**雙出口**：轉需求單 / 諮詢訂單。
> 這張卡負責什麼：寫諮詢單存在的業務理由、兩個出口怎麼走、哪些資料不直覺。完整欄位明細以下方 `implemented-by` 指向的 OpenSpec 規格 § 諮詢單實體與表單欄位為正本，本卡只解釋業務意義，不重抄欄位表。

## 這張卡要回答的問題

- 還沒成交，為什麼要先收一張諮詢單、還先收費？→ 看「營運背景」。
- 諮詢談完有哪幾種結局？金流怎麼處理？→ 看「業務流程（雙出口）」。
- 客戶原話和諮詢人員筆記為什麼要分兩欄記？→ 看「需要解釋的資料」。

## 營運背景

有些客戶需求複雜、還不確定要不要做，會先付一筆諮詢費請公司協助釐清需求。把這段「先收費、談需求、還沒到報價」的過程獨立成諮詢單，是因為它有自己的金流（先收的諮詢費要掛在某個對象上）、自己的認領流程（諮詢人員自我認領），而且結局不只一種——談完可能轉成正式報價（需求單），也可能就此收尾或取消。諮詢單就是承接這段前置互動、並讓那筆諮詢費後續能正確轉移或退還的對象。

## 需要解釋的資料

諮詢單上要記哪些欄位以 OpenSpec 正本為準，這裡只解釋不直覺的：

- **諮詢費**：含稅固定 NT$ 2000（規格與雛形皆為 2000）。先收的這筆錢後續會依結局轉移到對應訂單或部分退還（技術欄位：`consultation_fee`）。
- **客戶原話 vs 諮詢人員筆記（雙欄位設計）**：客戶從 surveycake 填來的原話唯讀、不可竄改（對齊 ISO 9001 客戶指示可追溯原則）；諮詢人員自己跟客戶溝通的記錄另記一欄、可多次編輯。兩者分開記，是為了保留「客戶原話」的可追溯性，不被內部筆記覆蓋（技術欄位：`consultation_topic` 唯讀 / `consultant_note` 可編輯）。
- **諮詢人員**：諮詢人員自我認領時寫入，主管可代為認領，業務主管可改派重新指定（限待諮詢，見 [[CR-4-諮詢單認領後改派狀態限制]] / add-sales-manager-reassign-owner change）（技術欄位：`consultant_id`）。
- **兩個出口的關聯欄位**：轉需求單則關聯 [[需求單]]；完成不做大貨 / 取消則關聯諮詢 [[訂單]]（技術欄位：`linked_quote_request_id` / `linked_consultation_order_id`）。

## 關鍵關聯

- 包含諮詢費收款紀錄（一開始掛在諮詢單上）
- 轉需求單時關聯 [[需求單]]，**收款紀錄轉移至主訂單**；客戶原話 + 諮詢人員筆記以雙區塊格式合併寫入需求單的需求說明
- 完成不做大貨 / 取消時關聯諮詢 [[訂單]]（來路為諮詢訂單）

## 業務流程（雙出口）

```
surveycake webhook → 自動建單（待諮詢）
  ↓ Slack 通知諮詢人員群組廣播待認領
諮詢人員自我認領（寫入諮詢人員；主管可代為認領）
  ↓ 諮詢人員可編輯溝通記錄，客戶原話唯讀
諮詢人員諮詢 → 選擇結局：
  ├── 轉需求單做大貨 → 建需求單（需求說明帶雙區塊預設值），收款轉移
  ├── 完成不做大貨 → 建諮詢訂單（含諮詢費 2000，訂單完成終態）
  └── 取消（待諮詢取消）→ 建諮詢訂單（已取消終態）+ 半額退 1000（收 2000 退 1000）
       + 自動建退款訂單異動（已核可、由系統建）；不自動開發票 / 折讓 / 待開發票
```
