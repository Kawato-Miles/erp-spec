# 售後服務（AfterSalesTicket）

> 一句話定位：訂單**已完成後**的售後容器。取代既有 `OrderAdjustment(phase=after_completion)` 雙重身份（add-after-sales-ticket change 引入）。
> 這張卡負責什麼：寫售後服務單存在的業務理由、會牽連到哪些單據、結案前要滿足什麼。完整欄位明細以下方 `implemented-by` 指向的 OpenSpec 規格 § AfterSalesTicket 實體與欄位為正本，本卡只解釋業務意義，不重抄欄位表。

## 這張卡要回答的問題

- 訂單異動單本來就能加錢退錢，為什麼訂單完成後還要另開一張售後容器？→ 看「為什麼要這張容器」。
- 「責任歸屬」怎麼影響後續處理？→ 看「責任歸屬如何影響後續」。
- 售後退款 / 補印分別怎麼走、要不要重審？→ 看「退款流程」「補印流程」。

## 為什麼要這張容器

訂單做完出貨後，客戶還是可能反映瑕疵、規格不符、物流問題。這些客訴往往一張單要同時牽動「退一筆錢」加「重印一件」加「跟客戶來回溝通」，而且要能反查是哪次客訴引發的。早期是用訂單異動單兼著記，但訂單異動單本來是管「訂單進行中加減錢」的，硬塞售後語意會讓它一單兩用、難追蹤。所以把「訂單已完成後的售後處理」獨立成售後服務單容器：它本身不直接改錢，而是當作一個受理 → 處理 → 結案的外殼，底下再掛 0..N 張退款 / 補收的訂單異動單、0..N 件補印印件。

## 關鍵關聯

- 屬於 `Order` → [[訂單]]（**Order 必須已完成**）
- 包含 `OrderAdjustment[]`：透過 `linked_after_sales_ticket_id` 反向關聯，0..N 個異動單
- 包含 `PrintItem[]`：補印用，透過 `related_after_sales_ticket_id`
- 包含 `Payment[]`：若決議為退款則關聯退款 Payment

## 核心邏輯

- **一個 Order 同時只能有 1 張未結案 ticket**
- **結案後可建新 ticket**
- 與訂單期間異動的區別：
  - 訂單期間：OrderAdjustment 獨立路徑（`phase=during_order`）
  - 訂單已完成後：透過 AfterSalesTicket 容器 → 內建關聯 OrderAdjustment / PrintItem

## 責任歸屬如何影響後續

售後服務單上記一個責任歸屬（公司認賠 / 客戶承擔 / 共同分擔），這項不是純標記——它決定後續要不要向客戶補收費用、還是公司吸收走退款。判到誰要付錢，下游掛的訂單異動單方向（補收 vs 退款）就跟著定。完整的責任歸屬與案件分類列舉值以 OpenSpec 正本為準（技術欄位：`responsibility` / `case_category`）。
