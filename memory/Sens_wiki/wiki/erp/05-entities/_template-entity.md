# 資料卡撰寫範本（entity 層）

> 設計依據：[[index]] § Wiki 憲章（本質、第一讀者、兩判準、BRD 自足、單元定位模式）＋ 與 `03-roles/_template-role.md`、`06-state-machines/_template-state-machine.md` 同構的治理機制。
> 本範本依「雙向開發」迭代：每張卡走查時發現的缺口回饋範本。

---

## 〇、實體卡是什麼（先懂定位再動筆）

> 本節目的：每條規則都從定位推導。Wiki 的本質、第一讀者、兩判準（紙本測試／商業語言）、BRD 自足與「範圍外＝防護欄」見 **[[index]] § Wiki 憲章**——本節只講實體單元的特定定位。

**實體卡是業務可見欄位的正本**——這個業務物件在介面上能看到哪些欄位、跟哪些單據相牽連、會有哪些狀態名。公司用紙本與口頭運作時這套單據結構就存在（訂單上記什麼、跟哪張單對應），完全過紙本測試。

**它是商業知識的「資料」視角投影**（與鄰層分工，投影不互相複寫）：

- 欄位承載的**決策邏輯與計算**（如金額怎麼算、什麼條件免審）正本在 `04-business-logic/`，本卡只記欄位本身與「為什麼要存這個欄位」一句，不複寫公式。
- **狀態怎麼流轉**（轉換條件、觸發事件）正本在 `06-state-machines/`，本卡只記「有哪些狀態名」並 wiki link 指過去。
- **業務過程**（誰先做誰後做）屬服務藍圖（型）與業務情境卡（實例），本卡不寫流程敘述。

**收什麼**：業務使用者在介面上看得到的欄位、實體間的關聯與基數、狀態名。**不收**：純技術欄位（系統識別碼、外鍵、時間戳記）、計算公式、狀態轉換邏輯、業務流程、Prototype 實作細節（資料層搬移、欄位遷移）——這些過不了紙本測試。

**北極星與已知陷阱**：欄位內容是**業務語言**為目標、wiki 維持 **BRD 定位**為原則（兩判準的落地）。最常見的失守——把欄位正本掛在 OpenSpec／本卡不寫欄位表（漂移成 PRD）；把技術欄位名（外鍵、列舉碼）當主詞寫進正文（違反商業語言）；把 change 迭代史用 callout 釘在正文（歷史屬 wiki/log.md）。

**本單元主要伺候的讀者情境**：商業自足理解（這張單上記什麼、業務看得到什麼）、規劃自查（設計功能時這個資料怎麼來、能不能改）、AI 載入依據（欄位詞彙正本、與鄰近實體的邊界）、訪談確認（與業務核對欄位）。

## 一、撰寫流程

```
辨識 → 歸位 → 確認前提 → 撰寫 → 稽核 → 定案
```

| 步驟 | 做什麼 | 這一步的產出 | 卡住時 |
|------|--------|------------|--------|
| **1. 辨識** | 這個業務物件需要開獨立實體卡嗎？依 § 二判斷 | 開卡與否 | 走不通 → § 六 |
| **2. 歸位** | 檔名＝實體名（繁中語意化），放 `05-entities/`；定 module 與 business-domain | 檔名與落點 | 走不通 → § 六 |
| **3. 確認前提** | 依 § 三逐項確認 | 已確認清單（任一缺 → 停下問） | 停下問使用者 |
| **4. 撰寫** | 依 § 八格式填入，遵守 § 四規則 | 卡草稿 | 需要假設 → § 七 |
| **5. 稽核** | 依 § 十二維度自檢（執行者／稽核者分離，見該節） | 逐項勾稽結果 | 不通過 → 回 3 或 4 |
| **6. 定案** | commit ＋ 追加 wiki/log.md（逐卡 `[[卡名]]`＋動機） | — | — |

## 二、判斷表

> 本節目的：實體層三個最常判錯的決策——該不該獨立成卡、某資訊屬哪一層、某欄位收不收。判錯就會長出售後卡那種「沒欄位段、技術 token 滿天、欄位正本掛 OpenSpec」的卡。

**判斷一：這個業務物件該獨立成實體卡，還是併入鄰近實體？**

| 順序 | 問自己 | 否的話 |
|------|--------|--------|
| 1 | 它有**自己的生命週期**嗎？（能被獨立建立、查詢、有自己的狀態） | 不開卡，當鄰近實體的欄位群或子表 |
| 2 | 各環節需要**共同指認**它嗎？（收款、生產、出貨都圍著它轉） | 不開卡，併入主實體 |
| 3 | 併入鄰近實體會造成**一單兩用**嗎？（如售後語意硬塞進訂單異動單） | 能乾淨併入就不獨立 |

**判斷二：某條資訊屬哪一層？**（這是售後卡失守的根源——把規則、狀態、流程都塞進實體卡）

| 這條資訊是…             | 歸屬                   | 本卡怎麼處理                  |
| ------------------ | -------------------- | ----------------------- |
| 介面上看得到的一筆資料        | **本卡欄位表**            | 寫進「欄位（業務可見）」            |
| 這筆資料由什麼組成（具體組成式） | **本卡欄位表**（貼著欄位寫） | 在說明／來源欄寫結論句（「商品小計＋其他費用－折扣」） |
| 為什麼這樣算／什麼條件生效（決策政策） | `04-business-logic/`（保持 high-level 商業用語） | wiki link 指規則卡，不複寫論證 |
| 這個狀態在什麼條件下變成下一個    | `06-state-machines/` | 只記狀態名，wiki link 指狀態機卡   |
| 誰先做誰後做的過程          | `07-scenarios/`      | 不寫，相關連結指情境卡             |

**判斷三：某欄位收進 BRD 欄位表嗎？**

| 問自己 | 是 | 否（不收本體） |
|--------|----|----|
| 業務使用者在介面上看得到嗎？ | 收進欄位表 | → 純技術欄位（外鍵、識別碼、時間戳記）留 prototype／OpenSpec |

## 三、前提確認（MUST 全部已確認才能動筆）

> 本節目的：答不出來的是默會知識缺口——停下問，不准編。

| 前提 | 確認狀態 |
|------|---------|
| 這個實體為什麼要獨立存在（不併入鄰近實體）？ | 已確認 / 待確認 → 停下 |
| 業務可見欄位清單從哪來？（訪談／既有介面／既有規格反向拆出） | 已確認 / 待確認 → 停下 |
| 每個欄位「業務看到什麼、從哪來、誰能改、何時鎖定」？ | 已確認 / 待確認 → 停下 |
| 哪些欄位是非顯然的、需要說明「為什麼要存」？ | 已確認 / 確認沒有 |
| 它跟哪些實體關聯？基數（1:1／1:N／N:M）？ | 已確認 / 待確認 → 停下 |
| 它有沒有狀態？狀態正本在哪張 state-machine 卡？ | 已確認 / 確認無狀態（省略狀態段） |

## 四、撰寫規則

> 本節目的：MUST 不容裁量、SHOULD 偏離須說明、MAY 自由。每條附理由。

| 強度 | 規則 | 理由 |
|------|------|------|
| MUST | 「欄位（業務可見）」段採五欄表：介面欄位／說明／欄位形態／來源／可否修改 | 欄位表是本卡核心職責，五欄是稽核勾稽基準 |
| MUST | **欄位正本在本卡，不指向 OpenSpec／Data Model**；`implemented-by` 只作導航指 prototype 型別檔 | 欄位的增修是商業決策（BRD 層），掛 OpenSpec 即漂移成 PRD |
| MUST | 介面欄位名用**業務在畫面看到的中文字**，不用底線／駝峰命名、不用列舉碼 | 商業語言判準；第一讀者是不做系統的 PM |
| MUST | 每段過紙本測試與商業語言兩判準（§ 〇）；正文零實作細節（資料層搬移、欄位遷移、store 名） | 本卡是 BRD，第一讀者讀不到實作文件 |
| MUST | 非顯然欄位在說明欄補「為什麼要存這個欄位」；欄位的**具體組成式**寫在說明／來源欄（如「印件單價 x 數量自動加總」）；背後的**決策政策**（為什麼、什麼條件）wiki link 指 `04-business-logic/`，不複寫論證 | 組成式屬欄位定義、貼著欄位最具體；政策只寫一次在 04——欄位卡答「有什麼、怎麼組成」、規則卡答「為什麼」 |
| MUST | 寫「範圍外／不收」訊號：易被下游腦補的技術欄位點名「存在但屬實作規格」 | 範圍外是防下游過度設計與捏造的防護欄 |
| SHOULD | 「為什麼獨立存在」附被駁回方案＋駁回理由（如「為何不把售後併回訂單異動」） | 防後人無意推翻有理由的拆分 |
| MAY | 欄位多的實體依業務主題分子表（基本資料／審核相關／金額組成／備註），各用 `###` 子標題 | 顯然分組提升可讀性，非強制 |

## 五、修改既有卡前的檢查

> 本節目的：欄位結構是商業決策，改動是商業行為不是編輯行為。

| 修改類型 | 改之前要確認什麼 |
|---------|----------------|
| 新增／移除欄位、改可否修改規則 | 商業決策依據（訪談定案／管理層拍板／OQ 拍板）；不是因 prototype 重構而動 |
| 欄位語意命名定位點 | 用業務語意命名（`#<欄位中文名>`），不重排、不重用——既有反向連結（backlink）不斷 |
| 純措辭修正 | 不需依據，但仍須通過語言紀律稽核 |

## 六、邊際情境處理

| 卡住的情境 | 標準處理 |
|-----------|---------|
| 無狀態實體（純參照／純子實體） | 省略「相關狀態機」段，其餘照寫 |
| 無業務可見欄位的純子實體（如僅承載關聯的中介實體） | 「欄位（業務可見）」段註明「本實體無獨立業務欄位，欄位隨主實體」，不硬編欄位 |
| 跨領域實體（多模組共用） | business-domain 標 `跨領域`，module 列全 |
| 某欄位疑似已在鄰近實體 | 先搜尋確認；已存在 → 不重複，於關聯段指過去 |
| 某資訊判不出屬欄位／規則／狀態／流程 | 走 § 二判斷二；仍判不準 → 停下問 |

## 七、error handle（出現任一即停下）

1. **無來源依據**：欄位的「來源／可否修改」只有推測 → 停下
2. **矛盾**：欄位定義與既有卡、狀態機、規則卡矛盾 → 停下，列出矛盾兩方
3. **欄位拆不出業務／技術**：欄位正本還掛在 OpenSpec，反向拆「業務可見 vs 純技術」拆不開 → 停下開 OQ（`oq-manage` mode B），不硬抄整份規格
4. **需要假設**：必須假設「這欄位應該是這樣用」才能繼續 → 停下
5. **該不該獨立判不準**：§ 二判斷一走完仍無法確定 → 停下問

處理方式：標記具體問題、附已知事實提問；**禁止**用「一般來說」「通常」填充。改寫舊卡發現的問題，保留舊內容另開 OQ。

## 八、產出格式

一張卡＝一個業務物件。**檔名即標題**；標題可附實體碼括號（`訂單（Order）`）對接 prototype 型別——這是**正文唯一允許英文之處**。**正文零迭代史**（歷史查 wiki/log.md 與 git）。

**段落結構**（八段；「為誰」＝伺候的讀者情境）：

| 段落 | 內容 | 為誰 | 必/選 |
|------|------|------|-------|
| 一句話定位＋規則歸屬聲明 | 這實體是什麼、屬哪領域、行為規則正本在哪張卡（本卡只引用不重述） | 第一次接觸的 PM | * |
| 這張卡要回答的問題 | 列關鍵問題，每題正文有對應結論（判斷卡完整＝每題有結論） | 規劃自查 | * |
| 營運背景 | 2-3 句業務語言：對應營運上什麼東西、為什麼獨立存在 | 商業自足理解 | * |
| 欄位（業務可見） | 五欄表（介面欄位／說明／欄位形態／來源／可否修改）；非顯然欄位補「為什麼要存」 | 規劃自查＋AI 載入 | * 無欄位實體依 § 六處理 |
| 關鍵關聯 | 實體間關聯＋基數（1:1／1:N／N:M）＋ wiki link，雙向可達 | 理解單據網絡 | * |
| 相關狀態機 | 有狀態才寫，wiki link 指狀態機卡，不重述狀態清單／轉換 | 定位狀態正本 | ? 無狀態省略 |
| 相關連結 | 依分類列（業務情境／角色／規則／實體），只列有連結的分類 | 互連防孤島 | * |
| 來源 | 欄位正本＝本卡欄位段；行為規格／prototype 為導航 | 溯源 | * |

**欄位形態參考值**：文字／文字（N 字）／單選／多選／日期／日期時間／金額（含稅／未稅）／數字／整數／是／否／連結（指向某實體）／檔案上傳（多檔）／單選（人員）／單選（唯讀）。

**frontmatter**：

```yaml
type: entity
module: [模組，中文值，見 wiki-schema § 二]
business-domain: [六領域之一或跨領域]
source:                # 往上＝正確性根據：承載其行為規則的 business-logic 正本卡或外部依據；禁指同層 entity、下層、OpenSpec
  - "[[<business-logic 規則卡>]] 或 <外部依據>"
implemented-by:        # 往下＝導航中繼資料：指 prototype 型別檔；欄位正本在本卡，不指 OpenSpec Data Model
  - "sens-erp-prototype/src/types/<...>.ts"
status: draft | active
last-reviewed: YYYY-MM-DD
```

**範例**：欄位段五欄、全中文、分子表、無技術 token、無迭代史，是憲章合規的標竿（§ 十二以它為通過樣本）。

| 介面欄位  | 說明               | 欄位形態              | 來源               | 可否修改          |
| ----- | ---------------- | ----------------- | ---------------- | ------------- |
| 訂單編號  | 系統自動產生的唯一編號      | 文字（唯讀）            | 系統自動             | 不可            |
| 訂單類型  | 一般 / 諮詢 / 線上（電商） | 單選                | 建立時依來源決定         | 不可            |
| 訂單狀態  | 訂單目前的生命週期階段      | 單選（依 [[訂單狀態]] 定義） | 系統自動推進或業務手動操作    | 依狀態機規則        |
| 客戶    | 下單的客戶公司          | 單選（廠客主檔）          | 需求單成交帶入          | 製作前可改（製作後不同步） |
| 案名    | 這張訂單的案名          | 文字                | 需求單標題帶入          | 業務可改          |
| 負責業務  | 這張訂單的業務窗口        | 單選（人員）            | 需求單成交時帶入（線上單可為空） | 業務主管可改派       |
| 窗口聯絡人 | 客戶端的對口窗口         | 單選（同客戶下的聯絡人）      | 需求單帶入            | 線下單可切換、線上單唯讀  |
| 帳務公司  | 用哪家公司名義開發票       | 單選（帳務公司主檔）        | 需求單帶入            | 建立後不可變更       |
| 是否急件  | 標記此訂單為急件         | 是 / 否             | 業務填寫             | 業務可改          |

## 九、不收清單

> 本節目的：越界內容的路由表。發現越界：判斷歸屬 → 確認目標卡存在 → 已有則刪、沒有則搬移後刪。

| 不收 | 歸屬 | 判斷訊號 |
|------|------|---------|
| 純技術欄位（識別碼、外鍵、時間戳記） | prototype／OpenSpec（不過紙本測試） | 看到底線命名、`_id`、`created_at` |
| 計算的決策政策（為什麼這樣算、什麼條件生效） | `04-business-logic/`（保持 high-level 商業用語）；具體組成式**本卡收**、貼著欄位寫 | 看到「因為…所以…」條件論證、核可分權敘述 |
| 程式碼層級公式（取整、精度、邊界處理） | OpenSpec（不過紙本測試） | 看到 `floor()`、`min()`、四捨五入精度 |
| 狀態枚舉值與轉換邏輯 | `06-state-machines/` | 看到狀態清單、「X 狀態 → Y 狀態」 |
| 業務流程（誰先做誰後做） | `07-scenarios/` | 看到「第 1 步…第 2 步…」、多角色接力 |
| change 迭代史 | wiki/log.md | 看到 `[!info] YYYY-MM-DD 更新（某 change）` callout |
| Prototype 實作細節 | 原型實作層（不進 BRD） | 看到 store 名、「資料層搬移」、「欄位遷移」 |
| 介面操作 | 原型實作層 | 看到按鈕、頁面、dialog |

## 十、撰寫紀律

1. **欄位正本只在本卡**：同一欄位不在 OpenSpec 另立正本；OpenSpec 引用本卡欄位段，本卡不複寫 OpenSpec 轉換規則
2. **規則回引不複寫**；**禁中英夾雜**；**禁技術術語當主詞**（外鍵、列舉碼、store 名）
3. **不確定項開 OQ**（`oq-manage` mode B），不留 inline 標注、不用 callout 釘正文
4. **檔名即標題**；**正文零迭代史**（change 歷史屬 wiki/log.md）
5. **不編造**：欄位的「來源／可否修改」只收訪談、既有介面、拍板來源，無則停下問

## 十一、擴張準則

| 情境 | 做法 |
|------|------|
| 新發現一個有獨立生命週期的業務物件 | 新開實體卡（過 § 二判斷一） |
| 新發現的資料只是既有實體的補充欄位 | 併入既有卡欄位表，不新建 |
| 某欄位群長出自己的狀態與 1:N 關聯 | 評估升級為獨立實體（過 § 二判斷一） |
| 卡明顯過長（粗標：正文超過 200 行） | 先過 § 九路由表外移越界內容；欄位多則分子表（§ 四 MAY），不輕易拆卡 |

## 十二、稽核維度（goal-driven；vault-audit 用，寫完逐項勾）

> 本節目的：把稽核從被動清單變成**可驗證的完成目標**（karpathy § 4 Goal-Driven）。一張實體卡的「完成」＝下列維度全部通過，以 [[訂單]] 為通過樣本對照。
> **執行者／稽核者分離**：撰寫卡的 agent ≠ 稽核的 agent（對齊 `erp-planning-pre-check` 既有模式）。稽核者拿本表獨立勾，不採信撰寫者的「大致 OK」。

| # | 維度 | 通過標準 | 標竿對照（[[訂單]]） |
|---|------|---------|---------------------|
| 1 | 結構對齊 | 八段齊備；檔名即標題；無正文來源段外的雜項 | 段落完整 |
| 2 | 欄位段存在 | 「欄位（業務可見）」段採五欄表（無欄位實體依 § 六註明） | 四個子表五欄 |
| 3 | **欄位正本在本卡** | 欄位正本是本卡欄位段，**未把正本指向 OpenSpec／Data Model** | frontmatter 不指 openspec 欄位 |
| 4 | **商業語言** | 介面欄位名全中文、無底線／駝峰、無列舉碼；說明用業務語言 | 「訂單編號」非 `order_no` |
| 5 | **無技術 token 當主詞** | 正文無外鍵、列舉碼、store 名當主詞（技術名僅必要時括號附註） | 正文無 `_id`／`store` |
| 6 | **正文零迭代史** | 無 `[!info] 某 change 更新` callout、無 inline OQ 措辭 | 無 callout |
| 7 | 規則唯一性 | 計算公式／轉換邏輯／流程未寫進本卡，wiki link 指正本 | 金額規則指 [[訂單異動規則]] |
| 8 | 關聯完整 | 關聯標基數＋ wiki link，雙向可達；無斷鏈、非孤島 | 七類相關單據 |
| 9 | 範圍外防護 | 純技術欄位點名「存在但屬實作規格」，未列入欄位表 | 技術欄位不列入 |
| 10 | 連結方向 | `source` 指 business-logic 正本卡或外部，未指同層／下層／OpenSpec | source 指規則卡 |
