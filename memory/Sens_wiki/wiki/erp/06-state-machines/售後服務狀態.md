# 售後服務狀態（AfterSalesTicketStatus）

> [[售後服務單]] 從客戶投訴受理到結案的進度狀態機。最大特色是**全由業務手動推進、無核可關卡、不阻擋主訂單**——主訂單維持「已完成」不回退。業務與業務主管的討論在 Slack 線下完成，ERP 只記錄結論。什麼情境該建售後單、決議怎麼分類的規則正本在 [[售後服務規則]]，本卡只定義狀態與轉換、不複述規則。

## 這張卡要回答的問題

> 每個問題在正文都有對應結論；判斷「卡完整了沒」就看每個問題是否都答到了。

- 售後服務單有哪幾種狀態？哪個是終態？
- 為什麼沒有主管核可關卡？
- 結案是人工還是系統自動？結案後可以重開嗎？
- 售後服務單會不會卡住主訂單或內部的訂單異動單？

## 狀態列舉（正本）

> 本段是售後服務單狀態的正本定義。狀態的新增與修改是商業決策，直接在此卡維護。轉換條件（Given/When/Then）留在 OpenSpec [after-sales-ticket/spec.md](../../../../openspec/specs/after-sales-ticket/spec.md) 行為規格中。

| 狀態 | 說明 | 對應營運需求 |
|------|------|------------|
| 受理中 | 業務已建單、尚未填入決議 | 給業務一個接收客訴的暫存空間，還沒決定怎麼處理 |
| 處理中 | 業務已填入決議並送出；下游關聯動作（退款異動單、補印印件）可開始建立與執行 | 標示這張單已有處理方向，下游可動工 |
| 已結案 | 業務手動點「結案」推進的終態 | 業務判斷處理完畢、明確收尾 |

> 判斷準則與決議分類見 [[售後服務規則]]，本卡不重述。

## 狀態轉換

售後服務單（業務全程手動推進，無系統自動）：

```
受理中 ─送出決議（填 resolution）─▶ 處理中 ─業務點結案─▶ 已結案
處理中 ─修改 resolution ──────────▶ 處理中（同態，允許變更）
已結案 ─不可重開 ─────────────────▶ X
```

## 轉換條件與觸發事件

| 轉換 | 觸發事件 | 條件 |
|------|---------|------|
| 受理中 → 處理中 | 業務填入決議並點「送出決議」 | 決議欄位不可為空 |
| 處理中 → 處理中 | 業務修改決議內容 | 同態，允許修改 |
| 處理中 → 已結案 | 業務點「結案」 | 人工判斷，系統不自動推進；若有未完結異動單會提示但允許強制結案 |

## 營運背景

客戶出貨後如果有品質問題或缺件要投訴，業務開一張售後服務單記錄。這張單刻意做得很輕——沒有主管核可關卡，是因為業務跟主管的溝通已經在 Slack 完成，ERP 這張單只記結論和決議，不重建線下已經跑完的核可流程。結案也刻意做成純手動，MVP 階段不靠系統自動判定，因為「處理完沒」只有業務看全貌才知道（例如客戶說可以了、或補印已出貨），系統沒有足夠資訊自動判斷。

已結案後不可重開，客戶再投訴就建新單——這避免已結案的紀錄被重開後語意混亂（到底結了沒）。

## 關鍵狀態 / 轉換的營運動機

- 受理中 → 處理中（送出決議）→ 動機：業務填完決議就能開始建退款異動單或補印印件，不必等到結案才動下游；決議送出即表示方向已定、可同步執行 → 例子：客戶投訴 #ORD-2026-0512 色差，業務填決議「退款 5000」送出，立即在單內建退款異動單 OA-015 送主管審核。
- 處理中 → 已結案（手動）→ 動機：MVP 不自動結案，因為「處理完沒」只有業務知道全貌；強制結案允許業務在下游動作未完結時仍可收尾（例如退款核可但客戶不急著拿錢）→ 例子：OA-015 已執行、客戶確認滿意，業務點「結案」收掉這張售後單。
- 不可重開 → 動機：避免已結案紀錄反覆開關導致稽核軌跡混亂；客戶再投訴就建新單，已結案的不算入「一訂單限一張未結案售後單」限制。

## 與其他狀態機 / 實體的關係

- 售後服務單處於任何狀態都不影響主 [[訂單狀態|訂單]]，訂單維持「已完成」不回退。
- 單內建立的退款 / 補印收費異動單走 [[訂單異動狀態]] 既有狀態機（含業務主管審核關卡），售後單結案前異動單可獨立運作。
- 一張訂單最多 1 張未結案售後服務單（已結案的不計入限制）。

## 邊界

- 退款 / 補印收費異動單本身的審核進度 → 走 [[訂單異動狀態]]，不在本卡。
- 決議分類（退款 / 補印 / 折讓 / 照常使用 / 混合）的完整規則 → 見 [[售後服務規則]]（規則正本）。
- 主訂單取消或完成的終態邏輯 → 見 [[訂單狀態]]。
