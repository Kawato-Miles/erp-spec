# 任務狀態（TaskStatus / WorkGroupStatus）

> [[任務]] 的狀態機：工單底下「同一工廠的所有生產任務」打包成一個任務（Work Group）後的進度。最大特色是**完全由 [[生產任務狀態]] 向上傳遞驅動**——任務本身不被主動操作。多筆生產任務怎麼彙整成任務完成度的規則正本在 [[齊套邏輯]]，本卡只記為什麼要有任務層、關鍵轉換的營運動機。

## 狀態列舉（正本）

> 本段是任務狀態的正本定義。狀態的新增與修改是商業決策，直接在此卡維護。轉換條件（Given/When/Then）留在 OpenSpec [production-task/spec.md](../../../../openspec/specs/production-task/spec.md) 行為規格中。

| 狀態 | 說明 | 階段 |
|------|------|------|
| 待交付 | 工單建立後，任務尚未交付產線 | 等待 |
| 已交付 | 工單交付後，任務同步交付 | 交付 |
| 製作中 | 底下首個生產任務開始製作，向上傳遞 | 製作 |
| 異動 | 印務發起異動，任務進入異動狀態 | 異動（可逆） |
| 已確認異動內容 | 生管確認收到異動安排（中間態） | 異動（可逆） |
| 已完成 | 底下所有生產任務皆完成 | 終態 |
| 作廢 | 底下所有生產任務皆已作廢，任務自動標記作廢（bottom-up 作廢） | 終態 |

> 異動路徑：已交付 / 製作中 → 異動 → 已確認異動內容 → 製作中（回到正常流程）。
>
> 作廢規則：當任務下**所有**生產任務皆為「已作廢」時，任務自動標記為作廢（bottom-up 作廢），不需手動操作。

## 營運背景

一張工單底下可能有好幾筆生產任務分散在不同工廠做。任務層的存在，是為了把「同一工廠的所有生產任務」歸成一組，方便生管對著一個工廠一次派工、一次追進度，而不必逐筆生產任務去管。所以任務狀態本身不開放手動操作——它只是上下兩層的彙整節點：上層工單交付就帶動任務交付，下層生產任務開做 / 完成就帶動任務製作中 / 完成。

之所以做成完全自動、不可手動，是因為任務只是分組視角，真正的進度事實在生產任務那層（現場報工）和工單那層（交付）。若允許手動改任務狀態，就會跟上下層的事實打架、對不上。

## 關鍵狀態 / 轉換的營運動機

- 待交付 → 已交付（自動向上傳遞）→ 動機：工單交付後，底下各工廠的任務跟著同步交付，生管不必逐組手動標交付，分組視角自動跟上工單。
- 已交付 → 製作中（自動向上傳遞）→ 動機：該組底下第一筆生產任務一報工，任務即顯示製作中，讓生管一眼看出哪個工廠已經動工。
- 製作中 → 已完成（自動）→ 動機：要等該組底下所有生產任務都完成，任務才算完成（取最少原則，見 [[齊套邏輯]]）；只要有一筆沒做完，這組就不算齊，避免分組進度虛報。
- 已交付 / 製作中 → 異動 → 已確認異動內容 → 製作中 → 動機：訂單成立後客戶改需求，異動由工單向下傳遞到任務，生管確認收到異動安排後再回製作中，讓分組層也能反映異動、不被跳過。

## 與其他狀態機 / 實體的關係

- 任務是向上傳遞鏈的中間層：底層 [[生產任務狀態|生產任務]] 報工 → 任務製作中 / 完成 → [[工單狀態|工單]] 跟進。上下兩層都會驅動任務，任務自己不主動。
- 任務按**工廠**分組，每個工廠對應一個任務，方便 [[生管]] 對「同一工廠的所有生產任務」整批派工；分組概念見 [[任務]]。
- 任務完成度取底下所有生產任務的最小值（min 聚合），是齊套統計的中間一層，完整公式見 [[齊套邏輯]]。

## 邊界

- 任務不開放手動操作：進度事實在生產任務（現場報工）與工單（交付）兩層，任務只彙整、不自設。
- 多層完成度怎麼算（取最少原則 / 四層計算）→ 見 [[齊套邏輯]]（規則正本），不在本卡。
- 派工排程怎麼安排 → 屬排程模組，不在本卡。
