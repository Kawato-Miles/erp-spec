# 生產任務狀態（ProductionTaskStatus）

> [[生產任務]] 的狀態機：一筆實際派給工廠或師傅做的活，從待處理、製作中到完成的進度。最大特色是**工廠類型決定狀態路徑**——自有 / 加工 / 外包 / 中國廠商四種，路徑長短不同。狀態怎麼往上彙整、怎麼算做齊的規則正本在 [[齊套邏輯]]，本卡只記為什麼要這樣分、關鍵轉換的營運動機。

## 狀態列舉（正本）

> 本段是生產任務狀態的正本定義。狀態的新增與修改是商業決策，直接在此卡維護。轉換條件（Given/When/Then）留在 OpenSpec [production-task/spec.md](../../../../openspec/specs/production-task/spec.md) 行為規格中。

### 共用狀態

| 狀態 | 說明 | 階段 |
|------|------|------|
| 待處理 | 生產任務已建立，等待師傅或供應商開始做；指派師傅不觸發狀態變更 | 等待 |
| 製作中 | 首次報工觸發，師傅或供應商已開始做 | 製作 |
| 已完成 | 製作完畢（或貨物送達）| 終態 |
| 已作廢 | 尚未開始製作即因異動取消（無成本） | 終態 |
| 報廢 | 製作中因異動或瑕疵中止 | 終態 |

### 依工廠類型分路徑

| 工廠類型 | 路徑 |
|---------|------|
| 自有工廠 | 待處理 → 製作中 → 已完成 |
| 加工廠 | 待處理 → 製作中 → 已完成（與自有工廠相同） |
| 外包廠 | 待處理 → 製作中 → 運送中 → 已完成 |
| 中國廠商 | 待處理 → 製作中 → 已送集運商 → 運送中 → 已完成 |

> 向上傳遞規則（生產任務 → 任務 → 工單 → 印件 → 訂單）見 [[齊套邏輯]]，本卡不重述。

## 營運背景

生產任務是生產流程裡真正動手做的最底層單位，它的進度是整條向上傳遞鏈的源頭——師傅或供應商一報工，這筆任務就推進，連帶把任務、工單、印件、訂單一路往上帶動。所以這層的狀態設計要貼著現場實況，不能只用一條通用流程套所有工廠。

不同工廠類型的物流節奏不同，因此路徑長短不一：自有工廠與加工廠做完就算完成（製作中 → 已完成）；外包廠做完還要送回來，多一段「運送中」；中國廠商最遠，做完先送集運商集中再運回，多「已送集運商 → 運送中」兩段。把這些中轉狀態顯式建出來，現場與業務才看得出貨卡在哪一段、不會把「在路上」誤當成「已到」。

## 關鍵狀態 / 轉換的營運動機

- 待處理 → 製作中（首次報工，自動推進上層）→ 動機：師傅或供應商一開始做就報工，這筆任務即推進，並自動往上帶動任務 / 工單 / 印件 / 訂單同步進入製作中；以報工為觸發點，現場不必另外去更新上層各層狀態。
- 外包 / 中國廠商多出「運送中」「已送集運商」中轉狀態 → 動機：遠端工廠做完不等於貨已到手，把在途狀態顯式建出來，業務與生管才追得出貨卡在哪一段、預估何時能進下一步，避免「做完了卻遲遲收不到貨」對不上。
- 待處理 → 已作廢、製作中 → 報廢 → 動機：任務還沒開做時派錯可作廢、做到一半因異動或瑕疵要中止可報廢，給兩個明確終態，不留無效或半成品任務污染進度統計。

## 與其他狀態機 / 實體的關係

- 生產任務是向上傳遞鏈的起點：任務首次報工 → [[任務狀態|任務]] 製作中 → [[工單狀態|工單]] 製作中 → [[印件狀態|印件]] 印製維度推進 → [[訂單狀態|訂單]] 製作中。完整多層鏈路見 [[印件狀態]]。
- 生產任務作廢會連帶讓關聯的品檢任務一併作廢（QC 已併入生產任務，type=qc / scope=print_item），避免源頭任務沒了、品檢任務還掛著。
- 任務做齊的數量往上彙整到工單與印件，背後的齊套統計見 [[齊套邏輯]]。

## 邊界

- 多筆生產任務怎麼彙整成任務 / 工單的完成度（取最少原則）→ 見 [[齊套邏輯]]，不在本卡。
- 數量在不同計量單位間的換算 → 見 [[數量換算規則]]。
- 派工怎麼分組、生管怎麼指派 → 見 [[任務狀態]]（按工廠分組的任務層）。
