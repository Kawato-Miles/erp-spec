# 訂單狀態

> [[訂單]] 從成立到結案的整體進度狀態機，分三條前段路徑（線下 / 線上 / 諮詢）再接同一段共用流程。屬跨領域的最上層訂單生命週期。狀態怎麼自動往前推、什麼時候算製作完成，背後的規則正本在 [[齊套邏輯]]；諮詢收尾的退費與終態規則正本在 [[付款發票邏輯]]，本卡只定義狀態與轉換、不複述規則。

## 這張卡要回答的問題

> 每個問題在正文都有對應結論；判斷「卡完整了沒」就看每個問題是否都答到了。

- 訂單有哪幾種狀態？哪些是收尾用的終態？
- 線下、線上、諮詢三條路徑為什麼分流？它們在哪裡會合？
- 諮詢訂單為什麼會有兩種不同的終態（訂單完成 vs 已取消）？
- 進入製作段後，狀態大多是誰在推（人工或系統自動）？條件是什麼？

## 營運背景

訂單從不同管道進來、收尾方式也不同：線下接單要先請客戶回簽報價單、線上接單要先收到款項、諮詢則是客戶先付諮詢費。三條路徑各有自己的前段，但只要進入大貨製作，後面的審稿、製作、出貨流程都一樣，所以三條路徑會合到同一段「共用流程」走到結案。

進入製作之後，整張訂單的進度大多不是人工一格一格手動改，而是隨著底下的工單、印件、生產任務做到哪裡，自動往上反映成訂單的進度——這樣業務不必盯著每張工單去手動更新訂單狀態。

## 狀態列舉（正本）

> 本段是訂單狀態的正本定義。狀態的新增與修改是商業決策，直接在此卡維護。轉換條件（Given/When/Then）留在 OpenSpec [order-management/spec.md](../../../../openspec/specs/order-management/spec.md) 行為規格中。

### 線下路徑（訂單類型 = 一般，來自需求單成交）

| 狀態 | 說明 | 段落 |
|------|------|------|
| 草稿 | 需求單成交轉訂單後的初始狀態，業務可自由調整帶入內容 | 審核段 |
| 待業務主管審核 | 業務點「送主管審核」後等待主管核准 | 審核段 |
| 審核通過 | 主管已核可成交條件，業務尚未送報價單給客戶 | 審核段 |
| 報價待回簽 | 已送報價單，等待客戶回簽確認 | 審核段 |
| 已回簽 | 客戶回簽完成（業務手動確認或上傳回簽檔案觸發） | 審核段 |

### 線上路徑（訂單類型 = 線上，來自電商下單）

| 狀態 | 說明 | 段落 |
|------|------|------|
| 等待付款 | 客戶在線上前台下單，尚未完成付款 | 付款段 |
| 已付款 | 客戶完成付款（電商系統自動觸發） | 付款段 |

### 共用段（線下路徑「已回簽」後 / 線上路徑「已付款」後合流）

| 狀態 | 說明 | 段落 |
|------|------|------|
| 稿件未上傳 | 進入審稿流程，等待業務 / 客戶上傳稿件 | 審稿段 |
| 等待審稿 | 稿件已上傳，等待審稿人員審核 | 審稿段 |
| 待補件 | 存在審稿不合格的印件，等待業務補件（與「等待審稿」可雙向互換） | 審稿段 |
| 製作等待中 | 審稿通過，等待印務主管建立工單 | 製作段 |
| 工單已交付 | 印務主管已建立工單並交付產線 | 製作段 |
| 製作中 | 至少一個生產任務已開始製作 | 製作段 |
| 製作完成 | 所有印件皆已完成製作（齊套性邏輯，見 [[齊套邏輯]]） | 製作段 |
| 出貨中 | 至少一張出貨單已建立 | 出貨段 |

### 終態

| 狀態 | 說明 | 適用訂單類型 |
|------|------|------------|
| 訂單完成 | 所有印件出貨完成（一般 / 線上）；諮詢訂單不做大貨 / 需求單流失收尾 | 一般 / 線上 / 諮詢（不做大貨） |
| 已取消 | 訂單取消（不可逆）；諮詢訂單待諮詢取消（半額退費） | 一般 / 線上 / 諮詢（取消） |

### 諮詢訂單路徑（訂單類型 = 諮詢）

諮詢訂單不進入共用段（無印件、無製作、無出貨）。只在以下兩種收尾情境建立：

- **不做大貨 / 需求單流失**：建立即推進至「訂單完成」終態
- **待諮詢取消（半額退費）**：建立即推進至「已取消」終態

### 免審稿快速路徑

當訂單下所有印件的審稿狀態皆為「合格」（含免審稿設定）時，訂單從「已回簽」或「已付款」直接進入「製作等待中」，跳過審稿段三個狀態。

### 段落不可逆規則

- 審核段 → 審稿段 → 製作段 → 出貨段，**段落之間不可回退**
- 審稿段內「等待審稿」與「待補件」**允許雙向互換**（補件完成回到等待審稿）
- 其他段落內子狀態維持單向推進

## 關鍵狀態 / 轉換的營運動機

- 線下前段插入「審核通過」把關（草稿 → 待業務主管審核 → 審核通過 → 報價待回簽）→ 動機：業務主管核可成交條件的當下就鎖定（收款條件、報價總額、客戶），避免「核可後、外發報價單前」業務偷改金額或條件繞過主管把關；草稿與待業務主管審核兩態本就可自由編輯、核可前不留稽核 → 例子：訂單經主管核可進「審核通過」後，業務想再加一筆運費已不能直接改報價總額，必須改走 [[訂單異動狀態|訂單異動單]]（含訂單額外費用凍結點亦前移至「審核通過」，見 [[ORD-027-訂單額外費用凍結時點與審核通過成交鎖定對齊]]）。
- 製作中 → 製作完成 → 動機：要全部印件都做齊才算完成、不能有一份漏做就出貨，否則客戶會收到缺件的訂單 → 例子：#ORD-2026-0512 下三款印件，前兩款先做完、第三款還在印，訂單仍停在「製作中」，等第三款也做完才一起轉「製作完成」。
- 諮詢取消 → 已取消（而非訂單完成）→ 動機：諮詢取消是沒談成的生意，若算成「訂單完成」會讓成交統計虛胖、業務月會數字失真 → 例子：客戶付了 2000 諮詢費後在諮詢階段喊停，系統建好諮詢訂單後直接收為「已取消」，並退一半（1000）給客戶，這筆不計入成交業績。
- 製作等待中 → 工單已交付 → 製作中 → 動機：訂單進度自動跟著底下工單與生產任務往上反映，業務不必逐張工單手動回填訂單狀態 → 例子：生產端把 #ORD-2026-0512 的第一張工單交付出去，訂單自動從「製作等待中」變「工單已交付」，第一個生產任務一開做又自動變「製作中」。

## 與其他狀態機 / 實體的關係

- 共用流程的製作段，訂單進度大多是跟著底下的 [[印件狀態|印件]] / [[工單狀態|工單]] / [[生產任務狀態|生產任務]] 做到哪裡自動往上反映——底層做到哪一步，訂單就顯示到哪一步。詳細的雙維度判定在 [[印件狀態]]，本卡只接收結果。
- 諮詢取消會連帶建立一張退費的 [[訂單異動狀態|訂單異動單]]（退費 1000），但那張單據自己的審核與認列進度走 [[訂單異動狀態]]，不影響訂單已收為「已取消」終態。
- 諮詢收尾、後續開票與收款的進度走 [[分期請款狀態]]；本卡只管訂單整體走到哪一格。

## 邊界

- 訂單成立後的金額增減（加印、補運費、退費）→ 走 [[訂單異動狀態]]，不在本卡。
- 開票與收款的進度（先收後開、頭尾款分期）→ 走 [[分期請款狀態]]，不在本卡。
- 諮詢階段本身（待諮詢、認領、結束諮詢）→ 走 [[諮詢單狀態]]；本卡只從諮詢收尾接手「諮詢訂單」這張訂單的終態。
- 製作完成的齊套判定準則、諮詢收尾退費與對帳的完整規則 → 見 [[齊套邏輯]] 與 [[付款發票邏輯]]。
