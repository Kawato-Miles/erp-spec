# 諮詢單狀態（ConsultationRequestStatus）

> [[諮詢單]] 的狀態機：客戶付費預約諮詢後，這張單從待諮詢到收尾的進度。最大特色是**雙出口**——諮詢完可轉需求單接著做大貨、也可結束不做大貨，外加待諮詢階段取消。退費與付款跨實體轉移的規則正本在 [[付款發票邏輯]]，本卡只記為什麼這樣分出口、關鍵轉換的營運動機。

> 完整狀態 enum（待諮詢 / 已轉需求單 / 完成諮詢 / 已取消）、轉換條件與觸發事件的列舉正本，見 [state-machines/spec.md § 諮詢單狀態機（v2 簡化）](../../../../openspec/specs/state-machines/spec.md)；分支與取消退費的行為規則見 [consultation-request/spec.md § 諮詢結束分支](../../../../openspec/specs/consultation-request/spec.md)、[§ 諮詢取消觸發建諮詢訂單與退費](../../../../openspec/specs/consultation-request/spec.md)。

## 營運背景

諮詢單由問卷工具（surveycake）的 webhook 自動建單，客戶已先付諮詢費。它不像需求單那樣有層層流程，而是諮詢人員認領後在「待諮詢」這一格直接決定走哪個出口——這是 v2 的刻意簡化：原本還有「諮詢中」過渡狀態，但實務上諮詢完當下就知道結局（要不要做大貨），多一格只是徒增手動點擊，故移除，認領後 status 仍維持「待諮詢」僅標示已分派。

之所以設計成雙出口，是因為諮詢這件事天然有兩種結局：談得成就轉需求單接著走報價、做大貨；談不成就結束諮詢、把已收的諮詢費結算掉。另有第三條路——客戶在待諮詢階段就反悔取消預約。三條出口對應三種真實收尾，少任何一條都會有一類客戶無處可去。

## 關鍵狀態 / 轉換的營運動機

- 待諮詢 → 已轉需求單（轉做大貨）→ 動機：諮詢談成、客戶要做大貨，建 [[需求單]] 接著走報價流程；此時諮詢費的付款紀錄維持綁諮詢單，等需求單結局明確（成交或流失）才轉移，避免錢提早算到還沒定案的單上。
- 待諮詢 → 完成諮詢（不做大貨）→ 動機：諮詢談完客戶不做大貨，仍要把已收的諮詢費（2000）結算為一張諮詢訂單收尾、不能讓這筆收入懸空；諮詢來源的需求單後續若流失，也自動觸發這條收尾。
- 待諮詢 → 已取消（取消預約）→ 動機：客戶在諮詢前就反悔，建諮詢訂單收為「已取消」終態並半額退費（收 2000 退 1000），系統自動開退款訂單異動單（已核可、核可人記系統）與退款付款紀錄（處理中）；取消必填取消原因（六選一），讓沒成交的預約有明確分類、不灌水成交數。
- 認領與改派不觸發狀態轉移 → 動機：諮詢人員自我認領、或業務主管代為認領 / 改派，都只是決定誰來諮詢，諮詢還沒發生、單據進度不該因此前進，故維持「待諮詢」僅標示分派對象。

## 與其他狀態機 / 實體的關係

- 轉需求單時把客戶往 [[需求單狀態]] 接走（建 QuoteRequest，MUST NOT 直接建訂單）。
- 完成諮詢 / 已取消時建一張諮詢 [[訂單]]（order_type = 諮詢），其終態由 [[訂單狀態]] 管轄（完成諮詢 → 訂單完成、取消 → 已取消）；本卡只負責諮詢這一段，訂單那張的善後走訂單狀態。
- 諮詢費的付款紀錄跨實體轉移、半額退費的金流規則 → 走 [[付款發票邏輯]]；退費單據本身的進度走 [[訂單異動狀態]]。

## 邊界

- 沒有底層自動推進：諮詢單不靠 bubble-up，唯一的自動事件是 webhook 自動建單（→ 待諮詢），其餘出口都是諮詢人員手動選。
- 諮詢訂單建立後的取消 / 完成終態、善後 → 走 [[訂單狀態]]，不在本卡。
- 半額退費的金額、退款 SLA、付款紀錄跨實體轉移的完整規則 → 見 [[付款發票邏輯]]（規則正本）。
