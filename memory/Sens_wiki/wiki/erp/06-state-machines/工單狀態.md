# 工單狀態（WorkOrderStatus）

> [[工單]] 的狀態機：印務把一張工單從建立、送審、交付生產到做完的整體進度。含兩條可逆路徑——收回（Withdraw）與異動（Modification）。狀態怎麼算做齊、底層怎麼往上反映的規則正本在 [[齊套邏輯]]，本卡只記為什麼要這樣分、關鍵轉換的營運動機。

## 狀態列舉（正本）

> 本段是工單狀態的正本定義。狀態的新增與修改是商業決策，直接在此卡維護。轉換條件（Given/When/Then）留在 OpenSpec [work-order/spec.md](../../../../openspec/specs/work-order/spec.md) 行為規格中。

| 狀態 | 說明 | 階段 |
|------|------|------|
| 草稿 | 印務建立工單後的初始狀態，可自由填寫製程 | 規劃 |
| 製程確認中 | 印務填完製程送印務主管審核 | 審核 |
| 重新確認製程 | 印務主管退回要求修改，印務修改後重新提交 | 審核 |
| 製程審核完成 | 印務主管核可製程 | 審核 |
| 工單已交付 | 印務交付產線（首個任務交付時觸發） | 製作 |
| 製作中 | 至少一個生產任務已開始製作（向上傳遞驅動） | 製作 |
| 異動 | 任務層有異動或已確認異動內容，工單 bubble-up 為異動 | 異動（可逆） |
| 已完成 | 工單齊套達標（見 [[齊套邏輯]]） | 終態 |
| 已取消 | 工單取消 | 終態 |

> 齊套達標判定準則見 [[齊套邏輯]]，本卡不重述。

## 營運背景

工單是生產排程的基本單位，由印務建立並填寫製程，送印務主管審核後才交付生產。狀態之所以前半段（草稿 → 製程確認中 → 製程審核完成）靠印務與主管手動推、後半段（工單已交付 → 製作中 → 已完成）改由底層生產任務自動往上反映，是因為前半段是人在規劃製程、需要主管把關，後半段是現場在做、進度該跟著實際報工走，不該讓印務再去手動回填工單做到哪。

工單同時掛兩條可逆路徑：印務發現製程填錯可「收回」回草稿重做；訂單成立後客戶改需求時，異動會從底層任務往上帶動工單進入「異動」狀態、處理完再回原狀態——這兩條讓工單不必作廢重開就能修正，符合印刷現場製程常臨時調整的實況。

## 關鍵狀態 / 轉換的營運動機

- 製程審核完成 → 工單已交付 → 製作中（自動）→ 動機：工單交付與開做之後改由系統跟著底層生產任務的報工自動往上反映，印務不必逐筆回填工單進度，避免現場做到哪、系統顯示對不上。
- 製作中 → 已完成（自動，齊套達標）→ 動機：要等這張工單累計入庫數量達到目標量（wo_target_qty）才算完成，不能做了一部分就標完成，否則訂單會以為這批已齊、提早往出貨推；怎麼算達標見 [[齊套邏輯]]。
- 製程確認中 / 製程審核完成 → 草稿（收回）→ 動機：印務送審後若發現製程填錯，給一條收回重做的路，不必把整張工單作廢重建、也不留無效單據。
- 任意狀態 ↔ 異動 → 動機：訂單成立後客戶改需求，異動由底層任務往上帶動工單；異動期間完成度計算持續運作（不暫停），確保現場照常生產、進度不因異動而中斷。

## 與其他狀態機 / 實體的關係

- 工單的後半段進度大量依靠**向上傳遞**：底層 [[生產任務狀態|生產任務]] 開做 → [[任務狀態|任務]] 製作中 → 工單製作中。底層做到哪一步，工單就反映到哪一步，本卡只接收結果；多層傳遞的完整鏈路見 [[印件狀態]]。
- 一張工單只對應一位印務，系統阻擋分派給第二位；要換印務時走「重新確認製程」流程改派，不直接改負責人。
- 工單做齊後的數量會往上彙整到 [[印件狀態]] 的印製維度，再反映到 [[訂單狀態]]；本卡只管單張工單走到哪一格。
