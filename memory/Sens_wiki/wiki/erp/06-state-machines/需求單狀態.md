# 需求單狀態（QuoteStatus）

> [[需求單]] 的狀態機：業務從蒐集客戶需求、送印務主管評估成本、報價議價到成交或流失的進度。特色是**業務獨立完成、這階段沒有業務主管 gate**（主管審核發生在訂單階段）。報價怎麼算的規則正本在 [[報價邏輯]]，本卡只記為什麼這樣分、關鍵轉換的營運動機。

## 狀態列舉（正本）

> 本段是需求單狀態的正本定義。狀態的新增與修改是商業決策，直接在此卡維護。轉換條件（Given/When/Then）留在 OpenSpec [quote-request/spec.md](../../../../openspec/specs/quote-request/spec.md) 行為規格中。

| 狀態 | 說明 | 階段 |
|------|------|------|
| 需求確認中 | 業務蒐集客戶需求，初始狀態 | 蒐集 |
| 待評估成本 | 業務提交需求單，等印務主管評估製程成本 | 評估 |
| 已評估成本 | 印務主管評估完成，業務拿到成本可報價 | 評估 |
| 議價中 | 業務對客戶報價、來回議價 | 議價 |
| 成交 | 客戶接受報價，後續自動建訂單 | 終態 |
| 流失 | 客戶拒絕報價或逾期未回覆 | 終態 |

> 報價計算規則、成本怎麼帶入見 [[報價邏輯]]，本卡不重述。

## 營運背景

需求單是接單的最前段：業務跟客戶確認要做什麼、請印務主管評估成本後對客戶報價、來回議價，最後談成轉訂單或談不成流失。這整段刻意讓業務獨立跑、不設業務主管簽核關卡——因為前段是業務在跟客戶互動、報價議價本就是業務的本職，每動一步都要主管核會拖垮回應客戶的速度。真正需要主管把關的是錢與承諾落定的訂單階段，所以 gate 放在那裡，不放這裡。

狀態之所以分這幾格，是因為這段有一次跨角色的交接：業務蒐集完需求要交給印務主管評估成本（待評估成本），主管評完業務才拿得到成本去報價（已評估成本）。把交接前後切成兩格，業務與印務主管才看得出球在誰手上、不會互等。

## 關鍵狀態 / 轉換的營運動機

- 待評估成本 → 已評估成本 → 動機：業務不會算製程成本，要先把需求送印務主管評估、拿到成本才報得出價；切成「等評估 / 評估完」兩格，雙方看得出評估卡在誰那、避免業務空等或主管漏接。
- 已評估成本 ↔ 待評估成本（重新評估）→ 動機：客戶改需求或成本要重抓時，業務可把單退回重新評估，不必作廢需求單重建。
- 議價中 → 成交 → 動機：客戶接受報價即成交、自動建訂單接著走，把確認落定的時點明確標出來，後續主管把關與生產才有起點。
- 議價中 → 流失 → 動機：客戶不接受報價，明確標「流失」收尾，讓沒談成的需求有歸屬、不懸在議價中污染在途需求統計。
- 全程無自動推進 → 動機：這段都是人在跟客戶互動（蒐集、報價、議價），沒有底層事件可自動驅動，故所有轉移由業務或印務主管手動觸發。

## 與其他狀態機 / 實體的關係

- 成交 → 自動建立 [[訂單]]，訂單初始狀態依來源不同：線下為報價待回簽、諮詢為建立、線上（從電商平台）為等待付款；後續走 [[訂單狀態]]，本卡只負責成交前的需求段。
- 業務在需求單階段就會填難易度與免審判定，供後續審稿與排程使用，規則見 [[難易度機制]] / [[免審決策樹]]。

## 邊界

- 業務主管審核 → 發生在訂單階段，不在需求單階段；本卡明示這段無 gate。
- 報價金額怎麼算、成本怎麼帶 → 見 [[報價邏輯]]（規則正本），不在本卡。
- 成交後訂單的回簽 / 付款 / 製作 / 出貨進度 → 走 [[訂單狀態]]。
