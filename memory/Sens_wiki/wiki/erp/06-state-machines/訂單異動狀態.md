# 訂單異動狀態

[[訂單]] 成立後遇到金額增減（追加印量、加運費、急件費、退費）時，用「訂單異動單」記錄這筆增減；本卡只講這張單據自己會經過哪些進度、什麼時候從一步走到下一步，跟主訂單分開算。誰能核可、錢什麼時候算進帳、退錯了怎麼還原，這些規則寫在 [[訂單異動規則]] 那張，這裡不重講。

## 這張卡要回答的問題

> 這段只講狀態怎麼分、什麼情況會跳下一步、是誰按的，每個問題在正文都有對應結論。規則對不對（為什麼多收錢不用審、退錢要核可）寫在 [[訂單異動規則]] 那張，不在本卡回答。

- 訂單異動單有哪幾種狀態？哪些是走到底就不再動的終態（已執行、已取消）？
- 補收（公司多收錢）、退款（公司退錢）、諮詢取消退費三種情況，為什麼走不同的路線？
- 每一步是誰按、還是系統自己往下走？要滿足什麼條件才會跳到下一步？

## 營運背景

這張異動單自己跑自己的流程：金額要加要減的審核、什麼時候算進帳，按它自己的步調走，不會卡住主訂單的生產跟出貨。狀態之所以分補收（公司多收錢）與退款（公司退錢出去）兩種走法，是因為兩個方向的審核強度不同——為什麼這樣分（誰能核可、為什麼這樣定）寫在 [[訂單異動規則]] 那張，本卡只記「哪個狀態、什麼情況會跳到下一個狀態」。

## 狀態清單

| 狀態 | 說明 | 對應營運需求 |
|------|------|------------|
| 草稿 | [[業務]]／[[諮詢|諮詢人員]] 建好異動單但還沒送出 | 給一個暫存空間，金額、原因填到一半可先存 |
| 待主管審核 | 業務送出後等 [[業務主管]] 核可（只有退款異動單會經過這一站）| 落實退款需主管把關，補收不經此站 |
| 已核可 | 主管已同意，等實際退款款項真的退出去 | 主管已經同意退這筆錢，但錢還沒真的退出去，所以先不算進帳 |
| 已退回 | 主管退回（附原因），業務改完可重送 | 讓主管擋下不合理退款，並給業務修正機會 |
| 已執行 | 終態；這筆增減正式計入訂單應收總額 | 等錢真的收進來（補收）或退出去（退款）以後，才算進這張訂單的帳 |
| 已取消 | 終態；業務在草稿或已退回階段主動撤掉 | 異動單建錯或不必要時可作廢，不留無效單據 |

> 各狀態怎麼判斷、金額怎麼算進帳、有哪些一定要守住的規則，都見 [[訂單異動規則]]，本卡不重複寫。

## 狀態怎麼一步步走

退款這條走法（公司退錢給客戶、由業務開單；系統裡把退款金額記成負數）：

```
草稿 ─提交審核─▶ 待主管審核 ─核可─▶ 已核可 ─這張單該退的錢全退完─▶ 已執行
待主管審核 ─退回─▶ 已退回 ─修改重交─▶ 待主管審核
草稿／已退回 ─取消─▶ 已取消
已執行 ─把已經退出去的某筆錢取消掉，這張單的退款就湊不齊了─▶ 已核可（退回上一步）
```

補收這條走法（公司向客戶多收錢、由業務開單、不用主管審）：

```
草稿 ─業務按「存檔並馬上生效」─▶ 已執行（跳過主管審核那兩站，直接生效）
```

諮詢取消退費這條走法（系統自動開的單，固定退 1000 元；系統裡記成負數）：

```
系統自動建單 ─▶ 已核可 ─這 1000 元真的退給客戶了─▶ 已執行
```

## 什麼情況會跳到下一步、是誰按的

| 從哪一步到哪一步 | 是誰按的、什麼事讓它跳 | 條件 |
|------|---------|------|
| 草稿 → 待主管審核 | 業務點「提交審核」 | 限退款異動單 |
| 草稿 → 已執行 | 業務點「存檔並馬上生效」 | 只有補收的單這樣走（不用審）。系統會把生效時間記成當下、把核可人記成開單的業務自己，這筆補收不用去對應到某一筆收款紀錄 |
| 草稿／已退回 → 已取消 | 業務點「取消」 | — |
| 待主管審核 → 已核可 | 業務主管核可 | — |
| 待主管審核 → 已退回 | 業務主管退回（附原因）| — |
| 已退回 → 待主管審核 | 業務修改後重交 | — |
| 系統自動建單 → 已核可 | 諮詢取消時系統自動開這張單 | 諮詢取消時系統自動開這張單，核可人就記成「系統」，生效時間先留空，等錢真的退出去才填 |
| 已核可 → 已執行 | 對應的那筆退款，狀態變成「已完成」（錢真的退出去了）| 等實際退出去的錢，累積到跟這張單該退的金額一樣多，系統就自動把狀態推到「已執行」、填上生效時間；還沒退到那麼多，就停在「已核可」 |
| 已執行 → 已核可（退回上一步）| 取消某筆已退出去的退款款項 | 把已經退出去的某筆錢取消掉、這張單該退的錢就湊不齊了，系統就自動退回「已核可」，並把原本填的生效時間清空；如果取消後錢還是退夠了，就維持已執行；如果取消的是還在處理、還沒退出去的款項，這張單狀態不變 |

> 「為什麼多收錢不用審、退錢要主管核可、諮詢取消由系統直接核可」這些誰能核可、錢什麼時候算帳的原因，寫在 [[訂單異動規則]] 那張。

## 邊界

- 異動單不卡主訂單：就算這張異動單還沒處理完（例如還在等主管核可），主訂單的生產、出貨進度照常往前跑，不會被這張單卡住。主訂單自己要不要取消、最後怎麼收尾（取消或完成）→ 看 [[訂單狀態]]，不在本卡。
- 這張單一旦「已執行」（錢真的收進來或退出去了），訂單該收的總金額會跟著重算，但系統不會自動去動 [[分期請款狀態|請款期次]]（不會自動幫你開一期或改一期）；每一期什麼時候開發票、收了多少錢，還有期次改過幾次的紀錄，都看 [[分期請款狀態]] 那張。請款期次有變動，也不會把這張異動單或主訂單退回上一步。
- 售後服務單本身的受理、處理、結案進度 → 走 [[售後服務]]；這裡只管：在售後服務單裡開出來的退款單、補印收費單，它們的進度怎麼跑。
- 金額怎麼算進帳、有哪些一定要守住的規則、補收跟退款各種情況怎麼判斷的完整內容 → 見 [[訂單異動規則]]（規則正本）與 [[付款發票邏輯]]（情境索引）。

## 相關連結

- 想看誰能核可、錢什麼時候算帳、退錯怎麼還原 → 看 [[訂單異動規則]]（規則寫那邊，這張只畫進度走法，規則的原因跟算法都回頭指這張）
- 相關實體：[[訂單]]（異動單依附的主實體）、[[售後服務]]（售後單內可建立關聯退款／補印收費異動單）
- 相關狀態：[[訂單狀態]]（異動單不會卡住主訂單往下走）、[[分期請款狀態]]（異動單就算生效了，系統也不會自動幫你開新的一期請款）
