# 印件狀態（PrintItemStatus，雙維度）

> [[印件]] 同時擁有**兩個維度**的狀態：審稿維度（稿件可不可印）+ 印製維度（東西做到哪、出貨沒）。兩維度獨立推進但相互關聯。怎麼算做齊、稿件鎖檔、免審稿等規則正本分別在 [[齊套邏輯]] / [[稿件管理規則]] / [[免審決策樹]]，本卡定義狀態列舉（正本）、拆兩維度的理由、關鍵轉換的營運動機。轉換條件（Given/When/Then）留在 OpenSpec 各模組 spec（審稿維度：[prepress-review/spec.md](../../../../openspec/specs/prepress-review/spec.md)、印製維度：[order-management/spec.md](../../../../openspec/specs/order-management/spec.md)）行為規格中。

## 狀態列舉（正本）

> 本段是印件雙維度狀態的正本定義。狀態的新增與修改是商業決策，直接在此卡維護。轉換條件（Given/When/Then）留在 OpenSpec 各模組 spec（審稿維度：[prepress-review/spec.md](../../../../openspec/specs/prepress-review/spec.md)、印製維度：[order-management/spec.md](../../../../openspec/specs/order-management/spec.md)）行為規格中。

### 審稿維度

| 狀態 | 說明 | 階段 |
|------|------|------|
| 稿件未上傳 | 印件建立後尚未上傳稿件 | 初始 |
| 等待審稿 | 稿件已上傳或補件完成，等待審稿人員審核 | 審稿中 |
| 不合格 | 審稿人員判定稿件不符合要求，等客戶補件 | 審稿中 |
| 已補件 | 客戶已重新上傳稿件，等原審稿人員重審 | 審稿中 |
| 合格 | 審稿人員品質判定通過，等業務確認可製作 | 業務確認 |
| 已確認可製作 | 業務確認可投產（B2C 自動 / B2B 手動），觸發工單建立 | 終態 |
| 待改稿 | 業務判斷客戶需改稿，退回等客戶重新上傳 | 退回重審 |

### 審稿維度路徑說明

- **正常審稿**：稿件未上傳 → 等待審稿 → 合格 → 已確認可製作
- **不合格迴圈**：等待審稿 → 不合格 → 已補件 → 等待審稿（可多次來回）
- **退回重審**：合格 → 待改稿 → 等待審稿（客戶上傳新稿後重審）
- **免審稿直通**：稿件未上傳 → 合格（B2C 再自動跳已確認可製作；B2B 等確認）
- **B2C 自動確認**：合格後系統自動推進至「已確認可製作」，業務不需手動操作
- **退回重審 vs 棄用 + clone**：退回重審用於「同一印件改稿」；棄用 + clone 用於「印件規格全換」。兩條路徑並存

### 印製維度

| 狀態 | 說明 | 階段 |
|------|------|------|
| 等待中 | 審稿維度尚未到達「已確認可製作」，生產不啟動 | 等待 |
| 工單已交付 | 印務主管已建立工單並交付產線 | 製作 |
| 部分工單製作中 | 部分工單已開始製作（多工單印件） | 製作 |
| 製作中 | 所有工單皆在製作中 | 製作 |
| 製作完成 | 所有工單皆完成（齊套達標，見 [[齊套邏輯]]） | 完成 |
| 出貨中 | 至少一張出貨單已建立 | 出貨 |
| 已送達 | 全部出貨完成 | 終態 |
| 已棄用 | 打樣 NG-稿件問題後棄用（保留稽核軌跡） | 終態 |

### 兩維度的關聯

- 印製維度在審稿維度到達「已確認可製作」後才開始推進
- 審稿維度推進不影響已建工單鎖定的檔案（見 [[稿件管理規則]]）
- 已棄用印件的雙維度狀態保留原值作為棄用前稽核軌跡

## 營運背景

一份印件在工廠裡其實同時在跑兩件事：一是「這份稿件審了沒、能不能印」（審稿維度），二是「實際做到哪一步、出貨沒」（印製維度）。這兩件事的負責人、節奏、可逆性都不同——審稿是審稿人員在看稿、可能來回退補件；印製是現場在做、進度跟著底層生產任務自動往上跑。若硬塞成一條狀態鏈，退一次補件就會把生產進度也攪亂，所以拆成兩個獨立維度各自記各自的進度。

兩維度不是完全平行：審稿維度「已確認可製作」是印製維度開跑的前提。稿件還沒審過、正在補件、或合格後業務尚未確認可製作，東西就不該先做下去，否則審完發現要改、已投產的就白做。因此印製維度在審稿維度到達「已確認可製作」前一律停在「等待中」。

## 關鍵狀態 / 轉換的營運動機

- 審稿維度「已確認可製作」後印製維度才推進 → 動機：稿件沒確認可印就投產，等於拿料和工時去賭客戶不改稿；先卡住印製、業務確認後再放行，避免做完才退稿的整批報廢。
- 審稿維度「合格」為業務確認前等待態，「已確認可製作」為終態 → 動機：合格代表審稿人員品質判定通過；業務確認可製作代表客戶不會再改稿、可投產。兩步分離品質判定與製作決策。B2C 訂單合格後系統自動跳到「已確認可製作」（行為不變）；B2B 線下單需業務手動確認。
- 審稿維度「合格 → 待改稿」退回重審路徑 → 動機：線下單客戶合格後仍可能改稿（換 Logo、改文字等），退回重審讓同一印件重新走審稿，避免「棄用 + clone」的操作繁瑣與多筆紀錄語意混亂。退回重審僅限「合格」狀態，「已確認可製作」後不可退回。
- 審稿維度「等待審稿 ↔ 不合格 ↔ 已補件 ↔ 再審」可來回 → 動機：稿件有問題本就要退回重給、補正後再審，這段來回是印前常態，做成可逆才不必每退一次就建新印件。
- 審稿維度「待改稿 → 等待審稿」與「不合格 → 已補件」的區別 → 「不合格」是審稿人員品質退件，「待改稿」是業務要求客戶改稿（品質已通過但內容需變更）。兩條路徑獨立但補件後都回到原審稿人員重審。
- 免審稿直通（稿件未上傳 → 合格）→ 動機：符合免審條件的印件（見 [[免審決策樹]]）不必走完整審稿，直接放行。B2C 再自動跳到「已確認可製作」；B2B 等業務確認。
- 與棄用 + clone 路徑並存 → 退回重審用於「同一印件改稿」（內容修改）；棄用 + clone 用於「印件規格全換」（如名片改成 DM、尺寸材質全變）。打樣 NG-稿件問題仍走棄用 + clone。
- 印製維度由底層生產任務多層向上傳遞驅動 → 動機：印件做到哪該跟著現場實際報工自動反映，不必有人逐張印件回填進度；底層做到哪一步，印件就顯示到哪一步。
- 任意狀態 → 已棄用 → 動機：打樣判定 NG 不做大貨時，原印件直接棄用、改建新印件接手，棄用的雙維度狀態保留作稽核軌跡（誰、為什麼棄）。

## 與其他狀態機 / 實體的關係

- 印製維度是向上傳遞鏈的中段：底層 [[生產任務狀態|生產任務]] → [[任務狀態|任務]] → [[工單狀態|工單]] → 印件印製維度 → [[訂單狀態|訂單]] 製作中。底層做到哪，印件就反映到哪，本卡是這條鏈的彙整節點。
- 審稿維度推進不會動到已建工單鎖定的檔案（避免改稿影響已投產批次），鎖檔規則見 [[稿件管理規則]]。
- 印件全部做齊（齊套達標）才會把「製作完成」往上帶到訂單，怎麼算齊見 [[齊套邏輯]]。
