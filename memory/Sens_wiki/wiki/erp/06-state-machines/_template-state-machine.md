# 狀態卡撰寫範本（state-machine）

> 給 Claude / Miles 撰寫 `06-state-machines/` 狀態機卡時複製套用，**同時**作為 [[vault-audit]] 事後稽核此類卡的勾稽尺（一份兩用：寫時照填、審時照勾）。
>
> 分層定位：依 [[wiki-architecture#分層體系（營運原則 → 驗收項目，由大到細）]]，state-machine 屬「狀態」這一層——把上層 [[business-logic-writing-guide#4.0 各類卡的單一職責|business-logic 規則]]展開為「資料層面的狀態變化」。**本卡承載狀態列舉（正本）與營運動機**，轉換規則（Given/When/Then + guard 條件）留在 OpenSpec 各模組 spec 的 Requirement。規則本體、計算公式、跨情境例子、營運動機長篇論述留 `04-business-logic`，本卡 wiki link 指過去。
>
> 上層約束（MUST 先讀）：[[business-logic-writing-guide#4.0 各類卡的單一職責]] 對應 type 職責 + [[business-logic-writing-guide#4.2 state-machine 卡（保留狀態 / 轉換，前置營運說明）]] 結構範式 + [[wiki-architecture]] 對應分層。本範本是上述兩份的合併展開，不與其衝突。
>
> 已落地的範例卡（可對照）：[[訂單異動狀態]]、[[分期請款狀態]]（皆已依 § 4.2 重寫，為正面範本）。

1. 複製下方「§ 三 範本本體」`--- 模板開始 ---` 至 `--- 模板結束 ---` 之間內容，存為 `06-state-machines/<狀態機中文名>.md`（命名見 [[wiki-schema#一般卡]]：繁體中文名詞片語、不用動詞 / 問號）。

4. 對照「§ 四 極簡填寫示意」確認結構落點正確（示意用佔位符，非真實內容）。

## 二、共同骨架在狀態卡怎麼落（撰寫前先掌握）

本範本依「所有卡共同的骨架」組裝（七項）。逐項說明在狀態卡怎麼落：

| 骨架 | 在 state-machine 卡的落法 |
|------|--------------------------|
| 一、Frontmatter 必填 | 共同 5 欄（type / module / business-domain / status / last-reviewed）+ 兩個方向欄（source 往上指依據 / implemented-by 往下指被誰實作），見 § 三 frontmatter 區佔位符與規約 |
| 二、正文共同 4 段 | 一句話定位（卡頂 `>`）+ `## 這張卡要回答的問題` + `## 營運背景` + `## 相關連結` + `## 來源`，見 § 三 |
| 三、定位點命名 | 狀態名 / 轉換用業務語意命名（如 `#已執行`、`#補收正項路徑`），不用流水號、不重排、不重用，見 § 三 § 狀態轉換指引 |
| 四、轉換→為什麼→一個例子 | 本層以「轉換 → 營運動機 → 1 例子」三件套呈現轉換動機；完整 if-then 規則論證留 business-logic 正本卡，本卡只引用 |
| 五、本層特有章節 | `## 狀態清單`（含「對應營運需求」欄）+ `## 狀態轉換`（ASCII 圖）+ `## 轉換條件與觸發事件` + `## 關鍵狀態 / 轉換的營運動機` + `## 與其他狀態機 / 實體的關係`，見 § 三 |
| 六、自審稽核點 | § 五 自檢清單（撰寫檢查表 = vault-audit 稽核維度同一份）|
| 七、硬規則 vs 彈性 | § 六 標注本層「不可違反」與「可因案調整」兩層 |

# <狀態機中文名>（<英文 Status 識別名，如 OrderAdjustmentStatus>）

> <一句話定位：這張卡是哪個實體的什麼狀態機、屬哪個業務領域；並聲明規則正本歸屬，如「轉換背後的分權／認列／回退規則，正本在 [[<規則正本卡>]]，本卡只定義狀態與轉換、不複述規則」。>

## 這張卡要回答的問題

> 這張卡要回答的關鍵狀態問題。每個問題在正文有對應明確結論；判斷「卡完整了沒」= 每個問題是否都有對應結論。狀態機卡的問題聚焦「狀態怎麼分、什麼條件轉、誰觸發」，**不問規則對錯**（規則對錯屬 business-logic 卡）。

- <問題 1：如「這個實體有哪幾種狀態？哪些是終態？」>
- <問題 2：如「補收與退款為什麼走不同轉換路徑？」（動機一句帶過，細則指 business-logic 正本）>
- <問題 3：如「狀態 X → Y 的觸發事件與條件是什麼？由人工或系統推進？」>

## 營運背景

> 該寫：這個流程在公司營運上要解決什麼（2-4 句、業務語言、無實作術語當主詞）。主管 / PM 入口，讀完應能判斷「狀態這樣分對不對」。
> 不該寫：規則本體、計算公式、長篇動機論述（→ business-logic 正本卡）。

<2-4 句營運背景。技術名詞須括號附註、不得當主詞；禁中英夾雜。>

## 狀態列舉（正本）

> **本段是該實體狀態的正本定義**。狀態的新增與修改是商業決策，直接在本卡維護，不經 OpenSpec change 工作流。轉換條件（Given/When/Then）留在 OpenSpec 各模組 spec 的 Requirement。
>
> 該寫：每個狀態的「狀態 / 說明 / 對應營運需求」三欄（或依路徑分子表）；對應營運需求講清楚「這個狀態存在是為了應付什麼營運情況」。
> 不該寫：認列公式、判斷準則細則、一定要成立的規則——只在表後加一行 wiki link 指 business-logic 正本卡。

| 狀態 | 說明 | 對應營運需求 |
|------|------|------------|
| <狀態 A> | <一句說明這個狀態是什麼> | <這個狀態為了應付什麼營運情況而存在> |
| <狀態 B> | <...> | <...> |
| <終態 X> | 終態；<...> | <...> |

> 各狀態的判斷準則、認列公式、一定要成立的規則見 [[<規則正本卡>]]，本卡不重述。

## 狀態轉換

> 該寫：ASCII 流程圖（一條路徑一塊）。狀態名與路徑名用業務語意命名（穩定、不流水號）。多條路徑（如補收 / 退款 / 系統內生）分塊呈現，標清楚發起方。
> 不該寫：UI 操作步驟（按鈕 / 彈窗）；轉換背後的規則論證（→ business-logic）。

<路徑一名稱>（<發起方 / 條件，如「金額 < 0，業務發起」>）：

\`\`\`
<狀態 A> ─<觸發動作>─▶ <狀態 B> ─<觸發動作>─▶ <終態 X>
<狀態 B> ─<退回動作>─▶ <狀態 C> ─<重交動作>─▶ <狀態 B>
\`\`\`

<路徑二名稱>（<發起方 / 條件>）：

\`\`\`
<狀態 A> ─<觸發動作>─▶ <終態 X>（<略過了哪些中間態的一句說明>）
\`\`\`

## 轉換條件與觸發事件

> 該寫：逐條轉換的「觸發事件」與「條件」（人工 / 系統自動推進）。系統自動推進的條件用業務語言描述判斷依據（如「對應生產任務全部完成」），不貼演算法。
> 不該寫：完整計算公式、跨情境窮舉例子。

| 轉換 | 觸發事件 | 條件 |
|------|---------|------|
| <狀態 A → 狀態 B> | <誰做了什麼 / 什麼系統事件> | <限定條件，或 ─> |
| <狀態 B → 終態 X> | <...> | <人工 / 系統自動；判斷依據一句> |

> 「為什麼這樣分權 / 認列 / 推進」的動機，見 [[<規則正本卡>]]。

## 關鍵狀態 / 轉換的營運動機

> 該寫：只針對「為什麼要有這個狀態 / 為什麼這樣轉換」需要展開的關鍵點，逐條「轉換 → 營運動機 → 1 個具體例子」。例子用真實格式（編號 / 金額 / 角色），停在業務語言。
> 不該寫：UI 措辭、抽象等價類、規則本體長篇論述、窮舉所有例子。

- <轉換 P → Q> → 動機：<營運上為什麼這樣轉，連回痛點 / 角色 / KPI 一句> → 例子：<真實格式情境，如「客戶對 #ORD-2026-0512 加印 500 份，業務直接執行應收 +6000」>。
- <轉換 R → S> → 動機：<...> → 例子：<...>。

## 與其他狀態機 / 實體的關係

> 該寫：本狀態機與哪些其他狀態機 / 實體互動、邊界在哪（什麼情況走別張卡）。用 wiki link。
> 不該寫：其他狀態機的內部轉換（只連過去，不複述）。

- <與狀態機 Y 的關係：如「本實體非終態時不阻擋主實體推進，主實體取消走 [[<主實體狀態卡>]]」>
- <與實體 Z 的關係：如「本狀態機推進到終態時重算某金額，但不自動建立 [[<另一狀態卡>]] 的期次」>

## 邊界

> 該寫：明確列出本狀態機不涵蓋什麼、什麼情況走哪張卡（防越界、防讀者誤用）。
> 不該寫：把別張卡的職責內容搬進來解釋。

- <邊界 1：某情境走哪張卡>
- <邊界 2：規則 / 公式 / 一定要成立的規則全貌 → 見 [[<規則正本卡>]]>

## 相關連結

## 五、自審稽核清單（撰寫完逐項打勾，同時是事後稽核基準）

> 一份兩用：作者撰寫完打勾；[[vault-audit]] 事後依此勾稽。共同項對齊共同骨架第六項，特有項對齊本層職責（[[wiki-schema#維度 14：卡類型內容職責邊界（2026-05-28 新增）]]）。

### 共同項（所有卡通用）

- [ ] frontmatter 共同 5 欄全填（type / module / business-domain / status / last-reviewed）；`source` 不指同層狀態機卡 / 下層 / OpenSpec。
- [ ] 「這張卡要回答的問題」段每個問題正文有對應結論（無懸而未答）。
- [ ] 「營運背景」段無實作術語當主詞、無中英夾雜（技術詞括號附註）。
- [ ] 正文零迭代史（迭代史在 [[business-logic-changelog]]）；正文零「待確認 / 待釐清 / 需確認 / 尚未確認 / 待補」inline OQ 措辭（識別到不確定項 → [[oq-manage]] mode B 開獨立卡，禁 `[!question]` callout）。
- [ ] 相關連結雙向可達、語意分類；沒有連到不存在的卡，也不是沒有任何卡連到它的孤島卡。
- [ ] 最後驗收：從「營運背景」抽一句，問「主管 / Miles 不看程式碼能否懂」——能，才算過。

### 本層特有項（state-machine 職責邊界）

- [ ] `## 狀態列舉（正本）` 表格含「對應營運需求」第三欄，每個關鍵狀態都有對應營運需求（不空泛）。狀態列舉為正本，新增 / 修改直接在本卡維護。
- [ ] `## 狀態轉換` 有 ASCII 流程圖；多條路徑分塊並標清楚發起方 / 條件。
- [ ] 狀態名 / 路徑名用業務語意命名（穩定可讀），**未用流水號（如 S1 / R1）當定位點、未重排、未重用**。

## 六、不可違反的硬規則 vs 可視情況調整的彈性

> 共同骨架第七項：明示「不可違反」與「可因案調整」兩層，禁作者私自偏離彈性以外的硬規則。

### 不可違反（硬規則）

- 禁中英夾雜（技術詞括號附註，不得當主詞）。
- 營運背景段禁工程術語 / 程式碼物件名當主詞。
- 識別到不確定項 MUST 開獨立 OQ（[[oq-manage]] mode B），禁 inline 標註 / 禁 `[!question]` callout。
- `source` 禁指下游（OpenSpec / prototype / test-case）或同層狀態機卡；只往上游指。
- 正文零迭代史（迭代史進 [[business-logic-changelog]]）。
- 規則單一正本不複寫：本卡禁複製 business-logic 規則本體 / 計算公式（只 wiki link 引用）。
- 本層禁退化成 business-logic（丟掉狀態 / 轉換、改寫成規則卡）。

### 可因案調整（彈性破口）

- **狀態極少且線性的簡單狀態機**（如僅 3 態無分支）：`## 狀態轉換` 可只一塊 ASCII；`## 關鍵狀態 / 轉換的營運動機` 若無「非顯然」轉換可精簡至 1-2 條或併入狀態清單「對應營運需求」欄。
- **無爭議的事實性狀態**（如「草稿 = 尚未送出」）：「這張卡要回答的問題」可不為其單列問題，聚焦真正需釐清的分流 / 推進問題。
- **`source` 暫補不出更高層卡**（如對應 business-logic 正本卡或營運原則 [[wiki-architecture#分層體系（營運原則 → 驗收項目，由大到細）|operating-principles]] 尚未建立）：過渡期允許僅填 `related-spec` 並在 frontmatter 標 `source-gap`，待專輪回填（回填依 `wiki-amend` skill 的 change 後增修機制與 [[business-logic-writing-guide]] 遷移誠實標注）；**新卡 / 被 change 異動的卡優先補 source**。
- **純措辭 / 排版修正**（不改狀態定義、不改轉換、不改觸發條件）：可不重跑完整自審清單，但仍須通過「無 inline OQ / 無中英夾雜」兩硬項。

> 彈性破口以外的偏離 MUST NOT 私自進行；有結構性疑問先走 [[oq-manage]] mode B 或回 [[business-logic-writing-guide]] 確認。
