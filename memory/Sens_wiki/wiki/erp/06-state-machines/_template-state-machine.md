# 狀態機卡撰寫範本

> 設計依據：BABOK 狀態建模（狀態表完整性）、Wiegers 多重表徵互核、UML 狀態機記法、SBVR／RuleSpeak 商業可讀原則、Business Rules 獨立原則（規則存在於軟體之外）。
> 本範本依「雙向開發」迭代：每張卡走查時發現的缺口回饋範本——範本不是一次寫完的完美文件，是隨真實撰寫經驗成長的工作標準。

---

## 〇、狀態機卡是什麼（先懂定位再動筆）

> 本節目的：每條規則都從這裡的定位推導出來。讀懂定位，遇到範本沒涵蓋的情況也能正確裁量。
> Wiki 的本質、第一讀者、讀者情境、兩個判準（紙本測試／商業語言）、BRD 自足與「範圍外＝防護欄」的完整定義見 **[[index]] § Wiki 憲章**——本節不重述，只講狀態機單元的特定定位。

**狀態機卡是兩件東西**：

1. **狀態詞彙的正本**——「製作中」「已收訖」是全公司協作的共同語言，業務、印務、會計說同一個詞必須同義。本卡首先是一份受控詞彙表：每個狀態＝一個業務詞＋它的定義＋它存在的營運理由。
2. **商業邏輯在「單據處境」維度的投影**——規則正本在 `04-business-logic/`，本層回答「這些決定反映在單據處境上是什麼樣子」：單據在什麼處境、因什麼事件、變成什麼處境。

**與鄰層的關係**（投影不互相複寫）：

- 對 `04-business-logic/`：規則的「為什麼」正本在 04，本卡只寫狀態怎麼變、引用不複述。
- 對服務藍圖：藍圖是**旅程視角**（時間軸、角色接力），本卡是**單據處境視角**。同一事件（主管核准訂單）在藍圖是一個步驟、在本卡是一條轉換。

**兩個判準在本單元的應用與已知陷阱**：

- 紙本測試：「主管核可後鎖定成交條件」紙本時代也成立 → 寫；歸納演算法的優先序只存在於軟體 → 不寫本體，但在「範圍外」點名（防實作腦補）。
- 已知陷阱：**把上游實體的旅程寫成本卡的「前段」**（訂單卡「三條前段路徑」教訓）——付費與諮詢過程屬諮詢單，不是諮詢訂單的前段。

**本單元主要伺候的讀者情境**（依憲章五情境）：商業自足理解（狀態詞彙表）、規劃自查（轉換與動機）、訪談確認（UML 圖）。

## 一、撰寫流程

> 本節目的：固定的管線讓每張卡品質一致；每步有明確產出，卡住有明確出口。

```
辨識 → 歸位 → 確認前提 → 撰寫 → 稽核 → 定案
```

| 步驟 | 做什麼 | 這一步的產出 | 卡住時 |
|------|--------|------------|--------|
| **1. 辨識** | 這個實體需要狀態機卡嗎？屬哪種結構變體？依 § 二判斷 | 開卡與否＋變體判定 | 走不通 → § 六 |
| **2. 歸位** | 一張卡＝一個實體的完整狀態機；雙維度併同一張卡 | 檔名與落點 | 走不通 → § 六 |
| **3. 確認前提** | 依 § 三逐項確認 | 已確認清單（任一缺 → 停下問） | 停下問使用者 |
| **4. 撰寫** | 依 § 八格式填入，遵守 § 四規則 | 卡草稿 | 需要假設 → § 七 |
| **5. 稽核** | 依 § 十二維度自檢 | 逐項勾稽結果 | 不通過 → 回 3 或 4 |
| **6. 定案** | commit ＋ 追加 wiki/log.md（逐卡 `[[卡名]]`＋動機） | — | — |

## 二、判斷表

> 本節目的：不是每個實體都值得開狀態機卡；開錯卡比缺卡更難收拾（會被引用）。

### 這個實體需要狀態機卡嗎（任一答「否」即不開）

| 順序 | 問自己 | 否的話 |
|------|--------|--------|
| 1 | 實體有生命週期嗎？（從建立走到結束，中間有可命名的處境） | 不開卡 |
| 2 | 狀態有三個以上，或處境變化有業務把關意義嗎？（角色審核、責任交接） | 純兩態開關寫在 `05-entities/` 實體卡 |
| 3 | 是獨立實體的狀態，而不是既有實體的第二條進度線嗎？ | 第二條進度線併入既有卡作雙維度 |

### 結構變體（三選一，第一個命中即答案）

| 順序 | 問自己 | 變體 | 既有範例 |
|------|--------|------|---------|
| 1 | 同一實體有兩條互不干擾、各自推進的進度線嗎？（負責人、節奏、可逆性不同） | **雙維度** | 分期請款（開票線 × 收款線）、印件（審稿線 × 印製線） |
| 2 | 狀態多且可分成幾個階段、階段間單向不可回頭嗎？ | **分段式** | 訂單（審核段 → 審稿段 → 製作段 → 出貨段） |
| 3 | 都不是 | **單鏈式** | 工單、需求單、諮詢單 |

### 歸位

- 檔名＝實體名＋「狀態」（如 `訂單狀態.md`），放 `06-state-machines/`（平目錄）。
- 一張卡＝一個實體的完整狀態機；雙維度的兩條進度線在同一張卡內分節。

## 三、前提確認（MUST 全部已確認才能動筆）

> 本節目的：把「寫的時候才發現不知道」提前到動筆前。答不出來的就是默會知識缺口——停下問，不准編。

| 前提 | 確認狀態 |
|------|---------|
| 狀態清單的來源依據在哪？（訪談／管理層拍板／法規／既有營運實務） | 已確認 / 待確認 → 停下 |
| 這個實體有幾種建立來路？各來路由誰、什麼事件觸發建立？建立之前的流程屬於哪個實體？（來路＝建立來源；前段＝本實體自身的狀態階段——兩回事） | 已確認 / 待確認 → 停下 |
| 初始狀態與終態是哪些？ | 已確認 / 待確認 → 停下 |
| 每個轉換由誰觸發？（人工角色／系統自動） | 已確認 / 待確認 → 停下（**不可假設觸發者**） |
| 轉換背後的規則正本在哪張商業規則卡？ | 已確認 / 找不到 → 先補 `04-business-logic/` 卡或開 OQ |
| 負向路徑（取消／退回／例外）怎麼走？ | 已確認 / 確認不存在（寫進範圍外） / 待確認 → 停下 |
| 與哪些其他狀態機有上下游互動？ | 已確認 / 暫無 |

## 四、撰寫規則

> 本節目的：規則強度分級——MUST 不容裁量、SHOULD 偏離須說明、MAY 自由。每條附理由，理解理由才能在沒涵蓋的情況正確裁量。

| 強度 | 規則 | 理由 |
|------|------|------|
| MUST | 每段內容過紙本測試與商業語言兩判準（§ 〇） | 本卡是 BRD；軟體細節寫進來會讓商業讀者讀不懂、讓正本與實作雙重維護 |
| MUST | 正文零實作規格引用（不出現 OpenSpec、程式檔名、演算法） | 第一讀者讀不到那些文件；BRD 必須自足 |
| MUST | 狀態列舉表含「狀態／說明／對應營運需求」三欄；初始標「初始；」、終態標「終態；」 | 詞彙表三要素：詞、定義、存在理由；起點終點是生命週期完整性的底線 |
| MUST | 狀態機圖（mermaid UML）與轉換表**雙表徵互核**：圖上每條轉換在表有對應列，反之亦然 | 兩種表徵互轉能暴露「沒有定義結果的處境與事件組合」（Wiegers 方法，走查實證有效） |
| MUST | 觸發事件寫「誰做了什麼」：「〔角色〕做〔動作〕」或「系統於〔業務事件〕自動推進」 | 固定句型全卡一致，業務讀者學一次讀懂所有卡；觸發者缺席＝責任不明 |
| MUST | 狀態名、角色名、單據名與狀態列舉、角色卡、實體卡**一字不差** | 詞彙先行：同一個詞在全 vault 同義，否則協作斷裂 |
| MUST | 概述敘事不越界：「前段／路徑／段落」只指本實體自身的狀態階段；上游實體流程以「來路」一句帶過；業務專名不挪用為泛描述；多來路用開放措辭（「現階段支援」）不寫死數量 | 投影混淆是已知陷阱；寫死數量會在新增類型時全面翻修 |
| MUST | 轉換背後的規則回引商業規則卡，不複寫規則本體與計算公式 | 規則只寫一次；複寫第二份必然失去同步 |
| MUST | 寫「範圍外」段：明確列出本卡不涵蓋什麼、屬於誰；**容易被實作腦補的主題點名「存在但不在本卡」** | 範圍外是防下游過度設計與捏造的防護欄 |
| SHOULD | 同一狀態因同一類事件分流到不同目標時，分流條件互斥不重疊 | 條件重疊＝同一情況兩種結果，營運會吵架 |
| SHOULD | 關鍵轉換的營運動機停在「轉換 → 動機一句 → 1 個例子」三件套；例子用真實業務格式 | 動機防止後人無意推翻有理由的設計；超過三件套的論述代表它夠格立 04 規則 |
| MAY | 簡單狀態機（三態以下無分支）省略動機段，動機併入列舉表的「對應營運需求」欄 | 顯然的事不必開段落 |

## 五、修改既有卡前的檢查

> 本節目的：狀態列舉是商業決策正本，改動是商業行為不是編輯行為。

| 修改類型 | 改之前要確認什麼 |
|---------|----------------|
| 新增／移除狀態 | 商業決策依據（管理層拍板／訪談定案）；沿引用反查連帶影響的卡 |
| 改轉換的觸發者或條件 | 同上；下游實作規格須由變更工作流同步（不在本卡處理） |
| 純措辭修正 | 不需商業依據，但仍須通過語言紀律稽核 |

## 六、邊際情境處理

> 本節目的：判斷表走不通時的標準出口，不准自行發明歸法。

| 卡住的情境 | 標準處理 |
|-----------|---------|
| 一個實體疑似有兩套狀態 | 問「兩條進度線的負責人、節奏、可逆性是否不同？」不同 → 雙維度；相同 → 其實是一條鏈，重新梳理 |
| 某狀態不確定該不該存在 | 開 OQ（`oq-manage` mode B），停下 |
| 某轉換講不出觸發者 | 停下問使用者，**禁止**寫「系統自動」充數 |
| 狀態跨實體向上反映（如工單做完帶動訂單前進） | 各實體各自開卡；上層卡只寫「接收下層結果」，不複寫下層內部轉換 |
| 不確定某內容歸本卡還是商業規則卡 | 問「這是『處境怎麼變』（本卡）還是『為什麼這樣定』（04）？」答不出來 → 停下 |
| 段落內子狀態越長越多 | 先與使用者評估，不擅自拆 |

## 七、error handle（出現任一即停下）

> 本節目的：高風險點的強制人工確認——讓 agent 亂猜風險高，停下來問風險可控。

1. **無來源依據**：狀態清單或轉換只有推測 → 停下
2. **矛盾**：與商業規則卡、服務藍圖、實作規格或既有營運實務矛盾 → 停下，列出矛盾兩方
3. **需要假設**：必須假設「應該是這樣轉」才能繼續 → 停下
4. **觸發者不明** → 停下
5. **結果未定義**：某處境遇到某事件的結果沒人說得出來 → 停下（這正是狀態機卡要逼出來的缺漏）
6. **超出已知**：要描述的生命週期超出已確認範圍 → 停下

處理方式：標記具體問題（「我不確定 X，因為 Y」）、向使用者提問附已知事實；**禁止**用「一般來說」「通常」「應該是」填充。改寫舊卡時發現的問題，保留舊內容另開 OQ。

## 八、產出格式

一張卡＝一個實體的狀態機。**檔名即標題**（卡內不寫 `# 標題`）。**正文零實作規格引用、無正文來源段**（正確性來源由 frontmatter `source` 承載；卡的迭代史查 wiki/log.md 與 git，不在卡內記）。

**段落結構**（七段；「為誰」＝這段伺候的讀者情境）：

| 段落 | 內容 | 為誰 | 必/選 |
|------|------|------|-------|
| 概述 | 哪個實體的狀態機＋英文識別名＋來路（開放措辭、條列）＋2-3 句營運背景＋規則正本歸屬聲明 | 第一次接觸的 PM | * |
| 狀態列舉（正本） | 「狀態／說明／對應營運需求」三欄表；初始與終態標注。**本段是該實體狀態的唯一正本，狀態新增修改直接在此維護** | 全公司共同詞彙 | * |
| 狀態機圖（UML） | mermaid `stateDiagram-v2` 依 UML 記法，圖前附一小段記法與分組語意說明 | 訪談確認、一眼掌握 | * |
| 轉換條件與觸發事件 | 表格：轉換／觸發事件（誰做了什麼）／條件；與圖互核 | 規劃自查、防誤解 | * |
| 關鍵轉換的營運動機 | 「轉換 → 動機一句 → 1 例子」，只挑不顯然的 | 防後人推翻有理由的設計 | ? 簡單狀態機可併入列舉表 |
| 與其他狀態機的關係＋範圍外 | 上下游互動與閘門（wiki link 指過去不複寫）；範圍外明列不涵蓋什麼、屬於誰、易腦補主題點名 | 邊界導航＋防下游捏造 | * |
| 相關卡 | 依分類列 wiki link（規則／流程／實體／狀態機／角色），只列有連結的分類 | wiki 互連 | * |

**frontmatter**：

```yaml
type: state-machine
module: [模組，中文值，見 wiki-schema § 二]
business-domain: [售前 / 訂單管理 / 印前審稿 / 生產執行 / 履約與售後 / 款項與發票 / 跨領域]
source:                # 往上＝正確性根據：商業規則卡／服務藍圖／外部依據（拍板、法規）；禁指同層、下層、實作文件
  - "[[<商業規則卡>]]"
implemented-by:        # 往下＝導航中繼資料（非正文內容）：實作規格檔層、原型型別檔；可多值、可留空
  - "openspec/specs/<模組>/spec.md"
status: draft | active
last-reviewed: YYYY-MM-DD
```

### 結構變體的段落調整

- **分段式**：狀態列舉依段落分子節（每段落一個三欄子表），列舉末尾寫「段落間可逆性規則」。
- **雙維度**：狀態列舉、狀態機圖、轉換表各分兩個子節（每維度自己的表與圖），加「兩維度的關聯」小節講清楚閘門關係與互不影響的部分。

### 狀態機圖（UML）記法規約

正面範本：[[訂單狀態]]、[[需求單狀態]]、[[諮詢單狀態]]。

- `direction LR`；初始 `[*] --> 初始狀態`、終態 `--> [*]`
- 建立分流用 `<<choice>>` 選擇偽狀態；轉換標籤「觸發事件 [守衛條件]」；狀態顯示名與列舉一字不差
- 段落＝複合狀態（`state "段落名" as 別名 { }`）；雙維度＝兩個複合狀態各框一條進度線
- **群組型轉換**（任一非終態皆可取消／流失）：**自段落複合狀態的邊界出發畫轉換**——依 UML 語意，從複合狀態邊界出發的轉換適用其內全部子狀態；圖前說明交代此語意
- **歸納型轉換**：落點由系統依下層狀態歸納時（向上反映驅動的常見模式），畫 `<<choice>>` 節點＋「落入對應狀態」概括；**詳細優先序不畫也不寫**（屬實作規格），但在「範圍外」點名
- 圖前說明段：一小段文字交代記法與分組語意

> 過渡註記：2026-06-10 前曾以同名 `.canvas` 檔承載狀態圖（訂單、需求單兩個 canvas 仍在本目錄），已由卡內 mermaid 段取代；既有 canvas 去留待使用者處置。

## 九、不收清單

> 本節目的：越界內容的路由表。發現越界：判斷歸屬 → 確認目標卡存在 → 已有則刪、沒有則搬移後刪。

| 不收 | 歸屬 | 判斷訊號 |
|------|------|---------|
| 規則本體、計算公式、認列邏輯 | `04-business-logic/` | 看到「為什麼這樣定」的完整論述、公式 |
