# 狀態機卡撰寫範本

> `06-state-machines/` 是 BRD 的狀態層——記錄實體的生命週期：有哪些狀態（**狀態列舉的唯一正本**）、什麼事件觸發轉換、由誰觸發。
> 核心紀律：**狀態列舉只寫一次**（本層是正本，其他卡引用不複寫）；轉換背後的規則不複寫（引用 `04-business-logic/` 商業規則卡）；轉換的系統行為規格不寫（歸 OpenSpec）。

---

## 一、撰寫流程（所有狀態機卡共用）

新狀態機（或既有狀態機變動）進來時，依以下步驟處理。**每一步都必須完成才能進入下一步。**

```
辨識 → 歸位 → 確認前提 → 撰寫 → 稽核 → 定案
```

| 步驟 | 做什麼 | 卡住時 |
|------|--------|--------|
| **1. 辨識** | 這個實體需要狀態機卡嗎？屬哪種結構變體？依 § 二判斷表走一遍 | 走不通 → 見 § 六邊際情境 |
| **2. 歸位** | 一張卡 = 一個實體的完整狀態機；雙維度併同一張卡 | 走不通 → 見 § 六邊際情境 |
| **3. 確認前提** | 依 § 三逐項確認前提 | 有未確認項 → 停下問使用者 |
| **4. 撰寫** | 依 § 八產出格式填入內容，遵守 § 四撰寫規則 | 需要假設才能寫 → 停下（§ 七） |
| **5. 稽核** | 依 § 十二稽核維度自檢 | 不通過 → 回步驟 3 或 4 修正 |
| **6. 定案** | commit | |

## 二、判斷表

### 這個實體需要狀態機卡嗎

依序問以下問題，**任一答「否」即不開卡**：

| 順序 | 問自己 | 否的話 |
|------|--------|--------|
| 1 | 實體有生命週期嗎？（從建立走到結束，中間經過可命名的階段） | 不開卡 |
| 2 | 狀態有三個以上，或轉換有業務把關意義嗎？（角色審核、系統自動推進） | 純兩態開關（啟用／停用）寫在 `05-entities/` 實體卡，不開卡 |
| 3 | 是獨立實體的狀態，而不是既有實體的第二條進度線嗎？ | 第二條進度線併入既有卡作雙維度，不另開卡 |

### 結構變體判斷（三選一）

依序問，**第一個命中的就是答案**：

| 順序 | 問自己 | 是 | 變體 | 既有範例 |
|------|--------|---|------|---------|
| 1 | 同一實體有兩條互不干擾、各自推進的進度線嗎？（負責人、節奏、可逆性不同） | 是 | **雙維度** | 分期請款（開票線 × 收款線）、印件（審稿線 × 印製線） |
| 2 | 狀態多且可分成幾個階段、階段之間單向不可回頭嗎？ | 是 | **分段式** | 訂單（審核段 → 審稿段 → 製作段 → 出貨段） |
| 3 | 都不是 | — | **單鏈式** | 工單、生產任務、需求單 |

### 歸位

- 檔名 = 實體名 + 「狀態」（如 `訂單狀態.md`），放 `06-state-machines/`（平目錄，不分子目錄）。
- 一張卡 = 一個實體的完整狀態機。雙維度的兩條進度線寫在同一張卡內各自分節。

## 三、前提確認（MUST 全部已確認才能開始撰寫）

| 前提 | 確認狀態 |
|------|---------|
| 狀態清單的來源依據在哪？（OpenSpec spec／訪談／管理層拍板） | 已確認 / 待確認 → 停下 |
| 這個實體有幾種建立來路？各來路由誰、什麼事件觸發建立？建立之前的流程屬於哪個實體？ | 已確認 / 待確認 → 停下（來路與前段是兩回事：來路 = 建立來源，前段 = 本實體自身的狀態階段） |
| 初始狀態與終態是哪些？（生命週期的起點與走到底不再動的狀態） | 已確認 / 待確認 → 停下 |
| 每個轉換由誰觸發？（人工角色／系統自動） | 已確認 / 待確認 → 停下（**不可假設觸發者**） |
| 轉換背後的規則正本在哪張商業規則卡？ | 已確認 / 找不到 → 先補 `04-business-logic/` 卡或開 OQ |
| 負向路徑（取消／退回／例外）怎麼走？ | 已確認 / 確認不存在（寫進「範圍外」） / 待確認 → 停下 |
| 與哪些其他狀態機有上下游互動？ | 已確認 / 暫無 |

## 四、撰寫規則

| 強度 | 規則 |
|------|------|
| MUST | 狀態列舉表含「狀態／說明／對應營運需求」三欄；初始狀態在說明欄開頭標「初始；」、終態標「終態；」 |
| MUST | 狀態轉換用文字流程圖呈現；多條路徑分塊，每塊標發起方與適用條件 |
| MUST | 轉換條件與觸發事件表逐條列「轉換／觸發事件（誰做了什麼）／條件」 |
| MUST | **圖與表互相核對**：文字流程圖的每條箭頭在轉換表都有對應列，轉換表的每列在圖上都有對應箭頭（兩種呈現互轉能暴露「沒有定義結果的狀態與事件組合」） |
| MUST | 命名：狀態用名詞或完成式（已Ｘ／待Ｘ／Ｘ中），觸發事件用動詞片語（誰做了什麼） |
| MUST | 轉換背後的規則回引商業規則卡，不複寫規則本體與計算公式 |
| MUST | 寫「範圍外」段（不涵蓋什麼、走哪張卡） |
| MUST | 概述敘事不越界：「前段／路徑／段落」只能指本實體自身的狀態階段；上游實體的流程以「來路」一句帶過、不展開（如付費與諮詢過程屬諮詢單，不是諮詢訂單的前段）；業務專名（如諮詢出口名「不做大貨」）不挪用為泛描述 |
| MUST | 每張狀態機卡含 `## 狀態機圖（UML）` 段：mermaid 狀態圖（stateDiagram-v2）依 UML 記法繪製，圖上狀態、轉換與狀態列舉、轉換表三方一致 |
| SHOULD | 同一狀態因同一類事件分流到不同目標時，分流條件互斥不重疊 |
| SHOULD | 關鍵轉換的營運動機停在「轉換 → 動機一句 → 1 個例子」三件套；例子用真實業務格式（編號／金額／角色）；要展開更多 → 回引 `04-business-logic/` 規則卡 |
| MAY | 簡單狀態機（三態以下且無分支）可省略「關鍵轉換的營運動機」段，動機併入狀態列舉表的「對應營運需求」欄 |

## 五、修改狀態機卡前的檢查

狀態列舉是商業決策、正本在本層。修改前依變動類型確認：

| 修改類型 | 改之前要確認什麼 |
|---------|----------------|
| 新增／移除狀態 | 商業決策依據（管理層拍板／訪談／OpenSpec change 定案）；沿用途檢查引用本卡狀態名的其他卡是否連帶 |
| 改轉換的觸發者或條件 | OpenSpec 對應模組 spec 的轉換規則是否需要同步 change |
| 純措辭修正（不動狀態與轉換） | 不需商業依據，但仍須通過 § 十二語言紀律維度 |

## 六、邊際情境處理

| 卡住的情境 | 標準處理 |
|-----------|---------|
| 一個實體疑似有兩套狀態 | 問「兩條進度線的負責人、節奏、可逆性是否不同？」不同 → 雙維度併同一張卡；相同 → 其實是一條鏈，重新梳理 |
| 某狀態不確定該不該存在 | 開 OQ（`oq-manage` mode B），停下 |
| 某轉換講不出觸發者 | 停下問使用者，**禁止**寫「系統自動」充數 |
| 狀態跨實體向上反映（如工單做完帶動訂單前進） | 各實體各自開卡；上層卡只寫「接收下層結果」，不複寫下層內部轉換 |
| 不確定某內容該寫本卡還是商業規則卡 | 問「這是『狀態怎麼變』（本卡）還是『為什麼這樣定』（`04-business-logic/`）？」答不出來 → 停下問使用者 |
| 段落內子狀態越長越多 | 先問使用者是否拆成下層實體的獨立狀態機卡，不擅自拆 |

## 七、error handle（卡住時的標準處理）

### 觸發條件（出現任一即停下）

1. **無來源依據**：狀態清單或轉換只有推測，沒有已確認的資訊支持 → 停下
2. **矛盾**：與商業規則卡、服務藍圖或 OpenSpec spec 矛盾 → 停下，列出矛盾兩方，問使用者哪個對
3. **需要假設**：必須假設「應該是這樣轉」才能繼續 → 停下
4. **觸發者不明**：講不出某條轉換是誰觸發 → 停下
5. **結果未定義**：某狀態遇到某事件的結果沒人說得出來 → 停下（這正是狀態機卡要逼出來的缺漏）
6. **超出已知**：要描述的生命週期超出目前已確認範圍 → 停下

### 處理方式

- 標記卡住的具體問題（「我不確定 X，因為 Y」）
- 向使用者提問，附上已知事實與不確定部分
- **禁止**用「一般來說」「通常」「應該是」填充未確認內容
- 如果是改寫舊卡時發現的問題，保留舊內容不動，另開 OQ 追蹤

## 八、產出格式

一張卡 = 一個實體的狀態機。**檔名即標題**（卡片內容不重複寫 `# 標題`，Obsidian 以檔名顯示）。

**段落結構**（必填標 *，選填標 ?）：

| 段落 | 內容 | 必/選 |
|------|------|-------|
| 概述 | 哪個實體的狀態機 + 英文識別名（如 OrderStatus）+ 2-4 句營運背景（這個生命週期在營運上解決什麼）+ 規則正本歸屬聲明（「轉換背後的規則正本在某張商業規則卡，本卡不複述」，用 wiki link 指向該卡） | * |
| 狀態列舉（正本） | 「狀態／說明／對應營運需求」三欄表；初始與終態在說明欄標注。**本段是該實體狀態的唯一正本**，狀態新增修改直接在此維護，不經 OpenSpec change | * |
| 狀態機圖（UML） | mermaid 狀態圖（stateDiagram-v2）依 UML 記法繪製，圖前附記法說明；記法規約見下方小節 | * |
| 狀態轉換 | 文字流程圖；多條路徑分塊，標發起方與條件 | * |
| 轉換條件與觸發事件 | 表格：轉換／觸發事件／條件；與流程圖互相核對 | * |
| 關鍵轉換的營運動機 | 只挑「為什麼這樣轉」不顯然的轉換，逐條「轉換 → 動機一句 → 1 個例子」 | ? 簡單狀態機可併入列舉表 |
| 與其他狀態機的關係 | 上下游互動與閘門關係，用 wiki link 指過去，不複寫對方內部轉換 | ? 有互動才寫 |
| 範圍外 | 明確排除不涵蓋的範圍 + 各走哪張卡 | * |
| 相關卡 | 依分類列出 wiki link（規則／實體／狀態機／角色／OQ），只列有連結的分類 | * |
| 來源 | OpenSpec spec 檔層 markdown link + 對應轉換規則的 Requirement 名稱用文字描述（不綁標題錨點）；末行「迭代脈絡見 [[changelog]]」 | * |

**frontmatter**：

```yaml
type: state-machine
module: [模組，英文 token，見 wiki-schema § 二 module Enum]
business-domain: [售前 / 訂單管理 / 印前審稿 / 生產執行 / 履約與售後 / 款項與發票 / 跨領域]
source:                          # 往上 = 正確性根據：商業規則卡或外部依據；禁指 OpenSpec／同層狀態機卡／下層
  - "[[<商業規則卡>]]"
implemented-by:                  # 往下 = 導航：OpenSpec spec 檔層，不綁 Requirement 標題錨點；可多值、可留空
  - "openspec/specs/<模組>/spec.md"
status: draft | active
last-reviewed: YYYY-MM-DD
```

### 結構變體的段落調整

**分段式**：「狀態列舉（正本）」依段落分子節（每段落一個三欄子表，如審核段／審稿段／製作段／出貨段），並在列舉末尾寫「段落間可逆性規則」（哪些段落間不可回頭、哪些狀態間可來回）。

**雙維度**：「狀態列舉（正本）」「狀態轉換」「轉換條件與觸發事件」各分兩個子節（每維度自己的表與圖），並加一個「兩維度的關聯」小節，講清楚閘門關係（如審稿維度到「已確認可製作」後印製維度才開跑）與互不影響的部分（如發票作廢不動收款進度）。

### 文字流程圖格式

一條路徑一塊，狀態名與卡內列舉一字不差。**生命週期起點（建立事件）也要畫**，否則圖表互核時「（建立）→ 初始狀態」這列必漏：

```
（<建立事件>）─▶ <初始狀態> ─<觸發動作>─▶ <狀態 B> ─<觸發動作>─▶ <終態>
<狀態 B> ─<退回動作>─▶ <狀態 C> ─<重交動作>─▶ <狀態 B>
```

**歸納型轉換的寫法**：當某轉換的落點由系統依下層狀態歸納（一個來源可能落入多個目標，如訂單回簽後依印件審稿現況落入審稿段不同狀態），圖與表都寫「來源 ─系統歸納─▶ 目標甲／目標乙／目標丙」一條概括，**歸納優先序細則指 OpenSpec 不複寫**。向上反映（bubble-up）驅動的狀態機常用此寫法。

### 狀態機圖（UML）記法規約

`## 狀態機圖（UML）` 段放在「狀態列舉（正本）」之後，用 mermaid `stateDiagram-v2` 依 UML 狀態機圖記法繪製（正面範本：[[訂單狀態]]、[[需求單狀態]]）：

- **方向**：`direction LR`（由左至右）
- **初始／終止**：實心圓 `[*] --> 初始狀態`、`終態 --> [*]`（UML initial／final pseudostate）
- **建立分流**：訂單類型等建立時分流用 `<<choice>>` 選擇偽狀態（菱形），轉換標籤寫「[守衛條件] 觸發事件」
- **段落 = 複合狀態**：分段式狀態機用複合狀態（`state "段落名" as 別名 { ... }`）框出段落；雙維度用兩個複合狀態各框一條進度線
- **轉換標籤**：「觸發事件 [守衛條件]」格式（UML trigger [guard]），與轉換表一致
- **群組型轉換**：「任一非終態皆可取消／流失」這類轉換，用一個外層複合狀態把進行中狀態框起來、從框畫一條轉換出去，並在圖前說明標注「外框是圖面分組，不是實際狀態」
- **圖前說明段**：一小段文字交代記法、圖面分組的語意、歸納細則指 OpenSpec
- 狀態顯示名用引號中文（`state "草稿" as s_draft`），與狀態列舉一字不差

## 十、撰寫紀律

1. **狀態列舉只寫一次**：本層是狀態正本。實體卡、情境卡、服務藍圖只能引用狀態名，不得另列狀態清單
2. **規則回引不複寫**：轉換背後的「為什麼」指向商業規則卡，本卡停在「狀態怎麼變、誰觸發」
3. **禁中英夾雜**：技術詞括號附註，不直接混入中文句子
4. **禁實作術語當主詞**：用角色、業務動作描述觸發事件，不用欄位名或程式碼物件
5. **不確定項開 OQ**：觸發 `oq-manage` 開獨立 OQ 卡，不在正文內留 inline 標注
6. **範圍外必寫**：明確說不涵蓋什麼、走哪張卡
7. **檔名即標題**：卡片內容不重複寫 `# 標題`
8. **正文零迭代史**：改了什麼、為什麼改進 [[changelog]]，正文只呈現當前生效內容
9. **命名一致**：狀態名詞、事件動詞；狀態名不用流水號、不重排、不重用
