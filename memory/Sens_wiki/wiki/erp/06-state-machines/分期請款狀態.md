# 分期請款狀態（BillingInstallmentStatus）

> [[訂單]] 的請款期次（BillingInstallment，介面別名「收款項目」）狀態機：管「這一期該收多少、何時開票、何時收款」的進度。

## 營運背景

印刷廠收錢與開發票的時點常常對不上：客戶可能先付訂金過幾週才要發票（先收後開）、也可能要先拿到發票去公司請款月底才匯款（先開後收），大宗訂單還會頭尾款分期。過去把「收款進度」和「開票進度」綁在同一條狀態鏈，碰到先收後開就卡住，業務只能用備註硬記。

因此請款期次同時掛兩條互不干擾的進度線：一條記開票走到哪、一條記收款收了多少。業務可以彈性規劃每期金額與時點，調整期次（改到期日、改預計開票日、拆期）不會驚動主管回頭重審整張訂單——這類調整在印刷廠是天天發生的家常事，每改一次就要主管簽核會拖垮交期。

## 雙維度概觀

請款期次同時維護兩個獨立的進度線（技術上稱「雙維度」），互不耦合：

| 進度線 | 欄位 | 性質 | 對應營運需求 |
|------|------|------|------------|
| 開票進度 | invoicing_status | 業務手動操作驅動 | 客戶何時要發票由業務按需開立，與收沒收到錢分開 |
| 收款進度 | payment_status | 由核銷明細自動累計推導（技術上稱「派生」）| 收了多少由實際入帳累加得出，業務不用手動標記避免漏記 |

## 開票進度（invoicing_status）

```
未開立 ─業務開立發票─▶ 已開立（記下對應發票編號）
已開立 ─對應發票作廢─▶ 已作廢（解除發票關聯，可重開）
已作廢 ─業務重新開立─▶ 已開立
```

| 狀態 | 說明 | 對應營運需求 |
|------|------|------------|
| 未開立 | 尚未為這期開發票 | 客戶還沒要票、或先收訂金階段 |
| 已開立 | 已開出發票並記下發票編號 | 客戶拿到發票可向其公司請款 |
| 已作廢 | 原發票作廢、關聯解除，可重新開立 | 統編填錯 / 抬頭錯需作廢重開，不必刪期次 |

## 收款進度（payment_status，自動推導）

依「未取消且已完成」的核銷明細（PaymentAllocation，記錄一筆收款分攤到哪一期）累計推導，業務不必手動標記：

| 累計已收 | payment_status | 對應營運需求 |
|---------|----------------|------------|
| 等於 0 | 未收 | 這期還沒收到任何錢 |
| 0 < 累計 < 該期應收 | 部分收款 | 客戶分批付、或只付了部分 |
| 累計 ≥ 該期應收 | 已收訖 | 這期收齊 |

## 規則與營運動機

| 規則 | 營運動機 |
|------|---------|
| 開票進度與收款進度互不耦合，支援「先收後開」與「先開後收」 | 對應印刷廠兩種真實收款節奏：訂金先進帳發票後補、發票先給客戶錢月底才到。綁在一起會讓其中一種流程卡住，業務只能靠備註硬記，對帳時對不上 |
| 收款進度由核銷明細自動累計推導，業務不手動標記 | 業務經手大量收款，手動勾「已收訖」必漏記；自動累加確保收款進度永遠等於實際入帳，餵 [[會計]] 月結對帳的數字才可信 |
| 發票作廢時開票進度回「已作廢」可重開、收款進度不動 | 統編 / 抬頭填錯在實務常見，作廢重開不該連帶抹掉已收到的錢的紀錄；保留收款歷史是稽核要求 |
| 拆期時原期次標記取消（不計入兩條進度推導）、兩筆新期次各自從零起算；期次與發票業務上一對一（一期一票，拆票即拆期）| 客戶臨時要把一期拆成頭尾款是常態。一期一票對齊發票實務（一張發票對一筆請款），拆票時若不拆期，發票與期次就對不齊，月結對帳會抓到差額 |
| 期次調整（改到期日 / 改預計開票日 / 拆期 / 取消）不觸發訂單回業務主管審核，改記操作軌跡 + 累計變更次數 | 這類調整在印刷廠天天發生，每改一次就回審會拖垮交期、淹沒主管。改採事後稽核（[[業務主管]] 看變更次數指標 + 完整操作軌跡），把主管從逐筆簽核解放，異常仍可事後追 |

## 雙進度獨立的四種組合

| 情境 | 開票進度 | 收款進度 | 對應營運需求 |
|------|---------|---------|------------|
| 先收後開 | 未開立 | 已收訖 | 客戶先付訂金、發票之後補 |
| 先開後收 | 已開立 | 未收 | 發票先給客戶、月底才匯款 |
| 開票後作廢重開 | 已開立 → 已作廢（可再開立）| 不動（保留收款歷史）| 統編誤填作廢重開，不抹收款紀錄 |
| 拆期 | 原期次取消不計入推導 | 兩新期次各自獨立推導 | 客戶臨時要拆頭尾款 |

## 具體例子

例子一（先收後開，正例）：客戶下單 #ORD-2026-0512，業務規劃一期 BI-001 應收 30,000。客戶先匯訂金 30,000，業務登錄收款並把這 30,000 全分攤到 BI-001、切為已完成——BI-001 收款進度自動推導為「已收訖」，但開票進度仍是「未開立」（客戶月底才要發票）。月底業務替 BI-001 開出發票，開票進度才轉「已開立」，收款進度維持「已收訖」。兩條進度線各走各的，業務不必為了開票去動已收到的 30,000 紀錄。

例子二（發票作廢重開，邊界）：BI-002 已開立發票 INV-002、收款進度為「已收訖」。會計核對時發現買受人統編填錯，業務作廢 INV-002（填作廢原因「統編誤填」）。此時開票進度從「已開立」退回「已作廢」、解除與 INV-002 的關聯，但收款進度仍維持「已收訖」——已收到的錢的事實不因發票作廢而消失。業務隨後重新開出正確發票，開票進度回到「已開立」。

例子三（拆期，邊界）：BI-003 應收 78,000，客戶臨時要求拆成頭款 2,500 + 尾款 75,500。業務拆期後，BI-003 標記取消（取消原因「拆兩期」）且不再計入任何進度推導；新生兩期 BI-003-A（2,500）、BI-003-B（75,500）各自從「未開立 / 未收」獨立起算，各對一張發票。

例子四（期次調整不回審，正例）：訂單 #ORD-2026-0512 已過業務主管審核、進入製作中，BI-020 原到期日 2026-06-01。客戶要求延後付款，業務把到期日改為 2026-06-15。系統記下這次調整（含操作人、前後值）並把該期變更次數累計為 1，訂單狀態維持「製作中」、不退回「業務主管審核」。業務主管事後從變更次數指標即可看出哪些期次被頻繁調整，不必逐筆簽核。

## 邊界

- 補收 / 退款的金額異動本身走 [[訂單異動狀態]]（OrderAdjustment）；補收正項免審直達已執行、退款負項走主管核可，與本期次狀態機分屬不同實體。補收的訂單異動已執行後「不」自動建期次（避免重複認列），由業務視需要另行規劃期次。
- 退款的最終實付走退款收款紀錄（Payment）切已完成，不在本期次的收款進度推導範圍內（本期次只累計正向應收的收款）。
- 訂單整體何時推進 / 取消由 [[訂單狀態]] 管轄；期次規劃不阻擋也不回退訂單狀態。
- 月結三方對帳（應收 = 發票淨額 = 收款淨額）的底線正本屬 [[對帳一致性]]，本卡只負責單期次的兩條進度線。
