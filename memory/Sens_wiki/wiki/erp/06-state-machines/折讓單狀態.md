# 折讓單狀態（SalesAllowanceStatus）

> [[折讓單]]（銷貨折讓證明單，依台灣統一發票使用辦法第 20 條）從開立到可能作廢的狀態機。Mockup 階段精簡為兩態，開立即確認、不設草稿。作廢後不參與三方對帳、發票可折讓金額回復。折讓計算規則與對帳邏輯的正本在 [[付款發票邏輯]]，本卡只定義狀態與轉換、不複述規則。

## 狀態列舉（正本）

> 本段是折讓單狀態的正本定義。狀態的新增與修改是商業決策，直接在此卡維護。轉換條件（Given/When/Then）留在 OpenSpec [order-billing/spec.md](../../../../openspec/specs/order-billing/spec.md) 行為規格中。

| 狀態 | 說明 | 對應營運需求 |
|------|------|------------|
| 已確認 | 業務開立折讓，Mockup 立即觸發確認；折讓正式生效，影響三方對帳；終態之一 | 折讓即時生效，發票可折讓金額扣減 |
| 已作廢 | 業務作廢已確認的折讓單（情境：金額打錯 / 客戶撤回投訴 / 開錯發票 / 雙重開立）；終態之一 | 折讓失效、發票可折讓金額回復、不參與對帳 |

> 折讓金額計算、發票可折讓金額的完整規則見 [[付款發票邏輯]]，本卡不重述。

## 狀態轉換

折讓單（Mockup 階段開立即確認，業務可作廢）：

```
（建立時直接寫入）─▶ 已確認 ─業務作廢折讓（填原因）─▶ 已作廢
```

## 轉換條件與觸發事件

| 轉換 | 觸發事件 | 條件 |
|------|---------|------|
| （建立）→ 已確認 | 業務開立折讓，系統 Mockup 呼叫藍新兩段式 API（開立 + 確認） | 自動；開立與確認同時完成 |
| 已確認 → 已作廢 | 業務點「作廢折讓」並填入原因 | 人工；該發票可折讓金額回復 |

## 營運背景

折讓單是台灣統一發票制度的一環（統一發票使用辦法第 20 條）：已開出的發票金額若需減少（退費、折讓），不能直接改發票，必須另開一張折讓證明單。Mockup 階段簡化為兩態——開立即確認、不停留在草稿——因為業務開折讓就是要即時生效，多一個草稿態只增加手動點擊而無實務意義。未來真實串接藍新 API 若需要「等買受人確認」流程，再另開變更補草稿狀態。

作廢機制是折讓的 undo：金額打錯、客戶撤回投訴、開錯發票引用的折讓，都可以作廢讓那張發票的可折讓金額回到作廢前。

## 關鍵狀態 / 轉換的營運動機

- 開立即確認（無草稿）→ 動機：Mockup 階段業務開折讓就是要即時生效影響對帳，多一個草稿態徒增操作步驟 → 例子：客戶投訴 #ORD-2026-0512 色差，業務開立 2000 元折讓，系統直接寫入「已確認」，對帳面板即時扣減 2000。
- 已確認 → 已作廢 → 動機：折讓開錯需要一條回退路徑，作廢後發票可折讓金額回復，確保帳目正確 → 例子：上述折讓金額應為 3000 非 2000，業務作廢後重開 3000 元折讓。

## 與其他狀態機 / 實體的關係

- 折讓單引用有效（開立）的 [[發票狀態|發票]]，一張發票可被多張折讓引用，但折讓總額不超過發票金額。
- 作廢的折讓不影響訂單狀態或異動單狀態。
- 三方對帳時，發票淨額 = 開立發票金額 - 已確認折讓金額；已作廢折讓不扣減。
