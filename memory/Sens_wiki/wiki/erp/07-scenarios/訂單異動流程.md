# 訂單異動流程（補收 / 退款端到端）

> 訂單成立後客戶要求改金額（加印補收 / 退費）時，從「業務開異動單」到「帳對平」的兩條跨模組從頭到尾旅程。這張卡只講「這件事在誰手上、走到哪個階段」；每一步該照哪條規定，會標出去那條規定的卡片（[[訂單異動規則]]）、這裡不再重抄一遍（規定本身對不對，去看那張規定的卡；這張卡只看整個流程從頭到尾接得順不順）。

## 這張卡要回答的問題

> 以下問句聚焦「整段流程串得對不對」（不是某條規則對不對——那屬規則卡）。每題在情境清單與對帳那一段都有對應結論，據此判斷這四條旅程驗證完整了沒。

- 客戶在製作中加印補收，從業務建單、開票、到客戶付款核銷，中間經過哪幾步？哪一步漏掉這筆收入就收不回來、對帳會湊不平？（旅程 A）
- 訂單取消退款，從業務取消、主管核可、實際退錢、到會計把帳對平，中間要經過哪幾個人的手？哪一步漏掉，帳面上看起來退了、其實錢根本沒退出去？（旅程 B）
- 退款牽動的那張已開出去的發票，什麼時候開折讓單沖銷、什麼時候直接作廢重開？這兩條分支在哪裡岔開、各自接到哪張卡？（旅程 B）
- 訂單已完成後才發生的客訴退款 / 補印補收，是沿用訂單期間那條同一套流程，還是另開一條例外？在哪一步交給售後服務單接手？（旅程 C）
- 諮詢取消退一半費用這條系統自動跑的旅程，跟業務手動發起的退款，流程差在哪一步？哪些後續動作對諮詢訂單是空轉（沒東西可做）？（旅程 D）
- 四條旅程最後是不是都回到同一個「應收 = 發票淨額 = 收款淨額」的對帳檢查點？哪些情況差額對不平、要會計回頭核對？（跨旅程對帳）

## 營運背景

印刷廠接單後金額常臨時變動：客戶在製作中追加印量、改急件、補運費（公司收更多錢，稱補收）；或因色差、規格不符要退費（公司退錢出去，稱退款）。這兩個方向的風險輕重不一樣——退出去的錢出錯難救、收進來的錢是好事，所以走兩條不同的核可路徑。退款還牽動已開出去的發票（要開折讓單沖銷，或作廢重開），最終都要在月結時讓「應收 = 發票淨額 = 收款淨額」三方對齊，差額不為零就要會計回頭核對。本卡把這兩條旅程從頭到尾串起，讓主管驗證「整段跨部門流程是否走得通」。

## 情境清單

> 本卡承接既有索引（[[wiki/erp/07-scenarios/README]]）的情境 8（加印追加補收）與情境 16（取消退款），並補上售後階段（情境 10 / 13）與諮詢取消（2026-05-30 諮詢取消收斂那次新增）兩條變體，集中呈現「訂單異動」這一主題的完整旅程。

### 旅程 A：訂單期間加印補收（情境 8）

- **觸發狀況**：訂單已成立、仍在製作中，客戶臨時加印 / 加運費 / 要求急件。
- **經手的人 / 流程走到哪一步**：
  1. **業務**（[[業務]]）於訂單開補收正項異動單（金額是正的），依 [[訂單異動規則#補收免審|補收免審規則]]，這筆補收單直接從「草稿」變成「已執行」（[[訂單異動狀態]]）——跳過「送主管審核」和「主管核可」這兩關、不用經過 [[業務主管]]、也不牽涉到退款那邊；單子一建好，這筆該收的錢當下就算進這張訂單的應收金額。
  2. **業務**依 [[訂單異動規則#應收總額認列公式|認列不自動改期次規則]] 自己決定這筆補收要併到哪一期去跟客戶收：於對應請款期次（[[分期請款狀態]]）開立發票（依 [[付款發票邏輯]] §C4 線下單補收、§A2 異動補收分類）。
  3. **業務**待客戶實際付款後，登錄收款核銷分配，把那一期的「收款狀態」往前帶（例如從沒收變成已收，[[分期請款狀態]]）。
- **具體例子**：訂單 #ORD-2026-0512 製作中，客戶把名片從 500 張加到 800 張。業務建補收異動單 +6000 元（類型：加印追加），直接成為已執行、這張單的核可人就掛業務自己的名字（等於沒經過別人核），應收當場 +6000；整批貨照常生產不停。業務隨後就這 +6000 開立一張發票併入尾款期次，客戶付款後核銷該期次。

### 旅程 B：訂單取消退款（情境 16）

- **觸發狀況**：訂單因故取消，且客戶已先付過款，需退費。
- **經手的人 / 流程走到哪一步**：
  1. **業務**（[[業務]]）取消訂單（訂單狀態改成「已取消」，這是最後狀態、不會再變，[[訂單狀態]]），並建退款負項異動單（金額是負的），依 [[訂單異動規則#退款送審|退款送審規則]] 送出（[[訂單異動狀態]]，單子從「草稿」送去「待主管審核」）。對應步驟 [[US-ORD-011-訂單取消與退款]]。
  2. **業務主管**（[[業務主管]]）核可（單子變成「已核可」）、或退回（單子變成「已退回」並寫退回原因，業務改完再送一次）；依 [[訂單異動規則#退款送審|退款送審規則]]，主管核可過後，業務還能改金額、不用再送一次審。
  3. **業務**依 [[訂單異動規則#退款已執行認列與回退|退款認列規則]] 建一筆退款紀錄（金額是負的或零、掛在這張異動單下面），客戶實際退款後補對帳資料並切「已完成」；當實際退的錢累積到跟異動單上的金額一樣多時，系統就在同一個動作裡把這張異動單變成「已執行」，並重新算一次該收的總金額。
  4. **業務 / 諮詢**依 [[訂單異動規則#來源售後服務單標記|發票後處理規則]] 看發票目前的狀態，二選一處理（這件事跟上面退款單那條線同時進行、互不影響）：發票已經開出去、又要留著這張發票 → 就在這張發票上開一張折讓單（金額是負的、折讓金額不能超過這張發票還能折讓的額度，依 [[付款發票邏輯]] §A3、§F8）並關聯這筆退款紀錄；發票開錯可作廢 → 把原本那張發票作廢（要填至少 6 個中文字的作廢原因，作廢掉的發票字軌號碼不能再用、要換下一個號碼重開，依 [[發票法規硬約束-ezPay-MIG|ezPay 法規硬約束]]）。折讓 / 作廢都不需要業務主管或會計核可。
  5. **會計**（[[會計]]）於月結對帳依 [[付款發票邏輯#六、三方對帳|三方對帳規則]] 核對：應收總額（把所有已執行的異動單金額全部加進去）= 發票淨額（已開發票 − 已確認折讓）= 收款淨額（只算已完成款項、退款算負的）；差額是零就代表帳對平了、過關，差額不是零，系統就跳出提醒要會計回頭查。對應步驟 [[US-ORD-013-會計執行退款處理]]。
- **訂單取消後系統自動連動**：訂單一取消，系統會由上往下、一路把底下還在跑的工單、任務、生產任務全部一起停掉（依 [[訂單狀態]]）；這串連動是系統自己做的，業務不用一個一個手動去關。
- **具體例子**：訂單 #ORD-2026-0488 因色差客訴取消，客戶已付 5000 元。業務取消訂單並建退款異動單 −5000 元送審 → 業務主管核可 → 業務實際匯款並建退款款項 −5000 切已完成，異動單變成已執行；該訂單先前已開 5000 元發票，業務於該發票開立 −5000 折讓單關聯這筆退款款項。會計月結時：應收 0（−5000 異動已執行）= 發票淨額 0（5000 − 5000 折讓）= 收款淨額 0（+5000 − 5000）→ 對帳通過。

### 旅程 C：已完成訂單售後退款（情境 10 / 13）

- **觸發狀況**：訂單已完成後客訴 / 不良 / 規格不符，不能像訂單還在進行時那樣、直接在訂單上開異動單。
- **經手的人 / 流程走到哪一步**：
  1. **業務**於售後服務單（[[售後服務]]）容器內處理（受理中 → 填處理結果 → 處理中）。
  2. 若處理結果為退款，**業務**於售後單內建關聯退款異動單（標記來源售後服務單），依 [[訂單異動規則#來源售後服務單標記|售後退款同流程規則]]：後面退款的流程，跟旅程 B 的第 2 到第 5 步走完全一樣的關卡（[[訂單異動狀態]]）和發票處理方式——售後不會另外搞一套特殊流程。
  3. 若處理結果為補印且責任歸屬為客戶承擔，**業務**於售後單內開一張補印的印件（依 [[打樣流程|補印跳過審稿規則]]，補印不用重新送審稿，系統自動算它審稿過關、直接沿用原本那份稿）並建補收異動單（依旅程 A 補收免審規則）；如果是公司認賠，補印不收錢、就不用開異動單（沒有金額變動）。
- **具體例子**：訂單 #ORD-2026-0455 完成後客戶反映色偏，業務於售後服務單決議退款 3000。售後單內建退款異動單 −3000（標記來源售後服務單），走業務主管核可 → 建退款款項 −3000 → 已執行，與旅程 B 走完全一樣的流程。

### 旅程 D：諮詢取消退一半費用（這條是系統自動跑的，2026-05-30 諮詢取消收斂那次定的）

- **觸發狀況**：客戶諮詢後於待諮詢階段取消，依公司規則退一半諮詢費。
- **經手的人 / 流程走到哪一步**：
  1. **諮詢人員 / 業務主管**於諮詢單（[[諮詢單]]）取消（選取消原因分類），系統自動開一張諮詢訂單（訂單狀態直接設成「已取消」，[[訂單狀態]]），並把諮詢費 2000 元這筆費用、還有客戶已付的 2000 元收款，一起轉到這張訂單上。
  2. 系統依 [[訂單異動規則#諮詢取消退費系統建已核可|諮詢取消退費規則]] 自動開一張退費異動單（退 1000 元、狀態直接是「已核可」、核可人掛「系統」、但還沒實際退錢），不用業務主管再審一次。
  3. **諮詢人員**實際退款 1000、建退款款項 −1000 切已完成，依 [[訂單異動規則#退款已執行認列與回退|退款認列規則]] 把異動單變成「已執行」；該收的錢扣掉退的 1000 後，剩 2000 − 1000 = 1000 元。系統不會自動開「待開發票」、不自動開發票、也不自動開折讓；剩下的 1000 元收入由業務自己去開發票，萬一忘了開，靠對帳時跳出的差額提醒來補抓。
- **系統自動連動（諮詢訂單沒東西可連動）**：諮詢訂單底下沒有印件、工單、生產任務，所以旅程 B 那種「由上往下一路停掉」的動作，對諮詢訂單來說沒東西可停、等於空跑。
- **具體例子**：客戶諮詢後取消，諮詢費原收 2000。系統自動建退費異動單 −1000（已核可、核可人系統），諮詢人員退款 1000 切已完成後異動單變成已執行；留存 1000 由業務後續手動開立發票，會計對帳時，這種「已取消但還有收進來的錢」的訂單還是會被抓出來查（以前只查已完成的訂單，這次改成連這種也要查）。

## 跨旅程對帳（會計把帳對平）

這四條流程最後都要回到同一個檢查點——三個數字（該收、已開發票、實際收到）要對得起來（依 [[付款發票邏輯#六、三方對帳|三方對帳規則]]，規則本身在規則卡，本段只描述會計的查核流程）：

- **業務 / 會計隨時對帳**：「該收的總金額（把已執行的異動單金額都加進來）」要等於「已開發票減掉已確認折讓的金額」、也要等於「實際收到的錢（只算已完成、退款算負的）」，三個數字一樣就過關。
- **訂單完成後才異動的提醒**：如果異動單實際做的時間，比訂單完成的日期還晚（不管是不是售後造成的，都一起算），對帳畫面會跳出一條提醒，要會計回頭去核對當初月結的紀錄。
- **已取消但還有收入的也要查**：諮詢取消還留著 1000 元、或一般取消但還留著部分收入又已經開過發票的訂單，都要被抓出來查（以前只查已完成的，這次改成連這些也查）；已取消但發票還沒開的，就靠「該收的比已開發票多」這個差額提醒來補抓。
- **會計批次對帳**：會計可以照帳務公司、開立期間、對帳狀態一次查一批，把「差額不是零」還沒對帳的訂單挑出來，把對帳檔匯出來（格式是每一張已開立的發票占一行、共 14 個欄位，這格式是之前帳務整合那次定的）、拿去藍新後台對帳。對應步驟 [[US-ORD-024-會計匯出14欄對帳CSV]]。

## 範疇外

- 異動單各個狀態之間怎麼轉、哪些規矩永遠要成立 → [[訂單異動狀態]]（那張是狀態的標準依據）。
- 補收不用審 / 退款要送審 / 諮詢取消由系統直接掛已核可——這些「誰能做什麼」的規定本身和怎麼算的公式 → [[訂單異動規則]]（規定的標準依據，本卡只引用）。
- 發票開立 / 折讓 / 作廢的法規硬約束（五欄、14 天作廢期、字軌）→ [[發票法規硬約束-ezPay-MIG]]。
- 請款期次「開了沒、收了沒」這兩種狀態怎麼走、拆期怎麼查 → [[分期請款狀態]]。
- 售後服務單本身的受理 / 處理 / 結案進度 → [[售後服務]]。
- 訂單取消後系統由上往下停掉工單 / 任務 / 生產任務時，各自停在哪個狀態的細節 → [[訂單狀態]] + 既有 [[wiki/erp/07-scenarios/README#情境 16：訂單取消與退款端到端（跨角色多模組連鎖）]]。
- 只改規格、金額沒變（不用開異動單）、或重新報價（那屬於需求單階段的議價）→ 不在這張卡的範圍。
