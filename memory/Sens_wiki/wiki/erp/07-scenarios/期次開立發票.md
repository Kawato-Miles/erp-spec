## 概述

[[業務]]（[[諮詢|諮詢人員]] 同權限）在客戶需要發票的時點，以一個請款期次為基礎一鍵開立發票，讓發票內容直接沿用期次既有的應收日、品項、買受人資訊，不必到開票表單從頭再貼一次。觸發是業務在某個期次按下「一鍵開立發票」這個動作，屬單一角色一次坐定完成的能力型過程。具現自 [[帳務流程]] 階段 2（期次開立發票）。

期次怎麼規劃、一個期次對應一張發票的對應規則、開票時機怎麼定，正本在 [[付款發票邏輯]]；發票品項五欄、作廢時限、字軌等外部硬性規定，正本在 [[發票法規硬約束-ezPay-MIG]]；發票狀態與作廢轉換正本在 [[發票狀態]]、期次的開票進度正本在 [[分期請款狀態]]。本卡只描述業務從一個期次把發票開出來的完成過程，不重述上述規則。

具體例子：訂單 #ORD-20260120-01（誠品年度活動小冊，應收 78,000）規劃成尾款一期 78,000。客戶要票時，業務在這期按一鍵開立發票，系統把訂單的開票公司、抬頭、統編與期次品項預填進發票表單，業務確認後送出，這期的開票進度轉為「已開立」，客戶拿著發票向其公司請款。

## 主成功過程

1. **業務**確認這張訂單的開票資訊（開票公司 / 抬頭 / 統編 / 地址）已維護齊全；尚缺或要改時，業務在訂單詳情的開票資訊區塊直接編輯後再開票（編輯能力見 [[US-ORD-005-訂單發票與配送資訊編輯]]）。判準：訂單上有一筆開票進度為「未開立」的請款期次（[[分期請款狀態]]），且開票公司與抬頭等欄位已填妥可供繼承。
2. **業務**在該期次按「一鍵開立發票」。判準：系統開出發票表單，並把來源期次的品項、應收日、業務備註與訂單的買受人資訊預填進去（依 [[付款發票邏輯#五E、請款與核銷四步流程|一鍵開票繼承規則]]）。
3. **業務**確認或微調發票內容（選有統編 B2B 或無統編 B2C、核對統編、調整品項），確保品項符合外部平台對品項欄位的硬性規定。判準：發票各品項通過品項欄位硬約束（[[發票法規硬約束-ezPay-MIG#二、發票品項五欄硬約束（核心，影響所有開票邏輯）|品項五欄硬約束]]），不合規時表單擋下不讓送出。
4. **業務**送出，系統經電子發票加值服務（藍新）完成開立。判準：發票進入「開立」狀態（[[發票狀態]]）、正式生效，並記下這張發票對應的是哪一個請款期次（一個期次只對應這一張發票）。

## 延伸岔路

- **2a 政府或學校預開發票、單張超過限額要拆**：客戶（多為學校或政府單位）走內部核銷流程須先預開發票，且單張發票金額不能超過平台限額時 → 業務在規劃階段先把這筆拆成多個期次（拆票即拆期、各期金額加總等於預收），再對每個期次各自走主成功過程開一張發票，每張單獨不超過限額。完整預開、拆票到後續實作金額調整退款的端到端走法見 [[US-ORD-030-F1預開發票拆票實作金額調整退款]]；退款段本身屬 [[訂單異動流程]]，不在本卡。
- **3a 已開的發票要改買受人（統編或抬頭打錯）**：發現原發票買受人資訊有誤 → 業務作廢原發票（填作廢原因，作廢屬 [[發票狀態]] 的轉換），再從同一期次重新開一張帶正確買受人的發票；作廢掉的發票字軌號碼不重用，新發票流水號往下遞增，是否能作廢的時限與字軌規則依 [[發票法規硬約束-ezPay-MIG#四、作廢 / 折讓 / 字軌規則|作廢與字軌規則]]。不另設「換開」捷徑。
- **3b 發票開出後要減額但不作廢（跨報稅期、客戶不需作廢）**：發票已生效、不走作廢只需沖掉部分金額時 → 業務在該發票上開折讓單，折讓金額不超過該發票剩餘可折讓額度，依 [[付款發票邏輯#五C、發票異動三情境（作廢、折讓、改買受人）|發票異動三情境規則]]（額度與適用時機正本在規則卡，本卡不抄公式）。折讓單自身的進度走 [[折讓單狀態]]。
- **4a 送出時平台回應作廢失敗（跨齊報稅期不可作廢）**：業務原想作廢重開、但發票已跨報稅期 → 業務不需自行判斷是否跨期，系統依第三方平台的失敗回應引導業務改開折讓單（規則正本見 [[發票法規硬約束-ezPay-MIG#四、作廢 / 折讓 / 字軌規則|作廢失敗引導折讓規則]]），改走 3b。

## 收斂狀態

發票開立成功後，這張發票處於「開立」狀態（[[發票狀態]]）、正式生效並計入三方對帳的發票淨額；對應的請款期次開票進度轉為「已開立」（[[分期請款狀態]]），系統不再對該期顯示一鍵開立入口。客戶取得發票後得以向其公司請款。期次的收款進度不受開票影響、維持原狀（兩條進度線各走各的）。

## 範圍外

- **期次怎麼規劃、一個期次對應一張發票的對應規則、開票時機怎麼定**：屬 [[付款發票邏輯]]（規則正本），實作時勿自行發明；本卡前提就是已有規劃好的期次可供開票。
- **發票狀態怎麼轉、作廢與折讓的轉換條件、字軌與配號機制**：屬 [[發票狀態]] 與 [[發票法規硬約束-ezPay-MIG]]，本卡只引用狀態名與規則，不重述轉換。
- **退款、補收等金額異動本身**（含預開發票後實作金額低於預估的退款）：屬 [[訂單異動流程]] 與 [[訂單異動規則]]，本卡只到把發票開出來為止，金額異動與後續折讓沖銷另案承接。
- **收款登錄與核銷分配**：開票不等於收到錢，收款怎麼登錄、怎麼勾選分攤到期次屬 [[付款發票邏輯]] § 五E 與收款核銷情境，存在但不在本卡。
- **多期合併開一張發票（合期）**：目前以一期一票為主軸，多期合併開一張票是否支援尚未拍板，見 [[BI-7-合期開一張發票是否支援]]；在拍板前不得自行設計合期開票路徑。
- **無對應期次的臨時或雜項收款發票**：本卡前提是以期次為基礎開票；業務是否需要不關聯任何期次的臨時發票、以及一鍵開票以外是否另開手動開票入口，尚未拍板，見 [[BI-18-手動開票是否一律需關聯期次]]。
- **電子發票加值服務的串接細節**（加密、上傳財政部時程、平台內部運作）：屬外部平台與實作規格，系統只呼叫其開立 / 作廢行為，規格見 [[發票法規硬約束-ezPay-MIG]]。
