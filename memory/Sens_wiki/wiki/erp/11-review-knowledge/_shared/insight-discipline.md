# Insight 規範（共用）

> 三視角審查 Agent 必讀。本卡定義「審查 Insight」的品質基準，避免空洞回應。

## 一、核心原則：Insight 不是讚美，是方向性的提醒

| 情況 | 該做 | 不該做 |
|------|------|--------|
| 設計方向正確 | 說明為什麼正確、還差什麼 | 「這個設計看起來合理」「需求看起來沒問題」 |
| 設計方向偏了 | 直接說偏在哪、為什麼偏 | 「建議再思考一下」「可以再優化」 |
| 業界有更好做法 | 附 URL 引用具體案例 | 「業界普遍做法」「最佳實踐」 |

**禁止輸出**以下類型句子：
- 「這個設計看起來合理」
- 「需求看起來沒問題」
- 「建議再思考一下」
- 「可以再優化」
- 「業界普遍做法」「最佳實踐」（沒有具體 URL）

## 二、業界參考必附 URL

提到「業界這樣做」「成熟做法」「最佳實踐」時 **MUST** 附 URL：

```
業界參考：
- [參考來源名稱](URL)：[一句話說明這個案例或做法的關鍵啟發]
```

若 WebSearch 後仍無找到具體案例，**MUST** 改用「依個人 N 年經驗判斷」並標明，**MUST NOT** 假裝有業界案例。

## 三、每問題必附解法

審查指出問題時 **MUST** 附解法或設計方向。不可只挑毛病不給方向：

| 情況 | 該做 |
|------|------|
| 有明確解法 | 「→ 解決方式：[具體建議怎麼調整]」 |
| 純商業決策需 Miles 拍板 | 「→ 需業務決策確認」並說明 A / B 方向的影響 |
| 需外部資訊才能定案 | 「→ 需確認 [具體外部資料] 後才能決定」 |

**MUST NOT** 空白帶過。輸出格式：

```
不合理之處 / 設計漏洞（必須指出，不能空白）：
1. [問題描述] → [具體影響 / 出問題情境] → [解決方式：...]
2. ...
```

## 四、所有意見基於已載入背景

審查意見 **MUST** 基於已載入的背景知識（Vault 卡、OpenSpec spec、Notion DB、WebSearch 結果），**MUST NOT** 憑空假設業務情境、系統行為、業界做法。

若需引用具體事實，**MUST** 標明來源：

```
依據：[Vault 卡名 / OpenSpec spec 路徑 / Notion 頁面 / Web URL]
```

## 五、決策確認後寫入 OQ

討論中確認設計決策時，**MUST** 確保對應 OQ 已填寫「決議與理由」：

```
決議：[採用方案 X]。理由：[判斷依據]。
```

BRD 中**只以行內括號引用 OQ**，**MUST NOT** 另立設計決策段落。
