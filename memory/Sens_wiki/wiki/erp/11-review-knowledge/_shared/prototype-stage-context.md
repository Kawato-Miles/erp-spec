# Prototype 階段背景（共用）

> 三視角審查 Agent（[senior-pm](../../../../../.claude/agents/senior-pm.md) / [ceo-reviewer](../../../../../.claude/agents/ceo-reviewer.md) / [erp-consultant](../../../../../.claude/agents/erp-consultant.md)）必讀。本卡取代三個 agent.md 內各自重述的階段背景段，集中維護以避免發散。

## 一、當前階段

本 ERP 系統處於 **prototype 探索階段**：

- 所有 spec / BRD / 流程設計都是工作版本，**尚未進入正式系統部署**
- Prototype 在 `sens-erp-prototype` repo 開發，用於驗證設計邏輯與業務情境
- spec 修訂、翻轉既有設計、修改剛歸檔的 change 都是**正常迭代過程**，不是 production 階段才需擔心的「政治成本」

## 二、不存在的事項（不應作為審查依據）

審查時 **MUST NOT** 以下列因素為由建議延後 / 不做 / 折衷設計：

- 「上線後政治成本」「翻轉既有 spec 會打臉前次決策」
- 「業務 / 印務 / 生管 / QC 等團隊習慣轉換成本」
- 「客戶教育負擔」「客戶習慣轉換」
- 「資料遷移成本」（前系統視為遷移前歷史基準，不納入新系統設計）
- 「上線時程壓力」（無上線時程）

## 三、Phase 在本專案的語意

**Phase 1 / Phase 2 / Phase 3** 在本專案語境是「**功能範圍切分**」（核心流程 / 優化 / 進階），**不是時間階段**。

- 反例（誤審）：「Phase 2 上線壓力大，這個功能建議延到 Phase 3」
- 正例：「這個功能屬於進階分析能力，建議排入 Phase 3 範疇切分」

## 四、應聚焦的審查角度

| 角度 | 說明 |
|------|------|
| 問題框架 | 真正要解決的業務痛點是什麼？是補功能還是解問題？|
| 設計邏輯正確性 | 流程能否走通？狀態機是否完整？資料是否一致？|
| 角色合理性 | 每個操作的執行者是對的人嗎？|
| 業務流程合理性 | 異常路徑（急單、改稿、退單、部分出貨）是否覆蓋？|
| 業界成熟做法對照 | 業界在這個問題上的解法是什麼？我們有沒有更根本的切入角度？|
| 系統一致性 | 與既有狀態機 / 商業流程 / 資料欄位是否矛盾？|
| KPI 對齊 | 設計是否真實貢獻 KPI？KPI 是否可量化？|

## 五、可指出但須加註的審查角度

「客戶感受 / 對帳邏輯 / 品牌觀感」這類正式運行才會浮現的問題**可以指出**，但 **MUST 標示為「上線前需驗證」**而非「立即修正」。

範例：「此設計可能造成客戶看不懂發票明細，建議標記為『上線前需驗證對外溝通方式』，不需在 prototype 階段強制修正。」
