# Sens 知識庫總目錄

Sens 多主題知識庫，依主題分流。本頁列**憲章、主題與入口**；各主題的詳細卡目錄由該主題自己維護，不在此重複列每張卡。

## Wiki 憲章（擴充任何主題、新增任何層級或範本前，先讀本節）

> 本節是全知識庫的內容原則正本——各單元範本直接引用本節，不重述。分工：本憲章管「BRD 是什麼、收什麼、怎麼寫」（why）；`.claude/rules/sens-wiki.md` 管操作與格式（frontmatter、連結、命名、log）；各主題 `00-meta/` 管主題專屬規則；各 `_template-*` 管該單元的執行標準。

### 本質與第一讀者

**Wiki 是公司的商業記憶——公司怎麼做生意的單一可信描述，不是任何系統的開發文件。** 系統（ERP、編輯器）只是它的消費者。**第一讀者是「不參與該系統開發的 PM」**：他要能只靠 wiki 讀懂公司怎麼運作。結構只服務於人；機器可解析的結構是下游實作規格的事，wiki 不為此妥協。

### 讀者情境（每段內容該為其中至少一個情境存在）

| 情境 | 使用者 | 需要什麼 |
|------|--------|---------|
| 商業自足理解（第一優先） | 不做該系統的 PM／新人 | 不依賴任何其他文件就能讀懂、純業務語言 |
| 規劃自查 | 規劃新功能前的 PM | 快查現況、知道**為什麼這樣設計**（防無意推翻有理由的設計） |
| AI 載入依據 | 撰寫下游規格／審查的 agent | 詞彙正本、商業條件、與他卡的邊界（防發明不存在的東西） |
| 訪談確認 | 與業務／管理層對流程 | 一張圖看懂、用公司的話講 |
| 自動稽核 | 健檢工具 | 結構一致、可勾稽 |

### 兩個判準（BRD 收什麼、怎麼寫）

| 判準 | 問法 | 管什麼 |
|------|------|--------|
| **紙本測試** | 「公司不用軟體、純手工作業時，這個知識還存在嗎？」存在 → 收；不存在（演算法、介面、技術欄位）→ 不收本體 | 內容收不收 |
| **商業語言** | 「不懂系統的業務主管讀得懂嗎？」 | 怎麼表達 |

依據：Business Rules 獨立原則——商業規則存在於軟體之外，是企業資產；下游系統需求從它派生。由此推導兩條鐵則：

- **BRD 自足**：卡的正文零實作文件引用（第一讀者讀不到那些文件）；往下游的導航由 frontmatter 中繼資料（`implemented-by`）承載，不進正文。
- **範圍外＝防護欄**：不過紙本測試的內容不寫本體，但**容易被下游實作者腦補的主題，必須在「範圍外」點名「存在但不在本層」**——BRD 沒劃清的邊界，實作時會被過度設計與捏造填滿。

### 單元定位模式（新增任何層級／範本時必做）

每個單元（狀態機、情境、使用者故事、驗收項目……）的範本，開頭 MUST 有「定位節」回答四題，其餘規則由定位推導：

1. 這個單元的卡**本質上是什麼**？（一句話，如「狀態機卡＝狀態詞彙的正本＋商業邏輯在單據處境維度的投影」）
2. 它是商業知識的**哪個視角投影**？與鄰層怎麼分工（投影不互相複寫，規則正本只有一處）？
3. 兩個判準在本單元的**具體應用與已知陷阱**？
4. 本單元伺候哪些讀者情境？

正面範例：`erp/06-state-machines/_template-state-machine.md` § 〇。新主題（如編輯器）開始維護 BRD 時，同樣從本憲章出發建立各單元範本，不重新調研。
