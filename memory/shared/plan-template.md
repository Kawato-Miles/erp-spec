## A. 業務段（PM 決策用）

### A1. As-is / To-be 對照

| 維度 | As-is（現況） | To-be（變更後） |
|------|---------------|----------------|
| <維度 1，例：觸發者> | <現況描述，純業務語言> | <變更後描述> |
| <維度 2，例：操作場景> | ... | ... |
| <維度 3，例：影響範圍> | ... | ... |

**禁用 anti-pattern**：
- 維度欄禁用欄位名 / interface 名 / 檔案路徑當主詞
- 內容禁中英夾雜：英文欄位名一律用介面中文（payment → 付款紀錄、printItem → 印件、orderAdjustment → 訂單異動、quoteRequest → 需求單、workOrder → 工單、productionTask → 生產任務、reviewRound → 審稿輪次、paymentPlan → 付款計畫、afterSalesTicket → 售後服務單 等）
- 至少 3 維度，少於則合併到一段純文字描述

### A2. PM 工作體驗的差異（如適用）

> 適用時機：本變更會改變 PM 平時工作流時填寫。
> 純內部規格變更可省略此段，但須在 A3 段明示「對 PM 工作流無影響」。

| 階段 | As-is（現況） | To-be（變更後） |
|------|---------------|----------------|
| <階段 1> | ... | ... |
| <階段 2> | ... | ... |

**禁用 anti-pattern**：
- 不要寫 Plan 撰寫流程本身（如「Claude 載入規範後寫 plan」），這是 meta 內容，PM 不關心
- 寫「PM 平時的工作」差異，不寫「系統內部處理」差異
- 「階段」欄禁用 Phase 1-5 之類的工程流程切分

### A3. 對 PM 平時工作流的影響

| PM 平時情境 | 變更後的影響 |
|------------|-------------|
| <情境 1，例：寫 OpenSpec change 探索時> | <影響說明> |
| <情境 2，例：改 Prototype 前> | ... |
| <情境 3，例：與 X 規範的相容性> | 不變 / 影響 |

**禁用 anti-pattern**：
- 情境欄禁用檔案路徑當主詞（反例：「改 spec 檔」→「修改規格時」）
- 影響欄禁用工程術語當主詞
- 須涵蓋「不影響」的既有規範 / 工作流，明示相容性（避免 PM 擔心衝突）

### A4. 風險與決策點（需 PM 拍板）

> 適用時機：本變更涉及取捨、有多個方案、或方向未定時填寫。
> 無決策點時可省略，但須在自審時聲明「無拍板議題」。

#### 議題 N：<議題標題（純業務問題）>

- **選項 A**：<簡述方案 A>
- **選項 B**：<簡述方案 B>
- **建議：<A 或 B>**。理由：<為何選這個，至少一句>

**禁用 anti-pattern**：
- 議題標題禁用技術名詞（反例：「改用 polymorphic 還是 inheritance」→「款項是否區分諮詢費與大貨費」）
- 選項禁只寫「方案 A / B」沒有具體描述
- 「建議」必填，禁丟給 PM 自選不給推薦
- 同一議題 ≤ 4 選項（超過拆成多議題）

### A5. 涉及的 OQ

- **既有 OQ**：<列出本變更觸及的 Vault `08-open-questions/` OQ ID，例：[[ORD-005]]、[[ORD-006]]，或聲明「無」>
- **新增 OQ**：<本次預計新開 N 個 OQ，例：「OQ AFT-9 最後活動時間 derived field 升級條件」，或聲明「無」>

**禁用 anti-pattern**：
- 不要寫「無 OQ」就跳過，須主動搜尋 Vault `08-open-questions/` 對應模組目錄確認後才聲明無
- 新增 OQ 須在 plan 拍板後立即觸發 oq-manage mode B 開檔，禁只 inline 標注

## B. 技術段（Agent 執行用，PM 可略）

### B1. 修改的關鍵檔案

| 路徑 | 動作 | 描述 |
|------|------|------|
| `<absolute path>` | Edit / Write / Delete | <一句話描述要做什麼> |
| `<absolute path>` | ... | ... |

**禁用 anti-pattern**：
- 路徑必須是絕對路徑（從 `/Users/...` 開始）
- 描述每個檔案動作不超過一行，細節留 design.md / spec
- 禁列「< 5 行修改」的瑣碎檔案（會稀釋 Agent 注意力）

### B2. 資料 / 結構變動（條件必填）

> 適用時機：涉及 schema 變動、API 介面變動、規範章節結構變動才填。

- <實體 / API / 章節> 變動：<新增 / 修改 / 移除什麼>

### B3. 可重用元件 / 既有函式（條件必填）

> 適用時機：用到既有 utility / 元件 / helper 才填，避免 Agent 重新造輪。

- `<absolute path>:<line>` — <符號名> — <為什麼可重用 / 用法簡述>

### B4. 驗證方式（Agent 自驗，每項含指令 + 成功條件）

| # | 驗證項 | 指令 | 成功條件 |
|---|--------|------|---------|
| 1 | <驗證目標 1> | `<具體指令>` | <可量化的成功條件> |
| 2 | ... | ... | ... |

**禁用 anti-pattern**（拍板前自審第 5 項）：
- 禁用「跑測試」「手動驗證」「確認檔案正確」等模糊描述
- 「指令」欄必須是可直接執行的 shell command 或具體操作（開啟某頁面 / 點某按鈕）
- 「成功條件」欄必須可量化（命令輸出 = N / 含字串 X / 畫面出現 Y / 測試 N 項通過）
- 觀察性驗證（如「下次使用時的行為」）可保留，但須說明何時觀察、看什麼判斷成功
- 至少 1 項可立即執行的指令型驗證（避免全部都是觀察性）
