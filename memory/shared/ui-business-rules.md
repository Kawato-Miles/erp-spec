# UI 業務規範（框架無關）

> **定位**：本檔記錄 ERP 介面的業務型行為規範，與 UI 框架（shadcn/ui / Ant Design / 其他）無關。
> **視覺規範正本**：`sens-erp-prototype/DESIGN.md`（tokens、元件、字型）。
> **適用範圍**：ERP 系統所有模組介面（Prototype / 實作）。
> **撰寫原則**：描述「該做什麼行為」而非「用哪個元件」。

---

## 1. 通用原則

### 1.1 禁用 Emoji

介面中任何位置均不得使用 Emoji，包含：
- 按鈕文字、說明文字、提示訊息（Banner / Alert / Toast）
- 側欄選單、表單 Label、表格內容
- 狀態標籤、空狀態說明

圖示一律使用圖示元件（如 `lucide-react` 或等價 SVG），不以 Emoji 替代。

### 1.2 側欄模組以實際規劃為準

側欄僅呈現已確認開發的模組。未定模組不得以佔位名稱、Emoji 或 placeholder 填充。

### 1.3 ERP 為管理階層中台

ERP 系統定位為 Golden Source 中台，使用者為管理層。其他角色（審稿人員、工廠師傅、客戶等）透過獨立入口（單獨 portal）登入，帶出各自所需作業，不直接進入 ERP 後台。此原則影響：

- RBAC 設計：ERP 不設 non-admin 角色
- 功能密度：可容忍高資訊密度（管理層為重度使用者）
- 移動版：不規劃行動裝置版（所有 ERP 模組適用）

## 2. 刪除操作流程

所有刪除操作必須有二次確認，禁止直接執行。

流程：
```
點「刪除」→ 彈出確認提示（「確定刪除？」+ 影響說明）
  ├─ 確認 → API 呼叫
  │         ├─ 成功 → 顯示 success 訊息 → 重新整理列表
  │         └─ 失敗 → 顯示 error 訊息，保留原資料
  └─ 取消 → 關閉確認，不執行
```

規範重點：
- 確認彈窗需說明「刪除影響」（例如「此操作無法復原，關聯的 N 筆工單將被解除」）
- 刪除成功後需明確告知（toast / message），不可靜默
- 危險操作（大量刪除、跨模組刪除）需強制輸入確認字串

## 3. Info Banner 使用規則

**用途**：頁面頂部的持續性說明（非一次性提示）。

**何時使用**：
- 頁面功能有前置條件（例如「需先完成 BOM 設定才能新增工單」）
- 頁面處於特殊模式（例如「此為草稿模式，資料尚未送出」）
- 需要引導使用者完成下一步（例如「請填寫必填欄位後送出」）

**何時不使用**：
- 操作成功/失敗回饋 → 用 Toast
- 欄位驗證錯誤 → 用欄位內錯誤訊息
- 需要使用者確認的詢問 → 用 Modal / Popover

**內容要求**：
- 文字需具體可行（說明使用者該做什麼），不可僅描述狀態
- 含圖示標示類型（info / warning / error）
- 可關閉（除非是永久警示）

---

## 4. 全域 Message / Toast 分類

| 類型 | 使用場景 | 持續時間 |
|------|---------|---------|
| success | 新增 / 編輯 / 刪除成功、儲存完成 | 2 秒 |
| error | API 失敗、系統錯誤、網路中斷 | 3 秒或需手動關閉 |
| warning | 非錯誤但需注意（例如「此操作將影響關聯資料」） | 2.5 秒 |
| loading | 提交中、長時間載入（搭配 async 操作） | 持續到操作結束 |
| info | 中性通知（例如「資料已自動儲存」） | 2 秒 |

規範重點：
- 重要 error（如資料遺失風險）不自動消失，需使用者關閉
- 連續操作成功可合併顯示（避免刷屏）
- 不用 Toast 做關鍵決策（例如「確認刪除？」→ 應用 Modal）

---

## 5. Loading 狀態規範

| 場景 | 處理方式 |
|------|---------|
| 表格初次載入 | 表格骨架（skeleton）或 loading spinner |
| 表格重新整理 | 表格上方或覆蓋層 loading indicator |
| 按鈕提交中 | 按鈕內 loading + disabled（禁止重複點擊） |
| 單欄位非同步驗證 | 欄位右側小 loading icon |
| 全頁載入（首次進入路由） | 中央 loading spinner + 「載入中...」文字 |
| 檔案上傳 | Progress bar + 可取消 |

規範重點：
- 任何 > 200ms 的操作必須有 loading 反饋
- loading 狀態下禁止觸發同一操作
- 失敗後 loading 消失，錯誤訊息取代

## 6. 表單提交流程

```
填寫表單 → 點「確認/儲存」→ 前端驗證
  ├─ 驗證失敗 → 欄位下方顯示紅色錯誤訊息，不送出
  │              焦點跳至第一個錯誤欄位
  └─ 驗證通過 → API 呼叫（按鈕進入 loading）
       ├─ 成功 → 關閉 Modal/Drawer → 重新整理資料 → 顯示 success Toast
       └─ 失敗 → 保持 Modal/Drawer 開啟
                 ├─ 欄位層錯誤（例如「名稱重複」） → 對應欄位下方顯示
                 └─ 系統層錯誤（例如「網路中斷」） → Modal 頂部 error banner
```

規範重點：
- 送出中禁止關閉 Modal / Drawer（除非提供明確取消按鈕）
- API 失敗後表單資料必須保留，不可清空
- 必填欄位以紅色 `*` 標示在 Label 前方

## 7. 頁面模板抽象結構

### 7.1 標準列表頁

適用：所有管理列表（訂單、工單、客戶、材料、庫存等）。

結構：
```
Page Header
  ├─ Breadcrumb
  └─ Title + (可選) 操作按鈕

[Info Banner]（選填，前置條件/特殊模式時）

Toolbar
  ├─ 左側：篩選器（下拉 / 搜尋 / 日期範圍）
  └─ 右側：刷新 + 新增 / 匯出

Main Content
  ├─ Table（含排序、分頁、欄位設定）
  │   └─ 最右欄：行內操作（編輯、刪除、檢視）
  └─ Footer：Pagination + 總筆數
```

### 7.2 單筆編輯頁面（單一頁面）

適用：需求單、訂單、工單的單筆深度編輯。

結構：
```
Page Header
  ├─ Breadcrumb
  └─ Title（含單號）+ Status Badge + Primary Action（儲存/送出）

[Info Banner]（選填，狀態說明或前置條件）

Form Body（單欄或分段容器）
  ├─ 基本資訊區
  ├─ 關聯資料區（例如印件明細表）
  └─ 附件 / 備註區

Sticky Footer（若 body 過長）
  └─ 取消 + 儲存 / 送出
```

### 7.3 表單彈窗（Modal / Drawer）

適用：新增/編輯資料、快速操作。

選擇原則：
- 欄位 ≤ 8 個、操作簡單 → 居中 Modal
- 欄位 > 8 個、需展示詳情 → 右側 Drawer

結構：
```
Header：標題 + 關閉 (×)
Body：Form（vertical layout，label 在上）
Footer：取消 + 確認（右對齊）
```

### 7.4 詳情 Drawer

適用：從列表點擊某筆查看完整資訊（不進入編輯頁）。

結構：
```
Header：標題（單號）+ 關閉 (×) + 編輯按鈕
Body
  ├─ Info Table（key-value 描述列表）
  ├─ 子資料 Table（若有關聯資料）
  └─ 操作紀錄 / 備註
Footer（可選）：關閉 + 編輯
```

## 8. 操作按鈕優先級

| 類型 | 使用場景 | 位置 |
|------|---------|------|
| Primary | 主要 CTA（送出、儲存、確認、新增） | 頁面右上、Modal Footer 右側 |
| Secondary / Default | 次要操作（取消、刷新、匯出、搜尋） | 緊鄰 Primary 左側 |
| Destructive | 刪除、取消訂單等破壞性操作 | 行內操作欄、Modal Footer 左側（遠離主 CTA） |
| Text / Link | 低優先操作（檢視詳情、前往頁面） | 行內、輔助位置 |

規範重點：
- 單一頁面同層級 Primary 按鈕只能有一個
- Destructive 與 Primary 不可緊鄰（防誤點）
- 按鈕順序（左→右）：Destructive → Secondary → Primary

## 9. 狀態標籤（Badge）呈現

所有業務狀態（需求單、訂單、工單、生產任務、QC、出貨等）必須用 Badge 呈現，不可用純文字。

規範重點：
- 狀態顏色對應使用專案 token（詳見 `sens-erp-prototype/src/index.css` 的 `--status-*`）
- 狀態文字需與狀態機定義一致（詳見 `openspec/specs/state-machines/spec.md`）
- 同一狀態在所有頁面的顏色必須一致
- 停用/作廢狀態使用 muted / neutral 色

---

## 10. 禁止事項總表

1. 不得使用 Emoji
2. 不得靜默刪除（必須二次確認）
3. 不得靜默操作成功（必須 Toast 回饋）
4. 不得阻塞 UI 但無 loading 反饋
5. 不得在同一頁面並列多個 Primary CTA
6. 不得用 Toast 承載需使用者決策的訊息
7. 不得在送出過程中允許關閉 Modal（除非提供明確取消）
8. 不得用 placeholder 填充未規劃模組
