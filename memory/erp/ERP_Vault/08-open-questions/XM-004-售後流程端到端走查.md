# XM-004：售後服務單收尾流程是否需要做一輪「使用者端到端走查 + 流程整體性審視」

## 背景

senior-pm 前期介入時識別出 **insight 2**：

> 從 v0.1 → v0.2 → v0.3 → 本 change（refine-after-sales-refund-and-add-supplementary-print），**連續四個 change 都在改 ticket 模組，且每次都暴露新的 gap**，這暗示規劃時可能少了「ticket 內部端對端流程走查」這個步驟。

連續四次都做 reactive 補丁，效率不高，且每次補丁都可能暴露新的 gap。

## 問題

是否要在 2026 Q3（v0.4 上線後 UAT 完成）做一輪「售後服務單收尾流程整體重構」的端到端走查 + 流程整體性審視？

候選做法：

1. **不做整體走查**：繼續 reactive 補丁，下次 Miles 發現新 gap 再開 change
2. **做一輪 PM-led 走查**：PM 與業務 / 諮詢 / 印務 / 會計各角色實際走過 ticket 完整流程（建立 → 處理 → 結案），識別所有 gap，列入下一輪 roadmap
3. **做完整重構 change**：開大 change 重構整個 ticket 模組（含折讓單 / 補費 OA 自動建議 / SLA / 業務離職轉派等），一次到位

## 影響範圍

- 不影響本 change（本 change 為 reactive 補丁）
- 影響 2026 Q3 roadmap（是否規劃 ticket 整體重構）
- 影響 ticket 模組相關 OQ（AFT-1 / AFT-2 / AFT-3 / AFT-4 / AFT-5 + XM-001 等）的處理時機

## 前置依賴

- [[XM-001-款項管理頁面業務最重要決策]]：款項管理頁面（含 ticket 動作）的訪談 + 設計尚未完成，是端到端走查的關鍵前置
- [[after-sales-ticket-AFT-1-業務離職轉派]] / [[after-sales-ticket-AFT-2-逾期分級]] / [[AFT-3-OA已核可改金額是否通知主管]] / [[AFT-4-補印優先度規則]] / [[AFT-5-補費OA由誰建立]]：所有未解 ticket 相關 OQ 都應在整體走查時納入考量

## 待釐清

- 端到端走查的範圍（單純 ticket vs ticket + 訂單 + 發票 + 退款全流程）
- 走查方式（PM 自走 vs 與業務 / 諮詢 / 印務 / 會計實地觀察）
- 走查時程（建議 2026 Q3，v0.4 UAT 完成後）
- 整體重構 vs reactive 補丁的 trade-off
