# PI-001：難易度分數 1-10 業務含義

## 問題描述

`PrintItem.difficultyLevel` 欄位為 1-10 整數，業務於需求單階段必填，用於審稿人員自動分配（依 `maxDifficultyLevel` 匹配）。

但 Notion / Spec 中**目前未明確定義「1-10 分各代表什麼產品類型 / 規格複雜度」**。目前由業務「經驗判斷」，**無共識量表**。

## 涉及範圍

- 模組：prepress-review、quote-request
- 相關卡：[[難易度機制]]、[[審稿分配規則]]、[[免審決策樹]]、[[需求單]]、[[印件]]
- 影響範圍：
  - 業務新人訓練（無量表可參考）
  - 自動分配命中率（KPI）— 業務評分不準 → 命中率低
  - 跨業務的評分一致性

## 討論記錄

Phase B 撰寫 04-business-logic/難易度機制.md 時識別出此議題。建議釐清以下情境：

- 1-3：簡單（如名片、單面 DM）？
- 4-6：中等（如型錄、多頁書冊）？
- 7-10：複雜（如特殊加工、燙金 / 模切組合）？

## 待解答

- [ ] 1-10 分各對應哪些產品類型 / 加工複雜度
- [ ] 是否建立量表 / 對照表
- [ ] 是否需要「自動建議分數」功能（依產品類型自動算）

## 候選方案

### 方案 A：建立明確量表（1-10 分對應產品類型）

- 行動：與業務 / 審稿主管討論共識量表，寫入 [[難易度機制]] 與 Notion 業務情境 DB
- 優點：新人友善、評分一致
- 缺點：產品多樣，量表覆蓋有限

### 方案 B：依現有訂單訓練「建議分數」

- 行動：用 mockOrders.ts 或實際歷史訂單，分析難易度分佈，訓練自動建議
- 優點：基於資料，精準
- 缺點：需有足夠資料

### 方案 C：維持現狀（業務經驗判斷）

- 不做量表，依賴業務經驗
- 優點：彈性最高
- 缺點：跨業務一致性低、新人難上手
