# XM-001：款項管理頁面業務日常的最重要決策

## 問題描述

業務人員每天打開款項管理三個頁面（Receivables 應收款項 / PendingInvoices 待開發票 / BillingAnomalies 帳務異常）時，**要做的最重要決策是什麼**？

當前 prototype UI 對齊（change `align-payment-list-pages-to-quote-list-pattern`）建立在「KPI 追蹤頁」假設上，提供：
- 上方 4 個 StatusCard（待收訂單 / 待收金額 / 逾期未收 / 逾 30 天等量化指標）
- 中段搜尋框 + 狀態 select 篩選器
- 下方訂單列表（按狀態 / 逾期天數排序）

但業界 AR 平台已轉向「actionable 追款列表」典範：
- aging bucket + 客戶歷史付款行為 + 爭議紀錄交叉排序
- 直接告訴使用者「今天該追誰」「下一步做什麼」
- 把 KPI dashboard 與 actionable list 切成兩種頁型（dashboard 給管理層，list 給操作人員）

→ 若業務真正要的是「今天該追款 / 該開發票 / 該處理異常的下一步」，目前的「靜態 KPI dashboard + 全列表」可能根本錯方向，需要重新設計為動態 actionable list。

## 涉及範圍

- **模組**：款項管理三模組（Receivables / PendingInvoices / BillingAnomalies）
- **跨模組**：影響 prototype UI 範式選擇（dashboard vs actionable list）
- **影響範圍**：
  - 若結論是「KPI dashboard 足夠」→ 目前 UI 對齊 change 已滿足，無需重做
  - 若結論是「需要 actionable list」→ 三個頁面的資訊架構需重新設計（含排序策略、客戶歷史付款資料、爭議紀錄整合）
  - 連帶影響：訂單 / 發票 / 收款 / 異動 / 退款等實體可能需擴充欄位（客戶評等、歷史付款行為、爭議標記等）

## 討論記錄

### CEO 三視角審查（2026-05-19）

於 change `align-payment-list-pages-to-quote-list-pattern` 規格期審查中，CEO 視角 reviewer 提出此問題：

> 「款項管理頁面到底要幫業務做什麼決策？這個沒先想清楚，現在花力氣統一 UI、補規範、做 KPI 動態化，未來真的要做 actionable 追款功能（『今天該打給哪 5 個客戶』）時，這套剛統一好的 UI 範式很可能又要翻一次。」

引用業界參考：
- [Versapay - 11 Accounts Receivable KPIs](https://www.versapay.com/resources/11-accounts-receivable-kpis-measure-ar-performance)：modern AR 工具把 KPI 與 actionable 收款列表分開，前者給 CFO 看，後者給收款人員用
- [ARDEM - Accounts Receivable KPIs 2026](https://ardem.com/bpo/accounts-receivable-kpis/)：明確指出「aging bucket 不應該是唯一指標，要結合付款歷史與爭議資料排序」

### 暫定決議（不阻斷本次 change）

Miles 確認本次 UI 對齊 change 繼續推進，理由：
- 本次定位為 `technical-debt-cleanup`（UI 規範一致性修補），不假設業務最終 UI 方向
- 本次成果（單一搜尋 Card / select 篩選 / 動態 KPI / 篩選中標記）作為 UI 系統一致性的階段性成果
- 若 OQ-0 後續結論是「需轉向 actionable list」，可在下個獨立 change 重新設計三個頁面，本次規範補強仍對其他列表頁有價值

## 候選方案

### 方案 A：維持當前 KPI dashboard 範式

- **適用前提**：業務在這三頁的決策主要是「看整體狀況 + 偶爾搜尋特定客戶」，不需要排序型 actionable 資訊
- **優點**：當前 UI 對齊 change 即足夠，無需額外資料模型擴充
- **缺點**：若實際需求是 actionable，UI 範式需要翻新

### 方案 B：轉向 actionable list 範式（業界主流）

- **適用前提**：業務每天的核心動作是「處理 list 中的下一個項目」（追款 / 開發票 / 處理異常）
- **優點**：對齊業界 AR 平台做法、直接驅動業務動作、降低決策成本
- **缺點**：
  - 需要額外資料（客戶評等、付款歷史、爭議標記）— 需擴充訂單 / 客戶實體
  - 需要排序演算法（用什麼分數排「今天該追誰」）
  - 三個頁面的 UI 範式需重新設計
  - 本次 change 完成的 UI 對齊在新範式下可能要再翻

### 方案 C：兩者並存（dashboard + actionable list 切換）

- **適用前提**：管理層需要 dashboard、操作人員需要 actionable list，兩種使用者並存
- **優點**：兼顧兩種 use case
- **缺點**：頁面複雜度倍增、需 role-based view 切換、實作成本最高

## 建議的下一步行動

1. **使用者訪談（必須先做）**：
   - 訪談 1-2 位業務、1 位會計
   - 開放式問題：「打開這個頁面，你做的下一個動作是什麼？」（觀察動線）
   - 記錄：實際停留時間、實際使用的功能、實際做的決策
2. **對照業界做法**：閱讀 Versapay / ARDEM / Bill.com 等 AR 平台的實際 UI，評估其 actionable list 設計是否適用
3. **產出決議**：依訪談結果與業界對照，選定方案 A / B / C
4. **若選方案 B 或 C**：另開 change（如 `redesign-payment-pages-as-actionable-list`），不重做本次 change 已完成的 UI 對齊
