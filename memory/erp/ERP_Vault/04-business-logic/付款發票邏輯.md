# 付款發票邏輯（總覽）

> 印刷業 ERP 訂單款項與發票的 **13 個關鍵情境**。詳細業務情境、實作解決方式、驗證重點與 Prototype 對應訂單見 [memory/erp/payment-invoice-scenarios.md](../../payment-invoice-scenarios.md)（475 行）。
>
> 本卡為**索引型 Zettel**，提供分類概覽與跨情境共用規則。

## 一、三大分類

| 分類 | 數量 | 範圍 |
|------|------|------|
| **特殊發票開立方式** | 8 個（F1-F8）| 預開、拆多筆、有 / 無統編、指定品名、指定日期、跨月、作廢重開、折讓 |
| **特殊收款方式** | 4 個（C1-C4）| 頭尾款、申請後付款、頭尾款指定日期、線下單補收（含異動補收）|
| **訂單異動情境** | 3 個（A1-A3）| 異動不收費（公司吸收）、異動補收、異動退款 |
| **不在範疇** | 2 個 | 跨訂單混合匯款（人工）、重新報價（屬需求單議價）|

## 二、特殊發票開立方式（F1-F8）

| 編號 | 情境 | 關鍵設計 |
|------|------|---------|
| F1 | 預開發票 + 拆 2 筆 + 實際金額調整 | Payment 可無 PaymentPlan 關聯；售後 OA + SalesAllowance + refundPaymentId |
| F2 | 拆多筆指定金額發票 | PaymentInvoice junction N:M |
| F3 | 拆多筆有統編 / 無統編 | Invoice.category 分 B2B / B2C |
| F4 | 指定品名 / 作廢重開 | 作廢需填原因（中文 6 字 / 英文 20 字內，藍新限制）|
| F5 | 指定日期開立 | PaymentPlan.expected_invoice_date 排程提醒；手動開立 |
| F6 | 分 2 筆不同月份 | PaymentPlan 拆 2 期、各填 expected_invoice_date |
| F7 | 作廢重開（多種子情況）| 統編 / 金額錯誤 / 已申報等 |
| F8 | 折讓單情況 | 過期發票不可作廢；SalesAllowance + refundPaymentId |

## 三、特殊收款方式（C1-C4）

| 編號 | 情境 | 關鍵設計 |
|------|------|---------|
| C1 | 頭尾款 50% 指定日期 | PaymentPlan 2 期、各 scheduled_date |
| C2 | 申請後付款（出貨後 N 天）| 先開後付，Invoice 可無 Payment 對應 |
| C3 | 頭尾款 + 指定開票日期 + 付款時間 | 兩 PaymentPlan 各自 scheduled_date + expected_invoice_date |
| C4 | 線下單補收（含異動）| OrderAdjustment（phase=during_order）+ 業務手動加 PrintItem |

## 四、訂單異動情境（A1-A3）

> **重要**：訂單**期間** vs **已完成後** 路徑不同。完成後走 AfterSalesTicket（add-after-sales-ticket change）。

| 編號 | 情境 | 訂單期間 | 訂單已完成後 |
|------|------|---------|-------------|
| A1 | 異動不收費（公司吸收）| 工單異動，不建 OrderAdjustment | AfterSalesTicket（responsibility=公司認賠）+ 補印免費 |
| A2 | 異動補收 | OrderAdjustment（補收）+ PaymentPlan + 補收發票 | AfterSalesTicket + 關聯 OA（補退）+ 補印 PrintItem |
| A3 | 異動退款 | OrderAdjustment（退款）+ 折讓 / 作廢重開 | AfterSalesTicket + 關聯 OA（退印）+ 折讓 / 作廢 + 退款 Payment |

## 五、跨情境共用實體

| 實體 | 用途 | 對應 [[05-entities]] 卡（Phase B 撰寫）|
|------|------|---------|
| PaymentPlan | 收款計畫，可分多期 | 訂單 § 付款 |
| Payment | 實際收款 / 退款紀錄（可負數） | 訂單 § 付款 |
| Invoice | 發票，可 B2B / B2C | 訂單 § 發票 |
| PaymentInvoice junction | N:M 關聯 Payment ↔ Invoice，可自由分攤 | 訂單 § 發票 |
| SalesAllowance | 折讓單，關聯 Invoice + refundPaymentId | 訂單 § 售後 |
| OrderAdjustment | 訂單異動單，phase=during_order / after_completion | 訂單 § 異動 |
| AfterSalesTicket | 售後容器（add-after-sales-ticket change）| 售後服務 |

## 六、三方對帳

每個情境的最終驗收都要對齊：

```
應收（訂單應收總額）= 發票淨額（已開發票合計 - 已確認折讓）= 收款淨額（Payment 加總，含負數退款）
```

差額為零代表「三方對齊」。差額不為零時觸發**對帳警示 banner**。

## 七、相關角色

- [[業務]] — 主要操作者，建 PaymentPlan / Payment / Invoice / SalesAllowance
- [[諮詢]] — 等同業務權限
- [[會計]] — 對帳查詢 + CSV 匯出，無發票操作權
- 業務主管（暫無 Vault 角色卡）— OrderAdjustment 審核（金額把關）

## 八、不做系統化的範疇外

| 情境 | 原因 |
|------|------|
| 跨訂單金額混合匯款對帳 | 走人工處理（業務 / 會計拆分銀行入帳）|
| 同訂單開指定數量發票 + 混單出貨 | 不做系統化 |
| 重新報價 / 重開報價單 / 重打樣 | 屬需求單議價階段，不在訂單模組範疇 |
