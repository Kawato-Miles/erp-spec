# 付款發票邏輯（總覽）

> 印刷業 ERP 訂單款項與發票的 **13 個關鍵情境**。詳細業務情境、實作解決方式、驗證重點與 Prototype 對應訂單見 [[payment-invoice-scenarios]]（同目錄，475 行）。
>
> 本卡為**索引型 Zettel**，提供分類概覽與跨情境共用規則。

## 一、三大分類

| 分類 | 數量 | 範圍 |
|------|------|------|
| **特殊發票開立方式** | 8 個（F1-F8）| 預開、拆多筆、有 / 無統編、指定品名、指定日期、跨月、作廢重開、折讓 |
| **特殊收款方式** | 4 個（C1-C4）| 頭尾款、申請後付款、頭尾款指定日期、線下單補收（含異動補收）|
| **訂單異動情境** | 3 個（A1-A3）| 異動不收費（公司吸收）、異動補收、異動退款 |
| **不在範疇** | 2 個 | 跨訂單混合匯款（人工）、重新報價（屬需求單議價）|

## 二、特殊發票開立方式（F1-F8）

| 編號 | 情境 | 關鍵設計 |
|------|------|---------|
| F1 | 預開發票 + 拆 2 筆 + 實際金額調整 | Payment 可無 PaymentPlan 關聯；售後 OA + SalesAllowance + refundPaymentId |
| F2 | 拆多筆指定金額發票 | PaymentInvoice junction N:M |
| F3 | 拆多筆有統編 / 無統編 | Invoice.category 分 B2B / B2C |
| F4 | 指定品名 / 作廢重開 | 作廢需填原因（中文 6 字 / 英文 20 字內，藍新限制）|
| F5 | 指定日期開立 | PaymentPlan.expected_invoice_date 排程提醒；手動開立 |
| F6 | 分 2 筆不同月份 | PaymentPlan 拆 2 期、各填 expected_invoice_date |
| F7 | 作廢重開（多種子情況）| 統編 / 金額錯誤 / 已申報等 |
| F8 | 折讓單情況 | 過期發票不可作廢；SalesAllowance + refundPaymentId |

## 三、特殊收款方式（C1-C4）

| 編號 | 情境 | 關鍵設計 |
|------|------|---------|
| C1 | 頭尾款 50% 指定日期 | PaymentPlan 2 期、各 scheduled_date |
| C2 | 申請後付款（出貨後 N 天）| 先開後付，Invoice 可無 Payment 對應 |
| C3 | 頭尾款 + 指定開票日期 + 付款時間 | 兩 PaymentPlan 各自 scheduled_date + expected_invoice_date |
| C4 | 線下單補收（含異動）| OrderAdjustment（phase=during_order）+ 業務手動加 PrintItem |

## 四、訂單異動情境（A1-A3）

> **重要**：訂單**期間** vs **已完成後** 路徑不同。完成後走 AfterSalesTicket（add-after-sales-ticket change）。

| 編號 | 情境 | 訂單期間 | 訂單已完成後 |
|------|------|---------|-------------|
| A1 | 異動不收費（公司吸收）| 工單異動，不建 OrderAdjustment | AfterSalesTicket（responsibility=公司認賠）+ 補印免費 |
| A2 | 異動補收 | OrderAdjustment（補收）+ PaymentPlan + 補收發票 | AfterSalesTicket + 關聯 OA（補退）+ 補印 PrintItem |
| A3 | 異動退款 | OrderAdjustment（退款）+ 折讓 / 作廢重開 | AfterSalesTicket + 關聯 OA（退印）+ 折讓 / 作廢 + 退款 Payment |

## 五、跨情境共用實體

| 實體 | 用途 | 對應 [[05-entities]] 卡（Phase B 撰寫）|
|------|------|---------|
| PaymentPlan | 收款計畫，可分多期 | 訂單 § 付款 |
| Payment | 實際收款 / 退款紀錄（可負數） | 訂單 § 付款 |
| Invoice | 發票，可 B2B / B2C | 訂單 § 發票 |
| PaymentInvoice junction | N:M 關聯 Payment ↔ Invoice，可自由分攤 | 訂單 § 發票 |
| SalesAllowance | 折讓單，關聯 Invoice + refundPaymentId | 訂單 § 售後 |
| OrderAdjustment | 訂單異動單，phase=during_order / after_completion | 訂單 § 異動 |
| AfterSalesTicket | 售後容器（add-after-sales-ticket change）| 售後服務 |

## 五B、七實體連帶矩陣（2026-05-28 新增）

> 任何「款項 / 發票 / 退款 / 補收 / OA」異動 MUST 連帶檢查下列影響範圍。
> 受 Miles 第六輪反饋拍板：「改退款的相關功能，就要去連帶檢查是否有影響收帳款、發票等規劃」。

### 5B.1 主動實體 → 連帶影響表

| 主動實體 | 連帶實體 | 影響規則 |
|---------|---------|---------|
| **Invoice 開立** | InvoiceItem（五欄硬約束）/ PaymentInvoice junction / PlannedInvoice（鏈式預填來源） | 五欄硬約束（ItemCount × ItemPrice = ItemAmt）+ B2B 未稅 / B2C 含稅切換 |
| **Invoice 作廢** | SalesAllowance（超過 14 天時限改折讓）/ Payment（退款 Payment 反向）| 14 天作廢期 + 超期 MUST 走折讓單 |
| **SalesAllowance 確認** | Invoice（折讓金額對應）/ Payment.refundPaymentId（退款 Payment 反向關聯）| 跨 Invoice / Payment 紀錄 + 隔日上傳財政部 |
| **Payment 退款（負值）** | OA.adjustment_type=退款（執行觸發）/ SalesAllowance（金額對齊）/ Invoice（作廢或折讓）| OA 推進條件 + SalesAllowance 反向 |
| **OrderAdjustment 已執行** | Payment（綁退款 Payment 建立事件）/ Invoice（作廢或折讓）/ SalesAllowance（建立）| Payment 累計 ≥ OA.amount 觸發推進 |
| **PlannedInvoice 自動建** | Order（諮詢訂單收尾**不做大貨 / 需求單流失**兩情境 / 補收 OA）/ Invoice（一鍵開立沿用 items[]）| 鏈式預填 + 不自動開立發票；**諮詢取消情境 MUST NOT 自動建待開發票**（2026-05-30 converge-consultation-cancel 收斂，留存 1000 收入由業務手動開票、未開票由對帳差額警示兜底）|
| **AfterSalesTicket 退款** | OA（responsibility=公司認賠 / 補退）/ Payment / SalesAllowance | 跨售後 ticket 容器 |

### 5B.2 七實體連帶網絡圖

```
                Payment ────────┐
              ↑      ↑          │
              │      │          ↓
              │   OrderAdjustment ────────┐
              │      ↑                    │
PaymentPlan ──┘      │                    │
                  AfterSalesTicket        │
              ┌──────┴──────┐             │
              ↓             ↓             ↓
            Invoice ←─ SalesAllowance ←──┤
              ↑                          │
              │                          │
       PaymentInvoice junction ─────────┘
              ↑
       PlannedInvoice
```

### 5B.3 連帶檢查紀律（規劃時必做）

任何 ERP 規劃 / 設計涉及下列議題 MUST 跑連帶檢查：

| 議題類型 | 必檢實體 |
|---------|---------|
| 改退款邏輯 | Payment（退款）+ OA（執行條件）+ SalesAllowance（金額對齊）+ Invoice（作廢或折讓）+ AfterSalesTicket（若已完成）|
| 改發票開立 | Invoice（五欄硬約束）+ InvoiceItem + PaymentInvoice junction + PlannedInvoice（鏈式預填）+ ezPay 法規 |
| 改 OA | Payment（累計推進）+ Invoice（補收 / 折讓）+ SalesAllowance + AfterSalesTicket（若已完成）|
| 改收款 | Payment（核銷分配）+ PaymentPlan（推進）+ Invoice（對應）+ OA（推進）|
| 改補印（after-sales）| AfterSalesTicket + OA（補退）+ Payment + SalesAllowance + 補印 PrintItem |
| 改 PlannedInvoice | Order（諮詢收尾不做大貨 / 需求單流失觸發、**諮詢取消不自動建**見 5B.1 / 補收觸發）+ Invoice（一鍵開立沿用）+ InvoiceItem（五欄） |

### 5B.4 外部硬約束連帶

任何 Invoice / InvoiceItem 異動 MUST 對齊 [[發票法規硬約束-ezPay-MIG]] 規範：
- 五欄硬約束（ItemName / ItemCount / ItemUnit / ItemPrice / ItemAmt）
- 作廢期限 14 天（超期走折讓）
- B2B 未稅 vs B2C 含稅切換
- 字軌 2 月為 1 期
- 上傳財政部時程（每日 01:00 上傳前日資料）

## 六、三方對帳

每個情境的最終驗收都要對齊：

```
應收（訂單應收總額）= 發票淨額（已開發票合計 - 已確認折讓）= 收款淨額（Payment 加總，含負數退款）
```

差額為零代表「三方對齊」。差額不為零時觸發**對帳警示 banner**。

**收款淨額限縮**（add-payment-status-and-decouple-oa-execution change 2026-05-21）：

`收款淨額 = ∑ Payment.amount WHERE paymentStatus = '已完成'`（含正值一般收款與負值已完成退款）。處理中 Payment 不計入收款淨額（業務先填一半時對帳不受影響）、不影響 PaymentPlan.status、不推進關聯 OA。對帳面板新增「處理中（合計，不計入）」資訊軸顯示，差額 hint 加註「另含處理中款項 X 元，齊備後將計入」。

詳見 [order-management spec § Requirement: 三方對帳檢視面板](../../../../openspec/specs/order-management/spec.md)。

## 七、相關角色

- [[業務]] — 主要操作者，建 PaymentPlan / Payment / Invoice / SalesAllowance
- [[諮詢]] — 等同業務權限
- [[會計]] — 對帳查詢 + CSV 匯出，無發票操作權

## 八、不做系統化的範疇外

| 情境 | 原因 |
|------|------|
| 跨訂單金額混合匯款對帳 | 走人工處理（業務 / 會計拆分銀行入帳）|
| 同訂單開指定數量發票 + 混單出貨 | 不做系統化 |
| 重新報價 / 重開報價單 / 重打樣 | 屬需求單議價階段，不在訂單模組範疇 |
