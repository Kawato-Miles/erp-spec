# 訂單異動規則（OrderAdjustment 規則正本）

> [[訂單]] 成立後遇到金額增減（補收／退款）時，「什麼情況套什麼規則」的決策正本。本卡承載補收與退款不對稱分權、已執行的認列與回退、應收重算公式、一致性條件（invariant）的完整定義；[[訂單異動狀態]]、[[售後服務]]、[[訂單]] 等卡只引用本卡，不重述規則。屬款項與現金（billing-cash）領域。

## 營運背景

印刷廠接單後金額常臨時變動：客戶在製作中追加印量、改急件、補運費，公司多收錢（稱補收）；或因品質、規格不符要退費，公司退錢出去（稱退款）。這兩個方向的營運風險不對稱——錢退出去一旦出錯難追回，客戶加單則是送上門的生意。所以公司的分權原則是：[[業務主管]] 把關「退款」這個現金流出方向，「補收」這個客戶加單方向由 [[業務]] 自己拍板、不必等主管，避免客戶等審核而流失加單機會。

另有一種特例：諮詢取消要退客戶一半費用（退 1000 元），這是公司既定規則、系統自動生成這筆退款異動單，不需主管再判斷該不該退，但實際退錢動作仍要有人真的把款項退出去才算數。

主管讀本卡要能驗證的是：這套「補收免審、退款送審」的分權，以及「錢真的動了才認列」的認列規則，是否符合公司實際的現金管控需求。

## 規則與營運動機

> 十條規則，依「分權（R1-R3）→ 認列與回退（R4-R5）→ 編輯與建單前提（R6-R7）→ 動機與來源標記（R8-R9）→ 事後監督（R10）」排列。每條規則的四要素（觸發條件 / 審核路徑 / 終態 / 一致性條件）見「具體例子」與「一致性條件」段。

### R1：補收正項免審，直達已執行

| 項目 | 內容 |
|------|------|
| 意圖 | 客戶主動加錢不被主管流程卡住，保住加單生意與出貨時程 |
| 規則 | 補收正項異動單（金額 > 0，且類型為加印追加／加運費／急件費／補退正項／規格變更正項）：免主管審核，從「草稿」直接推進到「已執行」（略過待主管審核與已核可）；核可人記為業務本人、執行時間記為當下；建立當下即認列應收 +N，且不綁任何退款款項 |
| 營運動機 | 客戶臨時加印是送上門的生意，等主管核可會拖延出貨、可能流失加單；加錢方向風險低（公司是收錢方），授權業務當場拍板 |

四要素：觸發＝客戶下單方向加錢；審核路徑＝免主管；終態＝已執行；一致性條件＝補收已執行不綁款項（六條已執行情境中唯一不綁款項者，見 R4 一致性條件）。

### R2：退款負項須主管核可且款項到位

| 項目 | 內容 |
|------|------|
| 意圖 | 退錢出去前先由主管把關金額與理由，且帳上「已退」必對應實際退款 |
| 規則 | 退款負項異動單（金額 < 0，業務發起）：走完整路徑——草稿 → 待主管審核 → 已核可 → 已執行；推進到已執行的條件是「已核可」後須綁實際退款款項，該退款款項切「已完成」且累計金額（含正負號）達到異動單金額 |
| 營運動機 | 錢退出去出錯難追回，由業務主管把關金額與理由是否合理；系統不讓「已執行」與「實際退款」脫鉤，避免帳上顯示已退、實際沒退 |

四要素：觸發＝品質／規格退費；審核路徑＝業務主管核可；終態＝已執行（須款項到位）；一致性條件＝退款已執行必有已完成退款款項累計達金額。

### R3：諮詢取消退費，系統建已核可

| 項目 | 內容 |
|------|------|
| 意圖 | 諮詢取消退一半是既定規則、不需人工判斷，但「該退」不等於「已退」 |
| 規則 | 諮詢取消退費異動單（系統內生、金額固定 −1000）：系統直接建為「已核可」（核可人記為系統、執行時間初始為空、免主管審核）；後續一樣要實際退款款項切「已完成」且累計達 −1000 才推進到已執行。此類型為系統專用，業務不可手動建立 |
| 營運動機 | 退一半是公司既定規則、不需主管再判斷該不該退（故免審）；建為已核可（而非直接已執行）讓業務在實際退款前仍能調整退款金額（沿用 R6 已核可可改），並修補「已執行卻配處理中退款款項」的一致性破洞 |

四要素：觸發＝諮詢取消（系統內生）；審核路徑＝免主管（系統建已核可）；終態＝已執行（須款項到位）；一致性條件＝同 R2（已執行必有已完成退款款項累計達金額）。

### R4：已執行的認列與回退

| 項目 | 內容 |
|------|------|
| 意圖 | 退款的「認列」與「撤銷」都要跟著實際退款款項走 |
| 規則 | 退款負項與諮詢取消退費：當關聯退款款項切「已完成」且累計金額（含正負號）等於異動單金額時，系統自動推進到已執行並記下執行時間。反向：若取消已完成的退款款項，導致累計金額不再等於異動單金額，系統同一筆交易把異動單從已執行退回已核可、清空執行時間、重算應收（見 R5）；若取消的非最後一筆、剩餘已完成退款款項仍滿足累計＝異動單金額，則維持已執行。取消「處理中」退款款項不影響異動單狀態 |
| 營運動機 | 防止「帳上已退、實際沒退」與「退款款項作廢後仍算在應收上」兩種對帳失真；處理中款項從未影響異動單，故取消它不連帶回退（resolve [[ORD-003-取消退款Payment是否回退OA\|ORD-003]]）|

一致性條件（invariant，系統恆守）：一張異動單若為「已執行」，必定有對應的已完成退款款項累計達到該單金額——補收正項除外（R1，補收不綁款項）。

### R5：應收總額認列公式

| 項目 | 內容 |
|------|------|
| 意圖 | 應收總額即時反映金額增減，但不替業務決定分幾期收 |
| 規則 | 應收總額 ＝ ∑印件費 ＋ ∑其他費用 ＋ ∑（已執行異動單金額）。諮詢取消為特例：認列「已核可」的諮詢取消退費異動單（因 R3 建為已核可而非已執行），公式 ＝ 其他費用 2000 ＋ 已核可異動單 −1000 ＝ 1000。異動單推進到已執行時觸發應收總額重算，但系統不自動建立或修改 [[分期請款狀態\|請款期次]] |
| 營運動機 | 應收要即時反映真實欠款／應退，但「分幾期收、何時開票」是業務依客戶狀況另行規劃，系統自動動期次反而打亂業務既有安排 |

### R6：金額編輯閘門按狀態分階段

| 項目 | 內容 |
|------|------|
| 意圖 | 退款執行前讓業務有最後校正空間，但不讓已生效的異動被竄改 |
| 規則 | 草稿：可改，改後重新送審。待主管審核：不可改。已退回：可改，改後重新送審。已核可：可改且不需重新送審（業務退款前最後校正，卡片同時顯示「主管核可金額 vs 當前金額」對照）。已執行：鎖定唯讀（不顯示編輯入口）。已取消：鎖定。每次改金額寫一筆稽核記錄，必含：異動時間／操作者／改前金額／改後金額／異動當時狀態 |
| 營運動機 | 已核可後現實常變（客戶談妥的退款數字微調），允許不重審以免流程空轉；但已執行代表錢已動、應收已認列，再改會破壞對帳，故鎖定。已核可可改是否需通知主管見 [[AFT-3-OA已核可改金額是否通知主管\|AFT-3]]、是否設金額閾值重審見 [[ORD-002-OA核可金額閾值閘門\|ORD-002]] |

### R7：金額為 0 不建單

| 項目 | 內容 |
|------|------|
| 意圖 | 沒有金額增減的規格調整不該佔用一張金額異動單 |
| 規則 | 純規格變更若金額 ＝ 0：不建立異動單。製作後的印件規格異動（非金額類，如規格備註／印件下單數量／單位／難易度）由 [[業務]] 直接編輯，系統自動通知相關 [[印務]]／[[印務主管]]／[[訂單管理人]] 並寫活動紀錄，不走異動單 |
| 營運動機 | 異動單是金額載具，規格調整無金額時走通知＋活動紀錄即可，避免無金額的空單據混淆對帳 |

### R8：補收／退款不對稱的營運動機（對稱破壞理由）

| 項目 | 內容 |
|------|------|
| 意圖 | 明確記錄「為什麼補收免審、退款送審」這條對稱破壞的營運根據，供主管驗證前提是否成立 |
| 規則 | 補收（R1）與退款（R2）採不對稱分權：主管把關現金流出方向（退款風險高、出錯難追回），不把關客戶下單追加方向（補收是送上門生意，等審核會拖延出貨、流失加單）。對齊台灣印刷業實務分權 |
| 營運動機 | 對稱設計（兩方向都送審或都免審）會在「擋不該擋的加單」與「放掉該管的退款」之間二擇一犯錯；不對稱分權讓風險與管控強度匹配。對稱破壞的表述方式見 [[BI-9-補收OA立即執行對稱破壞表述\|BI-9]] |

### R9：來源售後服務單標記

| 項目 | 內容 |
|------|------|
| 意圖 | 訂單期間與訂單完成後的退款／補收用同一套異動流程，僅以一個標記區分來源 |
| 規則 | 異動單可標記來源售後服務單（linked_after_sales_ticket_id）：未標記＝訂單期間 [[業務]] 直接建立；已標記＝從 [[售後服務]] 單內建立（退款或補印收費）。兩者共用同一狀態機與審核邏輯（R1／R2／R6 一致），此標記一經建立不可更動 |
| 營運動機 | 訂單完成後的客訴退費沿用同一套異動流程，不另開例外路徑，主管審核與認列邏輯一致、降低維護負擔；標記只供反查來源，不改變規則本身 |

### R10：大額補收事後監督

| 項目 | 內容 |
|------|------|
| 意圖 | 補收免審後，主管仍能在事後看到異常大額的補收，但不重新成為流程瓶頸 |
| 規則 | 補收金額超過大額閾值（建議起始 50000 元）：寫一筆活動紀錄並標紅，同時推送 Slack 通知給 [[業務主管]]。屬事後監督，不阻擋業務操作（補收仍免審直達已執行） |
| 營運動機 | 免審換到了出貨速度，但完全不可見會讓主管失去對大額現金流入異常（如誤填、舞弊）的察覺；事後紅標＋通知補上可見性，又不退回審核流程。大額閾值的最終取值見 [[BI-4-補收OA大額閾值定義\|BI-4]] |

## 具體例子

例子 1（R1 補收正項免審，正例）：訂單 #ORD-2026-0512 製作中，客戶臨時把名片從 500 張加到 800 張。業務建補收異動單 +6000 元（類型：加印追加），儲存後該單直接成為已執行，核可人記為業務本人，訂單應收當場 +6000。整批貨照常生產，不必停下來等主管，也不綁任何收款款項（收款另循正常流程）。

例子 2（R2 退款負項須款項到位，正例）：訂單 #ORD-2026-0488 因色差客訴，業務建退款異動單 −5000 元送審，業務主管核可後成為已核可。業務之後實際匯款並建退款款項 −5000 元、上傳匯款證明、切「已完成」，系統確認累計達 −5000，自動推進到已執行。

例子 3（R2／R4 分次退款未到位，邊界反例）：退款異動單 −10000 元已核可，業務先退第一筆 5000 元（退款款項 −5000 切已完成），累計 −5000 尚未達 −10000，異動單維持已核可、不推進已執行——錢只退了一半，不能算這筆退款已完成。

例子 4（R4 已執行回退，邊界反例）：訂單 #ORD-2026-0488 的退款異動單 −5000 元已是已執行，業務發現退款款項建錯、取消那筆已完成退款款項，系統重算累計變為 0、不再等於 −5000，於是自動把異動單退回已核可、清空執行時間，訂單應收同步把這 −5000 退回未認列狀態。

例子 5（R3／R5 諮詢取消特例，正例）：客戶諮詢後取消，諮詢費原收 2000 元，依規則退一半。系統自動建諮詢取消退費異動單 −1000 元、狀態已核可、核可人記為系統。此時應收 ＝ 2000（其他費用）＋（−1000，已核可異動單）＝ 1000。諮詢人員後續實際退款 1000 元、退款款項切已完成，系統確認累計 −1000，推進到已執行。

例子 6（R9 來源標記，正例）：訂單 #ORD-2026-0470 已完成後客戶反映外觀瑕疵，業務在售後服務單內建退款異動單 −3000 元，系統自動寫入來源售後服務單標記。這張異動單仍走「草稿 → 待主管審核 → 已核可 → 已執行」（與訂單期間退款同一套），主管在異動單層級核可金額。

例子 7（R10 大額補收，正例）：訂單 #ORD-2026-0531 客戶追加大量印量，業務建補收異動單 +80000 元（類型：加印追加）。金額超過 50000 閾值，系統照 R1 免審直達已執行（不卡業務），同時寫一筆紅標活動紀錄並 Slack 通知業務主管，主管事後可檢視這筆異常大額補收。

例子 8（R7 金額為 0 不建單，邊界反例）：訂單製作後客戶把某印件的紙張磅數從 200 改 250（不影響售價，金額差 ＝ 0），業務直接修改該印件規格，系統推通知給工單負責印務並寫活動紀錄，全程不建任何異動單。

## 一致性條件（invariant）

> 系統恆守、供對帳驗證的硬條件。

1. 異動單為「已執行」 → 必有對應已完成退款款項累計達該單金額（補收正項 R1 例外，不綁款項）。
2. 補收正項建立即認列應收 +N、且不綁款項（R1）。
3. 應收總額 ＝ ∑印件費 ＋ ∑其他費用 ＋ ∑（已執行異動單金額）；諮詢取消特例改認列已核可異動單（R5）。
4. 來源售後服務單標記一經建立不可更動（R9）。

## 邊界（不適用 / 範疇外）

- 主訂單本身的取消、終態收斂（含諮詢取消後諮詢訂單走「已取消」終態）→ [[訂單狀態]]，不在本卡。
- 異動單自身的狀態定義、轉換條件、觸發事件（草稿／待主管審核／已核可／已退回／已執行／已取消）→ [[訂單異動狀態]]（本卡寫規則與動機，狀態圖在該卡）。
- 請款期次的開票與收款維度狀態、期次變更稽核 → [[分期請款狀態]]；本卡只負責「已執行時重算應收總額」，不碰期次（R5）。
- 售後服務單本身的受理、處理、結案進度 → [[售後服務]]；本卡只負責售後單內建立的退款／補印收費異動單的規則。
- 退款後是否開折讓單或作廢重開發票、三方對帳收斂的整段端到端旅程 → [[訂單異動流程]]（補收／退款端到端 scenario）與 [[07-scenarios/README#情境 16：訂單取消與退款端到端（跨角色多模組連鎖）]] 與 [[付款發票邏輯]] 索引；本卡只管異動單這一段規則。
- 補收／退款各情境（公司吸收不建單／補收／退款）的分類全貌與七實體連帶影響 → [[付款發票邏輯#四、訂單異動情境（A1-A3）]] 與 [[付款發票邏輯#五B、七實體連帶矩陣（2026-05-28 新增）]]。
