# 數量換算規則

> 核心邏輯：印件 → 工單 → 生產任務 三層的數量倍數換算。確保「印件完成數」計算精確。

## 一、三個關鍵欄位

| 欄位 | 所屬實體 | 定義 | 範例值 |
|------|---------|------|-------|
| `quantity_per_print_item` | [[工單]] | 完成 1 份印件需要多少份此工單 | 1000、0.5 |
| `quantity_per_work_order` | [[生產任務]] | 完成 1 份工單需要多少份此生產任務 | 1、2、0.5 |
| `affects_product` | [[生產任務]] | 該生產任務的 QC 通過數是否計入工單完成度（TRUE / FALSE） | TRUE |

> 三欄位必須 > 0；`affects_product` 為布林值。

## 二、簡單 1:1:1 計算範例

印件含 1 工單（`quantity_per_print_item = 1`），工單含 1 任務，任務含 1 生產任務（`quantity_per_work_order = 1`、`affects_product = TRUE`），該生產任務 QC 通過數為 1000：

- 層級 1：`pt_qc_passed = 1000`
- 層級 2：`task_completion = floor(1000/1) = 1000`
- 層級 3：`wo_completion = 1000`
- 層級 4：`pi_completion = floor(1000/1) = 1000`

→ 1000 份印件完成。

## 四、邊界防呆

| 情境 | 系統行為 |
|------|---------|
| `quantity_per_print_item ≤ 0` 或 `quantity_per_work_order ≤ 0` | ERROR 阻擋儲存，提示「數值必須大於 0」 |
| `quantity_per_work_order > 10000` | WARNING 提醒印務確認，但允許儲存 |
| 一個任務內所有生產任務 `affects_product` 皆為 FALSE | 阻擋儲存，提示「至少需有一個生產任務影響成品」 |
| `pi_completion = 0` | UI 顯示缺口「已完成 X / Y（缺口 Z）」 |

## 五、異動流程數量重算

| 觸發點 | 系統行為 |
|--------|---------|
| 新增生產任務 | 以該任務現有生產任務的平均 `quantity_per_work_order` 為建議值，印務主管可覆寫 |
| 修改 `affects_product` | 修改前提示「修改將重新計算工單完成數，是否確認？」→ 確認後重算 |
