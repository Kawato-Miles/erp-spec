# 商業邏輯卡撰寫範本（共用規則 / 業務規則層）

> 給 Miles 手動撰寫、或 agent 在 `/opsx:propose` 背景對齊 / `erp-planning-pre-check` 修補階段，新建或重寫 `04-business-logic/` 卡時複製套用。
>
> **這份範本同時兩用**：寫的時候照著填、事後照著檢查。撰寫時依章節骨架填佔位符；事後 `doc-audit` / `vault-audit` 依底部「Lint 自審稽核清單」勾稽。完整撰寫範式（每節為什麼這樣寫）見 [[business-logic-writing-guide]]；卡類型邊界 lint 依據見 [[wiki-schema#十一、卡類型內容職責邊界]]；分層定位見 [[wiki-architecture#business-logic 內部分層：共用規則 vs 業務規則]]。本範本不重述範式內容，只提供可複製骨架——範式衝突時以 [[business-logic-writing-guide]] 為準。

## 一、這類卡的定位（撰寫前先判斷）

`business-logic` 這個 type 橫跨「共用規則」與「業務規則」兩層（[[wiki-architecture#business-logic 內部分層：共用規則 vs 業務規則]]），靠卡的角色區分、不另立 type。撰寫前先判斷本卡是哪一種，再選對應骨架變體（§ 三）：

| 卡角色 | 性質 | 該寫 | 例 | `source` 往上指誰 |
|--------|------|------|----|------------------|
| **共用規則卡** | 某領域所有規則共用的底線（一定要成立、可驗算）| 這個領域所有規則都一定要遵守的底線（如對帳一致性）+ 各業務規則須服從的硬條件 | 對帳一致性「應收 ＝ 發票淨額 ＝ 收款淨額」 | 營運原則（[[wiki-architecture\|operating-principles]]）／最上層的依據（Miles 拍板 + 印刷業實務）|
| **業務規則卡（規則型）** | 具體單點 if-then 要件 ＋ 計算公式 | 觸發條件 ＋ 判斷 ＋ 動作 ＋ 一定要成立的規則 ＋ 公式；該規則的單一正本 | [[訂單異動規則]]（補收免審／退款送審）| 本領域**共用規則卡**（如對帳一致性）|
| **業務規則卡（索引型）** | 一組相關情境的分類概覽 ＋ 跨情境共用規則 ＋ 連帶矩陣 | 分類表 ＋ 共用實體 ＋ 七實體連帶矩陣；明細連到 scenario / `payment-invoice-scenarios` | [[付款發票邏輯]] | 同業務規則層（指向本領域共用規則或最上層的依據）|

- **判斷口訣**：「這是一條可驗算、各業務規則都不得違反的領域底線嗎？」→ 是則共用規則卡；「這是一條 if-then 要件嗎？」→ 是則業務規則卡（規則型）；「這是一群情境的分類入口 ＋ 連帶矩陣嗎？」→ 是則業務規則卡（索引型）。
- **MUST NOT 越位**：共用規則卡不下放到單點 if-then（那是業務規則）；業務規則卡不上僭定領域底線（那是共用規則）。

## 營運背景

> 公司營運上遇到什麼情況 / 需求才有這張卡。2-4 句，業務語言，**無實作術語當主詞、無中英夾雜**（技術詞括號附註）。主管 / PM 的入口——讀完應能判斷「我們對商業的理解對不對」。
> 該寫：營運痛點、角色實務、現金 / 交期 / 客戶風險。
> 不該寫：欄位名 / 程式碼物件 / UI 操作 / 系統怎麼實作。

<2-4 句營運背景>

## 規則與營運動機

> 逐條規則，每條用「意圖 → 規則 → 為什麼」呈現（讓 doc-audit / 三視角審查可勾稽「推理有沒有跳過既有規則」）：
> 意圖（這條規則想達成的業務目的）→ 規則（精確陳述，可含條件 / 觸發 / 動作）→ 營運動機（連回痛點 / 角色 / KPI，並引用更上層卡作為既有一定要成立的規則）。
> **每條規則的定位點用業務語意命名**（如 `#補收免審`），不用流水號（R1）、不重排、不重用——改規則只改內容、定位點不變，連結不斷。
> 該寫：if-then 要件本體 + 動機。
> 不該寫：跨模組完整流程（→ scenario）/ 狀態轉換圖（→ state-machine）/ 角色職責清單（→ role）/ 完整實體 Data Model（→ entity）。

## 三、規則定位點命名規約（用業務語意命名，不用流水號）

- 規則 / 一定要成立的規則 / 狀態的定位點**用業務語意命名**（H3 標題即定位點，如 `### 補收免審` → 連結 `[[卡名#補收免審]]`）。
- **不用流水號**（R1 / R2）、**不重排、不重用**：改規則只改內容、定位點不變，連結指向的位置穩定、連結不斷。
- 定位點用業務語意命名（補收免審 / 退款送審 / 應收公式 / 三方對帳相等），不用編號或英文。
- 既有 `#R1` 定位點屬待遷移，新卡一律用業務語意命名。

## 四、不可違反的硬規則 vs 可視情況調整的彈性（撰寫時對照）

> 以下兩層明示，禁作者私自偏離。

### 不可違反（最上層硬規則，極少且硬）

- 禁中英夾雜（技術詞括號附註）；營運背景段禁工程術語當主詞（檔名 / interface / function / UI 措辭）。
- 識別不確定項 MUST 開獨立 OQ（觸發 `oq-manage` skill mode B），**禁 inline 標注、禁 `[!question]` callout**。
- `source` 禁指下游 / 同層業務規則 / OpenSpec（依據往上、實作往下，連結不繞回自己，[[wiki-architecture#依據往上、實作往下，連結不繞回自己]]）。
- 正文零迭代史（迭代史進 [[business-logic-changelog]]）。
- 規則單一正本不複寫（同一規則只在一張卡寫完整版，他卡 wiki link 引用）。
- 四要素（觸發 + 審核路徑 + 終態 + 不變條件）MUST 留同一卡，不可再拆。

### 可視情況調整（明示彈性破口）

- **無爭議純事實型規則**：「這張卡要回答的問題」段可用敘述代替問句（不強制每卡都列問句）。
- **單純常數**（如「諮詢費寫死 2000」「諮詢取消退一半」）：留母卡、不獨立成卡（要被多處引用才獨立成卡，不先建好備用）。
- **索引型 vs 規則型 vs 共用規則**：依 § 一判斷選對應骨架變體，不強求單一模板。
- **缺 source**：過渡期補不出真 source 時暫標 `source-gap`（frontmatter 加 `status-note: source-gap`），暫時補不出上層依據、標記待之後補，不阻擋建卡。

## 六、Lint 自審稽核清單（撰寫完逐項打勾 ＝ 事後稽核基準）

> 這份清單同時兩用：撰寫完作者逐項自審；`doc-audit` / `vault-audit`（維度 5 / 6 / 14）依同一份勾稽。任一項未過即不可 commit。

### 共同骨架（所有層）
- [ ] frontmatter 共同 5 欄全填（type / module / business-domain / status / last-reviewed）。
- [ ] `source` 已填且**不指同層業務規則卡 / 下層 / OpenSpec**；補不出真 source 暫標 `source-gap`。
- [ ] `implemented-by`（或過渡期 `related-spec`）指 OpenSpec Requirement 標題層 / prototype，不承載正確性。
- [ ] 「這張卡要回答的問題」段每個問句正文有對應明確結論（或屬無爭議純事實，已用敘述代替——彈性破口）。
- [ ] 營運背景段無實作術語當主詞、無中英夾雜（技術詞括號附註）、無 UI 措辭。
- [ ] 正文零迭代史（迭代史在 [[business-logic-changelog]] 對應 section）；「來源」段末行有「迭代脈絡見 [[business-logic-changelog#<卡名>]]」。
- [ ] 正文零「待確認 / 待釐清 / 需確認 / 尚未確認 / 待補」inline OQ 措辭、零 `[!question]` callout（→ `oq-manage` skill mode B）。
- [ ] 相關連結雙向可達、語意分類（上層情境 / 下層步驟 / 相關角色 / 相關規則 / 相關狀態 / 相關實體 / 相關 OQ）；無連到不存在的卡（dangling）、非沒有任何卡連到它的孤島卡（orphan）。
- [ ] 最後驗收：營運背景抽一句，問「Miles / 主管不看程式碼能否懂」。

### 本層特有（business-logic）
- [ ] 規則正本歸屬聲明已寫（卡頂 `>` 引言聲明「本卡承載完整定義、他卡引用不重述」）。
- [ ] 規則 / 一定要成立的規則的定位點用**業務語意命名**（H3 標題如 `### 補收免審`），不用流水號（R1）、不重排、不重用。
- [ ] 每條規則用「意圖 → 規則 → 為什麼」呈現；營運動機連回更上層卡作為既有一定要成立的規則（不跳過既有規則）。
- [ ] 具體例子用真實格式（編號 / 金額 / 角色），**禁 UI 措辭、禁占位符、禁抽象等價類、禁堆無關細節**；含正例 + 邊界反例。
- [ ] 不變條件可驗算（如對帳一致性）；業務規則卡不變條件服從本領域共用規則卡、不破壞共用規則。
- [ ] 共用規則卡：標明「各業務規則 source 可往上指本卡」+ `source` 只指營運原則 / 最上層的依據；業務規則卡：`source` 往上指本領域共用規則卡。
- [ ] 四要素（觸發 + 審核路徑 + 終態 + 不變條件）留同一卡，未過度拆分；單純常數留母卡未過度抽象。

### 越界檢查（[[wiki-schema#十一、卡類型內容職責邊界]] 維度 14，MUST NOT 寫入）
- [ ] 未寫**跨模組完整流程 / step-by-step**（→ [[../07-scenarios]] scenario 卡）。
- [ ] 未寫**狀態轉換圖 / ASCII 狀態機 / 狀態定義長篇**（→ [[../06-state-machines]] state-machine 卡）。
- [ ] 未寫**角色職責清單**（→ [[../03-roles]] role 卡）。
- [ ] 未寫**完整實體 Data Model 表格**（→ [[../05-entities]] entity 卡）。
- [ ] 未寫 **user-story 格式模板**（「作為 / 我希望 / 以便」連續三段）、未寫 **test-case 範本**（前置 / 步驟 / 預期結果）——產 user story / test case 一律 cross-reference [[../11-review-knowledge/pm/user-story-spec]] + `erp-user-story` / `erp-test-case` skill，禁複製模板進卡。
