# 免審決策樹

> **業務規則：業務於需求單階段決定每個印件是否免審**。免審路徑跳過審稿人員分配，直接產生 `source=免審稿` 的合格 Round。

## 一、定義

`skipReview`（是否免審稿）：布林值，由[[業務]]或[[諮詢]]於需求單階段對每個印件決定。

## 二、業務決策樹

```
業務評估印件 → 是否免審？
├── YES（skipReview = true）
│   └── 成交轉訂單時觸發「免審稿快速路徑」
│       ├── 系統建立 ReviewRound（source=免審稿、reviewerId=null、result=合格）
│       └── 印件狀態直接到「合格」
│
└── NO（skipReview = false）
    └── 走一般審稿流程
        ├── 訂單付款 hook 觸發自動分配（見 [[審稿分配規則]]）
        ├── 系統依難易度匹配審稿人員
        └── 印件狀態：等待審稿
```

## 三、典型免審情境

> [!question] OQ
> 目前 spec 未明確定義「哪些情境算免審」，由業務 / 諮詢主觀判斷。建議釐清以下情境是否屬免審：
>
> - 重複下單（同一規格已審過）？
> - B2B 大客戶免審政策？
> - 標準規格商品（無設計變動）？
> - EC 自動商品（無自由設計欄位）？

## 四、免審路徑技術細節

當 `skipReview = true` 時，系統執行：

| 步驟 | 行為 |
|------|------|
| 1. 不進行自動分配 | 跳過審稿人員候選集篩選 |
| 2. 直接建 ReviewRound | `source = '免審稿'`、`reviewerId = null`、`result = '合格'` |
| 3. 印件狀態推進 | 「稿件未上傳」→「合格」（直接跳過「等待審稿」） |
| 4. 開放下游工單 | 直接開放[[印務主管]]建工單草稿 |

## 五、跨層級繼承

| 階段 | 行為 |
|------|------|
| 需求單階段 | 業務評估並設定 `PrintItem.skipReview` |
| 成交轉訂單 | 自動帶入 `OrderPrintItem.skipReview`（**parity 欄位，需求 / 訂單跨層同步**）|
| 訂單階段 | 不可再修改（避免已轉訂單後改變審稿流程） |

> [!info] Parity 欄位
> `skipReview` 為 Quote → Order 跨層 parity 欄位，data-consistency-audit change 處理。與 [[印件檔案備註上限|client_note]]（脫鉤型）不同。
