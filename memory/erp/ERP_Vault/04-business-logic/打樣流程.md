# 打樣流程

> 核心：打樣工單與大貨工單透過 `WorkOrder.type` 欄位區分（打樣 / 大貨）。打樣 WorkOrder 推進至「已完成」後，業務於打樣 WorkOrder 詳情頁判定 `sampleResult`（四值 enum）。判定結果決定後續分支。

## 一、打樣與大貨的關係

- 若需打樣，**打樣印件與大貨印件 SHALL 同時建立**
- 打樣 / 大貨各為獨立的 PrintItem，但屬於同一張訂單
- 打樣 WorkOrder（`WorkOrder.type = 打樣`）對應打樣印件；大貨 WorkOrder（`type = 大貨`）對應大貨印件

## 二、打樣結果業務判定（resolve-prepress-review-gaps-ar-10-ar-12 對齊 Prototype 實作）

**觸發位置**：打樣 WorkOrder 詳情頁（**不是印件詳情頁**）
**觸發條件**：`WorkOrder.type = 打樣 && WorkOrder.status = 已完成 && PrintItem.sampleResult = 待確認`
**判定者**：業務（owner of 訂單）
**判定結果**：

| sampleResult | 處理 |
|-------------|------|
| `待確認`（預設）| 業務尚未判定 |
| `OK`（通過）| 業務後續手動建大貨工單（系統不自動觸發）|
| `NG-製程問題` | 業務 UI 自行建新打樣 WorkOrder 重做（系統不自動建；下游自動化待 [[AR-13-打樣NG製程問題下游處理機制]]）|
| `NG-稿件問題` | 系統自動觸發棄用 + clone 新打樣印件流程 |

判定不可逆（避免規避稽核軌跡）；ActivityLog 走訂單層弱型別文字「填打樣結果：{result}（工單 {workOrderNo}）」（對齊 Prototype 既有實作）。

## 三、典型情境

### 情境 1：打樣通過後業務手動建大貨工單

- `sampleResult = OK`
- 業務後續手動建大貨工單 / 進入大貨生產流程
- 系統 MUST NOT 自動建大貨工單（保留業務決定權）

### 情境 2：打樣失敗（NG-製程問題）

- `sampleResult = NG-製程問題`（製程問題，稿件本身無問題）
- 該打樣印件審稿維度維持「合格」不動
- 業務 UI 自行於同打樣印件下建立新打樣 WorkOrder 重做
- 系統 MUST NOT 自動建新打樣 WorkOrder（保留業務決定權）
- ng_process 下游自動化處理機制細節待 [[AR-13-打樣NG製程問題下游處理機制]] 解後實作

### 情境 3：打樣失敗（NG-稿件問題）— 自動棄用 + clone

- `sampleResult = NG-稿件問題`（稿件問題）
- 系統 atomic 自動觸發：
  - 原打樣印件 `printItemStatus = '已棄用'`（用既有 PrintItemStatus enum）+ notes 加註棄用說明
  - clone 原打樣印件至新打樣印件（沿用印件規格 / 客戶資訊 / 訂單關聯 / difficultyLevel；reset 審稿維度 + 印製維度；`sampleResult = '待確認'`）
  - 新印件 `derived_from_print_item_id` 指向原印件 ID（**結構化追溯 FK，本 change 新增**）+ notes 加註「（由 [原印件 ID] NG-稿件問題重建）」
  - 訂單層 ActivityLog 寫入「NG-稿件問題：棄用 [原印件號]，建立新印件 [新印件號]」
- 業務後續與客戶聯繫取得新稿件 → 於新印件上傳新稿件 → 進入審稿流程
- 對齊 [[US-AR-011-打樣後重新處理稿件]]、[[AR-12-打樣後新稿件實體機制與根因判定]]（已 closed）

## 四、與付款發票的關係

打樣相關的補收 / 退款情境見 [[付款發票邏輯]]：

- **打樣酌收**：原規格送打樣，但打樣後不製作大貨 → OrderAdjustment（打樣費用補收）
- **第二次打樣是因第一次出問題**：走原工單不補收（公司認賠）
