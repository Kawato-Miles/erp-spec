# 業務

> [!success] OpenSpec spec 狀態：**已有**
> `openspec/specs/user-roles/spec.md` 中已定義此角色。內容以 Notion DB 為 ground truth，OpenSpec 為交叉驗證。

## 基本資料

- **平台**：業務平台
- **利害關係程度**：高
- **參與階段**：評估 / 訂購 / 審稿 / 打樣印製 / 出貨（全程）
- **任務操作**：X
- **報價單 / 訂單**：R/W
- **工單**：X（僅查閱狀態、QC 紀錄）
- **需求單**：R/W

## 主要工作職責

Sales process（售前評估、成本評估、需求單管理、線下訂單管理）；款項、發票處理

- **訂單異動建立與執行**：於訂單詳情頁建立訂單異動單（記錄規格變更 / 加印追加 / 退印 / 折扣 / 加運費 / 急件費 / 補退等金額增減，可正可負）。補收正項異動單直接執行、免業務主管核可（客戶加單不該被審核卡住交期）；退款負項異動單須送業務主管核可，核可後由業務在異動單建關聯退款款項並切已完成以推進至已執行（退款是現金流出方向，需主管把關 + 實際退錢動作為證）。已核可後業務可調整退款金額不需重新送審。規則正本見 [[訂單異動規則]]

## 關切點

- 報價流程順暢
- 訂單狀態可追蹤
- 補單或規格變更有記錄
- 客戶改稿後的追加費用可在系統補記，避免漏收
- 接客戶來電查詢時，能在訂單列表快速定位並回覆訂單下特定印件的審稿與印製狀態（[[US-ORD-034-業務查訂單下印件狀態]]，add-print-item-subrows-to-order-list change）

## 預期阻力

- 補單與規格異動是日常常態，系統化後需輸入更多資訊，可能被認為增加操作負擔
- 改稿追加費用漏收的責任歸屬，可能引發內部核算摩擦

## 備註

- 需可「分享」單據給其他業務 / 諮詢（職務代理 / 檢視權限，不改負責人；沿用 PermissionManagement）；「轉派負責業務」（改 owner）收歸業務主管，業務不可自行轉派（[[../08-open-questions/XM-008-業務主管改派負責業務的權限邊界與機制|XM-008]]、add-sales-manager-reassign-owner change 2026-05-30 收斂：分享 ≠ 轉派）
- 工單僅查閱狀態、QC 紀錄

## 款項追款流程

> 2026-05-28 Miles 對談補充（來源：BI-13 第一輪稽核對談）。

- **追款動機**：業務的款項收訖狀況**影響其獎金** → 業務有主動追款誘因，會主動掌握哪些帳款待與客戶確認（不需主管 push）
- **現況做法**：業務**自己記錄**待確認 / 待追的帳款（無系統支援，靠個人 Excel / 筆記）
- **ERP 目標**：用系統「**待開發票**」清單 + 「**應收款項**」清單取代業務自行記錄，讓業務在系統內主動追蹤待追帳款
- **與會計協作**：會計月結對帳時若發現「發票已開立但未收款」，會提醒業務追款（見 [[會計#月結對帳節奏]]）
