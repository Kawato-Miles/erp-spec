# 審稿

> [!info] OpenSpec spec 狀態：**部分對應**
> `openspec/specs/user-roles/spec.md` 中有「品管」角色但與「審稿」實際職責不同：
> - **審稿**（本卡）：負責稿件合規確認與製作（製前工序）
> - **品管 / QC**（[[QC]]）：負責成品 QC（印刷後品質檢驗）
>
> OpenSpec 將兩者合併為「品管」，與 Notion DB 不一致。需後續評估是否拆分。詳見 [[_alignment-report]]。

## 基本資料

- **平台**：審稿平台
- **利害關係程度**：高
- **參與階段**：僅審稿階段
- **任務操作**：X
- **報價單 / 訂單**：R/W
- **工單**：X
- **需求單**：X

## 主要工作職責

確認稿件內容是否合規、製作稿件；EC 客服問題處理

## 關切點

- 明確知道哪些稿件需要合規確認
- 審稿結果有留存記錄
- 可追蹤待審稿件的等待時間

## 預期阻力

審稿流程若未整合進工單狀態機，仍可能以 Slack 進行，形成資訊斷點

## 備註

排除工單、營收資料
