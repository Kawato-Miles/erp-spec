# 商業邏輯卡撰寫範式（營運驗證導向）

> ERP_Vault 商業邏輯正本卡的撰寫範式，涵蓋 [[wiki-architecture#分層體系（營運原則 → 驗收項目，由大到細）|wiki-architecture § 分層體系]] 六層的全部六種載體：營運原則（最高層，`product-vision` type）/ 共用規則與業務規則（`business-logic`）/ 流程・狀態・角色・資料（`scenario` / `state-machine` / `role` / `entity`）/ 操作步驟（`user-story`）/ 驗收項目（`test-case`，UAT 業務層）。
>
> 核心目的：讓**管業務的主管**（業務主管 / CEO / 諮詢主管）**與 PM（Miles 本人也會讀卡，作為規劃與迭代的依據）**讀卡時能檢視「我們對商業的理解是否正確」，而不是只能讀懂「系統怎麼做」。PM 讀卡的訴求多一層：迭代某條規則時，要能沿關聯精準定位「這條規則的正本在哪張卡、改了會連帶影響哪些卡」（見 § 二原則 6、原則 7）。
>
> **三份管理文件的分工（讀前必懂）**：
> - [[wiki-architecture]]（**文件管理規範（最高層）/ 上位**）：定分層體系、依據往上指（`source`）／實作往下指（`implemented-by`）、引用方向、歸類決策樹（§ 10.3）、新增三門檻（§ 10.4）、各 type 範本清單（§ 10.1）、change 後增修機制（§ 10.5）。本卡是它在「各 type 卡正文怎麼寫」這一層的展開，**不重複**它的決策樹與門檻全文，只引用。
> - [[wiki-schema]]：規範 frontmatter（§ 四）與卡類型「邊界」（哪些內容該寫哪張卡，§ 十一 + 維度 14 lint）。本卡是 wiki-schema § 十一內容職責的撰寫範式展開。
> - 本卡（business-logic-writing-guide）：規範各 type 卡正文的「撰寫範式」（每張卡內部怎麼組織、怎麼注入營運驗證寫法、§ 4.0 各類卡的單一職責）。
>
> 三者不衝突：wiki-architecture 是「為什麼這樣分層 + 怎麼歸類 + 怎麼往上指依據」（上位管理），本卡是「每張卡正文怎麼下筆」（撰寫範式），wiki-schema 是「frontmatter 與邊界 lint」（機器可檢）。各 type 卡實際下筆時複製對應範本（§ 十清單），本卡是範本背後的範式說明。

## 一、為什麼需要這份範式

主管驗證的不是「系統怎麼做」，而是「這條規則是否符合我們的生意」。三個痛點促成本範式：

1. **規則只寫 what、不寫 why**：卡只寫「補收正項免審直達已執行」，主管讀了只能點頭或搖頭，無法指出「你假設錯了，超過五萬的補收我們要管」。記錄「為什麼這樣規劃」才讓主管揪出隱藏假設與錯誤前提。
2. **正文混入迭代史**：卡正文寫「v1.10 改半額（推翻 v1.7 全退）；v1.17 終態由訂單完成改已取消」，主管得在腦中重演整段演進史才拼出現況，且容易把廢止的舊規當現行。
3. **同概念多種叫法**：同一實體在不同卡叫「付款計畫 / 請款期次 / BillingInstallment」，主管無法確認在講同一件事，驗證失效。
4. **一條規則散在多張卡、且職責混淆**：補收 / 退款不對稱分權同時出現在訂單異動狀態卡、售後服務卡、訂單實體卡；狀態機卡又夾帶整段業務規則與計算公式。同一條規則散在多張卡、改一處漏其他處，正是「converge change 漏對齊 wiki 6 卡」「archive 後 wiki 卡停留舊版」的根因。PM 迭代時無法回答「這條規則的正本在哪、改它會連帶影響誰」。

對應 [[scope-boundary]]：Vault 只收商業層 WHAT / WHY（不收 UI 規範、演算法、step-by-step Requirement）。本範式是「WHY 怎麼寫得讓主管能驗證」的具體規約。

## 二、七條核心原則

> 原則 1-5 規範「每張卡內部怎麼寫」（讓主管能驗證）；原則 6-7 規範「卡與卡怎麼切、怎麼連」（讓 PM 能定位正本、做影響分析）。原則 6-7 是 2026-05-30 原子化要求新增。

### 原則 1：每條規則「意圖 → 規則 → 營運動機 → 例子 → 來源」分欄呈現

把「規則陳述（what）」與「為什麼（why）」分欄，不混在一段敘述裡。固定結構讓主管每張卡用同一套眼睛掃，降低審閱負擔。

- **意圖**：這條規則想達成的業務目的，一句話。
- **規則**：精確陳述（可含條件 / 觸發 / 動作），業務語言。
- **營運動機（rationale）**：公司營運上遇到什麼情況 / 需求，所以才這樣規劃。連回營運痛點、角色實務、或 KPI。
- **例子**：見原則 3。
- **來源**：連回可獨立驗證的外部出處（見原則 2）。

實例（諮詢取消半額退費）：
- 意圖：取消時公司已投入諮詢成本，需保留部分收入。
- 規則：取消觸發 50% 退費，不分時機、不分主動方。
- 營運動機：諮詢人員已出席投入工時，全退會侵蝕成本；CR-3 拍板半額。
- 來源：[[../08-open-questions/CR-3]]（待確認時的決策）+ converge-consultation-cancel change。

### 原則 2：往上指依據（source）、往下指實作（implemented-by），兩個方向不會接回自己

每張卡的依據與實作用**兩個語意不同、方向相反**的欄位（對齊 [[wiki-architecture#依據往上、實作往下，連結不繞回自己|wiki-architecture § 依據往上、實作往下]]）：

- **`source`（往上指依據 = 這張卡為什麼對）**：指向更上層卡或外部已驗證的依據——真實業務情境、訪談紀錄、法規硬約束、CEO 指標、拍板的 OQ、或更上層卡（業務規則往上指共用規則、共用規則往上指營運原則、營運原則的依據來自老闆拍板與印刷業實務、不再往上追）。**嚴禁指同層 / 下層 / OpenSpec**，也不指向另一張同類商業邏輯卡（規則卡之間互引會繞回自己、變成自己證明自己）。
- **`implemented-by`（往下指實作 = 被誰落實）**：指向 OpenSpec Requirement 標題層、Prototype 端對端測試、或 test-case。**不承載正確性**，只供「這條規則被哪裡落實」的導航。
- 因兩欄方向相反 → 結構上不可能 A.source→B 且 B.source→A，往上的依據鏈不會接回原點。

為什麼這樣設計：

- **正確性可驗**：當業務現實改變（例如退費政策調整），主管能沿 `source` 往上找到原始情境 / 最上層依據，確認規則是否該改。若規則 A 的 source 指規則 B、B 又指 C，繞成一圈、變成自己證明自己（這也是 AI 拿自己寫的東西當依據再生新東西的風險），主管無從驗證最初的業務假設。
- **導航不污染正確性**：把「這條規則由哪個 OpenSpec Requirement / prototype 實作」放 `implemented-by`，而非塞進 source。下游實作規格是上游需求文件的落地，不能反過來當上游規則的正確性來源（對齊 [[wiki-architecture#營運需求（Obsidian）與實作規格（OpenSpec）的分工|wiki-architecture § 營運需求與實作規格的分工]]）。
- **單一正本**：同一規則只在一張卡寫完整版，其餘卡 wiki link 引用，避免主管讀到不同版本的矛盾（見原則 6）。
- 與 erp-user-story skill 強制規則 3「source 禁指向其他 user-story 卡」、test-case 範本「source 指 user-story 不指同層」同源。
- **遷移誠實標注**：現況約 12+ 張 business-logic 卡的「來源」段系統性指向 OpenSpec（拿下游當上游、方向顛倒），且多數卡僅有 `related-spec`（implemented-by 的弱版）。前進標準採 `source` / `implemented-by` 雙欄，但**不要求一次全改**——新卡 / 被 change 異動的卡優先補，補不出 source 的標 `source-gap`（暫時補不出上層依據，標記待之後補）；`implemented-by` 留空 = 待實作（漸進、不假裝）。
- 實例：[[發票法規硬約束-ezPay-MIG]] 的硬約束理由連回外部 ezPay API 與財政部 MIG 法規（外部正本），而非連到另一張內部卡。

### 原則 3：用具體真實例子（含真實情境與具體數字）讓主管驗證

抽象規則（「取消時部分退費」）主管讀了會點頭，但點頭不代表理解一致——「部分」是多少？哪些情況算？具體例子強迫把模糊處攤開，主管一眼就能說「不對，超過 7 天我們不退」。

- 用真實格式的情境（真實金額 / 數量 / 角色 / 編號），不用占位符（對應 memory feedback「Prototype mock 要貼近真實情境」）。
- 一條規則配 1-3 個關鍵例子（含正例 + 邊界反例），不堆砌窮舉。
- 具體數字讓計算邏輯可被驗算（對應 CLAUDE.md 原則 2「邏輯驗證優於空談」）。
- 實例（對帳差錯偵測）：訂單 #ORD-2026-0512 已取消但已開發票 1000、應收 1000 → 仍須進對帳清單；若從對帳消失即為差錯。

這些例子同時是 erp-test-case 與 erp-user-story 的原始素材（[[payment-invoice-scenarios]] 13 情境即此用法），達成「一份具體情境餵多個下游產物」。

### 原則 4：例子停在業務語言，禁綁 UI / 禁抽象等價類 / 禁堆無關細節

避開三個反模式：

- **禁綁 UI 操作步驟**：寫「點訂單詳情頁 → 按新增訂單異動 → 選退款」，主管驗證的是 UI 流程不是業務規則，且 UI 一改例子就過時。例子 MUST 用角色 / 業務動作 / 金額 / 狀態描述（「業務對已完成訂單發起補收 5000，正項免審直達已執行」）。與 erp-user-story skill 強制規則 1「業務情境段禁含 UI 措辭」完全一致。
- **禁用抽象等價類**：寫「金額為負時」等於沒給例子；給一個具體值（「退款 −1000」）。
- **禁堆無關細節**：每個例子只保留驅動該規則的變數（金額正負、訂單狀態、角色），其餘（客戶地址、時間戳）省略，否則主管得自己過濾才看到重點。

UI 綁定的驗證留給 Prototype 端對端測試（Playwright），商業卡只管業務意圖。

### 原則 5：正文只寫現在的規則，改版歷史另放 changelog

主管驗證「需求理解是否正確」讀的是「現在規則是什麼」。

- 正文 MUST 只描述**當前生效**規則。
- 廢止 / 取代 / 哪個 change 改的 / 取代關係，MUST 抽離到獨立 changelog（見 § 五），指回對應卡。
- 想知道「為什麼改成這樣 / 哪版改的」再去 changelog 下鑽（分眾 cross-link：主管讀精簡現況，需要溯源才查 changelog）。
- 反例正是 [[../06-state-machines/分期請款狀態]] 的 `## 取代關係（廢止舊雙實體狀態）` 段（舊→新對照表混在正文）與 [[../06-state-machines/訂單異動狀態]] 的 inline change 標註（`(unify-billing 2026-05-28)` 散在多處）——本範式要把這些移出正文。

### 原則 6：規則正本歸屬——每條業務規則的完整定義只在一張卡，其餘卡引用不複製

把「同一條規則寫進多張卡」改為「一條規則一個正本，他處 wiki link 引用」。這是「一條規則只在一張卡寫完整、其他卡只連過去不重抄」在 Vault 的落實，也是 PM 迭代能精準定位的前提。

- **正本歸屬規則**：
  - 業務**規則**（觸發條件 + 判斷 + 動作 + 一定要成立的規則（frontmatter 欄位名 `invariant`）+ 計算公式）的完整定義 MUST 只在 `04-business-logic` 卡（規則正本）。
  - `06-state-machines` 卡只寫**狀態定義 / 轉換條件 / 觸發事件**；轉換背後「為什麼這樣分權 / 認列」的規則動機，以 wiki link 指向 business-logic 正本卡，**不重述規則本體、不寫計算公式、不放跨情境例子**。
  - `05-entities` 卡只寫**欄位 / 關聯 / 狀態名**；欄位承載的業務規則指向 business-logic 卡。
  - `03-roles` 卡只寫**該角色做什麼（職責動作）**；跨角色的分權規則細則（如補收免審 vs 退款送審的判斷準則）指向 business-logic 卡，role 卡只說「業務建立異動單、退款送主管核可」這層動作歸屬。
  - `07-scenarios` 卡只寫**跨模組端到端流程的角色傳遞與狀態鏈**；每步引用的規則指向 business-logic 卡。
- **判斷哪條規則該獨立成卡**（粒度準則，對齊「要被多處引用才獨立成卡、不先建好備用」紀律）：問「未來是否會有另一張卡 / 另一個 change / 另一條 OQ 需要單獨引用這條規則？」會 → 抽成獨立規則卡並給穩定 rule-id；只單處使用的常數（如「諮詢費寫死 2000」）留母卡，不獨立。
- **最小單位的下限**：一張規則卡至少要能被「一位角色在一個情境下獨立讀懂並執行」。一條規則的四要素（觸發條件 + 審核路徑 + 終態 + 一定要成立的規則）MUST 留在同一張卡，**不可再拆**（拆了讀者要拼多張才懂一條規則，反而失去脈絡）。
- **禁過度抽象**：對應 CLAUDE.md「禁新建抽象卡」——抽出獨立規則卡的前提是「已被多處引用」，不是「先建好備用」。
- 反例：補收 / 退款不對稱分權目前同時寫在 [[../06-state-machines/訂單異動狀態]] §一、[[../05-entities/售後服務]] L91、[[../05-entities/訂單]] OA 摘要三處（「已執行」這條一定要成立的規則三份重複定義），改造方向是建 business-logic 規則正本卡承載完整定義，三處改 wiki link 引用。

### 原則 7：完整關聯——卡內提到的每個概念都設 wiki link，雙向可達，且語意分類

把「卡內提到某概念卻只用純文字」改為「一律設 wiki link」，讓 backlink（反向連結）成為 PM 影響分析的查詢基礎設施（改 A → 立刻反查誰引用了 A），而非靠記憶。

- **提到即連**：卡內任何提到「另一個被 Vault 收錄的概念」（角色 / 實體 / 狀態機 / 規則 / 情境 / 期次 / OQ）MUST 設 wiki link。
  - 例：情境卡寫「業務會建立退款異動單」→「業務」連 [[../03-roles/業務]]、「退款異動單」連對應 business-logic / state-machine 卡，讓讀者可從任一概念跳轉。
- **雙向可達**：正本卡與引用卡互設連結（卡 A 在「相關連結」列卡 B、卡 B 也回連卡 A），讓 Obsidian graph 與 agent grep 皆能雙向追溯。user-story 卡用 frontmatter `related-*` 欄位回連（related-business-logic / related-scenarios / related-entities），避免孤立（現況 US-ORD-026/027 等 related 全空、孤立度偏高，須回填）。
- **語意分類**：連結 MUST 標明關聯方向 / 語意，不只丟一串連結。建議五類：
  | 語意類別 | 指向 | 對「本卡」的意義 |
  |---------|------|----------------|
  | 上層情境 | `07-scenarios` | 本卡的規則 / 狀態被哪個端到端情境串起 |
  | 下層步驟 | `13-user-stories` | 本卡的規則被哪些角色執行步驟（user story）落實 |
  | 相關角色 | `03-roles` | 哪些角色執行 / 把關本卡的規則 |
  | 相關規則 | `04-business-logic` | 本卡引用 / 被引用的業務規則正本 |
  | 相關狀態 | `06-state-machines` | 本卡規則對應的狀態變化 |
  | 相關實體 | `05-entities` | 本卡規則作用的資料實體 |
- **可達性的價值**：規則卡標明「上游依賴（depends-on）」與「下游被引用（affects）」雙向，使「改某規則」能沿連結查出連帶影響，把現行「靠 CLAUDE.md 長註解記憶七實體連帶矩陣」升級為「靠卡片雙向連結查詢」（對應 [[付款發票邏輯#五B、七實體連帶矩陣]]）。

## 三、術語單一正本（貫穿全部 type 卡）

主管驗證需求的前提是看得懂每個詞。

- 規則卡術語 MUST 對齊 [[../02-domain/glossary-erp|glossary]]（術語內部正本）。
- 技術名詞須以括號附註，不得當主詞（對應全域偏好「禁止中英文夾雜」與 CLAUDE.md 原則 9「術語業務友善」，如 Bubble-up → 狀態向上傳遞、聚合 → 統計、min() 聚合 → 最少工單原則）。
- 術語演進（如三實體合併後統一叫「請款期次」）先更新 glossary 正本，再同步各引用卡，避免主管讀到新舊並存的歧義。

## 四、各 type 卡章節結構範式

> 各 type 對齊 [[wiki-architecture#分層體系（營運原則 → 驗收項目，由大到細）|wiki-architecture § 分層體系]] 六層、保留既有職責（見 [[wiki-schema#11.1 各卡類型內容職責]]），但都注入「營運背景前置」。共通骨架見 § 4.1，各 type 差異見 4.2-4.7。
> **§ 4.0 是各類卡的單一職責定義，是 § 4.1-4.7 結構範式的上位約束。**
> **實際下筆時複製對應範本（§ 十清單）**：範本同時是「寫的時候照著填」和「事後照著檢查」（撰寫時照著填、事後 doc-audit / vault-audit 照著勾），本節是範本背後的範式說明。

### 4.0 各類卡的單一職責（六層由大到細、只連不重抄）

對齊 [[wiki-architecture]] 六層：每種卡單一職責，依「由大到細（營運原則 → 共用規則 → 業務規則 → 流程/狀態/角色/資料 → 操作步驟 → 驗收項目）= 由大到小（價值 → 一定要成立的規則 → 規則 → 流程/狀態/角色/實體 → 步驟 → 驗收）」相依，用 wiki link 串起，使迭代時可精準定位與影響分析。

| 分層 | type | 單一職責（一句話）| 該寫 | MUST NOT 寫（越界 → 移到哪張卡）|
|------|------|----------------|------|------------------------------|
| 營運原則（最高層）| `product-vision`（營運原則）| 公司最高**商業價值 / 分權方向**（不可驗算、Miles 拍板）| 原則陳述（一句話價值）+ 分權方向（誰把關 / 誰不被阻擋）+ 為什麼這是最上層、不再往上追 | 可驗算的規則底線（→ business-logic 共用規則）/ 具體 if-then 要件與門檻數字（→ business-logic 業務規則）|
| 共用規則 | `business-logic`（領域共用規則卡）| 某領域所有規則共用的**底線**（可驗算）| 這個領域所有規則都一定要遵守的底線（如對帳一致性「應收 = 發票淨額 = 收款淨額」）+ 服務於哪條營運原則 | 具體單點 if-then 要件（→ 業務規則）/ 跨模組流程（→ scenario）|
| 業務規則 | `business-logic`（規則正本卡）| 該情境的**規則**（什麼情況套什麼規則）| 觸發條件 + 判斷 + 動作 + 一定要成立的規則（frontmatter 欄位名 `invariant`）+ 計算公式；規則的正本 | 跨模組完整流程（→ scenario）/ 狀態轉換圖（→ state-machine）/ 角色職責清單（→ role）/ user-story 範本 / test-case 範本 |
| 流程 | `scenario` | 實際商業情境的**完整流程**（跨 user story 的端到端串接）| 跨模組角色傳遞 + 狀態鏈 + 觸發此流程的真實業務狀況 | 規則本體與計算公式（→ business-logic）/ 單一步驟細節（→ user-story）|
| 狀態 | `state-machine` | 資料層面的**狀態變化**（只描述狀態，不寫規則動機長篇）| 狀態清單 + 轉換條件 + 觸發事件 + 狀態轉換圖 | 業務規則本體 / 計算公式 / 跨情境例子 / 營運動機長篇論述（→ business-logic）|
| 角色 | `role` | 角色的**工作 / 權責**（這個角色做什麼、把關什麼）| 職責動作 + 權限 + 職責邊界 + 痛點 | 跨角色流程細節（→ scenario）/ 分權規則判斷準則細則（→ business-logic）/ 實體欄位（→ entity）|
| 資料 | `entity` | 資料實體的**欄位 / 關聯 / 狀態名** | 核心欄位 + 關鍵關聯 + 相關狀態名 | 業務流程敘述（→ business-logic / scenario）/ 狀態轉換細節（→ state-machine）/ 計算公式（→ business-logic）|
| 操作步驟 | `user-story` | role 執行 business-logic 的**步驟**（= test-case 依據）| 單一角色單一目標的業務情境（穩定層）+ UI 操作（易變層）| 跨多角色端到端流程（→ scenario）/ 規則本體（→ business-logic、以 source 引用）|
| 驗收項目 | `test-case` | 實作後的**驗收項目**（最小單位、案例之間不重疊合起來涵蓋所有情況、happy/edge）| 前置 / 步驟 / 預期結果（**正文存 Notion，Vault 不放**，user-story frontmatter `related-test-cases` 以 Notion URL 回連）| Vault 內不放 test-case 正文 / SIT / UT 技術測試 |

- **`business-logic` 橫跨共用規則與業務規則兩層**（對齊 [[wiki-architecture#business-logic 內部分層：共用規則 vs 業務規則|wiki-architecture § business-logic 內部分層]]）：靠卡的角色與內容區分，不另立 type——共用規則卡寫「這個領域所有規則都一定要遵守的底線」（可驗算，如對帳一致性），業務規則卡寫「具體 if-then 要件」（補收免審 / 退款送審 / 應收公式）；業務規則卡 `source` 可往上指共用規則卡。
- **`product-vision` 即營運原則（最高層）**：沿用既有 `product-vision` type（**不新增 type**，對齊 [[wiki-schema#一、type Enum（必填）]]：product-vision 對應 `01-products/`），正本檔 `01-products/erp/operating-principles.md`（待建）。撰寫風格見 § 4.7。

#### 4.0.1 規則卡 vs 流程卡怎麼分

兩者最易混淆，界線如下：

| 維度 | `business-logic`（規則）| `scenario`（流程）|
|------|----------------------|------------------|
| 回答的問題 | 「**什麼情況**，套**什麼規則**」（單一條件 → 動作）| 「一件事**從頭到尾怎麼走完**」（多步驟、跨時序）|
| 結構 | 規則條列，每條獨立可驗證 | 步驟串接，前一步觸發後一步 |
| 跨度 | 單一決策點（可跨實體，但聚焦一條規則）| 跨多 user story / 多模組 / 多角色 |
| 範例（訂單異動）| 「補收正項免審直達已執行」「退款負項須主管核可且綁退款款項」（各是一條規則）| 「客戶客訴 → 業務建退款異動單 → 主管核可 → 建退款款項 → 已開發票則開折讓 / 發票錯誤則作廢重開 → 三方對帳收斂」（一條完整流程，串起多條規則與多角色）|
| 判斷口訣 | 「**這是一條 if-then 規則嗎？**」是 → business-logic | 「**這是一段有先後順序、跨角色傳遞的旅程嗎？**」是 → scenario |

- **規則被流程引用，不被流程複製**：scenario 卡描述流程時，每一步「依某規則」MUST wiki link 指向 business-logic 正本卡，**不在 scenario 卡重述規則本體**。
- **同一主題會同時有兩種卡**：訂單異動既有 business-logic 規則卡（補退不對稱等 if-then 規則），也有 scenario（退款→折讓/作廢的端到端旅程）。兩者互補不重複——規則卡管「每條規則對不對」，情境卡管「整段流程串得對不對」。

### 4.6 business-logic 卡（共用規則型 / 業務規則型 / 索引型三種）

### 4.7 營運原則卡（最高層，`product-vision` type）

```

## 這張卡要回答的問題
本卡要回答的關鍵「價值取捨 / 分權方向」問題（非 if-then 規則層級）。每個問題正文有對應結論。

## 原則陳述
一句話價值 / 分權方向，MUST 不可驗算，業務語言、無實作術語當主詞。每條極少且硬。
（如「現金流出須把關、客戶加單不阻擋。」）

## 分權方向
誰把關什麼、誰不被阻擋——表達為「方向」非「規則」，不寫具體門檻數字（門檻屬業務規則）。

取捨理由先寫被駁回的對立方向 + 駁回理由，再承認被採納方向。

## 為什麼這是最上層、不再往上追
說明為何此原則是最上層依據、不再往上追（source 來自老闆拍板 + 印刷業實務，而非內部卡）。

## 相關連結 / 來源
主要往「下游被引用」連（哪些 business-logic 規則以本卡為最上層依據）；來源是最上層依據、禁指內部卡。
```

### 4.8 驗收項目卡（test-case，UAT 業務層，前提→動作→驗收結果三段）

**最具體層、正文存 Notion、Vault 只放索引與依據。** 撰寫風格對齊 [[wiki-architecture#各層怎麼寫|wiki-architecture § 各層怎麼寫]]：前提條件→做的動作→要驗收到的結果三段，`source` 往上指 user-story（操作步驟）、`implemented-by` 往下指 OpenSpec Requirement 標題層 / Prototype 端對端測試。

- **三段的正文存 Notion ERP Test Case DB，Vault 卡內不重抄**；Vault 卡承載 frontmatter（含 `source` 指 user-story + `notion-page-url`）+ 測試案例索引（happy / edge 分區 + Notion 連結）+ 相關連結。對應的 user-story 卡以 frontmatter `related-test-cases` 填 Notion URL 回連。
- 三段在 Notion 的撰寫規約（Vault 不重抄，僅供對照）：
  - **前提條件**：可觀測具體值，非抽象的一類值（「訂單 #ORD-2026-0512 狀態=訂單完成、已開發票 1000、應收 1000」，非「訂單已完成且有發票」）。
  - **做的動作**：業務動作，**禁 UI 點擊措辭**（「業務對該訂單發起補收 6000」，非「點訂單詳情頁 → 按新增訂單異動 → 選補收」）。
  - **要驗收到的結果**：可勾稽驗收值，**非「做了沒」**（「訂單異動狀態=已執行、應收由 1000 變 7000、未產生待主管審核項」，非「補收成功」）。
- happy path 與 edge case **實體分區**（兩張表分開）；案例之間不重疊、合起來涵蓋所有情況 + 明確結束點。
- **只收 UAT 業務層**，MUST NOT 放 SIT / UT 技術測試。

對應範本：[[../15-test-cases/_template-test-case]]（對應 `15-test-cases/<module>/` 目錄待建）。

## 五、Changelog 架構（獨立檔，以卡為主鍵）

> **定案（Miles 2026-05-30 拍板）**：迭代史集中於獨立檔 [[business-logic-changelog]]，正本卡正文零迭代史，只在「來源」段指向該檔對應 section。拍板條件：對 agent 搜尋而言要雙向可反查。

### 5.1 為什麼選獨立檔（而非卡底部區塊 / audit-log）

- **正文極致乾淨**：卡內零迭代史，卡不隨變更累積而變長（對應 Miles「內容不要有雜訊」核心訴求）。
- **agent 雙向可反查、且都在單一檔內 grep**：找某卡的變更 → 該卡 section；找某 change 影響哪些卡 → 全檔 grep change-id 跨 section 聚合。卡底部區塊法的弱點正是「change→卡」反查得跨整個 vault grep 所有卡。
- **不污染既有 audit-log**：audit-log 是 vault-audit / vault-insight 的稽核事件載體，商業卡迭代史另立檔，不混雜兩種用途。

### 5.2 組織與格式

- 檔案：[[business-logic-changelog]]（`00-meta/business-logic-changelog.md`，type: meta）。
- **以「卡」為主鍵分節**：每張商業邏輯卡一個 `## <卡名>` section，標題下放 `→ 正本卡：[[<卡名>]]`。
- 每節**反向時序**（最新在上），每筆三件套：**日期 +（change-id）+ 一句話變更 + 動機**。

### 5.3 卡↔log 雙向連結

- **卡 → 史**：正本卡「來源」段加一行 `迭代脈絡見 [[business-logic-changelog#<卡名>]]`（取代散在正文的 inline change 標註，正文不再留 Changelog 區塊）。
- **史 → 卡**：changelog 每節標題下 `→ 正本卡：[[<卡名>]]`。
- 兩者皆 wiki link，agent 與 Obsidian 皆可雙向跳轉。

### 5.4 與既有 audit-log / CLAUDE.md 清單的關係

- **audit-log**：仍記 skill 稽核事件（vault-audit / vault-insight），不混入商業卡迭代史；change-archive 事件不需補「影響卡」欄（「change→卡」反查由 [[business-logic-changelog]] grep change-id 達成）。
- **CLAUDE.md § Spec 規格檔清單「狀態」欄**：給工程 / 稽核的技術 changelog（exhaustive、版本級 vN 全史）。本檔是給主管驗證的精簡溯源（benefit-led，一句話變更 + 動機）。兩者分眾並存：主管讀本檔，工程查 OpenSpec change archive 與 CLAUDE.md 清單。本範式不要求改動 CLAUDE.md 清單。

## 六、營運驗證寫法 該做 / 不該做

| 維度 | 該做 | 不該做 |
|------|----|----|
| 規則陳述 | 意圖 / 規則 / 動機分欄 | 把 what 與 why 混成一段敘述 |
| 動機來源 | 連回業務情境 / 訪談 / 法規 / KPI / 拍板 OQ | 連到另一張商業邏輯卡（繞回自己證明自己 / AI 自迭代）|
| 例子 | 真實編號 + 具體金額 + 角色（「#ORD-2026-0512 退 1000」）| 占位符（demo-*）/ 抽象的一類值（「金額為負時」）|
| 例子層次 | 業務動作 / 角色 / 金額 / 狀態 | UI 措辭（按鈕 / 下拉 / 彈窗 / Tab / 點擊）|
| 例子數量 | 1-3 個關鍵例子（正例 + 邊界反例）| 窮舉所有金額組合 |
| 正文時態 | 只寫當前生效規則 | 正文寫「v1.x 推翻 v1.y」「廢止舊…」混改版歷史 |
| 改版歷史 | 移到獨立 [[business-logic-changelog]] 對應卡 section，每筆連回 change | inline 散在正文（如 `(unify-billing 2026-05-28)`）|
| 取代關係 | 移到 Changelog，正文只留新設計 | 正文留「舊→新對照表」（分期請款狀態反例）|
| 術語 | 對齊 glossary 單一正本，技術詞括號附註 | 同概念多種叫法 / 英文當主詞 |
| state-machine | 保留狀態 / 轉換 + 前置營運背景 + 每態對應營運需求 | 退化成 business-logic 卡（丟掉狀態機）/ 純系統規格無營運入口 |
| 職責邊界 | role 卡邊界附「為什麼這樣切」 | 只列邊界不附營運動機 |

## 八、新內容歸類與 change 後增修（指引引用，不複製全文）

> 「新內容該歸哪一層、掛既有卡還是新增、change 後怎麼回補」的完整方法論在 [[wiki-architecture]]，本節**只引用不複製**，避免兩份管理文件不同步。本範式（§ 一～七）管「卡正文怎麼下筆」，本節接力指向「下筆前怎麼歸類、下筆後怎麼隨 change 維護」。

### 8.1 歸類決策樹（哪一層 + 掛既有 vs 新增）

新內容進來，依 [[wiki-architecture#歸類與增修機制]] 四步判斷：

- **第 0 步守門**：是 UI 規範 / 演算法 / step-by-step Requirement → 留 DESIGN.md / OpenSpec（[[scope-boundary]]）；是不確定項 → 立即 [[oq-manage]] mode B 開獨立 OQ（禁 inline / 禁 callout）；是 AI 自編無外部來源 → 不寫（已驗證未精練進 `raw/`）。
- **第 1 步判分層**：沿「問題的抽象度」對照六層（見 § 4.0 表）。最易混的 business-logic vs scenario 用 § 4.0.1 口訣（「if-then 規則嗎」→ business-logic；「跨角色傳遞的旅程嗎」→ scenario）。
- **第 2 步判掛既有 vs 新增**：**預設先試掛既有卡**（呼應 CLAUDE.md「禁新建抽象卡」），過 8.2 三門檻才新增。
- **第 4 步後置**：`source` 往上指依據 / `implemented-by` 往下指實作（原則 2）、正本卡與引用卡雙向 wiki link（原則 7）、源自 change 則同步 changelog（§ 五）。

### 8.2 新增獨立卡 vs 掛既有卡的三道硬門檻

完整判準見 [[wiki-architecture#歸類與增修機制]]。三道硬門檻（AND，全過才新增）摘要——與本範式原則 6「規則正本歸屬」同源：

| 門檻 | 過 = 傾向新增 | 不過 = 掛既有 |
|------|--------------|--------------|
| 既有卡蓋不住 | 既有卡職責 + 規則語意讀不懂、蓋不住這條新內容 | 是既有規則的細化 / 例子 / 邊界 |
| 要被多處引用 | 未來將被另一張卡 / change / OQ 單獨引用 | 只單處使用（如寫死常數） |
| 分層 / 職責純度 | 塞進候選卡會越界（規則塞進 state-machine 等）| 同 type 同職責，可併 |

**最小單位下限**（即使過門檻也不可拆過頭）：呼應本範式原則 6「最小單位的下限」——一張卡至少能被「一位角色在一個情境下獨立讀懂並執行」；一條規則的四要素（觸發條件 + 審核路徑 + 終態 + 一定要成立的規則）MUST 留同一張卡。

### 8.3 change archive 後增修進各層（指向 wiki-amend skill）

change 結束後把 delta 增修進各層卡的**執行**，由 `wiki-amend` skill 跑（依 [[wiki-architecture#歸類與增修機制]] 六步流程：照新增・修改・刪除分類 → 改一處要連帶檢查會影響哪些卡的盤點 → 精準改到該段、不整卡重寫的單一正本回補 → 同步 changelog → 跟程式碼一起審的三層驗證 → Step 5 閉環）。本範式提供該 skill 的「卡正文怎麼寫」基準（§ 一～七 + § 4.0 越界表）。

- **觸發**：綁 `/opsx:archive` 當下（change 一歸檔當下就回補、不累積），非定期累積到 vault-audit（累積 = 「converge change 漏對齊 wiki 6 卡」的根因）。
- **執行者與稽核者分離**：偵測「回補清單」由 `doc-audit-archive` workflow 稽核 sub-agent 跑；實際增修由主對話 agent 跑 `wiki-amend`。

## 九、套用順序（給重寫 / 增修 agent）

1. 讀目標卡的 type → 對應 § 四選骨架（營運原則 product-vision / business-logic / state-machine / entity / scenario / role / user-story / test-case 各保留職責）。新內容先依 § 八歸類定分層與掛既有 vs 新增。
2. 把正文現有的 what 拆出 why → 補「營運背景」前置段 + 每規則「營運動機」。
3. 把例子改成真實格式（編號 / 金額 / 角色），刪 UI 措辭與無關細節。
4. 補依據與實作雙欄（原則 2）：`source` 往上指更上層卡 / 最上層依據（禁指同層 / 下層 / OpenSpec）、`implemented-by` 往下指 OpenSpec Requirement 標題層 / prototype / test-case；補不出 source 標 `source-gap`（暫時補不出上層依據，標記待之後補）。
5. 把 inline change 標註 / 取代關係 / 廢止舊設計，全部移到獨立 [[business-logic-changelog]] 對應卡 section（以卡為主鍵、反向時序、每筆 日期 + change-id + 一句話 + 動機，見 § 五），正文只留當前生效規則。
6. 正文「來源」段加一行「迭代脈絡見 [[business-logic-changelog#<卡名>]]」（卡→史）；changelog 該 section 標題下放「→ 正本卡：[[<卡名>]]」（史→卡）。
7. 補語意分類雙向 wiki link（原則 7）；術語對齊 glossary，技術詞括號附註。
8. 自審 § 六 該做 / 不該做 表逐列檢查（亦即對應範本末段「Lint 自檢清單」——這份範本同時是「寫的時候照著填」和「事後照著檢查」）。
