# 訂單異動流程（補收 / 退款端到端）

> 訂單成立後客戶要求變更金額（加印補收 / 退費）時，從「業務建異動單」到「三方對帳收斂」的兩條跨模組端到端旅程。本卡只描述角色傳遞與狀態鏈；每一步「依某規則」回引 [[04-business-logic/訂單異動規則]] 正本，不重述規則本體（規則對不對由規則卡驗證，本卡只驗證整段流程串得對不對）。

## 營運背景

印刷廠接單後金額常臨時變動：客戶在製作中追加印量、改急件、補運費（公司收更多錢，稱補收）；或因色差、規格不符要退費（公司退錢出去，稱退款）。這兩個方向的營運風險不對稱——退款一旦出錯難追回、補收是送上門的生意，所以走兩條不同的核可路徑。退款還牽動已開出去的發票（要開折讓單沖銷，或作廢重開），最終都要在月結時讓「應收 = 發票淨額 = 收款淨額」三方對齊，差額不為零就要會計回頭核對。本卡把這兩條旅程從頭到尾串起，讓主管驗證「整段跨部門流程是否走得通」。

## 情境清單

> 本卡承接既有索引（[[07-scenarios/README]]）的情境 8（加印追加補收）與情境 16（取消退款），並補上售後階段（情境 10 / 13）與諮詢取消（converge v1.17）兩條變體，集中呈現「訂單異動」這一主題的完整旅程。

### 旅程 A：訂單期間加印補收（情境 8）

- **觸發狀況**：訂單已成立、仍在製作中，客戶臨時加印 / 加運費 / 要求急件。
- **角色傳遞 / 狀態鏈**：
  1. **業務**（[[03-roles/業務]]）於訂單建補收正項異動單（金額 > 0），依 [[04-business-logic/訂單異動規則#R1：補收正項免審，直達已執行|補收免審規則]] 直接從「草稿」推進「已執行」（[[06-state-machines/訂單異動狀態]]）——略過待主管審核與已核可、不經 [[03-roles/業務主管]]、不綁退款款項；建立當下即認列訂單應收 +N。
  2. **業務**依 [[04-business-logic/訂單異動規則#R5：應收總額認列公式|認列不自動改期次規則]] 自行規劃這筆補收要併入哪一期請款：於對應請款期次（[[06-state-machines/分期請款狀態]]）開立發票（依 [[04-business-logic/付款發票邏輯]] §C4 線下單補收、§A2 異動補收分類）。
  3. **業務**待客戶實際付款後，登錄收款核銷分配，推進該期次收款維度（[[06-state-machines/分期請款狀態]]）。
- **具體例子**：訂單 #ORD-2026-0512 製作中，客戶把名片從 500 張加到 800 張。業務建補收異動單 +6000 元（類型：加印追加），直接成為已執行、核可人記為業務本人，應收當場 +6000；整批貨照常生產不停。業務隨後就這 +6000 開立一張發票併入尾款期次，客戶付款後核銷該期次。

### 旅程 B：訂單取消退款（情境 16）

- **觸發狀況**：訂單因故取消，且客戶已先付過款，需退費。
- **角色傳遞 / 狀態鏈**：
  1. **業務**（[[03-roles/業務]]）取消訂單（[[06-state-machines/訂單狀態]] 轉「已取消」終態），並建退款負項異動單（金額 < 0），依 [[04-business-logic/訂單異動規則#R2：退款負項須主管核可且款項到位|退款送審規則]] 送出（[[06-state-machines/訂單異動狀態]] 草稿 → 待主管審核）。對應步驟 [[13-user-stories/order-management/US-ORD-011-訂單取消與退款]]。
  2. **業務主管**（[[03-roles/業務主管]]）核可（→ 已核可）或退回（→ 已退回附原因，業務修改後重交）；依 [[04-business-logic/訂單異動規則#R2：退款負項須主管核可且款項到位|退款送審規則]]，已核可後業務仍可調整金額不需重審。
  3. **業務**依 [[04-business-logic/訂單異動規則#R4：已執行的認列與回退|退款認列規則]] 建關聯退款款項（金額 ≤ 0、綁該異動單），客戶實際退款後補對帳資料並切「已完成」；系統重算累計達異動單金額時，同一筆作業推進異動單至「已執行」、重算應收總額。
  4. **業務 / 諮詢**依 [[04-business-logic/訂單異動規則#R9：來源售後服務單標記|發票後處理規則]] 視發票狀態二選一（與異動單流程平行）：已開立且須保留發票 → 於該發票開立折讓單（金額為負、絕對值 ≤ 剩餘可折讓，依 [[04-business-logic/付款發票邏輯]] §A3、§F8）並關聯退款款項；發票開錯可作廢 → 作廢原發票（必填中文 6 字原因、字軌不可重用、流水號 +1，依 [[04-business-logic/發票法規硬約束-ezPay-MIG|ezPay 法規硬約束]]）重開。折讓 / 作廢皆不需業務主管或會計核可。
  5. **會計**（[[03-roles/會計]]）於月結對帳依 [[04-business-logic/付款發票邏輯#六、三方對帳|三方對帳規則]] 核對：應收總額（含 ∑ 已執行異動單金額）= 發票淨額（已開發票 − 已確認折讓）= 收款淨額（僅已完成款項、含退款負數）；差額為零即對帳通過，差額不為零觸發對帳警示。對應步驟 [[13-user-stories/order-management/US-ORD-013-會計執行退款處理]]。
- **系統連鎖（no-op 邊界）**：訂單取消時系統 top-down 連鎖終止下游工單 / 任務 / 生產任務（依 [[06-state-machines/訂單狀態]]）；此連鎖為系統自動行為、業務無須手動處理。
- **具體例子**：訂單 #ORD-2026-0488 因色差客訴取消，客戶已付 5000 元。業務取消訂單並建退款異動單 −5000 元送審 → 業務主管核可 → 業務實際匯款並建退款款項 −5000 切已完成，異動單推進已執行；該訂單先前已開 5000 元發票，業務於該發票開立 −5000 折讓單關聯這筆退款款項。會計月結時：應收 0（−5000 異動已執行）= 發票淨額 0（5000 − 5000 折讓）= 收款淨額 0（+5000 − 5000）→ 對帳通過。

### 旅程 C：已完成訂單售後退款（情境 10 / 13）

- **觸發狀況**：訂單已完成後客訴 / 不良 / 規格不符，無法走訂單期間的直接建單路徑。
- **角色傳遞 / 狀態鏈**：
  1. **業務**於售後服務單（[[05-entities/售後服務]]）容器內處理（受理中 → 填處理結果 → 處理中）。
  2. 若處理結果為退款，**業務**於售後單內建關聯退款異動單（標記來源售後服務單），依 [[04-business-logic/訂單異動規則#R9：來源售後服務單標記|售後退款同流程規則]] 沿用旅程 B 步驟 2-5 的同一套狀態機（[[06-state-machines/訂單異動狀態]]）與發票後處理——售後不另開例外路徑。
  3. 若處理結果為補印且責任歸屬為客戶承擔，**業務**於售後單內建補印印件（依 [[04-business-logic/打樣流程|補印跳過審稿規則]]，自動通過輪次沿用原稿）並建補收異動單（依旅程 A 補收免審規則）；公司認賠則補印免費、不建異動單（金額 0 不屬金額變動）。
- **具體例子**：訂單 #ORD-2026-0455 完成後客戶反映色偏，業務於售後服務單決議退款 3000。售後單內建退款異動單 −3000（標記來源售後服務單），走業務主管核可 → 建退款款項 −3000 → 已執行，與旅程 B 完全一致的狀態鏈。

### 旅程 D：諮詢取消退費（系統內生，converge v1.17）

- **觸發狀況**：客戶諮詢後於待諮詢階段取消，依公司規則退一半諮詢費。
- **角色傳遞 / 狀態鏈**：
  1. **諮詢人員 / 業務主管**於諮詢單（[[05-entities/諮詢單]]）取消（選取消原因分類），系統建諮詢訂單（[[06-state-machines/訂單狀態]] 推進「已取消」終態，2026-05-30 converge-consultation-cancel 收斂）+ 諮詢費其他費用（2000）+ 既有收款款項（+2000）轉移。
  2. 系統依 [[04-business-logic/訂單異動規則#R3：諮詢取消退費，系統建已核可|諮詢取消退費規則]] 自動建退費異動單（金額固定 −1000、狀態「已核可」、核可人記為系統、尚未執行），免業務主管再審。
  3. **諮詢人員**實際退款 1000、建退款款項 −1000 切已完成，依 [[04-business-logic/訂單異動規則#R4：已執行的認列與回退|退款認列規則]] 推進異動單至已執行；應收認列 −1000 後 = 2000 − 1000 = 1000。系統不自動建待開發票 / 不自動開發票 / 不自動建折讓，留存的 1000 收入由業務手動開票、未開票由對帳差額警示兜底。
- **連鎖 no-op**：諮詢訂單無印件 / 工單 / 生產任務，故旅程 B 的 top-down 連鎖對諮詢訂單為 no-op（無下游可終止）。
- **具體例子**：客戶諮詢後取消，諮詢費原收 2000。系統自動建退費異動單 −1000（已核可、核可人系統），諮詢人員退款 1000 切已完成後異動單推進已執行；留存 1000 由業務後續手動開立發票，會計對帳時已取消但有收入的此訂單仍納入差錯偵測（converge v1.17 取代「限訂單完成」）。

## 跨旅程對帳收斂（會計）

四條旅程的最終驗收都收斂到同一個三方等式（依 [[04-business-logic/付款發票邏輯#六、三方對帳|三方對帳規則]]，規則本體在規則卡，本段只描述會計的查核流程）：

- **業務 / 會計**即時對帳：應收總額（含 ∑ 已執行異動單金額）= 發票淨額（已開發票 − 已確認折讓）= 收款淨額（僅已完成款項、含退款負數），差額為零即通過。
- **跨完成日警示**：若異動單執行時間晚於訂單完成日（含售後與非售後不分桶），對帳面板顯示警示 banner 提醒會計回頭核對原月結紀錄。
- **已取消但有收入納入差錯偵測**：諮詢取消留存 1000 / 一般取消保留部分收入的已開發票訂單，皆納入差錯偵測（converge v1.17 取代限訂單完成）；已取消但未開票者由「應收 > 發票淨額」差額警示兜底。
- **會計批次對帳**：會計依帳務公司 / 開立期間 / 對帳狀態批次查詢，篩「差額 ≠ 0」的待對帳訂單，匯出對帳檔（unify-billing 定義一列 = 一張已開立發票的 14 欄格式）供藍新後台對帳。對應步驟 [[13-user-stories/order-management/US-ORD-024-會計匯出14欄對帳CSV]]。

- 售後服務單本身的受理 / 處理 / 結案進度 → [[05-entities/售後服務]]。
