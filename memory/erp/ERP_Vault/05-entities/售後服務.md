# 售後服務（AfterSalesTicket）

> 訂單**已完成後**的售後容器。取代既有 `OrderAdjustment(phase=after_completion)` 雙重身份（add-after-sales-ticket change 引入）。

## 核心欄位

| 欄位 | 說明 |
|------|------|
| `id` / `case_no` | 系統 ID / 編號 |
| `order_id` | 所屬訂單 → [[訂單]]（**強制 Order.status = 已完成**）|
| `status` | 受理中 → 處理中 → 已結案 |
| `opened_at` / `opened_by` | 開單時間 / 開單者 |
| `customer_complaint` | 客戶投訴內容 |
| `case_category` | 案件分類（7 enum）|
| `responsibility` | 責任歸屬（3 enum：公司認賠 / 客戶承擔 / 共同分擔）|
| `resolution` | 處理結果（不處理 / 補印 / 退款 / 退款+補印）|
| `slack_thread_url` | Slack 討論 thread URL |
| `closed_at` / `closed_by` | 結案時間 / 結案者 |
| `additional_complaint_log` | 後續投訴紀錄 |
| `customer_feedback_note` | 客戶反饋備註 |

## 關鍵關聯

- 屬於 `Order` → [[訂單]]（**Order 必須已完成**）
- 包含 `OrderAdjustment[]`：透過 `linked_after_sales_ticket_id` 反向關聯，0..N 個異動單
- 包含 `PrintItem[]`：補印用，透過 `related_after_sales_ticket_id`
- 包含 `Payment[]`：若決議為退款則關聯退款 Payment

## 核心邏輯

- **一個 Order 同時只能有 1 張未結案 ticket**
- **結案後可建新 ticket**
- 與訂單期間異動的區別：
  - 訂單期間：OrderAdjustment 獨立路徑（`phase=during_order`）
  - 訂單已完成後：透過 AfterSalesTicket 容器 → 內建關聯 OrderAdjustment / PrintItem

## 退款流程

售後服務單內可建立退款異動單（OrderAdjustment，adjustmentType = 退印 / 補退、預填 linked_after_sales_ticket_id）。退款的審核分權、已執行認列、一致性條件沿用規則正本 [[訂單異動規則]]（R2 退款送審 + R4 已執行綁退款款項 + R9 來源售後服務單標記）；端到端操作旅程見 [[訂單異動流程]]（旅程 C：已完成訂單售後退款）。本容器不重述規則本體（single source）。

**連帶關係**：售後退款異動會改變所屬訂單的應收總額——迭代售後退款功能時須連帶檢查 [[訂單異動規則]]，反向，異動規則變更亦影響本容器的退款行為。

## 補印流程（refine-supplementary-print-skip-review 2026-05-20）

業務 / 諮詢操作步驟：

1. ticket 內點「建立補印印件」對話框
2. 填：來源印件（從訂單既有印件選一個）+ 補印印件名稱 + 補印數量 + 補印備註
3. 提交 → 系統建立補印 PrintItem 自動完成：
   - `type = '補印印件'` + `related_after_sales_ticket_id` 寫入
   - **沿用來源印件最終合格稿件**（`reviewFiles` 完整拷貝）
   - **審稿自動通過**：新增「售後補印自動通過輪次」（ReviewRoundSource = '售後補印'、sourcePrintItemId = 來源 ID、result = '合格'）
   - `reviewStatus = '合格'`、`assignedReviewerId = null`、印製狀態 = `'待生產'`
4. 印務主管立即在「印件總覽」「等待中」分類看到此補印 → 分配工單 → 走完整生產流程

**例外情境**：補印改稿 → **不走補印路徑，走新訂單**（[AFT-6 answered](../08-open-questions/AFT-6-補印改稿情境入口設計.md)）；印務發現稿件問題 → 系統外處理（[AFT-7 answered](../08-open-questions/AFT-7-補印自動通過後稿件問題處理.md)，極端情境靠人工溝通）

詳見 OpenSpec [after-sales-ticket/spec.md § Requirement: 與 PrintItem 關聯（補印觸發）](../../../../openspec/specs/after-sales-ticket/spec.md)。

## 既有 OA 反向遷移

既有 `phase=after_completion` 歷史 OA 反向遷移為 `legacy_migrated` ticket（add-after-sales-ticket change 處理）。

## OrderAdjustment 移動

OrderAdjustment 移入 `useErpStore.orderAdjustments` 達跨頁面同步（Prototype 資料層調整）。
