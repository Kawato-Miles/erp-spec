# 售後服務（AfterSalesTicket）

> 訂單**已完成後**的售後容器。取代既有 `OrderAdjustment(phase=after_completion)` 雙重身份（add-after-sales-ticket change 引入）。

## 核心欄位

| 欄位 | 說明 |
|------|------|
| `id` / `case_no` | 系統 ID / 編號 |
| `order_id` | 所屬訂單 → [[訂單]]（**強制 Order.status = 已完成**）|
| `status` | 受理中 → 處理中 → 已結案 |
| `opened_at` / `opened_by` | 開單時間 / 開單者 |
| `customer_complaint` | 客戶投訴內容 |
| `case_category` | 案件分類（7 enum）|
| `responsibility` | 責任歸屬（3 enum：公司認賠 / 客戶承擔 / 共同分擔）|
| `resolution` | 處理結果（不處理 / 補印 / 退款 / 退款+補印）|
| `slack_thread_url` | Slack 討論 thread URL |
| `closed_at` / `closed_by` | 結案時間 / 結案者 |
| `additional_complaint_log` | 後續投訴紀錄 |
| `customer_feedback_note` | 客戶反饋備註 |

## 關鍵關聯

- 屬於 `Order` → [[訂單]]（**Order 必須已完成**）
- 包含 `OrderAdjustment[]`：透過 `linked_after_sales_ticket_id` 反向關聯，0..N 個異動單
- 包含 `PrintItem[]`：補印用，透過 `related_after_sales_ticket_id`
- 包含 `Payment[]`：若決議為退款則關聯退款 Payment

## 核心邏輯

- **一個 Order 同時只能有 1 張未結案 ticket**
- **結案後可建新 ticket**
- 與訂單期間異動的區別：
  - 訂單期間：OrderAdjustment 獨立路徑（`phase=during_order`）
  - 訂單已完成後：透過 AfterSalesTicket 容器 → 內建關聯 OrderAdjustment / PrintItem

## case_category（7 enum）

> Phase B 補完整列表（spec 未在當前萃取段提供）

## responsibility（3 enum）

- 公司認賠
- 客戶承擔
- 共同分擔

→ 影響後續是否補收費用、是否走退款

## 業務情境

詳見 [[付款發票邏輯]] § 訂單異動情境 A1-A3（訂單已完成後路徑）。

範例 Prototype 訂單：
- `AS-20260512-01`（不處理 demo）
- `AS-20260505-01`（補印免費 demo）
- `AS-20260503-01`（退款 demo）

## 既有 OA 反向遷移

既有 `phase=after_completion` 歷史 OA 反向遷移為 `legacy_migrated` ticket（add-after-sales-ticket change 處理）。

## OrderAdjustment 移動

OrderAdjustment 移入 `useErpStore.orderAdjustments` 達跨頁面同步（Prototype 資料層調整）。
