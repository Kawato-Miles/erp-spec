# 訂單異動狀態（OrderAdjustmentStatus）

[[訂單]] 成立後遇到金額增減（追加印量、加運費、急件費、退費）時，用「訂單異動單（OrderAdjustment）」記錄這筆增減；本卡只定義這張單據自己的進度狀態與轉換，獨立於主訂單。轉換背後的分權、認列、回退規則，正本在 [[訂單異動規則]]，本卡不複述。

## 營運背景

訂單異動單是一條獨立於主訂單的進度線：金額增減的審核與認列照自己的節奏走，不卡主訂單的生產與出貨。狀態之所以分補收（公司多收錢）與退款（公司退錢出去）兩種走法，是因為兩個方向的審核強度不同——這層分權的規則動機見 [[訂單異動規則]]，本卡只記「哪個狀態、什麼條件會轉到下一個狀態」。

## 狀態清單

| 狀態 | 說明 | 對應營運需求 |
|------|------|------------|
| 草稿 | [[業務]]／[[諮詢]] 建好異動單但還沒送出 | 給一個暫存空間，金額、原因填到一半可先存 |
| 待主管審核 | 業務送出後等 [[業務主管]] 核可（只有退款異動單會經過這一站）| 落實退款需主管把關，補收不經此站 |
| 已核可 | 主管已同意，等實際退款款項真的退出去 | 金額方向已背書，但錢還沒退、不能算數 |
| 已退回 | 主管退回（附原因），業務改完可重送 | 讓主管擋下不合理退款，並給業務修正機會 |
| 已執行 | 終態；這筆增減正式計入訂單應收總額 | 確認錢真的進來（補收）或退出去（退款）後才認列 |
| 已取消 | 終態；業務在草稿或已退回階段主動撤掉 | 異動單建錯或不必要時可作廢，不留無效單據 |

> 各狀態的判斷準則、認列公式、不變條件（invariant）見 [[訂單異動規則]]，本卡不重述。

## 狀態轉換

退款負項路徑（金額 < 0，業務發起）：

```
草稿 ─提交審核─▶ 待主管審核 ─核可─▶ 已核可 ─退款款項累計達金額─▶ 已執行
待主管審核 ─退回─▶ 已退回 ─修改重交─▶ 待主管審核
草稿／已退回 ─取消─▶ 已取消
已執行 ─取消已完成退款款項致累計不足─▶ 已核可（回退）
```

補收正項路徑（金額 > 0，業務發起，免主管審核）：

```
草稿 ─儲存並執行─▶ 已執行（略過待主管審核與已核可）
```

諮詢取消退費路徑（系統內生，金額固定 −1000）：

```
（系統建立）─▶ 已核可 ─退款款項累計達 −1000─▶ 已執行
```

## 轉換條件與觸發事件

| 轉換 | 觸發事件 | 條件 |
|------|---------|------|
| 草稿 → 待主管審核 | 業務點「提交審核」 | 限退款負項異動單 |
| 草稿 → 已執行 | 業務點「儲存並執行」 | 限補收正項異動單（免審）；執行時間記為當下、核可人記為業務本人、不綁款項 |
| 草稿／已退回 → 已取消 | 業務點「取消」 | — |
| 待主管審核 → 已核可 | 業務主管核可 | — |
| 待主管審核 → 已退回 | 業務主管退回（附原因）| — |
| 已退回 → 待主管審核 | 業務修改後重交 | — |
| （系統建立）→ 已核可 | 諮詢取消觸發系統建單 | 核可人記為系統、執行時間初始為空 |
| 已核可 → 已執行 | 關聯退款款項切「已完成」 | 對應款項累計（含正負號）等於異動單金額（系統自動推進、記執行時間）；累計未達則維持已核可 |
| 已執行 → 已核可（回退）| 取消某筆已完成退款款項 | 取消後累計不再等於異動單金額（系統自動回退、清執行時間）；取消後仍達金額則維持已執行；取消處理中款項則狀態不變 |

> 「為什麼補收免審、退款須核可、諮詢取消系統建已核可」的分權與認列動機，見 [[訂單異動規則]]。

## 邊界

- 主訂單在非終態時不被異動單阻擋：異動單處於非終態（如待主管審核）時，主訂單仍可由 [[工單狀態|工單]]／[[印件狀態|印件]] 層向上推進。主訂單本身的取消、終態收斂 → 走 [[訂單狀態]]，不在本卡。
- 異動單推進到已執行時會重算訂單應收總額，但不自動建立或修改 [[分期請款狀態|請款期次]]；期次的開票與收款維度狀態、期次變更稽核 → 走 [[分期請款狀態]]。請款期次變更也不會觸發本異動單或主訂單狀態回退。
- 售後服務單本身的受理、處理、結案進度 → 走 [[售後服務]]；本卡只負責售後單內建立的退款／補印收費異動單的狀態流轉。
- 認列公式、不變條件、補收／退款各情境的判定全貌 → 見 [[訂單異動規則]]（規則正本）與 [[付款發票邏輯]]（情境索引）。
