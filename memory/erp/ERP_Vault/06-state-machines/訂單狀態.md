# 訂單狀態

> [[訂單]] 從成立到結案的整體進度狀態機，分三條前段路徑（線下 / 線上 / 諮詢）再接同一段共用流程。屬跨領域的最上層訂單生命週期。狀態怎麼自動往前推、什麼時候算製作完成，背後的規則正本在 [[齊套邏輯]]；諮詢收尾的退費與終態規則正本在 [[付款發票邏輯]]，本卡只定義狀態與轉換、不複述規則。

## 這張卡要回答的問題

> 每個問題在正文都有對應結論；判斷「卡完整了沒」就看每個問題是否都答到了。

- 訂單有哪幾種狀態？哪些是收尾用的終態？
- 線下、線上、諮詢三條路徑為什麼分流？它們在哪裡會合？
- 諮詢訂單為什麼會有兩種不同的終態（訂單完成 vs 已取消）？
- 進入製作段後，狀態大多是誰在推（人工或系統自動）？條件是什麼？

## 營運背景

訂單從不同管道進來、收尾方式也不同：線下接單要先請客戶回簽報價單、線上接單要先收到款項、諮詢則是客戶先付諮詢費。三條路徑各有自己的前段，但只要進入大貨製作，後面的審稿、製作、出貨流程都一樣，所以三條路徑會合到同一段「共用流程」走到結案。

進入製作之後，整張訂單的進度大多不是人工一格一格手動改，而是隨著底下的工單、印件、生產任務做到哪裡，自動往上反映成訂單的進度——這樣業務不必盯著每張工單去手動更新訂單狀態。

## 狀態清單

> 各狀態的判斷準則、認列公式、一定要成立的規則，見 [[齊套邏輯]] 與 [[付款發票邏輯]]，本卡不重述。

線下路徑前段：

| 狀態 | 說明 | 對應營運需求 |
|------|------|------------|
| 報價待回簽 | 報價單已給客戶，等客戶回簽確認 | 線下接單要客戶白紙黑字確認報價才往下做，避免做完才有爭議 |
| 已回簽 | 客戶已回簽（上傳回簽檔或業務手動確認）| 確認單據成立後才進入製作前置流程 |

線上路徑前段：

| 狀態 | 說明 | 對應營運需求 |
|------|------|------------|
| 等待付款 | 線上訂單成立、等客戶在購物平台付款 | 線上接單先收到錢才做，不墊款 |
| 已付款 | 購物平台回報付款完成 | 確認款項到位後才進入製作前置流程 |

諮詢訂單路徑（無印件、無製作、無出貨，建立後直接收尾）：

| 狀態 | 說明 | 對應營運需求 |
|------|------|------------|
| 訂單完成 | 終態；諮詢正常收尾（不做大貨 / 需求單流失）| 有完整應收、不是取消，結算為完成才不會少算這筆收入 |
| 已取消 | 終態；客戶在諮詢階段取消（半額退費）| 沒成交的生意要算「已取消」，不能算成交、否則會灌水業績數字 |

共用流程（線下 / 線上接續，諮詢不走此段）：

| 狀態 | 說明 | 對應營運需求 |
|------|------|------------|
| 稿件未上傳 | 進入製作前置、等客戶 / 業務上傳設計稿 | 沒有稿件無法開始審稿與製作 |
| 等待審稿 | 已有稿件、等審稿人員看稿 | 印前要先確認稿件可印才下單生產 |
| 待補件 | 有印件審稿不合格，等補正確稿件 | 稿件有問題要退回重給，避免印錯整批 |
| 製作等待中 | 稿件都過了、等開立工單排生產 | 確認可印後進入生產排程 |
| 工單已交付 | 首張工單已交付給生產端 | 生產端開始接手製作 |
| 製作中 | 已有生產任務在做 | 反映實際正在生產的進度 |
| 製作完成 | 所有印件都做完 | 全部做齊才能進出貨，避免漏做某幾份 |
| 出貨中 | 已建立出貨單、貨在送 | 反映已出貨待客戶簽收 |
| 訂單完成 | 終態；所有出貨都送達 | 整張訂單收尾結案 |

## 狀態轉換

線下路徑（業務經手）：

```
報價待回簽 ─客戶回簽（上傳回簽檔或手動確認）─▶ 已回簽 ─▶ 進入共用流程（稿件未上傳）
```

線上路徑（購物平台觸發）：

```
等待付款 ─購物平台付款完成（系統自動）─▶ 已付款 ─▶ 進入共用流程（稿件未上傳）
```

諮詢訂單路徑（建立後直接收尾、不經中間態）：

```
（諮詢取消）─建立諮詢訂單─▶ 已取消
（不做大貨 / 需求單流失）─建立諮詢訂單─▶ 訂單完成
```

共用流程（線下 / 線上會合後）：

```
稿件未上傳 ─上稿─▶ 等待審稿 ─審稿不合格─▶ 待補件 ─補件完成─▶ 等待審稿
等待審稿 ─全部稿件通過─▶ 製作等待中 ─首張工單交付─▶ 工單已交付
工單已交付 ─首個生產任務開做─▶ 製作中 ─所有印件做齊─▶ 製作完成
製作完成 ─建立出貨單─▶ 出貨中 ─所有出貨送達─▶ 訂單完成
```

稿件全部合格（含免審稿）時走快速路徑：從「已回簽」或「已付款」直接跳到「製作等待中」，略過「稿件未上傳」「等待審稿」「待補件」。

## 轉換條件與觸發事件

| 轉換 | 觸發事件 | 條件 |
|------|---------|------|
| 報價待回簽 → 已回簽 | 業務上傳回簽檔，或手動點「確認回簽」 | 任一成立即推進（人工）|
| 等待付款 → 已付款 | 購物平台回報付款完成 | 系統自動 |
| 已回簽 / 已付款 → 稿件未上傳 | 進入共用流程 | 系統自動 |
| 稿件未上傳 → 等待審稿 | 客戶 / 業務上傳稿件 | 系統自動 |
| 等待審稿 ↔ 待補件 | 審稿不合格退回 / 補件完成 | 兩者可雙向互換（補件補齊後回等待審稿）|
| 等待審稿 → 製作等待中 | 所有印件審稿通過 | 系統自動 |
| 製作等待中 → 工單已交付 | 首張工單交付 | 系統自動往上反映 |
| 工單已交付 → 製作中 | 首個生產任務開始製作 | 系統自動往上反映 |
| 製作中 → 製作完成 | 所有印件製作完成 | 系統自動（所有印件都做齊了才推進，怎麼算做齊見 [[齊套邏輯]]）|
| 製作完成 → 出貨中 | 出貨單建立 | 系統自動 |
| 出貨中 → 訂單完成 | 所有出貨單送達 | 系統自動 |
| （建立諮詢訂單）→ 已取消 | 諮詢取消收尾 | 系統建立諮詢訂單即推進終態 |
| （建立諮詢訂單）→ 訂單完成 | 不做大貨 / 需求單流失收尾 | 系統建立諮詢訂單即推進終態 |

各段之間（付款段 → 審稿段 → 製作段 → 出貨段）一旦往前就不可回退；只有審稿段內的「等待審稿 ↔ 待補件」可以來回（補件補齊後回等待審稿，不算回退）。

> 「為什麼製作完成要等所有印件做齊」「諮詢收尾為什麼這樣分終態」的規則動機，見 [[齊套邏輯]] 與 [[付款發票邏輯]]。

## 關鍵狀態 / 轉換的營運動機

- 製作中 → 製作完成 → 動機：要全部印件都做齊才算完成、不能有一份漏做就出貨，否則客戶會收到缺件的訂單 → 例子：#ORD-2026-0512 下三款印件，前兩款先做完、第三款還在印，訂單仍停在「製作中」，等第三款也做完才一起轉「製作完成」。
- 諮詢取消 → 已取消（而非訂單完成）→ 動機：諮詢取消是沒談成的生意，若算成「訂單完成」會讓成交統計虛胖、業務月會數字失真 → 例子：客戶付了 2000 諮詢費後在諮詢階段喊停，系統建好諮詢訂單後直接收為「已取消」，並退一半（1000）給客戶，這筆不計入成交業績。
- 製作等待中 → 工單已交付 → 製作中 → 動機：訂單進度自動跟著底下工單與生產任務往上反映，業務不必逐張工單手動回填訂單狀態 → 例子：生產端把 #ORD-2026-0512 的第一張工單交付出去，訂單自動從「製作等待中」變「工單已交付」，第一個生產任務一開做又自動變「製作中」。

## 與其他狀態機 / 實體的關係

- 共用流程的製作段，訂單進度大多是跟著底下的 [[印件狀態|印件]] / [[工單狀態|工單]] / [[生產任務狀態|生產任務]] 做到哪裡自動往上反映——底層做到哪一步，訂單就顯示到哪一步。詳細的雙維度判定在 [[印件狀態]]，本卡只接收結果。
- 諮詢取消會連帶建立一張退費的 [[訂單異動狀態|訂單異動單]]（退費 1000），但那張單據自己的審核與認列進度走 [[訂單異動狀態]]，不影響訂單已收為「已取消」終態。
- 諮詢收尾、後續開票與收款的進度走 [[分期請款狀態]]；本卡只管訂單整體走到哪一格。

## 邊界

- 訂單成立後的金額增減（加印、補運費、退費）→ 走 [[訂單異動狀態]]，不在本卡。
- 開票與收款的進度（先收後開、頭尾款分期）→ 走 [[分期請款狀態]]，不在本卡。
- 諮詢階段本身（待諮詢、認領、結束諮詢）→ 走 [[諮詢單狀態]]；本卡只從諮詢收尾接手「諮詢訂單」這張訂單的終態。
- 製作完成的齊套判定準則、諮詢收尾退費與對帳的完整規則 → 見 [[齊套邏輯]] 與 [[付款發票邏輯]]（規則正本）。
