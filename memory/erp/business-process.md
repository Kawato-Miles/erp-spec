## 一、單據層級結構

四層單據依序串聯，上層建立後才能產生下層：

```
需求單
 └─ 訂單（含 N 個印件）
      └─ 工單（含 N 個任務）
           └─ 任務（紙本工單，含 N 個生產任務）
```

| 單據 | 核心內容 | 主要使用者 |
|------|----------|------------|
| 需求單 | 基本資料、印件規格、預計製程、報價紀錄、活動紀錄 | 業務 / 諮詢業務 |
| 訂單 | 基本資料、印件 × N（含稿件/預覽圖）、工單 × N、付款紀錄、出貨單、盈利資料、補收款訂單 | 業務、訂單管理 |
| 工單 | 任務 × N、生產任務（工序）× N、QC 紀錄 | 印務、印務主管 |
| 任務（紙本工單） | 印件、生產任務（工序）× N、報工紀錄 | 生產廠長、師傅、外包廠 |

**建立規則：**
- 審稿合格後，系統自動依印件中設定的製程與材料產生工單內容
- 若為全客製化商品，由印務自行建立工單
- 同廠商的工序歸類在同一任務中

## 二、業務流程五大階段

```
評估階段 → 打單階段 → 審稿階段 → 打樣/印製階段 → 出貨階段
```

| 階段 | 主要活動 | 主要角色 |
|------|----------|----------|
| 評估階段 | 客戶需求確認、成本評估、需求單建立、報價 | 業務、諮詢業務、製程審核、印務主管 |
| 打單階段 | 正式訂單建立（線上/線下）、款項確認 | 業務、諮詢業務、會計 |
| 審稿階段 | 稿件合規確認、Preflight 檢查、稿件退回/通過 | 審稿、審稿主管 |
| 打樣/印製階段 | 工單建立、生產任務分派、師傅執行、報工紀錄、QC 確認 | 印務主管、印務、生產廠長、師傅、外包廠、QC |
| 出貨階段 | 成品入庫、分批出貨、出貨單管理 | 出貨、業務 |

## 三、角色與資料權限

| 角色 | 主要職責 | 資料限制 |
|------|----------|----------|
| Supervisor | Dashboard、Forecast、KPI 追蹤 | 不直接編輯資料 |
| 業務 | Sales process、需求單、線下訂單、款項發票 | 只能看屬於自己或被授權的單；可轉單給其他業務/諮詢 |
| 諮詢業務 | 同業務，另含 EC 金額/資料異動客服 | 同業務限制 |
| 印務主管 | 製程審核、建立工單草稿、分派印務、追蹤工單 | BOM 表管理 |
| 印務 | 建立生產任務、工單追蹤與異動管理 | 僅可查閱印件規格；客戶資料變更需由業務/諮詢處理 |
| 生產廠長 | 工作任務安排、工單異動執行 | 以紙本為主；僅可編輯生產任務進度，不可改任務內容 |
| 師傅 | 執行生產任務、每日報工 | 以紙本為主；僅可回填進度 |
| 外包廠 / 自有廠 | 依任務報價、執行生產 | 僅可回填進度與是否接單報價 |
| 審稿 | 稿件合規確認、EC 初步客服 | 排除工單與營收資料 |
| 審稿主管 | 審稿案件分派 | 只能看基本資料與印件規格 |
| QC | 確認成品是否符合規格 | 僅可編輯 QC 紀錄 |
| 出貨 | 出貨單管理 | 訂單僅可編輯出貨單；工單僅可查閱狀態與 QC 紀錄 |
| 製程審核 | 售前成本評估、BOM 表管理、工單製程審核 | 主要為審核與 BOM 維護 |
| 會計 | 金流查帳、發票開立 | 只能看基本資料與付款紀錄 |
| 工程師 | BOM 表管理 | 主要業務為 BOM 維護 |
| 訂單管理 | 追蹤與管理全公司訂單進度 | R/W（評估、打單、印製、出貨階段） |

## 四、印件數量計算規則（核心邏輯）

### 4.1 前提假設

1. **以印件為單位**統計生產數量與出貨，不以生產任務數量計
2. **生產任務數量 ≠ 印件數量**，需經換算（如：生產 2 個壓克力才完成 1 個印件）
3. 工單建立時，可能僅製作部分部件（不含完整印件），此類工單不需出貨
4. 分批出貨：在工單尚未全部完成前即可出貨，這也是需要即時計算印件完成數量的原因

### 4.2 換算層級：逐層計算（非跨層）

```
印件（業務填寫數量）
 └─ 工單（每份印件生產數量 = 工單相對印件的比例）
      └─ 生產任務（每份工單需生產數量 = 任務相對工單的比例）
```

**關鍵設計決策：生產任務的比例參照直接上層（工單），而非印件。**
這樣每層只需了解與直接上層的關係，避免跨層換算的複雜性。

### 4.3 工單完成數量計算

```
工單完成數 = floor( min over 所有「影響成品」的生產任務 of
                   ( 生產任務 QC 通過數 ÷ 每份工單需生產數量 ) )
```

- **影響成品**：需設旗標標記，由印務建立工單時設定
- **不影響成品**（如採購紙張、開版）：不計入工單完成數計算
- 取 **floor（無條件捨去）**，零頭視為「製作中」，需在 UI 顯示以避免誤讀

### 4.4 印件完成數量計算

```
印件完成數 = floor( min over 所有工單 of
                   ( 工單完成數 ÷ 每份印件生產數量 ) )
```

### 4.5 入庫與出貨

```
入庫數量 = 生產任務 QC 通過後才計入（非生產完成即入庫）
出貨數量 ≤ 入庫數量（不可超出已入庫的成品）
```

### 4.6 數量計算範例

**情境：1式 = 1000 個專輯 + 300 本歌詞本**

| 層級 | 欄位 | 值 |
|------|------|----|
| 印件 | 數量 | 1（1式） |
| 工單A（專輯） | 每份印件生產數量 | 1000 |
| 工單A 生產任務X | 每份**工單**需生產數量 | 2 |
| 工單B（歌詞本） | 每份印件生產數量 | 300 |
| 工單B 生產任務Y | 每份**工單**需生產數量 | 1 |

計算過程（假設任務X QC通過 1200，任務Y QC通過 280）：
- 工單A完成數 = floor(1200 ÷ 2) = 600
- 工單B完成數 = floor(280 ÷ 1) = 280
- 印件完成數 = floor(min(600 ÷ 1000, 280 ÷ 300)) = floor(min(0.6, 0.933)) = floor(0.6) = **0**

（工單A尚未完成1000個目標，印件完成數為0）

### 4.7 預計生產數量

- 由**印務自行決定 buffer**，規則：預計生產數量 **不可小於**印件所需數量
- 預計生產數量設定於工單層，生產任務預計數量依工單預計數量 × 比例自動計算

### 4.8 報價單印件填寫原則

| 情境 | 建議做法 |
|------|----------|
| 同批商品確定一起出貨 | 合併為一個印件 |
| 各別商品獨立出貨 | 拆開為多個印件 |
| 業務填「1式」 | 建議轉換為實際數量（如 300 個），以利生產數量計算 |
| 打樣 vs 大貨 | 建議分開建立印件 |

## 五、打樣流程

### 5.1 工單類型

| 類型 | 說明 | 建立時機 |
|------|------|----------|
| **打樣工單** | 生產少量樣品，供業務與客戶確認成品 | 印件審稿通過後 |
| **大貨工單** | 正式大量生產 | 打樣確認 OK 後 |

**以 type 欄位區分**，兩種工單分別建立。

### 5.2 印件與工單的建立時機

- 報價單建立時，**打樣印件 + 大貨印件同時建立**（方案 A）
- 大貨印件在打樣確認前處於「等待打樣確認」狀態，不進入生產流程
- 打樣 OK 後，大貨印件才啟動後續審稿 → 工單建立流程

### 5.3 打樣決策點（業務確認後）

業務與印務討論後，共同決定重新打樣的原因，並在系統中觸發對應動作：

```
打樣完成 → 業務取得成品 → 業務 + 印務確認
    ├─ OK
    │    → 業務通知印務啟動大貨印件
    │
    ├─ 不 OK（製程問題）
    │    → 在同一打樣印件下，建立新的打樣工單
    │    → 印務調整製程設定後執行新工單
    │    → 舊打樣工單標記為「打樣不通過」保留紀錄
    │
    └─ 不 OK（稿件問題）
         → 建立新的打樣印件，走完整的印件流程
         → 新打樣印件：審稿 → 製程審核 → 打樣工單建立 → 生產
         → 原打樣印件標記為「作廢」保留紀錄
```

**重新打樣的判斷責任**：印務 + 業務共同討論決定，系統需提供明確的「觸發重新打樣」動作，要求選擇原因（製程問題 / 稿件問題）。

### 5.4 重新審稿規則

- 打樣後若需重新處理稿件，**建立全新的打樣印件**，重新跑完整印件流程
- 不修改原稿件，歷史版本保存於原打樣印件記錄中

## 六、工單分派流程

```
審稿通過
 → 印務主管建立工單草稿
 → 印務主管分派工單給印務（依負載評估）
 → 印務填寫工單詳細內容（製程、材料、生產任務）
 → 生管接手，分派生產任務給工廠 / 師傅
```

**關鍵規則**：
- 一個訂單可由**多位印務**分開負責
- 一張工單對應**一位負責印務**
- 工單異動：由印務提出需求，**印務主管執行**
