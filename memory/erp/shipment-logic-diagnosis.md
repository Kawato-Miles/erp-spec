# 出貨商業邏輯漏掉原因診斷與改善方案

> 診斷時間：2026-03-02
> 診斷對象：ERP 出貨相關討論（Spec / Prototype 設計）常見的邏輯漏洞

---

## 一、漏掉的核心商業邏輯點

根據 state-machines-ops.md § 出貨單狀態機 與 scenarios.md § 情境 7（分批出貨），以下邏輯點容易被忽略：

### 1.1 出貨單建立的時機規則及防呆條件

**漏掉的邏輯：**
- ❌ 討論時常說「打樣完成可出貨」或「大貨進行中可出貨」，**但未明確限制條件**

**實際規則（business-process.md § 5.2.5 + state-machines-ops.md）：**
```
✅ 打樣出貨：
  - 前置：打樣工單 QC 達標（累計 QC 入庫 ≥ 工單目標）
  - 可建立：任何時候，無印製狀態限制
  - 防呆：本次出貨數量 ≤ 該印件累計 QC 入庫數量 - 已出貨數量
  - 若 QC 入庫 = 0，無法建立出貨明細

✅ 大貨出貨：
  - 前置：製作進行中或製作完成皆可
  - 可建立：任何時候，無前置狀態要求
  - 防呆：本次出貨數量 ≤ 該印件累計 QC 入庫數量 - 已出貨數量
```

**影響：** 若不明確定義防呆條件，容易出現「建立出貨單成功，但無法填加出貨明細」或「出貨數量超過入庫」的邏輯矛盾。

---

### 1.2 分批出貨的累計計算邏輯

**漏掉的邏輯：**
- ❌ 說「可分批出貨」，但未驗證「多筆出貨單的累計數量如何控制」

**實際規則（state-machines-ops.md § 防呆規則 + scenarios.md 情境 7）：**
```
✅ 同一印件可出現在多筆出貨單
  - 出貨單 A：印件 X，500 個
  - 出貨單 B：印件 X，500 個
  - 合計 1000 個

✅ 累計限制（警告非強制阻擋）：
  Σ 出貨明細數量（跨所有出貨單）≤ 目標生產數量

✅ 累計送達判斷（觸發印件 → 已送達）：
  Σ 送達數量（跨所有出貨單）≥ 目標生產數量

例子：
  - 目標：1000 個
  - 出貨單 A（500 個）送達
  - 出貨單 B（500 個）仍運送中 → 印件狀態 = 出貨中（未達已送達）
  - 出貨單 B 送達 → 累計 1000 ≥ 1000 → 印件 → 已送達
```

**影響：** 若不定義「累計數量計算」與「何時判定全數送達」，容易出現：
- 印件被重複出貨超額
- 多筆出貨單進度追蹤混亂
- 「印件已送達」的判定時機錯誤

---

### 1.3 出貨單與印件的雙層狀態映射

**漏掉的邏輯：**
- ❌ 容易混淆「出貨單狀態」（未處理 → 打包 → 待出 → 出貨中 → 已送達）與「印件狀態」（製作中 → 製作完成 → 出貨中 → 已送達）

**實際規則（state-machines-ops.md § 連動規則 → 印件印製狀態）：**
```
出貨單狀態變化    →    印件印製狀態變化

出貨單 A 進入「出貨中」（自行）
    或「已出貨」（第三方）
                      →    印件 → 出貨中（從製作完成/製作中）

累計所有出貨單的「已送達」≥ 目標
                      →    印件 → 已送達

異常狀態（未送達/例外事故）
                      →    印件維持「出貨中」（不自動轉換，待業務介入）
```

**影響：** 若忽視出貨單 → 印件的映射規則，容易出現：
- Prototype 中出貨單狀態與印件狀態無法同步
- 多筆出貨單並行時，不知何時應推進印件狀態
- 異常情況處理不清（如「一筆出貨未送達，另一筆已送達」時的印件狀態）

---

### 1.4 出貨明細結構：多印件合併出貨

**漏掉的邏輯：**
- ❌ 容易認為「一筆出貨單 = 一個印件」，忽略「出貨單可包含多個印件」

**實際規則（state-machines-ops.md § 出貨明細）：**
```
出貨單 {
  type: 自行出貨 | 第三方物流
  出貨明細 [
    { 印件 ID: X, 本次出貨數量: 500 },
    { 印件 ID: Y, 本次出貨數量: 300 }
  ]
}
```

**應用場景：**
- 打樣 + 大貨同批出貨（scenarios.md 情境 1 step 16）
- 多個印件（不同工單）合併一張出貨單送達（cost 最佳化）

**影響：** 若未設計出貨明細結構，難以支持：
- 跨印件合併出貨
- 部分印件先送達，另一部分延後
- UI 中清楚顯示「本筆出貨單含哪些印件、各自數量」

---

### 1.5 一印件多工單時的出貨聚合（Open Question）

**漏掉的邏輯：**
- ❌ scenarios.md 情境 2（一印件多工單）與情境 7（分批出貨）未結合驗證

**當前狀態（待完善）：**
- 一印件有工單 A、B 兩個
- 出貨時：是否要求工單 A + B 均「製作完成」？
- 還是任何一個工單完成即可出貨其部分？

**可能規則（需確認）：**
```
? 方案 1（嚴格）
  出貨前置：所有工單均達「製作完成」
  缺點：柔性低，延遲出貨

? 方案 2（柔性）
  出貨前置：只需 QC 入庫達標
  印件完成條件：所有工單均完成 + 全數送達
  優點：支援先發後驗工序

[決策待確認]
```

**影響：** 若不明確規則，容易出現：
- 一印件多工單時出貨邏輯不一致
- Prototype 無法正確聚合 bubble-up
- 打樣後立即分批出貨時，大貨工單狀態判定混亂

## 二、為什麼容易漏掉？根本原因分析

### 原因 A：「概念層面」vs「實現層面」的鴻溝

**症狀：**
```
❌ 討論說法：「支援分批出貨」
✅ 實際需求：「同一印件可分多筆出貨單，累計數量控制、分別追蹤、獨立觸發印件狀態」
```

**原因：**
- 業務側說「可以分批」，但沒具體說明「多筆出貨單如何協調」
- 討論時跳過「細節驗證」，停留在需求陳述階段
- 缺少「反向推敲」：從「印件要如何標記已送達？」反推「出貨單須紀錄什麼？」

**改善：**
→ **進行「具體場景驗證」而非「概念背書」**

---

### 原因 B：文件結構分散，難以全面對照

**症狀：**
```
出貨建立時機 → business-process.md § 5.2.5（打樣）+ § 4.5（大貨）
出貨狀態機 → state-machines-ops.md § 出貨單狀態機
出貨防呆 → state-machines-ops.md § 防呆規則
出貨場景驗證 → scenarios.md § 情境 7
一印件多工單 → scenarios.md § 情境 2（但無出貨驗證）
```

**原因：**
- 出貨邏輯跨 3 個檔案，每份檔案講的層次不同
- 討論時往往只讀其中 1-2 份
- 缺少「出貨設計統一視圖」

**改善：**
→ **建立「出貨設計檢查清單」，強制載入所有相關段落**

---

### 原因 C：防呆規則隱含但不顯著

**症狀：**
```
state-machines-ops.md 有說：
「本次出貨數量 ≤ 累計 QC 入庫數量 - 已出貨數量」

但：
- 沒有粗體強調
- 沒有反面例子（「為什麼不能超過？」）
- 沒有驗算範例
```

**原因：**
- 文件是「規範描述」而非「教學文檔」
- 讀者容易掃過細節
- 沒有「違反此規則會發生什麼後果」的警示

**改善：**
→ **補充「反面範例」與「驗算步驟」**

---

### 原因 D：多工單情境複雜度未充分驗證

**症狀：**
```
scenarios.md 情境 1-7 涵蓋：
✅ 打樣流程（含出貨）
✅ 一印件多工單（無出貨驗證）
✅ QC 不通過補生產（無出貨驗證）
✅ 分批出貨（單工單）

❌ 缺少組合情境：
  - 一印件多工單 + 分批出貨
  - 一印件多工單 + 出貨進行中時發生補生產
  - 打樣出貨 + 大貨尚未完成時的狀態
```

**原因：**
- 情境數量已多，組合爆炸
- 優先驗證「核心路徑」，複雜邊界未全覆蓋

**改善：**
→ **補充「跨層組合情境」或「邊界假設說明」**

---

### 方案 5：補充「反面例子 + 驗算範例」（教學強化）

**更新檔案：** state-machines-ops.md § 出貨單狀態機 → 防呆規則

**新增段落：**

```markdown
## 防呆規則詳解與驗算

### 規則 1：本次出貨數量 ≤ 累計 QC 入庫 - 已出貨

**規則敘述：**
```
新出貨單的本次出貨數量 ≤ （該印件累計 QC 入庫數 - 該印件已出貨數）
```

**驗算範例：**

| 步驟 | 事件 | 累計入庫 | 已出貨 | 可出貨額度 | 結果 |
|------|------|--------|--------|-----------|------|
| 1 | QC 單 A 通過 500 | 500 | 0 | 500 | ✅ |
| 2 | 出貨單 X（500）交物流 | 500 | 500 | 0 | ⚠️ 無法再建明細 |
| 3 | QC 單 B 通過 300 | 800 | 500 | 300 | ✅ 可出貨 300 |
| 4 | 出貨單 Y（300）建立 | 800 | 800 | 0 | ✅ |
| 5 | 出貨單 X 失敗，物流退回 | 800 | 200 | 600 | ✅ 可重新出貨 600 |

**反面例子（違反規則）：**

❌ **情景 A：超出 QC 入庫**
- QC 入庫：500 個
- 嘗試出貨：600 個
- 系統防呆：拒絕，提示「可出貨額度不足」

❌ **情景 B：單筆出貨未到位卻再出貨**
- 累計入庫：1000，已出貨：0
- 出貨單 X：500 個（建立成功）
- 出貨單 Y：600 個（應失敗！）
- 原因：500 + 600 = 1100 > 1000
- 系統應警告：「Σ 出貨 ≤ 目標」

**UI 建議：**
- 建立出貨明細時，實時計算「該印件可出貨額度」
- 若輸入超過額度，禁用「保存」按鈕，顯示紅色提示

---

### 規則 2：Σ 出貨明細數量（跨所有出貨單）≤ 目標（警告）

**規則敘述：**
```
同一印件的所有出貨單的出貨數之和 ≤ 該印件目標生產數量（1000 個）
超過時：警告（非阻擋）
```

**驗算範例：**

| 出貨單 | 出貨數 | 累計 | 目標 | 狀態 |
|-------|-------|-----|------|------|
| A | 500 | 500 | 1000 | ✅ |
| B | 500 | 1000 | 1000 | ✅ |
| C | 100 | 1100 | 1000 | ⚠️ 警告 |

**意義：** 警告業務「可能過度出貨」，但允許操作（留給業務處理多餘品的判斷）
```
