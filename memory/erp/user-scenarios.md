# ERP 使用者情境（User Scenarios）

> 描述各角色在真實工作情境下的操作需求。
> 所有功能設計與 spec 需能對應到至少一個使用者情境。
>
> 最後更新：2026-02-23（v3：補充 QC / 出貨角色情境）
> 狀態：業務、審稿、印務、印務主管、訂單管理人、生管、QC、出貨 全部完成

---

## 業務

### 情境 1：建立需求單（售前評估）

**角色**：業務 / 諮詢業務
**觸發**：客戶詢問報價
**流程**：
1. 登打客戶基本資料
2. 填寫購買的印件（品項、規格、數量）
3. 與**印務主管**確認預計成本
4. 確認後可提供報價給客戶
5. 客戶確認後，將需求單**轉為訂單**

**成功條件**：需求單成功轉為訂單，訂單進入打單階段
**待確認**：需求單轉訂單時，資料是否全部帶入，還是需要業務再補填？

---

### 情境 2：訂單款項與發票管理

**角色**：業務
**觸發**：訂單成立後，依付款條件收款
**流程**：
1. 依照訂單付款條件（訂金、中期款、尾款）登記收款紀錄
2. 收款後開立發票（或由會計處理）
3. 追蹤尚未收款的款項

**成功條件**：所有款項收訖，發票開立完成
**連動角色**：會計負責金流查帳與發票開立

---

### 情境 3：確認打樣結果（打樣決策點）

**角色**：業務
**觸發**：印務完成打樣工單，打樣成品交由業務確認
**流程**：
1. 業務取得打樣成品，與客戶確認
2. **決策點 A：打樣 OK**
   - 業務通知印務可開始印製大貨
   - 進入大貨印製流程
3. **決策點 B：打樣不 OK — 需調整製程**
   - 業務 + 印務共同討論確認為製程問題
   - 在系統觸發「重新打樣（製程問題）」動作
   - 原打樣工單標記「打樣不通過」，在同一打樣印件下建立新打樣工單
   - 印務調整製程後執行新工單
4. **決策點 C：打樣不 OK — 稿件有問題**
   - 業務 + 印務共同討論確認為稿件問題
   - 在系統觸發「重新打樣（稿件問題）」動作
   - 原打樣印件標記「作廢」，建立新的打樣印件
   - 新打樣印件走完整流程：審稿 → 製程審核 → 打樣工單

**注意**：打樣印件與大貨印件在報價單建立時即同時存在；大貨印件在打樣確認前維持「等待打樣確認」狀態。

---

## 審稿

### 情境 1：審核稿件

**角色**：審稿人員
**觸發**：訂單中的印件進入審稿階段（稿件上傳後）
**流程**：
1. 依照訂單中的印件，逐一審核稿件是否合規（出血、解析度、色彩模式等）
2. **稿件合格**：標記合格，系統自動進入製程審核 → 工單建立流程
3. **稿件不合格**：退稿，通知業務 / 客戶修改

**成功條件**：稿件標記合格，進入下一階段

---

### 情境 2：打樣後需重新處理稿件

**角色**：審稿人員
**觸發**：打樣結果不 OK 且問題來自稿件（非製程）
**流程**：
1. 客戶 / 業務提供修改後的新稿件
2. **新稿件視為全新稿件**，重新跑一次完整的審稿流程（非修改原稿）
3. 審稿通過後，再進行打樣

**成功條件**：新稿件審核通過，可重新安排打樣
**注意**：每次重新審稿都是新的稿件紀錄，保留歷史版本

---

## 印務

### 情境 1：建立工單內容

**角色**：印務
**觸發**：印務主管將工單草稿分派給印務後
**前置條件**：印務主管已建立工單草稿並完成分派
**流程**：
1. 接收分派的工單草稿
2. 填寫工單詳細內容（製程、材料、生產任務規格）
3. 設定生產任務的數量與「是否影響成品」旗標
4. 工單建立完成後，生管接手分派生產任務給工廠

**成功條件**：工單內容填寫完畢，生管可接手執行
**注意**：一個訂單可由**多位印務**分開負責；一張工單對應**一位負責印務**

---

### 情境 2：追蹤與管理生產任務

**角色**：印務
**觸發**：工單進入生產中
**流程**：
1. 追蹤各生產任務的進度（透過生管 / 師傅的報工紀錄）
2. 若有異動需求（訂單需求變更、任務變更）：由印務提出，印務主管執行異動
3. 確認 QC 結果，紀錄 QC 通過數量

**成功條件**：所有生產任務完成，QC 通過數量達到目標

---

### 情境 3：打樣工單 vs 大貨工單

**角色**：印務
**說明**：工單分為兩種類型，流程不同

| 類型 | 說明 | 後續流程 |
|------|------|----------|
| **打樣工單** | 先行生產少量樣品供業務 / 客戶確認 | 打樣成品交業務確認 → 打樣 OK 後才建立大貨工單 |
| **大貨工單** | 正式大量生產 | 直接進入生產 → QC → 出貨 |

**標準流程**：報價單建立時打樣印件 + 大貨印件同時存在；打樣 OK 後大貨印件才啟動
**確認**：打樣工單與大貨工單以 type 欄位區分，分別建立

## 印務主管

### 情境 1：建立工單草稿

**角色**：印務主管
**觸發**：訂單審稿通過，進入工單建立階段
**流程**：
1. 建立工單草稿（依印件規格與製程自動帶入，或手動建立）
2. 工單草稿建立完成後，交由訂單管理人進行分配（見下方訂單管理人情境）
3. 執行工單製程審核（通過 / 退回要求調整）
4. 追蹤工單執行狀況，必要時執行工單異動

**成功條件**：工單草稿建立完成並可供訂單管理人分配
**注意**：工單分配由訂單管理人負責，印務主管不負責分配決策

---

## 訂單管理人

### 情境 1：工單分配給印務

**角色**：訂單管理人
**觸發**：印務主管完成工單草稿建立後
**前置條件**：工單草稿已由印務主管建立完成
**流程**：
1. 評估各印務目前負載與排程
2. 決定各工單的負責印務
3. 將工單草稿分派給對應的印務
4. 追蹤整體訂單進度，確認各工單均已被接手

**成功條件**：所有工單均已分派，各印務接手填寫工單詳細內容
**注意**：一個訂單內的不同工單可分派給不同印務

---

## 生管（生產廠長）

### 情境 1：接收任務並分派生產任務

**角色**：生管
**觸發**：印務完成工單內容填寫後
**流程**：
1. 接收印務建立完成的工單任務清單
2. 評估各工廠 / 師傅的產能與排程
3. 分派生產任務給對應的工廠或師傅（自有廠 / 外包廠）
4. 追蹤各生產任務進度（透過師傅每日報工紀錄）
5. 若有異動，回報印務處理

**成功條件**：所有生產任務分派完畢，工廠開始執行
**注意**：生管以紙本工單為主要作業依據；系統輔助追蹤

---

## 品管人員（QC）

### 情境 1：執行 QC 並記錄結果

**角色**：QC 人員
**觸發**：印務建立 QC 單後，指派給 QC 人員執行
**流程**：
1. 接收 QC 單，確認目標 QC 數量、抽驗項目與指派的工廠 / 師傅
2. 逐件抽驗，記錄通過數量與不通過數量
3. QC 完成後提交，系統產生 QC 紀錄並計算累計 QC 入庫數量
4. 若通過數量不足目標：印務建立補生產任務（在原任務下新增）→ 補產完成後再次建立 QC 單

**成功條件**：累計 QC 入庫數量 ≥ 工單目標生產數量
**連動角色**：印務（建立 QC 單 / 補生產任務）、生管（確認異動）
**詳細情境**：見 `scenarios.md` 情境 5

## 出貨（業務 / 印務）

### 情境 1：自行出貨

**角色**：業務 / 印務
**觸發**：印件製作完成（全部或部分），準備安排出貨
**流程**：
1. 在訂單下建立出貨單（選擇出貨類型：自行出貨；填寫收件人 / 地址）
2. 新增出貨明細（指定印件 + 本次出貨數量；系統防呆：數量 ≤ 可出貨餘量）
3. 備料打包完成，更新出貨單狀態至「待出貨」
4. 車輛出發，更新至「出貨中」→ 系統自動觸發印件印製狀態 = 出貨中
5. 確認送達，手動更新至「已送達」
6. 系統計算：印件累計已送達數量 ≥ 目標 → 印件印製狀態 → 已送達

**成功條件**：出貨單狀態到達「已送達」，印件印製狀態更新完成
**注意**：一張出貨單可包含多個印件（合併出貨）；同一印件也可分批建立多筆出貨單；UI 顯示已出 X / 目標 Y

---

### 情境 2：第三方物流出貨

**角色**：業務
**觸發**：同情境 1，但改由物流商配送
**流程**：
1. 在訂單下建立出貨單，選擇出貨類型：**第三方物流**；新增出貨明細
2. 備料打包完成，更新狀態至「待出貨」
3. 將貨物移交物流商，手動更新至「已出貨」→ 系統觸發印件印製狀態 = 出貨中
4. 物流商確認收件後，系統（API）自動更新至「運送中」
5. 物流商確認送達，系統（API）自動更新至「已送達」
6. 若發生異常（未送達 / 例外事故）：業務介入處理，出貨單維持異常狀態直到解決

**成功條件**：出貨單狀態到達「已送達」
**連動**：第三方物流狀態更新目前為手動模擬；API mapping 待實作（`open-questions.md` #28）
**詳細情境**：見 `scenarios.md` 情境 7
