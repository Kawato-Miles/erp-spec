## 任務（紙本工單）狀態機

> 任務是派發給各工廠的執行單位，每個任務含 1～N 筆生產任務。狀態依工廠類型有不同的終態路徑。

**角色對應：**
- **印務**：建立任務、交付給工廠
- **生管**：確認異動、追蹤生產進度
- **各工廠師傅 / 廠商**：執行生產

### 狀態一覽

| 狀態 | 適用工廠 | 類型 | 說明 |
|------|---------|------|------|
| 待交付 | 全部 | 初始 | 待印務提供工單給工廠 |
| 已作廢 | 全部 | 終止終態 | 取消，尚未交付前 |
| 已交付 | 全部 | 進行中 | 印務已提供工單 |
| 製作中 | 全部 | 進行中 | 任一生產任務進入製作中（由生產任務層 bubble-up） |
| 異動 | 全部 | 可逆 | 印務發起異動，待生管確認 |
| 已確認異動內容 | 全部 | 可逆 | 生管確認，需重新印製工單後繼續 |
| 運送中 | 外包廠 / 中國廠商 | 進行中 | 生產完成後運送回場 |
| 已送集運商 | 中國廠商 | 進行中 | 已送達中國集運商 |
| 已抵達 | 外包廠 / 中國廠商 | 進行中 | 材料抵達場廠 |
| 已完成 | 全部 | 正常終態 | 此廠商的工序結束，材料已回場 |

### 狀態流程

#### 主流程

```
待交付（待印務提供工單給工廠）
 → 已作廢（取消，尚未交付前）
 → 已交付（印務已提供工單）
 → 製作中（任一生產任務進入製作中時，由生產任務層 bubble-up）
   ↔ 異動（印務發起異動工單）
     → 已確認異動內容（生管確認後重新印製工單）
     → 回到製作中
 → [依工廠類型分流，見下]
```

> bubble-up 鏈：生產任務「製作中」→ 任務「製作中」→ 工單「製作中」→ 印件「製作中」→ 訂單「製作中」
> 任何一層「任一子項目製作中」即觸發上層進入製作中。

#### 終態路徑（依工廠類型）

| 工廠類型 | 製作中後的狀態路徑 |
|----------|-------------------|
| 自有工廠 / 加工廠 | → 已完成 |
| 外包廠 | → 運送中 → 已抵達 → 已完成 |
| 中國廠商 | → 已送集運商 → 運送中 → 已抵達 → 已完成 |

### 狀態轉換

| 從 | 觸發條件 | 到 | 執行角色 | 手動/自動 |
|----|---------|----|---------|-----|
| 待交付 | 尚未交付即取消 | 已作廢 | 印務 / 業務 | 手動 |
| 待交付 | 印務提供工單給工廠 | 已交付 | 印務 | 手動 |
| 已交付 | 任一生產任務進入製作中（bubble-up） | 製作中 | 系統 | 自動 |
| 製作中 | 印務發起異動 | 異動 | 印務 | 手動 |
| 異動 | 生管確認異動內容 | 已確認異動內容 | 生管 | 手動 |
| 已確認異動內容 | 重新印製工單後繼續製作 | 製作中 | 生管 | 手動 |
| 製作中（自有 / 加工廠） | 任務內所有生產任務達「已完成」（系統自動聚合） | 已完成 | 系統 | 自動 |
| 製作中（外包廠） | 生產完成，交物流 | 運送中 | 外包廠 | 手動 |
| 製作中（中國廠商） | 送達集運商 | 已送集運商 | 廠商 | 手動 |
| 已送集運商 | 集運商發出貨 | 運送中 | 廠商 / 系統 | 手動 |
| 運送中 | 材料抵達場廠 | 已抵達 | 生管 | 手動 |
| 已抵達 | 生管確認驗收 | 已完成 | 生管 | 手動 |

### 設計決策

> 「已完成」代表此廠商的工序結束，材料已回場。後續場內加工為另一筆獨立的生產任務（自有工廠執行），兩者間存在任務相依性。
>
> **相依性機制（已確認）：採手動 + UI 排序。** 生管手動決定後續任務的啟動時機，不自動觸發。UI 設計階段應提供任務排序 / 相依標記介面，方便生管追蹤前置任務完成情形。（決策依據：scenarios.md 情境 4 附錄）

#### 【新增】任務異動流程的邏輯與狀態轉換

**異動的觸發場景**：

1. **QC 不通過**：某生產任務 QC 失敗，需補建新生產任務重新製作
2. **製程異動**：客戶或印務要求調整製程，需增加 / 修改生產任務

**異動流程的角色分工**：

| 角色 | 責任 | 操作 |
|------|------|------|
| **印務** | 發起異動請求、新增或修改生產任務 | 任務「製作中」→「異動」 |
| **生管** | 確認異動內容、決定如何推進 | 「異動」→「已確認異動內容」；決定返回「製作中」時機 |

**狀態轉換流程**：

```
任務「製作中」
  ↓ （印務發起異動，如 QC 失敗或製程調整）
「異動」（暫停生產推進，等生管確認）
  ├─ 生管審視異動內容（新增生產任務、修改倍數等）
  ├─ 生管確認無誤，觸發「已確認異動內容」
  ↓
「已確認異動內容」（異動已確認，待回到製作中）
  ├─ 新生產任務已建立 / 倍數已調整
  ├─ 印務重新印製紙本工單（若為外包廠 / 中國廠商）
  ├─ 生管確認準備好，觸發返回「製作中」
  ↓
「製作中」（繼續原流程）
```

**異動期間的計算影響**：

- 任務在「異動」或「已確認異動內容」狀態時，不參與工單層完成數計算
- 生管返回「製作中」後，系統重新計算工單層完成數（考慮新增生產任務）
- 新增生產任務的 `quantity_per_work_order` 設定：
  - 建議值：參照原同工廠任務的平均倍數（系統提供建議）
  - 可覆寫：印務主管或生管確認製程變化後，可手動設定不同倍數
  - 詳見 `docs/quantity-calculation-rules.md § 4.4` 異動流程中倍數調整的完整驗算

**異動流程的防呆規則**：

1. **禁止推進至「已完成」**：
   - 任務在「異動」或「已確認異動內容」狀態時，不得直接轉為「已完成」
   - 必須先返回「製作中」，後續由自動聚合邏輯判定是否達成完成條件

2. **修改「影響成品」標誌時重新計算**：
   - 若異動流程中修改生產任務的 `affects_product` 標誌，系統提示：
     「該修改將重新計算工單完成數，可能影響出貨進度。請確認業務邏輯。」
   - 修改後需重新驗算工單層的完成倍數（見 quantity-calculation-rules.md § 4.5）

3. **異動期間的出貨保護**：
   - 任務在「異動」或「已確認異動內容」期間，該工單的出貨仍受限於已累計 QC 入庫數量
   - 不因異動而凍結出貨，但異動後重新計算可能改變印件完成數（若新生產任務導致倍數提高）

**UI / UX 設計建議**：

- 生管操作介面應清楚顯示：
  - 異動前後的生產任務對比（新增、修改、刪除）
  - 新生產任務的 `quantity_per_work_order` 建議值及其計算依據
  - 修改前後的工單完成數預測（若可行）
- 打印工單時，系統提示是否為「異動後重新打印」，標記版次號
- 生管確認返回「製作中」時，系統自動計算並顯示更新後的工單完成倍數

**與情境的對應**：

- 異動觸發：scenarios.md 情境 5（QC 失敗 → 補生產）
- 角色責任與狀態轉換驗算：scenarios.md 情境 5 的驗證重點
- 測試案例：test-cases.md TC-W004（任務異動完成），TC-W005（異動期間工單計算）

## 生產任務狀態機

> 生產任務是最小追蹤單位，用於計算報工完成數量。工廠類型於建立時由 BOM 參數設定決定，決定對應狀態路徑。
> 三種工廠類型共用一份狀態機；工廠類型控制可用的轉換路徑，UI 僅顯示對應路徑的操作按鈕。

**角色對應：**
- **生管**：分派生產任務、確認作廢與報廢
- **師傅 / 工廠**：報工（執行生產，填報完成數量）

### 狀態一覽

| 狀態 | 適用工廠 | 類型 | 說明 |
|------|---------|------|------|
| 待處理 | 全部 | 初始 | 生產任務建立，等待開始 |
| 已作廢 | 全部 | 終止終態 | 尚未進入生產即取消 |
| 製作中 | 全部 | 進行中 | 生產開始（bubble-up 觸發任務 → 工單 → 印件 → 訂單製作中） |
| 報廢 | 全部 | 終止終態 | 已進入生產後取消；費用仍以報工完成數量計算 |
| 已送集運商 | 中國廠商 | 進行中 | 已送達中國集運商 |
| 運送中 | 外包廠 / 中國廠商 | 進行中 | 生產完成後運送 |
| 已完成 | 全部 | 正常終態 | 所有報工完成數量 = 任務數量 |

### 狀態流程

```
待處理
 → 已作廢（尚未進入生產即取消）
 → 製作中（生產開始）
   → 已完成（自有工廠 / 加工廠；條件：所有報工完成數量 = 任務數量）
   → 運送中（外包廠；完成生產後交付物流）→ 已完成
   → 已送集運商 → 運送中（中國廠商）→ 已完成
   → 報廢（已進入生產但取消；以報工完成數量計算成本）
```

**工廠類型對應狀態路徑：**

| 工廠類型 | 路徑 |
|----------|------|
| 自有工廠 / 加工廠 | 製作中 → 已完成 |
| 外包廠 | 製作中 → 運送中 → 已完成 |
| 中國廠商 | 製作中 → 已送集運商 → 運送中 → 已完成 |

### 狀態轉換

| 從 | 觸發條件 | 到 | 執行角色 | 手動/自動 |
|----|---------|----|---------|-----|
| 待處理 | 尚未開始即取消 | 已作廢 | 生管 | 手動 |
| 待處理 | 師傅 / 工廠開始生產 | 製作中 | 師傅 / 工廠 | 手動 |
| 製作中 | 已進入生產後取消 | 報廢 | 生管 | 手動 |
| 製作中（自有 / 加工廠） | 報工完成數量 = 任務數量 | 已完成 | 師傅 | 手動 |
| 製作中（外包廠） | 生產完成，交物流 | 運送中 | 外包廠 | 手動 |
| 製作中（中國廠商） | 送達集運商 | 已送集運商 | 廠商 | 手動 |
| 已送集運商 | 集運商發出貨 | 運送中 | 廠商 / 系統 | 手動 |
| 運送中（外包廠） | 材料抵達 | 已完成 | 生管 | 手動 |
| 運送中（中國廠商） | 材料抵達 | 已完成 | 生管 | 手動 |

### 設計決策

> 生產任務 已完成 = 純生產進度達標。QC 另由 QC 紀錄追蹤（見下方），不在生產任務層設置 QC 狀態。

## QC 相關實體

### QC 單狀態機

由印務建立，指定本次核查的批次數量，指派 QC 人員執行盤點。

> 印件可分批出貨，因此 QC 也分批進行：每次 QC 單只針對本次印務指定的批次數量。

**角色對應：**
- **印務**：建立 QC 單，設定本次批次目標數量與指派對象
- **QC 人員**：執行盤點，填寫結果

#### 狀態一覽（QC 單）

| 狀態 | 類型 | 說明 |
|------|------|------|
| 待執行 | 初始 | 印務建立，指定任務 + 本次目標數量 + QC 人員 |
| 執行中 | 進行中 | QC 人員開始盤點 |
| 已完成 | 正常終態 | 本次指定批次核查完畢，通過 / 不通過數量已填入 |

#### 狀態流程（QC 單）

```
待執行（印務建立，指定任務、本次目標 QC 數量、指派 QC 人員）
 → 執行中（QC 人員開始執行盤點）
 → 已完成（QC 人員完成本次指定批次的核查，填入各生產任務通過 / 不通過數量）
```

> **QC 單完成條件 = 本次指定核查數量盤點完畢**，不是「缺口 = 0」。
> 缺口判斷在工單層計算（見下方）；QC 單本身完成後即關閉，後續批次另開新 QC 單。

#### 狀態轉換（QC 單）

| 從 | 觸發條件 | 到 | 執行角色 | 手動/自動 |
|----|---------|----|---------|----|
| — | 印務建立 QC 單 | 待執行 | 印務 | 手動 |
| 待執行 | QC 人員開始盤點 | 執行中 | QC 人員 | 手動 |
| 執行中 | 完成本次批次核查，填入通過 / 不通過數量 | 已完成 | QC 人員 | 手動 |

#### 工單層計算（QC 單完成後觸發）

```
系統計算（每次 QC 單完成後更新）：
  累計 QC 入庫數量 = Σ 所有 QC 單的通過數量

若 累計入庫 ≥ 工單目標生產數量：
  工單 → 已完成

若 累計入庫 < 工單目標生產數量：
  工單維持「製作中」，系統顯示缺口數量（= 目標 - 累計入庫）
  印務可視情況：建立新任務補足（補生產），或等待下一批 QC
```

### QC 紀錄（資料實體）

QC 單完成後產生，掛在任務下，記錄各「影響成品」生產任務的 QC 結果。

```
QC 紀錄 {
  任務 ID
  QC 單 ID
  明細（per 影響成品的生產任務）:
    - 生產任務 ID
    - 生產數量
    - QC 通過數量（入庫數量）
    - QC 不通過數量
}
```

**計算邏輯：**
- 不影響成品的生產任務（如採購紙張、開版）不建 QC 紀錄，不計入工單完成判斷
- 工單完成數量計算公式沿用 business-process.md Section 4.3，QC 通過數量來源改為 QC 紀錄
- 印件出貨條件：出貨數量 ≤ 累計 QC 入庫數量

**QC 完成判斷（層級傳遞，由下往上）：**
```
生產任務：QC 通過數 ≥ 生產任務目標 → 該生產任務 QC 達標

任務：所有「影響成品」的生產任務 QC 達標 → 該任務 QC 達標

工單 已完成：
  累計所有任務之 QC 入庫數量 ≥ 工單目標生產數量
  （等同：所有任務 QC 達標，且累計數量足夠）
```

## 出貨單狀態機

出貨單掛在**訂單**下，追蹤每一次出貨的物流狀態。一張出貨單可包含多個印件（合併出貨），亦可只包含單一印件；一個印件也可出現在多筆出貨單中（分批出貨）。

**出貨類型（type）：** `自行出貨 / 第三方物流`
- 自行出貨：業務 / 印務手動逐步更新狀態
- 第三方物流：物流商 API 自動更新（API mapping 細節待規劃；本節著重資料流設計）

**出貨明細（Shipment Line Items）：**
每張出貨單包含一或多筆出貨明細，每筆對應一個印件 + 本次出貨數量。
```
出貨明細 {
  印件 ID
  本次出貨數量
}
```
**防呆：** 每個印件的 `Σ 出貨數量（所有出貨單）≤ 目標生產數量`；`本次出貨數量 ≤ 累計 QC 入庫數量 - 已出貨數量`

**建立時機：** 訂單下只要有任一印件完成「製作完成」且 QC 入庫數量 > 0，業務即可建立出貨單

### 狀態一覽

| 狀態 | 適用路徑 | 類型 | 說明 |
|------|---------|------|------|
| 未處理 | 兩者 | 初始 | 出貨單建立後，尚未開始作業 |
| 打包中 | 兩者 | 進行中 | 開始備料、打包作業 |
| 待出貨 | 兩者 | 進行中 | 打包完成，等待出車或物流商取件 |
| 出貨中 | 自行出貨 | 進行中 | 我方車輛已出車，配送途中 |
| 已出貨 | 第三方物流 | 進行中 | 已移交物流商，等待追蹤更新 |
| 運送中 | 第三方物流 | 進行中 | 物流商確認收件，追蹤進行中（API 更新）|
| 已送達 | 兩者 | 終態 | 確認送達客戶 |
| 未送達 | 第三方物流 | 異常 | 嘗試送達失敗，需重新安排 |
| 例外事故 | 第三方物流 | 異常 | 遺失、損毀等運送異常，需業務介入 |
| 無追蹤中包裹 | 第三方物流 | 異常 | 物流追蹤資料失效，需業務介入 |
| 過期 | 第三方物流 | 異常 | 追蹤期限屆滿仍未送達，需業務介入 |
| 已作廢 | 兩者 | 終止終態 | 出貨單取消（限未出車 / 未交物流商前）|

> **「出貨中」vs「已出貨」語意區分**：「出貨中」= 我方車輛在配送途中（自行出貨）；「已出貨」= 貨物已移交物流商手中（第三方物流）

### 狀態流程

#### 自行出貨
```
未處理 → 打包中 → 待出貨 → 出貨中 → 已送達
```
> 中間步驟可由業務 / 印務手動跳過（如直接從「未處理」進入「待出貨」）

#### 第三方物流
```
未處理 → 打包中 → 待出貨 → 已出貨 → 運送中 → 已送達

[異常分支]
運送中 → 未送達（重新安排）→ 出貨中（自行補送）或 運送中（物流商再次嘗試）
運送中 / 已出貨 → 例外事故 / 無追蹤中包裹 / 過期（業務介入處理）
```

#### 共用終止
```
任意進行中狀態（未出車 / 未交物流商前）→ 已作廢
```

### 狀態轉換

| 從 | 觸發條件 | 到 | 執行角色 | 手動/自動 |
|----|---------|----|---------|----|
| — | 業務建立出貨單 | 未處理 | 業務 | 手動 |
| 未處理 | 開始備料打包 | 打包中 | 業務 / 印務 | 手動 |
| 打包中 | 打包完成 | 待出貨 | 業務 / 印務 | 手動 |
| 待出貨 | 出車（自行出貨）| 出貨中 | 業務 / 印務 | 手動 |
| 待出貨 | 移交物流商（第三方）| 已出貨 | 業務 | 手動 |
| 出貨中 | 確認送達（自行出貨）| 已送達 | 業務 | 手動 |
| 已出貨 | 物流商確認收件 | 運送中 | 系統（API）| 自動 |
| 運送中 | 物流商確認送達 | 已送達 | 系統（API）| 自動 |
| 運送中 | 送達失敗 | 未送達 | 系統（API）| 自動 |
| 未送達 | 重新安排出貨 | 出貨中 / 運送中 | 業務 / 系統 | 手動 / 自動 |
| 運送中 / 已出貨 | 物流異常通報 | 例外事故 | 系統（API）| 自動 |
| 運送中 / 已出貨 | 追蹤資料失效 | 無追蹤中包裹 | 系統（API）| 自動 |
| 運送中 / 已出貨 | 追蹤期限屆滿 | 過期 | 系統（API）| 自動 |
| 任意（未出車 / 未交物流商前）| 業務取消出貨 | 已作廢 | 業務 | 手動 |

> ⚠️ API 觸發狀態為預設設計；實際以物流商 webhook 規格為準，API mapping 細節另行規劃。

### 連動規則（出貨單 → 印件 → 訂單）

#### 出貨單 → 印件印製狀態

| 出貨單觸發 | 印件印製狀態變化 | 說明 |
|-----------|----------------|------|
| 含該印件的第一筆出貨單進入「出貨中」（自行）或「已出貨」（第三方）| 製作完成 → **出貨中** | 物品已實際開始出貨 |
| 累計已送達數量（跨所有出貨單之該印件明細）≥ 目標生產數量 | 出貨中 → **已送達** | 全數確認送達 |
| 出貨單進入「未送達 / 例外事故 / 無追蹤中包裹 / 過期」| 維持 **出貨中** | 業務介入處理，印件不自動轉換 |

#### 出貨單 → 訂單狀態（bubble-up via 印件）

| 觸發 | 訂單狀態 | 說明 |
|------|---------|------|
| 任一印件進入「出貨中」| 製作完成 → **出貨中** | 開始出貨，bubble-up |
| 所有印件進入「已送達」| 出貨中 → **訂單完成** | 全部印件全數送達 |

> 訂單層僅保留「出貨中」一個摘要狀態（合併原本的 出貨未處理 / 打包中 / 物流未完成 三個細粒度狀態），出貨細節改由出貨單管理。

### 設計決策

- **出貨單掛訂單層**：允許跨印件合併出貨；每筆出貨單透過「出貨明細」記錄各印件本次出貨數量
- **合併 vs 分開出貨**：業務依實際作業決定；系統不強制；UI 可提供「全部印件合併」快捷選項
- **分批出貨**：同一印件可出現在多筆出貨單；印件層以「累計已送達數量（跨所有出貨單）」判斷是否達目標
- **防呆邏輯**：建立出貨明細時，系統驗證「本次出貨數量 ≤ 該印件累計 QC 入庫數量 - 已出貨數量」
- **路徑分流**：自行出貨與第三方物流共用初始（未處理）與終態（已送達 / 已作廢）；中間段依 type 分岐，避免狀態混用
- **異常不自動推進**：未送達 / 例外事故等異常狀態，印件與訂單維持現況，業務需主動介入（重新出貨 or 填寫損毀報告）後才能推進
- **作廢限制**：已出車（出貨中）或已交物流商（已出貨）後不得直接作廢；運送中異常請走「例外事故」流程
- **API 設計原則**：第三方物流的進行中狀態（運送中 / 已送達 / 未送達等）預設由 API 自動更新；業務仍保有手動覆寫能力（如 API 未回調時手動確認送達）
