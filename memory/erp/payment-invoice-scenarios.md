# 訂單款項與發票業務情境（Notion 13 情境）

**來源**：[Notion follow-up「提供收款情境」](https://www.notion.so/3573886511fa80b39093d8c76b57737a)
**整理**：2026-05-12（refactor-order-adjustment-and-cleanup change archive 隨附）
**用途**：後續產生 user story / test case 推送至 Notion 給外部確認的原始材料

13 個情境分類：
- 特殊發票開立方式 8 個（F1-F8；F9 跨訂單混合匯款屬人工流程不做）
- 特殊收款方式 4 個（C1-C4）
- 訂單異動情境 3 個（A1-A3；A4 重新報價屬需求單議價）

每個情境結構：業務情境（Notion 原文）→ 實作解決方式 → 驗證重點 → Prototype 對應訂單。

## 特殊發票開立方式

### 1. 預開發票 + 拆 2 筆發票開立 + 實際製作金額調整

**業務情境**（Notion 原文）：
比較特殊的案例，幾乎都是學校或政府單位，需分開開立發票：
- 預先開發票請款（走內部核銷流程）
- 開立發票單筆不能超過限額
- 預開金額（如 15,000）→ 拆 2 張各 7,500 含稅
- 實際製作金額低於預估（如 9,135），需從預先支付款項退款（5,865）

**實作解決方式**：
1. 訂單建立後業務記錄預收款（Payment，可選不關聯 PaymentPlan）
2. 業務開立 N 張預開發票（自由填入金額），各張獨立但加總等於預收
3. 訂單完成後實際金額若低於預估：
   - 建立售後服務單（OrderAdjustment phase = after_completion，金額為差額負數）
   - 業務於發票區建立退款 Payment（負數）
   - 開立折讓單（SalesAllowance）關聯預開發票，refund_payment_id 連結退款 Payment
4. 對帳：應收 = 發票淨額 = 收款淨額（差額對齊）+ 對帳警示 banner

**驗證重點**：
- 預付款可記錄為「無 PaymentPlan 關聯」的 Payment
- 拆發票時各張獨立金額，加總等於預收
- 售後服務單 phase 自動為 after_completion，type 範圍縮限（不可加印 / 加運費 / 急件費）
- 折讓單關聯預開發票，發票剩餘可折讓金額正確扣減
- 退款 Payment 不入 PaymentInvoice junction，透過 SalesAllowance.refundPaymentId 連結
- 三方對帳差額 = 0
- 對帳警示 banner 顯示「訂單已於 [日期] 完成，異動於 [日期] 執行」

**Prototype 對應訂單**：`ORD-20260116-05`（中央大學系所招生簡介，預開 15K → 實做 9.1K → 退 5.9K）

### 2. 拆多筆指定金額發票開立

**業務情境**：客戶要求一筆訂單金額拆成多張不同金額的發票。

**實作解決方式**：
1. 訂單建立、業務收款
2. 業務於發票區開立 N 張發票，各張金額由業務自由填入
3. PaymentInvoice junction 自由勾選一筆 Payment 對應多張 Invoice 各自分攤金額

**驗證重點**：
- 開立發票時不強制金額等於訂單應收（業務自由填）
- 一筆 Payment 可分對應多張 Invoice（N:M）
- 多張 Invoice 加總 = 應收，差額為零
- 各張 Invoice 金額獨立，互不影響字軌

**Prototype 對應訂單**：`ORD-20260120-01`（誠品年度活動小冊 78K → 2,500 + 75,500）

---

### 3. 拆多筆有統編 / 無統編發票開立

**業務情境**：
- 同訂單客戶要求拆成兩張發票
- 一張有統編（B2B）+ 一張無統編（B2C）
- 部分客戶為個人但有公司同事報帳需要分開

**實作解決方式**：
1. 業務開立 INV-01 設 category = B2B、填入買方統編
2. 業務開立 INV-02 設 category = B2C、無統編
3. 兩張 Invoice 加總 = 訂單應收

**驗證重點**：
- Invoice.category 可分別設 B2B / B2C
- B2B 必填 buyerUbn 並驗證 8 碼數字
- B2C 不需 buyerUbn
- 兩張 Invoice 共用同一張 Payment（透過 PaymentInvoice junction 拆金額）
- 三方對帳對齊

**Prototype 對應訂單**：`ORD-20260125-03`（陳○○婚禮邀請卡，B2B 工作室 15K + B2C 個人 10K）

### 4. 指定發票品名 / 數量 + 作廢重開發票

**業務情境**：
- 系統預設以訂單印件名稱開立發票品名
- 部分客戶會「指定品名」（如「印刷品一批」）會計報帳用
- 需作廢原發票、重開正確品名

**實作解決方式**：
1. 業務開立 INV-01 系統預設品名（印件名稱）
2. 客戶要求改品名 → 業務作廢 INV-01（填入作廢原因）
3. 業務重開 INV-02 品名為客戶指定（如「印刷品一批」）
4. INV-01 字軌號碼不重用，INV-02 流水號 +1

**驗證重點**：
- 作廢發票需填入原因（中文 6 字內 / 英文 20 字內，對應藍新 InvalidReason 限制）
- 作廢後 Invoice.status = 作廢，invalidAt / invalidBy 寫入
- 重開時 ezpayMerchantOrderNo 流水號 +1
- 三方對帳：作廢 Invoice 不計入發票淨額
- 業務可自由填入 Invoice.items（品名 / 數量 / 單價）

**Prototype 對應訂單**：`ORD-20260205-08`（陳董科技年度報告書，INV-01「年度報告書」作廢 → INV-02「印刷品一批」）

### 5. 指定日期開立發票

**業務情境**：
- 客戶（特別是政府單位 / 大企業）需指定發票開立日期
- 需提前告知財務避免錯過該月開票批次
- 與「分多月開立」（情境 6）相關但不同

**實作解決方式**：
1. 訂單成立、收款完成
2. 業務於 PaymentPlan 填入 expected_invoice_date 作為排程提醒
3. 業務於指定日當天手動開立發票（spec 已決定不做自動排程）
4. 期間「待開發票模組」會列出此訂單（待 Change B 上線後驗證）

**驗證重點**：
- PaymentPlan.expected_invoice_date 可選填
- 訂單已收款但未開立發票時，對帳檢視顯示「待開發票」差額
- 業務於指定日手動開立發票，issued_at 為當天系統時間
- 待開發票模組以 expected_invoice_date 排序

**Prototype 對應訂單**：`ORD-20260210-02`（宜蘭縣文化局地方文宣，4/12 收款，5/15 開立發票）

---

### 6. 分 2 筆發票不同月份且指定時間

**業務情境**：
- 客戶要求發票拆兩張、分別在不同月份開立
- 常見於跨年度預算 / 跨月編列預算的政府或機構
- 業務需手動排程兩張發票各自的開立月份

**實作解決方式**：
1. 訂單成立、收款完成
2. 業務於 PaymentPlan 拆 2 期，各填 expected_invoice_date 對應月份
3. 業務於 4 月底開立 INV-01（一半金額）
4. 業務於 5 月中開立 INV-02（另一半金額）

**驗證重點**：
- 同訂單可開多張 Invoice（沒有上限）
- 各張 Invoice 開立月份獨立（issued_at 各自寫入）
- 跨月開立不影響字軌（系統流水號全局遞增）
- 兩張 Invoice 加總 = 訂單應收
- 三方對帳於兩張都開立後對齊

**Prototype 對應訂單**：`ORD-20260301-05`（高雄觀光協會旅遊摺頁 90K → 4/30 開 INV-01 45K + 5/15 開 INV-02 45K）

### 7. 作廢重開發票情況（多種子情況）

**業務情境**：
- 抬頭 / 統編更動 — 改統編重開
- 金額錯誤 — 業務報錯價、銷貨單金額錯
- 品名錯誤 / 調整（情境 4 已涵蓋）
- 客戶要求改月份（情境 6 已涵蓋）
- 調整規格數量改變 — 重新拋報 / 退簽折讓單後重開
- 自家失誤誤開（如點數已付款的訂單）
- 部分退款後 — 重開正確金額
- 客戶已申報過作廢發票 — 國稅局聯繫客人處理（屬流程外）

**實作解決方式**：
1. 業務於發票區點「作廢」並填入原因（如「統編誤填」「金額錯誤」）
2. 業務開立新發票，輸入正確內容
3. 舊發票字軌不重用，新發票流水號 +1

**驗證重點**：
- 作廢原因必填（限中文 6 字 / 英文 20 字內）
- Invoice.status 從「開立」變「作廢」
- 字軌流水號正確遞增
- 重開發票時 PaymentInvoice junction 重新關聯
- 對帳檢視時作廢 Invoice 不計入發票淨額

**Prototype 對應訂單**：`ORD-20260305-09`（傳承文創產品型錄，INV-01 統編誤填 54321987 作廢 → INV-02 重開 12345678）

---

### 8. 開折讓單情況（退款 / 調整金額）

**業務情境**：
- **前提條件：過期發票不可作廢，需開折讓單處理**
- 開立後要求指定月份日期
- 取消訂單退款
- 瑕疵退部分款項（單個 / 數量退）
- 規格調整需重開
- 多收款項退款（宅配改自取退運費）

**實作解決方式**：
1. 業務發現需退款時先建退款 Payment（金額為負數，paymentMethod = 退款）
2. 業務於對應 Invoice 點「開立折讓」，填入金額（負數）+ 原因
3. SalesAllowance 直接寫入「已確認」狀態（藍新 Mockup 兩段式 API）
4. SalesAllowance.refundPaymentId 手動關聯該退款 Payment

**驗證重點**：
- 折讓金額為負數，且絕對值 ≤ 發票剩餘可折讓金額
- 折讓不需業務主管核可（與 OrderAdjustment 不同）
- 一張發票可多筆折讓，累計上限為發票總額
- 已確認折讓可作廢（屬例外情境）
- 三方對帳：發票淨額 = 已開發票合計 - ∑ 已確認折讓
- 對帳警示 banner 因執行時點跨期觸發

**Prototype 對應訂單**：`ORD-20260118-04`（松山機場餐飲菜單 35K → 跨期不可作廢，折讓 -2K + 退款）

## 特殊收款方式

### 1. 指定日期分頭尾款 50%

**業務情境**：客戶要求頭款 50% 與尾款 50% 在指定日期到期。

**實作解決方式**：
1. 業務於 PaymentPlan 建立 2 期，各 50%，填入 scheduled_date
2. 客戶於指定日付頭款 → 業務記錄 Payment + 開立 INV-01
3. 客戶於指定日付尾款 → 業務記錄 Payment + 開立 INV-02
4. PaymentPlan.status 依累計 Payment 自動推導（未收 / 部分收款 / 已收訖）

**驗證重點**：
- PaymentPlan 兩期合計 = 訂單應收
- Payment 關聯到對應 PaymentPlan 期次
- 每期收訖後開立對應 Invoice
- 兩張 Invoice 加總 = 應收
- 三方對帳對齊

**Prototype 對應訂單**：`ORD-20260315-02`（典藏出版社年度文藝刊物 120K → 4/20 頭款 60K + 5/20 尾款 60K）

---

### 2. 申請後付款 — 出貨後指定時間內支付全額

**業務情境**：
- 申請後付款：先回簽 / 製作 / 出貨 / 開立發票
- 客戶於指定時間（如出貨後 30 天）內付款
- 演示「先開後付」流程

**實作解決方式**：
1. 業務於 PaymentPlan 建一期，scheduled_date 為出貨後 N 天
2. 訂單出貨同時開立 Invoice（PaymentInvoice junction 暫無 Payment 關聯）
3. 客戶於指定日期前付款 → 業務記錄 Payment 並關聯 PaymentInvoice junction

**驗證重點**：
- Invoice 可在無 Payment 對應時先開立（先開後付）
- PaymentInvoice junction 後續可手動關聯（業務操作）
- 對帳檢視應正確顯示「先開後付」狀態（待收款）
- 客戶付款後 PaymentPlan.status → 已收訖

**Prototype 對應訂單**：`ORD-20260408-01`（信義房屋名片 + DM 85K → 4/29 出貨開票 → 5/15 收款）

---

### 3. 申請頭尾款 — 且指定開發票日期 + 付款時間

**業務情境**：
- 頭款 30%：報價單回簽後 14 天內付款
- 尾款 70%：出貨同時開立發票，發票開立 45 天內付款
- 跨多月，業務需追蹤「該收款 / 該開票」

**實作解決方式**：
1. 業務於 PaymentPlan 建 2 期，各填 scheduled_date + expected_invoice_date
2. 收頭款 + 開 INV-01 訂金（業務手動執行）
3. 出貨 + 開 INV-02 尾款（業務手動執行）
4. 客戶於發票後 45 天內付尾款

**驗證重點**：
- PaymentPlan 兩期各自有 scheduled_date 與 expected_invoice_date
- INV-01 訂金、INV-02 尾款各自獨立
- 業務透過待收款模組追尾款（Change B 範疇）
- 對帳檢視於兩期均收訖後對齊

**Prototype 對應訂單**：`ORD-20260318-04`（金石堂書籍 100K → 訂金 3/20 開 30K + 尾款 4/26 開 70K）

---

### 4. 線下單補收款情境（含異動補收）

**業務情境**：
- 製作中印件追加加工（規格變更）
- 原訂單未評估運費出貨前補收運費
- 加印 / 規格升級 / 加工升級補收

**實作解決方式**：
1. 業務於訂單建立 OrderAdjustment（adjustment_type = 加印追加 / 規格變更 / 加運費 / 急件費）
2. 業務主管審核後執行 → 訂單應收總額增加
3. 業務手動於訂單詳情頁編輯 PrintItem（加印走原審稿 / 工單流程）
4. 業務新增一期 PaymentPlan 對應補收金額
5. 客戶付款 → 業務開立補收發票

**驗證重點**：
- OrderAdjustment.phase = during_order，可選 type 含加印 / 加運費 / 急件費
- 加印追加執行後系統提示「請至訂單詳情頁編輯印件」
- 業務手動加 PrintItem 走原審稿 / 工單流程
- 應收總額即時更新
- 三方對帳對齊

**Prototype 對應訂單**：`ORD-20260301-01`（霹靂 Q4 角色海報，加印 +8K + 折扣 -3K + 售後退款 -5K）

## 訂單異動情境

### 1. 異動不收費：公司吸收費用

**業務情境**：
- 工法目前無法克服 / 評估無法承接 → 全額退
- 紙廠停產 / 缺貨，客戶接受替代紙 → 直接換不補收
- 工廠瑕疵需補印（飛墨、背印、脫膜、污漬）→ 公司內部吸收
- 工單錯誤、規格寫錯 → 公司內部吸收
- 大貨「微調」如客戶補檔字距、輕微調整 → 不另收
- 第二次打樣是因第一次出問題 → 走原工單不補收

**實作解決方式**：
1. 業務 / 印務發起工單異動（修改紙材 / 工序 / 數量等）
2. 工單異動記錄變更內容（modifications 區塊）
3. **不建 OrderAdjustment**（amount=0 不屬於「金額變動」，違反 phase 判定邊界）
4. 訂單應收 / 發票 / 收款不變動

**驗證重點**：
- 工單異動流程獨立於 OrderAdjustment
- 工單.modifications 紀錄變更原因
- 訂單應收總額不變
- 三方對帳對齊（無新異動）
- 既有 PaymentPlan / Payment / Invoice 不受影響
- 不出現對帳警示 banner（無 OrderAdjustment 執行）

**Prototype 對應訂單**：`ORD-20260322-01`（田田餐飲菜單，紙廠停產換紙公司吸收，純走工單異動）

---

### 2. 異動補收：向客戶補收請款

**業務情境**：
- 原規格送打樣，但打樣後不製作大貨 → 打樣酌收
- 客戶臨時改規格 / 改數量 / 改加工 → 補收成本差
- 紙材成本 / 工法升級 → 重新報價補收
- 加工升級：新增刀模、燙金、上膜、雙面印 → 補收

**實作解決方式**：
與情境「特殊收款 4. 線下單補收款情境」相同：
1. 業務建立 OrderAdjustment（補收類型）
2. 業務主管審核執行
3. 業務手動於訂單詳情頁編輯 PrintItem（若涉及生產內容）
4. 業務新增 PaymentPlan + 開立補收發票

**驗證重點**：與情境「特殊收款 4」相同。

**Prototype 對應訂單**：`ORD-20260301-01`（已用於特殊收款 4 演示）

---

### 3. 異動退款：問題單最終手段

**業務情境**：
- 公司認同工法不夠好 + 客戶堅持 → 全額退 + 收回印件 / 減價驗收
- 不良率過高（如 21%）+ 客戶實測有疑慮 → 協助補印 + 提供備品 / 減價驗收
- 時間壓力下無法重新打樣 + 客戶不製作大貨 → 部分 / 全額退款
- 打樣實品 vs 客戶期待落差 → 退大貨費用終止合作
- 客戶臨時改規格 / 數量 / 加工 / 出貨方式 → 退部分異動款項
- 印刷瑕疵 → 退瑕疵數量款項
- 收款後發現檔案問題無法製作 → 全額退
- 下單後途中發現交期來不及 → 視情況退款

**實作解決方式**：
1. 業務建立售後服務單（OrderAdjustment phase = after_completion，type = 退印 / 補退）
2. 業務主管審核執行 → 應收總額降低
3. 業務於發票區建立退款 Payment（負數）
4. 視發票是否跨期：
   - 未跨期：作廢原 Invoice 重開正確金額
   - 已跨期：開立 SalesAllowance（折讓）關聯原 Invoice + refundPaymentId 連結退款 Payment

**驗證重點**：
- 售後服務單 phase 自動為 after_completion，type 範圍限退印 / 折扣 / 補退 / 其他
- 執行後對帳警示 banner 因執行時點跨期觸發
- 退款 Payment 不入 PaymentInvoice junction
- 折讓單關聯 Invoice 後發票剩餘可折讓金額減少
- 三方對帳：應收降低、發票淨額降低（折讓抵減）、收款淨額降低（退款抵減）→ 差額為零

**Prototype 對應訂單**：`ORD-20260331-02`（益生健康產品標籤 32K → 客戶投訴減價驗收 -8K）

## 不在 ERP 系統範疇

### 跨訂單金額混合匯款對帳 + 同訂單開指定數量發票 + 混單出貨（Notion 特殊發票 9）

**業務情境**：
- 同訂單分多次匯款且每次匯款混合其他訂單款項
- 例：訂單 A 分 3 次匯款，每次合併其他訂單款項一起匯
- 多訂單合併出貨

**spec 決定**：不做系統化支援，走人工處理。
- 業務 / 會計人工拆分銀行入帳到對應訂單
- 合併出貨業務人工協調

**未來考慮**：可建立會計後台「銀行入帳對帳台」工具，但不在本 change 範疇。

### 重新報價 / 重開報價單 / 重打樣案例（Notion 異動 4）

**業務情境**：
- 大貨重新確認規格 + 重出報價
- 大客戶大單議價
- 獨立簽打樣單 → 追加第二次打樣
- 客戶公司有請款及採購流程需要新報價單
- 議價單版本

**spec 決定**：屬需求單議價階段，不在訂單模組範疇。需求單模組另開 change 處理。
