# ERP 產品目標（Product Goals）

> 本文件定義 ERP 產品的商業目標、核心問題、成功指標與範疇。
> 所有功能規劃與優先排序需對照此文件。
>
> 最後更新：2026-02-23
> 狀態：更新中

---

## 一、產品背景

### 現況痛點
目前公司使用兩套系統處理訂單：

| 系統 | 用途 | 問題 |
|------|------|------|
| **Ragic** | 線下客戶訂單、工單管理 | 大量依賴 Slack / 紙本處理工單異動，人工記錄容易出錯 |
| **官網 EC** | 線上訂單 | 獨立系統，與 Ragic 資料結構不同，造成線上/線下作業流程分歧 |

兩套系統並存造成：資料無法統一、角色操作重複、問題發生時難以追溯。

### 自建 ERP 的動機
1. **印刷業特殊製程**需求（打樣流程、工廠類型路由、分批出貨等）無法完全依賴現成方案
2. **EC 整合需求**：自有電商與線下訂單需要統一資料結構，最終實現同一套流程管理
3. **Data Governance 優先**：先建立生產任務、QC、出貨的資料追蹤基礎，後續才能做生產效率優化分析

## 二、核心問題

### 問題 1：工單資訊散落，掉單與生產錯誤風險高

**現況**：規格異動、數量異動等工單變更，多依賴 Slack 討論或紙本修改，未統一進入系統。
**後果**：
- 工單更新後，執行端（師傅 / 工廠）未收到最新版本
- 容易掉單（工單未進入生產）或生產錯誤（照舊版規格執行）
- 異動記錄無留存，出問題時無法追溯

### 問題 2：產能不透明，管理層無法有效判斷工廠實際狀況

**現況**：工廠生產排程與進度不在系統內，管理層只能靠口頭回報。
**後果**：
- 工廠說做不完，管理層無法判斷是真的超載還是排程問題
- 無法做有效的產能調配與交期承諾
- 緊急訂單插單時缺乏數據依據

### 問題 3：角色分工不明確，問題發生時責任模糊

**現況**：工單操作、審核、分派等行為缺乏明確的系統記錄與角色綁定。
**後果**：
- 出問題時不知道要找誰
- 相同的工作可能被多人重複做，或沒有人做
- 無法建立績效追蹤基礎

### 問題 4：小量客製化印刷的工廠派工缺乏合批機制，換工成本高

**現況**：每張工單的生產任務各自派給工廠，未依工序統一合批。
**後果**：
- 工廠同一天需多次切換工序準備（換版、換設定），浪費準備時間
- 小量訂單無法累積到可有效安排的批次，單位成本偏高
- 排程人員每天手動整理各工單派工，重複勞力且容易遺漏
- 管理層無法一覽當天各工序的整體工作量，難以評估產能是否合理

## 三、兩階段範疇

### Phase 1 — 線下訂單完整流程（目標：6 個月內上線）

**目標**：讓線下訂單能在新 ERP 中走完整個生命週期，取代 Ragic + Slack + 紙本的組合。

```
評估階段 → 打單階段 → 審稿階段 → 打樣 / 印製階段 → 出貨階段
```

**定義「完成」的條件**：
- 業務能在系統建立需求單、轉訂單
- 工單的建立、審核、分派、異動全部在系統內完成，不走 Slack
- 印務 / 生管能在系統追蹤生產進度
- QC 結果在系統記錄並驅動入庫狀態
- 出貨可在系統操作，支援分批出貨
- **（產能優化）** 排程人員能在系統產出「當日工序合批派工表」，依工序 × 內容群組合並同天生產任務，統一派給工廠（XM-003 確認後設計；詳細規則見 `memory/erp/open-questions.md`）

### Phase 2 — EC 訂單整合（目標：Phase 1 上線後 6 個月）

**目標**：官網 EC 訂單使用與線下訂單相同的資料結構與生產流程，不需要另一套作業系統。

**定義「完成」的條件**：
- EC 訂單自動進入 ERP 工單流程
- 線上 / 線下訂單在同一個系統介面管理
- 生產端（師傅 / 工廠）不需區分訂單來源

## 四、成功指標

### North Star Metric

**訂單生命週期系統化率**：從需求建立到出貨，有多少比例的操作是在 ERP 系統內完成（而非 Slack / 紙本）。

> 目標（Phase 1 上線 3 個月後）：線下訂單系統化率 ≥ 90%

---

### Phase 1 KPIs（上線後 3 個月內評估）

| 指標 | 定義 | 目標 |
|------|------|------|
| **線下訂單系統化率** | 透過新 ERP 處理的線下訂單比例 | 100%（全量切換） |
| **工單異動系統化率** | 在系統內執行異動（非 Slack/紙本）的工單比例 | ≥ 90% |
| **掉單率** | 工單未進入生產或生產錯誤的訂單比例 | 趨近 0（較現況顯著下降） |
| **產能可見度** | 管理層能在系統即時查看工廠生產排程 | 100%（儀表板可用） |
| **責任可追溯率** | 工單相關操作有明確角色記錄的比例 | 100% |
| **合批派工覆蓋率** | 使用系統合批派工表處理的生產任務比例（⚠️ XM-003 確認後補目標值）| TBD |
| **換工次數減少率** | 工廠每日同工序任務合批後，相較於逐單派工的換工次數下降比例（⚠️ XM-003 確認後補基準與目標）| TBD |

### Phase 2 KPIs（EC 整合上線後 3 個月內評估）

| 指標 | 定義 | 目標 |
|------|------|------|
| **EC 訂單整合率** | EC 訂單透過新 ERP 生產流程處理的比例 | 100% |
| **線上/線下流程一致性** | 生產端無需區分訂單來源，操作流程相同 | 達成（定性驗收） |
| **人工對接作業時間** | 每張 EC 訂單需要額外人工處理的時間 | ≤ Phase 1 線下訂單水準 |

## 五、明確不做的事（Out of Scope）

| 項目 | 說明 |
|------|------|
| **完整財務 / 會計模組** | ERP 僅需產出對帳報表，不需完整財會系統 |
| **人事 / HR 模組** | 不在範疇內 |
| **EC 後台功能** | Phase 1 不含 EC 整合，僅處理線下訂單 |
| **採購自動化** | 目前人工採購流程維持，ERP 記錄採購單即可 |
| **客戶自助入口** | 客戶端操作（查單、上稿）不在此版本範疇 |

## 六、關係人

| 角色 | 關切點 |
|------|--------|
| **老闆 / 管理層** | 產能透明度、訂單不掉單、Phase 1 在 6 個月內上線 |
| **業務 / 諮詢業務** | 報價流程順暢、訂單狀態可追蹤 |
| **印務主管** | 工單審核有系統支援、異動有留存記錄 |
| **訂單管理人** | 工單分配效率、整體進度可見 |
| **生管 / 師傅** | 工單指示清楚、異動即時通知 |
| **開發團隊** | 資料結構統一（線下 / 線上共用），降低維護成本 |
