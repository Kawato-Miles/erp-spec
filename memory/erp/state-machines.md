# ERP 狀態機（上層：需求單 / 訂單 / 工單 / 印件）

> 涵蓋：需求單、訂單、工單、印件（審稿 + 印製）。
> 下層狀態機（任務 / 生產任務 / QC / 出貨單）見 `state-machines-ops.md`。
>
> 最後更新：2026-02-24（v10：新增工單完成計算與 狀態向上傳遞 機制的詳細設計決策；補充層級統計邏輯、計算時機、邊界情況防呆）
> 設計 spec 或 prototype 時，所有狀態必須與本文件及 ops 文件對齊。

---

**層級相依概覽（由上至下）：**
```
需求單 → 訂單 → 印件 → 工單 → 任務 → 生產任務
```
上層狀態由下層統計決定；下層需等上層建立後才能產生。

> 印件（Print Item）掛在訂單下，一張訂單可含多個印件（不同款式 / 尺寸 / 材料）。印件有兩個種類（打樣印件、大貨印件）。每個印件會產生 1 張以上的工單（打樣印件時，因需重新打樣建立多個工單 / 大貨印件因製作負責，拆分給多個印務負責，故建立多筆工單）。工單下再分拆為任務（依工廠分組）、任務下再為生產任務（最小追蹤單位）。

## 需求單狀態機

> 概念類似 Salesforce Sales Cloud 的機會管理（Opportunity）。成交後銜接訂單流程；流失則關單，不建立訂單。

**角色對應：**
- **業務 / 諮詢業務**：建立需求單、推進報價流程、與客戶議價
- **印務主管**：評估成本與售價（v0.8 簡化：直接填入成本預估，無需選擇製程）

### 狀態一覽

| 狀態 | 類型 | 說明 |
|------|------|------|
| 需求確認中 | 初始 | 釐清客戶需求規格中 |
| 待評估報價 | 進行中 | 待印務主管評估成本與售價 |
| 已評估報價 | 進行中 | 成本與售價已確認 |
| 已提供報價 | 進行中 | 已提供報價給客戶 |
| 議價中 | 進行中 | 待客戶回應，進入議價 |
| 成交 | 終態 | 需求成立，接訂單流程 |
| 流失 | 終態 | 需求不成立，關單保留紀錄 |

### 狀態流程

```
需求確認中
 → 待評估報價（需求確認完成）
 → 已評估報價（印務主管評估完成）
 → 已提供報價（業務對外提供）
 → 議價中（客戶回應，進入議價）
   → 業務申請重新評估：退回「待評估報價」，與印務主管討論成本調整
   → 成交：需求成立 → 接訂單流程（建立正式報價單）
   → 流失：需求不成立 → 關單，不建立訂單
```

### 狀態轉換

| 從 | 觸發條件 | 到 | 執行角色 | 手動/自動 |
|----|---------|----|---------|----|
| 需求確認中 | 規格確認完成，送印務評估 | 待評估報價 | 業務 | 手動 |
| 待評估報價 | 印務主管評估完成 | 已評估報價 | 印務主管 | 手動 |
| 已評估報價 | 業務對外提供報價 | 已提供報價 | 業務 | 手動 |
| 已提供報價 | 客戶回應，開始議價 | 議價中 | 業務 | 手動 |
| 議價中 | 業務申請重新評估，與印務主管討論成本調整 | 待評估報價 | 業務 | 手動 |
| 議價中 | 客戶確認成交 | 成交 | 業務 | 手動 |
| 議價中 | 客戶不成交 | 流失 | 業務 | 手動 |

### 設計決策

- 各階段成交機率 / Pipeline 管理功能：不在本版 ERP 範疇，為 Data Governance 後的 Dashboard 優化項目
- 流失後需求單保留紀錄，不得刪除
- **檢視權限（v2.0）**：需求單擁有者可授予他人檢視權限（檢視專用，不可編輯）。檢視權限為獨立維度，不影響狀態機：擁有者可進行所有狀態轉換；被授權人只能檢視，無法進行任何狀態轉換或編輯操作。成交/流失狀態不影響權限管理。

## 訂單狀態機

> 線下訂單以「已回簽」驅動生產流程，付款另外平行追蹤。
> 線上訂單（EC）以「已付款」驅動，付款即生產流程的觸發點。

**角色對應：**
- **業務 / 諮詢業務**：建立訂單、確認回簽、追蹤稿件與款項
- **會計**：確認付款、開立發票

### 狀態一覽

| 狀態 | 路徑 | 類型 | 說明 |
|------|------|------|------|
| 報價待回簽 | 線下 | 初始 | 報價單待客戶回簽 |
| 已回簽 | 線下 | 進行中 | 客戶已回簽 |
| 等待付款 | 線上 / 線下 | 初始 / 進行中 | 待客戶付款 |
| 已付款 | 線上 / 線下 | 進行中 | 客戶已付款 |
| 稿件未上傳 | 共用 | 進行中 | 訂單內至少一個印件未上傳稿件 |
| 等待審稿 | 共用 | 進行中 | 所有印件稿件已上傳，至少一個進行審稿 |
| 製作等待中 | 共用 | 進行中 | 所有印件審稿通過，工單尚未全數交付 |
| 工單已交付 | 共用 | 進行中 | 所有工單的所有任務都已交付 |
| 製作中 | 共用 | 進行中 | 任一工單進入製作中（狀態向上傳遞，不倒退） |
| 製作完成 | 共用 | 進行中 | 所有印件製作已完成 |
| 出貨中 | 共用 | 進行中 | 至少一個印件出貨中（狀態向上傳遞）|
| 訂單完成 | 共用 | 終態 | 所有印件全數送達 |

> 補收款（急件費用等）為平行追蹤維度，不列入主狀態機。

### 狀態流程

#### 線下訂單（以回簽驅動）
```
報價待回簽 → 已回簽 → [共用段]
```

**付款狀態（平行追蹤，不阻塞主流程）：**
```
等待付款 → 已付款
退款狀態（⏳ 待設計）
```

#### 線上訂單（以付款驅動）
```
等待付款 → 已付款 → [共用段]
```

#### 共用段（已回簽 / 已付款後）
```
稿件未上傳（訂單中至少有一個印件尚未上傳稿件）
 → 等待審稿（訂單中所有印件稿件都已上傳，至少有一個印件審稿進行中）
 → 製作等待中（所有印件審稿通過，工單草稿尚未全數交付）
 → 工單已交付（所有工單的所有任務都已交付）
 → 製作中（任一印件進入「製作中」；進入後狀態向前推進，不倒退）
 → 製作完成（所有印件製作完成）
 → 出貨中（任一印件開始出貨，狀態向上傳遞；細節由出貨單追蹤）
 → 訂單完成（所有印件全數送達）
```

#### 補收款（平行，不阻塞主流程）
```
補收款未付款 ↔ 補收款已付款
```
> 適用於急件費用、重新打樣補收等情境；與主訂單流程獨立追蹤，不影響生產推進。

### 狀態轉換

| 從 | 觸發條件 | 到 | 執行角色 | 手動/自動 |
|----|---------|----|---------|-----|
| 報價待回簽 | 客戶回簽確認 | 已回簽 | 業務 | 手動 |
| 已回簽 | 自動進入共用段 | 稿件未上傳 | 系統 | 自動 |
| 等待付款（線下） | 會計確認收款 | 已付款 | 會計 | 手動 |
| 等待付款（線上） | 客戶付款成功 | 已付款 | 系統 | 自動 |
| 已付款（線上） | 自動進入共用段 | 稿件未上傳 | 系統 | 自動 |
| 稿件未上傳 | 訂單中所有印件的稿件都已上傳，且至少有一個印件進入審稿 | 等待審稿 | 系統（偵測所有印件稿件狀態） | 自動 |
| 等待審稿 | 所有印件審稿通過 | 製作等待中 | 系統 | 自動 |
| 製作等待中 | 所有工單任務交付完成 | 工單已交付 | 系統（統計） | 自動 |
| 工單已交付 | 任一印件進入「製作中」（狀態向上傳遞；前置條件：訂單已進入「工單已交付」狀態） | 製作中 | 系統 | 自動 |
| 製作中 | 所有印件製作完成 | 製作完成 | 系統 | 自動 |
| 製作完成 | 任一印件第一筆出貨單出車 / 移交物流商 | 出貨中 | 系統（狀態向上傳遞）| 自動 |
| 出貨中 | 所有印件全數送達（累計已送達 ≥ 目標）| 訂單完成 | 系統（狀態向上傳遞）| 自動 |

### 設計決策（訂單狀態機）

**1. 訂單狀態向前推進，不可逆**

訂單狀態遵循單向流動原則，一旦進入某個狀態，不倒退回前期狀態。特別地：
- 訂單進入「製作中」後，即使有新印件的稿件後續上傳或仍在審稿，訂單也維持「製作中」；不倒退回「等待審稿」
- 訂單進入「製作等待中」後，不倒退回「等待審稿」
- 此原則確保訂單整體進度單向推進，避免狀態混亂

**2. 訂單「等待審稿」觸發條件**：

訂單進入「等待審稿」需：訂單中**所有印件**的稿件都已上傳。若打樣稿件已上傳但大貨未上傳 → 訂單停留「稿件未上傳」。

**3. 訂單狀態基於印件層聚合**（非工單層）

訂單「工單已交付」：所有工單的所有任務都已交付

訂單「製作中」：任一印件進入「製作中」（不是工單層直接推進）
  - 例：工單 A 進入製作中 → 印件進入「部分工單製作中」→ 訂單仍停留「工單已交付」
  - 工單 A、B 都進入製作中 → 印件進入「製作中」→ 訂單進入「製作中」

**4. 「稿件未上傳」語義**

訂單內至少一個印件未上傳稿件。大貨印件特例：建立時審稿狀態 = 稿件未上傳；直到打樣 OK 後業務才開放上傳。

## 工單狀態機

> type 欄位區分：打樣工單 / 大貨工單

**角色對應：**
- **印務**：填寫工單內容，送製程審核；建立 QC 單
- **印務主管**：執行製程審核（通過 / 退回）、執行工單異動
- **生管**：確認異動、分派生產任務
- **QC 人員**：執行 QC 盤點，填寫 QC 紀錄

### 狀態一覽

| 狀態 | 類型 | 說明 |
|------|------|------|
| 草稿 | 初始 | 工單已建立，待送審 |
| 製程確認中 | 進行中 | 待印務主管審核 |
| 重新確認製程 | 可逆 | 印務主管退回，待調整重送 |
| 製程審核完成 | 進行中 | 審核通過，待分派任務 |
| 工單已交付 | 進行中 | 所有任務都已交付給工廠 |
| 製作中 | 進行中 | 任一任務進入製作中（狀態向上傳遞） |
| 異動 | 可逆 | 任務需求異動，待生管確認 |
| 已完成 | 終態 | 累計 QC 入庫達工單目標生產數量 |

### 狀態流程

```
草稿
 → 製程確認中（印務填寫工單內容後送審）
   ↔ 重新確認製程（印務主管退回要求調整）
 → 製程審核完成（印務主管確認通過）
 → 工單已交付（條件：工單下所有任務皆已完成各自的交付作業）
     * 自有工廠任務：印務透過系統交付給生管
     * 外包廠 / 中國廠商任務：印務印製紙本工單（僅帶出該廠商所需執行的生產任務），交付給生管；生管決定時機後交付給工廠
 → 製作中（條件：任一任務進入製作中，由任務層 狀態向上傳遞）
   ↔ 異動（任務需求異動，需生管確認後恢復製作中）
 → 已完成（條件：所有「影響成品」任務之 QC 入庫累計數量 ≥ 工單目標生產數量）
```

### 狀態轉換

| 從 | 觸發條件 | 到 | 執行角色 | 手動/自動 |
|----|---------|----|---------|-----|
| 草稿 | 印務填寫完成，送審 | 製程確認中 | 印務 | 手動 |
| 製程確認中 | 印務主管退回 | 重新確認製程 | 印務主管 | 手動 |
| 重新確認製程 | 印務調整完成，重新送審 | 製程確認中 | 印務 | 手動 |
| 製程確認中 | 印務主管審核通過 | 製程審核完成 | 印務主管 | 手動 |
| 製程審核完成 | 工單下所有任務完成各自交付（系統統計） | 工單已交付 | 系統 | 自動 |
| 工單已交付 | 任一任務進入製作中（任務 狀態向上傳遞） | 製作中 | 系統 | 自動 |
| 製作中 | 印務發起異動 | 異動 | 印務 | 手動 |
| 異動 | 生管確認異動內容 | 製作中 | 生管 | 手動 |
| 製作中 | 累計 QC 入庫 ≥ 工單目標（系統計算） | 已完成 | 系統 | 自動 |

### 設計決策

> 工單不再設置 QC 狀態節點（待 QC / QC 已通過 / QC 不通過 / 重建任務 / 入庫）。
> 完成判斷改由 QC 紀錄中的入庫累計數量驅動（見本文件 QC 相關實體 / business-process.md Section 4.3）。

> 工單不設「打樣不通過」終態。打樣 OK/NG 以印件的「打樣結果」欄位管理（見印件狀態機）。

#### 【新增】工單完成數的計算與狀態向上傳遞機制

**計算觸發時機**：每次 QC 單完成時，系統自動重新計算工單層完成數。

**計算邏輯（四層統計，採用木桶原則 min()）**：

詳細公式見 `quantity-calculation-rules.md`；此處為邏輯摘要：

層級 1 - 生產任務層：`pt_completed = 報工累計數`

層級 2 - 任務層：篩選「影響成品」的生產任務 → 各自計算倍數 `floor(pt_completed ÷ quantity_per_work_order)` → 取最小值

層級 3 - 工單層：所有任務的倍數 → 取最小值

層級 4 - 印件層：各工單計算倍數 `floor(wo_completed ÷ quantity_per_print_item)` → 取最小值 → 若達目標則印件完成、狀態向上傳遞到訂單層

**完成判斷**：

若 累計 QC 入庫 ≥ 工單目標 → 工單自動「製作中」→「已完成」

若 累計 QC 入庫 < 工單目標 → 工單保持「製作中」，顯示缺口數；印務決定補生產、等下批 QC 或分批出貨

**狀態向上傳遞鏈**：

生產任務進入「製作中」→ 任務「製作中」→ 工單「製作中」→ 印件「製作中」→ 訂單「製作中」
（任一層級任一子項進入下層狀態，即自動觸發上層進入對應狀態）

**異動流程對計算的影響**：

任務進入「異動」時暫停計算 → 生管確認異動後返回「製作中」，系統重新計算工單完成數。新增生產任務的倍數由系統建議（參照原平均倍數），但允許手動覆寫。詳見 `quantity-calculation-rules.md`。

**邊界情況防呆**：

1. 工單必須至少有 1 個「影響成品」的生產任務，否則系統提示錯誤
2. 倍數範圍驗證：必須 > 0；超過 10,000 發出警告
3. UI 應顯示「缺口」（如 500/1000）而非隱藏小數，提升清晰度

**與 QC 層對應**：

QC 單完成 → 系統計算 `累計入庫 = Σ 所有 QC 單通過數` → 與工單目標對比判定完成。詳細計算流程與測試案例見 `quantity-calculation-rules.md` 與 `test-cases.md`。

## 印件狀態機

印件有兩個獨立的狀態維度：**審稿狀態**與**印製狀態**，分別追蹤。
觸發關係：審稿狀態達到「合格」→ 自動建立工單草稿 → 印製狀態啟動。

---

### 審稿狀態

**角色對應：**
- **審稿人員**：執行審稿，標記合格 / 不合格
- **印務**：選定稿件檔案建立工單
- **業務 / 客戶**：上傳稿件、補件

#### 狀態一覽（審稿）

| 狀態 | 類型 | 說明 |
|------|------|------|
| 稿件未上傳 | 初始 | 尚未上傳稿件 |
| 等待審稿 | 進行中 | 稿件待審核 |
| 已補件 | 進行中 | 審稿未通過，已重新上傳稿件 |
| 合格 | 終態 | 審稿通過，自動建立工單草稿 |

> 「等待審稿 → 已補件 → 等待審稿」為迴圈段，直到合格為止。

#### 狀態流程（審稿）

```
稿件未上傳
 → 等待審稿（上傳稿件檔案後）
   → 不合格（審稿人員標記）
       → 已補件（上傳新稿件檔案）→ 等待審稿（loop）
   → 合格（審稿人員標記）
       → 自動建立工單草稿（生產任務是否自動帶入，依印件類型決定）
```

#### 狀態轉換（審稿）

| 從 | 觸發條件 | 到 | 執行角色 | 手動/自動 |
|----|---------|----|---------|----|
| 稿件未上傳 | 業務 / 客戶上傳稿件 | 等待審稿 | 業務 / 客戶 | 手動 |
| 等待審稿 | 審稿人員標記不合格 | 已補件（待上傳） | 審稿人員 | 手動 |
| 已補件（待上傳） | 業務 / 客戶上傳補件稿 | 等待審稿 | 業務 / 客戶 | 手動 |
| 等待審稿 | 審稿人員標記合格 | 合格 | 審稿人員 | 手動 |
| 合格 | 系統自動觸發 | 工單草稿建立，印製狀態進入等待中 | 系統 | 自動 |

#### 設計決策（審稿）

**生產任務帶入規則（合格後建工單草稿時）：**

| 印件類型 | 生產任務帶入方式 |
|----------|----------------|
| EC 訂單印件 | ✅ 自動帶入（非客製化商品，製程預先定義） |
| 線下 — 半客製品 | ✅ 參照 EC 商品帶入，印務可修改 |
| 線下 — 參考其他訂單 | ✅ 參照指定訂單帶入，印務可修改 |
| 線下 — 全客製品 | ❌ 工單草稿建立，生產任務由印務手動建立 |

**稿件檔案管理規則：**
- 印件可上傳多個稿件檔案（不以版次號管理）
- 每份稿件含該印件所有款式設計內容；款式更新時整份檔案重新上傳
- 工單建立時鎖定使用的合格稿件；生產期間不受後續上傳異動
- 棄用 / 不合格的歷史稿件保留傳檔紀錄，不刪除
- 打樣 NG（稿件問題）→ 建立新打樣印件（不在同一印件補稿，見 business-process.md Section 5.3）

### 印製狀態

打樣印件與大貨印件共用一份狀態機；以 `type` 欄位區分，UI 依 type 控制初始狀態、欄位顯示與可用操作。

**打樣結果欄位**（掛在打樣印件上；僅打樣印件在 UI 顯示與開放填寫）：
`待確認 / OK / NG-製程問題 / NG-稿件問題`

後續動作說明：
- OK → 業務手動開放大貨印件上傳稿件，進入審稿流程（流程層控制，非狀態轉換）
- NG-製程問題 → 印務在同一打樣印件下建立新打樣工單；**系統自動觸發印製狀態 → 等待中**（打樣印件限定）
- NG-稿件問題 → 建立新打樣印件；業務 / 印務手動將舊打樣印件 → 已棄用

#### 狀態一覽（印製）

| 狀態 | 適用 | 類型 | 說明 |
|------|------|------|------|
| 等待中 | 兩者 | 初始 | 工單尚未交付給工廠 |
| 工單已交付 | 兩者 | 進行中 | 該印件下所有工單的所有任務都已交付 |
| 部分工單製作中 | 兩者 | 進行中 | 多工單場景：至少一個工單進入製作中 |
| 製作中 | 兩者 | 進行中 | 所有工單都進入製作中（狀態向上傳遞） |
| 製作完成 | 兩者 | 進行中 | 所有工單皆已完成 |
| 出貨中 | 兩者 | 進行中 | 出貨進行中 |
| 已送達 | 兩者 | 終態 | 全數送達確認（打樣另有特殊規則見下） |
| 已棄用 | 打樣限定 | 終態 | 打樣結果為 NG-稿件問題 |

#### 狀態流程（印製）

```
[打樣印件 / 大貨印件 — 共同初始狀態：印件建立時即為等待中]

等待中（工單尚未交付；工單可能未建立、草稿建立中、或製程審核中）
 → 工單已交付（該印件下所有工單的所有任務都交付）
   ↓
   [單工單場景]
   → 製作中

   [多工單場景]
   → 部分工單製作中（至少一個工單進入製作中）
    → 製作中（所有工單都進入製作中或之後狀態）
   ↓
 製作中（出貨單可在此階段建立；防呆：數量 ≤ 累計 QC 入庫數量）
 → 製作完成（條件：印件中所有工單皆已完成）
 → 出貨中（第一筆出貨單進入「出貨中」（自行）或「已出貨」（第三方）時，系統自動觸發）
 → 已送達

[打樣印件特殊規則]
已送達 + NG-製程問題 → 等待中（建新工單，系統自動觸發）

```

|----|---------|----|---------|----|
| — | 印件建立（初始狀態） | 等待中 | 系統 | 自動 |

#### 設計決策（印製）

**部分工單製作中狀態（多工單場景）：**

第 1 個工單進入「製作中」→ 印件「部分工單製作中」；最後 1 個工單進入「製作中」→ 印件「製作中」。單工單場景無此中間態，工單進入製作中時印件直接「製作中」。

出貨仍受木桶原則限制（min() 入庫數），即使一工單已完成，出貨額度仍受所有工單最小值限制。

**出貨鏈設計：**

出貨細節由**出貨單**管理；印件僅記錄摘要。出貨單可在製作中建立（防呆：出貨數 ≤ 累計 QC 入庫數）。分批出貨進度由 UI 展示。

**打樣印件特殊語意：**

「已送達」= 樣品已送達，待客戶填寫打樣結果。NG-製程問題時狀態退回「等待中」（正向流程，非逆向），意為本次打樣結束、重新開始下一輪。

**初始印製狀態：**

印件建立時狀態即為「等待中」（涵蓋工單建立至交付前的整個等待期）。

**大貨印件打樣控制：**

大貨建立時審稿停在「稿件未上傳」；打樣 OK 後業務才開放上傳。此為流程層控制（非狀態轉換）；大貨在等待打樣期間印製狀態維持「等待中」。

**工單延遲交付：**

工單可在審核完成後暫不交付，直到決定啟動時再交付。只有所有工單都交付後，印件才進入「工單已交付」。見 scenarios.md § 情境 2。
