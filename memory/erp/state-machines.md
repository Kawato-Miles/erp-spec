# ERP 狀態機（State Machines）

> 所有單據的狀態轉換規則。設計 spec 或 prototype 時，所有狀態必須與本文件對齊。
>
> 最後更新：2026-02-22（v6：修正主從關係（印件層級位置）；工單已交付條件改由任務層驅動；修正工單 / 任務製作中 bubble-up 描述；QC 單完成條件改為本次指定數量核查完畢）
> 狀態：六層相依鏈 + QC 實體全數確認完成

---

**層級相依概覽（由上至下）：**
```
需求單 → 訂單 → 印件 → 工單 → 任務 → 生產任務
```
上層狀態由下層聚合決定；下層需等上層建立後才能產生。

> 印件（Print Item）掛在訂單下，一張訂單可含多個印件（不同款式 / 尺寸 / 材料）。每個印件會產生 1 張以上的工單（打樣工單 / 大貨工單）。工單下再分拆為任務（依工廠分組）、任務下再為生產任務（最小追蹤單位）。

## 需求單狀態機

> 概念類似 Salesforce Sales Cloud 的機會管理（Opportunity）。成交後銜接訂單流程；流失則關單，不建立訂單。

```
需求確認中（收到客戶需求，釐清規格中）
 → 待評估報價（需求確認，待印務主管評估成本與售價）
 → 已評估報價（成本與售價確認，可對外報價）
 → 已提供報價（已將報價提供給客戶）
 → 議價中（待客戶回應，可能需調整方案）
   → 重新提供方案：退回「待評估報價」，與印務主管重新討論
   → 成交：需求成立 → 接訂單流程（建立正式報價單）
   → 流失：需求不成立 → 關單，不建立訂單
```

**角色對應：**
- **業務 / 諮詢業務**：建立需求單、推進報價流程、與客戶議價
- **印務主管**：評估製程成本與售價

**注意事項：**
- 各階段可設定成交機率，供 pipeline 管理與營收預估使用（功能待設計）
- 流失後需求單保留紀錄，不得刪除

## 訂單狀態機

> 線下訂單以「已回簽」驅動生產流程，付款另外平行追蹤。
> 線上訂單（EC）以「已付款」驅動，付款即生產流程的觸發點。

#### 線下訂單（以回簽驅動）

```
報價待回簽 → 已回簽 → [共用段]
```

**付款狀態（平行追蹤，不阻塞主流程）：**
```
等待付款 → 已付款
退款狀態（⏳ 待設計）
```

#### 線上訂單（以付款驅動）

```
等待付款 → 已付款 → [共用段]
```

#### 共用段（已回簽 / 已付款後）

```
稿件未上傳（尚未有任何印件上傳稿件）
 → 等待審稿（至少一個印件稿件已上傳，審稿進行中）
 → 製作等待中（所有印件審稿通過，工單草稿尚未建立）
 → 工單已交付（工單草稿分派後）
 → 製作中（任一工單開始製作）
 → 製作完成（所有印件製作完成）
 → 出貨未處理 → 打包中 → 物流未完成
 → 訂單完成
```

**角色對應：**
- **業務 / 諮詢業務**：建立訂單、確認回簽、追蹤稿件與款項
- **會計**：確認付款、開立發票

#### 補收款（平行，不阻塞主流程）

```
補收款未付款 ↔ 補收款已付款
```

> 補收款適用於急件費用、重新打樣補收等情境；與主訂單流程獨立追蹤，不影響生產推進。

## 工單狀態機

```
草稿
 → 製程確認中（印務填寫工單內容後送審）
   ↔ 重新確認製程（印務主管退回要求調整）
 → 製程審核完成（印務主管確認通過）
 → 工單已交付（條件：工單下所有任務皆已完成各自的交付作業）
     * 自有工廠任務：透過系統交付
     * 外包廠 / 中國廠商任務：印製紙本工單，僅帶出該廠商所需執行的生產任務，逐份交付
 → 製作中（條件：任一任務進入製作中，由任務層 bubble-up）
   ↔ 異動（任務需求異動，需生管確認後恢復製作中）
 → 已完成（條件：所有「影響成品」任務之 QC 入庫累計數量 ≥ 工單目標生產數量）
```

**角色對應：**
- **印務**：填寫工單內容，送製程審核；建立 QC 單
- **印務主管**：執行製程審核（通過 / 退回）、執行工單異動
- **生管**：確認異動、分派生產任務
- **QC 人員**：執行 QC 盤點，填寫 QC 紀錄

> 工單不再設置 QC 狀態節點（待 QC / QC 已通過 / QC 不通過 / 重建任務 / 入庫）。
> 完成判斷改由 QC 紀錄中的入庫累計數量驅動（見本文件 QC 相關實體 / business-process.md Section 4.3）。

## 印件狀態機

印件有兩個獨立的狀態維度：**審稿狀態**與**印製狀態**，分別追蹤。
觸發關係：審稿狀態達到「合格」→ 自動建立工單草稿 → 印製狀態啟動。

---

### 審稿狀態

```
稿件未上傳
 → 等待審稿（上傳稿件檔案後）
   → 不合格（審稿人員標記）
       → 已補件（上傳新稿件檔案）→ 等待審稿（loop）
   → 合格（審稿人員標記）
       → 自動建立工單草稿（生產任務是否自動帶入，依印件類型決定）
```

**生產任務帶入規則（合格後建工單草稿時）：**

| 印件類型 | 生產任務帶入方式 |
|----------|----------------|
| EC 訂單印件 | ✅ 自動帶入（非客製化商品，製程預先定義） |
| 線下 — 半客製品 | ✅ 參照 EC 商品帶入，印務可修改 |
| 線下 — 參考其他訂單 | ✅ 參照指定訂單帶入，印務可修改 |
| 線下 — 全客製品 | ❌ 工單草稿建立，生產任務由印務手動建立 |

**稿件檔案管理規則：**
- 印件可上傳多個稿件檔案（不以版次號管理）
- 每份稿件含該印件所有款式設計內容；款式更新時整份檔案重新上傳
- 工單建立時鎖定使用的合格稿件；生產期間不受後續上傳異動
- 棄用 / 不合格的歷史稿件保留傳檔紀錄，不刪除
- 打樣 NG（稿件問題）→ 建立新打樣印件（不在同一印件補稿，見 business-process.md Section 5.3）

**角色對應：**
- **審稿人員**：執行審稿，標記合格 / 不合格
- **印務**：選定稿件檔案建立工單

---

### 印製狀態

打樣印件與大貨印件共用一份狀態機；以 `type` 欄位區分，UI 依 type 控制初始狀態、欄位顯示與可用操作。

**打樣結果欄位**（掛在打樣印件上；僅打樣印件在 UI 顯示與開放填寫）：
`待確認 / OK / NG-製程問題 / NG-稿件問題`

後續動作由業務 / 印務手動觸發，不自動化：
- OK → 大貨印件可開始上傳稿件，進入審稿流程（流程層控制，非狀態轉換）
- NG-製程問題 → 在同一打樣印件下建立新打樣工單
- NG-稿件問題 → 建立新打樣印件；舊打樣印件 → 已棄用

```
[打樣印件 / 大貨印件 — 初始狀態]
等待中

[共用段]
等待中（工單草稿已建立，尚未交付）
 → 工單已交付
 → 製作中（任一工單開始執行）
   → 部分已出貨（出貨數量 > 0，分批出貨中）
 → 製作完成（條件：印件中所有工單皆已完成）
 → 打包中 → 待出貨 → 出貨中 → 已出貨 → 運送中 → 已送達

[打樣印件限定終態]
製作完成 → 已棄用（打樣結果 = NG-稿件問題，建立新打樣印件後手動觸發）
```

**分批出貨說明：**
- 同一印件可分批出貨；出貨數量 ≤ 累計 QC 入庫數量（不可超出已入庫成品）
- 出貨數量 > 0 時進入「部分已出貨」；全數出貨後轉為「已出貨」
- 打樣印件出貨 = 樣品交客戶確認；大貨印件出貨 = 正式交貨

> 舊版流程圖中「已出貨 ↔ 運送中」雙向箭頭為錯誤，正確為單向：已出貨 → 運送中 → 已送達。

## 任務（紙本工單）狀態機

> 任務是派發給各工廠的執行單位，每個任務含 1～N 筆生產任務。狀態依工廠類型有不同的終態路徑。

#### 主流程

```
待交付（待印務提供工單給工廠）
 → 已作廢（取消，尚未交付前）
 → 已交付（印務已提供工單）
 → 製作中（任一生產任務進入製作中時，由生產任務層 bubble-up）
   ↔ 異動（印務發起異動工單）
     → 已確認異動內容（生管確認後重新印製工單）
     → 回到製作中
 → [依工廠類型分流，見下]
```

> bubble-up 鏈：生產任務「製作中」→ 任務「製作中」→ 工單「製作中」→ 印件「製作中」→ 訂單「製作中」
> 任何一層「任一子項目製作中」即觸發上層進入製作中。

#### 終態路徑（依工廠類型）

| 工廠類型 | 製作中後的狀態路徑 |
|----------|-------------------|
| 自有工廠 / 加工廠 | → 已完成 |
| 外包廠 | → 運送中 → 已抵達 → 已完成 |
| 中國廠商 | → 已送集運商 → 運送中 → 已抵達 → 已完成 |

> 「已完成」代表此廠商的工序結束，材料已回場。後續場內加工為另一筆獨立的生產任務（自有工廠執行），兩者間存在任務相依性。相依機制（自動 vs 手動觸發）待 UX / 開發階段設計。

## 生產任務狀態機

> 生產任務是最小追蹤單位，用於計算報工完成數量。工廠類型於建立時由 BOM 參數設定決定，決定對應狀態路徑。
> 三種工廠類型共用一份狀態機；工廠類型控制可用的轉換路徑，UI 僅顯示對應路徑的操作按鈕。

```
待處理
 → 已作廢（尚未進入生產即取消）
 → 製作中（生產開始）
   → 已完成（自有工廠 / 加工廠；條件：所有報工完成數量 = 任務數量）
   → 運送中（外包廠；完成生產後交付物流）→ 已完成
   → 已送集運商 → 運送中（中國廠商）→ 已完成
   → 報廢（已進入生產但取消；以報工完成數量計算成本）
```

**工廠類型對應狀態路徑：**

| 工廠類型 | 路徑 |
|----------|------|
| 自有工廠 / 加工廠 | 製作中 → 已完成 |
| 外包廠 | 製作中 → 運送中 → 已完成 |
| 中國廠商 | 製作中 → 已送集運商 → 運送中 → 已完成 |

**終止狀態說明：**
- `已作廢`：尚未進入生產即取消
- `報廢`：已進入生產後取消，仍以報工完成數量計算成本
- `已完成`：所有報工完成數量 = 任務數量

> 生產任務 已完成 = 純生產進度達標。QC 另由 QC 紀錄追蹤（見下方），不在生產任務層設置 QC 狀態。

## QC 相關實體

### QC 單狀態機

由印務建立，指定本次核查的批次數量，指派 QC 人員執行盤點。

> 印件可分批出貨，因此 QC 也分批進行：每次 QC 單只針對本次印務指定的批次數量。

```
待執行（印務建立，指定任務、本次目標 QC 數量、指派 QC 人員）
 → 執行中（QC 人員開始執行盤點）
 → 已完成（QC 人員完成本次指定批次的核查，填入各生產任務通過 / 不通過數量）
```

> **QC 單完成條件 = 本次指定核查數量盤點完畢**，不是「缺口 = 0」。
> 缺口判斷在工單層計算（見下方）；QC 單本身完成後即關閉，後續批次另開新 QC 單。

**角色對應：**
- **印務**：建立 QC 單，設定本次批次目標數量與指派對象
- **QC 人員**：執行盤點，填寫結果

**QC 單完成後，工單層計算：**
```
系統計算（每次 QC 單完成後更新）：
  累計 QC 入庫數量 = Σ 所有 QC 單的通過數量

若 累計入庫 ≥ 工單目標生產數量：
  工單 → 已完成

若 累計入庫 < 工單目標生產數量：
  工單維持「製作中」，系統顯示缺口數量（= 目標 - 累計入庫）
  印務可視情況：建立新任務補足（補生產），或等待下一批 QC
```

---

### QC 紀錄（資料實體）

QC 單完成後產生，掛在任務下，記錄各「影響成品」生產任務的 QC 結果。

```
QC 紀錄 {
  任務 ID
  QC 單 ID
  明細（per 影響成品的生產任務）:
    - 生產任務 ID
    - 生產數量
    - QC 通過數量（入庫數量）
    - QC 不通過數量
}
```

**計算邏輯：**
- 不影響成品的生產任務（如採購紙張、開版）不建 QC 紀錄，不計入工單完成判斷
- 工單完成數量計算公式沿用 business-process.md Section 4.3，QC 通過數量來源改為 QC 紀錄
- 印件出貨條件：出貨數量 ≤ 累計 QC 入庫數量

**QC 完成判斷（層級傳遞，由下往上）：**
```
生產任務：QC 通過數 ≥ 生產任務目標 → 該生產任務 QC 達標

任務：所有「影響成品」的生產任務 QC 達標 → 該任務 QC 達標

工單 已完成：
  累計所有任務之 QC 入庫數量 ≥ 工單目標生產數量
  （等同：所有任務 QC 達標，且累計數量足夠）
```
