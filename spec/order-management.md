## 1. 問題陳述（Problem Statement）

### 1.1 背景與痛點

訂單從報價成交到最終出貨，橫跨業務、審稿、印務、生管、QC、出貨等多個角色。現有流程分散於 Ragic + Slack + 紙本，導致以下問題：

- **訂單進度不透明**：業務無法即時掌握稿件上傳狀態、工單生產進度與出貨情況，需主動追問各部門，每筆訂單耗費 5–10 分鐘確認進度。
- **工單分配無系統化記錄**：訂單管理人分配工單給印務透過口頭或 Slack 通知，無正式紀錄；分配錯誤或遺漏時無法追責。
- **打樣決策點缺乏系統入口**：打樣結果（OK / NG）由業務口頭通知印務，系統無記錄；開放大貨稿件上傳的動作無系統化觸發，容易混亂。
- **出貨數量防呆缺失**：分批出貨時，業務手動計算可出貨額度，多工單場景下（木桶原則）計算易出錯，偶有超出入庫數出貨的情況。
- **付款狀態與生產進度脫鉤**：收款紀錄散落於 Slack 或 Excel，業務無法快速判斷某訂單的應收款項餘額。

### 1.2 受影響的使用者

| 使用者角色 | 受影響方式 | 頻率 |
|------------|------------|------|
| **業務** | 無法即時查看訂單進度；打樣確認動作無系統化入口；出貨數量需手動計算 | 每日 10–30 次 |
| **訂單管理人** | 工單分配無系統紀錄；無法一覽全公司待分配工單清單 | 每日 5–15 次 |
| **會計** | 收款紀錄分散，對帳耗時 | 每日 3–5 次 |

### 1.3 目前解法與缺陷

- **現有解法**：Ragic 記錄訂單基本資料；生產進度透過口頭回報或 Slack；出貨由業務手工記錄；付款由 Excel 追蹤。
- **缺陷**：
  - 多系統分散，無法集中查詢；
  - 打樣結果無系統化觸發機制，容易漏觸發後續動作（如開放大貨上傳）；
  - 出貨防呆依賴人工計算，多工單場景出錯率高；
  - 工單分配無正式授權記錄，責任歸屬不清。

## 2. 目標與成功指標（Goals & Success Metrics）

### 2.1 目標

- **主要目標**：訂單全生命週期在新 ERP 系統內完整追蹤，業務能即時掌握任一訂單的稿件、生產與出貨進度。
- **次要目標**：系統化工單分配流程、出貨數量自動防呆、打樣決策點有明確系統入口。

### 2.2 成功指標（KPI）

| 指標 | 現況基準 | 目標值 | 衡量方式 |
|------|----------|--------|----------|
| 訂單進度查詢時間 | 需主動追問，平均 5–10 分鐘 | ≤ 30 秒（系統內直接查看） | 使用者問卷（上線後 2 週） |
| 工單分配系統化率 | 0%（全口頭 / Slack） | ≥ 90% | 系統日誌：有分配紀錄的工單比例 |
| 出貨數量超額錯誤率 | TBD（估計每月 1–3 件） | 0 件 | 系統防呆攔截次數 |
| 打樣結果系統記錄率 | 0% | 100% | 打樣 OK/NG 填寫率 |
| 業務滿意度 | N/A | ≥ 80% | 上線後 2 週問卷 |

### 2.3 非目標（Out of Scope）

- **工單詳細管理**：製程審核、生產任務管理、異動流程見 `spec/work-order.md`
- **QC 單管理**：QC 盤點流程獨立規劃
- **EC 訂單整合**：Phase 2 目標，本 spec 僅含線下訂單
- **完整財務 / 會計模組**：僅追蹤訂單層級收款紀錄，不包含發票開立的完整流程
- **採購 / 倉儲模組**：獨立規劃
- **客戶自助入口**：客戶端操作不在此版本範疇

## 3. 使用者故事（User Stories）

### 主線故事（業務視角）

**US-OM-A1**（P0）— 業務：從需求單轉訂單

作為 **業務**，我想要 在需求單成交後，一鍵觸發「轉訂單」，系統自動帶入需求單資料（客戶、印件規格、交期等），以便 快速完成訂單建立，不需重複輸入。

**驗收條件**：
- 點擊「轉訂單」後，系統依 `business-process.md § 2.1` 的帶入規則自動填入訂單欄位
- 帶入的欄位依規則區分唯讀（如印件規格）與可編輯（如交期）
- 訂單建立時同步建立打樣印件 + 大貨印件（依需求單印件項目逐筆建立）
- 建立完成後訂單狀態進入「報價待回簽」

---

**US-OM-A2**（P0）— 業務：查看訂單完整進度

作為 **業務**，我想要 在訂單詳情頁一次看到每個印件的稿件狀態、審稿狀態、工單生產進度與出貨狀態，以便 快速掌握訂單現況，判斷是否需要催件或協調。

**驗收條件**：
- 訂單詳情頁顯示所有印件，每個印件呈現：審稿狀態 + 印製狀態
- 可展開印件查看對應工單的生產進度
- 出貨單列表顯示每筆出貨的狀態（草稿 / 出貨中 / 已送達）
- 頁面資料即時更新（不需重新整理）

---

**US-OM-A3**（P0）— 業務：確認打樣結果並觸發後續動作

作為 **業務**，我想要 在取得打樣成品後，在系統填寫打樣結果（OK / NG-製程問題 / NG-稿件問題），系統自動執行對應後續動作，以便 確保打樣決策被正式記錄，並正確觸發後續流程。

**驗收條件**：
- 打樣印件印製狀態 = 已送達 後，業務可填寫「打樣結果」
- OK → 系統開放大貨印件稿件上傳（`file_upload_enabled` = true）
- NG-製程問題 → 系統在同一打樣印件下自動建立新打樣工單草稿，印製狀態回到「等待中」
- NG-稿件問題 → 系統提示業務將舊打樣印件標記「已棄用」，並引導建立新打樣印件流程
- 打樣結果填寫後不可修改（需聯繫印務主管）

---

**US-OM-A4**（P0）— 業務：建立出貨單（分批出貨）

作為 **業務**，我想要 在訂單製作中或製作完成後，建立出貨單並新增出貨明細（指定印件 + 數量），系統自動防呆出貨數量，以便 安排分批出貨，確保不超出已入庫數量。

**驗收條件**：
- 出貨單可在訂單「製作中」後任意時點建立
- 出貨明細防呆：本次出貨數量 ≤ 印件層級統計入庫數 - 已出貨數（依 SHP-002 規則）
- 超出時系統顯示錯誤訊息並阻止送出
- 一筆出貨單可包含多個印件明細（合併出貨）
- 出貨單完成後，系統更新印件印製狀態（出貨中 / 已送達）

---

**US-OM-A5**（P1）— 業務：記錄收款紀錄

作為 **業務**，我想要 在訂單付款頁籤新增收款紀錄（金額、日期、付款類型），以便 追蹤訂單的已收款與待收款情況，不需再靠 Excel 管理。

**驗收條件**：
- 付款頁籤顯示：訂單總額、已收款合計、待收款餘額
- 支援新增收款紀錄（金額、日期、付款方式、類型：訂金 / 中期款 / 尾款 / 補收款）
- 系統自動計算已收款合計與餘額

---

### 主線故事（訂單管理人視角）

**US-OM-B1**（P0）— 訂單管理人：分配工單草稿給印務

作為 **訂單管理人**，我想要 在「待分配工單」清單中看到所有已建立草稿但尚未指定印務的工單，並能為每張工單指定負責印務，以便 確保所有工單都有明確的執行人，不遺漏。

**驗收條件**：
- 提供「待分配工單」清單，支援按訂單 / 客戶 / 日期篩選
- 每張工單可指定一位負責印務（單選）
- 分配後系統記錄操作人、分配時間
- 分配完成後工單從待分配清單移除

---

**US-OM-B2**（P1）— 訂單管理人：查看全公司訂單進度概覽

作為 **訂單管理人**，我想要 在訂單列表中看到每筆訂單的目前狀態與生產進度摘要，以便 快速識別哪些訂單需要關注或協調。

**驗收條件**：
- 訂單列表支援按狀態、客戶、負責業務、交期篩選
- 每筆訂單顯示：訂單號、客戶、狀態、交期、工單完成進度摘要（X / Y 工單完成）

---

### 主線故事（會計視角）

**US-OM-C1**（P1）— 會計：確認線下訂單收款

作為 **會計**，我想要 在訂單付款頁籤確認業務登記的收款紀錄，更新訂單付款狀態，以便 確保金流記錄與帳本一致。

**驗收條件**：
- 會計可查看業務登記的收款紀錄並標記「已確認」
- 訂單付款狀態（`payment_status`）由會計手動更新（未付款 → 已付款）
- 付款確認後系統自動記錄操作人與時間

---

### 邊緣情境故事

**US-OM-E1**（P1）— 業務：作廢出貨單並回算數量

作為 **業務**，當出貨單需要取消（運送異常 / 客戶變更）時，我想要 作廢出貨單，系統自動回算印件可出貨額度，以便 正確反映實際已出貨數量，不造成額度浪費。

**驗收條件**：
- 出貨單「待出貨」狀態可作廢
- 作廢後系統回算：印件可出貨額度 = 入庫數 - (已出貨 - 作廢數)（依 SHP-003）
- 作廢需填寫原因，記錄於活動日誌

| FR-OM-056 | P1 | 系統應顯示每個印件的出貨進度 | 格式：已出貨 X / 目標 Y（可出貨 Z）；即時更新 |

## 5. 非功能需求（Non-Functional Requirements）

| 類型 | 需求 | 標準 |
|------|------|------|
| **效能** | 訂單詳情頁載入 | ≤ 2 秒（含印件 + 工單進度） |
| **效能** | 出貨數量防呆計算 | ≤ 1 秒（即時回應） |
| **效能** | 訂單狀態自動推進 | ≤ 1 分鐘（系統觸發後） |
| **穩定性** | 系統可用率 | ≥ 99.5%（工作時段） |
| **安全性** | 角色權限控管 | 業務僅可查看自己負責的訂單（除非被授予檢視權限）；訂單管理人可查看全公司 |
| **資料保存** | 操作紀錄保存期限 | ≥ 12 個月 |
| **資料保存** | 出貨紀錄、付款紀錄 | 永久保存（支援審計） |

## 6. UX / 互動設計

### 6.1 操作流程（User Flow）

#### 需求單轉訂單流程

```
需求單「成交」狀態
  → 業務點擊「轉訂單」
  → 系統自動帶入欄位（依 business-process.md § 2.1）
  → 業務確認 / 調整可編輯欄位（交期、付款條件等）
  → 訂單建立，同步建立打樣印件 + 大貨印件
  → 訂單進入「報價待回簽」
```

#### 打樣確認流程

```
打樣印件印製狀態 = 已送達
  → 業務點擊「填寫打樣結果」
  → 選擇結果：
      OK → 系統開放大貨印件稿件上傳
      NG-製程問題 → 系統建立新打樣工單草稿
      NG-稿件問題 → 提示標記舊印件已棄用 → 引導建立新打樣印件
```

#### 出貨單建立流程

```
訂單「製作中」（或之後狀態）
  → 業務點擊「建立出貨單」
  → 選擇出貨類型（自行 / 第三方）
  → 填寫收件資訊
  → 新增出貨明細（選擇印件 + 填寫數量）
  → 系統防呆：超額時顯示錯誤
  → 確認儲存 → 出貨單狀態「草稿」
  → 備料完成 → 更新至「待出貨」
  → 出車 / 交物流 → 更新至「出貨中 / 已出貨」
  → 確認送達 → 更新至「已送達」
```

### 6.2 介面需求

- [ ] 需要設計稿（Wireframe / Mockup）：**訂單詳情頁（含印件進度展開）、出貨單建立 / 管理、打樣結果確認對話框**
- [ ] 需要設計師介入：**是**（配合 UI 設計系統）
- [ ] 需要 UX 審查：**是**（確認印件二層展開的資訊架構）

> 參考：`code/prototype-order-management.html` v1.3 已呈現訂單列表與工單印件合併生產進度二層展開，可作為設計起點。

### 6.3 錯誤處理與例外情境

| 情境 | 系統行為 | 顯示訊息 |
|------|----------|----------|
| 出貨數量超過可出貨額度 | 阻擋儲存 | 「出貨數量超過可出貨額度（最多可出 X 件），請調整後再試。」 |
| 大貨印件尚未開放上傳時嘗試上傳稿件 | 隱藏上傳入口 / 阻擋操作 | 「大貨稿件尚未開放上傳，請等待打樣確認 OK 後由業務開放。」 |

| 出貨單作廢但已有部分明細送達 | 提示需主管確認 | 「此出貨單包含已送達的明細，作廢需主管確認。請選擇：全數作廢或部分明細作廢。」 |

### 9.2 風險與緩解策略

| 風險 | 機率 | 影響 | 緩解策略 |
|------|------|------|----------|
| 出貨防呆計算依賴 QC 入庫數，QC 模組延後可能影響出貨功能上線 | 中 | 高 | Phase 1 先以手動輸入入庫數的方式暫代，待 QC 模組完成後切換自動計算 |
| 多工單木桶原則計算邏輯複雜，開發實現有誤 | 中 | 高 | 提早進行單元測試，對照 quantity-calculation-rules.md 案例驗算 |
| 打樣結果確認後的連鎖動作（開放大貨上傳 / 建新工單）若觸發失敗 | 低 | 高 | 設計補救機制：業務可手動觸發「開放大貨上傳」；加強後端事務處理（atomic） |
| ORD-001（訂單取消）尚未確認，取消流程暫缺 | 高 | 中 | Phase 1 暫不開放訂單取消功能；確認後補充 Spec |
