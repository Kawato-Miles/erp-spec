## 1. 問題陳述（Problem Statement）

### 1.1 背景與痛點

目前需求與報價流程分散在 **Ragic + Slack** 兩個工具上：

- **Ragic**：需求單建立與報價紀錄，但狀態管理依賴人工更新、缺乏明確的角色動作觸發，業務只靠 Slack 討論確認報價細節
- **Slack**：報價討論、成本評估、議價溝通全部發生在 Slack，但無法與 Ragic 資料結構連動，事後回填 URL 常有遺漏
- **成本評估**：印務主管評估製程成本後，人工填入報價金額，無稽核軌跡

核心問題：**需求單的狀態不代表真實流程位置**，業務看狀態無法判斷「現在要做什麼」；管理層無法從系統判斷案件進度或流失原因。

### 1.2 受影響的使用者

| 使用者角色 | 受影響方式 | 頻率 |
|------------|------------|------|
| 業務 / 諮詢 | 需手動追蹤每筆需求單進度；客戶催報價時需翻 Slack 查歷史 | 每日 |
| 印務主管 | 不清楚哪些需求單待評估成本；評估後需手動通知業務 | 每日 |
| 管理層 | 無法從系統看出本月詢價量、成交率、流失原因 | 每週 |

### 1.3 目前解法與缺陷

- **現有解法（Pre-ERP）**：業務在 Ragic 建需求單 → Slack 通知印務主管評估 → Slack 議價 → 業務手動更新 Ragic 狀態（全程無紙本）
- **缺陷**：
  - 狀態更新依賴人工，漏更新時系統資料不可信
  - Slack 與 Ragic 分離，查案需跨工具來回
  - 流失原因無結構化記錄，無法累積為報價策略參考
  - 成本評估無時間戳，歷史版本無跡可查
  - 相似案件無法系統化參照，每次報價都需重新計算

> **注意**：Ragic 為遷移至新 ERP 前的歷史系統，**不納入 ERP 設計考量**。Ragic 資料保留用途為：遷移前後的指標對比基準（如系統化率、成交率等），不需要系統整合。

## 2. 目標與成功指標（Goals & Success Metrics）

### 2.1 目標

- **主要目標**：讓業務與印務主管透過新 ERP 完成需求確認 → 成本評估 → 報價提供 → 議價 → 成交 / 流失的全流程，取代 Ragic + Slack 組合
- **次要目標**：沉澱詢價與成交資料，為後續報價策略與 Pipeline 分析奠基

### 2.2 成功指標（KPI）

| 指標 | 現況基準 | 目標值（Phase 1 上線 3 個月） | 衡量方式 |
|------|----------|------------------------------|----------|
| 需求單系統化操作率 | 低（多數在 Slack 完成） | ≥ 90% 的狀態轉換在 ERP 系統內操作 | 系統狀態轉換記錄 |
| 成本評估回填率 | TBD（估計 < 70%） | ≥ 95%（必填驗證） | 欄位填寫完整率 |
| 需求單 → 訂單轉換追蹤覆蓋率 | 0%（無結構化紀錄） | 100% 的流失需填寫流失原因 | open-questions.md 待確認 |

> 注意：成交率本身不列入 Phase 1 KPI（屬 Data Governance 後的 Dashboard 優化項目，見 product-goals.md Section 3 設計決策）

### 2.3 非目標（Out of Scope）

- Pipeline 管理功能（各階段轉化率分析、Sales Forecast）
- 客戶自助查詢報價狀態
- 自動化報價生成（AI 報價）
- 財務會計整合（發票開立、收款記錄）— 僅記錄發票開立方式
- 與 Slack 的雙向同步（ERP 單向發送通知至 Slack，不支援從 Slack 回寫至 ERP）

## 3. 使用者故事（User Stories）

### 主線故事

**US-001**（P0）
作為**業務**，我想要在系統內建立需求單並填寫客戶需求規格，以便讓印務主管知道有待評估的案件，不再靠 Slack 通知。

**US-002**（P0）
作為**印務主管**，我想要在系統內看到所有「待評估報價」的需求單並填入成本評估與建議售價，以便業務可以直接對外報價，不再靠人工通知。

**US-003**（P0）
作為**業務**，我想要在系統內記錄報價提供與議價過程，並標記最終結果（成交 / 流失），以便管理層可以追蹤需求單進度。

**US-004**（P1）
作為**業務**，我想要在一張需求單下建立多個印件項目，並讓每個印件有獨立的規格描述與報價，以便一次回覆客戶的複合需求（如 6 款明信片含不同規格）。

**US-005**（P1）
作為**業務**，我想要在需求單的印件項目中上傳參考圖片或檔案，以便印務主管評估時有具體參考依據，減少來回溝通次數。

**US-006**（P1）
作為**業務**，我想要在成交後一鍵將需求單轉為訂單，並帶入需求單已填寫的客戶、印件、報價資料，以便不需要重複填寫。

**US-009**（P1）
作為**業務 / 諮詢**，我想要在新需求單中搜尋並連結過去相似的需求單作為參考，以便下次報相似案件時，可以直接查閱歷史成本與報價，不需重新估算。

**US-010**（P1）
作為**印務主管**，我想要在印件項目中選擇「預計製程（工序）」，系統自動帶入該工序的標準成本供我確認，以便不需要每次手動計算製程成本，同時提高成本估算一致性。

### 邊緣情境故事

**US-007**（P1）
作為**業務**，當議價後需要調整方案（如更換紙材降低成本），我想要讓需求單退回「待評估報價」狀態，以便印務主管重新評估，並透過報價紀錄查看歷史評估。

**US-008**（P2）
作為**業務 / 諮詢**，當客戶不成交時，我想要填寫流失原因（如價格太高、找其他廠商），以便累積報價策略參考。

## 4. 功能需求（Functional Requirements）

### 4.1 需求單基本資料管理

| ID | 優先級 | 需求描述 | 驗收條件 |
|----|--------|----------|----------|
| FR-001 | P0 | 系統應允許業務建立需求單，並自動產生流水號（格式：Q-YYYYMMDD-XX） | 建立後顯示唯一需求單號，同日第二筆為 -02 |
| FR-002 | P0 | 系統應要求填寫「需求案名、客戶、接單業務、詢價來源」為必填欄位 | 未填必填欄位時，送出按鈕顯示錯誤提示，不允許儲存 |
| FR-003 | P0 | 系統應記錄所有欄位變更至活動紀錄（actor、field、old_value、new_value、timestamp），不可人工修改 | 活動紀錄顯示每次欄位變更的前後值與操作者 |
| FR-004 | P1 | 系統應支援填寫「報價截止日」，並在截止日前 1 天通知接單業務 | 報價截止日前 1 天，業務透過 Slack webhook 收到提醒通知 |
| FR-005 | P1 | 系統應於需求單建立時，自動透過 Slack Webhook 發送通知至指定頻道，並將 Slack 訊息 permalink 回寫至 `slack_thread_url` 欄位 | 建立需求單後，`slack_thread_url` 自動填入；若通知發送失敗，業務可手動補填；點擊可正確跳轉至 Slack 訊息 |
| FR-006 | P1 | 系統應支援在需求單中搜尋並連結過去的需求單作為「相似案件參考」，一張需求單可連結多筆歷史需求單 | 搜尋框可依案名 / 客戶 / 印件類型查詢；連結後顯示被參考需求單的案名、報價時間、各印件成本與售價摘要；可解除連結 |

### 4.2 印件項目與規格管理

| ID | 優先級 | 需求描述 | 驗收條件 |
|----|--------|----------|----------|
| FR-010 | P0 | 系統應支援在一張需求單下新增多個印件項目，每個項目有獨立編號（格式：{quote_no}-{項次流水號}，例：Q-20260127-02-1） | 需求單可含至少 10 個印件項目，各項目編號唯一；編號隨 quote_no 自動產生 |
| FR-011 | P0 | 系統應要求每個印件項目必填「項目名稱、規格備註」；成本預估在觸發「評估完成」時必須有值 | 成本預估欄位若空白，不允許進入「已評估報價」狀態 |
| FR-011a | P0 | 印務主管可為每個印件項目選擇「預計製程（工序）」：有選擇時，系統自動帶入該工序的標準成本（`工序.標準成本/單位 × 數量`）至成本預估欄位（可覆寫）；未選擇時，系統以 `報價單價（未稅）× 50%` 作為建議成本（顯示提示，可覆寫或手動填入）| 預計製程下拉選單支援搜尋；更換工序後自動重新計算；提示文字：「建議成本（未選工序，以售價 50% 估算）」 |
| FR-012 | P0 | 系統應自動計算每個印件項目的金額：`費用(未稅) = 報價單價(未稅) × 數量`，以及含稅總金額 `= 費用(未稅) × (1 + 稅率)` | 修改報價單價或數量後，費用與總金額即時自動更新 |
| FR-013 | P1 | 系統應支援填寫每個印件項目的「出貨方式、包裝說明」 | 欄位為選填，可儲存空值 |
| FR-014 | P1 | 系統應支援在印件項目填寫「倍數」（加成倍數），並顯示「報價單價 = 成本預估 × 倍數」的計算參考 | 修改倍數後，UI 即時顯示對應建議報價，但報價單價可由業務覆寫 |
| FR-015 | P1 | 系統應顯示需求單的印件項目金額彙總（所有項目金額小計加總、稅額加總、含稅總金額） | 新增 / 修改 / 刪除印件項目後，彙總金額即時更新 |
| FR-016 | P1 | 系統應允許業務在每個印件項目上傳參考附件（圖片 / PDF），每個項目最多支援 10 個附件 | 上傳後顯示檔案名稱與縮圖（圖片）；印務主管查看需求單時可直接預覽或下載附件 |

### 4.3 需求單狀態管理（State Machine）

> 完整狀態設計見 `memory/erp/state-machines.md` 需求單狀態機段落。以下為功能性需求。

| ID | 優先級 | 需求描述 | 驗收條件 |
|----|--------|----------|----------|
| FR-020 | P0 | 系統應依角色限制狀態轉換按鈕的可見性與可操作性 | 業務看不到「已評估報價」的確認按鈕；印務主管看不到「成交」按鈕 |
| FR-021 | P0 | 業務執行「送印務評估」時，系統應將需求單狀態從「需求確認中」轉為「待評估報價」，並透過 Slack webhook 通知印務主管 | 狀態轉換後，印務主管收到 Slack 通知，需求單出現在印務主管的待辦清單 |
| FR-022 | P0 | 印務主管執行「評估完成」時，系統應將狀態從「待評估報價」轉為「已評估報價」，並自動建立一筆報價紀錄（QuoteRequestPriceLog） | 必須填寫所有印件項目的成本預估才允許執行；轉換後自動建立報價紀錄，通知業務 |
| FR-023 | P0 | 業務執行「對外提供報價」時，系統應將狀態從「已評估報價」轉為「已提供報價」，並記錄時間戳 | 狀態轉換後顯示「報價提供時間」欄位（系統自動記錄，不可人工修改） |
| FR-024 | P0 | 業務執行「成交」時，系統應將狀態從「議價中」轉為「成交」，並允許業務建立對應訂單 | 成交後顯示「轉建訂單」按鈕；需求單狀態變更為「成交」（狀態鎖定，不可再轉換）；其餘欄位仍可異動 |
| FR-025 | P0 | 業務執行「流失」時，系統應將狀態從「議價中」轉為「流失」，並要求填寫流失原因（選單 + 補充說明） | 未選擇流失原因不允許確認；流失後需求單狀態鎖定，不允許刪除 |
| FR-026 | P1 | 印務主管執行「退回重評估」時，系統應將狀態從「議價中」退回「待評估報價」，並在活動紀錄記錄退回原因 | 退回後需求單出現在印務主管待辦清單；歷史報價紀錄保留，可在「報價紀錄」區塊展開查看 |
| FR-027 | P1 | 系統應支援業務在「已提供報價」後直接執行「客戶回應，進入議價」，狀態轉為「議價中」 | 狀態轉換後活動紀錄自動新增一筆事件 |

## 6. UX / 互動設計

### 6.1 主線操作流程（User Flow）

```
[業務]
建立需求單（填基本資料 + 印件項目 + 可上傳參考附件）
 → 「送印務評估」→ 狀態：待評估報價
   → Slack 自動通知印務主管

[印務主管]
選擇各印件項目「預計製程」（系統自動帶入標準成本）→ 確認 / 調整成本預估 → 填寫建議售價
 → 「評估完成」→ 狀態：已評估報價（自動建立報價紀錄，含製程資訊快照）

[業務]
確認售價 → 「對外提供報價」→ 狀態：已提供報價
客戶回應 → 「進入議價」→ 狀態：議價中
  ├ 需調整方案 → 退回待評估報價（印務主管重評，新增報價紀錄）
  ├ 客戶成交 → 「成交」→ 一鍵建立訂單
  └ 客戶流失 → 「流失」（填寫流失原因）
```

### 6.2 需求單詳情頁結構

詳情頁分為以下區塊（對應截圖中的 tab/section 設計）：

| 區塊 | 說明 |
|------|------|
| **基本資料** | 需求案名、客戶、業務 / 諮詢、來源、發票方式、截止日、Slack 串、相似案件參考 |
| **印件規格** | 印件項目清單，每項含：規格備註、參考附件、數量、預計製程選擇（帶入成本）、成本預估、倍數、報價單價、費用計算；製程與售價即時連動，不拆為獨立 Tab |
| **報價紀錄** | 每次「評估完成」時的歷史快照（時間、評估者、各項目成本 / 製程 / 售價）|
| **活動紀錄** | 所有狀態變更、欄位修改的系統日誌 |

> **設計注意**：印件規格與預計製程整合於同一 Tab，讓印務主管可即時看到「選工序 → 帶入成本 → 調整售價」的連動效果，避免切換頁面造成計算中斷。

### 6.3 需求單列表頁

- 預設顯示欄位：需求單號、需求案名、客戶名稱、接單業務、狀態、報價截止日、最後更新時間
- 預設排序：最後更新時間（由新到舊）
- 支援依狀態篩選（多選）、依業務篩選、依客戶搜尋
- 印務主管視角：預設顯示「待評估報價」的需求單待辦清單

### 6.4 錯誤處理與例外情境

| 情境 | 系統行為 | 顯示訊息 |
|------|----------|----------|
| 必填欄位為空 | 阻擋送出 | 「[欄位名稱] 為必填，請填寫後再繼續。」 |
| 成本預估未填就觸發「評估完成」 | 阻擋狀態轉換 | 「請先填寫每個印件項目的成本預估，才能完成評估。」 |
| 流失未填流失原因 | 阻擋確認 | 「請選擇流失原因後才能標記為流失。」 |
| 嘗試刪除已流失的需求單 | 阻擋刪除 | 「需求單已流失，保留紀錄不允許刪除。」 |
| 報價截止日早於建立日期 | 阻擋儲存 | 「報價截止日不能早於建立日期。」 |
| 附件上傳格式不支援 | 阻擋上傳 | 「僅支援 JPG、PNG、PDF 格式，請確認後重新上傳。」 |
| Slack webhook 通知發送失敗 | 記錄失敗事件，不阻擋主流程 | 「Slack 通知發送失敗，請手動補填報價串連結。」 |

## 7. 資料模型（Data Model）

> 完整欄位定義見 [`docs/data-model.md`](docs/data-model.md)（需求單模組節）。
> 本節為 **Spec 層級補充**：必填時機、業務規則與實作注意事項；欄位結構以 data-model.md 為準。

### 本 Spec 新增的資料表（全新模組）

`需求單`（QuoteRequest）、`需求單印件項目`（QuoteRequestItem）、`印件項目附件`（QuoteRequestItemAttachment）、`報價紀錄`（QuoteRequestPriceLog）、`活動紀錄`（QuoteRequestActivityLog）、`相似案件連結`（QuoteRequestReference）

### 欄位必填時機

| 欄位（中文）| 所在資料表 | 必填時機 | 觸發 FR |
|------------|-----------|---------|---------|
| 案名、客戶、接單業務、詢問來源 | 需求單 | 建立時（前端驗證）| FR-002 |
| 成本預估 | 需求單印件項目 | 觸發「評估完成」時（所有印件項目必填）| FR-011、FR-022 |
| 流失原因 | 需求單 | 執行「流失」時（未填阻擋）| FR-025 |
| 印件快照 | 報價紀錄 | 系統自動建立（印務主管執行「評估完成」時快照，含預計製程）| FR-022 |
| 提供報價時間 | 需求單 | 系統自動記錄（執行「對外提供報價」），不可人工修改 | FR-023 |

### 業務規則補充

- **客戶 FK**：永遠取廠客管理最新資料，**不在需求單快照**（見 FR-030）
- **關聯訂單 ID**：由系統在成交轉訂單時自動寫入，不可人工設定
- **操作者**：活動紀錄中系統自動事件填 `system`
- **印件類型**：在印件項目層（非需求單主表）；待 QR-003 確認選項
- **發票類型**：繼承自客戶預設值，可於需求單層覆寫；待 QR-001 確認完整選項
- **成本預估計算規則**：有選預計製程 → 工序標準成本 × 數量（可覆寫）；未選 → 售價 × 50%（建議值，可覆寫）
- **報價快照**：`印件快照` JSON 需記錄各印件的成本預估、預計製程、倍數、報價單價
- **相似案件連結**：多對多；一張需求單可連結多筆歷史需求單，反向查詢亦顯示（被誰參考）

## 9. 相依性與風險

### 9.1 相依的功能 / 系統

- **廠客管理模組（CRM）**：業務建立需求單時從廠客管理選取客戶；需求單顯示時永遠抓最新廠客資料，廠客管理模組須先完成
- **使用者角色模組（RBAC）**：狀態轉換的角色限制依賴角色設定完成
- **訂單模組**：成交後轉建訂單功能，需訂單模組 API 完成後才能實作
- **Slack Webhook 設定**：需事先設定 Slack Incoming Webhook URL（對應各通知場景的頻道）

### 9.2 風險與緩解策略

| 風險 | 機率 | 影響 | 緩解策略 |
|------|------|------|----------|
| 業務不接受新流程，仍習慣 Slack 操作 | 高 | 高 | Phase 1 上線期間同步停用 Ragic（全面遷移至 ERP），配合操作培訓；以「需求單系統化操作率」為指標持續監控 |
| 印件規格備註為自由文字，後續難以搜尋與分析 | 中 | 中 | P2：提供常用規格快填模板；Phase 2 再評估結構化欄位 |
| 流失原因選單設計不完整，業務選「其他」率高 | 中 | 低 | 第一版參考 Slack 歷史案件定義清單，上線後 2 週回顧修正 |
| 成本評估由印務主管集中處理，形成瓶頸 | 低 | 中 | 監控「待評估報價」的平均等待時間，超過 2 天 SLA 則提醒管理層 |
| Slack Webhook 通知失敗導致印務主管未收到通知 | 低 | 中 | 失敗時記錄事件並顯示警告，業務可手動通知；定期檢查 webhook 健康狀態 |
