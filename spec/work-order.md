## 1. 問題陳述（Problem Statement）

### 1.1 背景與痛點

審稿通過後工單建立，但現有流程無法自動化追蹤製程進度、廠商分派、QC 入庫與完成度計算。具體痛點如下：

- **紙本工單分派低效**：無系統化廠商篩選機制，易遺漏重複分派。生管手工打印紙本工單，每份耗時 10–15 分鐘；按廠商分類準備多份工單需額外人工作業。
- **完成度計算複雜**：一印件多工單、多生產任務場景下，手工計算工單完成度容易出錯（目前準確度約 95%）。需採木桶原則計算，單靠人工無法即時判斷。
- **異動流程無紀錄**：QC 不通過需補生產、製程變更時，系統無法追蹤異動內容、倍數調整、完成度重算過程，難以追責。

### 1.2 受影響的使用者

| 使用者角色 | 受影響方式 | 頻率 |
|------------|------------|------|
| **印務主管** | 無法快速檢視工單製程內容、批量通過或退回；異動執行無系統確認 | 每日 10–20 次 |
| **印務** | 手動建立工單、生產任務；無法快速估算完成度；異動發起後無系統追蹤 | 每日 5–10 次 |
| **生管** | 領取紙本工單依賴印務準備；無法自動篩選廠商任務；異動確認流程無紀錄 | 每日 20–30 次 |

### 1.3 目前解法與缺陷

- **現有解法**：
  - 工單由 Ragic 系統紀錄；生管接收紙本工單或郵件通知。
  - 完成度由印務手工計算，對照 QC 紀錄估算。
  - 異動時手寫備註於紙本或口頭通知，無系統紀錄。

- **缺陷**：
  - 紙本工單無廠商篩選，生管需自行判斷哪些任務屬於自己負責的廠商。
  - 手工計算易出錯；多工單場景時，無法自動應用木桶原則。
  - 異動紀錄散亂（紙本+口頭），難以追蹤完整流程，無法審計。

## 2. 目標與成功指標（Goals & Success Metrics）

### 2.1 目標

- **主要目標**：自動化工單狀態流轉、廠商分派、完成度計算，系統化異動流程，提升製程追蹤效率 & 準確性。
- **次要目標**：減少人工誤差，提升三個角色（印務主管、印務、生管）的操作效率。

### 2.2 成功指標（KPI）

| 指標 | 現況基準 | 目標值 | 衡量方式 |
|------|----------|--------|----------|
| 工單狀態轉換自動化率 | 0%（全手工） | ≥ 95% | 系統日誌記錄自動推進比例 |
| 廠商紙本工單生成耗時 | 10–15 分鐘（手工） | ≤ 30 秒 | 記錄 UI 點擊到 PDF 生成時間 |
| 完成度計算準確度 | 95%（手工計算） | 100% | 對比測試案例 TC-Q-001～005 |
| 異動流程追蹤完整度 | 70%（紙本記錄） | 100% | 系統審計日誌（0 遺漏） |
| 印務滿意度 | N/A | ≥ 80% | 上線後 2 週問卷調查 |

### 2.3 非目標（Out of Scope）

- **與訂單 / QC 單 / 出貨單的集成邏輯**：v1.0 假設訂單、印件已定義完整；後續規劃與其他模組的集成。
- **打樣工單特殊流程**：邏輯與大貨工單相同，暫不特殊處理。
- **工單成本估算與利潤分析**：屬 Dashboard / BI 範疇，非工單管理核心。
- **多語言支持**：v2.0 規劃。

## 3. 使用者故事（User Stories）

### 主線故事

**US-A1**（P0）— 印務主管：審核工單製程

作為 **印務主管**，我想要 檢視工單草稿的製程詳情（所有任務、生產任務、倍數、affects_product 標記），以便 快速判斷製程邏輯是否正確，並及時通過或退回。

**驗收條件**：
- 工單詳情頁顯示所有任務與生產任務（含廠商、倍數、設備、影響成品標記）
- 支援批量操作（同時通過/退回多個工單）
- 退回時強制附帶備註

---

**US-B1**（P0）— 印務：建立工單與生產任務

作為 **印務**，我想要 在工單中新增/編輯/刪除生產任務，並為每個生產任務設定廠商、倍數、設備、affects_product 標記，以便 準確描述製程需求，確保生管與工廠理解正確的執行步驟。

**驗收條件**：
- 工單草稿支持新增多個生產任務（按廠商自動分類為任務層）
- 每個生產任務可填：名稱、廠商、倍數、設備（單選，可選）、affects_product
- 驗證：至少有 1 個生產任務 affects_product=TRUE

---

**US-B2**（P0）— 印務：交付生產任務給生管

作為 **印務**，我想要 按廠商篩選工單內的生產任務，自動生成 PDF 紙本工單，支持列印，以便 快速交付給對應的生管/工廠，減少誤傳或重複分派。

**驗收條件**：
- UI 提供「按廠商篩選生成紙本工單」功能
- 生成的紙本工單僅包含該廠商的生產任務
- 支援 PDF 下載、瀏覽器列印；自動記錄交付時間

---

**US-B3**（P0）— 印務：監控工單完成度

作為 **印務**，我想要 在首頁或工單詳情頁實時查看完成度進度（已完成 X / 目標 Y，缺口 Z），並支援按廠商/客戶/狀態篩選，以便 掌握最新進度，及時判斷是否需要補生產。

**驗收條件**：
- 工單詳情頁顯示「已完成 X / 目標 Y（缺口 Z）」格式
- 實時更新（每次 QC 紀錄提交後重算）
- 支援按廠商/客戶/狀態篩選、導出 CSV

---

**US-B4**（P0）— 印務：發起工單異動

作為 **印務**，我想要 在工單進入「已完成」前發起異動（新增生產任務、修改倍數、修改 affects_product），附帶異動原因備註，以便 因應 QC 不通過或客戶需求變更，系統自動記錄完整異動軌跡。

**驗收條件**：
- 工單製作中時，點擊「發起異動」按鈕
- 異動類型選項：新增生產任務、修改倍數、修改 affects_product
- 提交異動時強制填寫備註；異動提交後任務狀態自動變為「異動」
- 系統自動計算修改前後的完成度差異

---

**US-C1**（P0）— 生管：領取並排序生產任務

作為 **生管**，我想要 在系統中查看分配給我的生產任務列表（含廠商、工序、交期、優先級），並支援按狀態篩選、導出 CSV，以便 掌握當前待派工的任務，並依優先度安排生產排程。

**驗收條件**：
- 生管首頁顯示「待派工」、「進行中」、「已完成」三個分類的任務列表
- 每筆任務顯示：廠商、工序、目標數量、交期、優先級
- 支援按廠商/狀態/交期篩選；支援導出 CSV

---

**US-C2**（P0）— 生管：確認異動並決定補生產時機

作為 **生管**，我想要 檢視印務發起的異動內容（新增生產任務、修改倍數等），確認邏輯正確，然後點擊「返回製作中」決定何時啟動補生產，以便 確保異動內容符合業務邏輯，及時推進補生產流程。

**驗收條件**：
- 異動通知進入生管的「待確認異動」清單
- 異動詳情頁顯示：原任務內容、新增的生產任務、修改的倍數/affects_product
- 生管確認後，任務狀態變為「已確認異動內容」
- 點擊「返回製作中」後，系統自動重算工單完成度

---

### 邊緣情境故事

**US-005**（P1）— 異動歷史查詢

作為 **印務或印務主管**，我想要 按日期/操作人/異動類型篩選查詢異動紀錄，以便 快速追蹤製程變更歷史、確認異動原因。

## 4. 功能需求（Functional Requirements）

### 4.1 工單狀態管理

| ID | 優先級 | 需求描述 | 驗收條件 |
|----|--------|----------|----------|
| FR-001 | P0 | 系統應支持工單狀態完整流轉 | 工單可依序轉換：草稿 → 製程確認中 ↔ 重新確認 → 製程審核完成 → 工單已交付 → 製作中 ↔ 異動 → 已完成 |
| FR-002 | P0 | 系統應在狀態轉換時記錄時間戳與操作人員 | 每筆狀態轉換在系統日誌記錄時間、操作人、轉換原因 |
| FR-003 | P0 | 系統應自動判斷工單完成條件 | 當累計 QC 入庫 ≥ 工單目標時，自動轉為「已完成」狀態 |

### 4.2 廠商分類與任務聚合

| ID | 優先級 | 需求描述 | 驗收條件 |
|----|--------|----------|----------|
| FR-010 | P0 | 系統應根據廠商類型自動分類任務 | 按廠商類型（自有/加工/外包/中國）自動分類；同廠商 = 一個任務；任務內可包含多個生產任務 |
| FR-011 | P0 | 系統應支持按廠商篩選生成紙本工單 | 選擇廠商後一鍵生成 PDF，僅含該廠商的生產任務；生成耗時 ≤ 30 秒 |
| FR-012 | P0 | 系統應自動記錄工單交付時間與交付人員 | 工單交付後自動記錄時間戳、交付人、交付時間 |

### 4.3 生產任務與倍數設定

| ID | 優先級 | 需求描述 | 驗收條件 |
|----|--------|----------|----------|
| FR-020 | P0 | 系統應支持生產任務完整欄位定義 | 每個生產任務包含：名稱、廠商、倍數（quantity_per_work_order）、設備（可選）、affects_product（布林） |
| FR-021 | P0 | 系統應驗證至少 1 個生產任務 affects_product=TRUE | 若無任何生產任務標記為影響成品，系統應阻止提交審核 |
| FR-022 | P1 | 系統應建議新增生產任務的倍數值 | 新增時系統建議倍數（參照原任務平均值），供使用者參考或手動覆寫 |

### 4.4 工單完成度計算

| ID | 優先級 | 需求描述 | 驗收條件 |
|----|--------|----------|----------|
| FR-030 | P0 | 系統應實裝四層統計邏輯計算完成度 | 生產任務 → 任務 → 工單 → 印件，採木桶原則（min()），支援 affects_product 篩選 |
| FR-031 | P0 | 系統應篩選影響成品的生產任務計算倍數 | 僅 affects_product=TRUE 的生產任務納入完成度計算 |
| FR-032 | P0 | 系統應每次 QC 紀錄提交時自動重算完成度 | QC 紀錄確認後系統自動觸發重算，記錄時間戳與結果 |
| FR-033 | P1 | 系統應在 UI 顯示「缺口」而非隱藏小數 | 例：「已完成 500 / 1000（缺口 500）」，清晰呈現進度 |

### 4.5 異動流程

| ID | 優先級 | 需求描述 | 驗收條件 |
|----|--------|----------|----------|
| FR-040 | P0 | 系統應支持印務發起異動（新增生產任務、修改倍數、修改 affects_product） | 提交異動後任務狀態自動變為「異動」 |
| FR-041 | P0 | 系統應記錄異動內容的完整軌跡 | 異動類型、操作人、時間、前值、後值、備註全部記錄於系統日誌 |
| FR-042 | P0 | 系統應在異動發起時計算影響 | 提交異動時顯示修改前後的完成度差異 |
| FR-043 | P0 | 系統應支持生管確認異動並決定返回時機 | 生管確認後狀態變「已確認異動內容」；點擊「返回製作中」後自動重算完成度 |

## 6. UX / 互動設計

### 6.1 操作流程（User Flow）

#### 印務工單建立流程

```
【印務主管建立工單草稿】
     ↓
【印務接收分派】→ 【填寫生產任務】 → 【提交審核】
     ↓
【印務主管審核】 ① 通過 ──→ 【按廠商生成紙本工單】 → 【交付給生管】
          ↓
        ② 退回 → 【印務修改】 → 【重新提交】
```

#### 異動流程

```
【工單進入製作中】
     ↓
【QC 檢驗】 → 通過 → 【入庫】
     ↓
    不通過 / 客戶變更需求
     ↓
【印務發起異動】 → 【備註異動原因】 → 【生管確認】 → 【返回製作中】
     ↓
【系統重算完成度】 → 【補生產任務執行】
```

- [ ] 需要設計稿（Wireframe / Mockup）：**工單詳情頁、紙本工單預覽、異動確認對話框**
- [ ] 需要設計師介入：**是**（配合 UI 設計系統）
- [ ] 需要 UX 審查：**是**（確認異動流程的清晰性）

### 6.3 錯誤處理與例外情境

| 情境 | 系統行為 | 顯示訊息 |
|------|----------|----------|
| 使用者提交工單但未勾選任何 affects_product=TRUE | 阻擋送出 | 「至少需要 1 個生產任務標記為『影響成品』，請確認後再試。」 |
| 工單進入「已完成」後嘗試發起異動 | 阻擋操作 | 「工單已完成，無法再進行異動。若需特殊處理，請聯繫印務主管。」 |
| QC 紀錄提交時發生計算錯誤 | 顯示錯誤提示，異動不通過 | 「完成度計算出錯，請稍後再試或聯繫技術支援。」 |
| 使用者發起異動但未填寫備註 | 阻擋送出 | 「異動原因為必填欄位，請說明異動理由後再試。」 |

### 7.4 ProductionTask（生產任務）

| 欄位名稱 | 英文鍵名 | 必填 | 型別 | 說明 |
|---------|---------|:--:|------|------|
| 系統 ID | `id` | ✓ | UUID | PK；系統自動生成 |

| 工序 | `process_id` | ✓ | FK | → Process |

| 目標數量 | `target_quantity` | ✓ | int | |
| 完成數量 | `completed_quantity` | ✓ | int | 報工累計；系統自動計算 |
| 預計執行日 | `scheduled_date` | ✓ | date | |
| 建立時間 | `created_at` | ✓ | datetime | 系統自動 |
| 更新時間 | `updated_at` | ✓ | datetime | 系統自動 |
