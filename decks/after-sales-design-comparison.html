售後服務單設計方向對比

針對「訂單異動：售後服務需要主管討論 + 結案追蹤」需求，本頁對比兩個設計維度的方案，並以三個具體場景演示流程差異與業務感受。

背景共識（不論選哪條都不變）
  • 業務跟主管討論走 Slack，系統不放核可關卡
  • 退款動作走既有「退款 Payment」流程
  • 補印動作走既有「印件 + 審稿 + 工單」流程
  • 不新增 AfterSalesCase 上層容器（諮詢是售前 ≠ 售後）
  • 一個訂單通常一次售後，不像保固有多次案件

維度 1：售後服務單實體要不要從 OrderAdjustment 拆出來

方向 A沿用 OrderAdjustment，加新欄位

售後服務單 = OrderAdjustment phase=after_completion，原表加新欄位與新 enum。

資料模型變更：
OrderAdjustment 既有欄位不動 新增： adjustment_type enum 加值「不處理」「補印觸發」 resolution = 不處理 / 退款 / 補印 / 退款+補印 closure_status = 處理中 / 已結案 closed_at = 結案時間 closed_by = 結案人 允許 amount = 0（破壞既有 「OrderAdjustment 必為金額異動」邊界）
取捨：實體最少改動，但 OrderAdjustment 同時承擔「金額異動」與「售後事件容器」雙重角色，語意變肥；amount=0 也打破現有 scenarios.md 設計邊界。

方向 B獨立 AfterSalesTicket 實體

售後服務單 = 新實體 AfterSalesTicket；OrderAdjustment 回歸「純金額異動」（during_order 用）。

資料模型變更：
新實體 AfterSalesTicket： id, order_id opened_at, opened_by customer_complaint (text) resolution = 不處理 / 退款 / 補印 / 組合 related_adjustments = 0..N OrderAdjustment related_print_items = 0..N PrintItem (補印) closure_status, closed_at, closed_by OrderAdjustment 改動： adjustment_phase 移除 after_completion （或保留但僅用於跨期執行警示）
取捨：多一個實體，但語意乾淨；「不處理」「補印觸發」「結案追蹤」都有專屬欄位承載，OrderAdjustment 不變肥。
關於「沿用既有流程」原則：方向 B 雖然新增 AfterSalesTicket 實體，但下游動作（退款 Payment、補印 PrintItem、SalesAllowance）完全沿用既有流程。AfterSalesTicket 只是「案件容器」，不新建任何售後專用的金額 / 退款 / 補印流程。所以兩個方向都符合「沿用既有」原則，只是抽象層次不同。

三個場景演示

場景 1：客戶投訴瑕疵，公司認賠不處理
已完成訂單 ORD-001，客戶反映印件有輕微瑕疵但不嚴重，業務跟主管討論後決定「公司認賠、不退款不補印」。

方向 A建一張 amount=0 的 OrderAdjustment

業務於訂單詳情頁點「建立售後服務單」 ↓ 填表單： adjustment_type = 不處理 amount = 0 resolution = 不處理 reason = 「客戶投訴瑕疵，公司認賠」 ↓ 業務點「執行」（無主管核可） ↓ closure_status 自動 = 已結案 （不處理沒有下游動作，立刻結案） ↓ 訂單應收總額不變
痛點：三方對帳檢視會看到一張 amount=0 的「已執行 OrderAdjustment」，這違反「OrderAdjustment 必為金額異動」的 scenarios.md 設計邊界。對帳警示 banner 觸發條件（executed_at > completed_at）會誤觸發。

方向 B建一張 AfterSalesTicket，零關聯動作

業務於訂單詳情頁點「建立售後服務單」 ↓ 填表單： customer_complaint = 「客戶投訴瑕疵」 resolution = 不處理 ↓ 業務點「結案」 ↓ closure_status = 已結案 （無 OrderAdjustment、無 PrintItem、無 Payment） ↓ 訂單應收 / 對帳完全不動 活動紀錄寫入「售後服務：不處理」
優點：沒有任何「假裝有動作」的 OrderAdjustment 污染對帳。售後服務單是獨立 ticket，不處理是合法結局。
場景 2：客戶要求退款 5,000 元
已完成訂單 ORD-002（應收 100,000、已開發票、已收款），客戶投訴部分瑕疵，業務跟主管討論後決定退 5,000。

方向 AOrderAdjustment 同時是案件 + 動作

業務建「售後服務單」 adjustment_type = 退印 amount = -5,000 resolution = 退款 ↓ 業務點「執行」 → OrderAdjustment.status = 已執行 → 訂單應收 = 95,000 ↓ 業務於發票區建退款 Payment(-5,000) 業務視需要開 SalesAllowance ↓ 業務於同一張 OrderAdjustment 點「結案」 → closure_status = 已結案 UI：訂單詳情頁有一張售後服務單， 點開可看到金額 + 結案狀態 + 關聯退款 Payment
痛點：同一張 OrderAdjustment 同時表達「金額 -5,000」與「案件結案」兩個概念。如果業務忘記點「結案」，OrderAdjustment 永遠掛著未結案，但對帳上又已執行，狀態混淆。

方向 BTicket 是案件，OrderAdjustment 是動作

業務建「售後服務單」(AfterSalesTicket) customer_complaint = 「客戶投訴瑕疵」 resolution = 退款 ↓ 系統提示「請建立退款異動單」 業務點「建立關聯異動單」 → OrderAdjustment 自動帶入 ticket_id → adjustment_type = 退印 → amount = -5,000 → 直接「已執行」(無核可關卡) → 訂單應收 = 95,000 ↓ 業務建退款 Payment(-5,000) 業務視需要開 SalesAllowance ↓ 業務於 AfterSalesTicket 點「結案」 → closure_status = 已結案 UI：訂單詳情頁有一張 ticket， ticket 內看到關聯的退款異動單 + 退款 Payment
優點：金額異動 / 案件結案兩個概念實體分離。OrderAdjustment 仍是純粹的金額異動單，三方對帳邏輯不受影響。Ticket 結案與否完全獨立追蹤。
場景 3：客戶要求補印（公司認賠 100 份）
已完成訂單 ORD-003，客戶反映 100 份瑕疵需重做，公司認賠補印。業務跟主管討論後決定補印 100 份不收費。

方向 A用 OrderAdjustment 觸發補印

業務建「售後服務單」 adjustment_type = 補印觸發 (新 enum) amount = 0 (公司認賠不收費) resolution = 補印 closure_status = 處理中 ↓ 業務點「執行」 → 系統提示「請手動建立補印印件」 ↓ 業務於訂單詳情頁手動建 PrintItem(補印100份) PrintItem 跟 OrderAdjustment 沒有直接關聯 （僅靠人腦記憶連結） ↓ PrintItem 走審稿 → 工單 → 出貨 ↓ 業務確認客戶收件滿意後 回到 OrderAdjustment 點「結案」 → closure_status = 已結案 UI 痛點：補印印件與 OrderAdjustment 在資料上沒有關聯，只能靠日期猜
痛點：(1) OrderAdjustment 又出現 amount=0；(2) 補印 PrintItem 跟 OrderAdjustment 沒有資料關聯，無法自動推導「補印完成 → 案件結案」；(3)「補印觸發」這個 type 語意奇怪——OrderAdjustment 本來是金額異動載具，不該管生產動作。

方向 BTicket 直接關聯補印印件

業務建「售後服務單」(AfterSalesTicket) customer_complaint = 「客戶要求補印100份」 resolution = 補印 closure_status = 處理中 ↓ 業務於 ticket 內點「建立補印印件」 → 系統建 PrintItem(補印100份) → PrintItem.related_after_sales_ticket_id = ticket → 不建任何 OrderAdjustment（amount=0 公司認賠） ↓ PrintItem 走審稿 → 工單 → 出貨 系統可自動偵測：補印印件出貨完成 ↓ 方案 a：自動結案 ticket.closure_status = 已結案 方案 b：通知業務確認結案 「補印印件已出貨，請確認客戶滿意後結案」 ↓ ticket 結案 UI 優點：ticket 內直接看到關聯的補印印件 狀態，一頁掌握進度
優點：(1) 沒有 amount=0 OrderAdjustment 污染；(2) ticket 與補印印件有明確 FK 關聯，自動推導結案邏輯可行；(3) 補印若客戶承擔費用，加掛一張正常 OrderAdjustment(amount=+補印費) 即可，與 ticket 平行。

維度 2：訂單狀態怎麼表達「售後處理中」

解讀 1狀態機新增「售後處理中」階段

Order.status 推進： ... 出貨中 → 已完成 → 售後處理中 → 售後完成
影響：v1.2 的 adjustment_phase 推算邏輯（依 Order.status=已完成 判定）要重寫；對帳警示 banner 的「executed_at > completed_at」需重新定義 completed_at 是哪個時間點。多個邊界要動。

解讀 2狀態不變，UI 徽章推導

Order.status = 已完成 (永遠不變) UI 推導： 訂單有未結案售後 → 顯示「售後中」徽章 訂單售後全結案 → 顯示「已結案」徽章
影響最小：不動狀態機、不動 adjustment_phase、不動對帳邏輯。售後處理時間戳記在 ticket 上，要查歷史用「訂單關聯的 ticket 列表」。

解讀 3parent + sub-status

Order.status = 已完成 (parent, 不變) Order.sub_status = 無 / 售後處理中 / 售後完成 兩維度獨立追蹤
影響：新欄位、UI 多一層顯示。比解讀 2 多了正式的 sub-status 概念，但對印刷業 ERP 場景而言可能過度設計（一個訂單通常一次售後）。

UI 上長什麼樣（訂單列表頁）

解讀 1狀態欄就是售後狀態

訂單列表
ORD-001    已完成    客戶 A
ORD-002    售後處理中    客戶 B
ORD-003    售後完成    客戶 C

優點：狀態欄一目了然
缺點：「售後完成」是不是訂單終態？業務模型彈性差

解讀 2主狀態 + 售後欄位

訂單列表
ORD-001    已完成    ——    客戶 A
ORD-002    已完成    售後中    客戶 B
ORD-003    已完成    已結案    客戶 C

優點：主狀態保持完整，售後是附加維度
缺點：列表多一欄

解讀 3主狀態 + sub-status

訂單列表
ORD-001    已完成 / ——    客戶 A
ORD-002    已完成 / 售後處理中    客戶 B
ORD-003    已完成 / 售後完成    客戶 C

優點：與解讀 2 視覺相近
缺點：要新增 sub_status 欄位，正式化代價較高

我的傾向組合

建議組合：方向 B + 解讀 2

方向 B（獨立 AfterSalesTicket 實體）— 因為三個場景中只有方向 B 的語意是乾淨的：「不處理」不會建出 amount=0 OrderAdjustment 污染對帳；補印有明確 FK 關聯方便自動推導結案；OrderAdjustment 回歸純金額異動，scenarios.md 既有設計邊界不被破壞。

解讀 2（訂單狀態不變，UI 徽章推導）— 不動 v1.2 既有設計邊界（adjustment_phase、對帳警示、三方對帳）。售後狀態靠「訂單關聯 ticket 的結案狀態」推導，列表頁加一欄「售後」徽章即可。
如果你想最小變更：方向 A + 解讀 2

保留 OrderAdjustment 雙重身份不拆，純粹加 resolution / closure_status 欄位。代價是接受三個瑕疵：(1) amount=0 OrderAdjustment 進對帳；(2) 補印無 FK 關聯；(3)「補印觸發」這個 type 語意尷尬。

下一步建議

  1. 看完此頁，決定要走 方向 A 或 方向 B，以及 解讀 1 / 解讀 2 / 解讀 3
  2. 確認「結案」是否要支援「自動推導」（補印印件出貨完成 → ticket 自動結案）還是統一手動結案
  3. 確認「業務跟主管 Slack 討論」要不要在系統留下任何痕跡（如 ticket 上的「決議筆記」欄位）
  4. 確認後啟動 /opsx:propose 建立 change，含 spec 變更 + tasks
