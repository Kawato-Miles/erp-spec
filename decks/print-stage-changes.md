# 印製階段流程升級 — 認知對齊簡報

> 給印務、生管、師父三方的認知對齊文件
>
> 涵蓋三大主軸：EC / Ragic 工單整合、工單層級變化、報工與齊套性規則

---

## P1 — 三大變化總覽

**過去三個月，印製階段在系統層級有三件大事**

| 主軸 | 一句話說明 | 影響最深的角色 |
|------|----------|---------------|
| 1. **EC / Ragic 工單整合** | 線上單與線下單從兩套系統合併到 ERP 一套 | **[印務]** |
| 2. **工單層級變化** | 一印件從拆 N 張工單變成最多兩張（打樣 + 大貨）；任務按廠商自動拆分；紙本工單按生產任務聚合 | **[印務]** **[生管]** |
| 3. **報工與齊套性規則** | 數量統計從以工單為單位改為以印件為單位；引入「每份印件需生產數量」與「不影響成品」標記 | **[師父]** **[印務]** **[業務]** |

**為什麼一次改三件**

這三件事是**一套事**：
- 工單來源不統一（主軸 1）→ 工單層級無法標準化（主軸 2）→ 數量無法被正確統計（主軸 3）
- 反過來：先把工單層級理順（主軸 2）→ 才能規範報工與齊套性（主軸 3）→ 才需要把工單來源統一進 ERP（主軸 1）

之前各種「漏單、數量對不上、印件交期說不準、業務報價跟生產對不起來」的痛點，根源都在這三層結構性問題。

**這份簡報的目標**

讓三方在新流程上線前，先理解架構長什麼樣子，避免上線時的認知斷層。**不教操作細節**，操作教學會在上線前另外做。

## P2 — 主軸一：EC / Ragic 工單整合

**現況（As-is）**

```
線上接單（規格品 / 半客製品）：
  EC 訂單 → 印件 → EC 工單  ───►  Ragic 工單（人工複製）→ 材料 / 製程 / 工序

線下訂單（業務單）：
  Ragic 訂單 → 印件 → Ragic 工單 → 材料 / 製程 / 工序
```

**兩個痛點**

1. **EC 工單必須人工搬到 Ragic** — 線上單在 EC 建立後，印務需要把每張工單再人工複製進 Ragic、補製作細節，才能交給工廠
2. **半客製需求要再開分支** — 一筆 EC 工單常被印務拆成多筆 Ragic 工單（打樣 / 大貨 / 多廠商），中間還可能再插更多工單，全靠印務判斷

```
EC 工單 ─┬─► Ragic 工單 1
        ├─► Ragic 工單 2
        └─► Ragic 工單 3 ...（印務憑經驗拆）
```

**改造後（To-be）**

```
線上接單：
  EC 訂單 → 印件 → ERP 工單（系統自動承接，無需人工搬運）

線下訂單：
  ERP 訂單 → 印件 → ERP 工單（依商品內容自動建立）
```

**三個改進**

1. 取消 EC 工單，**所有工單統一在 ERP**，廠商一律依 ERP 的圖檔資料外包加工
2. **以印件層級**記錄並控制工單流通（誰負責、做到哪、何時交）
3. 強化工單描述能力，**每筆工單獨立決算**，不再共用 EC 工單同一筆紀錄

## P3 — 主軸二：工單層級變化（4 層 → 6 層）

**現況（As-is）4 層**

```
訂單 → 商品（印件）→ 工單 → 工序
```

一個印件按「打樣 / 大貨 × 廠商」拆成多筆工單。例：「壓克力+貼紙」會拆出：

- 自有工廠 + 加工廠 F **打樣工單**
- 外包 B 廠商 **打樣工單**
- 外包 C 廠商 **打樣工單**
- 中國 D 廠商 **打樣工單**
- 中國 E 廠商 **打樣工單**
- 自有工廠 + 加工廠 F **大貨工單**
- 外包 B 廠商 **大貨工單**（以下略同）

→ 一個印件可能對應 **10+ 張工單**，印務每張都得手動建。

**改造後（To-be）6 層**

```
訂單 → 印件 → 工單 → 任務 → 生產任務（工序）→ 紙本工單
```

同一個「壓克力+貼紙」印件變成只兩張工單：

- **打樣工單**（一張）
- **大貨工單**（一張）

各工單下：

- **任務**（按廠商自動歸類）：自有工廠+加工廠 F / 外包 B / 外包 C / 中國 D / 中國 E
- **生產任務**（工序）：A 工作、B 工作、C 工作、D 工作、E 工作、F 工作
- **紙本工單**：依生產任務輸出，**同製程內容的多印件可合併一張紙本**（省派工次數）

**三個關鍵差異**

| 項目 | As-is | To-be |
|------|-------|-------|
| 一印件的工單數 | 依廠商 × 打樣 / 大貨拆 N 張 | 最多 2 張（打樣 + 大貨） |
| 廠商歸類 | 印務手動 | 系統依工序設定自動拆分為「任務」 |
| 紙本工單 | 一工單一張 | 多印件同製程可樞紐成一張紙本 |

## P4 — 主軸三：報工與齊套性規則

**核心觀念變化：以印件為統計單位**

過去：以工單為單位算數量、出貨；改後：**以印件為單位**統計生產數量、入庫、出貨。

**新增兩個概念**

1. **每份印件需生產數量**
   - 例：1 個印件「壓克力+貼紙」 = 2 個壓克力 + 1 個立牌
   - 即生產任務 A x 2 + 生產任務 B x 1 = 1 個成品印件
2. **不影響成品**標記
   - 採購類生產任務（如：包材、附件）標記為「不影響成品」
   - 不參與齊套性計算，不卡印件完成度

**齊套性計算（印件層）**

```
印件實際生產數量 = min(各影響成品生產任務的 QC 通過數 ÷ 每份印件需生產數量)
```

**範例**：印件 1000 份，含兩個生產任務

| 生產任務 | 每份需要 | QC 通過 | 換算可成印件 |
|----------|---------|---------|-------------|
| A | 2 | 980 | 980 ÷ 2 = 490 |
| B | 1 | 1060 | 1060 ÷ 1 = 1060 |

→ 印件可成數 = min(490, 1060) = **490**

**師父端報工要記錄**

| 欄位 | 說明 |
|------|------|
| 完成數量 | 本次做完了幾個（生產任務數，不是印件數） |
| 報工紀錄 | 一個生產任務可有多筆報工（500 + 500 = 1000） |
| QC 通過 / 不通過 | QC 階段填，不是師父填 |

**業務 / 印務報價建議**

- 報價單**改用實際數量**（不再用「1 式」）
- 打樣 / 大貨建議**分開建立印件**
- 各別出貨需求時建議**分開建立印件**

## P5 — 三方日常影響

**[印務] 五件事**

1. 線上單**不用再手動搬到 Ragic** — EC 訂單進 ERP，工單系統自動建
2. 一個印件**最多兩張工單**（打樣 + 大貨），廠商分流由系統依工序自動拆「任務」，不用印務手拆
3. **任務內容看廠商歸類**，紙本工單看生產任務（同製程多印件可合併一張）
4. **報價單寫法要調整**：用實際數量、打樣 / 大貨拆開、分批出貨拆開
5. 系統提供**工單引用 / 複製功能**，相似工單建立時間降低

**[生管] 三件事**

1. **派工單位是「任務」**（按廠商歸類），不是工單
2. **紙本工單可合併**：同製程內容的多印件能樞紐成一張紙本，可單獨提供紙本、一次派工
3. 派工後系統會記錄狀態，工單內容表格顯示**製程 / 廠商 / 狀態 / 預計完成時間**

**[師父] 三件事**

1. **完成生產任務進度時必須填寫報工紀錄**（完成數量）
2. 同一個生產任務可分多次報工（如：500 + 500），系統會累加
3. 報工數量以**生產任務的數**填，不是印件數（系統會自動換算）

## P6 — 情境一：壓克力+貼紙訂單（工單拆分對照）

**訂單內容**：客戶 A 訂購「壓克力+貼紙」一批，含打樣與大貨；製作要橫跨自有工廠、外包 B、外包 C、中國 D、中國 E、加工廠 F。

**As-is 現況：印務需建 10+ 張工單**

```
訂單
└─ 印件「壓克力+貼紙」
   ├─ 打樣：自有 + 加工廠 F 工單 → A、F 工序
   ├─ 打樣：外包 B 工單 → B 工序
   ├─ 打樣：外包 C 工單 → C 工序
   ├─ 打樣：中國 D 工單 → D 工序
   ├─ 打樣：中國 E 工單 → E 工序
   ├─ 大貨：自有 + 加工廠 F 工單 → A、F 工序
   ├─ 大貨：外包 B 工單 → B 工序
   ├─ 大貨：外包 C 工單 → C 工序
   ├─ 大貨：中國 D 工單 → D 工序
   └─ 大貨：中國 E 工單 → E 工序
```

→ **印務手動建 10 張工單**，每張都要寫廠商、工序、製作細節。

**To-be 改造後：印務只建 2 張工單**

```
訂單
└─ 印件「壓克力+貼紙」
   ├─ 打樣工單（1 張）
   │   ├─ 任務：自有 + 加工廠 F → 生產任務 A、F
   │   ├─ 任務：外包 B → 生產任務 B
   │   ├─ 任務：外包 C → 生產任務 C
   │   ├─ 任務：中國 D → 生產任務 D
   │   └─ 任務：中國 E → 生產任務 E
   └─ 大貨工單（1 張）
       └─（同上結構）
```

→ **印務只建 2 張工單**；任務按廠商自動分；紙本工單按生產任務聚合輸出給生管派工。

**省了什麼**

- 印務工單建立時間從 10 次降到 2 次
- 廠商分流不再憑印務記憶（系統依工序設定）
- 印件層數量統計可直接彙整（不用跨 10 張工單對帳）

## P7 — 情境二：分批出貨數量計算（齊套性實例）

**情境**：訂單印件「壓克力+貼紙」共 **1000 份**，師父分兩次報工，要分批出貨 300 份。

**生產進度（單一工單下）**

| 生產任務 | 每份印件需生產數量 | 預計生產量 | 報工紀錄 | QC 通過 | QC 不通過 |
|---------|-------------------|-----------|---------|---------|----------|
| A（壓克力） | 2 | 2150 | 500 + 500 = 1000 | 980 | 20 |
| B（貼紙） | 1 | 1060 | 1060 | 1060 | 0 |
| C（包材，**不影響成品**） | — | 3 | 3 | 3 | 0 |

**齊套性計算（印件可成數）**

```
A 換算 = 980 ÷ 2 = 490
B 換算 = 1060 ÷ 1 = 1060
C 不影響成品，跳過
印件可成數 = min(490, 1060) = 490
```

→ 印件目前**可生成 490 份**，實際生產數量 = 490。

**分批出貨**

- 已出貨：300 份
- 入庫數量（QC 通過後入庫，假設 490 都入庫，扣損耗 10）：480
- 剩餘可出貨 = 入庫 − 已出貨 = 490 − 300 = **190**

**為什麼這個算法重要**

- 過去：印務憑感覺說「應該做了 X 份」，業務無法準確答客戶交期
- 改後：系統依齊套性公式即時算，每個生產任務報工後**印件實際生產量自動更新**
- 採購類（C）標「不影響成品」，不會因為 C 還沒到貨就卡住印件出貨

## P8 — 情境三：多印件樞紐成生產任務（紙本工單合併）

**情境**：訂單含三個印件，A 印件 100 張、B 印件 100 張、C 印件 200 張，**前兩者都要做「01 數位印刷 - 大台 - 大張」、尺寸都是 2x2**；C 印件做「02 數位印刷 - 大台 - 大張」、尺寸 3x3。

**工單行項目（印務輸入）**

| 製程 | 印件 | 內容 | 尺寸 | 數量 |
|------|------|------|------|------|
| 數位 | A 印件 | 01 數位印刷 - 大台 - 大張 | 2x2 | 100 |
| 數位 | B 印件 | 01 數位印刷 - 大台 - 大張 | 2x2 | 100 |
| 數位 | C 印件 | 02 數位印刷 - 大台 - 大張 | 3x3 | 200 |

**系統依「製程內容」樞紐出生產任務**

| 生產任務 | 涵蓋印件 | 製程內容 | 尺寸 | 合計數量 |
|---------|---------|---------|------|---------|
| 生產任務 1 | A 印件 + B 印件 | 01 數位印刷 - 大台 - 大張 | 2x2 | 200 |
| 生產任務 2 | C 印件 | 02 數位印刷 - 大台 - 大張 | 3x3 | 200 |

**紙本工單輸出**

- **紙本 1**：生產任務 1（A + B 印件，01 製程，2x2，200 張） — **可單獨提供紙本工單，與一次派工**
- **紙本 2**：生產任務 2（C 印件，02 製程，3x3，200 張）

**為什麼重要**

- 過去：A、B、C 各印件分別派工，師父要切三次機台
- 改後：A 與 B 同製程同尺寸，**合併一次派工、一次上機**，C 另外一次
- 報工時師父填生產任務 1 的完成數，**系統自動回推 A 印件、B 印件各自完成多少**（依各印件的需求量分攤）

## P9 — 常見 QA（一）：通用 + 印務

**Q1（通用）：為什麼要動這麼大？以前不就好好的？**

A：以前能跑是因為印務記憶力很好、業務跟印務私下對得起來。但只要有一筆訂單漏掉、一個印件數量對不上、客戶問交期答不出來，就要追三天。這次改不是為了改而改，是把**「靠人記」改為「靠系統算」**，讓新人也能上手、量大也不會炸。

**Q2（通用）：什麼時候上線？我手上的活會被卡到嗎？**

A：分階段推進。EC / Ragic 工單整合與工單層級變化的核心結構（一印件兩張工單、任務按廠商自動拆）已逐步在 ERP 上線。報工與齊套性規則需要報價單格式調整與業務配合，會在準備好後再正式啟動。**手上既有的活會繼續用舊流程跑完**，新流程只用在新訂單。

**Q3（印務）：我以前每次線上單都要手動拆 N 張 Ragic 工單，現在還會這樣嗎？**

A：**不會。** EC 訂單進來後 ERP 系統會直接建工單，半客製需求改在 ERP 工單內**調整內容**而不是再開分支。你需要做的：確認 EC 帶進來的內容、補製作細節、確認工序廠商設定。

**Q4（印務）：一個印件只能兩張工單（打樣 + 大貨）真的夠用嗎？萬一客戶要重新打樣呢？**

A：重新打樣時建議**獨立成新訂單**（重新打樣訂單），不在原打樣工單上累加。這樣兩張工單原則仍成立、過去版本的數量也不會被新打樣干擾。如果遇到「兩張不夠」的真實情境，請帶到流程討論再評估，但目前看到的所有案例都能用兩張處理。

**Q5（印務）：報價單我要怎麼改寫，業務會配合嗎？**

A：三條原則：
1. 用**實際數量**（如「300 個」）取代「1 式」
2. 打樣與大貨**分開建印件**（不要塞在同一個印件名稱下）
3. 分批出貨需求時也**分開建印件**

業務這邊會同步收到調整指引，配合度高低決定報工與齊套性是否能跑準 — 這是**業務、印務、工廠的共同責任**，不是任一單一角色決定。

## P10 — 常見 QA（二）：生管 + 師父

**Q6（生管）：派工單位變成「任務」（按廠商歸類），跟我以前派的工單有什麼差？**

A：本質沒差，但**自動化程度提高**：
- 以前：你看工單，自己判斷哪幾筆給哪個師父
- 改後：系統已按廠商把生產任務分到對應「任務」，你以**任務**為單位派工
- 紙本工單還是會有，但會按生產任務聚合（同製程內容多印件合併一張紙本）

**Q7（生管）：多印件合併一張紙本，師父怎麼知道每張稿要做幾張？**

A：紙本工單會列出**該生產任務涵蓋哪些印件、每個印件需多少張**。例如：
- 紙本 1：生產任務 1（A 印件 100 張 + B 印件 100 張，合計 200 張）
- 師父按紙本上機，做完 200 張後報工填「完成數量 200」
- 系統依各印件需求量自動分攤回 A、B 印件

**Q8（師父）：報工要填什麼？以前都口頭講就好**

A：**完成數量**一個欄位（生產任務數，不是印件數）。例：紙本工單寫 200 張，你做完報工填 200；如果一次只做一半，填 100 即可，下次再報剩下的 100。**口頭講的好處是快**，但發票對不起來、客戶問起來沒人答得出來；報工系統就是把這件事補上。

**Q9（師父）：「每份印件需生產數量」是什麼意思？我做 1000 張就是 1000 嗎？**

A：你**不用管「每份印件需生產數量」**那欄是給系統算的，不是給你看的。你只需要按紙本工單上的數字做，做完報實際做了幾張。例：
- 紙本寫「壓克力 2150 張」，你做完報「2150」
- 系統會自動算：2150 ÷ 2 = 1075 個成品印件可生成（扣 QC 不通過再算）
- **你只負責做與報，不用做心算**

**Q10（師父）：不影響成品的生產任務（如包材、附件）也要報工嗎？**

A：**要**。系統雖然把它標記「不影響成品」（不卡印件完成度），但仍要報工，因為：
- 採購類也要追進度（避免主成品做完了，包材還沒到）
- 數量入庫紀錄完整，會計才能核對
- 報工本身就是「完成什麼工作」的紀錄，不是「對印件貢獻多少」的紀錄
